Section 01

Spring vs Spring Boot

Spring is the framework. Spring Boot is the shortcut to using that framework without drowning in configuration. Understanding the difference is the first step.

What is Spring?

Spring Framework is a massive Java framework created in 2003 by Rod Johnson. It provides everything you need to build enterprise applications: dependency injection, database access, security, web MVC, transactions, and more. It is incredibly powerful — but it comes with a cost: configuration.

With plain Spring, you have to:

  • Write XML configuration files (sometimes hundreds of lines)
  • Manually configure a web server (download Tomcat, set it up, deploy your WAR file)
  • Manually set up database connections, transaction managers, view resolvers
  • Manually resolve dependency versions and make sure everything is compatible
Analogy: Building a House

Spring = You get all the raw materials (bricks, cement, wood, wiring, pipes). You can build anything. But you have to lay every brick, run every wire, and connect every pipe yourself.

Spring Boot = You get a pre-built house with plumbing and wiring already done. Move in and start living. If you need to change the wall color, you can. But you do not have to build from scratch.

What is Spring Boot?

Spring Boot was created in 2014 by Pivotal. It is NOT a different framework — it is an opinionated, auto-configured wrapper around Spring. It makes the same Spring Framework easy to use by:

  • Eliminating XML: No XML configuration files. Everything done through annotations and properties files.
  • Auto-configuration: It detects what libraries you have and configures them automatically.
  • Embedded server: Tomcat is built into your app. No separate install. Just run main().
  • Starters: Pre-packaged dependency bundles. Add spring-boot-starter-web and you get Spring MVC + Tomcat + Jackson in one line.
  • Sensible defaults: Port 8080, JSON as default response format, auto component scanning — all out of the box.
Plain Spring

Download Tomcat separately

Write web.xml (50+ lines)

Write dispatcher-servlet.xml

Configure DataSource bean in XML

Build WAR file, deploy to Tomcat

Resolve 20+ dependency versions manually

Spring Boot

Tomcat is embedded — just run main()

No XML files at all

One annotation: @SpringBootApplication

Set 3 lines in application.properties

Run as JAR file: java -jar app.jar

Starters handle all dependency versions

Key Insight

Spring Boot does not replace Spring. It uses Spring under the hood. Every Spring concept (beans, DI, MVC) still applies. Spring Boot just eliminates the boilerplate so you can focus on your business logic.

Section 02

Why Spring Boot?

There are many Java web frameworks. Here is why Spring Boot dominates the industry.

The 5 Killer Features

No XML
Zero XML configuration. Everything is done through Java annotations (@RestController, @Autowired, @Service) and application.properties.
Embedded Tomcat
Apache Tomcat (web server) is embedded inside your JAR file. Run your app with java -jar nimbus.jar or click Run in your IDE. No separate server install.
Auto-Config
Spring Boot scans your classpath. See MySQL driver? Auto-configures a DataSource. See Jackson? Auto-configures JSON serialization. You do not configure anything manually unless you want to override defaults.
Starters
Pre-built dependency bundles. spring-boot-starter-web pulls in Spring MVC, Tomcat, Jackson, validation — all with compatible versions. No dependency hell.
Production Ready
Built-in health checks (/actuator/health), metrics, logging, and externalized configuration. Ready for production from day one.

Industry Usage

Spring Boot is the most used Java framework in the world. Companies using it include Netflix, Amazon, Google, Uber, and almost every bank and enterprise in India. When companies say "Java developer," they mean "Spring Boot developer."

In Nimbus

Nimbus uses Spring Boot because:

  • We need a REST API backend — spring-boot-starter-web gives us everything
  • We need MySQL access — spring-boot-starter-jdbc handles it
  • We need JSON serialization — Jackson comes bundled with the web starter
  • We need to run the app easily — embedded Tomcat, just run main()
  • Zero XML. Just annotations and properties.

Click "Add Dependencies" and add these three:

spring-boot-starter-web — Spring MVC, embedded Tomcat, JSON handling with Jackson. This is the core dependency for building REST APIs.
Spring JDBC
spring-boot-starter-jdbc — JdbcTemplate, DataSource auto-configuration, connection pooling (HikariCP). For database operations without JPA complexity.

Common Mistake

Do not add dependencies you do not need yet. Start with Web + JDBC + MySQL. You can always add more later by editing pom.xml. Adding too many starters at the start leads to confusing auto-configuration errors.

pom.xml breakdown
1
<parent> ... spring-boot-starter-parent ... </parent>
Inherits from Spring Boot's parent POM. This manages ALL dependency versions for you — you never specify version numbers for Spring starters.
2
<groupId>com.nimbus</groupId>
Your organization's reverse domain name. Used as the base package for your Java code.
3
<artifactId>nimbus-api</artifactId>
The project name. This becomes the JAR file name: nimbus-api-0.0.1-SNAPSHOT.jar
4
spring-boot-starter-web (no version)
Notice: no <version> tag. The parent POM handles version resolution. This starter pulls in Spring MVC + Tomcat + Jackson.
5
<scope>runtime</scope>
The MySQL driver is only needed at runtime, not at compile time. Your code uses JDBC interfaces, not MySQL-specific classes.
6
spring-boot-maven-plugin
This plugin creates a "fat JAR" — a single JAR that contains your code + all dependencies + embedded Tomcat. Run with: java -jar nimbus-api.jar

application.properties — Configuration

This file lives in src/main/resources/. It is where you configure your app without touching Java code.

src/main/resources/application.properties 
# ====== SERVER ======
server.port=8080

# ====== DATABASE ======
spring.datasource.url=jdbc:mysql://localhost:3306/nimbus_db
spring.datasource.username=root
spring.datasource.password=yourpassword
spring.datasource.driver-class-name=com.mysql.cj.jdbc.Driver

# ====== CONNECTION POOL (HikariCP) ======
spring.datasource.hikari.maximum-pool-size=10
spring.datasource.hikari.minimum-idle=5

# ====== JSP VIEW (if using JSP) ======
spring.mvc.view.prefix=/WEB-INF/views/
spring.mvc.view.suffix=.jsp

# ====== LOGGING ======
logging.level.root=INFO
logging.level.com.nimbus=DEBUG
Properties explained
1
server.port=8080
The port your app runs on. Access at http://localhost:8080. Change to 9090 if 8080 is taken.
2
spring.datasource.url=jdbc:mysql://localhost:3306/nimbus_db
JDBC connection URL. Format: jdbc:mysql://host:port/database_name. Spring auto-creates a DataSource bean from this.
3
spring.datasource.username / password
MySQL login credentials. In production, these would be in environment variables, not hardcoded.
4
spring.datasource.hikari.maximum-pool-size=10
HikariCP (the fastest Java connection pool) maintains up to 10 database connections. Spring Boot uses HikariCP by default.
5
spring.mvc.view.prefix / suffix
If using JSP views, this tells Spring where to find them. For REST APIs (JSON only), you can skip these.
6
logging.level.com.nimbus=DEBUG
Sets logging level for your package. DEBUG shows detailed logs for your code while keeping third-party logs at INFO level.
Section 05

@SpringBootApplication

This single annotation is the magic that makes Spring Boot work. It combines three powerful annotations into one.

The Main Class

NimbusApiApplication.java 
package com.nimbus.nimbusapi;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class NimbusApiApplication {

    public static void main(String[] args) {
        SpringApplication.run(NimbusApiApplication.class, args);
    }
}

That is the entire main class. 8 lines of code to start a fully configured web application with an embedded server, database connection, and JSON handling. Let us break down what @SpringBootApplication actually does.

The Three-in-One

@SpringBootApplication is a shortcut for three annotations combined:

@Configuration
Marks this class as a source of bean definitions. It tells Spring: "This class contains configuration. Read it." Replaces the old XML configuration files.
@ComponentScan
Tells Spring to scan the current package and all sub-packages for components (@Controller, @Service, @Repository). It automatically finds and registers all your classes. This is why your package structure matters — everything must be under or inside the main class's package.
@EnableAutoConfiguration
The magic. Looks at your classpath (what JARs are in your project) and automatically configures things. MySQL driver present? Configure a DataSource. Jackson present? Configure JSON serialization. Tomcat present? Start a web server.
  @SpringBootApplication
  ├── @Configuration          --> "I am a config class"
  ├── @ComponentScan          --> "Scan com.nimbus.nimbusapi.** for beans"
  └── @EnableAutoConfiguration --> "Auto-configure based on classpath"

  Package structure matters!
  ┌─────────────────────────────────────────────────────────┐
  │  com.nimbus.nimbusapi                  <-- Main class   │
  │  com.nimbus.nimbusapi.controller       <-- Scanned ✓   │
  │  com.nimbus.nimbusapi.service          <-- Scanned ✓   │
  │  com.nimbus.nimbusapi.dao              <-- Scanned ✓   │
  │  com.other.package                     <-- NOT scanned │
  └─────────────────────────────────────────────────────────┘
ComponentScan only scans the main class's package and sub-packages
Common Mistake

If you put your main class in com.nimbus but your controllers in com.controllers, Spring Boot will NOT find them. Always keep your main class in the root package, and all other classes in sub-packages.

What Happens When main() Runs

SpringApplication.run() triggers the entire boot sequence:

  1. Creates the application context — the container that holds all your beans
  2. Component scan — finds all @Controller, @Service, @Repository classes
  3. Auto-configuration — configures DataSource, Tomcat, Jackson, etc. based on classpath
  4. Reads application.properties — applies your configuration overrides
  5. Starts embedded Tomcat — web server starts listening on port 8080
  6. App is ready — you see "Started NimbusApiApplication in X seconds" in the console

Section 06

Embedded Tomcat

One of Spring Boot's best features: the web server is inside your application, not the other way around.

Traditional vs Embedded

Traditional (Plain Spring)
  1. Download Apache Tomcat
  2. Install it on your machine
  3. Configure server.xml
  4. Build your code as a WAR file
  5. Copy the WAR to Tomcat's webapps/ folder
  6. Start Tomcat separately
  7. Debug by looking at Tomcat logs
Embedded (Spring Boot)
  1. Add spring-boot-starter-web
  2. Write your code
  3. Run main()
  4. Tomcat starts automatically on port 8080
  5. Done.

The embedded Tomcat is packaged inside the spring-boot-starter-web dependency. When you call SpringApplication.run(), Spring Boot starts Tomcat as part of your application process. Your app IS the server.

Key Insight

With embedded Tomcat, your application is a single JAR file that contains everything. Deploy to any server with java -jar nimbus-api.jar. No Tomcat install needed on the production server. This is why Spring Boot is perfect for microservices and Docker containers.

Changing the Port

application.properties 
# Default is 8080. Change if something else is using that port.
server.port=9090

# Use port 0 for random port (useful for testing)
# server.port=0
Section 07

Spring Boot Starters

Starters are pre-packaged dependency bundles. Each starter pulls in all the libraries you need for a specific task — with guaranteed compatible versions.

What Problem Do They Solve?

Without starters, building a REST API requires you to manually add:

  • spring-webmvc (version 6.1.5)
  • spring-web (version 6.1.5)
  • tomcat-embed-core (version 10.1.19)
  • tomcat-embed-el (version 10.1.19)
  • jackson-databind (version 2.16.1)
  • jackson-core (version 2.16.1)
  • jackson-annotations (version 2.16.1)
  • ... and more

And you have to make sure ALL versions are compatible with each other. One version mismatch = hours of debugging NoSuchMethodError or ClassNotFoundException.

With starters, you add ONE dependency and get everything:

Just one line in pom.xml 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-web</artifactId>
</dependency>

The Three Starters You Need to Know

StarterWhat It IncludesUsed For
spring-boot-starter-web Spring MVC, embedded Tomcat, Jackson (JSON), Bean Validation Building REST APIs and web apps. The most important starter.
spring-boot-starter-jdbc JdbcTemplate, HikariCP connection pool, Spring transaction management Database access using SQL queries. Simpler than JPA for straightforward operations.
spring-boot-starter-data-jpa Hibernate ORM, Spring Data JPA (repositories), JdbcTemplate Database access using Java objects instead of SQL. More powerful but more complex. We use JDBC in Nimbus.

Other Common Starters

StarterPurpose
spring-boot-starter-securityAuthentication, authorization, CSRF protection, session management
spring-boot-starter-validationBean Validation (JSR 380) with Hibernate Validator — @NotNull, @Size, @Email
spring-boot-starter-mailSending emails via SMTP (JavaMail)
spring-boot-starter-testJUnit 5, Mockito, Spring Test — everything for testing
spring-boot-devtoolsAutomatic restart on code change, live reload, development-time features
In Nimbus

Nimbus uses exactly 3 starters:

  • spring-boot-starter-web — for REST controllers, JSON, embedded Tomcat
  • spring-boot-starter-jdbc — for JdbcTemplate database access
  • spring-boot-devtools — for auto-restart during development

Plus mysql-connector-j as the JDBC driver (this is not a starter — it is a third-party library).

Section 08

Auto-Configuration

Auto-configuration is what makes Spring Boot feel magical. It looks at what is on your classpath and configures everything automatically.

How It Works

When your app starts, Spring Boot runs through a checklist:

Classpath ScanWhat JARs are present?
→
Condition CheckIs MySQL driver present?
→
Auto-ConfigureCreate DataSource bean
Spring Boot auto-configures based on what it finds
What Spring Boot FindsWhat It Auto-Configures
mysql-connector-j on classpath + spring.datasource.url in propertiesCreates a DataSource bean with HikariCP connection pool
spring-boot-starter-web on classpathStarts embedded Tomcat, configures DispatcherServlet, enables JSON with Jackson
spring-boot-starter-jdbc + DataSource existsCreates a JdbcTemplate bean, auto-injects into your DAOs
jackson-databind on classpath (comes with web starter)Configures ObjectMapper for JSON serialization/deserialization

The Conditional Magic

Auto-configuration uses conditional annotations under the hood:

  • @ConditionalOnClass — "Only configure this IF this class is on the classpath"
  • @ConditionalOnProperty — "Only configure this IF this property is set"
  • @ConditionalOnMissingBean — "Only configure this IF the user has NOT already configured it"

That last one is important. If you manually define a DataSource bean, Spring Boot's auto-configured one backs off. Your explicit configuration always wins over auto-configuration.

Key Insight

Auto-configuration is not magic — it is a giant checklist of "if this exists, configure that." You can see exactly what was auto-configured by adding debug=true to your application.properties. The startup log will show every auto-configuration that was applied and every one that was skipped.

Section 09

Spring Boot DevTools

DevTools makes development faster by automatically restarting your app whenever you change code. No more manual stop-and-start.

What DevTools Does

  • Automatic Restart: Detects file changes in src/main/java and restarts the app in ~2 seconds (much faster than a cold start)
  • LiveReload: Automatically refreshes your browser when static files change (HTML, CSS, JS)
  • Relaxed Binding: Disables template caching during development so you see changes immediately
  • Auto-disabled in Production: DevTools automatically deactivates when running as a packaged JAR. It only works during development.

Adding DevTools

pom.xml 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-devtools</artifactId>
    <scope>runtime</scope>
    <optional>true</optional>
</dependency>

<optional>true</optional> means this dependency is NOT passed to other projects that depend on yours. It is only for local development.

Eclipse/IntelliJ Setup

Eclipse: DevTools works out of the box. Save a file (Ctrl+S) and it auto-restarts.

IntelliJ IDEA: You need to enable "Build project automatically" in Settings → Build → Compiler, and also enable "Allow auto-make to start even if developed application is currently running" in Advanced Settings.

Step 4: A Quick Health Check Controller

Before building the full API, create a simple controller to verify the app is working:

src/main/java/com/nimbus/nimbusapi/controller/HealthController.java 
package com.nimbus.nimbusapi.controller;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;

import java.time.LocalDateTime;
import java.util.Map;

@RestController
public class HealthController {

    @GetMapping("/api/health")
    public Map<String, Object> health() {
        return Map.of(
            "status", "UP",
            "application", "nimbus-api",
            "timestamp", LocalDateTime.now().toString()
        );
    }
}
Line-by-line explanation
1
@RestController
Tells Spring: "This class handles HTTP requests and returns data directly (JSON), not HTML pages."
2
@GetMapping("/api/health")
When someone sends a GET request to /api/health, run this method.
3
public Map<String, Object> health()
Returns a Map. Spring Boot automatically converts this Map to a JSON object using Jackson.
4
Map.of("status", "UP", ...)
Java's immutable Map factory. Creates {"status": "UP", "application": "nimbus-api", "timestamp": "..."}.

Running the Application

Three ways to run a Spring Boot application. Use whichever fits your workflow.

Method 1: From IDE (Recommended for Development)

Eclipse:

  1. Right-click on NimbusApiApplication.java
  2. Select Run As → Java Application
  3. App starts. Console shows the Spring Boot banner and "Started" message.

IntelliJ IDEA:

  1. Open NimbusApiApplication.java
  2. Click the green play button next to the main() method
  3. Or press Shift + F10

Method 2: Maven Command (Terminal)

Terminal 
# Navigate to your project directory
cd nimbus-api

# Run with Maven
mvn spring-boot:run

# Or on Windows with the Maven wrapper
mvnw.cmd spring-boot:run

# On Linux/Mac with the Maven wrapper
./mvnw spring-boot:run

Method 3: JAR File (Production)

Terminal 
# Step 1: Build the JAR
mvn clean package

# Step 2: Run the JAR
java -jar target/nimbus-api-0.0.1-SNAPSHOT.jar

# Override port at runtime
java -jar target/nimbus-api-0.0.1-SNAPSHOT.jar --server.port=9090

# Override database at runtime
java -jar target/nimbus-api-0.0.1-SNAPSHOT.jar --spring.datasource.url=jdbc:mysql://prod-server:3306/nimbus_db
Key Insight

mvn clean package creates a "fat JAR" (also called "uber JAR") in the target/ folder. This JAR contains your code + all dependencies + embedded Tomcat. You can copy this single file to any server with Java 17 installed and run it. No Maven, no Tomcat install, no IDE needed.

Stopping the App

  • IDE: Click the red stop button
  • Terminal: Press Ctrl + C
