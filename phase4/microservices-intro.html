Section 01

Monolith vs Microservices

The monolith is one big application that does everything. Microservices break it into small, independent services that each do one thing well.

The Restaurant vs Food Court Analogy

Monolith = One Restaurant

One kitchen handles everything — starters, main course, desserts, drinks. One chef falls sick? The entire restaurant slows down. Want to change the dessert menu? You risk breaking the main course workflow.

One codebase. One deployment. Everything together.

Microservices = Food Court

Separate stalls for pizza, biryani, juice, dessert. Pizza stall breaks? Biryani stall keeps running. Want to add a new sushi stall? Just add it without touching the others.

Separate codebases. Independent deployments. Each does one thing.

In Nimbus

Our Nimbus project started as a monolith (one Spring Boot app with all endpoints). In this section, we learn how to split it into microservices — a User Service, Booking Service, and Notification Service — each running independently.

Section 02

When to Use Microservices

Microservices are not always better. They add complexity. The decision depends on your team size, project scale, and rate of change.

Monolith is Better When...

  • Small team (1-5 developers) — microservices overhead is not worth it
  • Early stage — you do not know your domain boundaries yet
  • Simple application — a blog, portfolio, or basic CRUD app
  • Speed matters — monolith is faster to build and deploy initially

Microservices are Better When...

  • Large team (10+ developers) — teams can work on different services independently
  • Different scaling needs — your booking service gets 100x more traffic than your notification service
  • Different tech stacks — one service in Java, another in Python, another in Go
  • Independent deployments — you want to deploy a fix to bookings without touching users
  • Fault isolation — if notifications break, bookings should still work
Reality Check

Most companies start with a monolith and migrate to microservices later when they hit scaling problems. Amazon, Netflix, and Uber all started as monoliths. Do not start with microservices just because "big companies use them." Start simple, evolve when needed.

FactorMonolithMicroservices
ComplexitySimple to build, simple to deployComplex to build, complex to manage

Section 03

Service Boundaries — How to Split

The hardest part of microservices is deciding where to draw the lines. Each service should own one business domain and its data.

The Rule: One Business Capability Per Service

A service should map to a business function, not a technical layer. You do NOT split by "controller service" and "database service." You split by what the business does.

Wrong: Technical Split

Database Service

Controller Service

Validation Service

Email Service

(Every service depends on every other)

Correct: Business Split

User Service (auth, profiles)

Booking Service (create, track, update)

Notification Service (email, SMS)

(Each service is self-contained)

Nimbus Service Boundaries

User Service
Owns: User registration, login, JWT token generation, user profiles. Database: users table. Port: 8081
Booking Service
Owns: Create bookings, update status, track parcels, booking history. Database: bookings table. Port: 8082
Notification Service
Owns: Send email/SMS notifications when booking status changes. Database: notifications table. Port: 8083
Key Insight

Each service has its own database. The User Service does not read from the bookings table directly. If it needs booking data, it calls the Booking Service's API. This is called database per service — a core microservices principle.

Section 04

Communication — REST Calls Between Services

In a monolith, one class calls another class directly. In microservices, services talk to each other over the network using REST APIs.

Synchronous Communication (REST)

The most common approach: Service A calls Service B's API and waits for the response.

BookingService calling UserService via REST 
// In BookingService (port 8082)
// Need to validate user before creating booking

@Service
public class BookingService {

    private final RestTemplate restTemplate;

    public BookingService(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    public Booking createBooking(BookingRequest request) {
        // Call User Service to verify the user exists
        String userServiceUrl = "http://localhost:8081/api/users/" + request.getUserId();
        User user = restTemplate.getForObject(userServiceUrl, User.class);

        if (user == null) {
            throw new RuntimeException("User not found");
        }

        // User exists — proceed with booking creation
        Booking booking = new Booking();
        booking.setSenderName(user.getFullName());
        // ... save booking
        return booking;
    }
}
Angular FrontendPOST /api/bookings
→
Booking Service:8082
→
User Service:8081 (verify user)
↓
Booking ServiceSaves booking to DB
→
Notification Service:8083 (send confirmation)
Service-to-service communication via REST APIs
The Downside of Synchronous Calls

If the User Service is slow or down, the Booking Service is stuck waiting. This is called cascading failure. Solutions include circuit breakers (Resilience4j), timeouts, and asynchronous messaging (RabbitMQ, Kafka) — but those are advanced topics.

Section 06

API Gateway

The API Gateway is the single entry point for all client requests. It routes each request to the correct microservice so the frontend only talks to one URL.

Why Do We Need It?

Without a gateway, the Angular frontend would need to know the URL and port of every service. With 3 services, that is 3 different URLs. With 20 services, that is 20 URLs. The gateway solves this:

Without Gateway

http://localhost:8081/api/login

http://localhost:8082/api/bookings

http://localhost:8083/api/notifications

Frontend knows every service URL. Tight coupling.

With Gateway

http://localhost:8080/api/login

http://localhost:8080/api/bookings

http://localhost:8080/api/notifications

Frontend talks to ONE URL. Gateway routes.

What Else Can a Gateway Do?

  • Routing — Send /api/users/* to User Service, /api/bookings/* to Booking Service
  • Authentication — Validate JWT tokens before forwarding requests
  • Rate Limiting — Block clients making too many requests (prevent abuse)
  • Load Balancing — Distribute requests across multiple instances of a service
  • CORS Handling — Manage cross-origin requests in one place
Key Insight

In Spring Boot, the most popular API Gateway is Spring Cloud Gateway. It is itself a Spring Boot application that routes requests based on URL patterns. Think of it as a receptionist in an office — all visitors go to the reception desk first, and the receptionist directs them to the right department.

Section 07

Service Discovery

In production, services do not always run on the same ports or machines. Service discovery lets services find each other dynamically without hardcoded URLs.

The Problem

In development, we hardcode http://localhost:8081 for the User Service. But in production, the User Service might run on 10.0.1.5:8081 today and 10.0.2.3:8081 tomorrow (after a restart or scale-up). Hardcoded URLs break.

The Solution: Registry

User ServiceRegisters itself: "I'm at 10.0.1.5:8081"
→
Eureka ServerService Registry (phone book)
←
Booking ServiceAsks: "Where is User Service?"
Each service registers itself. Other services look up the registry to find it.

Netflix Eureka is the most common service registry in the Spring Boot ecosystem. Each service registers itself on startup, and other services can look up the registry to find the current address.

Analogy: Phone Book

Imagine you move to a new house. You update your address in the phone book (Eureka). Anyone who wants to visit you looks up the phone book instead of memorizing your address. If you move again, you update the phone book, and everyone automatically knows your new address.

Section 08

Challenges of Microservices

Microservices solve some problems but create new ones. Understanding these challenges is as important as understanding the benefits.

The Big 5 Challenges

Network Failures
In a monolith, method calls cannot fail. In microservices, every service call goes over the network, which can time out, drop packets, or fail. You must handle this with retries, circuit breakers, and fallback logic.
Data Consistency
With separate databases, there is no single transaction spanning services. If the Booking Service saves a booking but the Notification Service fails to send the email, you have eventual consistency — the system is temporarily in an inconsistent state.
Debugging
A single request may flow through 3 services. When something breaks, you need distributed tracing (like Zipkin or Jaeger) to follow the request across all services and find where it failed.
Deployment Complexity
Instead of deploying one application, you deploy 3 (or 30). You need CI/CD pipelines, container orchestration (Docker + Kubernetes), and infrastructure management for each service.
Testing
Testing one service in isolation is easy. Testing the interaction between 5 services is hard. You need integration tests, contract tests, and end-to-end tests across services.
The Golden Rule

"If your team is not big enough to have dedicated DevOps, you probably do not need microservices." Microservices trade development simplicity for operational complexity. Make sure you can handle the operational cost before adopting them.

Section 09

Docker & Containers (Brief)

Docker packages your microservice into a container — a lightweight, portable box that includes everything the service needs to run: code, dependencies, runtime.

Why Containers?

Without Docker, deploying 3 microservices means installing Java, setting up databases, and configuring environments on every server. With Docker, each service is packaged into a container that runs the same way everywhere — on your laptop, on a test server, or in production.

Without Docker

"It works on my machine."

Java 17 on dev, Java 11 on server.

Different OS, different configs.

Hours spent on environment setup.

With Docker

"It runs in a container."

Same Java version everywhere.

Container is identical on all machines.

docker run nimbus-user-service — done.

Dockerfile (Nimbus User Service) 
FROM openjdk:17-slim
COPY target/user-service.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java", "-jar", "app.jar"]
Dockerfile explained
1
FROM openjdk:17-slim
Start with a base image that has Java 17 installed. slim means a minimal version (smaller download).
2
COPY target/user-service.jar app.jar
Copy the compiled JAR file from your project into the container.
3
EXPOSE 8081
Tell Docker that this container listens on port 8081.
4
ENTRYPOINT ["java", "-jar", "app.jar"]
The command to run when the container starts. Same as running java -jar app.jar in your terminal.
Key Insight

Docker is not microservices-specific. You can containerize a monolith too. But microservices + Docker is the standard combination because each service gets its own container, deployed and scaled independently.
