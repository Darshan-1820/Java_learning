Section 01

What are Services?

A service is a class that handles reusable business logic — fetching data from APIs, calculations, shared state. Components handle the UI. Services handle everything else.

The Separation

In Angular, components are responsible for displaying data and handling user interactions. But where does the data come from? Components should NOT contain HTTP calls, data processing, or business rules directly. That is the job of services.

Without Services

Every component fetches its own data. If 3 components need booking data, the HTTP call is written 3 times. Change the API URL? Fix it in 3 places.

With Services

One BookingService handles all booking API calls. All 3 components use the same service. Change the API URL? Fix it once in the service.

BookingListComponentDisplays list of bookings
→
BookingServicegetBookings(), createBooking()
→
Spring Boot API/api/bookings
↑
TrackingComponentShows booking status
→
Same BookingServicegetBookingById()
 
 
Multiple components share one service. The service is the single source of truth for API calls.
Analogy: The Library System

Think of components as readers and services as the librarian. Readers do not go into the storage room to find books themselves. They ask the librarian (service), who knows where everything is and handles all the fetching. Multiple readers use the same librarian.

Section 02

@Injectable() and Dependency Injection

Angular services use the @Injectable() decorator, and components receive them through dependency injection — a pattern you already know from Spring!

Creating a Service

booking.service.ts 
import { Injectable } from '@angular/core';

@Injectable({
  providedIn: 'root'   // Available everywhere in the app
})
export class BookingService {

  getBookings(): string {
    return "Fetching bookings...";
  }
}
Line-by-line explanation
1
@Injectable({ providedIn: 'root' })
This decorator tells Angular: "This class can be injected into other classes." providedIn: 'root' means Angular creates ONE instance for the entire app (singleton). Same concept as @Service in Spring Boot.
2
export class BookingService
export so other files can import it. The class contains all booking-related business logic and API calls.

Injecting a Service into a Component

booking-list.component.ts 
import { Component } from '@angular/core';
import { BookingService } from '../services/booking.service';

@Component({
  selector: 'app-booking-list',
  template: '<h2>Bookings</h2>'
})
export class BookingListComponent {

  // Angular automatically creates and injects the service
  constructor(private bookingService: BookingService) { }

  loadBookings() {
    this.bookingService.getBookings();
  }
}

Spring DI vs Angular DI — Side by Side

Spring Boot
@Service
public class BookingService {
  // ...
}

@RestController
public class BookingController {
  @Autowired
  private BookingService service;
}
Angular
@Injectable({ providedIn: 'root' })
export class BookingService {
  // ...
}

@Component({ ... })
export class BookingListComponent {
  constructor(
    private service: BookingService
  ) { }
}
In Nimbus

The pattern is identical. In Spring Boot, @Service + @Autowired handles DI. In Angular, @Injectable + constructor injection handles DI. Same concept, same purpose, different framework. If you understood Spring DI, you already understand Angular DI.

Section 03

HttpClient — Making API Calls

Angular's HttpClient module is how the frontend talks to the Spring Boot backend. It supports GET, POST, PUT, and DELETE — all the REST methods you learned.

Setup

app.module.ts 
import { HttpClientModule } from '@angular/common/http';

@NgModule({
  imports: [
    HttpClientModule   // Add this to enable HTTP calls
  ]
})
export class AppModule { }

All 4 HTTP Methods

HTTP Methods in Angular 
import { HttpClient } from '@angular/common/http';
import { Injectable } from '@angular/core';
import { Observable } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class ApiService {

  private baseUrl = 'http://localhost:8080/api';

  constructor(private http: HttpClient) { }

  // GET — fetch data
  getBookings(): Observable<Booking[]> {
    return this.http.get<Booking[]>(`${this.baseUrl}/bookings`);
  }

  // POST — create data
  createBooking(booking: BookingRequest): Observable<Booking> {
    return this.http.post<Booking>(`${this.baseUrl}/bookings`, booking);
  }

  // PUT — update data
  updateStatus(id: number, status: string): Observable<Booking> {
    return this.http.put<Booking>(`${this.baseUrl}/bookings/${id}/status`, { status });
  }

  // DELETE — remove data
  deleteBooking(id: number): Observable<void> {
    return this.http.delete<void>(`${this.baseUrl}/bookings/${id}`);
  }
}
Key patterns explained
1
this.http.get<Booking[]>(url)
<Booking[]> is a generic type parameter. It tells TypeScript what type the response will be. The HTTP call returns an Observable that will emit an array of Booking objects.
2
Observable<Booking[]>
HTTP calls in Angular return Observables, not Promises. An Observable is a stream of data. You must subscribe() to it to actually trigger the HTTP request and receive data.
3
`${this.baseUrl}/bookings/${id}/status`
Template literal (backticks) with variable interpolation. Builds the full URL dynamically. Same as Java string concatenation but cleaner.
Section 04

Observables — The Subscribe Pattern

Angular uses Observables (from RxJS) for all asynchronous operations. An Observable is like a newspaper subscription — you sign up, and data gets delivered to you when it arrives.

How It Works

ComponentCalls service method
→
ServiceReturns Observable
→
ComponentSubscribes to Observable
→
Data ArrivesCallback runs
The HTTP request only fires when you subscribe. No subscribe = no request.
Subscribing to an Observable 
export class BookingListComponent {
  bookings: Booking[] = [];
  errorMessage: string = '';

  constructor(private bookingService: BookingService) { }

  loadBookings() {
    this.bookingService.getBookings().subscribe({
      next: (data) => {
        // Success — data arrived from the API
        this.bookings = data;
        console.log('Loaded', data.length, 'bookings');
      },
      error: (err) => {
        // Error — API call failed
        this.errorMessage = 'Failed to load bookings';
        console.error(err);
      }
    });
  }
}
Common Mistake

Forgetting to subscribe(). If you call this.bookingService.getBookings() without .subscribe(), nothing happens. The HTTP request never fires. Always subscribe.

Analogy: Pizza Delivery

Observable = the pizza order. Subscribe = placing the order. next = the pizza arrives. error = the delivery failed. If you never call subscribe(), you are just looking at the menu — no pizza is coming.

Section 05

Building the UserService

The UserService handles authentication — login, register, and token management. It connects directly to our Spring Boot /api/login and /api/register endpoints.

services/user.service.ts 
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';
import { tap } from 'rxjs/operators';

interface LoginRequest {
  username: string;
  password: string;
}

interface LoginResponse {
  token: string;
  userId: number;
  username: string;
  role: string;
  expiresIn: number;
}

interface RegisterRequest {
  username: string;
  email: string;
  password: string;
  fullName: string;
  phone: string;
}

@Injectable({ providedIn: 'root' })
export class UserService {

  private baseUrl = 'http://localhost:8080/api';

  constructor(private http: HttpClient) { }

  // POST /api/login
  login(credentials: LoginRequest): Observable<LoginResponse> {
    return this.http.post<LoginResponse>(`${this.baseUrl}/login`, credentials)
      .pipe(
        tap(response => {
          // Store the token in localStorage on successful login
          localStorage.setItem('token', response.token);
          localStorage.setItem('userId', response.userId.toString());
          localStorage.setItem('role', response.role);
        })
      );
  }

  // POST /api/register
  register(user: RegisterRequest): Observable<any> {
    return this.http.post(`${this.baseUrl}/register`, user);
  }

  // Check if user is logged in
  isLoggedIn(): boolean {
    return !!localStorage.getItem('token');
  }

  // Get stored token
  getToken(): string | null {
    return localStorage.getItem('token');
  }

  // Get user role
  getRole(): string | null {
    return localStorage.getItem('role');
  }

  // Logout — clear everything
  logout(): void {
    localStorage.removeItem('token');
    localStorage.removeItem('userId');
    localStorage.removeItem('role');
  }
}
Key patterns explained
1
.pipe(tap(response => { ... }))
pipe lets you chain operators on an Observable. tap is a side-effect operator — it runs code (storing the token) without modifying the data flowing through. Think of it as "peek" in a stream.
2
localStorage.setItem('token', response.token)
localStorage is browser storage that persists across page refreshes. We store the JWT token here so every subsequent API request can include it in the Authorization header.
3
isLoggedIn(): boolean { return !!localStorage.getItem('token'); }
!! converts any value to a boolean. If the token exists, it returns true. If it is null, it returns false. Used by route guards to protect pages.
Section 06

Building the BookingService

The BookingService handles all parcel booking operations. It connects to every booking endpoint in our Spring Boot API.

services/booking.service.ts 
import { Injectable } from '@angular/core';
import { HttpClient } from '@angular/common/http';
import { Observable } from 'rxjs';

export interface Booking {
  id: number;
  trackingNumber: string;
  senderName: string;
  senderAddress: string;
  receiverName: string;
  receiverAddress: string;
  weight: number;
  parcelType: string;
  status: string;
  cost: number;
  createdAt: string;
  updatedAt: string;
}

export interface BookingRequest {
  senderName: string;
  senderAddress: string;
  senderPhone: string;
  receiverName: string;
  receiverAddress: string;
  receiverPhone: string;
  weight: number;
  parcelType: string;
  description?: string;
}

@Injectable({ providedIn: 'root' })
export class BookingService {

  private baseUrl = 'http://localhost:8080/api';

  constructor(private http: HttpClient) { }

  // GET /api/bookings?userId=X
  getBookings(userId: number): Observable<Booking[]> {
    return this.http.get<Booking[]>(`${this.baseUrl}/bookings?userId=${userId}`);
  }

  // GET /api/bookings/:id
  getBookingById(id: number): Observable<Booking> {
    return this.http.get<Booking>(`${this.baseUrl}/bookings/${id}`);
  }

  // POST /api/bookings
  createBooking(booking: BookingRequest): Observable<Booking> {
    return this.http.post<Booking>(`${this.baseUrl}/bookings`, booking);
  }

  // PUT /api/bookings/:id/status
  updateStatus(id: number, status: string, remarks?: string): Observable<Booking> {
    return this.http.put<Booking>(`${this.baseUrl}/bookings/${id}/status`, {
      status,
      remarks
    });
  }

  // PUT /api/bookings/:id/pickup
  markPickup(id: number, agentId: number): Observable<Booking> {
    return this.http.put<Booking>(`${this.baseUrl}/bookings/${id}/pickup`, {
      agentId,
      pickupTime: new Date().toISOString()
    });
  }

  // GET /api/bookings/history
  getHistory(): Observable<Booking[]> {
    return this.http.get<Booking[]>(`${this.baseUrl}/bookings/history`);
  }
}
In Nimbus

Notice how every method maps directly to a Spring Boot endpoint:

  • getBookings() → GET /api/bookings?userId=X
  • getBookingById() → GET /api/bookings/{id}
  • createBooking() → POST /api/bookings
  • updateStatus() → PUT /api/bookings/{id}/status
  • markPickup() → PUT /api/bookings/{id}/pickup
  • getHistory() → GET /api/bookings/history

The frontend service is the mirror of the backend controller. Same endpoints, same data shapes.

Section 07

Error Handling

API calls can fail — the server might be down, the network might drop, or the user might not be authorized. Angular provides operators to handle these gracefully.

Error Handling with catchError and retry 
import { catchError, retry } from 'rxjs/operators';
import { throwError } from 'rxjs';

@Injectable({ providedIn: 'root' })
export class BookingService {

  constructor(private http: HttpClient) { }

  getBookings(userId: number): Observable<Booking[]> {
    return this.http.get<Booking[]>(`${this.baseUrl}/bookings?userId=${userId}`)
      .pipe(
        retry(2),  // If the request fails, retry up to 2 more times
        catchError(error => {
          // Handle the error
          if (error.status === 401) {
            console.error('Not authenticated — redirect to login');
          } else if (error.status === 404) {
            console.error('Bookings not found');
          } else if (error.status === 500) {
            console.error('Server error — try again later');
          }
          // Re-throw so the component knows it failed
          return throwError(() => error);
        })
      );
  }
}
OperatorWhat It DoesWhen to Use
retry(n)Re-attempts the HTTP call up to n times on failureNetwork glitches, temporary server issues
catchErrorCatches the error and lets you handle it (log, show message, redirect)Always — every API call should have error handling
throwErrorCreates an error Observable to re-throw the error to the subscriberWhen the component also needs to know about the error
Section 08

Interceptors — Adding Auth Token Automatically

An interceptor sits between your service and the actual HTTP request. It can modify every outgoing request — perfect for automatically attaching the JWT token.

The Problem

Without an interceptor, you would have to manually add the Authorization header to every single HTTP call. With an interceptor, you write it once and it applies to all requests.

interceptors/auth.interceptor.ts 
import { Injectable } from '@angular/core';
import { HttpInterceptor, HttpRequest, HttpHandler, HttpEvent } from '@angular/common/http';
import { Observable } from 'rxjs';
import { UserService } from '../services/user.service';

@Injectable()
export class AuthInterceptor implements HttpInterceptor {

  constructor(private userService: UserService) { }

  intercept(request: HttpRequest<any>, next: HttpHandler): Observable<HttpEvent<any>> {

    // Get the token from localStorage
    const token = this.userService.getToken();

    if (token) {
      // Clone the request and add the Authorization header
      const authRequest = request.clone({
        setHeaders: {
          Authorization: `Bearer ${token}`
        }
      });
      return next.handle(authRequest);
    }

    // No token — send the original request (for login/register)
    return next.handle(request);
  }
}
Interceptor explained
1
implements HttpInterceptor
This class implements Angular's HttpInterceptor interface, which requires an intercept() method. Every HTTP request passes through this method before being sent.
2
request.clone({ setHeaders: { ... } })
HTTP requests are immutable in Angular. You cannot modify the original request. Instead, you clone() it with the new header added. This is a safety pattern.
3
next.handle(authRequest)
Passes the modified request to the next handler in the chain (or to the actual HTTP layer). Without this call, the request would never be sent.

Registering the Interceptor

app.module.ts 
import { HTTP_INTERCEPTORS } from '@angular/common/http';
import { AuthInterceptor } from './interceptors/auth.interceptor';

@NgModule({
  providers: [
    {
      provide: HTTP_INTERCEPTORS,
      useClass: AuthInterceptor,
      multi: true   // Allows multiple interceptors
    }
  ]
})
export class AppModule { }

→

→

→

The interceptor is invisible to your components and services. They never deal with tokens directly.
In Nimbus

Our Nimbus Angular app uses one AuthInterceptor that automatically adds the JWT token to every request. This means our BookingService methods are clean — they just call this.http.get() without worrying about authentication. The interceptor handles it.
