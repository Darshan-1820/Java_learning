Section 01

@RestController vs @Controller

Two annotations, two different jobs. Understanding this distinction is the first step to building REST APIs.

@Controller — Returns Views

In Phase 3, you used @Controller with Spring MVC. It returns view names — Spring looks up a JSP or Thymeleaf template and renders HTML:

@Controller — Returns HTML views 
@Controller
public class BookingController {

    @GetMapping("/bookings")
    public String showBookings(Model model) {
        List<Booking> bookings = bookingService.getAll();
        model.addAttribute("bookings", bookings);
        return "booking-list";  // Returns a VIEW NAME
        // Spring finds booking-list.jsp and renders HTML
    }
}

@RestController — Returns Data

In Phase 4, you use @RestController. It returns data directly — no view, no HTML. The return value gets automatically converted to JSON:

@RestController — Returns JSON data 
@RestController
public class BookingController {

    @GetMapping("/api/bookings")
    public List<Booking> getAllBookings() {
        return bookingService.getAll();
        // Returns the LIST DIRECTLY
        // Spring converts it to JSON automatically
        // [{"bookingId":1001,...},{"bookingId":1002,...}]
    }
}

The Technical Difference

@RestController = @Controller + @ResponseBody on every method. The @ResponseBody annotation tells Spring: "Don't look for a view — write the return value directly to the HTTP response body."

@Controller
  • Returns view names (JSP, Thymeleaf)
  • Renders HTML on the server
  • Used in MVC web applications
  • Need @ResponseBody for JSON
@RestController
  • Returns data directly (objects, lists)
  • Auto-converts to JSON via Jackson
  • Used for REST APIs
  • @ResponseBody is built-in
In Nimbus

Phase 4 Nimbus uses @RestController for everything. The backend is a pure API — it only serves JSON data. The Angular frontend consumes this data and builds the UI. This is how modern production apps work: the backend doesn't generate any HTML.

Section 02

HTTP Method Mapping Annotations

Each CRUD operation maps to an HTTP method, and each HTTP method has its own Spring annotation.

The Four Core Mappings

HTTP Method Annotation CRUD Nimbus Example
GET @GetMapping Read Get booking by ID, list all bookings
POST @PostMapping Create Create new booking, register user
PUT @PutMapping Update Update booking status, update pickup date
DELETE @DeleteMapping Delete Cancel a booking
All four mappings in one controller 
@RestController
@RequestMapping("/api/bookings")  // Base path for all endpoints
public class BookingController {

    @GetMapping           // GET /api/bookings
    public List<Booking> getAll() { ... }

    @GetMapping("/{id}")  // GET /api/bookings/1001
    public Booking getById(@PathVariable int id) { ... }

    @PostMapping          // POST /api/bookings
    public Booking create(@RequestBody Booking booking) { ... }

    @PutMapping("/{id}")  // PUT /api/bookings/1001
    public Booking update(@PathVariable int id, @RequestBody Booking booking) { ... }

    @DeleteMapping("/{id}")  // DELETE /api/bookings/1001
    public void delete(@PathVariable int id) { ... }
}
Line-by-line explanation
2
@RequestMapping("/api/bookings")
Sets the base URL path for this controller. Every method's path is appended to this. So @GetMapping("/{id}") becomes GET /api/bookings/{id}.
5
@GetMapping
No extra path — handles GET /api/bookings (the base path). Returns all bookings.
8
@GetMapping("/{id}")
Handles GET /api/bookings/1001. The {id} is a path variable — a dynamic segment. The value gets extracted and passed to the method via @PathVariable.
11
@PostMapping
Handles POST /api/bookings. POST is for creating new resources. The booking data comes in the request body as JSON, captured by @RequestBody.
14
@PutMapping("/{id}")
Handles PUT /api/bookings/1001. PUT is for updating. You need both the ID (which booking to update) and the new data (in the request body).
Section 03

@PathVariable

Extracting values from the URL path itself — like getting the booking ID from /api/bookings/1001.

Basic Usage

@PathVariable — Single variable 
@GetMapping("/api/bookings/{id}")
public Booking getBooking(@PathVariable int id) {
    // URL: GET /api/bookings/1001
    // id = 1001 (extracted from URL, converted to int)
    return bookingService.findById(id);
}

The {id} in the URL pattern is a placeholder. Spring extracts the value and passes it to the method parameter. The name must match: {id} matches @PathVariable int id.

Multiple Path Variables

@PathVariable — Multiple variables 
@GetMapping("/api/users/{userId}/bookings/{bookingId}")
public Booking getUserBooking(@PathVariable int userId,
                              @PathVariable int bookingId) {
    // URL: GET /api/users/1/bookings/1001
    // userId = 1, bookingId = 1001
    return bookingService.findByUserAndId(userId, bookingId);
}

Custom Parameter Name

When the URL placeholder name doesn't match the Java parameter name:

@PathVariable with explicit name 
@GetMapping("/api/bookings/{booking-id}")
public Booking getBooking(@PathVariable("booking-id") int bookingId) {
    // "booking-id" has a hyphen, can't be a Java variable name
    // So we map it explicitly
    return bookingService.findById(bookingId);
}
@PathVariable vs @RequestParam

@PathVariable is for identifying a specific resource: /api/bookings/1001 (which booking?).
@RequestParam is for filtering or optional parameters: /api/bookings?status=Delivered (which bookings?).
If the value identifies a unique resource, put it in the path. If it filters or modifies the query, use a query parameter.

Section 04

@RequestParam

Query parameters in the URL — the part after the ? mark. Used for filtering, searching, pagination, and optional parameters.

Basic Usage

@RequestParam — Required parameter 
@GetMapping("/api/bookings")
public List<Booking> getByUser(@RequestParam int userId) {
    // URL: GET /api/bookings?userId=1
    // userId = 1
    return bookingService.findByUserId(userId);
}

Optional Parameters with Default Values

@RequestParam — Optional with defaults 
@GetMapping("/api/bookings")
public List<Booking> searchBookings(
        @RequestParam int userId,
        @RequestParam(required = false) String status,
        @RequestParam(defaultValue = "10") int limit,
        @RequestParam(defaultValue = "0") int offset) {
    // URL: GET /api/bookings?userId=1&status=Delivered&limit=5
    // userId = 1 (required, must be present)
    // status = "Delivered" (optional, could be null)
    // limit = 5 (provided, overrides default of 10)
    // offset = 0 (not provided, uses default)
    return bookingService.search(userId, status, limit, offset);
}
Line-by-line explanation
3
@RequestParam int userId
Required by default. If the URL doesn't include ?userId=X, Spring returns a 400 Bad Request error automatically.
4
@RequestParam(required = false) String status
Optional parameter. If not provided, status will be null. Your code must handle the null case.
5
@RequestParam(defaultValue = "10") int limit
If limit isn't in the URL, it defaults to 10. Note: defaultValue is always a String — Spring converts "10" to int 10.

Multiple Values for One Parameter

@RequestParam — List of values 
@GetMapping("/api/bookings")
public List<Booking> getByStatuses(
        @RequestParam List<String> status) {
    // URL: GET /api/bookings?status=BOOKED&status=IN_TRANSIT
    // status = ["BOOKED", "IN_TRANSIT"]
    return bookingService.findByStatuses(status);
}
Section 05

@RequestBody

Receiving JSON in the request body and having Jackson automatically convert it to a Java object.

How It Works

Client sends JSON POST /api/bookings
→
Spring receives body raw JSON string
→
Jackson deserializes JSON → Java object
→
Your method gets a Booking object
@RequestBody tells Spring to read the HTTP body and convert it from JSON to the specified Java type
@RequestBody — Creating a booking 
@PostMapping("/api/bookings")
public Booking createBooking(@RequestBody Booking booking) {
    // The client sends this JSON in the request body:
    // {
    //   "senderName": "Darshan",
    //   "receiverName": "Amit",
    //   "weight": 2.5
    // }
    //
    // Jackson reads the JSON, creates a new Booking object,
    // and calls setters: setSenderName("Darshan"), etc.
    // By the time your code runs, 'booking' is a fully populated Java object.

    return bookingService.create(booking);
}
Content-Type Header Required

For @RequestBody to work, the client must send the HTTP header Content-Type: application/json. This tells Spring: "the body contains JSON, use Jackson to parse it." If this header is missing, Spring returns a 415 Unsupported Media Type error.

Common Mistake

@RequestBody is for POST and PUT requests that have a body. GET and DELETE requests typically do NOT have a body. Never use @RequestBody with @GetMapping — use @PathVariable or @RequestParam instead.

Section 06

ResponseEntity

Full control over the HTTP response — status code, headers, and body. The professional way to build API responses.

Why Not Just Return Objects?

Returning a Java object directly works, but Spring always uses 200 OK as the status code. That's wrong for many cases:

  • Creating a resource should return 201 Created
  • Deleting should return 204 No Content
  • Not found should return 404 Not Found
  • Bad input should return 400 Bad Request

ResponseEntity lets you set the exact status code, custom headers, and the response body:

ResponseEntity Builder Methods

ResponseEntity — Common patterns 
// 200 OK with body
ResponseEntity.ok(booking);
// equivalent to: ResponseEntity.status(200).body(booking)

// 201 Created with body and Location header
ResponseEntity.created(URI.create("/api/bookings/1001")).body(booking);

// 204 No Content (no body)
ResponseEntity.noContent().build();

// 404 Not Found (no body)
ResponseEntity.notFound().build();

// 400 Bad Request with error body
ResponseEntity.badRequest().body(errorResponse);

// Custom status code
ResponseEntity.status(HttpStatus.CONFLICT).body(errorResponse);  // 409

Full Example with Proper Status Codes

BookingController — With ResponseEntity 
@RestController
@RequestMapping("/api/bookings")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    @GetMapping("/{id}")
    public ResponseEntity<Booking> getById(@PathVariable int id) {
        Booking booking = bookingService.findById(id);
        if (booking == null) {
            return ResponseEntity.notFound().build();        // 404
        }
        return ResponseEntity.ok(booking);                   // 200
    }

    @PostMapping
    public ResponseEntity<Booking> create(@RequestBody Booking booking) {
        Booking created = bookingService.create(booking);
        URI location = URI.create("/api/bookings/" + created.getBookingId());
        return ResponseEntity.created(location).body(created);  // 201
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable int id) {
        bookingService.delete(id);
        return ResponseEntity.noContent().build();           // 204
    }
}
Line-by-line explanation
9
ResponseEntity<Booking>
The generic type <Booking> tells Java (and anyone reading the code) what type the response body will contain. For delete, we use <Void> because there's no body.
12
ResponseEntity.notFound().build()
Returns HTTP 404 with no body. .build() is used when there's no response body. .body() is used when there is one.
20-21
ResponseEntity.created(location).body(created)
Returns HTTP 201 with a Location header pointing to the new resource's URL, plus the created booking as the response body. This is the REST standard for resource creation.

Status Code Reference

Code Meaning When to Use
200 OK GET success, PUT success
201 Created POST success (new resource created)
204 No Content DELETE success (nothing to return)
400 Bad Request Invalid input data
404 Not Found Resource doesn't exist
409 Conflict Duplicate (e.g., email already registered)
500 Internal Server Error Unhandled exception (bug)
Section 07

Full Nimbus REST Controllers

The complete UserController and BookingController for Nimbus. Every endpoint, production-ready.

UserController

com/nimbus/controller/UserController.java 
package com.nimbus.controller;

import com.nimbus.dto.UserRegistrationRequest;
import com.nimbus.dto.LoginRequest;
import com.nimbus.dto.LoginResponse;
import com.nimbus.model.User;
import com.nimbus.service.UserService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.net.URI;

@RestController
@RequestMapping("/api/users")
public class UserController {

    @Autowired
    private UserService userService;

    // POST /api/users/register
    @PostMapping("/register")
    public ResponseEntity<User> register(
            @RequestBody UserRegistrationRequest request) {
        User created = userService.register(
            request.getName(),
            request.getEmail(),
            request.getPassword(),
            request.getRole()
        );
        URI location = URI.create("/api/users/" + created.getUserId());
        return ResponseEntity.created(location).body(created);
    }

    // POST /api/users/login
    @PostMapping("/login")
    public ResponseEntity<LoginResponse> login(
            @RequestBody LoginRequest request) {
        LoginResponse response = userService.authenticate(
            request.getEmail(),
            request.getPassword()
        );
        return ResponseEntity.ok(response);
    }
}

BookingController

com/nimbus/controller/BookingController.java 
package com.nimbus.controller;

import com.nimbus.dto.BookingRequest;
import com.nimbus.dto.BookingResponse;
import com.nimbus.dto.StatusUpdateRequest;
import com.nimbus.dto.PickupUpdateRequest;
import com.nimbus.model.Booking;
import com.nimbus.service.BookingService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.net.URI;
import java.util.List;

@RestController
@RequestMapping("/api/bookings")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    // POST /api/bookings — Create a new booking
    @PostMapping
    public ResponseEntity<BookingResponse> create(
            @RequestBody BookingRequest request) {
        BookingResponse created = bookingService.createBooking(request);
        URI location = URI.create("/api/bookings/" + created.getBookingId());
        return ResponseEntity.created(location).body(created);
    }

    // GET /api/bookings/{id} — Get booking by ID
    @GetMapping("/{id}")
    public ResponseEntity<BookingResponse> getById(@PathVariable int id) {
        BookingResponse booking = bookingService.findById(id);
        if (booking == null) {
            return ResponseEntity.notFound().build();
        }
        return ResponseEntity.ok(booking);
    }

    // GET /api/bookings?userId=1&status=DELIVERED — Get bookings for a user
    @GetMapping
    public ResponseEntity<List<BookingResponse>> getByUser(
            @RequestParam int userId,
            @RequestParam(required = false) String status) {
        List<BookingResponse> bookings;
        if (status != null) {
            bookings = bookingService.findByUserAndStatus(userId, status);
        } else {
            bookings = bookingService.findByUserId(userId);
        }
        return ResponseEntity.ok(bookings);
    }

    // PUT /api/bookings/{id}/status — Update booking status
    @PutMapping("/{id}/status")
    public ResponseEntity<BookingResponse> updateStatus(
            @PathVariable int id,
            @RequestBody StatusUpdateRequest request) {
        BookingResponse updated = bookingService.updateStatus(
            id, request.getStatus()
        );
        return ResponseEntity.ok(updated);
    }

    // PUT /api/bookings/{id}/pickup — Update pickup date
    @PutMapping("/{id}/pickup")
    public ResponseEntity<BookingResponse> updatePickup(
            @PathVariable int id,
            @RequestBody PickupUpdateRequest request) {
        BookingResponse updated = bookingService.updatePickupDate(
            id, request.getPickupDate()
        );
        return ResponseEntity.ok(updated);
    }
}
In Nimbus

Notice that the controllers are thin — they don't contain business logic. Their job is to: (1) receive the HTTP request, (2) extract the data (@PathVariable, @RequestParam, @RequestBody), (3) call the service, (4) wrap the result in a ResponseEntity with the right status code. All the real work happens in the service layer.

Section 08

DTO Pattern

Why you never expose your entity directly through the API, and how DTOs solve this.

The Problem — Exposing Entities

If you return your User entity directly, every field goes to the client — including passwords, internal IDs, and fields the client doesn't need:

Without DTO (Dangerous)

Client receives:

{
  "userId": 1,
  "name": "Darshan",
  "email": "darshan@nimbus.com",
  "password": "hashed_pw_here",
  "internalNotes": "VIP customer",
  "createdAt": "..."
}

Password leaked!

With DTO (Safe)

Client receives:

{
  "userId": 1,
  "name": "Darshan",
  "email": "darshan@nimbus.com",
  "role": "CUSTOMER"
}

Only what the client needs

What is a DTO?

A Data Transfer Object is a simple class that defines exactly what data goes in and out of the API. You create separate DTOs for requests and responses:

BookingRequest.java — What the client sends 
package com.nimbus.dto;

public class BookingRequest {
    private int userId;
    private String senderName;
    private String senderAddress;
    private String receiverName;
    private String receiverAddress;
    private double weight;
    private String pickupDate;

    // Getters and setters only
    // No business logic, no database annotations
    // Just the fields the client needs to send
}
BookingResponse.java — What the client receives 
package com.nimbus.dto;

import com.fasterxml.jackson.annotation.JsonFormat;
import java.time.LocalDateTime;

public class BookingResponse {
    private int bookingId;
    private String senderName;
    private String receiverName;
    private double weight;
    private String status;
    private String pickupDate;
    private String estimatedDelivery;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime createdAt;

    // Getters and setters
    // Notice: no userId, no internal tracking fields
    // Only what the frontend needs to display
}

Why DTOs Matter

Entity vs DTO
  • Entity = database representation. Has all fields, JPA annotations, relationships. Lives in the model package.
  • Request DTO = what the client sends. Only the fields needed to create/update. Lives in the dto package.
  • Response DTO = what the client receives. Only the fields the client should see. Lives in the dto package.

The service layer converts between entities and DTOs. This gives you complete control over your API contract — you can change your database schema without breaking the API.

Section 09

CORS

Cross-Origin Resource Sharing — why your Angular app on localhost:4200 can't talk to your Spring Boot API on localhost:8080 without configuration.

The Problem

Browsers block requests between different origins by default. An origin is the combination of protocol + domain + port:

  • http://localhost:4200 (Angular dev server) and http://localhost:8080 (Spring Boot) are different origins
  • The browser blocks Angular's API calls to Spring Boot — even though both are on your laptop
  • This is a security feature, not a bug

Solution 1: @CrossOrigin on Controller

@CrossOrigin — Per-controller 
@RestController
@RequestMapping("/api/bookings")
@CrossOrigin(origins = "http://localhost:4200")  // Allow Angular
public class BookingController {
    // All endpoints in this controller now accept requests from Angular
}

Solution 2: Global CORS Configuration (Recommended)

WebConfig.java — Global CORS config 
package com.nimbus.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")              // All API endpoints
                .allowedOrigins("http://localhost:4200")  // Angular dev server
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("*")
                .allowCredentials(true);
    }
}
Line-by-line explanation
11
registry.addMapping("/api/**")
Apply CORS rules to all URLs starting with /api/. The ** means any sub-path.
12
.allowedOrigins("http://localhost:4200")
Only allow requests from this origin. In production, this would be your real domain like "https://nimbus.com".
13
.allowedMethods("GET", "POST", "PUT", "DELETE")
Which HTTP methods to allow. Without this, browsers might block PUT/DELETE requests even if the origin is allowed.
15
.allowCredentials(true)
Allows cookies and auth headers to be sent. Needed for session-based or token-based authentication.
In Nimbus

Use the global config approach. One WebConfig class handles CORS for all controllers. During development, allow localhost:4200. In production, change it to your real domain. Never use allowedOrigins("*") in production — that opens your API to requests from any website.

Section 10

Exception Handling

When things go wrong, return structured JSON errors instead of Spring's default ugly HTML error page.

The Problem

Without custom exception handling, when your code throws an exception, Spring returns something like this:

Default Spring error (ugly, exposes internals) 
{
  "timestamp": "2026-04-23T10:35:00.000+00:00",
  "status": 500,
  "error": "Internal Server Error",
  "trace": "java.lang.NullPointerException at com.nimbus...",
  "path": "/api/bookings/9999"
}

The trace field exposes your stack trace to the client — a security risk. And the format isn't consistent across different error types.

Step 1: Custom Error Response Class

ErrorResponse.java 
package com.nimbus.exception;

import com.fasterxml.jackson.annotation.JsonFormat;
import java.time.LocalDateTime;

public class ErrorResponse {
    private int status;
    private String error;
    private String message;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime timestamp;

    public ErrorResponse(int status, String error, String message) {
        this.status = status;
        this.error = error;
        this.message = message;
        this.timestamp = LocalDateTime.now();
    }

    // Getters and setters
}

Step 2: Custom Exceptions

Custom Nimbus exceptions 
// Thrown when a resource isn't found
public class ResourceNotFoundException extends RuntimeException {
    public ResourceNotFoundException(String message) {
        super(message);
    }
}

// Thrown when input validation fails
public class InvalidInputException extends RuntimeException {
    public InvalidInputException(String message) {
        super(message);
    }
}

Step 3: @ControllerAdvice — Global Exception Handler

GlobalExceptionHandler.java 
package com.nimbus.exception;

import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

@ControllerAdvice  // Applies to ALL controllers
public class GlobalExceptionHandler {

    @ExceptionHandler(ResourceNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            ResourceNotFoundException ex) {
        ErrorResponse error = new ErrorResponse(
            404, "Not Found", ex.getMessage()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    @ExceptionHandler(InvalidInputException.class)
    public ResponseEntity<ErrorResponse> handleBadInput(
            InvalidInputException ex) {
        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", ex.getMessage()
        );
        return ResponseEntity.badRequest().body(error);
    }

    @ExceptionHandler(Exception.class)  // Catch-all for unexpected errors
    public ResponseEntity<ErrorResponse> handleGeneral(Exception ex) {
        ErrorResponse error = new ErrorResponse(
            500, "Internal Server Error",
            "Something went wrong. Please try again."
            // Don't expose ex.getMessage() for 500 errors!
        );
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
                             .body(error);
    }
}
Line-by-line explanation
8
@ControllerAdvice
Tells Spring: "This class handles exceptions for ALL controllers." Without it, you'd need @ExceptionHandler methods in every controller — duplication.
10
@ExceptionHandler(ResourceNotFoundException.class)
When any controller method throws a ResourceNotFoundException, this method catches it. Spring routes the exception here instead of returning its default error page.
29
@ExceptionHandler(Exception.class)
Catch-all handler. If no specific handler matches the exception type, this one catches it. Notice we return a generic message — never expose internal error details (like stack traces or SQL queries) to the client.

Now the Controller Stays Clean

With @ControllerAdvice in place, your controllers can throw exceptions freely — the handler catches them:

BookingService — Throws exceptions, handler catches them 
@Service
public class BookingService {

    public Booking findById(int id) {
        Booking booking = bookingDAO.findById(id);
        if (booking == null) {
            throw new ResourceNotFoundException(
                "Booking not found with ID: " + id
            );
            // GlobalExceptionHandler catches this
            // Returns 404 with: {"status":404,"error":"Not Found","message":"Booking not found with ID: 9999"}
        }
        return booking;
    }
}
In Nimbus

One GlobalExceptionHandler class handles errors for the entire API. The controller and service code never catches exceptions or builds error responses — they just throw. This keeps the code clean and ensures every error response has the same consistent JSON format that the Angular frontend can reliably parse.

  2. When do you use @PathVariable vs @RequestParam? (Path for resource identity, param for filtering/optional)
  3. Why use DTOs instead of entities? (Security, API contract stability, control over what's exposed)
  4. What does @ControllerAdvice do? (Global exception handling across all controllers)
  5. Why is CORS needed? (Browsers block cross-origin requests by default for security)
