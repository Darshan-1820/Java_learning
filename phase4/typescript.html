Section 01

What is TypeScript?

TypeScript is JavaScript with types. It was created by Microsoft and is the language Angular is built with. You write TypeScript, and it compiles (transpiles) down to plain JavaScript that browsers can run.

The Core Idea

JavaScript is dynamically typed — you can put any value into any variable and JavaScript will not complain until your code actually runs and breaks. TypeScript adds a type system on top of JavaScript so errors are caught at compile time, before your code ever runs.

JavaScript (Runtime Error)
let price = 100;
price = "free";   // No error
// ... 500 lines later ...
let total = price * 2;  // NaN! Bug found
// at RUNTIME by a user
TypeScript (Compile Error)
let price: number = 100;
price = "free";
// ERROR: Type 'string' is not
// assignable to type 'number'
// Caught BEFORE code runs

How TypeScript Works

You Write.ts files (TypeScript)
→
TSC CompilerChecks types, reports errors
→
Output.js files (JavaScript)
→
Browser / NodeRuns the JS
TypeScript is a development tool. Browsers never see TypeScript — they only run the compiled JavaScript.

Every valid JavaScript file is also valid TypeScript. TypeScript is a superset of JavaScript — it adds features on top, but everything from JavaScript still works.

In Nimbus

Our Angular frontend is written entirely in TypeScript. Every component, service, and model uses TypeScript types to catch bugs before the code ever reaches the browser. When you see a .ts file in Angular, that is TypeScript.

Section 02

Why TypeScript?

JavaScript works fine for small scripts. But when your project has 50+ files, dozens of functions, and multiple developers — types become essential.

The 4 Big Wins

Catch Bugs Early
Errors appear as red squiggles in your editor while you type, not after deployment when a user hits the bug.
Better IDE Support
VS Code gives you autocompletion, parameter hints, and inline docs because it knows every variable's type. No more guessing what a function returns.
Self-Documenting
When you see function getUser(id: number): User, you instantly know it takes a number and returns a User object. No need to read the function body.
Safer Refactoring
Rename a property and TypeScript shows you every place that breaks. In JavaScript, you find out at runtime — if you are lucky.
Analogy: Building a House

JavaScript is like building a house without blueprints. You can do it — but you might discover the bathroom is too small after the walls are up. TypeScript is the blueprint. It catches design flaws before you pour the concrete.

Who Uses TypeScript?

TypeScript is not a niche tool. It is used by:

  • Angular — built entirely in TypeScript (the framework itself)
  • VS Code — written in TypeScript by Microsoft
  • Slack, Airbnb, Google, Stripe — all use TypeScript in production
  • React projects — increasingly migrating to TypeScript
Key Insight

Angular requires TypeScript. Unlike React (where TypeScript is optional), you cannot build an Angular app without TypeScript. Learning TypeScript is not optional for Nimbus — it is a prerequisite.

Section 03

Basic Types

TypeScript has the same types as JavaScript, but you explicitly declare them. This is the foundation of everything else.

The Core Types

TypeWhat It HoldsExample
stringTextlet name: string = "Darshan";
numberAll numbers (int + float)let age: number = 22;
booleantrue or falselet isActive: boolean = true;
anyAnything (disables type checking)let data: any = "hello";
voidNo return value (for functions)function log(): void { }
nullIntentionally emptylet x: null = null;
undefinedNot yet assignedlet y: undefined = undefined;
Avoid any

Using any defeats the entire purpose of TypeScript. It tells the compiler "do not check this variable." Use it only as a last resort when you genuinely do not know the type (e.g., data from an unknown third-party API). In our Nimbus code, we avoid any completely.

Section 04

Type Annotations

Type annotations are how you tell TypeScript what type a variable, parameter, or return value should be. The syntax is : type after the name.

Variables

Variable Annotations 
// Explicit type annotation
let name: string = "Darshan";
let age: number = 22;
let isAdmin: boolean = false;

// Type inference — TypeScript figures it out automatically
let city = "Mumbai";       // TypeScript infers: string
let weight = 2.5;          // TypeScript infers: number
let active = true;         // TypeScript infers: boolean

// const is always inferred to the literal type
const role = "CUSTOMER";   // Type is literally "CUSTOMER", not string
Line-by-line explanation
1
let name: string = "Darshan";
: string is the type annotation. It tells TypeScript this variable can ONLY hold strings. Assigning a number later would cause a compile error.
2
let city = "Mumbai";
No annotation needed here. TypeScript sees the value "Mumbai" and infers the type as string automatically. This is called type inference.
3
const role = "CUSTOMER";
const cannot be reassigned, so TypeScript narrows the type to the exact string literal "CUSTOMER" instead of generic string.

Functions

Function Annotations 
// Parameter types + return type
function calculateCost(weight: number, type: string): number {
  let base = weight * 50;
  if (type === "FRAGILE") base += 30;
  return base;
}

// Arrow function with types
const greet = (name: string): string => {
  return `Hello, ${name}!`;
};

// void return — function does not return anything
function logBooking(id: number): void {
  console.log(`Booking ${id} logged`);
}
Key Insight

In Java, the type comes before the name: String name. In TypeScript, the type comes after the name: name: string. Different order, same idea.

Section 05

Interfaces

An interface defines the shape of an object — what properties it must have and what types those properties are. It is TypeScript's most powerful feature for keeping code organized.

Defining an Interface

User Interface 
interface User {
  id: number;
  username: string;
  email: string;
  role: string;
  isActive: boolean;
}

// Now TypeScript enforces the shape
let user: User = {
  id: 1,
  username: "darshan",
  email: "darshan@example.com",
  role: "CUSTOMER",
  isActive: true
};

// ERROR: Property 'email' is missing
let badUser: User = {
  id: 2,
  username: "rahul",
  role: "ADMIN",
  isActive: true
  // Missing email — TypeScript catches this!
};

Optional Properties

Optional Properties 
interface Booking {
  id: number;
  trackingNumber: string;
  senderName: string;
  receiverName: string;
  weight: number;
  status: string;
  description?: string;    // Optional — the ? means it can be missing
  remarks?: string;        // Optional
}

// Valid — description and remarks are optional
let booking: Booking = {
  id: 42,
  trackingNumber: "NIM-2026-0042",
  senderName: "Darshan",
  receiverName: "Rahul",
  weight: 2.5,
  status: "BOOKED"
};

Interfaces for Function Parameters

Using Interfaces in Functions 
interface LoginRequest {
  username: string;
  password: string;
}

interface LoginResponse {
  token: string;
  userId: number;
  role: string;
}

function login(request: LoginRequest): LoginResponse {
  // The function signature tells you everything:
  // - Takes a LoginRequest (username + password)
  // - Returns a LoginResponse (token + userId + role)
  return {
    token: "eyJhbGci...",
    userId: 1,
    role: "CUSTOMER"
  };
}
In Nimbus

In our Angular app, we define interfaces for every API model:

  • User interface — matches the JSON from /api/register and /api/login
  • Booking interface — matches the JSON from /api/bookings
  • LoginRequest / LoginResponse interfaces — define exact shapes for API calls

This way, if the backend changes a field name, TypeScript immediately shows every place in the frontend that breaks.

Section 06

Classes

TypeScript classes are very similar to Java classes. If you know Java classes, you already know 80% of TypeScript classes.

Java vs TypeScript — Side by Side

Java
public class User {
  private int id;
  private String name;
  private String role;

  public User(int id, String name,
              String role) {
    this.id = id;
    this.name = name;
    this.role = role;
  }

  public String getName() {
    return this.name;
  }
}
TypeScript
class User {
  private id: number;
  private name: string;
  private role: string;

  constructor(id: number,
    name: string, role: string) {
    this.id = id;
    this.name = name;
    this.role = role;
  }

  getName(): string {
    return this.name;
  }
}

TypeScript Shorthand (Constructor Parameters)

TypeScript has a shorthand that Java does not — you can declare and assign properties directly in the constructor parameters:

Shorthand Constructor 
// The long way
class Booking {
  private id: number;
  private sender: string;
  private receiver: string;

  constructor(id: number, sender: string, receiver: string) {
    this.id = id;
    this.sender = sender;
    this.receiver = receiver;
  }
}

// The shorthand — does exactly the same thing
class Booking {
  constructor(
    private id: number,
    private sender: string,
    private receiver: string
  ) { }
  // TypeScript auto-creates the properties AND assigns them
}
Shorthand explained
1
constructor(private id: number, ...)
Adding private (or public) before a constructor parameter automatically creates a class property with that name and assigns the value. No need for this.id = id; lines.
2
{ }
The constructor body is empty because all assignment is handled by the shorthand. This is extremely common in Angular services and components.

Access Modifiers

ModifierAccessJava Equivalent
publicAccessible from anywhere (default in TypeScript)public
privateOnly inside the classprivate
protectedInside the class and subclassesprotected
readonlyCan only be set once (in constructor)final
Key Difference from Java

In Java, the default access is package-private. In TypeScript, the default is public. If you write name: string in a class without a modifier, it is public.

Section 07

Enums

Enums let you define a set of named constants. They work almost identically to Java enums.

Nimbus Enums 
// Numeric enum (default — values are 0, 1, 2, ...)
enum BookingStatus {
  BOOKED,         // 0
  PICKED_UP,      // 1
  IN_TRANSIT,     // 2
  OUT_FOR_DELIVERY, // 3
  DELIVERED,      // 4
  CANCELLED       // 5
}

// String enum (explicit values — preferred for APIs)
enum ParcelType {
  STANDARD = "STANDARD",
  FRAGILE = "FRAGILE",
  HEAVY = "HEAVY",
  DOCUMENT = "DOCUMENT"
}

enum UserRole {
  CUSTOMER = "CUSTOMER",
  OFFICER = "OFFICER",
  ADMIN = "ADMIN"
}

// Using enums
let status: BookingStatus = BookingStatus.IN_TRANSIT;
let type: ParcelType = ParcelType.FRAGILE;

if (status === BookingStatus.DELIVERED) {
  console.log("Parcel delivered!");
}
String Enums vs Numeric Enums

Always use string enums when the values are sent to/from an API. If you use numeric enums, the API sends 0 instead of "BOOKED", which is unreadable and breaks if you reorder the enum.

Section 08

Arrays and Tuples

TypeScript arrays are typed — you declare what type of elements they can hold. Tuples are fixed-length arrays where each position has a specific type.

Typed Arrays

Arrays 
// Array of numbers
let weights: number[] = [1.5, 2.0, 3.5, 0.8];

// Array of strings
let cities: string[] = ["Mumbai", "Delhi", "Bangalore"];

// Array of objects using an interface
interface Booking {
  id: number;
  status: string;
}

let bookings: Booking[] = [
  { id: 42, status: "DELIVERED" },
  { id: 43, status: "IN_TRANSIT" }
];

// Alternative syntax (same thing)
let names: Array<string> = ["Darshan", "Rahul"];

Tuples

Tuples 
// Tuple — fixed positions with specific types
let userInfo: [number, string, boolean] = [1, "darshan", true];
//             id       name     isActive

// Accessing tuple values
let userId = userInfo[0];    // number
let userName = userInfo[1];  // string

// Java does not have tuples — this is unique to TypeScript
// Tuples are rarely used in Angular; interfaces are preferred
In Nimbus

When our Angular frontend calls GET /api/bookings?userId=1, the response is an array of booking objects. We type it as Booking[] so TypeScript knows every element in the array has id, trackingNumber, status, etc.

Section 09

Functions with Types

TypeScript functions can have typed parameters, return types, optional parameters, and default values.

Function Patterns 
// 1. Basic function with types
function getBooking(id: number): Booking {
  // returns a Booking object
  return { id: id, trackingNumber: "NIM-001", status: "BOOKED" };
}

// 2. Arrow function (common in Angular)
const calculateCost = (weight: number, type: string): number => {
  return weight * 50 + (type === "FRAGILE" ? 30 : 0);
};

// 3. Optional parameter (add ? after name)
function createBooking(sender: string, receiver: string, description?: string): void {
  // description may or may not be provided
  console.log(`From ${sender} to ${receiver}`);
  if (description) {
    console.log(`Description: ${description}`);
  }
}

createBooking("Darshan", "Rahul");                    // OK
createBooking("Darshan", "Rahul", "Glass frames");    // Also OK

// 4. Default parameter
function getBookings(userId: number, limit: number = 10): Booking[] {
  // If limit is not provided, it defaults to 10
  return [];
}

getBookings(1);      // limit = 10 (default)
getBookings(1, 50);  // limit = 50

Java vs TypeScript Functions

Java
// Must use method overloading
public Booking getBooking(int id) {
  return bookingDao.findById(id);
}

// No default params in Java
// Need overloaded method
public List<Booking> getBookings(
    int userId) {
  return getBookings(userId, 10);
}
public List<Booking> getBookings(
    int userId, int limit) {
  // actual logic
}
TypeScript
// Clean and simple
function getBooking(id: number):
    Booking {
  return this.http.get(`/api/${id}`);
}

// Default params — no overloading
function getBookings(
    userId: number,
    limit: number = 10
): Booking[] {
  // one function handles both cases
}
Section 10

Modules — Import / Export

TypeScript uses the same module system as modern JavaScript. You export things from one file and import them in another. This is how Angular organizes code across hundreds of files.

Export and Import

models/booking.model.ts 
// Export — makes it available to other files
export interface Booking {
  id: number;
  trackingNumber: string;
  senderName: string;
  receiverName: string;
  status: string;
}

export enum BookingStatus {
  BOOKED = "BOOKED",
  IN_TRANSIT = "IN_TRANSIT",
  DELIVERED = "DELIVERED"
}

export function calculateCost(weight: number): number {
  return weight * 50;
}
services/booking.service.ts 
// Import — bring in what you need
import { Booking, BookingStatus, calculateCost } from '../models/booking.model';

class BookingService {
  getBookingLabel(booking: Booking): string {
    if (booking.status === BookingStatus.DELIVERED) {
      return "Delivered";
    }
    let cost = calculateCost(2.5);
    return `Cost: ${cost}`;
  }
}

Java vs TypeScript Modules

ConceptJavaTypeScript
Making code availablepublic class User { }export class User { }
Using code from another fileimport com.nimbus.model.User;import { User } from './models/user';
File = module?No (packages contain multiple classes)Yes (each .ts file is a module)
Path stylePackage names: com.nimbus.modelFile paths: ./models/user
In Nimbus

In our Angular project, every component, service, and model is in its own .ts file. The module system connects them:

  • booking.model.ts exports the Booking interface
  • booking.service.ts imports Booking and uses it in API calls
  • booking-list.component.ts imports BookingService and displays bookings

Each file has a single responsibility, just like Java classes.

Same Concept, Different Syntax

Java Booking Model 
public class Booking {
    private int id;
    private String trackingNumber;
    private String senderName;
    private String receiverName;
    private double weight;
    private String status;

    public Booking(int id, String trackingNumber, String senderName,
                   String receiverName, double weight, String status) {
        this.id = id;
        this.trackingNumber = trackingNumber;
        this.senderName = senderName;
        this.receiverName = receiverName;
        this.weight = weight;
        this.status = status;
    }

    public int getId() { return id; }
    public String getTrackingNumber() { return trackingNumber; }
    public String getStatus() { return status; }
    // ... more getters and setters
}
TypeScript Booking Model 
// TypeScript — interface is enough (no getters/setters needed)
export interface Booking {
  id: number;
  trackingNumber: string;
  senderName: string;
  receiverName: string;
  weight: number;
  status: string;
}

// That's it. 8 lines vs 20+ lines in Java.
// Interfaces define shape. No boilerplate.
Key Insight

Java uses classes with getters/setters for data objects. TypeScript uses interfaces — just the shape, no implementation. This is because JavaScript objects are plain key-value pairs. You do not need a class just to hold data.
