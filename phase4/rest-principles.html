Section 01

What is an API?

API stands for Application Programming Interface. It is the contract that lets two pieces of software talk to each other without knowing what is happening inside the other.

The Waiter Analogy

You are sitting in a restaurant. You want food. The kitchen has the food. But you cannot walk into the kitchen yourself. You need a waiter.

  • You = the client (your app, your browser, your phone)
  • The kitchen = the server (database, business logic, backend)
  • The waiter = the API
  • The menu = the API documentation (what you can order)
  • Your order = the request
  • The dish you get back = the response
     YOU (Client)          WAITER (API)          KITCHEN (Server)
    +-------------+      +--------------+      +----------------+
    |             | ---->|  Takes your  | ---->|  Processes     |
    | "I want     |      |  order to    |      |  the order,    |
    |  chicken    |      |  the kitchen |      |  cooks it      |
    |  biryani"   |      |              |      |                |
    |             | <----|  Brings your | <----|  Sends the     |
    |             |      |  food back   |      |  dish out      |
    +-------------+      +--------------+      +----------------+
       REQUEST --->         API LAYER          <--- RESPONSE
The waiter never cooks. The kitchen never serves. Each does its own job.

The waiter does not cook. The kitchen does not serve. The waiter is the interface between you and the kitchen. That is exactly what an API does — it sits between two systems, takes requests from one, and delivers responses from the other.

APIs in the Real World

APIs are everywhere. Every time you use an app, APIs are working behind the scenes:

What You DoAPI Call Behind the Scenes
Open Zomato and see restaurants near youApp calls Google Maps API for your location + Zomato's own API for restaurant data
Pay with UPI on PhonePePhonePe calls NPCI's UPI API to process the payment
Check weather on your phoneApp calls a weather API (like OpenWeatherMap) with your city name
Log in with Google on any websiteWebsite calls Google's OAuth API to verify your identity
Track a parcel on NimbusFrontend calls GET /api/bookings/{id} to get booking details
Key Insight

An API is not a database. It is not a server. It is the rules and mechanisms that allow one application to request data or actions from another. The API defines what you can ask for, how to ask, and what format the answer comes in.

Section 02

What is REST?

REST stands for REpresentational State Transfer. It is not a technology or a library — it is an architectural style, a set of rules for how APIs should be designed.

REST = A Set of Rules

REST was defined by Roy Fielding in his PhD dissertation in 2000. He observed how the web worked so well — billions of pages, millions of servers, all working together — and distilled the principles that made it work into 6 constraints. Any API that follows these constraints is called a RESTful API.

Think of REST like traffic rules. You can drive however you want on a private road, but on public roads, everyone follows the same rules (drive on the left in India, stop at red lights, yield at roundabouts). REST is the traffic rules for APIs.

The 6 REST Constraints

1. Client-Server

The client (frontend, mobile app) and the server (backend, database) are completely separate. The client only knows how to make requests. The server only knows how to process them and send responses. Neither knows the internal details of the other.

Why it matters: You can change the entire frontend from Angular to React without touching a single line of backend code. You can swap MySQL for PostgreSQL without the frontend ever knowing.

2. Stateless

Every request from the client must contain all the information the server needs to process it. The server does not remember anything about previous requests. Each request is treated as if it is the first one ever.

Stateful (NOT REST)

Request 1: "I'm user Darshan"

Request 2: "Show me my bookings"

Server remembers who you are from Request 1. If the server restarts, it forgets and breaks.

Stateless (REST)

Every request: "I'm user Darshan (here's my token), show me my bookings"

Each request is self-contained. Server can restart, scale, load-balance — nothing breaks.

3. Cacheable

Responses must declare whether they can be cached or not. If a response says "this data is valid for 60 seconds," the client can reuse it without making another request. This reduces server load and makes apps faster.

Example: A list of Indian states rarely changes. An API can mark this response as cacheable for 24 hours. Your app fetches it once and uses the cached version all day.

4. Uniform Interface

This is the most important constraint. Every REST API follows the same conventions:

  • Resources are identified by URIs (/api/bookings/42)
  • You manipulate resources through representations (JSON)
  • Messages are self-descriptive (the HTTP method + headers + body tell you everything)
  • Hypermedia (responses can contain links to related resources)

Why it matters: Once you learn how one REST API works, you can use any REST API. The patterns are the same everywhere.

5. Layered System

The client does not know (and does not care) if it is talking to the actual server, a load balancer, a cache, or a proxy. You can put layers between the client and server without the client knowing.

  Client  --->  Load Balancer  --->  API Gateway  --->  Server
                                          |
                                        Cache
                                          |
                                     Rate Limiter
The client just talks to a URL. What is behind it is invisible.

6. Code on Demand (Optional)

The server can optionally send executable code to the client. For example, a web server sending JavaScript to the browser. This is the only optional constraint — most REST APIs do not use it.

In Nimbus

Our Nimbus API follows all 5 mandatory REST constraints:

  • Client-Server: Angular frontend is completely separate from Spring Boot backend
  • Stateless: Every request carries a JWT token — server never stores session state
  • Cacheable: GET responses for booking history can be cached
  • Uniform Interface: All endpoints follow /api/resource pattern with standard HTTP methods
  • Layered: We could put Nginx in front of Spring Boot and nothing changes
Section 03

Resources and URIs

In REST, everything is a resource. A user is a resource. A booking is a resource. A parcel is a resource. Each resource has a unique address called a URI.

What is a Resource?

A resource is any piece of data or concept that your API manages. It is a noun, not a verb. You do not have an endpoint called /getUser or /createBooking. Instead, you have resources like /users and /bookings, and you use HTTP methods (GET, POST, PUT, DELETE) to act on them.

Wrong (Verb-Based)

/getUsers

/createBooking

/deleteUser/5

/updateBookingStatus

Correct (Resource-Based)

GET /api/users

POST /api/bookings

DELETE /api/users/5

PUT /api/bookings/3/status

URI Structure

URI stands for Uniform Resource Identifier. It is the address of a resource. Here is how a REST URI is structured:

  https://api.nimbus.com/api/bookings/42
  \___/   \_____________/ \__/ \______/ \_/
    |           |           |      |      |
  scheme      host       prefix resource  id

  /api/bookings          -->  ALL bookings (collection)
  /api/bookings/42       -->  ONE specific booking (single resource)
  /api/bookings?userId=5 -->  Filtered bookings (query parameter)
  /api/users/5/bookings  -->  Bookings belonging to user 5 (nested)
URIs are hierarchical. They read like a file path.

URI Design Rules

RuleGoodBad
Use nouns, not verbs/api/bookings/api/getBookings
Use plural nouns/api/users/api/user
Use lowercase/api/bookings/api/Bookings
Use hyphens for readability/api/booking-history/api/bookingHistory
Use path params for IDs/api/bookings/42/api/bookings?id=42
Use query params for filters/api/bookings?status=delivered/api/bookings/delivered
In Nimbus

All Nimbus endpoints follow these URI conventions:

  • /api/register — user registration resource
  • /api/login — authentication resource
  • /api/bookings — bookings collection
  • /api/bookings/{id} — single booking by ID
  • /api/bookings?userId=5 — filtered bookings for a specific user
  • /api/bookings/{id}/status — sub-resource (status of a booking)
  • /api/bookings/{id}/pickup — sub-resource (pickup action)
  • /api/bookings/history — historical bookings collection
Section 04

HTTP Methods and CRUD

REST uses HTTP methods to define what action you want to perform on a resource. There are 4 main methods, and they map directly to the 4 CRUD operations.

The CRUD Mapping

CRUD stands for Create, Read, Update, Delete — the four basic operations you can do with any data. REST maps each one to an HTTP method:

CRUDHTTP MethodWhat It DoesNimbus Example
CreatePOSTCreates a new resourcePOST /api/bookings — create a new parcel booking
ReadGETRetrieves a resource (no changes)GET /api/bookings/42 — get booking #42 details
UpdatePUTUpdates an existing resourcePUT /api/bookings/42/status — change booking status
DeleteDELETERemoves a resourceDELETE /api/users/5 — delete a user account

Deep Dive: Each Method

GET — Read Data

Purpose: Retrieve data. Never changes anything on the server. Called safe and idempotent (calling it 10 times gives the same result as calling it once).

GET Request 
GET /api/bookings/42

// No request body (GET never has a body)

// Response: 200 OK
{
  "id": 42,
  "senderName": "Darshan",
  "receiverName": "Rahul",
  "status": "IN_TRANSIT",
  "weight": 2.5
}

POST — Create Data

Purpose: Create a new resource. The server assigns an ID and returns the created resource. Not idempotent — calling it twice creates two resources.

POST Request 
POST /api/bookings
Content-Type: application/json

// Request body:
{
  "senderName": "Darshan",
  "senderAddress": "Mumbai",
  "receiverName": "Rahul",
  "receiverAddress": "Delhi",
  "weight": 2.5,
  "parcelType": "FRAGILE"
}

// Response: 201 Created
{
  "id": 43,
  "trackingNumber": "NIM-2026-0043",
  "status": "BOOKED",
  "createdAt": "2026-04-23T10:30:00"
}

PUT — Update Data

Purpose: Update an existing resource. You send the updated data. Idempotent — calling it 10 times with the same data produces the same result.

PUT Request 
PUT /api/bookings/42/status
Content-Type: application/json

// Request body:
{
  "status": "DELIVERED"
}

// Response: 200 OK
{
  "id": 42,
  "status": "DELIVERED",
  "updatedAt": "2026-04-23T14:00:00"
}

DELETE — Remove Data

Purpose: Remove a resource. Idempotent — deleting the same resource twice should not cause an error (first call deletes, second call returns 404 or 204).

DELETE Request 
DELETE /api/users/5

// No request body

// Response: 204 No Content
// (empty body — the resource is gone)

Safety and Idempotency

MethodSafe?Idempotent?Has Body?
GETYes (no changes)YesNo
POSTNo (creates data)NoYes
PUTNo (modifies data)YesYes
DELETENo (removes data)YesNo
Analogy: The Library

Think of a library:

  • GET = Looking up a book in the catalog. You read information. The catalog does not change.
  • POST = Donating a new book. The library adds it to the shelves with a new catalog number.
  • PUT = Updating a book's information (correcting the author's name). The book stays, the info changes.
  • DELETE = Removing a damaged book from the catalog. It is gone.
Section 05

Request & Response Anatomy

Every HTTP communication has two parts: the request (what you send) and the response (what you get back). Understanding their structure is essential.

Request Anatomy

An HTTP request has 4 parts:

URL
Where to send the request. Example: https://api.nimbus.com/api/bookings/42
Method
What action to perform. One of: GET, POST, PUT, DELETE
Headers
Metadata about the request. Content-Type, Authorization, Accept, etc.
Body
The data you are sending (only for POST and PUT). Usually JSON.
Full HTTP Request Example 
POST /api/bookings HTTP/1.1          <-- Method + URL + HTTP version
Host: api.nimbus.com                  <-- Header: which server
Content-Type: application/json        <-- Header: body format is JSON
Authorization: Bearer eyJhbGci...     <-- Header: authentication token
Accept: application/json              <-- Header: I want JSON back

{                                     <-- Body starts here
  "senderName": "Darshan",
  "receiverName": "Rahul",
  "weight": 2.5
}

Response Anatomy

An HTTP response has 3 parts:

Status Code
A 3-digit number telling you what happened. 200 = OK, 404 = Not Found, 500 = Server Error
Headers
Metadata about the response. Content-Type, Cache-Control, Content-Length, etc.
Body
The actual data. Usually a JSON object or array.
Full HTTP Response Example 
HTTP/1.1 201 Created                 <-- Status line
Content-Type: application/json        <-- Header: body is JSON
Location: /api/bookings/43            <-- Header: where the new resource lives

{                                     <-- Body starts here
  "id": 43,
  "trackingNumber": "NIM-2026-0043",
  "status": "BOOKED",
  "createdAt": "2026-04-23T10:30:00"
}

Section 06

HTTP Status Codes

Status codes are 3-digit numbers that tell the client exactly what happened with their request. They are grouped into 5 families.

The 5 Families

RangeFamilyMeaning
1xxInformationalRequest received, still processing (rarely used)
2xxSuccessRequest worked. Everything is good.
3xxRedirectionResource moved. Go somewhere else.
4xxClient ErrorYOU made a mistake. Fix your request.
5xxServer ErrorSERVER made a mistake. Not your fault.

The Status Codes You Must Know

Success Codes (2xx)

CodeNameWhen It Is UsedNimbus Example
200 OK Request succeeded. Data is in the response body. GET /api/bookings/42 returns booking details
201 Created A new resource was successfully created. POST /api/bookings creates a new booking
204 No Content Request succeeded but there is nothing to return. DELETE /api/users/5 — user deleted, no body needed

Client Error Codes (4xx)

CodeNameWhen It Is UsedNimbus Example
400 Bad Request Your request is malformed or missing required fields. POST booking without senderName — validation fails
401 Unauthorized You are not authenticated. No token or invalid token. Calling GET /api/bookings without a JWT token
403 Forbidden You are authenticated but do not have permission. A regular user trying to access admin-only endpoint
404 Not Found The resource you requested does not exist. GET /api/bookings/99999 — no booking with that ID

Server Error Codes (5xx)

CodeNameWhen It Is UsedNimbus Example
500 Internal Server Error Something went wrong on the server. A bug, crash, or unhandled exception. Database connection fails while processing a booking
Analogy: The Traffic Light

Think of status codes like traffic signals:

  • 2xx (Green): Go ahead, everything worked perfectly
  • 3xx (Yellow): Redirect — the road you want moved, follow the new sign
  • 4xx (Red for you): You ran a red light. You made the mistake. Fix your driving.
  • 5xx (Red for the road): The road itself is broken. Not your fault. Wait for repairs.
Common Mistake

401 vs 403: These are often confused.

  • 401 Unauthorized = "Who are you? I don't know you." (Not authenticated. Log in.)
  • 403 Forbidden = "I know who you are, but you're not allowed in here." (Authenticated but insufficient permissions.)
Section 07

JSON Format

JSON (JavaScript Object Notation) is the standard format for sending and receiving data in REST APIs. It is human-readable and machine-parseable.

JSON Syntax Rules

  • Data is in key-value pairs: "name": "Darshan"
  • Keys are always strings in double quotes
  • Values can be: string, number, boolean, null, object, or array
  • Objects use curly braces: { }
  • Arrays use square brackets: [ ]
  • Pairs are separated by commas
  • No trailing commas (the last item has no comma after it)
  • No comments allowed in JSON

JSON Data Types

All JSON Data Types 
{
  "name": "Darshan",              // String  (always double quotes)
  "age": 22,                      // Number  (no quotes)
  "weight": 2.5,                  // Number  (decimals work)
  "isActive": true,               // Boolean (true or false, no quotes)
  "middleName": null,             // Null    (no value, no quotes)
  "address": {                    // Object  (nested curly braces)
    "city": "Mumbai",
    "pincode": "400001"
  },
  "phones": [                     // Array   (square brackets)
    "9876543210",
    "9123456789"
  ]
}

Objects vs Arrays

Object { }

A single thing with named properties.

{ "name": "Darshan", "age": 22 }

Like a single row in a database table.

Array [ ]

A list of things (ordered, can repeat).

[{"name":"A"}, {"name":"B"}]

Like multiple rows from a database query.

Section 10

Authentication Concepts

REST APIs are stateless — the server does not remember who you are between requests. So how do you prove your identity? With tokens.

The Problem

Without authentication, anyone can call your API. Someone could create fake bookings, read other people's data, or delete accounts. You need a way to verify: "Who is making this request, and are they allowed to?"

Token-Based Authentication

The most common approach for REST APIs is token-based authentication:

  1. User sends username + password to the login endpoint
  2. Server verifies credentials. If correct, it generates a token (a long encrypted string) and sends it back
  3. Client stores the token (in localStorage, cookie, or memory)
  4. For every subsequent request, client sends the token in the Authorization header
  5. Server reads the token, verifies it, and knows who the user is

JWT Preview

JWT stands for JSON Web Token. It is the most popular token format for REST APIs. A JWT has 3 parts, separated by dots:

JWT Structure

|_________________________|  |______________________________________________|  |___________________|
         HEADER                              PAYLOAD                               SIGNATURE
   (algorithm + type)              (user data — userId, role)                 (server's secret seal)
Header
Says which algorithm was used to sign the token. Usually HS256 (HMAC-SHA256).
Payload
Contains the user data (called "claims"): userId, username, role, exp (expiry time). This is NOT encrypted — anyone can decode it. Never put passwords in here.
Signature
The server uses a secret key to sign the token. If anyone tampers with the header or payload, the signature won't match and the server rejects the token.
Important

JWT payload is base64-encoded, not encrypted. Anyone can decode it and read the data inside. The signature only prevents tampering — it does not prevent reading. Never put sensitive data (passwords, credit card numbers) in a JWT.

In Nimbus

Nimbus uses JWT for authentication:

  • POST /api/login returns a JWT token containing userId, username, and role
  • Every protected endpoint checks for Authorization: Bearer <token>
  • Token expires in 24 hours (86400 seconds)
  • The server validates the token's signature with its secret key before processing any request
