Section 01

Splitting Nimbus into Microservices

The monolithic Nimbus app becomes three independent services, each with its own responsibilities and port.

The Monolith vs Microservices

Monolith (Current Nimbus)

One Spring Boot app on port 8080:

  • UserController + UserService + UserDAO
  • BookingController + BookingService + BookingDAO
  • NotificationService
  • All share one database
  • One pom.xml, one jar file
Microservices (New)

Three separate Spring Boot apps:

  • User Service — port 8081
  • Booking Service — port 8082
  • Notification Service — port 8083
  • Each can have its own database
  • Three pom.xml files, three jar files

Section 02

Database Per Service

Each microservice owns its data. No shared database tables.

Two Approaches

Approach How When to Use
Separate databases User Service uses users_db, Booking uses bookings_db Production — full isolation, independent scaling
Shared database, separate schemas Same MySQL server, different schemas: nimbus_users, nimbus_bookings Development — simpler setup, still logically separated

Nimbus Database Setup

User Service — application.properties 
server.port=8081
spring.datasource.url=jdbc:mysql://localhost:3306/nimbus_users
spring.datasource.username=root
spring.datasource.password=root
Booking Service — application.properties 
server.port=8082
spring.datasource.url=jdbc:mysql://localhost:3306/nimbus_bookings
spring.datasource.username=root
spring.datasource.password=root
Notification Service — application.properties 
server.port=8083
# No database — this service only sends notifications
# It calls User Service to get user email/phone
The Key Rule

Booking Service must never directly query the users table. If it needs user data (like validating that a userId exists), it calls User Service's API: GET http://localhost:8081/api/users/1. This is the fundamental microservices rule — services communicate through APIs, not shared databases.

Section 03

Inter-Service REST Calls

When Booking Service needs user data, it makes an HTTP call to User Service. Two ways to do it.

Option 1: RestTemplate (Classic)

BookingService.java — Calling User Service with RestTemplate 
package com.nimbus.booking.service;

import com.nimbus.booking.dto.UserResponse;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import org.springframework.web.client.RestTemplate;
import org.springframework.web.client.HttpClientErrorException;

@Service
public class BookingService {

    @Autowired
    private RestTemplate restTemplate;

    @Autowired
    private BookingDAO bookingDAO;

    private static final String USER_SERVICE_URL = "http://localhost:8081/api/users";

    public BookingResponse createBooking(BookingRequest request) {
        // Step 1: Validate user exists by calling User Service
        try {
            UserResponse user = restTemplate.getForObject(
                USER_SERVICE_URL + "/" + request.getUserId(),
                UserResponse.class
            );
        } catch (HttpClientErrorException.NotFound e) {
            throw new InvalidBookingException(
                "User not found with ID: " + request.getUserId()
            );
        }

        // Step 2: Create the booking in our own database
        Booking booking = new Booking();
        booking.setUserId(request.getUserId());
        booking.setSenderName(request.getSenderName());
        booking.setReceiverName(request.getReceiverName());
        booking.setWeight(request.getWeight());
        booking.setStatus("BOOKED");
        booking.setPickupDate(request.getPickupDate());

        bookingDAO.save(booking);

        // Step 3: Notify the notification service
        restTemplate.postForObject(
            "http://localhost:8083/api/notify",
            new NotificationRequest(request.getUserId(), "BOOKING_CREATED", booking.getBookingId()),
            Void.class
        );

        return convertToResponse(booking);
    }
}

RestTemplate Bean Configuration

AppConfig.java — RestTemplate bean 
package com.nimbus.booking.config;

Option 2: WebClient (Modern, Reactive)

BookingService.java — Using WebClient 
@Service
public class BookingService {

    private final WebClient userServiceClient;

    public BookingService(WebClient.Builder webClientBuilder) {
        this.userServiceClient = webClientBuilder
            .baseUrl("http://localhost:8081")
            .build();
    }

    public BookingResponse createBooking(BookingRequest request) {
        // Call User Service to validate user
        UserResponse user = userServiceClient.get()
            .uri("/api/users/{id}", request.getUserId())
            .retrieve()
            .onStatus(HttpStatusCode::is4xxClientError,
                response -> Mono.error(new InvalidBookingException(
                    "User not found with ID: " + request.getUserId())))
            .bodyToMono(UserResponse.class)
            .block();  // .block() makes it synchronous

        // ... create booking
    }
}
RestTemplate
  • Synchronous (blocking)
  • Simple to use
  • Being deprecated in newer Spring
  • Good for learning
WebClient
  • Async by default (non-blocking)
  • Fluent API, more options
  • Recommended by Spring team
  • Needs spring-webflux dependency
Section 04

API Gateway

One entry point for all clients. The gateway routes requests to the correct service.

The Problem Without a Gateway

Without a gateway, the Angular frontend needs to know every service's URL and port:

  • User endpoints → http://localhost:8081/api/...
  • Booking endpoints → http://localhost:8082/api/...
  • Notification endpoints → http://localhost:8083/api/...

If you add a new service or change a port, the frontend must be updated. The gateway solves this:

Angular Frontend localhost:4200
→
API Gateway localhost:8080
→
Routes to correct service
Gateway — application.yml (Spring Cloud Gateway) 
server:
  port: 8080

spring:
  cloud:
    gateway:
      routes:
        - id: user-service
          uri: http://localhost:8081
          predicates:
            - Path=/api/users/**, /api/register, /api/login

        - id: booking-service
          uri: http://localhost:8082
          predicates:
            - Path=/api/bookings/**

        - id: notification-service
          uri: http://localhost:8083
          predicates:
            - Path=/api/notify/**

Now the frontend only knows http://localhost:8080. The gateway inspects the URL path and forwards to the right service.

Section 05

Service Discovery

In production, services don't have fixed URLs. Service discovery lets them find each other dynamically.

The Problem

Hardcoding http://localhost:8081 works on your laptop. But in production, services run on different servers with dynamic IPs. If User Service moves to a new server, Booking Service's hardcoded URL breaks.

Netflix Eureka

Eureka is a service registry. Each service registers itself on startup ("I'm User Service, running at 192.168.1.10:8081"). Other services ask Eureka: "Where is User Service?" and get the current address.

Eureka Server Service Registry
↓
User Service Registers: "I'm at :8081"
  
Booking Service Registers: "I'm at :8082"
  
Notification Registers: "I'm at :8083"
All services register with Eureka on startup. To call another service, ask Eureka for its address.
With Eureka — Service calls by name instead of URL 
// Without Eureka (hardcoded URL):
restTemplate.getForObject("http://localhost:8081/api/users/1", UserResponse.class);

// With Eureka (service name):
restTemplate.getForObject("http://USER-SERVICE/api/users/1", UserResponse.class);
// Eureka resolves "USER-SERVICE" to the actual IP and port at runtime
When Do You Need This?

For a learning project or small deployment, hardcoded URLs are fine. Eureka becomes essential when you have multiple instances of the same service (3 copies of Booking Service for load balancing), auto-scaling (services spin up/down dynamically), or cloud deployment where IPs change on every restart.

user-service/.../controller/UserController.java 
package com.nimbus.user.controller;

import com.nimbus.user.dto.*;
import com.nimbus.user.model.User;
import com.nimbus.user.service.UserService;
import jakarta.validation.Valid;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.net.URI;

@RestController
public class UserController {

    @Autowired
    private UserService userService;

    @PostMapping("/api/register")
    public ResponseEntity<UserResponse> register(
            @Valid @RequestBody RegistrationRequest request) {
        UserResponse created = userService.register(request);
        return ResponseEntity.created(
            URI.create("/api/users/" + created.getUserId())
        ).body(created);
    }

    @PostMapping("/api/login")
    public ResponseEntity<LoginResponse> login(
            @Valid @RequestBody LoginRequest request) {
        LoginResponse response = userService.authenticate(request);
        return ResponseEntity.ok(response);
    }

    @GetMapping("/api/users/{id}")
    public ResponseEntity<UserResponse> getById(@PathVariable int id) {
        UserResponse user = userService.findById(id);
        return ResponseEntity.ok(user);
    }
}

booking-service/.../controller/BookingController.java 
package com.nimbus.booking.controller;

import com.nimbus.booking.dto.*;
import com.nimbus.booking.service.BookingService;
import jakarta.validation.Valid;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;
import java.net.URI;
import java.util.List;

@RestController
@RequestMapping("/api/bookings")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    @PostMapping
    public ResponseEntity<BookingResponse> create(
            @Valid @RequestBody BookingRequest request) {
        BookingResponse created = bookingService.createBooking(request);
        return ResponseEntity.created(
            URI.create("/api/bookings/" + created.getBookingId())
        ).body(created);
    }

    @GetMapping("/{id}")
    public ResponseEntity<BookingResponse> getById(@PathVariable int id) {
        BookingResponse booking = bookingService.findById(id);
        return ResponseEntity.ok(booking);
    }

    @GetMapping
    public ResponseEntity<List<BookingResponse>> getByUser(
            @RequestParam int userId) {
        return ResponseEntity.ok(bookingService.findByUserId(userId));
    }

    @PutMapping("/{id}/status")
    public ResponseEntity<BookingResponse> updateStatus(
            @PathVariable int id,
            @Valid @RequestBody StatusUpdateRequest request) {
        return ResponseEntity.ok(bookingService.updateStatus(id, request));
    }

    @PutMapping("/{id}/pickup")
    public ResponseEntity<BookingResponse> updatePickup(
            @PathVariable int id,
            @Valid @RequestBody PickupUpdateRequest request) {
        return ResponseEntity.ok(bookingService.updatePickup(id, request));
    }

    @GetMapping("/history")
    public ResponseEntity<List<BookingResponse>> getHistory(
            @RequestParam int userId) {
        return ResponseEntity.ok(bookingService.getHistory(userId));
    }
}

notification-service/.../NotificationController.java 
package com.nimbus.notification.controller;

import com.nimbus.notification.dto.NotificationRequest;

import org.springframework.http.ResponseEntity;

@RestController

public class NotificationController {

    @Autowired
    private NotificationService notificationService;

Section 07

Docker Basics

Package each microservice into a container so it runs the same everywhere — your laptop, a server, or the cloud.

What is Docker?

Docker packages your application with everything it needs (JDK, dependencies, config) into a container. The container runs identically on any machine. No more "it works on my machine" problems.

Dockerfile for Each Service

user-service/Dockerfile 
# Stage 1: Build
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

# Stage 2: Run
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/user-service-0.0.1-SNAPSHOT.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java", "-jar", "app.jar"]
booking-service/Dockerfile 
# Same structure, different jar name and port
FROM maven:3.9-eclipse-temurin-17 AS build
WORKDIR /app
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=build /app/target/booking-service-0.0.1-SNAPSHOT.jar app.jar
EXPOSE 8082
ENTRYPOINT ["java", "-jar", "app.jar"]

docker-compose.yml — Run Everything Together

docker-compose.yml 
version: '3.8'
services:
  mysql:
    image: mysql:8
    environment:
      MYSQL_ROOT_PASSWORD: root
    ports:
      - "3306:3306"

  user-service:
    build: ./user-service
    ports:
      - "8081:8081"
    depends_on:
      - mysql
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/nimbus_users

  booking-service:
    build: ./booking-service
    ports:
      - "8082:8082"
    depends_on:
      - mysql
      - user-service
    environment:
      SPRING_DATASOURCE_URL: jdbc:mysql://mysql:3306/nimbus_bookings

  notification-service:
    build: ./notification-service
    ports:
      - "8083:8083"

One command starts everything: docker-compose up. MySQL, all three services, all wired together.

In Nimbus

Each service gets its own Dockerfile with a multi-stage build (compile in one stage, run in another — keeps the final image small). The docker-compose.yml defines the entire system. A new developer clones the repo, runs docker-compose up, and the entire Nimbus platform is running in minutes.

Section 08

Architecture Summary

The complete picture of Nimbus as a microservices system.

Client
Angular app on port 4200. All HTTP calls go to the API Gateway on port 8080.
Gateway
Spring Cloud Gateway on port 8080. Routes /api/users/** to 8081, /api/bookings/** to 8082, /api/notify/** to 8083.
Services
User Service (8081) — registration, login, user lookup. Booking Service (8082) — CRUD, status, history. Notification Service (8083) — email/SMS.
Discovery
Eureka Server (8761) — service registry. Services register on startup, look up each other by name instead of hardcoded URLs.
Data
MySQL — nimbus_users schema (User Service), nimbus_bookings schema (Booking Service). Notification Service has no database.
When to Use Microservices

Microservices add complexity: inter-service calls, distributed transactions, deployment orchestration. For a small team or a project with a few thousand users, a well-structured monolith is simpler and faster to build. Microservices shine when you have multiple teams working independently, different scaling needs per feature, or need to deploy parts of the system independently. Learn the pattern — but start with a monolith and split when the pain justifies it.
