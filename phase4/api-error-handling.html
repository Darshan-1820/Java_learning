Section 01

Why APIs Need Structured Errors

Your API should never leak stack traces to the client. Structured error responses are a security and usability requirement.

What Happens Without Error Handling

When your Spring Boot API throws an unhandled exception, the client receives something like this:

Default Spring error response (BAD) 
{
  "timestamp": "2026-04-23T10:35:00.000+00:00",
  "status": 500,
  "error": "Internal Server Error",
  "trace": "java.lang.NullPointerException\n\tat com.nimbus.service.BookingService.findById(BookingService.java:42)\n\tat com.nimbus.controller.BookingController.getById(BookingController.java:28)\n\tat sun.reflect.NativeMethodAccessorImpl...",
  "path": "/api/bookings/9999"
}

Three problems with this:

  • Security risk: The trace field exposes your package names, class names, line numbers, and the full call stack. An attacker can map your entire backend architecture from this.
  • Inconsistent format: Validation errors, 404s, and 500s all return different JSON structures. The frontend can't parse errors reliably.
  • No useful message: "Internal Server Error" tells the client nothing about what went wrong or how to fix it.

What a Good Error Response Looks Like

Structured error response (GOOD) 
{
  "timestamp": "2026-04-23 10:35:00",
  "status": 404,
  "error": "Not Found",
  "message": "Booking not found with ID: 9999",
  "path": "/api/bookings/9999"
}

Clean, consistent, no stack traces. The frontend knows exactly what happened and can show the user a helpful message.

The Rule

Every error response from your API should have the same JSON structure, regardless of whether it's a 400, 404, 409, or 500. The only things that change are the values. This lets the Angular frontend handle all errors with one piece of code.

Section 02

Custom ErrorResponse Class

A simple POJO that defines the exact JSON structure for every error your API returns.

com/nimbus/exception/ErrorResponse.java 
package com.nimbus.exception;

import com.fasterxml.jackson.annotation.JsonFormat;
import java.time.LocalDateTime;

public class ErrorResponse {

    private int status;
    private String error;
    private String message;
    private String path;

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime timestamp;

    public ErrorResponse(int status, String error, String message, String path) {
        this.status = status;
        this.error = error;
        this.message = message;
        this.path = path;
        this.timestamp = LocalDateTime.now();
    }

    // Getters
    public int getStatus() { return status; }
    public String getError() { return error; }
    public String getMessage() { return message; }
    public String getPath() { return path; }
    public LocalDateTime getTimestamp() { return timestamp; }

    // Setters
    public void setStatus(int status) { this.status = status; }
    public void setError(String error) { this.error = error; }
    public void setMessage(String message) { this.message = message; }
    public void setPath(String path) { this.path = path; }
    public void setTimestamp(LocalDateTime timestamp) { this.timestamp = timestamp; }
}
Line-by-line explanation
8
private int status;
The HTTP status code (400, 404, 500). Included in the body so the frontend can access it without parsing response headers.
9
private String error;
Human-readable error category: "Not Found", "Bad Request", "Internal Server Error". Matches standard HTTP reason phrases.
10
private String message;
The specific error message: "Booking not found with ID: 9999". This is what the frontend displays to the user.
11
private String path;
The URL that caused the error: "/api/bookings/9999". Useful for debugging which endpoint failed.
13
@JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
Tells Jackson how to format the timestamp in JSON. Without this, LocalDateTime serializes as an array like [2026,4,23,10,35,0] instead of a readable string.

When Jackson serializes this class, the JSON output looks exactly like the structured example above. Every error response from Nimbus will use this same class.

Section 03

@ExceptionHandler on Controllers

The first approach: handling exceptions directly inside a controller. Simple but limited.

How @ExceptionHandler Works

You put a method annotated with @ExceptionHandler inside your controller. When any method in that controller throws the specified exception, Spring routes it to your handler instead of returning a default error:

BookingController.java — Local exception handler 
@RestController
@RequestMapping("/api/bookings")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    @GetMapping("/{id}")
    public ResponseEntity<BookingResponse> getById(@PathVariable int id) {
        // If booking doesn't exist, service throws BookingNotFoundException
        BookingResponse booking = bookingService.findById(id);
        return ResponseEntity.ok(booking);
    }

    // This catches BookingNotFoundException thrown by ANY method in THIS controller
    @ExceptionHandler(BookingNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(
            BookingNotFoundException ex, HttpServletRequest request) {
        ErrorResponse error = new ErrorResponse(
            404, "Not Found", ex.getMessage(), request.getRequestURI()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }
}

The Limitation

This approach only works for the controller it's in. If UserController also needs to handle ResourceNotFoundException, you'd copy the same handler method there. And in every other controller. That's duplication.

When to Use Local @ExceptionHandler

Only when the exception is truly specific to one controller and you want different handling than the global handler. For example, a controller that returns a redirect instead of JSON for certain errors. For everything else, use @ControllerAdvice.

Section 04

@ControllerAdvice — Global Handling

One class catches exceptions from ALL controllers. Write once, handle everywhere.

How It Works

Controller method throws exception
→
Spring intercepts checks for handler
→
@ControllerAdvice matches exception type
→
ErrorResponse JSON returned to client
Spring routes every unhandled exception through the @ControllerAdvice class before sending a response

@ControllerAdvice tells Spring: "This class provides exception handling for all controllers." You combine it with @ExceptionHandler methods, one per exception type:

Structure of a @ControllerAdvice class 
@ControllerAdvice          // 1. Apply globally to all controllers
public class GlobalExceptionHandler {

    @ExceptionHandler(SomeException.class)  // 2. Which exception to catch
    public ResponseEntity<ErrorResponse> handleSome(
            SomeException ex,               // 3. The thrown exception
            HttpServletRequest request) {    // 4. The original request (for path)

        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", ex.getMessage(),
            request.getRequestURI()
        );
        return ResponseEntity.badRequest().body(error);  // 5. Return structured error
    }

    // Add more @ExceptionHandler methods for other exception types...
}
Priority Order

When an exception is thrown, Spring looks for a handler in this order: (1) local @ExceptionHandler in the same controller, (2) @ControllerAdvice class. If a local handler exists, it takes priority over the global one. This lets you override global behavior for specific controllers when needed.

Section 05

Nimbus GlobalExceptionHandler

The complete exception handling setup for Nimbus. Custom exceptions + one handler class = clean error responses everywhere.

Step 1: Custom Exception Classes

com/nimbus/exception/DuplicateEmailException.java 
package com.nimbus.exception;

public class DuplicateEmailException extends RuntimeException {
    public DuplicateEmailException(String email) {
        super("An account with email '" + email + "' already exists");
    }
}
com/nimbus/exception/BookingNotFoundException.java 
package com.nimbus.exception;

public class BookingNotFoundException extends RuntimeException {
    public BookingNotFoundException(int bookingId) {
        super("Booking not found with ID: " + bookingId);
    }
}
com/nimbus/exception/InvalidBookingException.java 
package com.nimbus.exception;

public class InvalidBookingException extends RuntimeException {
    public InvalidBookingException(String message) {
        super(message);
    }
}

Step 2: The Global Handler

com/nimbus/exception/GlobalExceptionHandler.java 
package com.nimbus.exception;

import jakarta.servlet.http.HttpServletRequest;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ControllerAdvice;
import org.springframework.web.bind.annotation.ExceptionHandler;

@ControllerAdvice
public class GlobalExceptionHandler {

    // 400 — Duplicate email during registration
    @ExceptionHandler(DuplicateEmailException.class)
    public ResponseEntity<ErrorResponse> handleDuplicateEmail(
            DuplicateEmailException ex, HttpServletRequest request) {
        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", ex.getMessage(), request.getRequestURI()
        );
        return ResponseEntity.badRequest().body(error);
    }

    // 400 — Invalid booking data
    @ExceptionHandler(InvalidBookingException.class)
    public ResponseEntity<ErrorResponse> handleInvalidBooking(
            InvalidBookingException ex, HttpServletRequest request) {
        ErrorResponse error = new ErrorResponse(
            400, "Bad Request", ex.getMessage(), request.getRequestURI()
        );
        return ResponseEntity.badRequest().body(error);
    }

    // 404 — Booking not found
    @ExceptionHandler(BookingNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleBookingNotFound(
            BookingNotFoundException ex, HttpServletRequest request) {
        ErrorResponse error = new ErrorResponse(
            404, "Not Found", ex.getMessage(), request.getRequestURI()
        );
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(error);
    }

    // 500 — Catch-all for unexpected errors
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(
            Exception ex, HttpServletRequest request) {
        // NEVER expose internal exception messages in 500 responses
        ErrorResponse error = new ErrorResponse(
            500, "Internal Server Error",
            "Something went wrong. Please try again later.",
            request.getRequestURI()
        );
        // Log the real error server-side (for debugging)
        ex.printStackTrace();
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR).body(error);
    }
}
Line-by-line explanation
13
@ExceptionHandler(DuplicateEmailException.class)
When UserService.register() finds the email already exists, it throws DuplicateEmailException. This handler catches it and returns a 400 with a clear message. The frontend shows "An account with email 'x@y.com' already exists".
34
@ExceptionHandler(BookingNotFoundException.class)
When GET /api/bookings/9999 hits a non-existent booking, the service throws this. Returns 404 with the booking ID in the message.
43
@ExceptionHandler(Exception.class)
Catch-all. Handles NullPointerException, SQLException, or any other unexpected error. Returns a generic message — never the real exception. The real error is logged server-side for developer debugging.

In Nimbus

The controllers and services never build ErrorResponse objects or set error status codes. They throw exceptions and move on. The GlobalExceptionHandler is the single place in the entire codebase where error responses are constructed. Change the error format once, every endpoint is updated.

Section 06

Bean Validation

Automatically validate request data using annotations like @NotBlank, @Size, and @Email. No manual if-checks needed.

The Problem: Manual Validation

Without bean validation, you'd write code like this for every field:

Manual validation (tedious) 
@PostMapping("/api/register")
public ResponseEntity<?> register(@RequestBody UserRegistrationRequest request) {
    if (request.getName() == null || request.getName().isBlank()) {
        return ResponseEntity.badRequest().body("Name is required");
    }
    if (request.getEmail() == null || !request.getEmail().contains("@")) {
        return ResponseEntity.badRequest().body("Valid email is required");
    }
    if (request.getPassword() == null || request.getPassword().length() < 6) {
        return ResponseEntity.badRequest().body("Password must be at least 6 characters");
    }
    // ... and this goes on for every field
}

This is repetitive, error-prone, and clutters your controller. Bean Validation replaces all of it with annotations on the DTO class.

Step 1: Add Dependency

pom.xml — Add validation starter 
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>

Step 2: Annotate the DTO

UserRegistrationRequest.java — With validation annotations 
package com.nimbus.dto;

import jakarta.validation.constraints.*;

public class UserRegistrationRequest {

    @NotBlank(message = "Name is required")
    @Size(min = 2, max = 50, message = "Name must be 2-50 characters")
    private String name;

    @NotBlank(message = "Email is required")
    @Email(message = "Email must be valid")
    private String email;

    @NotBlank(message = "Password is required")
    @Size(min = 6, message = "Password must be at least 6 characters")
    private String password;

    @NotBlank(message = "Role is required")
    private String role;

    // Getters and setters
    public String getName() { return name; }
    public void setName(String name) { this.name = name; }
    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }
    public String getPassword() { return password; }
    public void setPassword(String password) { this.password = password; }
    public String getRole() { return role; }
    public void setRole(String role) { this.role = role; }
}
BookingRequest.java — With validation annotations 
package com.nimbus.dto;

import jakarta.validation.constraints.*;

public class BookingRequest {

    @NotNull(message = "User ID is required")
    private Integer userId;

    @NotBlank(message = "Sender name is required")
    private String senderName;

    @NotBlank(message = "Sender address is required")
    private String senderAddress;

    @NotBlank(message = "Receiver name is required")
    private String receiverName;

    @NotBlank(message = "Receiver address is required")
    private String receiverAddress;

    @Positive(message = "Weight must be greater than 0")
    private double weight;

    @NotBlank(message = "Pickup date is required")
    private String pickupDate;

    // Getters and setters...
}

Common Validation Annotations

Annotation What It Checks Example
@NotNull Field must not be null @NotNull Integer userId
@NotBlank String must not be null, empty, or whitespace @NotBlank String name
@NotEmpty String/Collection must not be null or empty @NotEmpty List<String> items
@Size String/Collection length must be within range @Size(min=2, max=50)
@Email Must be a valid email format @Email String email
@Min / @Max Numeric value must be at least / at most @Min(1) @Max(100) int quantity
@Positive Number must be greater than 0 @Positive double weight
@Pattern Must match a regex pattern @Pattern(regexp="^[A-Z]{3}$")

Step 3: Add @Valid to the Controller

UserController.java — @Valid triggers validation 
@PostMapping("/register")
public ResponseEntity<User> register(
        @Valid @RequestBody UserRegistrationRequest request) {
    //  ^^^^^^
    // @Valid tells Spring: "Before calling this method, validate the
    // request object using the annotations on its fields."
    // If validation fails, Spring throws MethodArgumentNotValidException
    // BEFORE this method is even called.

    User created = userService.register(request);
    return ResponseEntity.created(URI.create("/api/users/" + created.getUserId()))
                         .body(created);
}
@Valid is the Trigger

Just adding @NotBlank annotations to the DTO does nothing by itself. You must also add @Valid before @RequestBody in the controller method. Without @Valid, Spring skips validation entirely and passes whatever JSON the client sent.

Section 07

Handling MethodArgumentNotValidException

When @Valid fails, Spring throws MethodArgumentNotValidException. Here's how to catch it and return clean error messages.

What Happens When Validation Fails

If the client sends this request to register:

Invalid registration request 
{
  "name": "",
  "email": "not-an-email",
  "password": "ab",
  "role": ""
}

Spring validates every field, collects all violations, and throws MethodArgumentNotValidException. Without a handler, the client gets Spring's default error page. We need a handler in GlobalExceptionHandler:

GlobalExceptionHandler.java — Add validation handler 
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ErrorResponse> handleValidation(
        MethodArgumentNotValidException ex, HttpServletRequest request) {

    // Collect all validation error messages
    String errors = ex.getBindingResult()
        .getFieldErrors()
        .stream()
        .map(fieldError -> fieldError.getField() + ": " + fieldError.getDefaultMessage())
        .collect(Collectors.joining("; "));

    // Example output: "name: Name is required; email: Email must be valid; password: Password must be at least 6 characters"

    ErrorResponse error = new ErrorResponse(
        400, "Validation Failed", errors, request.getRequestURI()
    );
    return ResponseEntity.badRequest().body(error);
}

The Response

Validation error response 
{
  "timestamp": "2026-04-23 10:40:00",
  "status": 400,
  "error": "Validation Failed",
  "message": "name: Name is required; email: Email must be valid; password: Password must be at least 6 characters; role: Role is required",
  "path": "/api/users/register"
}
How validation error collection works
1
ex.getBindingResult()
Gets the validation result object that contains all field errors.
2
.getFieldErrors()
Returns a list of FieldError objects, one for each annotation that failed. If name fails @NotBlank and @Size, you get two entries.
3
.stream().map()
Converts each error to the format "fieldName: error message". The getDefaultMessage() returns the message you set in the annotation.
4
.collect(Collectors.joining("; "))
Joins all error strings with a semicolon separator into one message string.
In Nimbus

This handler is added to the same GlobalExceptionHandler class. Now the complete handler has four @ExceptionHandler methods: DuplicateEmailException → 400, BookingNotFoundException → 404, MethodArgumentNotValidException → 400 with field details, Exception → 500 catch-all.

Section 08

HTTP Status Code Best Practices

Choosing the right status code for each situation. A reference you'll use constantly.

Success Codes (2xx)

Code Meaning Nimbus Usage
200 OK Request succeeded GET booking, GET bookings list, PUT status update
201 Created New resource created POST register, POST create booking
204 No Content Succeeded, nothing to return DELETE booking (if implemented)

Client Error Codes (4xx)

Code Meaning Nimbus Usage
400 Bad Request Invalid input data Validation failed, duplicate email, invalid booking data
401 Unauthorized Not authenticated No token or expired token
403 Forbidden Authenticated but not allowed Customer tries to update status (officer-only)
404 Not Found Resource doesn't exist Booking ID not found, user ID not found
409 Conflict Conflicting state Trying to book when already booked

Server Error Codes (5xx)

Code Meaning Nimbus Usage
500 Internal Server Error Unexpected server failure Database down, NullPointerException, any unhandled error
503 Service Unavailable Server temporarily unavailable Maintenance mode, overloaded
The Golden Rules
  • 400 vs 404: 400 means "I understood your request but the data is wrong." 404 means "I can't find what you asked for." Sending POST /api/register with no name = 400. Requesting GET /api/bookings/9999 that doesn't exist = 404.
  • 401 vs 403: 401 means "I don't know who you are" (no token). 403 means "I know who you are, but you're not allowed to do this" (customer trying officer actions).
  • Never return 200 for errors. Some APIs return 200 with {"success": false, "error": "..."}. This is an anti-pattern. HTTP status codes exist for this exact purpose.

Self-Check Questions
  1. Why should APIs never return stack traces? (Security risk — exposes internal class names, file paths, and application structure)
