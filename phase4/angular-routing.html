Section 01

RouterModule & Routes

Angular routing maps URL paths to components. When the URL changes, Angular swaps the displayed component without reloading the page.

How Routing Works

User clicks link /bookings/1001
→
Router matches finds matching route
→
Component loads BookingDetailComponent
→
Renders in <router-outlet>
No page reload — Angular swaps the component in the router-outlet placeholder

Defining Routes

app-routing.module.ts 
import { NgModule } from '@angular/core';
import { RouterModule, Routes } from '@angular/router';

import { LoginComponent } from './components/login/login.component';
import { RegisterComponent } from './components/register/register.component';
import { DashboardComponent } from './components/dashboard/dashboard.component';
import { BookingFormComponent } from './components/booking-form/booking-form.component';
import { BookingDetailComponent } from './components/booking-detail/booking-detail.component';
import { BookingHistoryComponent } from './components/booking-history/booking-history.component';

const routes: Routes = [
  { path: '', redirectTo: '/login', pathMatch: 'full' },
  { path: 'login', component: LoginComponent },
  { path: 'register', component: RegisterComponent },
  { path: 'dashboard', component: DashboardComponent },
  { path: 'bookings/new', component: BookingFormComponent },
  { path: 'bookings/history', component: BookingHistoryComponent },
  { path: 'bookings/:id', component: BookingDetailComponent },
  { path: '**', redirectTo: '/login' }  // Wildcard: catch-all for unknown URLs
];

@NgModule({
  imports: [RouterModule.forRoot(routes)],
  exports: [RouterModule]
})
export class AppRoutingModule { }
Line-by-line explanation
12
const routes: Routes = [...]
An array of route objects. Each maps a URL path to a component. Angular checks them top-to-bottom and uses the first match.
13
{ path: '', redirectTo: '/login', pathMatch: 'full' }
When the URL is exactly / (empty path), redirect to /login. The pathMatch: 'full' ensures this only matches the exact empty path, not every URL that starts with nothing.
18
{ path: 'bookings/:id', component: BookingDetailComponent }
The :id is a route parameter. /bookings/1001 matches this route and passes 1001 as the id parameter to the component.
19
{ path: '**', redirectTo: '/login' }
Wildcard route. Matches any URL that didn't match above. Redirects to login. Always put this last.
23
RouterModule.forRoot(routes)
forRoot() registers the routes at the application level. Only called once in the root module. Feature modules use forChild().

The Router Outlet

app.component.html — Where routed components render 
<!-- This stays on every page -->
<app-navbar></app-navbar>

<!-- This is where Angular puts the routed component -->
<router-outlet></router-outlet>

<!-- When URL is /login, LoginComponent renders here -->
<!-- When URL is /dashboard, DashboardComponent renders here -->
<!-- The navbar stays, only the content below it changes -->
Route Order Matters

Angular checks routes top-to-bottom and stops at the first match. Put specific routes before general ones. bookings/new must come before bookings/:id, otherwise Angular would match "new" as an ID parameter.

Section 02

routerLink for Navigation

Navigate between pages without reloading. Never use regular <a href> in Angular.

navbar.component.html — Navigation links 
<nav>
  <!-- Static routes -->
  <a routerLink="/dashboard" routerLinkActive="active">Dashboard</a>
  <a routerLink="/bookings/new" routerLinkActive="active">New Booking</a>
  <a routerLink="/bookings/history" routerLinkActive="active">History</a>

  <!-- Dynamic route with parameter -->
  <a [routerLink]="['/bookings', booking.bookingId]">View Details</a>
  <!-- Generates: /bookings/1001 -->
</nav>
<a href> (Wrong)

<a href="/dashboard">

Causes a full page reload. The browser downloads everything again. Angular app restarts from scratch. Slow and loses component state.

routerLink (Correct)

<a routerLink="/dashboard">

No page reload. Angular swaps the component instantly. State is preserved. Fast SPA navigation.

Programmatic Navigation

login.component.ts — Navigate after login 
import { Router } from '@angular/router';

export class LoginComponent {
  constructor(private router: Router) {}

  onLogin() {
    this.authService.login(this.email, this.password).subscribe({
      next: (response) => {
        // Login successful — navigate to dashboard
        this.router.navigate(['/dashboard']);
      },
      error: (err) => {
        this.errorMessage = 'Invalid email or password';
      }
    });
  }
}
Section 03

Route Parameters

Pass data through the URL. The booking ID in /bookings/1001 is a route parameter.

ActivatedRoute — Reading Parameters

booking-detail.component.ts 
import { Component, OnInit } from '@angular/core';
import { ActivatedRoute } from '@angular/router';
import { BookingService } from '../../services/booking.service';

@Component({
  selector: 'app-booking-detail',
  templateUrl: './booking-detail.component.html'
})
export class BookingDetailComponent implements OnInit {
  booking: any;
  loading = true;

  constructor(
    private route: ActivatedRoute,
    private bookingService: BookingService
  ) {}

  ngOnInit() {
    // Get the :id parameter from the URL
    const id = Number(this.route.snapshot.paramMap.get('id'));

    // Fetch booking from the API
    this.bookingService.getById(id).subscribe({
      next: (data) => {
        this.booking = data;
        this.loading = false;
      },
      error: (err) => {
        console.error('Booking not found', err);
        this.loading = false;
      }
    });
  }
}

Reactive Parameter Subscription

If the user navigates from /bookings/1001 to /bookings/1002, Angular reuses the same component instance. The snapshot approach won't detect the change. Use paramMap observable instead:

Reactive route params — Detects parameter changes 
ngOnInit() {
  this.route.paramMap.subscribe(params => {
    const id = Number(params.get('id'));
    this.loadBooking(id);  // Fires every time :id changes
  });
}

Query Parameters

Reading query parameters: /bookings?status=DELIVERED 
// Setting query parameters
this.router.navigate(['/bookings'], {
  queryParams: { status: 'DELIVERED', userId: 1 }
});
// URL becomes: /bookings?status=DELIVERED&userId=1

// Reading query parameters
this.route.queryParamMap.subscribe(params => {
  const status = params.get('status');   // 'DELIVERED'
  const userId = params.get('userId');    // '1' (always a string)
});

Child / Nested Routes

Routes within routes. The dashboard has sub-pages, each rendered inside the dashboard layout.

app-routing.module.ts — Nested routes 
const routes: Routes = [
  { path: 'login', component: LoginComponent },
  {
    path: 'dashboard',
    component: DashboardComponent,
    children: [
      { path: '', component: DashboardHomeComponent },         // /dashboard
      { path: 'bookings', component: BookingListComponent },   // /dashboard/bookings
      { path: 'profile', component: ProfileComponent },        // /dashboard/profile
    ]
  }
];
dashboard.component.html — Has its own router-outlet 
<div class="dashboard-layout">
  <aside class="sidebar">
    <a routerLink="/dashboard" routerLinkActive="active"
       [routerLinkActiveOptions]="{exact: true}">Home</a>
    <a routerLink="/dashboard/bookings" routerLinkActive="active">Bookings</a>
    <a routerLink="/dashboard/profile" routerLinkActive="active">Profile</a>
  </aside>
  <main>
    <!-- Child components render HERE -->
    <router-outlet></router-outlet>
  </main>
</div>

The dashboard layout (sidebar + main area) stays fixed. Only the content inside the nested <router-outlet> changes as the user navigates between dashboard sub-pages.

Section 05

CanActivate Guard — Auth Protection

Prevent unauthorized users from accessing protected pages. If not logged in, redirect to login.

Creating an Auth Guard

guards/auth.guard.ts 
import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { AuthService } from '../services/auth.service';

export const authGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  if (authService.isLoggedIn()) {
    return true;  // Allow access
  }

  // Not logged in — redirect to login with return URL
  router.navigate(['/login'], {
    queryParams: { returnUrl: state.url }
  });
  return false;  // Block access
};

Applying the Guard to Routes

app-routing.module.ts — Protected routes 
const routes: Routes = [
  { path: 'login', component: LoginComponent },
  { path: 'register', component: RegisterComponent },

  // These routes require authentication
  { path: 'dashboard', component: DashboardComponent, canActivate: [authGuard] },
  { path: 'bookings/new', component: BookingFormComponent, canActivate: [authGuard] },
  { path: 'bookings/history', component: BookingHistoryComponent, canActivate: [authGuard] },
  { path: 'bookings/:id', component: BookingDetailComponent, canActivate: [authGuard] },

  { path: '**', redirectTo: '/login' }
];
In Nimbus

The authGuard checks if the user has a valid session (typically a token in localStorage). If not, it redirects to /login with a returnUrl query parameter. After successful login, the login component reads returnUrl and navigates the user back to where they originally wanted to go.

Section 06

Role-Based Guard

Customers see their bookings. Officers update statuses. Different roles, different access.

guards/role.guard.ts 
import { inject } from '@angular/core';
import { CanActivateFn, Router } from '@angular/router';
import { AuthService } from '../services/auth.service';

export const roleGuard: CanActivateFn = (route, state) => {
  const authService = inject(AuthService);
  const router = inject(Router);

  // Get required roles from route data
  const requiredRoles = route.data['roles'] as string[];
  const userRole = authService.getUserRole();

  if (requiredRoles.includes(userRole)) {
    return true;  // User has the required role
  }

  // Wrong role — redirect to dashboard
  router.navigate(['/dashboard']);
  return false;
};
app-routing.module.ts — Role-protected routes 
const routes: Routes = [
  // Both customers and officers can see the dashboard
  {
    path: 'dashboard',
    component: DashboardComponent,
    canActivate: [authGuard]
  },

  // Only customers can create bookings
  {
    path: 'bookings/new',
    component: BookingFormComponent,
    canActivate: [authGuard, roleGuard],
    data: { roles: ['CUSTOMER'] }
  },

  // Only officers can update booking status
  {
    path: 'officer/update-status',
    component: StatusUpdateComponent,
    canActivate: [authGuard, roleGuard],
    data: { roles: ['OFFICER', 'ADMIN'] }
  }
];
Frontend Guards Are Not Security

Route guards only control what the user sees. They don't prevent API access. A tech-savvy user could bypass Angular entirely and call the API with curl. Real authorization must happen on the backend too — the Spring Boot controller should check the user's role before performing restricted actions. Frontend guards are for UX, backend checks are for security.

Section 07

Lazy Loading Modules

Don't load everything upfront. Load feature modules only when the user navigates to them.

The Problem

Without lazy loading, Angular bundles every component into one large JavaScript file. The user downloads the officer dashboard, admin panel, and booking history — even if they're a customer who never visits those pages.

Lazy Loading a Feature Module

app-routing.module.ts — Lazy loaded routes 
const routes: Routes = [
  { path: 'login', component: LoginComponent },

  // Lazy load the officer module only when an officer navigates here
  {
    path: 'officer',
    loadChildren: () => import('./officer/officer.module')
      .then(m => m.OfficerModule),
    canActivate: [authGuard, roleGuard],
    data: { roles: ['OFFICER'] }
  },

  // Lazy load booking module
  {
    path: 'bookings',
    loadChildren: () => import('./bookings/bookings.module')
      .then(m => m.BookingsModule),
    canActivate: [authGuard]
  }
];
bookings/bookings-routing.module.ts 
const routes: Routes = [
  { path: '', component: BookingListComponent },         // /bookings
  { path: 'new', component: BookingFormComponent },      // /bookings/new
  { path: 'history', component: BookingHistoryComponent }, // /bookings/history
  { path: ':id', component: BookingDetailComponent }     // /bookings/1001
];

@NgModule({
  imports: [RouterModule.forChild(routes)],  // forChild, not forRoot
  exports: [RouterModule]
})
export class BookingsRoutingModule { }
When to Lazy Load

Lazy load feature modules that not every user needs: admin panel, officer dashboard, reports section. Keep the login, registration, and main dashboard in the main bundle since every user visits them. Angular CLI generates separate chunk files for lazy-loaded modules automatically.

The complete routing setup for the Nimbus Angular frontend.

app-routing.module.ts — Complete Nimbus routing 
import { NgModule } from '@angular/core';
import { RouterModule, Routes } from '@angular/router';
import { authGuard } from './guards/auth.guard';
import { roleGuard } from './guards/role.guard';

import { LoginComponent } from './components/login/login.component';

import { BookingHistoryComponent } from './components/booking-history/booking-history.component';
import { StatusUpdateComponent } from './components/status-update/status-update.component';

const routes: Routes = [
  // Public routes (no guard)
  { path: '', redirectTo: '/login', pathMatch: 'full' },

  { path: 'register', component: RegisterComponent },

  // Customer routes (auth required)
  {
    path: 'dashboard',
    component: DashboardComponent,
    canActivate: [authGuard]
  },
  {
    path: 'bookings/new',
    component: BookingFormComponent,
    canActivate: [authGuard, roleGuard],

  },
  {
    path: 'bookings/history',
    component: BookingHistoryComponent,
    canActivate: [authGuard]
  },
  {
    path: 'bookings/:id',
    component: BookingDetailComponent,
    canActivate: [authGuard]
  },

  // Officer routes (auth + role required)
  {
    path: 'officer/update-status',
    component: StatusUpdateComponent,
    canActivate: [authGuard, roleGuard],
    data: { roles: ['OFFICER', 'ADMIN'] }
  },

  // Wildcard
  { path: '**', redirectTo: '/login' }
];

@NgModule({
  imports: [RouterModule.forRoot(routes)],

Practice Exercises

Build routing from scratch, step by step.

Exercise 1: Basic Routing (Easy)

Exercise 1

Create a new Angular app with ng new nimbus-frontend --routing. Add three components: LoginComponent, DashboardComponent, BookingListComponent. Configure routes so /login, /dashboard, and /bookings each show the correct component. Add routerLink navigation between them.

Exercise 2: Route Parameters (Medium)

Exercise 2

Add a BookingDetailComponent with route /bookings/:id. In the component, use ActivatedRoute to read the :id parameter and display it on the page. From the booking list, add [routerLink] links that navigate to /bookings/1001, /bookings/1002, etc.

Exercise 3: Auth Guard (Medium)

Exercise 3

Create an AuthService with an isLoggedIn() method that checks localStorage for a token. Create an authGuard using the functional guard pattern. Apply it to the dashboard and booking routes. Test: accessing /dashboard without logging in should redirect to /login.

Exercise 4: Role-Based Access (Hard)

Exercise 4

Extend Exercise 3 with role-based access. Store the user's role in localStorage after login. Create a roleGuard that reads required roles from route.data. Create an officer-only page at /officer/update-status. Test: a CUSTOMER trying to access this route should be redirected to /dashboard.

Self-Check Questions
  1. Why use routerLink instead of href? (routerLink navigates without page reload, keeping Angular SPA state intact)
  2. What's the difference between snapshot.paramMap and paramMap.subscribe()? (Snapshot reads once; subscribe detects parameter changes when the component is reused)
  3. What does canActivate do? (Runs a guard function before allowing navigation to a route; returns true/false)
  4. Why do you need backend authorization in addition to frontend guards? (Frontend guards only hide UI; the API is still accessible directly via curl or Postman)
