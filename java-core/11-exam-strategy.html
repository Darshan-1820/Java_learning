Exam Strategy

How to Approach PRA Questions (Beginner's Guide)

You're new to Java. The question looks like 3 pages of text. The timer is running. Your mind goes blank. This guide is for exactly that moment.

The Truth About PRA Questions

Every PRA question follows the same pattern. Once you see the pattern, the 3-page question becomes a 5-step recipe. Here's the pattern:

  1. Create classes with fields, constructor, getters
  2. Read input using Scanner (build objects)
  3. Store objects in a List or Map
  4. Call methods that filter/calculate on those collections
  5. Print results in the exact format shown

That's it. Every single question. The only things that change are the class names, the number of fields, and what the methods calculate.

Step 1

How to Read & Understand the Question

Don't read top to bottom like a novel. Read it in this order:

  • Step 1: Skip the warning text. The plagiarism warning at the top is NOT part of the question. Skip it completely.
  • Step 2: Read the class definitions. This tells you what objects you'll create. Write down the class name and its fields. Example: Song has songId, songName, duration, genre.
  • Step 3: Read the method signatures. Look at the method names and parameters. This tells you what you need to compute. Example: findSongsByGenre(List<Musician>, String genre) — OK, I need to filter songs by genre.
  • Step 4: Jump to Sample Input/Output. This is GOLD. The sample I/O tells you the exact format. Read it carefully. Count the lines. Match each input line to a field.
  • Step 5: Read the output format. Copy the exact output text. Spaces, colons, capitalization — everything matters.
  • Step 6: Now read the details. Go back and read the full method descriptions. Now they'll make sense because you've seen the sample.
The #1 Reason People Fail

They read the question once, panic, and start typing random code. Spend 5-10 minutes understanding the question. Trace through the sample input with pen and paper. If you understand the sample, you can write the code.

Step 2

Creating Classes (The Template)

Every class follows the exact same structure. Memorize this template:

The Universal Class Template 
class Song {
    // Step 1: Private fields (from the question)
    private int songId;
    private String songName;
    private int duration;
    private String genre;

    // Step 2: Constructor (same order as fields)
    public Song(int songId, String songName, int duration, String genre) {
        this.songId = songId;
        this.songName = songName;
        this.duration = duration;
        this.genre = genre;
    }

    // Step 3: Getters (one per field)
    public int getSongId() { return songId; }
    public String getSongName() { return songName; }
    public int getDuration() { return duration; }
    public String getGenre() { return genre; }
}
Speed Trick

For EVERY class in the question, just follow this template:

  1. Copy the field names from the question
  2. Make them private
  3. Write a constructor that takes ALL fields as parameters
  4. Write getXxx() for each field

Don't think. Just apply the template. You'll write 3-4 classes in 5 minutes.

Core Skill

Traversing From One List to Another

This is the most common thing in PRA. You have a Musician who has a List of Songs. Or a Brand that has a List of Mobiles. How do you get from the outer object to the inner list?

  Musician                    Songs (List)
  +------------------+       +------------------+
  | musicianId: 1    |       | songId: 101      |
  | musicianName:    |       | songName: "Rise" |
  |   "Alice"        | ----> | duration: 200    |
  | songs: [Song,    |       | genre: "Pop"     |
  |         Song]    |       +------------------+
  +------------------+       | songId: 102      |
                              | songName: "Fall" |
                              | duration: 180    |
                              | genre: "Rock"    |
                              +------------------+
One Musician "has" many Songs. This is called Composition.

The Pattern: Nested For Loops

Traversing Outer List → Inner List 
// I have a List<Musician>. I want to find all Pop songs.

// OUTER LOOP: Go through each Musician
for (Musician m : musicians) {

    // INNER LOOP: Go through each Song of THIS musician
    for (Song s : m.getSongs()) {

        // Now I can check each song's properties
        if (s.getGenre().equalsIgnoreCase("Pop")) {
            System.out.println(s.getSongName());
        }
    }
}
How to Read This Code

Think of it like opening boxes:

  1. Big box: musicians — a list of Musician objects
  2. Open one box: m — one Musician. Now I can call m.getSongs()
  3. Inside that box, smaller boxes: m.getSongs() — returns a List of Songs
  4. Open each small box: s — one Song. Now I can call s.getGenre()

More Examples

Brand → Mobiles (find mobiles in storage range) 
for (Brand b : brands) {                      // outer: each brand
    for (Mobile mob : b.getMobileList()) {     // inner: each mobile of this brand
        if (mob.getStorage() >= min && mob.getStorage() <= max) {
            // Found a mobile in range! Print it with brand name
            System.out.println("Brand: " + b.getBrandName() + ", Model: " + mob.getModelName());
        }
    }
}
Subscription → Customers (build customer-to-subscriptions map) 
// Build a Map: customerId -> List of their Subscriptions
Map<Integer, List<Subscription>> map = new LinkedHashMap<>();

for (Subscription sub : allSubscriptions) {           // outer: each subscription
    for (Customer c : sub.getSubscribedCustomers()) { // inner: each customer of this sub
        // Add this subscription to this customer's list
        map.computeIfAbsent(c.getCustomerId(), k -> new ArrayList<>()).add(sub);
    }
}
The Rule

Whenever you see "Class A has a List of Class B" in the question, you WILL need nested loops. Outer loop = Class A. Inner loop = call a.getListOfB().

Where to Print: In the Method or in Main?

This confuses everyone. Here's the simple rule.

Print in the Method (void methods)

When the question says the method should "print" or "display" the results.

The method return type is void. It prints directly.

// Question says: "print the averages"
public static void getAveragesByType(
    List<Car> cars, String type) {
    // ... calculate ...
    System.out.println("avgTankCapacity: " + avg);
    // prints INSIDE the method
}

// In main:
getAveragesByType(cars, searchType);
// Just call it. No System.out.println() needed.
Return from Method, Print in Main

When the question says the method should "return" a value/list/map.

The method has a return type (List, Map, double, etc). Main receives it and prints.

// Question says: "return the list of songs"
public static List<String> findSongsByGenre(
    List<Musician> musicians, String genre) {
    List<String> result = new ArrayList<>();
    // ... add matching songs to result ...
    return result;  // returns, doesn't print
}

// In main:
List<String> songs = findSongsByGenre(musicians, genre);
if (songs.isEmpty()) {
    System.out.println("Songs in " + genre + " genre are not present.");
} else {
    System.out.println("Songs in " + genre + " genre :");
    for (String s : songs) {
        System.out.println(s);
    }
}
How to Decide: 3 Clues
Clue in QuestionWhere to PrintMethod Return Type
"print the result" / "display"Inside the methodvoid
"return the list/map/value"In main (after calling)List, Map, double, etc.
"return null if not found"In main (check null, then print)Object type
"the main method should print"In mainReturn type given in question
Common Trap

Some questions have methods that return a value, but then say "if the returned value is null/zero, main should print an error message." This means:

  • The method RETURNS (doesn't print)
  • Main receives the result
  • Main checks if it's null/zero/empty
  • Main prints the success output OR the error message
Core Skill

Custom Exceptions & Try-Catch

Some questions ask you to throw custom exceptions. It sounds scary but it's only 3 lines of code.

Creating a Custom Exception (2 lines)

Custom Exception — It's just a class that extends Exception 
// That's literally it. Two lines.
class AverageCannotBeCalculatedException extends Exception {
    public AverageCannotBeCalculatedException(String message) {
        super(message);  // passes the message to the parent Exception class
    }
}

Throwing a Custom Exception

Inside your method — throw when condition is met 
// Method signature MUST say "throws" the exception
public static Map<String, Double> calculateAveragePrice(List<Brand> brands)
    throws AverageCannotBeCalculatedException {

    // Check if something is wrong
    if (brands.isEmpty() || /* no mobiles */ ) {
        // THROW the exception with the message from the question
        throw new AverageCannotBeCalculatedException("Average cannot be calculated");
    }

    // Normal logic here...
    return result;
}

Catching in Main (try-catch)

In main — wrap method calls in try-catch 
// In main method:
try {
    Map<String, Double> result = calculateAveragePrice(brands);
    // If we reach here, no exception was thrown
    System.out.println("Average Price of each brand:");
    for (Map.Entry<String, Double> entry : result.entrySet()) {
        System.out.println(entry.getKey() + ": " + entry.getValue());
    }
} catch (AverageCannotBeCalculatedException e) {
    // If exception was thrown, we land here
    // Print EXACTLY what the question asks
    System.out.println("AverageCannotBeCalculatedException: " + e.getMessage());
}
The Output Format for Exceptions

Read the question carefully. The output usually follows one of these patterns:

Question SaysPrint This
"throw exception with message 'Average cannot be calculated'"AverageCannotBeCalculatedException: Average cannot be calculated
"print the name of the exception"e.getClass().getSimpleName() → prints just ArithmeticException
"print the exception message"e.getMessage() → prints just the message text

Simple Exception Handling (Division-type questions)

When the question says "handle all exceptions" 
try {
    Scanner sc = new Scanner(System.in);
    int a = sc.nextInt();
    int b = sc.nextInt();
    System.out.println(a / b);
} catch (Exception e) {
    // Catches ANY exception (ArithmeticException, InputMismatchException, etc.)
    System.out.println(e.getClass().getSimpleName());
    // For 15/0: prints "ArithmeticException"
    // For "abc"/5: prints "InputMismatchException"
}
Survival Guide

What to Do When You're Stuck

Your mind is blank. You've read the question 3 times. Nothing is clicking. Here's your emergency protocol:

  • Step 1: Write the classes first. Don't think about logic. Just look at the class definitions in the question and write them using the template (private fields, constructor, getters). This is free marks and gets your fingers moving.
  • Step 2: Write the Scanner input code. Look at the sample input. Count the lines. Write code to read each line in order. Create objects as you go. This is mechanical — no thinking required.
  • Step 3: Write an empty method with the correct signature. Copy the method name and parameters from the question. Set the return type. Return null/0/empty list for now. At least the code compiles.
  • Step 4: Focus on ONE method at a time. Pick the simpler method. Look at the sample output for that method. Think: "What data goes in? What comes out?" Write a for loop. Filter. Calculate. Return.
  • Step 5: Match the output EXACTLY. Run the sample input mentally. Does your output match? Check every space, colon, and capital letter.
The "I Have No Idea" Emergency Plan

If you truly cannot figure out the method logic, at least do this:

  1. Write all classes correctly (20-30% marks)
  2. Write the input reading in main (10-20% marks)
  3. Write the method signature with correct return type
  4. In the method, write a for loop that iterates through the input list
  5. Inside the loop, write an if-statement comparing with equalsIgnoreCase

Even if the logic isn't perfect, you'll get partial marks for structure. Never submit a blank method.

Time Management: The 15-60 Rule

Do the minor question first (15 min). It's usually simple — String manipulation, basic math, simple loops. Bank those marks.

Then spend the remaining 60 minutes on the major question. Even if you don't finish the major, you've secured the minor marks.

Scanner Input: The Reading Pattern

Most PRA questions use nextLine() for everything and then parse. Here's the safe pattern:

Safe Scanner Pattern (works for ALL PRA questions) 
Scanner sc = new Scanner(System.in);

// Read a number (how many objects to create)
int n = Integer.parseInt(sc.nextLine().trim());

for (int i = 0; i < n; i++) {
    // Read each field on its own line
    String name = sc.nextLine().trim();
    double price = Double.parseDouble(sc.nextLine().trim());
    int count = Integer.parseInt(sc.nextLine().trim());

    // Read nested objects
    int numInner = Integer.parseInt(sc.nextLine().trim());
    List<InnerClass> innerList = new ArrayList<>();
    for (int j = 0; j < numInner; j++) {
        // read inner object fields...
        innerList.add(new InnerClass(...));
    }

    // Create the outer object with its inner list
    outerList.add(new OuterClass(name, price, count, innerList));
}
The nextInt() + nextLine() Bug

If you mix sc.nextInt() with sc.nextLine(), the nextLine() will read the leftover newline from nextInt() and give you an empty string. Solution: Use Integer.parseInt(sc.nextLine().trim()) for everything. It's longer but it never bugs out.

Critical Rule

String Comparison: equalsIgnoreCase

Every single PRA question says "All string comparisons must be case-insensitive." This means:

NEVER do this
// Using == compares references, not values
if (genre == "Pop")         // WRONG

// Using .equals() is case-sensitive
if (genre.equals("Pop"))    // WRONG (won't match "pop" or "POP")
ALWAYS do this
// Case-insensitive comparison
if (genre.equalsIgnoreCase("Pop"))  // CORRECT

// When comparing two variables
if (s.getGenre().equalsIgnoreCase(searchGenre))  // CORRECT
Critical Rule

Insertion Order: LinkedHashMap & LinkedHashSet

Almost every PRA question says "Preserve insertion order." This means:

NeedUseNOT
Map with insertion ordernew LinkedHashMap<>()new HashMap<>()
Set with insertion ordernew LinkedHashSet<>()new HashSet<>()
List (already ordered)new ArrayList<>()—

If you use HashMap instead of LinkedHashMap, the output order will be random and your test cases will fail even though the values are correct.

Final

Exam Day Checklist

Before you hit Submit, check these:

  • Class name is Solution (not Main, not your custom name)
  • All string comparisons use equalsIgnoreCase()
  • Maps use LinkedHashMap (not HashMap)
  • Output format matches EXACTLY (spaces, colons, capitalization)
  • Scanner reads in the correct order (match sample input line by line)
  • No extra prints (debug System.out.println lines removed)
  • "not found" messages match exactly (copy from question, don't paraphrase)
  • Minor question is done (don't leave free marks on the table)
