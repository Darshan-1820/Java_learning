Mock 7 — Question 1

Exception Handling — Division

Minor 10–15 min

Question Statement

Write a program that divides two given integers a and b provided as input and print the integral value after division.

The program should handle all possible exceptions that might occur during runtime, such as dividing by zero, invalid inputs, or other runtime issues.

If an exception occurs, the program should print the name of the exception, ensuring the program does not terminate abnormally.

Method Used: (exception_variable_name).getClass().getSimpleName()

Analysis

What This Question Tests

This is a straightforward exception handling question. Three core concepts are being tested:

Core Concepts Tested
  • try-catch block: Wrapping risky code inside try and catching failures in catch — the foundation of Java exception handling.
  • getClass().getSimpleName(): Instead of printing a full error message, they want just the exception class name (e.g., ArithmeticException, not / by zero).
  • Integer division: Java integer division truncates toward zero. 7 / 3 = 2, not 2.33. -7 / 3 = -2, not -3.
Walkthrough

Tracing Through Each Test Case

Let's follow the code path for every input.

  try block runs → a = 10, b = 2

  No exception thrown. Output: 5

  15 / 0 → BOOM! Java throws ArithmeticException
  Jumps to catch block → e.getClass().getSimpleName()
  Output: ArithmeticException

Hidden TC4: Input = -7 3
  try block runs → a = -7, b = 3
  -7 / 3 = -2 (truncates toward zero, NOT toward negative infinity)
  No exception thrown. Output: -2

Hidden TC5: Input = 0 5
  try block runs → a = 0, b = 5
  0 / 5 = 0 (zero divided by anything is 0)
  No exception thrown. Output: 0
All 5 test cases traced through the code
Watch: Integer Division Truncates Toward Zero

-7 / 3 = -2, not -3. Java truncates toward zero, not toward negative infinity. This is different from Python's // operator which floors toward negative infinity.

Algorithm

The Algorithm Step by Step

  • Step 1: Wrap everything inside a try block — read two integers from Scanner and divide them.
  • Step 2: Print the result of a / b (integer division).
  • Step 3: In the catch block, catch the general Exception class and print e.getClass().getSimpleName().
Why Catch General Exception?

The question says "handle all possible exceptions." By catching Exception (the parent class), you catch everything: ArithmeticException (divide by zero), InputMismatchException (non-integer input), and any other runtime issue. One catch handles them all.

Full Code

Complete Solution

This is possibly the shortest solution you'll ever write. 10 lines total.

Solution.java — Complete 
import java.util.Scanner;

public class Solution {
    public static void main(String[] args) {
        try {
            Scanner sc = new Scanner(System.in);
            int a = sc.nextInt();
            int b = sc.nextInt();
            System.out.println(a / b);
        } catch (Exception e) {
            System.out.println(e.getClass().getSimpleName());
        }
    }
}
Line-by-line explanation
1
try { ... }
Everything that could throw an exception goes inside the try block. If any line inside throws an exception, execution immediately jumps to the catch block.
2
int a = sc.nextInt(); int b = sc.nextInt();
Read two integers from input. If the user types a non-integer (like "abc"), nextInt() throws InputMismatchException — caught by our catch block.
3
System.out.println(a / b);
Integer division. If b is 0, Java throws ArithmeticException. Otherwise, it prints the truncated result (no decimals).
4
catch (Exception e)
Catches ANY exception. Exception is the parent class of all checked and unchecked exceptions. This single catch handles ArithmeticException, InputMismatchException, and anything else.
5
e.getClass().getSimpleName()
getClass() returns the runtime class of the exception object. getSimpleName() returns just the class name without the package prefix. So you get "ArithmeticException" instead of "java.lang.ArithmeticException".

Watch Out

Common Mistakes

Wrong: getMessage()

Using e.getMessage() prints the error description like "/ by zero".

The question explicitly asks for the exception name, not the message.

Output would be: / by zero — WRONG.

Correct: getClass().getSimpleName()

Using e.getClass().getSimpleName() prints just the class name.

This is exactly what the question specifies.

Output would be: ArithmeticException — CORRECT.

Mistake What Goes Wrong Fix
Using e.getMessage() Prints "/ by zero" instead of "ArithmeticException" Use e.getClass().getSimpleName() as the question specifies
Catching specific exceptions only Catches ArithmeticException but misses InputMismatchException or others Catch the general Exception class to handle all possible exceptions
Using double division 7 / 3 gives 2.3333 instead of 2 Keep both variables as int — integer division truncates automatically
Forgetting integer division truncates toward zero Expecting -7 / 3 = -3 instead of -2 Java truncates toward zero: -7 / 3 = -2, not -3
