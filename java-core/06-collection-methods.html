Pattern 01

The Filter Pattern

The most common method type you'll see in PRA. The question gives you a collection and a condition — you return only the items that match. This pattern shows up in almost every PRA question as one of the two required methods.

The structure is always the same, no matter what class or condition:

  1. Create an empty result list — this will hold the matches
  2. Loop through every item in the input collection
  3. Check the condition — does this item match what we're looking for?
  4. If yes, add it to the result list
  5. Return the result list — even if it's empty

Real PRA Example: Find Investors with Negative Gains

From Mock 1 Q1 — the question asks: "return all investors whose gains are less than 0."

Solution.java — Filter Method 
public static ArrayList<Investor> negativeGainsInvestorNames(Collection<Investor> investors) {
    ArrayList<Investor> result = new ArrayList<>();  // 1. Empty result list
    for (Investor inv : investors) {                    // 2. Loop through all
        if (inv.getGains() < 0) {                      // 3. Check condition
            result.add(inv);                            // 4. Add match
        }
    }
    return result;                                      // 5. Return results
}
Line-by-line explanation
1
ArrayList<Investor> result = new ArrayList<>();
Create a fresh empty list. This is where we'll put all investors that match our condition. We always start with an empty list.
2
for (Investor inv : investors)
Enhanced for loop — go through every Investor object in the collection one by one. The variable inv holds the current investor.
3
if (inv.getGains() < 0)
The filter condition. We call the getter method to get the gains value and check if it's negative. Only investors with negative gains pass this check.
4
result.add(inv);
If the condition was true, add this investor to our result list. If the condition was false, we skip this investor and move to the next one.
5
return result;
Return the result list. It might have 0 items, 1 item, or all items — depends on how many matched the condition.

💡 Think About It

The filter pattern is like a security guard at a gate. Every person (object) in the queue (collection) walks up. The guard checks their ID (condition). If they pass, they go through to the VIP area (result list). If they don't, they stay out. At the end, you have a list of only the people who passed.

Pattern 02

The Aggregate Pattern

The second most common PRA method type. Instead of returning a list of matching items, you calculate a single value: a sum, average, count, or product from the collection.

The structure follows a slightly different flow:

  1. Initialize accumulators — sum = 0, count = 0
  2. Loop through every item in the collection
  3. Check condition (if applicable) — not all aggregates need a filter
  4. Accumulate — add to sum, increment count
  5. Calculate final result — divide sum by count for average
  6. Return the result

Real PRA Example: Average of Positive Gains

Solution.java — Aggregate Method 
public static double averageOfPositiveGains(Collection<Investor> investors) {
    double sum = 0;     // accumulator for total
    int count = 0;      // accumulator for how many
    for (Investor inv : investors) {
        if (inv.getGains() > 0) {       // only positive gains
            sum += inv.getGains();       // add to running total
            count++;                      // one more positive investor
        }
    }
    return count > 0 ? sum / count : 0;  // safe division
}
⚠ Critical Warning

ALWAYS check count > 0 before dividing! If no items match your condition, count stays at 0. Dividing by zero crashes your program on HackerRank and you get 0 marks. The ternary operator count > 0 ? sum / count : 0 is your safety net.

Common Aggregate Operations

Operation Accumulators Inside Loop Final Return
Sum double sum = 0 sum += item.getX() return sum
Average double sum = 0; int count = 0 sum += item.getX(); count++ return sum / count
Count int count = 0 count++ return count
Max double max = Double.MIN_VALUE if (item.getX() > max) max = item.getX() return max
Min double min = Double.MAX_VALUE if (item.getX() < min) min = item.getX() return min
Filter Pattern

Returns: ArrayList<T> (a list)

Purpose: "Give me all items that match"

Keywords: "find all", "return collection of", "filter by"

Result: Multiple objects

Aggregate Pattern

Returns: double, int, or Double

Purpose: "Calculate a single value from all items"

Keywords: "average of", "sum of", "total", "count"

Result: Single number

Pattern 03

Searching by String (Case-Insensitive)

Many PRA questions ask you to find items by matching a string value. This gets more complex when you have nested objects (like a Cricketer who has a list of Statistics). You need nested loops to search inside inner collections.

Nested Loop Search — Cricketer by League Name

This pattern appears when one class contains a List of another class. You need two loops: the outer loop goes through each parent object, the inner loop searches through the child objects.

Solution.java — Nested Search 
public static void findCricketerByLeagueName(Collection<Cricketer> cricketers, String leagueName) {
    boolean found = false;
    for (Cricketer c : cricketers) {
        for (Statistics stat : c.getStatistics()) {           // inner loop
            if (stat.getLeagueName().equalsIgnoreCase(leagueName)) {
                System.out.println(c.getCricketerName());
                found = true;
                break;  // found in this cricketer's stats, move to next cricketer
            }
        }
    }
    if (!found) {
        System.out.println("No Cricketer Found");
    }
}
Line-by-line explanation
1
boolean found = false;
A flag to track whether we found ANY match at all. We need this to print "No Cricketer Found" if nothing matches.
2
for (Cricketer c : cricketers)
Outer loop — go through each cricketer one by one.
3
for (Statistics stat : c.getStatistics())
Inner loop — for the current cricketer, go through all their statistics entries. getStatistics() returns a List<Statistics>.
4
if (stat.getLeagueName().equalsIgnoreCase(leagueName))
Case-insensitive string comparison. "IPL" matches "ipl", "Ipl", etc. Always use equalsIgnoreCase in PRA.
5
break;
Once we found a match in this cricketer's stats, no need to check the rest of their stats. Break out of the inner loop and move to the next cricketer.
6
if (!found) System.out.println("No Cricketer Found");
After checking ALL cricketers, if found is still false, no one matched. Print the error message (exact wording from the question).
🎯 PRA Tip

ALWAYS use equalsIgnoreCase() for string comparisons in PRA. The question almost always says "case-insensitive" somewhere in the problem statement. Even if it doesn't explicitly say it, using equalsIgnoreCase() is the safe choice — you'll never lose marks for being case-insensitive, but you CAN lose marks for being case-sensitive when the test cases use different cases.

📚 Analogy

Think of it like searching a library. The outer loop is walking through each shelf (Cricketer). The inner loop is checking each book (Statistics) on that shelf. Once you find the book you want on a shelf, you note the shelf name and move to the next shelf. If you go through the entire library and find nothing, you tell the user "Book not found."

Pattern 04

Sum by Category

A combination of the aggregate pattern with nested searching. You sum a numeric value, but only for items that match a specific category (usually a string parameter).

Real PRA Example: Total Performance Average by Country

Solution.java — Sum by Category 
public static Double findTotalOfPerformanceAverageByCountryName(
        Collection<Cricketer> cricketers, String countryName) {
    double total = 0;
    boolean found = false;
    for (Cricketer c : cricketers) {
        for (Statistics stat : c.getStatistics()) {
            if (stat.getCountryName().equalsIgnoreCase(countryName)) {
                total += stat.getPerformanceAverage();
                found = true;
            }
        }
    }
    if (!found) {
        System.out.println("No Statistics are available With the Country : " + countryName);
        return null;
    }
    return total;
}

Key Differences from Simple Aggregate

  • Return type is Double (wrapper class, not primitive double) — because we need to return null when nothing is found
  • Uses boolean found flag — to distinguish between "total is 0 because values add up to 0" vs "total is 0 because nothing matched"
  • No break in inner loop — because we want to sum ALL matching statistics, not just the first one
  • Prints error AND returns null — two actions when nothing is found
⚠ Watch Out

Notice we do NOT use break in the inner loop here. In the search pattern (Section 3), we break because we just need to know IF a cricketer plays in that league. Here, we need to SUM ALL statistics for that country, so we must check every single one.

Search Pattern (Section 3)

Goal: Find IF something exists

Inner loop: Uses break on first match

Action: Print the parent's name

Why break: One match per parent is enough

Sum Pattern (Section 4)

Goal: Calculate a total from matches

Inner loop: No break — check ALL

Action: Add value to running total

Why no break: Every match contributes to the sum

Pattern 05

Discount Calculation

A common PRA variation: filter items by a condition, modify their values, and return the modified items. This combines filtering with calculation.

Real PRA Example: Calculate Discounted Price by Genre

Solution.java — Discount Calculation 
public static List<Book> calculateDiscountedPrice(List<Book> books,
        String genre, int discountPercent) {
    List<Book> result = new ArrayList<>();
    boolean found = false;
    for (Book book : books) {
        if (book.getGenre().equalsIgnoreCase(genre)) {
            double newPrice = book.getPrice() - (book.getPrice() * discountPercent / 100.0);
            book.setPrice(newPrice);       // modify the original object
            result.add(book);              // add to result
            found = true;
        }
    }
    if (!found) {
        System.out.println("Discounted books are unavailable in the given genre.");
    }
    return result;
}
Line-by-line explanation
1
double newPrice = book.getPrice() - (book.getPrice() * discountPercent / 100.0);
The discount formula: Original price minus (original price times discount percentage divided by 100). The 100.0 (not 100) ensures we get decimal division, not integer division.
2
book.setPrice(newPrice);
We modify the ORIGINAL book object using its setter. This is important — we're not creating a new Book, we're updating the existing one's price.
3
result.add(book);
Add the modified book to the result list. Now it has the new discounted price.
🎯 PRA Tip

Use 100.0 not 100 in discount formulas. In Java, 20 / 100 gives 0 (integer division), but 20 / 100.0 gives 0.2 (decimal division). This one character difference can fail all your test cases.

The Discount Formula Visualized

  Original Price: 500.0
  Discount:       20%

  Step 1: 500.0 * 20 = 10000.0
  Step 2: 10000.0 / 100.0 = 100.0    (discount amount)
  Step 3: 500.0 - 100.0 = 400.0      (new price)

  Formula: newPrice = price - (price * discount / 100.0)
Always use 100.0 to force decimal division
Templates

Reusable Method Templates

These are copy-paste templates you can use in any PRA question. Just replace the class name, field names, and condition. These three templates cover 90% of all PRA methods.

Template 1: Filter by Condition

📋 Copy-Paste Template

Use when the question says "find all", "return all", "filter", or "return a collection of items matching..."

Template — Filter 
public static ArrayList<ClassName> filterByCondition(Collection<ClassName> list) {
    ArrayList<ClassName> result = new ArrayList<>();
    for (ClassName item : list) {
        if (/* your condition here */) {
            result.add(item);
        }
    }
    return result;
}

Template 2: Aggregate (Average)

📋 Copy-Paste Template

Use when the question says "average of", "sum of", "total", "calculate the mean"...

Template — Aggregate 
public static double calculateAverage(Collection<ClassName> list) {
    double sum = 0;
    int count = 0;
    for (ClassName item : list) {
        if (/* condition - optional */) {
            sum += item.getValue();
            count++;
        }
    }
    return count > 0 ? sum / count : 0;
    // NEVER forget the count > 0 check!
}

Template 3: Search Nested List

📋 Copy-Paste Template

Use when one class has a List of another class inside it, and you need to search or sum across the inner list.

Template — Nested Search 
public static void searchNested(Collection<Parent> parents, String searchValue) {
    boolean found = false;
    for (Parent p : parents) {
        for (Child c : p.getChildren()) {
            if (c.getField().equalsIgnoreCase(searchValue)) {
                System.out.println(p.getName());
                found = true;
                break;  // move to next parent
            }
        }
    }
    if (!found) {
        System.out.println("No results found");
    }
}

Quick Reference: Which Template to Use?

Question Says Template Return Type
"Find all investors with gains < 0" Filter ArrayList<Investor>
"Average of positive gains" Aggregate double
"Find cricketers who play in league X" Nested Search void (prints directly)
"Total performance by country" Nested Search + Sum Double (nullable)
"Calculate discounted price for genre X" Filter + Modify List<Book>

Critical Concept

Return vs Print — The #1 PRA Confusion

This single confusion costs more marks than any other mistake. Here's how to NEVER get it wrong again.

How to Read the Question

Every PRA method description uses specific words. Those words tell you EXACTLY what to do:

Question Says What It Means Your Method Does Main Method Does
"returns a list/map/value to the main method" Method gives data BACK to main return result; Receives result, then prints it
"print the details" or "display" Method prints directly System.out.println(...) Just calls the method, nothing else
"throw exception with message" Method creates and throws an exception throw new MyException("msg") try-catch, prints "MyException: msg"

Example — "returns a Map"

When the question says: "This method will calculate the average price of mobiles for each brand and return a map of brand names with their corresponding average mobile price."

WRONG — void method
// Method prints directly
public static void calculateAvg(List<Brand> brands) {
    for (Brand b : brands) {
        double avg = ...;
        System.out.println(b.getName() + ": " + avg);
    }
}

// Main just calls it
calculateAvg(brands);

Problem: Question said "return a map". void returns nothing. Test cases fail because the expected behavior is to return data.

CORRECT — returns Map
// Method RETURNS the map
public static Map<String, Double> calculateAvg(List<Brand> brands) {
    Map<String, Double> result = new LinkedHashMap<>();
    for (Brand b : brands) {
        double avg = ...;
        result.put(b.getName(), avg);
    }
    return result;  // <-- gives data back
}

// Main RECEIVES and prints
Map<String, Double> avgMap = calculateAvg(brands);
for (Map.Entry<String, Double> e : avgMap.entrySet()) {
    System.out.println(e.getKey() + ": " + e.getValue());
}

Example — "throw exception"

When the question says: "If average cannot be determined, then throw custom exception AverageCannotBeCalculatedException with message 'Average cannot be calculated'"

WRONG — Exception crashes the program
// Method throws
throw new AverageCannotBeCalculatedException("Average cannot be calculated");

// Main does NOT catch it
calculateAvg(brands);  // CRASHES HERE!
// Code below never runs
findMobiles(brands, min, max);  // NEVER REACHED

Problem: Program crashes with "Exception in thread main". The second method never gets called. You lose marks for BOTH methods.

CORRECT — Catch and print
// In main — WRAP each method call in try-catch
try {
    Map<String, Double> avgMap = calculateAvg(brands);
    // print the map...
} catch (AverageCannotBeCalculatedException e) {
    System.out.println("AverageCannotBeCalculatedException: " + e.getMessage());
}

// This STILL runs even if above threw exception!
try {
    List<Mobile> mobiles = findMobiles(brands, min, max);
    // print mobiles...
} catch (NoMobileFoundException e) {
    System.out.println("NoMobileFoundException: " + e.getMessage());
}
The Golden Rule

Each method call in main gets its OWN try-catch. This way, if method 1 throws an exception, method 2 still runs. If you put both in ONE try block, the first exception skips the second method entirely.

Integer vs int — Quick Rule

This is simpler than it seems:

  • Match the question. If it says Integer brandId, use Integer. If it says int count, use int.
  • Collections NEED wrappers: List<Integer> works. List<int> does NOT compile.
  • They auto-convert (autoboxing), so Integer x = 5; and int y = x; both work fine.
  • When in doubt, use what the question says. PRA is not testing whether you know the difference — it's testing if you can follow instructions.

Reading Inputs — Follow the Sample Input Explanation

Every PRA question has a section called "Sample Input Explanation" that tells you the EXACT reading order. Follow it step by step:

InputReadingPattern.java 
Scanner sc = new Scanner(System.in);

// Step 1: "First read the number of Brand objects"
int numBrands = Integer.parseInt(sc.nextLine().trim());
List<Brand> brands = new ArrayList<>();

for (int i = 0; i < numBrands; i++) {
    // Step 2: "Read brand details (brandId, brandName, number of mobiles)"
    int brandId = Integer.parseInt(sc.nextLine().trim());
    String brandName = sc.nextLine().trim();
    int mobileCount = Integer.parseInt(sc.nextLine().trim());

    // Step 3: "For each mobile: read model number, model name, price, storage"
    List<Mobile> mList = new ArrayList<>();
    for (int j = 0; j < mobileCount; j++) {
        int modelNumber = Integer.parseInt(sc.nextLine().trim());
        String modelName = sc.nextLine().trim();
        double price = Double.parseDouble(sc.nextLine().trim());
        int storage = Integer.parseInt(sc.nextLine().trim());
        mList.add(new Mobile(modelNumber, modelName, price, storage));
    }

    brands.add(new Brand(brandId, brandName, mList));
}

// Step 4: "Next, read the parameters for the methods"
int minStorage = Integer.parseInt(sc.nextLine().trim());
int maxStorage = Integer.parseInt(sc.nextLine().trim());
Pro tip: Use Integer.parseInt(sc.nextLine().trim())

Instead of sc.nextInt(), use Integer.parseInt(sc.nextLine().trim()). This avoids the newline trap completely. You NEVER need sc.nextInt() or sc.nextDouble() if you use this approach. Every input becomes sc.nextLine().trim() and you parse it yourself.
