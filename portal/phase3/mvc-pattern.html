Section 01

What is MVC?

MVC stands for Model-View-Controller. It is an architecture pattern that separates your application into three distinct responsibilities, so each part can change independently.

The Three Components

ComponentResponsibilityIn Java Web Apps
Model Data and business logic. Knows nothing about how data is displayed. Java classes: User.java, Booking.java, UserDAO.java, BookingService.java
View Display layer. Shows data to the user. No logic beyond presentation. JSP pages: login.jsp, dashboard.jsp, tracking-result.jsp
Controller Traffic cop. Receives requests, calls the Model, picks the View. Servlets: LoginServlet, BookingServlet, TrackingServlet
Analogy: The Restaurant

Model = Kitchen. The chefs prepare the food (data). They don't care how it's served or who ordered it. They just cook what's requested and hand it over.

View = Plate & Presentation. The food is plated beautifully on a dish (HTML page). The plate doesn't cook the food — it just displays it attractively.

Controller = Waiter. The waiter takes the customer's order (HTTP request), tells the kitchen what to make (calls Service/DAO), gets the prepared dish (Model data), and brings it to the table on a plate (forwards to JSP).

If the restaurant changes its menu (Model), the waiter's job stays the same. If they redesign the plates (View), the kitchen doesn't change. Each layer is independent.

Section 02

Why MVC? — Separation of Concerns

Without MVC, your code becomes a tangled mess where HTML, database queries, and business logic live in the same file. MVC separates them so each can evolve independently.

What Happens Without MVC

Everything in one JSP — the WRONG way 
// booking.jsp — this is a DISASTER
<%
    // Database connection in JSP!
    Connection conn = DriverManager.getConnection("jdbc:mysql://...");
    String name = request.getParameter("senderName");
    double weight = Double.parseDouble(request.getParameter("weight"));

    // Business logic in JSP!
    double cost = weight * 50;
    if ("EXPRESS".equals(request.getParameter("type"))) {
        cost *= 1.5;
    }

    // Database insert in JSP!
    PreparedStatement ps = conn.prepareStatement("INSERT INTO bookings...");
    ps.setString(1, name);
    ps.setDouble(2, cost);
    ps.executeUpdate();
%>
<html>
    <h1>Booking Confirmed!</h1>
    <p>Cost: Rs. <%= cost %></p>
</html>
What's Wrong With This?
  • Database code in JSP — if you change the database, you edit every JSP page.
  • Business logic in JSP — the pricing formula is buried in HTML. No one can find it.
  • No reuse — if another Servlet needs cost calculation, you copy-paste it.
  • Untestable — you can't unit test a JSP. You need to deploy to Tomcat to check anything.
  • Team chaos — the frontend person and backend person are editing the same file.

Benefits of MVC

BenefitWhat It Means
Separation Each layer has one job. A change in one layer doesn't break the others.
Testability You can test Models (DAOs, Services) with plain JUnit — no Tomcat needed.
Reuse The same BookingService works for web, mobile API, and command-line.
Parallel work Frontend developer works on JSPs. Backend developer works on Servlets/DAOs. No conflicts.
Maintainability Bug in cost calculation? Go to BookingService. Display issue? Go to the JSP. You always know where to look.
Section 03

The MVC Flow

Every request follows the same path through the three layers. Once you understand this flow, you understand how every Java web application works.

Complete Request Flow

BrowserPOST /book (form data)
→
ControllerBookingServlet.doPost()
→
ServiceBookingService.createBooking()
↓
DAOBookingDAO.insert()
→
ModelBooking object with data
→
ControllersetAttribute() + forward()
→
Viewbooking-result.jsp
↓
BrowserSees the confirmation page
Request → Controller → Service → DAO → Model → Controller → View → Response

Step-by-Step Breakdown

  1. Browser sends request. User fills out the booking form and clicks Submit. The browser sends a POST request with form data to /book.
  2. Controller (Servlet) receives it. BookingServlet.doPost() reads the form parameters using request.getParameter().
  3. Controller calls Service. The Servlet doesn't do the business logic itself. It calls BookingService.createBooking(senderName, weight, type).
  4. Service calls DAO. The Service applies business rules (calculate cost, validate data) and calls BookingDAO.insert(booking) to save to the database.
  5. DAO returns Model. The DAO executes the SQL, gets the generated ID, and returns a Booking object (the Model) with all the data.
  6. Controller prepares the View. The Servlet puts the Booking object into the request: request.setAttribute("booking", booking).
  7. Controller forwards to View. request.getRequestDispatcher("/WEB-INF/booking-result.jsp").forward(request, response).
  8. View renders HTML. The JSP reads ${booking.cost}, ${booking.senderName}, and generates the HTML that goes back to the browser.
Section 04

Model — Data Classes

The Model layer contains your Java classes that represent data. These are the same POJO classes from Phase 2 — User, Booking, Parcel. Nothing changes.

com/nimbus/model/Booking.java 
package com.nimbus.model;

public class Booking {
    private int bookingId;
    private int senderId;
    private String senderName;
    private String receiverName;
    private String receiverAddress;
    private double weight;
    private String parcelType;  // STANDARD, EXPRESS, FRAGILE
    private double cost;
    private String status;      // BOOKED, PICKED_UP, IN_TRANSIT, DELIVERED

    // Constructors, getters, setters
    // ... same as Phase 2
}
Model Classes Are Platform-Independent

Your Booking.java class has no idea whether it's being used in a console app, a web app, or a REST API. It just holds data. That's the whole point — the Model doesn't know about HTTP, Servlets, or JSPs. It works everywhere.

Model Also Includes Service & DAO

In MVC, the "Model" is not just the data classes. It includes everything that deals with data and business logic:

Model Classes
User, Booking, Parcel — plain Java objects holding data (POJOs)
DAO Layer
UserDAO, BookingDAO — talk to the database (CRUD operations)
Service Layer
BookingService, TrackingService — business rules, validation, orchestration
Section 05

View — JSP Pages

The View layer is JSP pages that display data. A JSP should only contain HTML and EL expressions (${}). No database code. No business logic. Display only.

WEB-INF/booking-result.jsp — Clean View 
<%@ taglib prefix="c" uri="jakarta.tags.core" %>
<html>
<head><title>Booking Confirmation</title></head>
<body>

    <h1>Booking Confirmed!</h1>

    <!-- Display only. No database code. No calculations. -->
    <table>
        <tr><td>Booking ID:</td><td>NB-${booking.bookingId}</td></tr>
        <tr><td>Sender:</td><td>${booking.senderName}</td></tr>
        <tr><td>Receiver:</td><td>${booking.receiverName}</td></tr>
        <tr><td>Weight:</td><td>${booking.weight} kg</td></tr>
        <tr><td>Type:</td><td>${booking.parcelType}</td></tr>
        <tr><td>Cost:</td><td>Rs. ${booking.cost}</td></tr>
        <tr><td>Status:</td><td>${booking.status}</td></tr>
    </table>

    <!-- Conditional display using JSTL -->
    <c:if test="${not empty successMessage}">
        <div class="success">${successMessage}</div>
    </c:if>

    <a href="book">Book Another Parcel</a>

</body>
</html>
The Golden Rule of Views

A JSP page should be able to answer: "Where did this data come from?" with "I don't know, it was in the request attributes." If a JSP knows about the database, about JDBC, about SQL — it's doing too much. Move that code to the Controller or Model.

Section 06

Controller — Servlets

The Controller is the Servlet. It receives the request, decides what to do, calls the right Service/DAO, and forwards to the right JSP. It does not contain business logic or HTML.

What a Controller Should Do

  1. Read input. Extract parameters from the request: request.getParameter("email")
  2. Validate input. Check for null, empty, wrong format. Return error to View if invalid.
  3. Call the Model. Pass data to the Service layer: bookingService.createBooking(data)
  4. Prepare the View. Put results into request attributes: request.setAttribute("booking", booking)
  5. Choose the View. Forward to the right JSP or redirect to another URL.

What a Controller Should NOT Do

  • Write SQL queries or call DriverManager.getConnection()
  • Calculate business rules (pricing, discounts, status transitions)
  • Generate HTML using response.getWriter().println("<html>...")
  • Hold state between requests (Servlets are shared across threads)
Section 07

Full Nimbus MVC Example — Booking Flow

Let's trace a complete booking request through all three MVC layers. This is how a real user booking a parcel flows through your code.

Controller: BookingController Servlet

com/nimbus/servlet/BookingServlet.java 
package com.nimbus.servlet;

import com.nimbus.model.Booking;
import com.nimbus.model.User;
import com.nimbus.service.BookingService;
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.*;
import java.io.IOException;

@WebServlet("/book")
public class BookingServlet extends HttpServlet {

    private BookingService bookingService;

    @Override
    public void init() throws ServletException {
        bookingService = new BookingService();  // Model layer
    }

    // GET /book — show the booking form (View)
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        HttpSession session = request.getSession(false);
        if (session == null || session.getAttribute("loggedInUser") == null) {
            response.sendRedirect("login");
            return;
        }

        request.getRequestDispatcher("/WEB-INF/booking.jsp")
               .forward(request, response);
    }

    // POST /book — process booking (Controller logic)
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        HttpSession session = request.getSession(false);
        if (session == null || session.getAttribute("loggedInUser") == null) {
            response.sendRedirect("login");
            return;
        }

        User loggedInUser = (User) session.getAttribute("loggedInUser");

        try {
            // 1. READ INPUT (Controller job)
            String senderName = request.getParameter("senderName");
            String receiverName = request.getParameter("receiverName");
            String receiverAddress = request.getParameter("receiverAddress");
            double weight = Double.parseDouble(request.getParameter("weight"));
            String parcelType = request.getParameter("parcelType");

            // 2. CALL THE MODEL (Service does business logic)
            Booking booking = bookingService.createBooking(
                loggedInUser.getUserId(),
                senderName, receiverName, receiverAddress,
                weight, parcelType
            );

            // 3. CHOOSE THE VIEW (redirect for PRG)
            session.setAttribute("successMessage",
                "Parcel booked! ID: NB-" + booking.getBookingId());
            response.sendRedirect("invoice?bookingId=" + booking.getBookingId());

        } catch (NumberFormatException e) {
            request.setAttribute("error", "Invalid weight value");
            request.getRequestDispatcher("/WEB-INF/booking.jsp")
                   .forward(request, response);
        } catch (Exception e) {
            request.setAttribute("error", "Booking failed: " + e.getMessage());
            request.getRequestDispatcher("/WEB-INF/booking.jsp")
                   .forward(request, response);
        }
    }
}
Controller responsibilities only
1
bookingService = new BookingService()
The Controller creates the Service once in init(). It delegates ALL business logic to the Service. The Servlet never calculates cost or validates business rules.
2
request.getParameter("senderName")
Controller's first job: read input. It extracts raw data from the HTTP request and passes it to the Model layer.
3
bookingService.createBooking(...)
Controller's second job: call the Model. The Service handles cost calculation, validation, and database insertion. The Controller doesn't know or care how.
4
response.sendRedirect("invoice?bookingId=...")
Controller's third job: choose the View. For successful POST, it uses PRG (redirect). For errors, it forwards back to the form with an error message.

Model: BookingService

com/nimbus/service/BookingService.java 
package com.nimbus.service;

import com.nimbus.dao.BookingDAO;
import com.nimbus.model.Booking;

public class BookingService {

    private BookingDAO bookingDAO = new BookingDAO();

    public Booking createBooking(int senderId, String senderName,
                                  String receiverName, String receiverAddress,
                                  double weight, String parcelType) {

        // BUSINESS LOGIC lives here, not in the Servlet
        Booking booking = new Booking();
        booking.setSenderId(senderId);
        booking.setSenderName(senderName);
        booking.setReceiverName(receiverName);
        booking.setReceiverAddress(receiverAddress);
        booking.setWeight(weight);
        booking.setParcelType(parcelType);
        booking.setStatus("BOOKED");

        // Cost calculation — the business rule
        double cost = weight * 50;  // Rs. 50 per kg
        if ("EXPRESS".equals(parcelType)) {
            cost *= 1.5;  // 50% extra for express
        } else if ("FRAGILE".equals(parcelType)) {
            cost *= 1.3;  // 30% extra for fragile handling
        }
        booking.setCost(cost);

        // Save to database via DAO
        int bookingId = bookingDAO.insertBooking(booking);
        booking.setBookingId(bookingId);

        return booking;
    }
}

Model: BookingDAO

com/nimbus/dao/BookingDAO.java 
package com.nimbus.dao;

import com.nimbus.model.Booking;
import java.sql.*;

public class BookingDAO {

    public int insertBooking(Booking booking) {
        String sql = "INSERT INTO bookings (sender_id, sender_name, " +
                     "receiver_name, receiver_address, weight, " +
                     "parcel_type, cost, status) " +
                     "VALUES (?, ?, ?, ?, ?, ?, ?, ?)";

        try (Connection conn = DBUtil.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql,
                     Statement.RETURN_GENERATED_KEYS)) {

            ps.setInt(1, booking.getSenderId());
            ps.setString(2, booking.getSenderName());
            ps.setString(3, booking.getReceiverName());
            ps.setString(4, booking.getReceiverAddress());
            ps.setDouble(5, booking.getWeight());
            ps.setString(6, booking.getParcelType());
            ps.setDouble(7, booking.getCost());
            ps.setString(8, booking.getStatus());

            ps.executeUpdate();

            ResultSet rs = ps.getGeneratedKeys();
            if (rs.next()) {
                return rs.getInt(1);  // Return generated booking ID
            }
        } catch (SQLException e) {
            throw new RuntimeException("Failed to insert booking", e);
        }
        return -1;
    }

    public Booking findById(int bookingId) {
        String sql = "SELECT * FROM bookings WHERE booking_id = ?";

        try (Connection conn = DBUtil.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setInt(1, bookingId);
            ResultSet rs = ps.executeQuery();

            if (rs.next()) {
                Booking booking = new Booking();
                booking.setBookingId(rs.getInt("booking_id"));
                booking.setSenderName(rs.getString("sender_name"));
                booking.setReceiverName(rs.getString("receiver_name"));
                booking.setWeight(rs.getDouble("weight"));
                booking.setParcelType(rs.getString("parcel_type"));
                booking.setCost(rs.getDouble("cost"));
                booking.setStatus(rs.getString("status"));
                return booking;
            }
        } catch (SQLException e) {
            throw new RuntimeException("Failed to find booking", e);
        }
        return null;
    }
}
In Nimbus — The Full Picture

Notice the clean separation:

  • BookingServlet (Controller): reads form data, calls service, picks JSP. Zero SQL, zero HTML.
  • BookingService (Model - logic): calculates cost, validates rules. Zero HTTP, zero SQL.
  • BookingDAO (Model - data): executes SQL, maps ResultSet to objects. Zero HTTP, zero business rules.
  • booking-result.jsp (View): displays ${booking.cost}. Zero Java logic, zero SQL.

If the pricing formula changes from Rs. 50/kg to Rs. 65/kg, you change one line in BookingService. Nothing else changes.

Section 08

Common MVC Mistakes

These are the mistakes every beginner makes. Knowing them upfront saves you hours of debugging and refactoring later.

Mistake 1: Database Code in JSP

Wrong — SQL in JSP
<%
Connection conn = DriverManager
  .getConnection("jdbc:mysql://...");
ResultSet rs = conn.createStatement()
  .executeQuery("SELECT * FROM users");
while (rs.next()) {
%>
<p><%= rs.getString("name") %></p>
<% } %>

Database in the View layer. Impossible to test, impossible to reuse, impossible to maintain.

Right — JSP reads attributes
<c:forEach var="user"
           items="${users}">
  <p>${user.name}</p>
</c:forEach>

JSP only displays. The Servlet loaded the users from the DAO and put them in request attributes.

Mistake 2: HTML in Servlet

Wrong — HTML in Servlet
PrintWriter out = response.getWriter();
out.println("<html><body>");
out.println("<h1>Welcome " + name + "</h1>");
out.println("<table>");
for (Booking b : bookings) {
  out.println("<tr><td>" + b.getId()
    + "</td></tr>");
}
out.println("</table></body></html>");

HTML buried in Java strings. No syntax highlighting, no designer can touch it, no CSS preview.

Right — Forward to JSP
request.setAttribute("name", name);
request.setAttribute("bookings", bookings);
request.getRequestDispatcher(
  "/WEB-INF/dashboard.jsp")
  .forward(request, response);

Servlet passes data. JSP handles all HTML/CSS. A designer can edit the JSP without touching Java.

Mistake 3: Business Logic in Controller

Wrong — Cost calc in Servlet
// In BookingServlet.doPost()
double cost = weight * 50;
if ("EXPRESS".equals(type)) cost *= 1.5;
if (weight > 10) cost -= cost * 0.1;
booking.setCost(cost);
bookingDAO.insert(booking);

Pricing formula embedded in the Controller. If a mobile API also creates bookings, you'd have to duplicate this code.

Right — Delegate to Service
// In BookingServlet.doPost()
Booking booking = bookingService
  .createBooking(
    senderId, senderName,
    receiverName, weight, type);

// BookingService handles cost,
// validation, and DB insertion.
// One place, used by all entry points.

Controller just passes data to Service. Business logic lives in one place. Web and API both use the same Service.

Section 09

Without MVC vs With MVC

A side-by-side look at how the same feature is built with and without MVC architecture.

Feature: Show Booking History

Without MVC
  • One file: history.jsp
  • JSP connects to database
  • JSP runs SQL query
  • JSP loops through ResultSet
  • JSP generates HTML rows
  • Everything in one place

Problems: Can't test without Tomcat. Can't reuse the query. Frontend dev can't edit without knowing SQL. Change database? Edit every JSP.

With MVC
  • HistoryServlet (Controller) reads params, calls DAO
  • BookingDAO (Model) runs query, returns List<Booking>
  • history.jsp (View) loops ${bookings} with JSTL
  • Three files, three responsibilities

Benefits: Unit test the DAO alone. Frontend edits JSP only. Change DB? Edit DAO only. Add a REST API? Reuse the same DAO.

File Count Comparison

FeatureWithout MVCWith MVC
Login 1 file (login.jsp with scriptlets) 3 files: LoginServlet + UserDAO + login.jsp
Booking 1 file (booking.jsp) 4 files: BookingServlet + BookingService + BookingDAO + booking.jsp
Tracking 1 file (tracking.jsp) 4 files: TrackingServlet + TrackingService + BookingDAO + tracking.jsp
More Files = Better?

Yes — for anything beyond a toy project. With 1 file per feature, you save time in week 1. By month 3, you're drowning. With MVC, each file is small, focused, testable, and you always know exactly where to fix a bug. Every professional Java application uses MVC or a variant of it.
