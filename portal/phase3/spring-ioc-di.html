Section 01

Constructor Injection

The recommended way. Dependencies are passed via the constructor, making the object immutable and always in a valid state.

Why Constructor Injection Wins

  • Immutable — Fields are final. Once set, they can't change.
  • Required dependencies — The object can't be created without them. No null surprises.
  • Testable — Pass mocks directly in the constructor. No reflection hacks needed.
  • No circular dependency hiding — Fails fast at startup if there's a circular dependency.
UserService.java 
@Service
public class UserService {

    private final UserDAO userDAO;        // final = immutable
    private final EmailService emailService;

    @Autowired  // optional if single constructor (Spring 4.3+)
    public UserService(UserDAO userDAO, EmailService emailService) {
        this.userDAO = userDAO;
        this.emailService = emailService;
    }

    public User registerUser(User user) {
        userDAO.save(user);
        emailService.sendWelcome(user.getEmail());
        return user;
    }
}
In Nimbus

UserService depends on UserDAO (database access) and EmailService (notifications). Both are injected via constructor. The service can't exist without them — exactly what we want.

Single Constructor Rule

If your class has only one constructor, Spring automatically uses it for injection. No @Autowired needed. With multiple constructors, mark the one Spring should use with @Autowired.

Section 02

Setter Injection

For optional dependencies that have sensible defaults.

BookingService.java 
@Service
public class BookingService {

    private final BookingDAO bookingDAO;  // required
    private NotificationService notifier; // optional

    @Autowired
    public BookingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    @Autowired(required = false)  // won't crash if no bean found
    public void setNotifier(NotificationService notifier) {
        this.notifier = notifier;
    }

    public Booking createBooking(Booking booking) {
        bookingDAO.save(booking);
        if (notifier != null) {
            notifier.send("Booking created: " + booking.getTrackingNumber());
        }
        return booking;
    }
}

When to Use Setter vs Constructor

ScenarioUseWhy
Required dependencyConstructorObject is invalid without it
Optional dependencySetterHas a fallback if not provided
Immutable serviceConstructorFields can be final
Reconfigurable at runtimeSetterCan change after creation
Section 03

Field Injection (Why It's Discouraged)

Looks clean but hides dependencies and makes testing harder.

Field Injection (Avoid)
@Service
public class TrackingService {
    @Autowired
    private BookingDAO dao;

    @Autowired
    private CacheService cache;
}
Constructor Injection (Prefer)
@Service
public class TrackingService {
    private final BookingDAO dao;
    private final CacheService cache;

    public TrackingService(BookingDAO dao,
                           CacheService cache) {
        this.dao = dao;
        this.cache = cache;
    }
}

Problems with Field Injection

  • Hidden dependencies — You can't tell what a class needs just by looking at its constructor.
  • Not testable with new — new TrackingService() has null fields. You need Spring or reflection to inject.
  • Can't be final — Fields must be mutable for reflection to set them.
  • Circular dependencies hidden — Field injection lets circular deps slip through. Constructor injection catches them at startup.
IDE Warning

IntelliJ and Eclipse both show a warning on @Autowired fields: "Field injection is not recommended." They suggest constructor injection. Listen to your IDE.

Section 04

Circular Dependencies

A depends on B, and B depends on A. Spring can't create either. How to fix it.

The Problem

Circular Dependency 
// A needs B to be created
@Service
public class UserService {
    private final BookingService bookingService;
    public UserService(BookingService bookingService) {
        this.bookingService = bookingService;
    }
}

// B needs A to be created — DEADLOCK
@Service
public class BookingService {
    private final UserService userService;
    public BookingService(UserService userService) {
        this.userService = userService;
    }
}
// Result: BeanCurrentlyInCreationException

Fixes

Fix 1: Redesign (Best)

Extract the shared logic into a third service. If A and B both need each other, they're probably doing too much.

Fix 2: @Lazy

Lazy Break 
@Service
public class UserService {
    private final BookingService bookingService;

    public UserService(@Lazy BookingService bookingService) {
        this.bookingService = bookingService;  // proxy injected, not real bean
    }
}

Fix 3: Setter Injection on One Side

Use constructor injection for A and setter injection for B. Breaks the cycle because B can be partially created first.

Section 05

Bean Lifecycle

From birth to death — what happens to a Spring bean.

Instantiate new UserService()
→
Populate Inject dependencies
→
Init @PostConstruct
→
Ready Bean in use
→
Destroy @PreDestroy
Spring manages the entire lifecycle. You hook into init and destroy.
Lifecycle Hooks 
@Service
public class TrackingService {

    private final BookingDAO bookingDAO;
    private Map<String, Booking> cache;

    public TrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    @PostConstruct  // runs AFTER constructor + injection
    public void init() {
        // Load frequently accessed bookings into cache
        cache = new HashMap<>();
        System.out.println("TrackingService initialized, cache ready");
    }

    @PreDestroy  // runs BEFORE bean is destroyed
    public void cleanup() {
        cache.clear();
        System.out.println("TrackingService cache cleared");
    }
}
Section 06

@Lazy Initialization

By default, Spring creates all singleton beans at startup. @Lazy delays creation until first use.

Lazy Bean 
@Service
@Lazy  // not created until someone asks for it
public class ReportService {

    public ReportService() {
        System.out.println("ReportService created");  // only prints on first use
    }

    public byte[] generatePDF(int bookingId) {
        // heavy PDF generation
        return new byte[0];
    }
}

When to Use

  • Expensive beans — If a bean takes 5 seconds to initialize and is rarely used, make it lazy.
  • Faster startup — Fewer beans at startup = faster app launch.
  • Breaking circular deps — @Lazy on a constructor parameter injects a proxy first.
Trade-off

Lazy beans delay errors to runtime. If a lazy bean has a config problem, you won't know until someone actually uses it. Eager (default) beans fail fast at startup — which is usually better for catching bugs early.

Section 07

@Profile for Dev/Prod

Different beans for different environments. Dev uses an in-memory DB, prod uses MySQL.

Profile-Specific Beans 
// Only active when profile = "dev"
@Repository
@Profile("dev")
public class InMemoryUserDAO implements UserDAO {
    private final List<User> users = new ArrayList<>();

    @Override
    public User findById(int id) {
        return users.stream()
            .filter(u -> u.getId() == id)
            .findFirst().orElse(null);
    }
}

// Only active when profile = "prod"
@Repository
@Profile("prod")
public class MySQLUserDAO implements UserDAO {
    private final JdbcTemplate jdbc;

    public MySQLUserDAO(JdbcTemplate jdbc) {
        this.jdbc = jdbc;
    }

    @Override
    public User findById(int id) {
        return jdbc.queryForObject("SELECT * FROM users WHERE id = ?",
            new UserRowMapper(), id);
    }
}

Activating a Profile

application.properties 
# In application.properties
spring.profiles.active=dev

# Or via command line
java -jar nimbus.jar --spring.profiles.active=prod

# Or via environment variable
export SPRING_PROFILES_ACTIVE=prod
In Nimbus

Dev profile uses InMemoryUserDAO (no MySQL needed) for quick testing. Prod profile uses MySQLUserDAO with the real database. Same UserService code works in both — it just gets a different UserDAO implementation injected.

Section 08

@Value and @PropertySource

Inject configuration values from properties files into your beans.

application.properties 
# Database config
db.url=jdbc:mysql://localhost:3306/nimbus_db
db.user=root
db.password=secret

# App config
app.name=Nimbus Parcel Management
app.max-parcels-per-user=50
app.tracking-prefix=NMB
AppConfig.java 
@Configuration
@PropertySource("classpath:application.properties")
public class AppConfig {

    @Value("${db.url}")
    private String dbUrl;

    @Value("${db.user}")
    private String dbUser;

    @Value("${app.max-parcels-per-user}")
    private int maxParcels;

    @Value("${app.tracking-prefix:TRK}")  // default value after colon
    private String trackingPrefix;

    @Bean
    public DataSource dataSource() {
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setUrl(dbUrl);
        ds.setUsername(dbUser);
        return ds;
    }
}
Default Values

@Value("${some.key:defaultValue}") — the part after the colon is the fallback if the property isn't found. Without a default, Spring throws an error if the property is missing.

Section 09

Full Nimbus with Proper DI

All Nimbus services with dependencies properly wired using constructor injection.

In Nimbus

The complete dependency tree: BookingController → BookingService → BookingDAO + TrackingService. TrackingService → BookingDAO. All wired via constructor injection.

com/nimbus/dao/UserDAO.java 
public interface UserDAO {
    User findById(int id);
    User findByEmail(String email);
    void save(User user);
    void update(User user);
    List<User> findAll();
}
com/nimbus/dao/UserDAOImpl.java 
@Repository
public class UserDAOImpl implements UserDAO {

    private final JdbcTemplate jdbc;

    public UserDAOImpl(JdbcTemplate jdbc) {
        this.jdbc = jdbc;
    }

    @Override
    public User findById(int id) {
        return jdbc.queryForObject(
            "SELECT * FROM users WHERE id = ?",
            new UserRowMapper(), id
        );
    }

    @Override
    public void save(User user) {
        jdbc.update("INSERT INTO users (name, email, role) VALUES (?, ?, ?)",
            user.getName(), user.getEmail(), user.getRole());
    }

    // ... other methods
}
com/nimbus/service/UserService.java 
@Service
public class UserService {

    private final UserDAO userDAO;
    private final BookingDAO bookingDAO;

    @Value("${app.max-parcels-per-user:50}")
    private int maxParcelsPerUser;

    public UserService(UserDAO userDAO, BookingDAO bookingDAO) {
        this.userDAO = userDAO;
        this.bookingDAO = bookingDAO;
    }

    public User getUser(int id) {
        return userDAO.findById(id);
    }

    public boolean canBookMore(int userId) {
        int current = bookingDAO.countByUserId(userId);
        return current < maxParcelsPerUser;
    }
}
com/nimbus/service/TrackingService.java 
@Service
public class TrackingService {

    private final BookingDAO bookingDAO;

    @Value("${app.tracking-prefix:NMB}")
    private String prefix;

    public TrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    public String generateTrackingNumber() {
        String date = LocalDate.now().format(DateTimeFormatter.BASIC_ISO_DATE);
        int seq = bookingDAO.getNextSequence();
        return String.format("%s-%s-%03d", prefix, date, seq);
    }

    public Booking track(String trackingNumber) {
        return bookingDAO.findByTrackingNumber(trackingNumber);
    }
}
Section 10

Exercises

Practice DI patterns with these challenges.

Exercise 1 — Constructor vs Setter

Create a PaymentService with a required PaymentGateway (constructor injection) and an optional FraudDetector (setter injection). Write a processPayment() method that checks fraud only if the detector is available.

Exercise 2 — Profile Switching

Create two implementations of BookingDAO: one @Profile("dev") using an in-memory ArrayList, one @Profile("prod") using JDBC. Inject into BookingService and verify switching profiles changes the DAO used.

Exercise 3 — @Value Configuration

Create application.properties with app.name, app.version, app.admin-email. Inject all three into an AppInfoService bean. Add a method getAbout() that returns "Nimbus v1.0 | admin@nimbus.com".

Exercise 4 — Lifecycle Hooks

Add @PostConstruct to TrackingService that pre-loads the last 100 bookings into a HashMap cache. Add @PreDestroy that clears the cache and logs the number of entries cleared.

Exercise 5 — Fix the Circular Dependency

Given: OrderService needs InventoryService, and InventoryService needs OrderService. Both use constructor injection. The app crashes on startup. Fix it using three different approaches: redesign, @Lazy, and setter injection.
