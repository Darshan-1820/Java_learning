Section 01

What are Annotations?

Annotations are special instructions you write above classes, methods, or fields using the @ symbol. They tell Spring what to do with your code — without you writing the boilerplate yourself.

Analogy

Think of annotations as sticky notes you put on your classes. You walk into a kitchen (your codebase) and stick a note on a jar that says "Sugar." Now anyone (Spring) who walks in knows what that jar is and how to use it — without opening it.

@Service on a class = a sticky note saying "This class handles business logic. Spring, manage it for me."

The @ Symbol

Every annotation starts with @ followed by the annotation name. It goes above the class, method, or field it describes.

Basic Annotation Syntax 
@Service                          // annotation on a class
public class UserService {

    @Autowired                    // annotation on a field
    private UserDAO userDAO;

    @PostConstruct                // annotation on a method
    public void init() {
        System.out.println("UserService is ready!");
    }
}

How Java Processes Annotations (Brief)

You don't need to master this, but it helps to know there are two meta-annotations that control how annotations work:

  • @Retention — tells Java how long to keep the annotation. RUNTIME means Spring can read it while the app is running.
  • @Target — tells Java where the annotation can be placed. TYPE = on classes, METHOD = on methods, FIELD = on fields.
How @Service is Defined Internally (You Won't Write This) 
@Retention(RetentionPolicy.RUNTIME)   // available at runtime
@Target(ElementType.TYPE)              // can only go on classes
@Component                             // @Service IS a @Component
public @interface Service {
    String value() default "";
}
Key Takeaway

You use annotations. Spring reads them at runtime and acts on them. That's the entire mental model. The rest of this page is learning which annotation does what.

Section 02

Stereotype Annotations (The Big 4)

These four annotations tell Spring: "Hey, this class is a bean — create it, manage it, and make it available for dependency injection." Each one marks a specific role in your application.

@ComponentGeneric bean
▼
@ServiceBusiness logic
+
@RepositoryData access
+
@ControllerWeb requests
All three are specialized versions of @Component

@Component — "Hey Spring, Manage This Class"

@Component is the most generic stereotype annotation. It tells Spring: "Create an instance of this class and keep it in the container (ApplicationContext) so other classes can use it."

When to use: On helper/utility classes that don't fit neatly into service, repository, or controller categories.

EmailValidator.java 
package com.nimbus.util;

import org.springframework.stereotype.Component;

@Component
public class EmailValidator {

    public boolean isValid(String email) {
        return email != null && email.contains("@") && email.contains(".");
    }
}
In Nimbus

@Component could be used on a utility class like EmailValidator or a TrackingIdGenerator. These are helpers that don't hold business logic and don't talk to the database — they're just reusable tools.

What happens if you DON'T use it? Spring will have no idea your class exists. It won't create an instance, it won't manage it, and any class that tries to @Autowired it will get a "No qualifying bean" error.

@Service — "This Class Handles Business Logic"

@Service is a specialized @Component. Functionally it does the exact same thing — registers the class as a bean. But it communicates intent: this class contains business rules, calculations, validations, and orchestration logic.

When to use: On every class that contains business logic. Services typically call DAOs/Repositories to get data, process it, and return results.

UserService.java — Full Nimbus Example 
package com.nimbus.service;

import com.nimbus.dao.UserDAO;
import com.nimbus.model.User;
import com.nimbus.exception.DuplicateEmailException;
import org.springframework.stereotype.Service;
import org.springframework.beans.factory.annotation.Autowired;

@Service
public class UserService {

    private final UserDAO userDAO;

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }

    public User registerUser(User user) throws DuplicateEmailException {
        // Business rule: check if email already exists
        if (userDAO.findByEmail(user.getEmail()) != null) {
            throw new DuplicateEmailException(
                "Email " + user.getEmail() + " is already registered"
            );
        }

        // Business rule: set default role if not specified
        if (user.getRole() == null || user.getRole().isEmpty()) {
            user.setRole("CUSTOMER");
        }

        return userDAO.save(user);
    }

    public User getUserById(int userId) {
        return userDAO.findById(userId);
    }

    public User login(String email, String password) {
        User user = userDAO.findByEmail(email);
        if (user != null && user.getPassword().equals(password)) {
            return user;
        }
        return null; // login failed
    }
}
Line-by-line explanation
1
@Service
Tells Spring: "This is a business logic class. Create it as a bean and manage its lifecycle."
2
private final UserDAO userDAO;
Declares a dependency on UserDAO. The final keyword means it MUST be set in the constructor and can never change — this is a best practice.
3
@Autowired public UserService(UserDAO userDAO)
Constructor injection — Spring looks in the container for a bean of type UserDAO and passes it here when creating UserService.
4
if (userDAO.findByEmail(...) != null)
Business rule: before registering, check if the email is already in use. The service DECIDES, the DAO just QUERIES.
5
throw new DuplicateEmailException(...)
If the email exists, throw a custom exception. The service enforces rules — it doesn't just pass data through.
6
user.setRole("CUSTOMER")
Another business rule: default role assignment. This logic belongs in the service, NOT in the DAO or controller.
In Nimbus

@Service goes on: UserService (registration, login, profile), TrackingService (tracking parcels), CustomerTrackingService (customer-specific tracking), OfficerTrackingService (officer-specific tracking), BaseHistoryService (shared history logic), CustomerHistoryService (customer history), and BookingThread (async booking processing).

What happens if you DON'T use it? Same as @Component — Spring won't create the bean. But also: anyone reading your code won't know this class is supposed to hold business logic. You lose readability and intent.

@Repository — "This Class Talks to the Database"

@Repository is another specialized @Component, but it has a superpower: it automatically translates database-specific exceptions (like SQLException) into Spring's unified DataAccessException hierarchy. This means your service layer doesn't need to know if you're using MySQL, PostgreSQL, or Oracle — the exception type is always the same.

When to use: On every class that directly interacts with the database — your DAO (Data Access Object) classes.

UserDAO.java — Full Nimbus Example 
package com.nimbus.dao;

import com.nimbus.model.User;
import org.springframework.stereotype.Repository;
import org.springframework.beans.factory.annotation.Autowired;
import javax.sql.DataSource;
import java.sql.*;

@Repository
public class UserDAO {

    private final DataSource dataSource;

    @Autowired
    public UserDAO(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    public User findById(int userId) {
        String sql = "SELECT * FROM tbl_UserProfile WHERE user_id = ?";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setInt(1, userId);
            ResultSet rs = ps.executeQuery();

            if (rs.next()) {
                User user = new User();
                user.setUserId(rs.getInt("user_id"));
                user.setName(rs.getString("name"));
                user.setEmail(rs.getString("email"));
                user.setRole(rs.getString("role"));
                return user;
            }
        } catch (SQLException e) {
            // @Repository converts this to DataAccessException automatically
            throw new RuntimeException("Error finding user by ID", e);
        }
        return null;
    }

    public User findByEmail(String email) {
        String sql = "SELECT * FROM tbl_UserProfile WHERE email = ?";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setString(1, email);
            ResultSet rs = ps.executeQuery();

            if (rs.next()) {
                User user = new User();
                user.setUserId(rs.getInt("user_id"));
                user.setName(rs.getString("name"));
                user.setEmail(rs.getString("email"));
                user.setRole(rs.getString("role"));
                return user;
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error finding user by email", e);
        }
        return null;
    }

    public User save(User user) {
        String sql = "INSERT INTO tbl_UserProfile (name, email, password, role) VALUES (?, ?, ?, ?)";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {

            ps.setString(1, user.getName());
            ps.setString(2, user.getEmail());
            ps.setString(3, user.getPassword());
            ps.setString(4, user.getRole());
            ps.executeUpdate();

            ResultSet keys = ps.getGeneratedKeys();
            if (keys.next()) {
                user.setUserId(keys.getInt(1));
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error saving user", e);
        }
        return user;
    }
}
Line-by-line explanation
1
@Repository
Marks this as a data access class. Spring creates the bean AND enables automatic exception translation from SQLException to DataAccessException.
2
private final DataSource dataSource;
Instead of using DBConnection utility directly, we inject a DataSource bean. Spring manages the connection pool for us.
3
dataSource.getConnection()
Gets a database connection from the pool. The try-with-resources block ensures it's returned to the pool when done.
4
PreparedStatement ps = conn.prepareStatement(sql)
Uses parameterized queries (the ? placeholders) — prevents SQL injection attacks. NEVER concatenate user input into SQL strings.
5
Statement.RETURN_GENERATED_KEYS
Tells MySQL to return the auto-generated user_id after INSERT so we can set it on the User object.
In Nimbus

@Repository goes on: UserDAO (queries tbl_UserProfile) and BookingDAO (queries the Booking table). These are the ONLY classes that should contain SQL statements.

Why @Repository and Not Just @Component?

If you use plain @Component on a DAO, the bean still gets created — but you lose the automatic exception translation. A MySQL-specific SQLException would leak into your service layer, coupling your business logic to your database vendor. With @Repository, Spring wraps it in DataAccessException — a standard Spring exception your service can catch without knowing which database is underneath.

@Controller — "This Handles Web Requests"

@Controller is the fourth specialized @Component. It marks a class as an MVC controller — one that handles incoming HTTP requests and returns views (JSP pages) or data (JSON).

When to use: On classes that receive web requests from users. This is the entry point of your web application.

UserController.java — Preview (Covered in Spring MVC) 
package com.nimbus.controller;

import com.nimbus.service.UserService;
import com.nimbus.model.User;
import org.springframework.stereotype.Controller;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.*;
import org.springframework.ui.Model;

@Controller
@RequestMapping("/users")
public class UserController {

    private final UserService userService;

    @Autowired
    public UserController(UserService userService) {
        this.userService = userService;
    }

    @GetMapping("/register")
    public String showRegisterForm() {
        return "register";  // returns register.jsp
    }

    @PostMapping("/register")
    public String registerUser(@ModelAttribute User user, Model model) {
        try {
            userService.registerUser(user);
            return "redirect:/users/login";
        } catch (Exception e) {
            model.addAttribute("error", e.getMessage());
            return "register";
        }
    }
}
In Nimbus

@Controller will be used when we build the web layer. Nimbus will have controllers for: UserController (register, login, profile), BookingController (create/view bookings), and TrackingController (track parcels). These are covered in depth in the Spring MVC topic.

What happens if you DON'T use it? Spring MVC won't recognize the class as a handler. No URL will be mapped to your methods, and your web endpoints will return 404.

Comparison: When to Use Which?

Annotation Layer Purpose Nimbus Classes Special Power
@Component Any Generic Spring bean EmailValidator, TrackingIdGenerator None (basic bean registration)
@Service Service Business logic UserService, TrackingService, BookingThread Communicates intent (business layer)
@Repository DAO / Data Database access UserDAO, BookingDAO Auto exception translation (SQLException → DataAccessException)
@Controller Web / Presentation Handle HTTP requests UserController, BookingController Maps URLs to methods, returns views
Simple Rule

Touches the database? @Repository. Contains business rules? @Service. Handles web requests? @Controller. None of the above? @Component.

Section 03

@Autowired — Dependency Injection

This is the annotation that makes Spring's dependency injection actually work. It tells Spring: "I need an instance of this type — find the matching bean in the container and inject it here."

Three Places to Use @Autowired

1. Constructor Injection (BEST Practice)

Put @Autowired on the constructor. Spring will pass the required beans as constructor arguments when creating the object.

Constructor Injection — Recommended 
@Service
public class UserService {

    private final UserDAO userDAO;       // marked final — can't change after construction

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;          // Spring passes the UserDAO bean here
    }
}

Why it's the best:

  • Dependencies are final — they can't be changed after the object is created (immutable)
  • You can't create the object without its dependencies — catches missing beans early
  • Easy to test — just pass mocks in the constructor
  • If there's only ONE constructor, you can even skip @Autowired — Spring will auto-detect it

2. Setter Injection

Put @Autowired on a setter method. Spring calls the setter after creating the object.

Setter Injection 
@Service
public class UserService {

    private UserDAO userDAO;            // NOT final — can be changed

    @Autowired
    public void setUserDAO(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
}

When to use: When a dependency is optional or when you might want to swap it out later. Rare in practice.

3. Field Injection (Easiest but NOT Recommended)

Put @Autowired directly on the field. Spring uses reflection to set it — no constructor or setter needed.

Field Injection — Avoid This 
@Service
public class UserService {

    @Autowired
    private UserDAO userDAO;     // Spring injects via reflection — no constructor needed
}
Why Field Injection Is Not Recommended
  • Can't make the field final — so it could theoretically be changed
  • Hard to test — you can't pass in a mock without using reflection or a Spring test context
  • Hides dependencies — you can't tell what the class needs just by looking at its constructor
  • The class can be created without its dependencies (they'll be null until Spring injects them)

It works, and you'll see it in tutorials, but professional Spring projects almost always use constructor injection.

Constructor Injection (Do This)
  • Dependencies are final
  • Required at creation time
  • Easy to unit test
  • Explicit — you see all deps in the constructor
Field Injection (Avoid This)
  • Dependencies can change
  • Object created without deps (they're null briefly)
  • Hard to unit test
  • Hidden — deps buried in field declarations

What Happens Behind the Scenes

When Spring starts your application and encounters @Autowired, here's the internal process:

Spring scans classesfinds @Service, @Repository, etc.
→
Creates beansinstantiates each annotated class
→
Resolves @Autowiredlooks for matching bean by type
→
Injects dependencypasses bean via constructor/setter/field
  1. Spring scans your packages and finds all classes annotated with @Component, @Service, @Repository, or @Controller.
  2. It creates instances (beans) of each one and stores them in the ApplicationContext (the container).
  3. For each @Autowired point, Spring looks in the container for a bean whose type matches the required parameter/field.
  4. If it finds exactly ONE match, it injects it. If it finds ZERO, it throws an error. If it finds MORE THAN ONE, it also throws an error (unless you use @Qualifier).

Common Error: "No Qualifying Bean"

Error Message 
NoSuchBeanDefinitionException: No qualifying bean of type
'com.nimbus.dao.UserDAO' available: expected at least 1 bean
which qualifies as autowire candidate.

This means: Spring is trying to inject UserDAO somewhere, but it can't find a bean of that type in the container.

Common fixes:

  • Did you forget @Repository on UserDAO? Add it.
  • Is UserDAO in a package that @ComponentScan doesn't reach? Check your scan base package.
  • Is the class spelled correctly? Java is case-sensitive.
  • Did the class fail to compile? Check for compilation errors in the DAO.

Full Before/After: UserService Without and With @Autowired

Without @Autowired (Manual Wiring)
public class UserService {
    private UserDAO userDAO;

    public UserService() {
        // Manually creating the dependency
        this.userDAO = new UserDAO();
    }
}

Problems: Tight coupling. Can't swap implementations. Can't test without real DB. UserService controls UserDAO's lifecycle.

With @Autowired (Spring DI)
@Service
public class UserService {
    private final UserDAO userDAO;

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
}

Benefits: Loose coupling. Spring injects the right bean. Easy to test with mocks. Spring controls the lifecycle.

Section 04

@Configuration & @Bean

Sometimes you can't put @Component on a class — maybe it's from a 3rd-party library, or maybe you need custom setup logic. That's when you use @Configuration and @Bean to create beans manually.

@Configuration — "This Class Defines Beans"

A class annotated with @Configuration is a configuration class. It's like a recipe book — it tells Spring how to create specific beans that can't be auto-detected via component scanning.

@Bean — "This Method Creates a Bean"

A method annotated with @Bean inside a @Configuration class tells Spring: "Call this method, take whatever it returns, and register it as a bean in the container."

AppConfig.java — Nimbus Configuration 
package com.nimbus.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.ComponentScan;
import org.apache.commons.dbcp2.BasicDataSource;
import javax.sql.DataSource;

@Configuration
@ComponentScan("com.nimbus")
public class AppConfig {

    @Bean
    public DataSource dataSource() {
        BasicDataSource ds = new BasicDataSource();
        ds.setDriverClassName("com.mysql.cj.jdbc.Driver");
        ds.setUrl("jdbc:mysql://localhost:3306/nimbus_db");
        ds.setUsername("root");
        ds.setPassword("password");
        ds.setInitialSize(5);            // start with 5 connections in the pool
        ds.setMaxTotal(10);              // max 10 connections
        return ds;
    }
}
Line-by-line explanation
1
@Configuration
Tells Spring: "This class contains bean definitions. Process it specially." Configuration classes are themselves beans, and Spring proxies them to ensure @Bean methods return singletons.
2
@ComponentScan("com.nimbus")
Tells Spring: "Scan the com.nimbus package and all its sub-packages for classes with @Component, @Service, @Repository, @Controller." Without this, Spring won't find your annotated classes.
3
@Bean public DataSource dataSource()
This method creates and returns a DataSource. Spring calls this method once, takes the returned object, and registers it as a bean named "dataSource" (the method name becomes the bean name).
4
BasicDataSource ds = new BasicDataSource()
BasicDataSource is from Apache Commons DBCP — a 3rd-party library. We can't put @Component on it because we didn't write it. That's WHY we need @Bean — to create it manually.
5
ds.setInitialSize(5); ds.setMaxTotal(10);
Connection pool settings. Instead of creating a new DB connection for every query (slow), the pool pre-creates 5 connections and reuses them. Max 10 connections at once.
In Nimbus

The AppConfig class replaces the DBConnection utility class. Instead of manually getting connections via DriverManager.getConnection(), we define a DataSource bean with connection pooling. Every @Repository class can then @Autowired the DataSource and get connections from the pool.

When to Use @Bean vs @Component?

Use @Component/@Service/@Repository: On classes YOU write. Just annotate the class directly.

Use @Bean: On 3rd-party classes you can't annotate (DataSource, RestTemplate, ObjectMapper, etc.) or when you need custom creation logic.

Section 05

@ComponentScan

Spring doesn't magically know where your annotated classes live. @ComponentScan tells it which packages to search.

How It Works

When Spring starts, it needs to find all classes with @Component, @Service, @Repository, and @Controller. @ComponentScan says: "Start looking in THIS package, and also check all its sub-packages."

Default Scan — Package of the Configuration Class 
package com.nimbus;

@Configuration
@ComponentScan          // scans com.nimbus and ALL sub-packages
public class AppConfig { }
Custom Scan — Specific Packages 
@Configuration
@ComponentScan(basePackages = {"com.nimbus.service", "com.nimbus.dao"})
public class AppConfig { }
In Nimbus

Our package structure is com.nimbus with sub-packages: com.nimbus.model, com.nimbus.dao, com.nimbus.service, com.nimbus.exception, com.nimbus.config, com.nimbus.controller. Using @ComponentScan("com.nimbus") on the config class scans ALL of them — finding UserDAO, BookingDAO, UserService, TrackingService, etc.

Nimbus Package Scanning Diagram

@ComponentScan("com.nimbus")
        |
        v
com.nimbus
  |-- config/
  |     AppConfig.java        (@Configuration -- already found)
  |-- model/
  |     User.java             (no annotation -- skipped, it's a plain POJO)
  |     Booking.java          (no annotation -- skipped)
  |-- dao/
  |     UserDAO.java          (@Repository -- FOUND, bean created)
  |     BookingDAO.java       (@Repository -- FOUND, bean created)
  |-- service/
  |     UserService.java      (@Service -- FOUND, bean created)
  |     TrackingService.java  (@Service -- FOUND, bean created)
  |     BookingThread.java    (@Service -- FOUND, bean created)
  |-- exception/
  |     DuplicateEmailException.java  (no annotation -- skipped)
  |     BookingNotFoundException.java (no annotation -- skipped)
  |-- controller/
        UserController.java   (@Controller -- FOUND, bean created)
Spring scans all sub-packages and registers every annotated class as a bean
Common Mistake

If your @Configuration class is in com.nimbus.config but you forget @ComponentScan, Spring will ONLY scan com.nimbus.config and its sub-packages. Your DAO and Service classes in other packages won't be found. Always set the base package to the root: @ComponentScan("com.nimbus").

Section 06

@Scope — Bean Lifecycle Control

By default, Spring creates one instance of each bean and shares it everywhere. @Scope lets you change that behavior.

The Four Scopes

Scope Instances When Use Case
singleton ONE instance total Default — always UserDAO, UserService — one instance shared by all requests
prototype NEW instance every time Every time someone asks for it Report generators, form objects that should be fresh each time
request ONE per HTTP request Web apps only Shopping cart data for the current request
session ONE per user session Web apps only User preferences, logged-in user info persisting across pages
Singleton (Default) — One Instance Shared Everywhere 
@Repository
// @Scope("singleton")   <-- this is the DEFAULT, you don't need to write it
public class UserDAO {
    // Only ONE UserDAO instance exists. All services share it.
}
Prototype — Fresh Instance Every Time 
@Component
@Scope("prototype")
public class BookingReportGenerator {

    private List<Booking> bookings = new ArrayList<>();

    public void addBooking(Booking b) {
        bookings.add(b);
    }

    public String generateReport() {
        // Each time someone requests this bean, they get a fresh list
        return "Report for " + bookings.size() + " bookings";
    }
}
In Nimbus

UserDAO and BookingDAO should be singleton (default) — one instance reused for all database operations. They don't hold state specific to any user.

If you needed a fresh report generator for each request (so data from one user's report doesn't leak into another), you'd use @Scope("prototype").

Request & Session Scopes (Web Apps Only) 
@Component
@Scope("request")
public class RequestLogger {
    // A new instance is created for EACH HTTP request
    // Dies when the request is complete
}

@Component
@Scope("session")
public class UserSessionData {
    // A new instance is created for EACH user session
    // Lives as long as the user's session (until logout/timeout)
    private String loggedInUsername;
    private String role;
}
Section 07

@Qualifier — Picking the Right Bean

When you have two or more beans of the same type, Spring doesn't know which one to inject. @Qualifier tells it exactly which one you want.

The Problem

In Nimbus, TrackingService has two implementations: CustomerTrackingService and OfficerTrackingService. If another class asks Spring to inject a TrackingService, Spring finds two candidates and panics.

The Error You'll Get 
NoUniqueBeanDefinitionException: No qualifying bean of type
'com.nimbus.service.TrackingService' available: expected single
matching bean but found 2: customerTrackingService, officerTrackingService

The Solution: @Qualifier

TrackingService.java — Interface 
package com.nimbus.service;

public interface TrackingService {
    String trackParcel(String trackingId);
}
CustomerTrackingService.java 
package com.nimbus.service;

import org.springframework.stereotype.Service;

@Service("customerTracker")
public class CustomerTrackingService implements TrackingService {

    @Override
    public String trackParcel(String trackingId) {
        // Customer sees: status + estimated delivery
        return "Your parcel " + trackingId + " is in transit. ETA: 2 days.";
    }
}
OfficerTrackingService.java 
package com.nimbus.service;

import org.springframework.stereotype.Service;

@Service("officerTracker")
public class OfficerTrackingService implements TrackingService {

    @Override
    public String trackParcel(String trackingId) {
        // Officer sees: full route, handler info, timestamps
        return "Parcel " + trackingId + " | Hub: Mumbai-Central | Handler: Officer #42 | Scanned: 14:32";
    }
}
Using @Qualifier to Pick the Right One 
@Controller
public class TrackingController {

    private final TrackingService trackingService;

    @Autowired
    public TrackingController(
            @Qualifier("customerTracker") TrackingService trackingService) {
        this.trackingService = trackingService;
    }

    // This controller will use CustomerTrackingService
}
Line-by-line explanation
1
@Service("customerTracker")
Registers this bean with the name "customerTracker". Without the name, the default would be "customerTrackingService" (class name with first letter lowercase).
2
@Qualifier("customerTracker")
Tells Spring: "I know there are multiple TrackingService beans. I specifically want the one named 'customerTracker'." This eliminates the ambiguity.

Alternative: @Primary

If one implementation should be the default choice (and the other is only used explicitly), mark the default with @Primary.

Using @Primary Instead 
@Service
@Primary     // This becomes the default when no @Qualifier is specified
public class CustomerTrackingService implements TrackingService {
    // ...
}

@Service
public class OfficerTrackingService implements TrackingService {
    // ...
}

// Now this works WITHOUT @Qualifier — Spring picks the @Primary one
@Autowired
public TrackingController(TrackingService trackingService) {
    // Gets CustomerTrackingService (the @Primary)
}
@Qualifier vs @Primary

@Primary = "This is the default. Use it unless told otherwise." Set it once on the implementation.

@Qualifier = "I specifically want THIS bean." Used at the injection point. Overrides @Primary.

Section 08

@Value — Injecting Configuration Values

Hardcoding database URLs, passwords, and API keys in your Java code is a bad idea. @Value lets you read values from a properties file and inject them into your beans.

The Properties File

application.properties 
# Database Configuration
db.driver=com.mysql.cj.jdbc.Driver
db.url=jdbc:mysql://localhost:3306/nimbus_db
db.username=root
db.password=nimbus2026

Using @Value to Read Properties

AppConfig.java — Using @Value for Database Config 
package com.nimbus.config;

import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.PropertySource;
import org.apache.commons.dbcp2.BasicDataSource;
import javax.sql.DataSource;

@Configuration
@PropertySource("classpath:application.properties")
public class AppConfig {

    @Value("${db.driver}")
    private String dbDriver;

    @Value("${db.url}")
    private String dbUrl;

    @Value("${db.username}")
    private String dbUsername;

    @Value("${db.password}")
    private String dbPassword;

    @Bean
    public DataSource dataSource() {
        BasicDataSource ds = new BasicDataSource();
        ds.setDriverClassName(dbDriver);
        ds.setUrl(dbUrl);
        ds.setUsername(dbUsername);
        ds.setPassword(dbPassword);
        return ds;
    }
}
Line-by-line explanation
1
@PropertySource("classpath:application.properties")
Tells Spring: "Load the file application.properties from the classpath (src/main/resources/) and make its key-value pairs available for @Value injection."
2
@Value("${db.url}")
The ${...} syntax is a placeholder. Spring looks up the key "db.url" in the loaded properties file and injects the value into this field. If the key doesn't exist, the app fails to start.
3
private String dbUrl;
This field will contain "jdbc:mysql://localhost:3306/nimbus_db" at runtime — read from the properties file, NOT hardcoded.

Default Values

You can provide a fallback value in case the property is missing:

@Value with Defaults 
@Value("${app.max-bookings-per-day:100}")    // default is 100 if property missing
private int maxBookingsPerDay;

@Value("${app.name:Nimbus}")                 // default is "Nimbus"
private String appName;
In Nimbus

The current DBConnection class has the database URL, username, and password hardcoded in the Java file. With @Value, these move to application.properties. Benefits:

  • Change database config without recompiling Java code
  • Different properties for dev vs production (different files)
  • Sensitive values (passwords) not in source code

Lifecycle Annotations

Sometimes you need to run code right after a bean is created or just before it's destroyed. That's what @PostConstruct and @PreDestroy do.

@PostConstruct — "Run This After the Bean is Ready"

A method annotated with @PostConstruct runs once, right after the bean is created and all its dependencies are injected. Think of it as an initialization hook.

→

DBConnection with @PostConstruct — Nimbus Example 
package com.nimbus.config;

import org.springframework.stereotype.Component;
import org.springframework.beans.factory.annotation.Value;
import javax.annotation.PostConstruct;
import javax.annotation.PreDestroy;
import javax.sql.DataSource;
import java.sql.Connection;

@Component
public class DatabaseHealthChecker {

    private final DataSource dataSource;

    @Value("${db.url}")
    private String dbUrl;

    public DatabaseHealthChecker(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    @PostConstruct
    public void checkConnectionOnStartup() {
        System.out.println("=== Nimbus Database Health Check ===");
        try (Connection conn = dataSource.getConnection()) {
            if (conn.isValid(5)) {
                System.out.println("Database connection OK: " + dbUrl);
            } else {
                System.out.println("WARNING: Database connection is NOT valid!");
            }
        } catch (Exception e) {
            System.out.println("CRITICAL: Cannot connect to database: " + e.getMessage());
        }
    }

    @PreDestroy
    public void onShutdown() {
        System.out.println("=== Nimbus Shutting Down — Closing DB pool ===");
    }
}
Line-by-line explanation
1
@PostConstruct public void checkConnectionOnStartup()
This method runs automatically after the bean is created and DataSource is injected. Perfect for startup validation — "Can we even reach the database?"
2
conn.isValid(5)
Tests if the connection is valid with a 5-second timeout. If the database is down, this returns false instead of hanging forever.
3
@PreDestroy public void onShutdown()
This method runs when the application is shutting down (e.g., server stop). Use it for cleanup — closing connections, flushing caches, saving state.

@PreDestroy — "Run This Before the Bean is Destroyed"

A method annotated with @PreDestroy runs just before Spring destroys the bean. Used for cleanup tasks:

  • Closing database connection pools
  • Flushing caches to disk
  • Logging shutdown messages
  • Stopping background threads
In Nimbus

@PostConstruct is ideal for DBConnection (now DatabaseHealthChecker) — test the MySQL connection at startup and fail fast if the database is unreachable. @PreDestroy could be used on BookingThread to cleanly stop background processing when the application shuts down.

Import Note

In Java 9+, javax.annotation.PostConstruct and javax.annotation.PreDestroy were removed from the JDK. You need to add the dependency: javax.annotation:javax.annotation-api:1.3.2 in your pom.xml. Or use Jakarta: jakarta.annotation:jakarta.annotation-api:2.1.1.

Why No Annotation on Model Classes?

Model classes (User, Booking) are data holders. You create new instances of them all the time (new User()). They're not beans managed by Spring — they're just plain Java objects. Spring manages service classes, DAO classes, and controllers. Models are passed through them.

BookingDAO.java

com/nimbus/dao/BookingDAO.java 
package com.nimbus.dao;

import com.nimbus.model.Booking;
import org.springframework.stereotype.Repository;
import org.springframework.beans.factory.annotation.Autowired;
import javax.sql.DataSource;
import java.sql.*;
import java.util.ArrayList;
import java.util.List;

@Repository
public class BookingDAO {

    private final DataSource dataSource;

    @Autowired
    public BookingDAO(DataSource dataSource) {
        this.dataSource = dataSource;
    }

    public Booking findById(int bookingId) {
        String sql = "SELECT * FROM Booking WHERE booking_id = ?";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setInt(1, bookingId);
            ResultSet rs = ps.executeQuery();

            if (rs.next()) {
                return mapRowToBooking(rs);
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error finding booking", e);
        }
        return null;
    }

    public List<Booking> findByUserId(int userId) {
        String sql = "SELECT * FROM Booking WHERE user_id = ? ORDER BY booking_date DESC";
        List<Booking> bookings = new ArrayList<>();
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql)) {

            ps.setInt(1, userId);
            ResultSet rs = ps.executeQuery();

            while (rs.next()) {
                bookings.add(mapRowToBooking(rs));
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error finding bookings by user", e);
        }
        return bookings;
    }

    public Booking save(Booking booking) {
        String sql = "INSERT INTO Booking (user_id, sender_address, receiver_address, weight, status) VALUES (?, ?, ?, ?, ?)";
        try (Connection conn = dataSource.getConnection();
             PreparedStatement ps = conn.prepareStatement(sql, Statement.RETURN_GENERATED_KEYS)) {

            ps.setInt(1, booking.getUserId());
            ps.setString(2, booking.getSenderAddress());
            ps.setString(3, booking.getReceiverAddress());
            ps.setDouble(4, booking.getWeight());
            ps.setString(5, "BOOKED");
            ps.executeUpdate();

            ResultSet keys = ps.getGeneratedKeys();
            if (keys.next()) {
                booking.setBookingId(keys.getInt(1));
            }
        } catch (SQLException e) {
            throw new RuntimeException("Error saving booking", e);
        }
        return booking;
    }

    private Booking mapRowToBooking(ResultSet rs) throws SQLException {
        Booking b = new Booking();
        b.setBookingId(rs.getInt("booking_id"));
        b.setUserId(rs.getInt("user_id"));
        b.setSenderAddress(rs.getString("sender_address"));
        b.setReceiverAddress(rs.getString("receiver_address"));
        b.setWeight(rs.getDouble("weight"));
        b.setStatus(rs.getString("status"));
        return b;
    }
}

NimbusApp.java — The Main Class

com/nimbus/NimbusApp.java 
package com.nimbus;

import com.nimbus.config.AppConfig;
import com.nimbus.model.User;
import com.nimbus.service.UserService;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class NimbusApp {

    public static void main(String[] args) {

        // Step 1: Create the Spring container using our config class
        AnnotationConfigApplicationContext context =
            new AnnotationConfigApplicationContext(AppConfig.class);

        // Step 2: Get beans from the container (NOT "new UserService()")
        UserService userService = context.getBean(UserService.class);

        // Step 3: Use the service — Spring has already injected UserDAO into it
        try {
            User newUser = new User("Darshan", "darshan@nimbus.com", "pass123", "CUSTOMER");
            User saved = userService.registerUser(newUser);
            System.out.println("Registered: " + saved.getName() + " (ID: " + saved.getUserId() + ")");
        } catch (Exception e) {
            System.out.println("Error: " + e.getMessage());
        }

        // Step 4: Fetch a user
        User found = userService.getUserById(1);
        if (found != null) {
            System.out.println("Found: " + found.getName() + " | Role: " + found.getRole());
        }

        // Step 5: Close the context (triggers @PreDestroy methods)
        context.close();
    }
}
Line-by-line explanation
1
new AnnotationConfigApplicationContext(AppConfig.class)
Creates the Spring container. Reads AppConfig, processes @ComponentScan, finds all annotated classes, creates beans, resolves @Autowired dependencies. Everything starts here.
2
context.getBean(UserService.class)
Asks the container: "Give me the UserService bean." Spring returns the singleton instance it already created, complete with UserDAO injected.
3
userService.registerUser(newUser)
Uses the service normally. Internally, registerUser() calls userDAO.findByEmail() and userDAO.save() — but we never created UserDAO ourselves. Spring did it all.
4
context.close()
Shuts down the Spring container. This triggers @PreDestroy methods on all beans, closes the connection pool, and cleans up resources. Always close the context.

Annotation Cheat Sheet

Every annotation on one page. Bookmark this.

Annotation What It Does Where to Put It Nimbus Usage
@Component Registers class as a generic Spring bean On a class Utility/helper classes (EmailValidator)
@Service Registers class as a business logic bean On a class UserService, TrackingService, BookingThread
@Repository Registers class as a data access bean + exception translation On a class UserDAO, BookingDAO
@Controller Registers class as a web request handler On a class UserController, BookingController
@Autowired Injects a dependency (bean) from the container Constructor, setter, or field UserService needs UserDAO, BookingDAO needs DataSource
@Configuration Marks class as a bean definition source On a class AppConfig.java
@Bean Method return value becomes a bean On a method inside @Configuration DataSource creation in AppConfig
@ComponentScan Tells Spring which packages to scan for annotated classes On @Configuration class @ComponentScan("com.nimbus") on AppConfig
@Scope Controls how many instances Spring creates On a class or @Bean method Default singleton for DAOs/services; prototype for report generators
@Qualifier Picks a specific bean when multiple match Next to @Autowired Choosing CustomerTrackingService vs OfficerTrackingService
@Primary Makes a bean the default when multiple match On a class CustomerTrackingService as default TrackingService
@Value Injects a value from a properties file On a field DB URL, username, password from application.properties
@PropertySource Loads a properties file On @Configuration class Loading application.properties
@PostConstruct Runs method after bean creation + injection On a method Database health check on startup
@PreDestroy Runs method before bean destruction On a method Cleanup on shutdown (close pools, stop threads)
@RequestMapping Maps URL to a controller class/method On class or method @RequestMapping("/users") on UserController
@GetMapping Maps GET request to a method On a method View profile, view bookings
@PostMapping Maps POST request to a method On a method Submit registration, create booking

Common Mistakes and Fixes

Mistake Error You'll See Fix
Forgot @Service/@Repository on class NoSuchBeanDefinitionException Add the annotation to the class
Class not in scanned package NoSuchBeanDefinitionException Check @ComponentScan base package
Two beans of same type, no @Qualifier NoUniqueBeanDefinitionException Add @Qualifier("name") or @Primary
Misspelled property key in @Value IllegalArgumentException: Could not resolve placeholder Check the key name matches the properties file exactly
Missing @PropertySource @Value fields are null or placeholder string Add @PropertySource("classpath:application.properties")
Circular dependency (A needs B, B needs A) BeanCurrentlyInCreationException Redesign classes or use setter injection on one side
Using new UserService() instead of getting from context NullPointerException (dependencies not injected) Always get beans from ApplicationContext, never new
