Section 01

What Is API Testing?

You built REST endpoints. Now you need to test them. The browser can't do it all.

Why the Browser Isn't Enough

You can type http://localhost:8080/api/bookings/1001 in a browser and see the JSON response. That works for GET requests. But browsers can only make GET requests from the address bar. You can't:

  • Send a POST request with a JSON body to create a booking
  • Send a PUT request to update a status
  • Set custom headers like Content-Type: application/json
  • Test error responses (400, 404, 500) systematically

You need a tool that lets you craft any HTTP request with any method, headers, and body. That's what Postman and curl do.

Browser
  • Only GET requests
  • Can't set custom headers
  • Can't send request body
  • Good for quick checks
Postman / curl
  • Any HTTP method (GET, POST, PUT, DELETE)
  • Custom headers, auth tokens
  • JSON request body
  • Save requests, organize in collections
Section 02

Postman — Setup & Basics

Install Postman, create your first request, and learn the interface.

Installation

  1. Go to postman.com/downloads and download the desktop app (free)
  2. Install and open it. You can use it without an account (skip sign-in)
  3. Click the + tab to create a new request

The Postman Interface

+------------------------------------------------------------------+
|  [GET  v]  http://localhost:8080/api/bookings/1001     [  Send  ] |
+------------------------------------------------------------------+
|  Params | Authorization | Headers | Body | Pre-request | Tests   |
+------------------------------------------------------------------+
|                                                                  |
|  Body tab:  ( none )  ( raw )  ( JSON v )                        |
|  +-------------------------------------------------------------+ |
|  | {                                                            | |
|  |   "senderName": "Darshan",                                   | |
|  |   "receiverName": "Amit",                                    | |
|  |   "weight": 2.5                                              | |
|  | }                                                            | |
|  +-------------------------------------------------------------+ |
+------------------------------------------------------------------+
|  Response:   Status: 200 OK    Time: 45ms    Size: 234 B        |
|  +-------------------------------------------------------------+ |
|  | {                                                            | |
|  |   "bookingId": 1001,                                         | |
|  |   "status": "BOOKED",                                        | |
|  |   ...                                                        | |
|  | }                                                            | |
|  +-------------------------------------------------------------+ |
+------------------------------------------------------------------+
Postman request builder — method dropdown, URL, tabs for headers/body, response panel below

Key Concepts

  • Method dropdown: Switch between GET, POST, PUT, DELETE
  • URL bar: Enter the full endpoint URL
  • Headers tab: Set Content-Type: application/json (Postman often auto-sets this)
  • Body tab: For POST/PUT, select "raw" and "JSON", then write your JSON
  • Response panel: Shows status code, response time, and the response body
  • Collections: Save requests into folders. Create a "Nimbus API" collection with all 8 endpoints

Section 04

curl Commands

Test APIs from the terminal. No GUI needed. Every developer should know curl.

What is curl?

curl is a command-line tool for making HTTP requests. It comes pre-installed on macOS, Linux, and Windows 10+. It's the fastest way to test an API without opening any application.

All 8 Nimbus Endpoints via curl

1. Register 
curl -X POST http://localhost:8080/api/register \
  -H "Content-Type: application/json" \
  -d '{"name":"Darshan Mohan","email":"darshan@nimbus.com","password":"secret123","role":"CUSTOMER"}'
2. Login 
curl -X POST http://localhost:8080/api/login \
  -H "Content-Type: application/json" \
  -d '{"email":"darshan@nimbus.com","password":"secret123"}'
3. Create Booking 
curl -X POST http://localhost:8080/api/bookings \
  -H "Content-Type: application/json" \
  -d '{"userId":1,"senderName":"Darshan","senderAddress":"Mumbai","receiverName":"Amit","receiverAddress":"Delhi","weight":2.5,"pickupDate":"2026-04-25"}'
4. Get Booking by ID 
curl http://localhost:8080/api/bookings/1001
5. Get User's Bookings 
curl "http://localhost:8080/api/bookings?userId=1"
6. Update Status 
curl -X PUT http://localhost:8080/api/bookings/1001/status \
  -H "Content-Type: application/json" \
  -d '{"status":"IN_TRANSIT"}'
7. Update Pickup Date 
curl -X PUT http://localhost:8080/api/bookings/1001/pickup \
  -H "Content-Type: application/json" \
  -d '{"pickupDate":"2026-04-26"}'
8. Booking History 
curl "http://localhost:8080/api/bookings/history?userId=1"

curl Flags Reference

Flag Purpose Example
-X Set HTTP method -X POST, -X PUT
-H Add a header -H "Content-Type: application/json"
-d Request body data -d '{"name":"Darshan"}'
-v Verbose (shows headers) curl -v http://...
-i Include response headers curl -i http://...
Windows Note

On Windows CMD/PowerShell, use double quotes for JSON and escape inner quotes: -d "{\"name\":\"Darshan\"}". Or use Git Bash, which supports single quotes like Linux/macOS.

Section 05

Automated Testing Concept

Manual testing with Postman works, but real projects need automated tests that run every time you build.

Why Automate?

Imagine you have 8 endpoints and each needs 3-5 test cases (success, validation error, not found, etc.). That's 30+ tests. Running them manually in Postman after every code change is tedious and error-prone.

Spring Boot provides MockMvc for writing automated API tests in Java:

BookingControllerTest.java — Automated API test 
@SpringBootTest
@AutoConfigureMockMvc
public class BookingControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Test
    void getBookingById_Returns200() throws Exception {
        mockMvc.perform(get("/api/bookings/1001"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.bookingId").value(1001))
            .andExpect(jsonPath("$.status").value("BOOKED"));
    }

    @Test
    void getBookingById_Returns404_WhenNotFound() throws Exception {
        mockMvc.perform(get("/api/bookings/9999"))
            .andExpect(status().isNotFound())
            .andExpect(jsonPath("$.error").value("Not Found"));
    }

    @Test
    void createBooking_Returns201() throws Exception {
        String json = "{\"userId\":1,\"senderName\":\"Darshan\",\"receiverName\":\"Amit\",\"weight\":2.5}";

        mockMvc.perform(post("/api/bookings")
                .contentType(MediaType.APPLICATION_JSON)
                .content(json))
            .andExpect(status().isCreated())
            .andExpect(jsonPath("$.bookingId").exists());
    }
}
MockMvc

MockMvc simulates HTTP requests without starting a real server. It calls your controller methods directly, applies all filters and exception handlers, and returns the response. Tests run in milliseconds instead of seconds. You run them with mvn test and they execute automatically in CI/CD pipelines.

Section 06

Swagger / OpenAPI Preview

Auto-generate interactive API documentation from your code. The industry standard.

What is Swagger?

Swagger (now called OpenAPI) auto-generates a web page that lists all your API endpoints, their parameters, request bodies, and response formats. You can even test endpoints directly from the page.

Add to Spring Boot

pom.xml — Add springdoc-openapi 
<dependency>
    <groupId>org.springdoc</groupId>
    <artifactId>springdoc-openapi-starter-webmvc-ui</artifactId>
    <version>2.3.0</version>
</dependency>

That's it. No configuration needed. Start your app and visit:

  • http://localhost:8080/swagger-ui.html — Interactive UI
  • http://localhost:8080/v3/api-docs — Raw OpenAPI JSON spec

Swagger scans your @RestController classes and generates documentation for every endpoint, including parameter types, request body schemas, and response codes.

In Nimbus

Add the springdoc dependency to your pom.xml and you instantly get a professional API documentation page. When you hand off the API to the Angular team (or any frontend developer), you share the Swagger URL instead of writing documentation manually. It's always up-to-date because it reads directly from your code.

Section 07

Practice Exercises

Hands-on API testing practice.

Exercise 1: Postman Collection (Easy)

Exercise 1

Create a Postman collection called "Nimbus API". Add all 8 endpoints. For each, set the correct method, URL, headers, and body. Run them in sequence: register → login → create booking → get booking → update status → get history. Save the collection.

Exercise 2: Error Testing (Medium)

Exercise 2

Test every error path in Postman:

  • Register with a duplicate email → expect 400
  • Login with wrong password → expect 401
  • Get booking with non-existent ID → expect 404
  • Create booking with empty sender name → expect 400 (validation)
  • Create booking with missing fields → expect 400

For each, verify the response uses the ErrorResponse JSON structure.

Exercise 3: curl Mastery (Medium)

Exercise 3

Run all 8 Nimbus curl commands from your terminal. Use the -i flag to see response headers. Verify the Content-Type header is application/json and the status codes are correct (201 for POST, 200 for GET/PUT).

Exercise 4: Swagger Setup (Easy)

Exercise 4

Add the springdoc-openapi-starter-webmvc-ui dependency to your Nimbus pom.xml. Start the app and open http://localhost:8080/swagger-ui.html. Explore the auto-generated docs. Try testing an endpoint directly from the Swagger UI page.

Self-Check Questions
  1. Why can't you test POST/PUT endpoints in a browser? (Browser address bar only sends GET requests)
  2. What does the -H "Content-Type: application/json" header tell the server? (The request body is JSON format, so use Jackson to parse it)
  3. What's the difference between Postman and curl? (Postman is a GUI tool; curl is command-line. Both send HTTP requests)
  4. What does Swagger auto-generate? (Interactive API documentation from your @RestController annotations)
