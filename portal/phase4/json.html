Section 01

What is JSON?

JSON (JavaScript Object Notation) is the universal language APIs use to send and receive data. When your Angular frontend talks to your Spring Boot backend, the conversation happens in JSON.

The Simplest Definition

JSON is a text-based data format that's both human-readable and machine-parseable. It looks like this:

A JSON object 
{
  "name": "Darshan",
  "age": 23,
  "developer": true,
  "email": null
}

That's it. Curly braces, key-value pairs, comma-separated. Every API you'll ever build or consume uses this format.

Analogy

Think of JSON as a parcel label. It has structured fields (sender, receiver, weight, address) that both the sender and the delivery system can read. JSON is the label format that both frontend and backend agree on.

Why JSON Matters for Nimbus

In Phase 3, your servlets returned HTML pages. In Phase 4, your REST APIs return JSON data. The Angular frontend receives this JSON and builds the UI from it. This separation is what makes REST powerful — the backend doesn't care if the consumer is a web app, mobile app, or another service.

Angular Frontend sends JSON request
→
Spring Boot API processes request
→
JSON Response data, not HTML
REST APIs exchange data as JSON — not HTML pages
Section 02

JSON Syntax

Six data types. Two structures. That's the entire language.

The Six Data Types

Type Example Notes
String "hello" Always double quotes. Never single quotes.
Number 42, 3.14, -7 No quotes. Integers and decimals.
Boolean true, false Lowercase only. No quotes.
Null null Means "no value." Lowercase, no quotes.
Object { "key": "value" } Curly braces. Unordered key-value pairs.
Array [1, 2, 3] Square brackets. Ordered list of values.

Objects — Key-Value Pairs

An object is wrapped in { }. Keys are always strings. Values can be any of the six types, including other objects and arrays:

JSON Object with nested data 
{
  "bookingId": 1001,
  "senderName": "Darshan",
  "receiverName": "Amit",
  "weight": 2.5,
  "fragile": true,
  "specialInstructions": null,
  "address": {
    "street": "MG Road",
    "city": "Bangalore",
    "pincode": "560001"
  }
}
Line-by-line explanation
2
"bookingId": 1001
Number value — no quotes around 1001. JSON numbers can be integers or decimals.
3
"senderName": "Darshan"
String value — must be in double quotes. Both the key AND the value are quoted for strings.
6
"fragile": true
Boolean — lowercase true or false. No quotes. "true" would be a string, not a boolean.
7
"specialInstructions": null
Null means "this field exists but has no value." Different from omitting the field entirely.
8-12
"address": { ... }
Nested object — the value of "address" is another JSON object with its own key-value pairs. You can nest as deep as needed.

Arrays — Ordered Lists

An array is wrapped in [ ]. It holds an ordered list of values (any type, including mixed types):

JSON Arrays 
{
  "tags": ["express", "fragile", "insured"],
  "trackingHistory": [
    { "status": "Booked", "timestamp": "2026-04-23T10:00:00" },
    { "status": "In Transit", "timestamp": "2026-04-23T14:30:00" },
    { "status": "Delivered", "timestamp": "2026-04-24T09:15:00" }
  ]
}

tags is an array of strings. trackingHistory is an array of objects. Arrays of objects are the most common pattern in API responses — it's how you return a list of bookings, users, or any collection.

Syntax Rules — Common Mistakes

JSON Gotchas
  • No trailing commas: {"a": 1, "b": 2,} is INVALID. The comma after 2 breaks it.
  • No single quotes: {'name': 'John'} is INVALID. Only double quotes.
  • No comments: You cannot add // comment or /* comment */ in JSON.
  • Keys must be strings: {name: "John"} is INVALID. Must be {"name": "John"}.
  • No undefined: JavaScript has undefined, but JSON does not. Use null.
Section 03

JSON vs XML

Before JSON, XML was the standard data format. Here's why JSON replaced it in most modern APIs.

The Same Data, Two Formats

JSON
{
  "name": "Darshan",
  "age": 23,
  "skills": ["Java", "Angular"]
}

68 characters

XML
<person>
  <name>Darshan</name>
  <age>23</age>
  <skills>
    <skill>Java</skill>
    <skill>Angular</skill>
  </skills>
</person>

156 characters

Comparison Table

Feature JSON XML
Readability Clean, compact Verbose, tag-heavy
File size Smaller (less overhead) Larger (opening + closing tags)
Parsing speed Faster Slower
Data types String, number, boolean, null, object, array Everything is a string (no native types)
Arrays Native [ ] syntax No native array — repeated elements
Comments Not supported Supported
Schema validation JSON Schema (optional) XSD, DTD (mature)
Used by REST APIs, modern web SOAP APIs, enterprise/legacy systems
Why JSON Won

JSON is native to JavaScript — you can parse it with one line: JSON.parse(text). XML needs a parser, DOM traversal, and type conversion. As the web moved to JavaScript-heavy frontends, JSON became the natural choice. Today, over 90% of public APIs use JSON as their primary format.

Section 05

Jackson Library

Jackson is the library that converts Java objects to JSON and back. Spring Boot includes it automatically — you don't need to add any dependency.

What Jackson Does

Java Object User user = new User()
→
Serialization Java → JSON
→
JSON String {"name": "Darshan"}
↓ and the reverse
JSON String {"name": "Darshan"}
→
Deserialization JSON → Java
→
Java Object User user
Serialization = Java → JSON (response) | Deserialization = JSON → Java (request)

ObjectMapper — The Core Class

ObjectMapper is Jackson's main class. It handles both directions:

ObjectMapper — Serialization (Java → JSON) 
import com.fasterxml.jackson.databind.ObjectMapper;

ObjectMapper mapper = new ObjectMapper();

// Create a Java object
User user = new User(1, "Darshan", "darshan@nimbus.com");

// Convert to JSON string
String json = mapper.writeValueAsString(user);
System.out.println(json);
// Output: {"userId":1,"name":"Darshan","email":"darshan@nimbus.com"}
Line-by-line explanation
3
ObjectMapper mapper = new ObjectMapper();
Creates the Jackson converter. In Spring Boot, you never do this manually — Spring creates one for you. But in standalone Java, this is how you start.
8
mapper.writeValueAsString(user)
writeValueAsString takes any Java object, reads its getter methods, and builds a JSON string from the field values. User's getName() becomes "name": "Darshan".
ObjectMapper — Deserialization (JSON → Java) 
String json = "{\"userId\":1,\"name\":\"Darshan\",\"email\":\"darshan@nimbus.com\"}";

// Convert JSON string to Java object
User user = mapper.readValue(json, User.class);

System.out.println(user.getName());   // "Darshan"
System.out.println(user.getEmail());  // "darshan@nimbus.com"
Line-by-line explanation
4
mapper.readValue(json, User.class)
readValue takes a JSON string and a target class. Jackson reads the JSON keys, matches them to setter methods on User, and populates a new User object. "name": "Darshan" calls setName("Darshan").

How Jackson Matches Fields

Jackson uses getter/setter method names to determine JSON key names. It strips "get"/"set" and lowercases the first letter:

Java Method JSON Key
getName() / setName() "name"
getUserId() / setUserId() "userId"
isFragile() "fragile" (boolean getters use "is")
Jackson Requirements

For Jackson to work, your Java class needs:

  • A no-arg constructor (for deserialization — Jackson needs to create an empty object first)
  • Getter methods (for serialization — Jackson reads values through getters)
  • Setter methods (for deserialization — Jackson sets values through setters)

If your Nimbus User and Booking classes already follow the JavaBean convention with getters/setters, Jackson works out of the box.

Section 06

Jackson Annotations

When the default behavior isn't enough, Jackson annotations give you fine-grained control over serialization and deserialization.

@JsonProperty — Custom Key Names

When you want the JSON key to be different from the Java field name:

@JsonProperty 
public class User {
    @JsonProperty("user_id")     // JSON key will be "user_id" not "userId"
    private int userId;

    @JsonProperty("full_name")   // JSON key will be "full_name" not "name"
    private String name;

    private String email;        // No annotation = uses "email" (default)
}

// Produces:
// { "user_id": 1, "full_name": "Darshan", "email": "darshan@nimbus.com" }

@JsonIgnore — Hide Fields

Some fields should never appear in JSON responses — like passwords:

@JsonIgnore — Security-sensitive fields 
public class User {
    private int userId;
    private String name;
    private String email;

    @JsonIgnore                  // This field will NOT appear in JSON output
    private String password;

    @JsonIgnore                  // Internal field, not needed by frontend
    private String internalNotes;
}

// Produces:
// { "userId": 1, "name": "Darshan", "email": "darshan@nimbus.com" }
// password and internalNotes are excluded
In Nimbus

The User entity has a password field. You must use @JsonIgnore on it. Without it, every API response that includes a user would expose their password in the JSON. This is the #1 security mistake in REST APIs.

@JsonFormat — Date Formatting

Dates in Java are complex objects. @JsonFormat controls how they appear in JSON:

@JsonFormat — Date patterns 
import com.fasterxml.jackson.annotation.JsonFormat;
import java.time.LocalDateTime;
import java.time.LocalDate;

public class Booking {
    private int bookingId;

    @JsonFormat(pattern = "yyyy-MM-dd")
    private LocalDate pickupDate;
    // Produces: "pickupDate": "2026-04-23"

    @JsonFormat(pattern = "yyyy-MM-dd HH:mm:ss")
    private LocalDateTime createdAt;
    // Produces: "createdAt": "2026-04-23 10:30:00"

    // Without @JsonFormat, LocalDateTime becomes:
    // "createdAt": [2026, 4, 23, 10, 30, 0]  <-- ugly array!
}

@JsonInclude — Skip Null/Empty Fields

@JsonInclude 
import com.fasterxml.jackson.annotation.JsonInclude;

@JsonInclude(JsonInclude.Include.NON_NULL)  // Skip ALL null fields
public class Booking {
    private int bookingId;
    private String senderName;
    private String specialInstructions;  // often null
}

// If specialInstructions is null:
// { "bookingId": 1001, "senderName": "Darshan" }
// specialInstructions is omitted entirely instead of showing null

Annotation Reference

Annotation Purpose Example
@JsonProperty Custom JSON key name @JsonProperty("user_id")
@JsonIgnore Exclude field from JSON @JsonIgnore private String password
@JsonFormat Date/time format pattern @JsonFormat(pattern = "yyyy-MM-dd")
@JsonInclude Skip null/empty fields @JsonInclude(Include.NON_NULL)
@JsonIgnoreProperties Ignore unknown fields on deserialization @JsonIgnoreProperties(ignoreUnknown = true)
Section 07

Parsing JSON in Java

Complete ObjectMapper operations — from single objects to lists to nested structures.

writeValueAsString() — Java to JSON

Serializing different types 
ObjectMapper mapper = new ObjectMapper();

// Single object
User user = new User(1, "Darshan", "darshan@nimbus.com");
String json = mapper.writeValueAsString(user);
// {"userId":1,"name":"Darshan","email":"darshan@nimbus.com"}

// List of objects
List<User> users = List.of(
    new User(1, "Darshan", "darshan@nimbus.com"),
    new User(2, "Amit", "amit@nimbus.com")
);
String jsonArray = mapper.writeValueAsString(users);
// [{"userId":1,"name":"Darshan",...},{"userId":2,"name":"Amit",...}]

// Pretty print (indented JSON)
String pretty = mapper.writerWithDefaultPrettyPrinter()
                      .writeValueAsString(user);
// {
//   "userId" : 1,
//   "name" : "Darshan",
//   "email" : "darshan@nimbus.com"
// }

readValue() — JSON to Java

Deserializing different types 
ObjectMapper mapper = new ObjectMapper();

// Single object
String json = "{\"userId\":1,\"name\":\"Darshan\"}";
User user = mapper.readValue(json, User.class);

// List of objects (needs TypeReference because of Java generics)
String jsonArray = "[{\"userId\":1,\"name\":\"Darshan\"},{\"userId\":2,\"name\":\"Amit\"}]";
List<User> users = mapper.readValue(jsonArray, new TypeReference<List<User>>() {});

// From a file
User user = mapper.readValue(new File("user.json"), User.class);

// From an InputStream (useful for HTTP requests)
User user = mapper.readValue(inputStream, User.class);
Line-by-line explanation
5
mapper.readValue(json, User.class)
Pass the JSON string and the target class. Jackson creates a new User, matches JSON keys to setter methods, and populates the fields.
9
new TypeReference<List<User>>() {}
Java's generics get erased at runtime, so List<User>.class doesn't exist. TypeReference is Jackson's workaround — it preserves the generic type information so Jackson knows to create a List of User objects, not just a list of raw maps.
You Won't Use ObjectMapper Directly

In Spring Boot, you never manually call writeValueAsString() or readValue(). When a controller method returns a Java object, Spring Boot automatically uses Jackson to serialize it to JSON. When a request body contains JSON, Spring Boot automatically deserializes it to a Java object via @RequestBody. The ObjectMapper knowledge is for understanding what happens under the hood.

Error Response

When something goes wrong, the API returns an error in JSON too:

Error Response (400 Bad Request) 
{
  "status": 400,
  "error": "Bad Request",
  "message": "Sender name is required",
  "timestamp": "2026-04-23T10:35:00"
}
In Nimbus

Every Nimbus API endpoint follows this pattern: request body is the data the client sends (JSON), response body is the data the server sends back (JSON), error body is a consistent error object (JSON). The format never changes — it's always JSON. This makes the Angular frontend's job straightforward: parse the JSON, display the data.

Section 09

JSON Schema Basics

JSON Schema is a way to describe and validate the structure of JSON data. It's like a blueprint that says "this JSON must have these fields with these types."

What is JSON Schema?

Just as a database has a schema that defines table columns and their types, JSON can have a schema that defines expected fields and their types. This is used for:

  • API documentation: Tell frontend developers exactly what to send
  • Validation: Reject invalid requests before they hit your business logic
  • Code generation: Auto-generate Java/TypeScript classes from the schema

A Simple Schema

JSON Schema for a Nimbus User 
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "type": "object",
  "properties": {
    "name": {
      "type": "string",
      "minLength": 1,
      "maxLength": 100
    },
    "email": {
      "type": "string",
      "format": "email"
    },
    "phone": {
      "type": "string",
      "pattern": "^[0-9]{10}$"
    },
    "role": {
      "type": "string",
      "enum": ["CUSTOMER", "OFFICER", "ADMIN"]
    }
  },
  "required": ["name", "email", "role"]
}
Line-by-line explanation
3
"type": "object"
The top-level JSON value must be an object (not an array, string, etc.).
6-7
"type": "string", "minLength": 1
The "name" field must be a string with at least 1 character. An empty string "" would fail validation.
16
"pattern": "^[0-9]{10}$"
Regex pattern validation. Phone must be exactly 10 digits. "98765" would fail, "9876543210" would pass.
20
"enum": ["CUSTOMER", "OFFICER", "ADMIN"]
The value must be one of these three strings. Any other value (like "MANAGER") would fail validation.
23
"required": ["name", "email", "role"]
These three fields must be present. Phone is optional — it's not in the required list.
Schema in Practice

You won't write JSON schemas manually in Nimbus — Spring Boot's @Valid annotations handle validation on the Java side (covered in the API Error Handling topic). But understanding schemas helps when reading API documentation or working with tools like Swagger/OpenAPI that auto-generate schemas from your code.
