Basics

HackerRank Input Pattern

PRA runs on HackerRank. The input is provided via standard input (stdin) — no prompts, no GUI, no "Enter name:" messages. You just read values directly in the order the question specifies.

The Setup — Always the Same

Scanner Setup 
import java.util.Scanner;

Scanner sc = new Scanner(System.in);
⚠ No Prompts!

Do NOT write System.out.println("Enter name:"); or any prompt messages. HackerRank sends input automatically. Any extra output you print will cause your answer to fail because the expected output won't match.

Key Rules for HackerRank

  • Always import: import java.util.Scanner; (or import java.util.*; which covers everything)
  • One Scanner: Create it once at the start of main. Don't create multiple scanners.
  • No prompts: Just sc.nextLine(), never println("Enter:"); before it
  • Read in order: The question tells you the exact order of inputs. Follow it precisely.
Methods

Reading Primitives

Scanner has different methods for reading different data types. Using the wrong one causes runtime errors or reads incorrect values.

Scanner Read Methods 
int n = sc.nextInt();       // reads an integer
double d = sc.nextDouble(); // reads a double
String s = sc.nextLine();   // reads entire line (including spaces)
String word = sc.next();    // reads single word (stops at space)
Method Reads Stops At Use For
nextInt() Integer number Whitespace (leaves \n in buffer!) IDs, counts, quantities
nextDouble() Decimal number Whitespace (leaves \n in buffer!) Prices, gains, averages
nextLine() Entire line Newline (\n) — consumes it Names, genres, addresses
next() Single word Any whitespace Single-word inputs (rarely needed in PRA)
💡 Key Difference

nextLine() reads the ENTIRE line including spaces and consumes the newline character at the end. nextInt() and nextDouble() only read the number and leave the newline sitting in the buffer. This leftover newline is the source of the #1 bug in PRA.

Critical Bug

THE NEWLINE TRAP

This is the #1 bug that fails PRA solutions. If you understand and fix this one thing, you'll avoid the most common cause of "wrong answer" on correct logic. Read this section carefully — it will save your marks.

The Problem

When you call sc.nextInt(), it reads the number but leaves the newline character (\n) in the input buffer. When you then call sc.nextLine(), it reads that leftover newline and returns an EMPTY string instead of the actual next line.

Visualizing the Bug

Let's say the input is:

Input 
1001
Savings
  INPUT BUFFER:  [ 1  0  0  1  \n  S  a  v  i  n  g  s  \n ]
                  ~~~~~~~~~~~~  ~~  ~~~~~~~~~~~~~~~~~~~~~~~~~
                  nextInt reads  leftover   nextLine should read this
                  "1001"         \n          but reads \n instead!

  STEP 1: int x = sc.nextInt();
  Buffer after:  [ \n  S  a  v  i  n  g  s  \n ]
                   ^^
                   This \n is still here!

  STEP 2: String str = sc.nextLine();
  Buffer after:  [ S  a  v  i  n  g  s  \n ]
                 str = "" (EMPTY! It read the \n and stopped)

  RESULT: x = 1001, str = "" (NOT "Savings"!)
The leftover \n from nextInt() poisons the next nextLine() call

The Buggy Code

BUG — Will Fail
int x = sc.nextInt();
String str = sc.nextLine();
// str = "" (EMPTY!)
// Reads leftover \n, not "Savings"
FIX — Works Correctly
int x = sc.nextInt();
sc.nextLine();  // consume leftover \n
String str = sc.nextLine();
// str = "Savings" (CORRECT!)

The Fix Visualized

  INPUT BUFFER:  [ 1  0  0  1  \n  S  a  v  i  n  g  s  \n ]

  STEP 1: int x = sc.nextInt();        // reads 1001
  Buffer:  [ \n  S  a  v  i  n  g  s  \n ]

  STEP 2: sc.nextLine();                // GARBAGE COLLECTOR! Eats the \n
  Buffer:  [ S  a  v  i  n  g  s  \n ]

  STEP 3: String str = sc.nextLine();   // NOW reads "Savings"
  Buffer:  [ ]  (empty)

  RESULT: x = 1001, str = "Savings" (CORRECT!)
The extra sc.nextLine() acts as a garbage collector for the leftover newline
🎯 THE GOLDEN RULE

Every time you call sc.nextInt() or sc.nextDouble(), and the NEXT read is sc.nextLine(), you MUST add an extra sc.nextLine() in between to consume the leftover newline.

This is not optional. This is not "good practice." This is the difference between your code working and failing every test case.

When You DON'T Need the Extra nextLine()

After Next Read Need Extra nextLine()? Why
nextInt() nextLine() YES nextLine reads the leftover \n
nextDouble() nextLine() YES Same problem as nextInt
nextInt() nextInt() No nextInt skips whitespace automatically
nextDouble() nextDouble() No nextDouble also skips whitespace
nextLine() nextLine() No nextLine already consumed the \n
nextInt() nextDouble() No nextDouble skips whitespace
📚 Simple Rule

Only add the extra sc.nextLine() when going from a number read (nextInt/nextDouble) to a line read (nextLine). Number-to-number is fine. Line-to-line is fine. Only number-to-line causes the trap.

Pattern

Reading N Objects in a Loop

The most common PRA input pattern: first read how many objects (n), then read each object's fields in a loop. The newline trap appears at every transition from number to string.

Reading N Investor Objects 
int n = sc.nextInt();                    // how many investors?
ArrayList<Investor> list = new ArrayList<>();

for (int i = 0; i < n; i++) {
    int id = sc.nextInt();               // read ID (number)
    sc.nextLine();                       // CONSUME leftover \n
    String name = sc.nextLine();         // read name (string)
    double gains = sc.nextDouble();      // read gains (number)
    list.add(new Investor(id, name, gains));
}
Reading order explained
1
int n = sc.nextInt();
First line of input is always the count. "How many objects am I about to read?"
2
int id = sc.nextInt();
First field of each object. In the loop, we read one complete object per iteration.
3
sc.nextLine(); // CONSUME
CRITICAL! We just read an int (id), and the next read is nextLine (name). We MUST consume the leftover \n here.
4
String name = sc.nextLine();
Now reads the actual name string correctly, because we already consumed the leftover \n.
5
double gains = sc.nextDouble();
Reading a number after a line — no extra consume needed. nextDouble skips whitespace automatically.

Sample Input (n=2 Investors)

What Each Line Reads

Pattern

Reading Method Parameters After Objects

After reading all objects in the loop, PRA often asks you to read additional parameters that will be passed to your methods (like a genre to search for, or a discount percentage).

Reading Parameters After Object Loop 
// After the object reading loop ends...
// The last read in the loop was probably nextDouble() or nextInt()
// So we MUST consume the leftover \n before reading strings

sc.nextLine();                           // consume leftover \n from loop's last read
String searchGenre = sc.nextLine();      // parameter for method 1
String discountGenre = sc.nextLine();    // parameter for method 2
int discount = sc.nextInt();             // discount percentage
⚠ Easy to Forget

The sc.nextLine() between the loop and the parameters is the one most people forget. Your loop's last read was a number (like gains = sc.nextDouble()). When the loop ends, that leftover \n is still in the buffer. If you immediately call sc.nextLine() for the search parameter, you'll get an empty string.

Full Input Reading Flow

Read nsc.nextInt()
→
Loop n timesRead each object's fields
→
Consume \nsc.nextLine()
→
Read paramsMethod arguments
The consume step between loop and parameters is the most forgotten sc.nextLine()
Advanced

Reading Composed Objects

When one class contains another (like Author inside Book), you read ALL fields for BOTH classes in a single loop iteration. The reading order follows the input specification exactly.

Author-Book Pattern

The question says: "For each entry, read Author fields first, then Book fields." Your code must follow this exact order.

Reading Composed Objects (Author + Book) 
int n = sc.nextInt();
ArrayList<Book> books = new ArrayList<>();

for (int i = 0; i < n; i++) {
    // Read Author fields first (as question specifies)
    int authorId = sc.nextInt();
    sc.nextLine();                       // consume \n after int
    String authorName = sc.nextLine();   // read author name

    // Then read Book fields
    int bookId = sc.nextInt();
    sc.nextLine();                       // consume \n after int
    String title = sc.nextLine();        // read book title
    String genre = sc.nextLine();        // read genre (line after line = no consume needed)
    double price = sc.nextDouble();      // read price

    // Create objects
    Author author = new Author(authorId, authorName);
    Book book = new Book(bookId, title, genre, price, author);
    books.add(book);
}
Reading order with consume points marked
1
int authorId = sc.nextInt();
Read author ID (number). Leaves \n in buffer.
2
sc.nextLine(); // consume
Number-to-line transition: MUST consume. Without this, authorName would be empty.
3
String authorName = sc.nextLine();
Reads the author's full name correctly now.
4
int bookId = sc.nextInt();
Line-to-number: no consume needed. nextInt skips whitespace automatically.
5
sc.nextLine(); // consume
Number-to-line transition again: MUST consume before reading title.
6
String genre = sc.nextLine();
Line-to-line: no consume needed. nextLine already ate the previous \n.
7
double price = sc.nextDouble();
Line-to-number: no consume needed. nextDouble skips whitespace.

Templates

Scanner Templates

Three copy-paste templates for the three input patterns you'll see in PRA. Pick the one that matches your question and fill in the field names.

Template 1: Simple Object Reading

📋 Copy-Paste Template

Use for flat objects with no composition (e.g., Investor with id, name, gains).

Template — Simple Object Input 
Scanner sc = new Scanner(System.in);
int n = sc.nextInt();
ArrayList<ClassName> list = new ArrayList<>();

for (int i = 0; i < n; i++) {
    int id = sc.nextInt();
    sc.nextLine();                  // consume \n after int
    String name = sc.nextLine();
    double value = sc.nextDouble();
    list.add(new ClassName(id, name, value));
}

Template 2: Composed Object Reading

📋 Copy-Paste Template

Use when one class contains another (e.g., Book has an Author inside it).

Template — Composed Object Input 
Scanner sc = new Scanner(System.in);
int n = sc.nextInt();
ArrayList<Child> list = new ArrayList<>();

for (int i = 0; i < n; i++) {
    // Parent fields
    int parentId = sc.nextInt();
    sc.nextLine();                      // consume \n
    String parentName = sc.nextLine();

    // Child fields
    int childId = sc.nextInt();
    sc.nextLine();                      // consume \n
    String childField1 = sc.nextLine();
    String childField2 = sc.nextLine(); // line-to-line: no consume
    double childValue = sc.nextDouble();

    Parent parent = new Parent(parentId, parentName);
    Child child = new Child(childId, childField1, childField2, childValue, parent);
    list.add(child);
}

Template 3: Object Reading + Method Parameters

📋 Copy-Paste Template

Use when the question reads objects in a loop, then additional parameters for the methods at the end.

Template — Objects + Parameters 
Scanner sc = new Scanner(System.in);
int n = sc.nextInt();
ArrayList<ClassName> list = new ArrayList<>();

// Read objects
for (int i = 0; i < n; i++) {
    int id = sc.nextInt();
    sc.nextLine();
    String name = sc.nextLine();
    double value = sc.nextDouble();
    list.add(new ClassName(id, name, value));
}

// Read method parameters
sc.nextLine();                          // CRITICAL: consume \n from loop's last read
String param1 = sc.nextLine();          // e.g., genre to search
String param2 = sc.nextLine();          // e.g., genre for discount
int param3 = sc.nextInt();              // e.g., discount percentage

// Call methods with parameters
method1(list, param1);
method2(list, param2, param3);
🎯 Memory Aid

Count the sc.nextLine() consume calls in your code. Every time you see nextInt() or nextDouble() followed by nextLine() on the next read, there should be a consume in between. If the count doesn't match, you have a bug.
