Section 01

The PRA Class Pattern

Every PRA question starts with "Create a class X with the following attributes." This is the most predictable part of the exam — the structure is ALWAYS the same. Learn it once, use it forever.

The Universal POJO Template

POJO stands for Plain Old Java Object. It is a class that only holds data — no complex logic, no database connections, just fields with getters and setters. Here is the exact template every PRA class follows:

POJOTemplate.java 
class ClassName {
    // Step 1: Private fields
    private type field1;
    private type field2;
    private type field3;

    // Step 2: Parameterized constructor
    public ClassName(type field1, type field2, type field3) {
        this.field1 = field1;
        this.field2 = field2;
        this.field3 = field3;
    }

    // Step 3: Getter and Setter for EACH field
    public type getField1() {
        return field1;
    }
    public void setField1(type field1) {
        this.field1 = field1;
    }

    public type getField2() {
        return field2;
    }
    public void setField2(type field2) {
        this.field2 = field2;
    }

    public type getField3() {
        return field3;
    }
    public void setField3(type field3) {
        this.field3 = field3;
    }
}
Line-by-line explanation
1
class ClassName {
Declares a new class. Replace "ClassName" with whatever the question asks (Investor, Book, etc.)
2-4
private type field1;
private means only this class can directly access the field. Others must use getters/setters. This is called encapsulation.
7-11
public ClassName(type field1, ...) { this.field1 = field1; }
The constructor — called when you create an object with new ClassName(...). this.field1 refers to the class field, field1 without "this" refers to the parameter.
14-16
public type getField1() { return field1; }
The getter — allows other classes to READ the private field. Always starts with "get" + field name (capitalized).
17-19
public void setField1(type field1) { this.field1 = field1; }
The setter — allows other classes to WRITE the private field. Returns void (nothing). Always starts with "set" + field name.
Analogy — Think of a Bank Locker

Your fields are like items inside a bank locker (private — only you can access directly). The getter is like requesting the bank to show you what is inside. The setter is like requesting the bank to put something new inside. The bank (class) controls access — nobody can just reach in and grab things.

Section 02

Field Types in PRA

PRA questions use a small set of data types. Here is every type you will encounter and when it appears.

Type Used For Example Declaration Getter Returns
int IDs, counts, age, rank, quantity private int investorId; int
String Names, categories, gender, country, genre private String name; String
double Money, scores, averages, prices, gains private double gains; double
boolean Active/inactive, yes/no flags (rare in PRA) private boolean isActive; boolean (getter uses is prefix)
List<T> Nested collections (composition) private List<Statistics> stats; List<Statistics>
ClassName Composition — one object inside another private Author author; Author
Key Point

Notice that PRA mostly uses just 3 types: int, String, and double. The last two types (List<T> and ClassName) only appear when the question involves composition — covered in detail on Page 05.

Section 03

Parameterized Constructor — Deep Dive

The constructor is called every time you create a new object. Understanding the this keyword here is critical.

Why Do We Use this?

Look at this constructor:

ThisKeyword.java 
class Investor {
    private int investorId;     // This is the CLASS FIELD
    private String name;        // This is the CLASS FIELD

    public Investor(int investorId, String name) {
        //    ↑ parameter name SAME as field name
        //    Java gets confused: which "investorId" do you mean?

        this.investorId = investorId;
        // ↑ "this." says "I mean the CLASS FIELD"
        //                  ↑ without "this" means "the PARAMETER"

        this.name = name;
    }
}

When the parameter name matches the field name (which it ALWAYS does in PRA), Java cannot tell which one you mean. The this keyword resolves this ambiguity:

this.investorId = investorId;
│                 │
│                 └── The VALUE coming in (parameter)
│
└── The FIELD inside the object (class variable)

Translation: "Set MY field to the value that was passed in"
Analogy

Imagine you are filling out a form. The form field (left side, this.name) says "Name:" and you write in the value someone told you (right side, name). this always points to "my own form" — the object being created.

What If You Forget this?

ForgotThis.java 
// WRONG — assigns parameter to itself, field stays null/0!
public Investor(int investorId, String name) {
    investorId = investorId; // Does NOTHING useful
    name = name;             // Does NOTHING useful
}

// After this constructor, investor.getInvestorId() returns 0
// and investor.getName() returns null. Your code breaks silently.
Warning

This mistake does NOT cause a compilation error. Your code compiles fine but produces wrong output. The fields remain at their default values (0 for int, null for String, 0.0 for double). Always use this. in the constructor.

Section 04

Getters and Setters

For every private field, you write one getter (to read the value) and one setter (to change the value). The naming convention is strict.

Naming Convention

Field Getter Setter
private int investorId; public int getInvestorId() public void setInvestorId(int investorId)
private String name; public String getName() public void setName(String name)
private double gains; public double getGains() public void setGains(double gains)
private boolean active; public boolean isActive() (note: is, not get) public void setActive(boolean active)
private List<Statistics> statistics; public List<Statistics> getStatistics() public void setStatistics(List<Statistics> statistics)

The Pattern

GetterSetterPattern.java 
// GETTER pattern:
// public [return-type] get[FieldName]() { return fieldName; }

public double getGains() {
    return gains;
}

// SETTER pattern:
// public void set[FieldName]([type] fieldName) { this.fieldName = fieldName; }

public void setGains(double gains) {
    this.gains = gains;
}
PRA Exam Tip

In PRA, you almost never use setters in your solution logic. You always create objects via the constructor and read values via getters. But you MUST still write the setters because the question asks for them, and hidden test cases may test that they exist.

Notice the Pattern

Every single class above follows the exact same structure: private fields, parameterized constructor with this, getters, setters. The only things that change are the class name, field names, and field types. Once your fingers know this pattern, writing a POJO class in PRA takes under 5 minutes.

Section 06

Speed Trick — Write the POJO First

A tactical tip that makes the rest of the exam easier.

PRA Speed Trick

Always write the POJO class FIRST. It is the easiest 5 minutes of your exam. While your fingers are busy typing the class (which requires zero thinking — it is pure mechanical work), your brain is processing the method logic in the background.

By the time you finish writing the class, you already have a mental plan for how the methods should work. This is called "productive autopilot" — your hands do easy work while your brain does hard work.

The Order of Typing

  1. Write import java.util.*; at the top. Do this FIRST so you never forget it.
  2. Write the POJO class(es). If there are two classes (Author + Book), write the simpler one first (Author), then the one that contains it (Book).
  3. Write the Solution class shell — class Solution { } with empty method signatures inside.
  4. Write main method — Scanner, input reading, method calls, output.
  5. Fill in method logic — now you know exactly what data you have and what you need to return.
Practice Exercise

Right now, without looking at the examples above, try writing a POJO class for a Student with these fields:

  • studentName (String)
  • marks (double)
  • department (String)

Time yourself. Your goal: under 3 minutes. If you can do it in under 3 minutes, the POJO section of PRA is locked in.
