Section 01

Which Collections Appear in PRA

Java has many collection types, but PRA only uses a small handful. Here is everything you need to know about which one to pick and when.

Collection PRA Frequency Maintains Order? Allows Duplicates? When to Use
ArrayList 90% of PRA Yes Yes Default choice. Always safe.
LinkedList Rare Yes Yes Sometimes used as parameter type. Works identically to ArrayList for PRA.
HashMap Occasional No Keys: No, Values: Yes When you need key-value pairs (e.g., name → score mapping).
HashSet Rare No No When you need unique values only (remove duplicates).
PRA Decision Rule

Use ArrayList unless the question specifically asks for something else. If you are unsure, ArrayList is always a safe choice. It maintains insertion order, allows duplicates, and works with Collection<T> parameters.

The Collection Hierarchy — What You Need

Java collections form a hierarchy. You do not need to memorize the entire thing, just this subset:

        Collection (interface)
        ├── List (interface)
        │   ├── ArrayList  (class) ← USE THIS
        │   └── LinkedList (class)
        ├── Set (interface)
        │   ├── HashSet    (class)
        │   └── LinkedHashSet (class) ← if you need unique + ordered
        └── Queue (interface)
            └── (not used in PRA)

        Map (separate interface — NOT a Collection)
        ├── HashMap       (class)
        └── LinkedHashMap (class) ← if you need key-value + ordered
Why This Matters

When a PRA method parameter says Collection<Investor>, it can accept ANY of these types — ArrayList, LinkedList, HashSet, etc. This is because they all implement the Collection interface. But when you CREATE a collection inside your method, you always create an ArrayList because it is the most straightforward.

Section 02

ArrayList — Your Best Friend

ArrayList is the collection you will use in almost every PRA question. Here is every operation you need to know.

Creating an ArrayList

CreateArrayList.java 
import java.util.*;

// Create an empty ArrayList of Investors
ArrayList<Investor> investors = new ArrayList<>();

// Create an empty ArrayList of Strings
ArrayList<String> names = new ArrayList<>();

// The <> (diamond operator) tells Java what TYPE of objects this list holds
// You CANNOT put an Investor in a list of Strings, or vice versa

Essential Operations

ArrayListOperations.java 
ArrayList<Investor> investors = new ArrayList<>();

// ===== ADDING ELEMENTS =====
// Add one item to the END of the list
investors.add(new Investor(1001, "Simon", -50000));
investors.add(new Investor(1002, "Raj", 30000));
investors.add(new Investor(1003, "Priya", 75000));
// List is now: [Simon, Raj, Priya]

// ===== SIZE =====
int count = investors.size(); // 3

// ===== GETTING BY INDEX =====
Investor first = investors.get(0);  // Simon (index starts at 0)
Investor last = investors.get(investors.size() - 1); // Priya

// ===== ITERATING (enhanced for loop — USE THIS IN PRA) =====
for (Investor inv : investors) {
    System.out.println(inv.getInvestorName() + ": " + inv.getGains());
}
// Output:
// Simon: -50000.0
// Raj: 30000.0
// Priya: 75000.0

// ===== CHECK EMPTY =====
if (investors.isEmpty()) {
    System.out.println("No investors");
}

// ===== REMOVE =====
investors.remove(0); // removes first element (Simon)
// List is now: [Raj, Priya]
Operations you will actually use in PRA
1
list.add(item)
Used in: Creating the input list in main(), and building result lists in filter methods. You will use this in EVERY PRA solution.
2
for (Type item : list) { }
Used in: Every method body. The enhanced for loop is the standard way to iterate in PRA. Never use a traditional for(i=0;...) loop unless you need the index.
3
list.size()
Used in: Checking if the list has items, calculating averages (dividing by size).
4
list.isEmpty()
Used in: Edge case handling — check if the result list is empty before printing.
Analogy — ArrayList is a Train

Think of ArrayList as a train with numbered compartments. Each compartment (index) holds one passenger (object). Passengers board in order (insertion order is maintained). You can walk through every compartment from front to back (for-each loop). You can check how many compartments are occupied (size()). You can look at a specific compartment number (get(index)).

Section 03

Collection as Parameter Type

PRA method signatures say Collection<Investor> — not ArrayList<Investor>. This confuses many students. Here is why and what it means for you.

Why Collection Instead of ArrayList?

Collection is an interface — a contract that says "I can add items, remove items, iterate, check size." ArrayList is a class that fulfills (implements) this contract.

When a method parameter says Collection<Investor>, it means: "I accept ANY type that is a Collection — ArrayList, LinkedList, HashSet, whatever." This is called polymorphism.

CollectionParameter.java 
// Method accepts Collection (interface)
public static double averageGains(Collection<Investor> investors) {
    // Inside, you use it exactly like an ArrayList
    // The for-each loop works on ANY Collection
    double sum = 0;
    for (Investor inv : investors) {
        sum += inv.getGains();
    }
    return sum / investors.size();
}

// In main, you pass an ArrayList (which IS a Collection)
ArrayList<Investor> list = new ArrayList<>();
list.add(new Investor(1, "Simon", 5000));
double avg = averageGains(list); // ArrayList works as Collection parameter
The Practical Takeaway

For PRA, the important thing is: it does not change how you write the method body. Whether the parameter is Collection<Investor> or ArrayList<Investor>, you use the for-each loop the same way. You just cannot call .get(index) on a generic Collection — but you never need to in PRA because for-each is always sufficient.

The Hierarchy — Visual

Collection
Interface (contract)
↓
List
Interface (ordered collection)
↓
ArrayList
Class (actual implementation)
An ArrayList IS a List, and a List IS a Collection.
So an ArrayList IS a Collection.
Section 04

Insertion Order

Some PRA questions say "The collection should maintain insertion order." This determines which collection type you must use.

What is Insertion Order?

Insertion order means: the order in which you add items is the order they stay in. If you add Simon first, then Raj, then Priya — when you iterate, you get Simon, Raj, Priya (not some random order).

Collection Maintains Insertion Order? Safe for PRA?
ArrayList Yes Always safe
LinkedList Yes Safe
LinkedHashSet Yes Safe (unique values only)
LinkedHashMap Yes Safe (key-value pairs)
HashMap NO Dangerous — order is random
HashSet NO Dangerous — order is random
TreeMap NO (sorted, not insertion) Sorts by key, not insertion order
PRA Exam Tip

When the question says "insertion order," just use ArrayList. It always works. Do not overthink it with LinkedHashSet or LinkedHashMap unless the question explicitly requires unique values or key-value pairs.

What Happens If You Use HashMap When Order Matters?

OrderProblem.java 
// You add items in this order: Simon, Raj, Priya
HashMap<String, Double> gains = new HashMap<>();
gains.put("Simon", -50000.0);
gains.put("Raj", 30000.0);
gains.put("Priya", 75000.0);

// When you iterate, you might get: Priya, Simon, Raj
// The order is UNPREDICTABLE — HashMap does not care about insertion order
// This will fail hidden test cases that expect a specific order
Section 05

Creating Collections from Scanner Input

This is the pattern you will use in the main method of every PRA solution. It reads N objects from Scanner input and builds an ArrayList.

The Universal Input Pattern

InputPattern.java 
Scanner sc = new Scanner(System.in);

// Step 1: Read how many objects to create
int n = sc.nextInt();

// Step 2: Create empty list
ArrayList<Investor> investors = new ArrayList<>();

// Step 3: Loop n times, reading each object's fields
for (int i = 0; i < n; i++) {
    int id = sc.nextInt();
    sc.nextLine();              // CONSUME NEWLINE after nextInt()!
    String name = sc.nextLine();
    double gains = sc.nextDouble();

    // Step 4: Create object and add to list
    investors.add(new Investor(id, name, gains));
}
Line-by-line explanation
1
int n = sc.nextInt();
The first line of input is ALWAYS the count of objects. Read it as an int.
2
ArrayList<Investor> investors = new ArrayList<>();
Create an empty list to hold the objects. It starts with size 0.
3
for (int i = 0; i < n; i++) {
Loop exactly n times — once for each object in the input.
4
sc.nextLine(); // after nextInt()
CRITICAL: consume the leftover newline. Without this, the next sc.nextLine() reads an empty string instead of the name.
5
investors.add(new Investor(id, name, gains));
Create a new Investor object with the values we just read, and immediately add it to the list. We do not need a separate variable for the object.

Input Order Matters

The order in which you read fields must exactly match the input format described in the question. If the question says:

Typical Input Format

Line 1: Number of investors (N)
For each investor:
  Line 1: investorId (int)
  Line 2: investorName (String)
  Line 3: gains (double)

Then your Scanner reads MUST be in this order: nextInt() → nextLine() → nextLine() → nextDouble().

Warning — The #1 Input Bug

If you read fields in the wrong order, every subsequent read gets the wrong data. Your objects will have names in the ID field and numbers in the name field. The code compiles fine but produces completely wrong output.

Always read the input format section of the question FIRST.

Section 06

Common Collection Mistakes

These are the collection-specific bugs that trip up PRA candidates. Each one includes the mistake, why it happens, and the fix.

Mistake 1: Forgetting the Import

Warning — Compilation Error

Without import java.util.*;, Java does not know what ArrayList, Collection, or Scanner are. Your code will not compile.

AlwaysImport.java 
// FIRST LINE of every PRA solution — no exceptions
import java.util.*;

Mistake 2: Using Raw Types

Warning — Type Safety Issue

A raw type means using ArrayList without the <Type> specification. While it compiles, it removes type safety and can cause ClassCastException at runtime.

RawTypes.java 
// WRONG — raw type, no type safety
ArrayList list = new ArrayList();
list.add(new Investor(1, "Simon", 5000));
list.add("This is a string"); // No error! But it will crash later

// CORRECT — generic type specified
ArrayList<Investor> list = new ArrayList<>();
list.add(new Investor(1, "Simon", 5000));
// list.add("This is a string"); // Compiler ERROR — caught early

Mistake 3: Modifying a Collection While Iterating

Warning — ConcurrentModificationException

If you try to add or remove elements from a list WHILE looping through it with a for-each loop, Java throws a ConcurrentModificationException.

ConcurrentMod.java 
// WRONG — removing from list while iterating
for (Investor inv : investors) {
    if (inv.getGains() < 0) {
        investors.remove(inv); // CRASH! ConcurrentModificationException
    }
}

// CORRECT — build a SEPARATE list of results
ArrayList<Investor> result = new ArrayList<>();
for (Investor inv : investors) {
    if (inv.getGains() < 0) {
        result.add(inv); // Add to a DIFFERENT list
    }
}
// Now "result" has the filtered items, original list is untouched

Mistake 4: Not Checking If Collection is Empty

Warning — Logic Errors

If your filter method returns an empty list, and your main method tries to print "No results found" but the question does not ask for that message, you will fail test cases. Conversely, if it DOES expect a message and you do not check, you fail too.

Fix: Always read what the question says about empty results. Some say "print nothing", some say "print a message", some say "the test cases guarantee at least one match."
