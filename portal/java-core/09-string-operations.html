equalsIgnoreCase — The PRA Essential

If there's one string method you MUST know for PRA, it's this one. Almost every PRA question involves comparing strings, and almost every one says "case-insensitive." Using equals() instead of equalsIgnoreCase() will fail test cases silently — your logic is right but the answer is wrong.

WRONG — Case Sensitive
if (genre.equals("Crime"))
// Only matches "Crime"
// Fails on "crime", "CRIME", "CrImE"
CORRECT — Case Insensitive
if (genre.equalsIgnoreCase("Crime"))
// Matches "Crime", "crime", "CRIME"
// Matches "CrImE", "cRiMe" - any case

How It Works

equalsIgnoreCase() converts both strings to the same case internally before comparing. It doesn't modify your original string — it just ignores case during the comparison.

equalsIgnoreCase Examples 
"Crime".equalsIgnoreCase("crime")   // true
"Crime".equalsIgnoreCase("CRIME")   // true
"Crime".equalsIgnoreCase("CrImE")   // true
"Crime".equalsIgnoreCase("Drama")   // false
"Crime".equals("crime")             // FALSE! Case sensitive!
🎯 PRA Rule

Always use equalsIgnoreCase() for every string comparison in PRA. There is zero downside. You'll never lose marks for being case-insensitive, but you WILL lose marks if a test case uses "crime" and your code only matches "Crime".

⚠ Never Use == for Strings

== compares memory addresses, not string content. Two strings with the same text can be at different memory addresses. Always use .equals() or .equalsIgnoreCase() for string comparison in Java.

  • "abc" == "abc" — might work (string pool), but unreliable
  • new String("abc") == "abc" — false! Different objects
  • "abc".equals("abc") — always true. Use this.
Output

String Concatenation for Output

PRA expects exact output format. You need to build output strings that match the expected output character-by-character. The + operator joins strings together.

Concatenation with +

String Concatenation Examples 
// Simple concatenation
System.out.println("Name: " + investor.getName());
// Output: Name: John Smith

// Multiple fields
System.out.println("AuthorName: " + book.getAuthor().getAuthorName()
    + ", Title: " + book.getTitle());
// Output: AuthorName: J.K. Rowling, Title: Harry Potter

// Number with unit
System.out.println(area + " sq.m");
// Output: 50 sq.m

System.out.println(perimeter + " m");
// Output: 30 m

How + Works with Different Types

Expression Result Why
"Hello" + " World" "Hello World" String + String = concatenation
"Price: " + 500 "Price: 500" String + int = int becomes string
"Value: " + 3.14 "Value: 3.14" String + double = double becomes string
50 + " sq.m" "50 sq.m" int + String = int becomes string
5 + 3 + " items" "8 items" 5+3 = 8 first (both ints), then 8 + " items"
"items: " + 5 + 3 "items: 53" "items: " + 5 = "items: 5", then + 3 = "items: 53"
💡 Order Matters

Java evaluates + from left to right. If it starts with a String, everything after becomes concatenation. If it starts with numbers, those numbers add normally until a String appears. The last two rows in the table above show this difference. If you need math inside a print, use parentheses: "Sum: " + (5 + 3) gives "Sum: 8".

Conversion

Parsing Strings to Numbers

Sometimes you receive a value as a String but need to use it as a number for calculations. Java provides parsing methods to convert between types.

Parsing Methods 
// String to int
int num = Integer.parseInt("42");          // num = 42

// String to double
double val = Double.parseDouble("3.14");   // val = 3.14

// Number to String
String s1 = String.valueOf(42);            // s1 = "42"
String s2 = Integer.toString(42);          // s2 = "42"
String s3 = "" + 42;                       // s3 = "42" (quick trick)
⚠ NumberFormatException

If you try to parse a non-numeric string, Java crashes with NumberFormatException:

  • Integer.parseInt("hello") — crashes
  • Integer.parseInt("3.14") — crashes (3.14 is not an integer)
  • Double.parseDouble("3.14") — works fine

In PRA, you rarely need to parse manually since Scanner handles type conversion. But it's good to know.

The Universal Pattern

  object.getStringField().equalsIgnoreCase(parameterFromInput)

  Examples:
  book.getGenre().equalsIgnoreCase(searchGenre)
  stat.getLeagueName().equalsIgnoreCase(leagueName)
  stat.getCountryName().equalsIgnoreCase(countryName)
  investor.getCategory().equalsIgnoreCase(category)

  Structure is ALWAYS the same:
  [object].[getter]().equalsIgnoreCase([variable])
Every PRA string comparison follows this exact pattern
Critical

Output Format Matching

PRA is STRICT about output format. HackerRank compares your output character-by-character against the expected output. One extra space, one missing comma, one wrong word — and your test case fails. Even if your logic is 100% correct.

Error Messages — Must Be Exact

Error Messages (copy exactly from question) 
// Different questions use different error messages
System.out.println("No such Investors found");    // one wording
System.out.println("No Investors found");          // different wording!
System.out.println("No Cricketer Found");          // yet another
System.out.println("No Statistics are available With the Country : " + countryName);
// Note: "With the Country : " has specific capitalization and spacing!
⚠ The #1 Reason for Wrong Answers

Output format mismatch is the number one reason people get wrong answers on HackerRank despite having correct logic. When you read the question:

  1. Find the "Output Format" or "Expected Output" section
  2. Copy the exact format — every comma, space, colon, and word
  3. Check capitalization: "No Cricketer Found" is not the same as "No cricketer found"
  4. Check spacing: "Country : " (with spaces around colon) is not "Country:"
Gotcha

int vs double in Output

This subtle difference has caught many people. When you print a number, Java formats it differently depending on whether it's an int or a double. The expected output tells you which type to use.

int vs double Output 
// int output
int area = 50;
System.out.println(area + " sq.m");       // prints: 50 sq.m

// double output
double area2 = 50.0;
System.out.println(area2 + " sq.m");      // prints: 50.0 sq.m
                                           //         ^^^^
                                           //         Note the .0!
int

int x = 50;

Prints as: 50

No decimal point

Use when expected output shows: 50 sq.m

double

double x = 50.0;

Prints as: 50.0

Always has decimal point

Use when expected output shows: 50.0 sq.m

🎯 PRA Tip

Check whether the expected output shows 50 or 50.0. This tells you whether the field should be int or double. If the expected output shows 50.0 and you use int, your output will be 50 (no .0) and HackerRank will mark it wrong.

Common Number Output Scenarios

Expected Output Use Type Code
50 sq.m int int area = length * breadth;
50.0 sq.m double double area = length * breadth;
3500.5 double System.out.println(gains);
1001 int System.out.println(id);
479.992 double System.out.println(discountedPrice);
🏃 Quick Check

If the question says "Calculate the area of a rectangle" and the expected output shows 50 sq.m, use int for both length, breadth, and area. If it shows 50.0 sq.m, use double. This one decision affects whether your entire solution passes or fails.
