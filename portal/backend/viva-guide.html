Overview

Architecture Overview

The application has two entry points — Servlets (for JSP web pages) and REST Controllers (for JSON APIs). Both use the same Services and Repositories underneath.

 TWO ENTRY POINTS, ONE BACKEND:

 Browser → Servlets (JSP Pages) → Services → Repositories → H2 Database
 Browser → REST Controllers (JSON) → Services → Repositories → H2 Database

 Servlets return HTML (rendered by JSP).
 REST Controllers return JSON (consumed by frontend JS or Postman).
Both paths share the Service and Repository layers

Think About It

Each package is a layer. Data flows top to bottom: Controller/Servlet → Service → Repository → Database. A controller NEVER talks to the database directly. A repository NEVER has business logic. This is Separation of Concerns.

Layer 2

DAOs — Sprint 2 Approach

DAO = Data Access Object. This was our Sprint 2 approach — writing SQL manually using JdbcTemplate. In Sprint 3, we replaced DAOs with JPA Repositories (much less code).

Why Learn DAOs?

DAOs teach you how database access actually works under the hood. JPA Repositories auto-generate SQL, but someone at your viva might ask: "What if you didn't have JPA?" DAOs are the answer.

Key Concepts

ConceptWhat It Does
@RepositoryMarks the class as a data access component. Spring manages it as a bean.
JdbcTemplateSpring's JDBC helper. Simplifies database calls from 7-8 lines of raw JDBC to 1 line.
RowMapperInterface that converts one SQL ResultSet row into a Java object. You define how each column maps to each field.
Analogy

Without JdbcTemplate: You personally go to the warehouse, find the shelf, pick up the box, carry it to the counter, open it, hand the item to the customer. With JdbcTemplate: You tell the warehouse robot "get item X" and it does everything. Same result, way less work.

Layer 2

UserDAO Example

This is what the DAO looks like. Note the RowMapper that manually maps each column to a Java field. DAOs are now commented out (@Repository removed) but kept as Sprint 2 reference.

src/main/java/com/nimbus/dao/UserDAO.java (Sprint 2 — now commented out) 
// @Repository  <-- REMOVED in Sprint 3 (replaced by JPA Repository)
public class UserDAO {

    private final JdbcTemplate jdbc;

    public UserDAO(JdbcTemplate jdbc) {
        this.jdbc = jdbc;
    }

    // RowMapper: converts SQL ResultSet row → User object
    private final RowMapper<User> userMapper = (rs, rowNum) -> {
        User u = new User();
        u.setUserId(rs.getString("user_id"));
        u.setFirstName(rs.getString("first_name"));
        u.setLastName(rs.getString("last_name"));
        u.setFullName(rs.getString("full_name"));
        u.setEmail(rs.getString("email"));
        u.setMobile(rs.getString("mobile"));
        u.setPassword(rs.getString("password"));
        u.setRole(rs.getString("role"));
        u.setCreatedAt(rs.getTimestamp("created_at").toLocalDateTime());
        return u;
    };

    // Find user by email
    public User findByEmail(String email) {
        String sql = "SELECT * FROM tbl_user_profile WHERE email = ?";
        List<User> users = jdbc.query(sql, userMapper, email);
        return users.isEmpty() ? null : users.get(0);
    }

    // Save a new user
    public void save(User user) {
        String sql = "INSERT INTO tbl_user_profile (user_id, first_name, last_name, "
                   + "full_name, email, mobile, password, role, created_at) "
                   + "VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?)";
        jdbc.update(sql, user.getUserId(), user.getFirstName(),
                    user.getLastName(), user.getFullName(), user.getEmail(),
                    user.getMobile(), user.getPassword(), user.getRole(),
                    user.getCreatedAt());
    }
}
Important for Viva

DAOs were our Sprint 2 approach. In Sprint 3, we replaced them with JPA Repositories. The @Repository annotation is removed from all DAO classes so Spring doesn't load them. They're kept in the codebase as learning reference.

Layer 3

JPA Repositories — Sprint 3 Approach

JpaRepository is Spring Data's magic. You write an interface, Spring auto-generates the SQL. No RowMapper. No SQL strings. Just method names.

src/main/java/com/nimbus/repository/UserRepository.java 
package com.nimbus.repository;

import com.nimbus.model.User;
import org.springframework.data.jpa.repository.JpaRepository;
import java.util.List;
import java.util.Optional;

public interface UserRepository extends JpaRepository<User, String> {

    Optional<User> findByEmail(String email);

    Optional<User> findByUserIdAndPassword(String userId, String password);

    List<User> findByRole(String role);

    boolean existsByEmail(String email);
}
// That's it. No SQL. No RowMapper. Spring generates everything.
How Does Spring Know the SQL?

Spring reads the method name and generates SQL from it. findByEmail(String email) becomes SELECT * FROM tbl_user_profile WHERE email = ?. findByUserIdAndPassword() becomes ... WHERE user_id = ? AND password = ?. The method name IS the query.

Layer 4

UserService

Handles registration, login, smart user ID generation, and user lookup.

src/main/java/com/nimbus/service/UserService.java (key methods) 
@Service
public class UserService {

    private final UserRepository userRepository;

    // Constructor injection (preferred over @Autowired on field)
    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    // Register a new customer
    public User register(User user) {
        // Validation: check if email already exists
        if (userRepository.existsByEmail(user.getEmail())) {
            throw new RuntimeException("Email already registered");
        }

        // Generate smart user ID: YYMMDD-Initials-Sequence
        user.setUserId(generateUserId(user));
        user.setRole("customer");
        user.setFullName(buildFullName(user));
        user.setCreatedAt(LocalDateTime.now());

        return userRepository.save(user);
    }

    // Smart User ID: 260401-AK01 (YYMMDD + initials + sequence)
    private String generateUserId(User user) {
        String date = LocalDateTime.now()
            .format(DateTimeFormatter.ofPattern("yyMMdd"));
        String initials = ("" + user.getFirstName().charAt(0)
                             + user.getLastName().charAt(0)).toUpperCase();
        long count = userRepository.count() + 1;
        return date + "-" + initials + String.format("%02d", count);
    }

    // Login validation
    public User validateLogin(String userId, String password) {
        return userRepository.findByUserIdAndPassword(userId, password)
                .orElse(null);  // Returns null if no match
    }

    // Register admin/officer/support (admin-only action)
    public User registerAdmin(User user, String role) {
        user.setUserId(generateUserId(user));
        user.setRole(role);
        user.setFullName(buildFullName(user));
        user.setCreatedAt(LocalDateTime.now());
        return userRepository.save(user);
    }
}
Layer 4

BookingService

Handles booking creation with a pricing engine, parcel tracking with access control, and delivery status updates.

src/main/java/com/nimbus/service/BookingService.java (pricing engine) 
@Service
public class BookingService {

    private final BookingRepository bookingRepository;
    private final UserRepository userRepository;

    public BookingService(BookingRepository bookingRepository,
                          UserRepository userRepository) {
        this.bookingRepository = bookingRepository;
        this.userRepository = userRepository;
    }

    // Create booking with pricing engine
    public Booking createBooking(Booking booking) {
        booking.setBookingId(generateBookingId());
        booking.setParStatus("Booked");
        booking.setCreatedAt(LocalDateTime.now());
        booking.setUpdatedAt(LocalDateTime.now());

        // PRICING ENGINE: weight × rate + packaging cost
        double cost = calculateCost(booking);
        booking.setParServiceCost(cost);

        return bookingRepository.save(booking);
    }

    private double calculateCost(Booking booking) {
        double weightKg = booking.getParWeightGram() / 1000.0;

        // Rate per kg based on delivery type
        double ratePerKg;
        switch (booking.getParDeliveryType().toLowerCase()) {
            case "express":   ratePerKg = 80.0;  break;
            case "overnight": ratePerKg = 120.0; break;
            default:          ratePerKg = 50.0;  break; // standard
        }

        double baseCost = weightKg * ratePerKg;

        // Packaging premium
        if ("fragile".equalsIgnoreCase(booking.getParPackingPreference())) {
            baseCost += 50.0;
        }

        return Math.round(baseCost * 100.0) / 100.0;
    }

    // Track with access control (customer sees own, officer sees all)
    public Booking trackBooking(String bookingId, String userId, String role) {
        Booking booking = bookingRepository.findById(bookingId)
                .orElseThrow(() -> new RuntimeException("Booking not found"));

        // Access control
        if ("customer".equals(role) && !booking.getUserId().equals(userId)) {
            throw new RuntimeException("Access denied");
        }
        return booking;
    }

    // Update delivery status (officer only)
    public Booking updateDeliveryStatus(String bookingId, String newStatus) {
        List<String> validStatuses = List.of(
            "Booked", "Picked Up", "In Transit", "Out for Delivery", "Delivered"
        );
        if (!validStatuses.contains(newStatus)) {
            throw new RuntimeException("Invalid status: " + newStatus);
        }

        Booking booking = bookingRepository.findById(bookingId)
                .orElseThrow(() -> new RuntimeException("Booking not found"));
        booking.setParStatus(newStatus);
        booking.setUpdatedAt(LocalDateTime.now());
        return bookingRepository.save(booking);
    }
}

        return issueRepository.findByAssignedTo(staffUserId);
    }

Layer 5

Servlets — Handling HTTP Requests

A Servlet is a Java class that handles HTTP requests. It extends HttpServlet, overrides doGet() and doPost(), and is managed by Tomcat (not Spring).

Servlet Lifecycle

init()Called once at startup
→
service()Every request (routes to doGet/doPost)
→
destroy()Server shutdown
Servlet lifecycle: created once, serves many requests, destroyed once

How Servlets Are Registered

Servlets are not Spring beans — they're managed by Tomcat. We register them in ServletConfig.java using @Bean methods that return ServletRegistrationBean objects with URL mappings.

Why Can't Servlets Use @Autowired?

Servlets are created by Tomcat, not by Spring. Spring's @Autowired only works on objects that Spring creates. To get Spring beans inside a Servlet, we use WebApplicationContextUtils.getRequiredWebApplicationContext(getServletContext()) to manually look up the Spring context.

Forward vs Redirect

Forward (Server-Side)

request.getRequestDispatcher("/WEB-INF/jsp/login.jsp").forward(req, res)

Same request object

URL in browser does NOT change

Use for: showing a page with data

Redirect (Browser-Side)

response.sendRedirect("/servlet/dashboard")

New request created by browser

URL in browser CHANGES

Use for: after POST (PRG pattern)

Layer 5

LoginServlet — Complete Code

The LoginServlet handles the login flow: GET shows the form (JSP), POST validates credentials and creates a session.

src/main/java/com/nimbus/servlet/LoginServlet.java 
package com.nimbus.servlet;

import com.nimbus.model.User;
import com.nimbus.service.UserService;
import jakarta.servlet.ServletException;
import jakarta.servlet.http.*;
import org.springframework.web.context.WebApplicationContext;
import org.springframework.web.context.support.WebApplicationContextUtils;
import java.io.IOException;

public class LoginServlet extends HttpServlet {
    private static final long serialVersionUID = 1L;

    private UserService userService;

    @Override
    public void init() throws ServletException {
        // Get Spring context manually (Servlets aren't Spring beans)
        WebApplicationContext ctx = WebApplicationContextUtils
                .getRequiredWebApplicationContext(getServletContext());
        this.userService = ctx.getBean(UserService.class);
    }

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {
        // Show the login page
        request.getRequestDispatcher("/WEB-INF/jsp/login.jsp")
               .forward(request, response);
    }

    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {
        String userId = request.getParameter("userId");
        String password = request.getParameter("password");

        User user = userService.validateLogin(userId, password);

        if (user != null) {
            // Login success → create session → redirect to dashboard
            HttpSession session = request.getSession();
            session.setAttribute("userId", user.getUserId());
            session.setAttribute("fullName", user.getFullName());
            session.setAttribute("role", user.getRole());
            response.sendRedirect("/servlet/dashboard");
        } else {
            // Login failed → show error on login page
            request.setAttribute("error", "Invalid User ID or Password");
            request.getRequestDispatcher("/WEB-INF/jsp/login.jsp")
                   .forward(request, response);
        }
    }
}
Login Flow Summary

GET /servlet/login → Shows login.jsp form.
POST /servlet/login → Validates credentials → If valid: create HttpSession + redirect to dashboard. If invalid: forward back to login.jsp with error message.

@RestController
@RequestMapping("/api/auth")
public class AuthController {

    private final UserService userService;

    public AuthController(UserService userService) {
        this.userService = userService;
    }

    @PostMapping("/login")
    public ResponseEntity<ApiResponse> login(@RequestBody LoginRequest req) {
        User user = userService.validateLogin(req.getUserId(), req.getPassword());

        if (user != null) {
            LoginResponse loginResp = new LoginResponse(true, "Login successful",
                new LoginResponse.UserDTO(user.getUserId(), user.getFullName(),
                                          user.getEmail(), user.getRole()));
            return ResponseEntity.ok(new ApiResponse(true, "Login successful", loginResp));
        }
        return ResponseEntity.status(401)
                .body(new ApiResponse(false, "Invalid credentials", null));
    }

    @PostMapping("/register")
    public ResponseEntity<ApiResponse> register(@RequestBody RegisterRequest req) {
        User user = new User();
        user.setFirstName(req.getFirstName());
        user.setLastName(req.getLastName());
        user.setEmail(req.getEmail());
        user.setMobile(req.getMobile());
        user.setPassword(req.getPassword());
        // ... set other fields

        User saved = userService.register(user);
        return ResponseEntity.ok(new ApiResponse(true, "Registration successful", saved));
    }

    @GetMapping("/users")
    public ResponseEntity<ApiResponse> getAllUsers() {
        return ResponseEntity.ok(
            new ApiResponse(true, "Users retrieved", userService.getAllUsers()));
    }

    @GetMapping("/user/{userId}")
    public ResponseEntity<ApiResponse> getUser(@PathVariable String userId) {
        User user = userService.getUserById(userId);
        if (user == null) {
            return ResponseEntity.status(404)
                    .body(new ApiResponse(false, "User not found", null));
        }
        return ResponseEntity.ok(new ApiResponse(true, "User found", user));
    }
}

DTOs & ApiResponse

DTOs (Data Transfer Objects) define the shape of API requests and responses. They protect internal model structure and strip sensitive data like passwords.

ApiResponse.java — consistent wrapper for all API responses 
public class ApiResponse {
    private boolean success;
    private String message;
    private Object data;

    // Every API returns: { "success": true/false, "message": "...", "data": {...} }
    // This gives a CONSISTENT structure to all responses.
}

// LoginResponse has a nested UserDTO that strips the password:
public class LoginResponse {
    private boolean success;
    private String message;
    private UserDTO user;

    public static class UserDTO {
        private String userId;
        private String fullName;
        private String email;
        private String role;
        // NO password field — never expose passwords in API responses!
    }
}
Why DTOs?

Without DTOs, your API would return the full User object including the password. DTOs let you control exactly what data leaves your server. LoginResponse.UserDTO has userId, fullName, email, role — but NO password.

CORS Configuration

CORS (Cross-Origin Resource Sharing) allows the frontend running on a different port to call our API. Without it, the browser blocks the request.

src/main/java/com/nimbus/config/CorsConfig.java 
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")        // Apply to all API endpoints
                .allowedOrigins("*")           // Allow any frontend origin
                .allowedMethods("GET", "POST", "PUT", "DELETE")
                .allowedHeaders("*");
    }
}
// Without this, Chrome blocks: "Access to XMLHttpRequest has been blocked by CORS policy"
Real Experience

Errors We Hit & How We Fixed Them

These are real errors we encountered while building Nimbus. Viva panels love asking "what problems did you face?" — these are your answers.

Error 1: "JpaRepository cannot be resolved to a type"

When: Creating Repository interfaces in Sprint 3.

Cause: Missing spring-boot-starter-data-jpa dependency in pom.xml.

Fix: Added the dependency to pom.xml, then ran Maven Update (Right-click project → Maven → Update Project, or Alt+F5 in Eclipse).

Error 2: "The method trackBooking() is undefined for BookingService"

When: Sprint 3 Servlets tried calling methods that didn't exist yet.

Cause: The Sprint 2 BookingService only had basic CRUD. Sprint 3 Servlets needed trackBooking(), updateDeliveryStatus(), getHistory().

Fix: Upgraded BookingService to Sprint 3 version with all the new methods using Repositories.

Error 3: "The method getFullName() is undefined for type User"

When: JSP pages tried calling user.getFullName().

Cause: Sprint 2 User model had getName() but Sprint 3 Servlets expected getFullName().

Fix: Upgraded User.java to Sprint 3 version with firstName, middleName, lastName, fullName and all JPA annotations.

Warning 4: "serialVersionUID" in Servlet classes

When: Eclipse showed yellow warning on all Servlet classes.

Cause: Servlets implement Serializable but don't declare serialVersionUID.

Fix: This is a warning, not an error. Does not affect running. Can ignore or add private static final long serialVersionUID = 1L;

Error 5: "Port 8080 was already in use"

When: Trying to run the app when a previous instance was still running.

Cause: Previous run didn't stop properly. Port 8080 is still occupied.

Fix: Stop it from Eclipse (click the red square button in Console). Or from terminal: taskkill /PID <pid> /F. Or change port in application.properties: server.port=8081.

Error 6: "Table TBL_USER_PROFILE not found"

When: App starts, Hibernate tries to query before schema.sql runs.

Cause 1: schema.sql wasn't executing before JPA initialized.

Fix 1: Added spring.jpa.defer-datasource-initialization=true to application.properties.

Cause 2: Hibernate converts table names to lowercase (tbl_UserProfile → tbl_user_profile) but schema.sql used mixed case.

Fix 2: Changed schema.sql to use lowercase table names matching Hibernate's naming convention.

Error 7: DAO compile errors after model upgrade

When: After upgrading User.java to Sprint 3 with new field names.

Cause: DAOs referenced old model methods (setName(), setPhone()) that no longer exist in the upgraded model.

Fix: Commented out @Repository on all DAOs so Spring doesn't load them. Services now use JPA Repositories. DAOs kept in codebase as Sprint 2 reference.

Priority: Learn These First

These were shared by someone who already gave the viva. Focus on these before anything else.

Q: What is a Class Loader?

Class Loader is the part of JVM that loads .class files (compiled Java bytecode) into memory at runtime. There are 3 types: Bootstrap (loads core Java classes like java.lang), Extension (loads classes from ext directory), and Application (loads your project classes from classpath). When you run NimbusApplication.java, the Application ClassLoader loads all your com.nimbus.* classes.

Q: What are the Four Pillars of OOP?

1. Encapsulation: Private fields + public getters/setters. Our User.java — fields are private, accessed only via getUserId(), setEmail(), etc.
2. Inheritance: A class extends another. Our repositories extend JpaRepository to inherit save(), findAll(), delete() methods.
3. Polymorphism: Same method behaves differently. doGet() in LoginServlet shows a form, doGet() in DashboardServlet shows a dashboard. Same method name, different behavior.
4. Abstraction: Hide complexity, show only what's needed. JpaRepository hides all SQL — you just call findByEmail() and it works.

Q: What is Servlet Lifecycle?

init() → Called once when the servlet is first loaded. Tomcat creates the servlet instance.
service() → Called on every HTTP request. Routes to doGet() for GET requests and doPost() for POST requests. You override doGet/doPost, not service().
destroy() → Called once when the server shuts down. Cleanup resources.
In Nimbus: When you hit /servlet/login, Tomcat calls LoginServlet.doGet() to show the form. When you submit, it calls doPost() to validate credentials.

Q: What are Types of Joins? How are they different?

INNER JOIN: Returns only matching rows from both tables. SELECT * FROM tbl_booking b INNER JOIN tbl_user_profile u ON b.user_id = u.user_id — only bookings that have a valid user.
LEFT JOIN: All rows from left table + matching from right. Shows all users even if they have no bookings (bookings columns will be NULL).
RIGHT JOIN: All rows from right table + matching from left. Opposite of LEFT JOIN.
FULL OUTER JOIN: All rows from both tables. NULLs where no match.
CROSS JOIN: Every row of table A combined with every row of table B (cartesian product). Rarely used.

Q: What are Types of Polymorphism?

1. Compile-time (Static) — Method Overloading: Same method name, different parameters. Example: calculatePrice(int weight) and calculatePrice(int weight, String type). Compiler decides which to call based on arguments.
2. Runtime (Dynamic) — Method Overriding: Subclass redefines a parent method. Example: LoginServlet and BookingServlet both override doGet() from HttpServlet. JVM decides which to call at runtime based on the actual object type.

Q: What is @Controller in Spring Boot?

@Controller marks a class as a web controller that returns HTML views (JSP pages). It works with ViewResolver to find JSP files.
@RestController = @Controller + @ResponseBody. It returns JSON data directly, not views.
In Nimbus: Our Servlets handle the JSP views (equivalent to @Controller). Our AuthController, BookingController use @RestController to return JSON for the API.

Q: What is @ComponentScan?

@ComponentScan tells Spring: "Search this package and all sub-packages for classes with @Component, @Service, @Repository, @Controller, @RestController annotations and register them as Spring beans."
In Nimbus: @SpringBootApplication on NimbusApplication.java already includes @ComponentScan. Since NimbusApplication is in com.nimbus, Spring automatically scans all sub-packages: com.nimbus.service, com.nimbus.controller, com.nimbus.repository, etc. You don't need to declare it separately.

Q: What is IoC (Inversion of Control)?

IoC means you don't create objects — the framework creates them for you. Normally you write UserService service = new UserService(). With Spring IoC, Spring creates UserService and manages its lifecycle.
The "inversion" is: instead of YOUR code controlling object creation, the FRAMEWORK controls it. The Spring IoC Container reads your annotations (@Service, @Repository, etc.) and creates all the beans at startup. This is the foundation of Spring.

Q: What is Dependency Injection?

DI is how IoC works in practice. Instead of a class creating its own dependencies, Spring injects them.
In Nimbus: AuthController needs UserService. Instead of writing new UserService(), we declare it in the constructor: public AuthController(UserService userService). Spring sees this, finds the UserService bean it already created, and passes it in. This makes code testable and loosely coupled.

Q: Why is Spring Boot better than Spring?

Spring requires tons of XML configuration — web.xml, applicationContext.xml, dispatcher-servlet.xml. You manually configure every bean, every URL mapping, every dependency.
Spring Boot = Spring + Auto-Configuration + Embedded Server. Zero XML. @SpringBootApplication does everything. Tomcat is embedded — no separate server install. application.properties replaces hundreds of lines of XML.
In Nimbus: We wrote ZERO XML files. One annotation (@SpringBootApplication) and everything works. That's Spring Boot's value.

Q: Primary Key vs Foreign Key difference?

Primary Key: Uniquely identifies each row in a table. Cannot be NULL. Cannot be duplicate. One per table.
In Nimbus: user_id in tbl_user_profile, booking_id in tbl_booking, issue_id in tbl_issue.

Foreign Key: A column that references a primary key in another table. Creates a relationship between tables. Can be NULL. Can be duplicate.
In Nimbus: user_id in tbl_booking is a foreign key → references user_id in tbl_user_profile. This means every booking must belong to a valid user. If you try to insert a booking with a user_id that doesn't exist, the database rejects it.

Q: What is JPA and Hibernate?

JPA (Java Persistence API) is a specification — it defines the rules for mapping Java objects to database tables. It defines annotations like @Entity, @Table, @Id, @Column. JPA is just the interface — it doesn't do the actual work.
Hibernate is the implementation — it's the library that actually does the work. It generates SQL, manages connections, handles caching. Think of JPA as the blueprint and Hibernate as the builder.
In Nimbus: We use JPA annotations (@Entity, @Table) on our models. Spring Boot auto-configures Hibernate as the JPA provider. When we call userRepository.save(user), Hibernate generates the INSERT SQL and executes it.

Core Java Q&A

Quick answers for core Java questions. Each answer is 2-3 sentences — enough for a viva.

Q: What is a POJO?

A: Plain Old Java Object. A class with private fields, public getters/setters, and no framework dependency. Our User.java, Booking.java, and Issue.java are POJOs annotated with JPA.

Q: What is Encapsulation?

A: Fields are private, accessed only via getters and setters. This protects data integrity — you can add validation inside a setter. Every model class in Nimbus uses encapsulation.

Q: What are Java Editions?

A: Java SE = core language (our models, collections, streams). Java EE = enterprise features (Servlets, JSP, JPA). Spring Boot simplifies Java EE so you don't need a full application server.

Q: Abstract Class vs Interface?

A: Abstract class can have method bodies + fields, supports single inheritance only. Interface is a pure contract (method signatures only), supports multiple inheritance. Our UserRepository extends JpaRepository (an interface).

Q: What are Class Relationships?

A: Booking has a userId field (Association with User). Issue has userId + bookingId. This is composition/aggregation — one object references another by ID rather than containing it directly.

Spring Q&A

Q: What is Spring Boot?

A: A framework that simplifies Java web development. It auto-configures everything — database connections, server setup, dependency injection. Comes with an embedded Tomcat server so you don't install one separately.

Q: What is Dependency Injection?

A: Instead of creating objects with new, Spring creates them and injects them where needed. Our UserService gets UserRepository injected via its constructor. Spring manages the lifecycle of all these objects (called "beans").

Q: What are Spring Stereotypes?

A: Annotations that register classes as Spring beans: @Component (generic), @Service (business logic), @Repository (data access), @Controller (web endpoints). They all do the same thing internally but communicate the class's purpose.

Q: What is @Autowired?

A: Tells Spring to inject a dependency. With constructor injection (our approach), it's implicit — you don't even need the @Autowired annotation. Spring sees the constructor parameter and injects the matching bean.

Q: What is @Configuration?

A: Marks a class as a source of bean definitions. Contains @Bean methods that return objects Spring should manage. Our ServletConfig (registers Servlets) and CorsConfig (allows cross-origin requests) use it.

Q: What is the 3-tier architecture?

A: Controller → Service → Repository → Database. Each layer talks only to the one below it. Controllers handle HTTP, Services have business logic, Repositories handle data. This is Separation of Concerns.

Servlets & JSP Q&A

Q: What is a Servlet?

A: A Java class that handles HTTP requests. It extends HttpServlet and overrides doGet() and doPost(). Tomcat (the web server) invokes these methods when a matching URL is requested.

Q: What is the Servlet lifecycle?

A: init() runs once at startup. service() runs for every request and routes to doGet()/doPost(). destroy() runs once when the server shuts down. The Servlet object is created once and reused for all requests.

Q: What is JSP?

A: JavaServer Pages — HTML files with embedded Java code. The server compiles JSP into a Servlet behind the scenes. Uses ${variable} for printing values and JSTL tags like <c:if> and <c:forEach> for logic.

Q: Forward vs Redirect?

A: Forward is server-side — same request, URL doesn't change in browser. Redirect sends a response telling the browser to make a new request — URL changes. Use redirect after POST to prevent form resubmission (PRG pattern).

Q: What is HttpSession?

A: Server-side storage per user. After login, we store userId, fullName, and role in the session. On every subsequent request, the Servlet reads session data to know who the user is. Created with request.getSession().

Viva Cheat Sheet

REST API Q&A

Q: What is REST?

A: Representational State Transfer. An architectural style that uses HTTP methods (GET, POST, PUT, DELETE) to perform CRUD operations on resources. Returns JSON. Our /api/auth, /api/bookings, /api/issues are REST endpoints.

Q: @RestController vs @Controller?

A: @RestController returns JSON data directly to the client. @Controller returns HTML views (JSP pages). @RestController = @Controller + @ResponseBody on every method.

Q: What is @RequestBody?

A: Parses the JSON request body into a Java object automatically. Spring uses the Jackson library to map JSON keys to Java field names. Example: {"userId": "260401-AK01"} becomes a LoginRequest object.

Q: What is @PathVariable?

A: Extracts a value from the URL path. In /api/auth/user/{userId}, the {userId} part is a variable. If someone calls /api/auth/user/260401-AK01, the method gets "260401-AK01" as the parameter.

Q: What is ResponseEntity?

A: Wraps the response body with an HTTP status code. ResponseEntity.ok() = 200, ResponseEntity.status(401) = Unauthorized, ResponseEntity.status(404) = Not Found. Gives the client a proper status code, not just data.

Q: What is CORS?

A: Cross-Origin Resource Sharing. Browsers block requests between different origins (e.g., frontend on port 3000 calling API on port 8080). Our CorsConfig allows all origins to call /api/** endpoints.

Q: What is a DTO?

A: Data Transfer Object. Defines the shape of API requests and responses separately from the database model. Our LoginResponse has a nested UserDTO that includes userId, fullName, email, role — but strips the password for security.

Viva Cheat Sheet

Database Q&A

Q: What is H2?

A: An embedded Java database. Zero installation — just add the Maven dependency and it runs. Supports the same SQL as MySQL. Data persists in a file (nimbusdb.mv.db). Perfect for development and learning. Has a built-in web console at /h2-console.

Q: What is JPA?

A: Java Persistence API. A specification that maps Java objects to database tables using annotations. @Entity marks a class as a table, @Id marks the primary key, @Column maps fields to columns. Hibernate is the JPA implementation we use.

Q: DAO vs Repository?

A: DAO uses JdbcTemplate with manual SQL and RowMapper (Sprint 2 approach — 30+ lines per DAO). Repository extends JpaRepository and auto-generates SQL from method names (Sprint 3 approach — 5 lines). Repository is less code for the same result.

Q: What is a RowMapper?

A: An interface used with JdbcTemplate that converts one row of a SQL ResultSet into a Java object. You define how each column maps to each field. This is the DAO pattern — in JPA Repositories, this mapping is automatic via annotations.

Microservices Q&A

Q: What is a Microservice?

A: A small, independent service that does one thing well. It has its own codebase, its own database, and can be deployed independently. Our Nimbus app is monolithic (one deployable), but it's structured in layers that could be split.

Q: Monolith vs Microservice?

A: Monolith = one codebase, one deployment, one database (our Nimbus app). Microservice = many small services, each deployed independently with its own database. Monolith is simpler for small teams. Microservices scale better for large teams and high traffic.

Q: How would you split Nimbus into microservices?

A: Three services: Auth Service (/api/auth/*), Booking Service (/api/bookings/*), Issue Service (/api/issues/*). Each gets its own database. They communicate via REST APIs. An API Gateway routes requests to the right service.
