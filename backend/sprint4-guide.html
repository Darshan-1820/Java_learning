SPRINT 4

REST APIs & Frontend Integration

The modern way to connect your frontend to your backend. Your existing HTML pages stay the same — they just get their data from Java instead of LocalStorage.

THE BIG IDEA

In Sprint 3, your Java code generated HTML pages (JSP). That's the old way. Modern web apps work differently: the backend sends DATA (JSON), and the frontend renders it. This is what REST APIs do.

Sprint 3 (Old Way)

Browser → Servlet → JSP → HTML

Server generates the HTML page. Every time you need new data, the server builds an entire new page.

Sprint 4 (Modern Way)

Browser → JavaScript → REST API → JSON

Server sends raw data. The frontend JavaScript decides how to display it. Much faster, much cleaner.

Your nimbus/ frontend already has all the HTML. It just needs DATA. Right now it gets data from LocalStorage. In Sprint 4, it will get data from REST APIs instead.

The complete login flow in Sprint 4
  1. User clicks "Login" on login.html
  2. JavaScript calls: fetch('http://localhost:8080/api/auth/login', { method: 'POST', body: JSON.stringify({userId, password}) })
  3. Spring Boot routes the request to AuthController.login()
  4. AuthController calls UserService.validateLogin()
  5. UserService calls UserRepository.findByUserIdAndPassword()
  6. Repository returns the User object from H2 database
  7. AuthController wraps User in a LoginResponse DTO (strips the password)
  8. Returns JSON: { "success": true, "user": {...} }
  9. JavaScript gets the JSON, stores the session, redirects to dashboard
CONCEPT 1

What is a REST API?

REST = Representational State Transfer. Sounds scary, means simple.

A REST API is just a URL that returns data instead of a web page.

Normal URL
http://localhost:8080/servlet/login

Returns an HTML page that the browser displays.

REST API URL
http://localhost:8080/api/auth/login

Returns JSON data that JavaScript reads and uses.

Here is what a REST API response looks like:

REST API Response (JSON) 
{
  "success": true,
  "message": "Login successful.",
  "user": {
    "userId": "260401-AK01",
    "fullName": "Amit Kumar",
    "role": "customer"
  }
}

Your frontend JavaScript calls these URLs using fetch(), gets JSON back, and updates the page. No page reload needed.

REAL-WORLD ANALOGY

Think of it like ordering food. The old way (Servlets + JSP): you go to a restaurant that cooks your meal AND plates it for you. The REST API way: you order from a kitchen that sends you the raw ingredients, and you plate it yourself however you want. The kitchen (backend) focuses on the food (data), and you (frontend) decide the presentation.

CONCEPT 2

What is JSON?

The format REST APIs use to send data back and forth. You already know it.

JSON (JavaScript Object Notation) is the universal language between backend and frontend. It is the same format as JavaScript objects:

JSON Example — A Booking 
{
  "bookingId": "NMB-260401-0001",
  "userId": "260401-AK01",
  "recName": "Vikram Reddy",
  "parStatus": "In Transit",
  "parServiceCost": 150
}

Java sends this, JavaScript reads this. Simple key-value pairs. Curly braces for objects, square brackets for arrays. That is all JSON is.

IN NIMBUS

Every single API endpoint in Nimbus returns JSON. When you create a booking, the backend calculates the cost and sends back the full booking object as JSON. When you login, it sends back the user object as JSON. The format never changes.

CONCEPT 3

What are DTOs?

DTOs define the SHAPE of data going in and out of your API.

DTO = Data Transfer Object. It is a simple Java class that defines exactly what fields the API accepts or returns.

LoginRequest DTO:   { "userId": "...", "password": "..." }        → goes INTO the API
LoginResponse DTO:  { "success": true, "user": {...} }            → comes OUT of the API
DTOs control the shape of API input and output

Why not just use the User model directly?

Because the User model has the password field. You do not want to send passwords back to the frontend. DTOs let you control exactly what data gets sent.

Without DTOs (Dangerous)

API returns the full User object including password: "Amit@123". Anyone can see it in the browser Network tab.

With DTOs (Safe)

API returns a UserDTO with only safe fields: userId, fullName, role, etc. Password is never sent.

CONCEPT 4

What is CORS?

A browser security feature that blocks cross-origin requests by default.

CORS = Cross-Origin Resource Sharing. When your frontend (localhost:5500) tries to call your backend (localhost:8080), the browser blocks it by default because they are different origins (different ports = different origins).

WITHOUT CORS CONFIG

You open login.html in the browser. You type your credentials and click Login. JavaScript calls fetch('http://localhost:8080/api/auth/login'). The browser says: "Blocked by CORS policy." Nothing works.

CorsConfig.java tells Spring Boot: "Allow requests from any origin." One small config class, problem solved. You will see the code for it later in the Config section.

CONCEPT 5

Servlets vs REST Controllers

Both handle HTTP requests. REST is the modern standard.

Servlets (Sprint 3) REST Controllers (Sprint 4)
Returns HTML pages (JSP) JSON data
Annotation @WebServlet @RestController
Frontend Server generates HTML Frontend generates HTML
Used by Browser directly JavaScript fetch()
Data format HTML embedded in JSP Pure JSON
Page reload? Yes, every action No, updates in-place

Both work. REST is the modern standard used by almost every real-world application today. When you hear "API" in the industry, they almost always mean REST APIs.

DTO 1

LoginRequest.java

Defines the shape of login request data: what the frontend sends when a user tries to log in.

When the frontend sends { "userId": "260401-AK01", "password": "Amit@123" }, Spring Boot automatically converts that JSON into this Java object. That is what @RequestBody does in the controller.

src/main/java/com/nimbus/dto/LoginRequest.java 
package com.nimbus.dto;

/**
 * LoginRequest — Data Transfer Object for login API.
 * Receives: { "userId": "260401-AK01", "password": "Amit@123" }
 */
public class LoginRequest {
    private String userId;
    private String password;

    public LoginRequest() {}

    public String getUserId() { return userId; }
    public void setUserId(String userId) { this.userId = userId; }

    public String getPassword() { return password; }
    public void setPassword(String password) { this.password = password; }
}
Line-by-line explanation
1
private String userId;
Matches the "userId" key in the JSON. Spring maps JSON keys to field names automatically.
2
public LoginRequest() {}
Empty constructor. Spring needs this to create an instance and then fill in the fields from JSON.
3
Getters and Setters
Spring uses setUserId() and setPassword() to fill in the values from the JSON request body.
DTO 2

LoginResponse.java

Defines the shape of login response data. Contains a nested UserDTO that strips the password.

This is the most important DTO. It has a nested class called UserDTO. The outer LoginResponse wraps the result with success and message. The inner UserDTO contains the user's details without the password.

WHY A NESTED CLASS?

The UserDTO is only used inside LoginResponse. Putting it inside keeps things organized. You do not need a separate file for something that only one class uses. The keyword static means you can create a UserDTO without creating a LoginResponse first.

DTO 5

ApiResponse.java

Generic wrapper for ALL API responses. Every endpoint returns this same structure.

Instead of every controller returning different shapes of data, they all return ApiResponse. This means the frontend always knows what to expect: success, message, and data.

src/main/java/com/nimbus/dto/ApiResponse.java 
package com.nimbus.dto;

/**
 * ApiResponse — Generic API response wrapper.
 * Every REST endpoint returns this format:
 * { "success": true/false, "message": "...", "data": {...} }
 */
public class ApiResponse {
    private boolean success;
    private String message;
    private Object data;

    public ApiResponse() {}

    public ApiResponse(boolean success, String message) {
        this.success = success;
        this.message = message;
    }

    public ApiResponse(boolean success, String message, Object data) {
        this.success = success;
        this.message = message;
        this.data = data;
    }

    // Static factory methods for convenience
    public static ApiResponse ok(String message, Object data) {
        return new ApiResponse(true, message, data);
    }

    public static ApiResponse ok(Object data) {
        return new ApiResponse(true, "Success", data);
    }

    public static ApiResponse error(String message) {
        return new ApiResponse(false, message);
    }

    public boolean isSuccess() { return success; }
    public void setSuccess(boolean success) { this.success = success; }

    public String getMessage() { return message; }
    public void setMessage(String message) { this.message = message; }

    public Object getData() { return data; }
    public void setData(Object data) { this.data = data; }
}
Line-by-line explanation
1
private Object data;
The type is Object — it can hold ANY Java object. A User, a List of Bookings, a single Booking, anything. Spring converts it to JSON automatically.
2
public static ApiResponse ok(Object data)
A static factory method. Instead of writing new ApiResponse(true, "Success", data) every time, you just write ApiResponse.ok(data). Shorter and cleaner.
3
public static ApiResponse error(String message)
For error responses. Creates { "success": false, "message": "..." } with no data field. Used in catch blocks when something goes wrong.
CONSISTENT API DESIGN

Every endpoint in Nimbus returns either ApiResponse.ok(data) or ApiResponse.error(message). The frontend always checks data.success first. If true, the actual payload is in data.data. If false, the error message is in data.message. Consistent patterns make code easier to write and debug.

BEFORE WE BEGIN

REST Controller Annotations

Every controller uses these annotations. Understand these six and you understand all the controllers.

Annotation What it does
@RestController "This class handles REST API requests." It tells Spring: methods in this class return data (not HTML views). Spring automatically converts the return value to JSON.
@RequestMapping("/api/auth") "All URLs in this class start with /api/auth." So if a method has @PostMapping("/login"), the full URL becomes /api/auth/login.
@PostMapping("/login") "Handle POST requests to /api/auth/login." POST is used when the client is sending data (like login credentials or a new booking).
@GetMapping("/users") "Handle GET requests to /api/auth/users." GET is used when the client is fetching data (like getting a list of users).
@RequestBody "Parse the JSON request body into this Java object." When you write @RequestBody LoginRequest request, Spring reads the JSON from the request, creates a LoginRequest object, and fills in the fields.
@PathVariable "Extract a value from the URL path." In /api/auth/user/{userId}, if the request is /api/auth/user/260401-AK01, then userId = "260401-AK01".
@RequestParam "Extract a query parameter from the URL." In /api/bookings/history?userId=X&startDate=Y, userId and startDate are request params.
ResponseEntity<T> "Lets you set HTTP status codes." ResponseEntity.ok() = 200 OK. ResponseEntity.status(401) = 401 Unauthorized. ResponseEntity.status(404) = 404 Not Found.

AuthController.java

Handles login, registration, and user lookup. The authentication hub of the app.

Endpoints in this Controller

src/main/java/com/nimbus/controller/AuthController.java 
package com.nimbus.controller;

import com.nimbus.dto.ApiResponse;
import com.nimbus.dto.LoginRequest;
import com.nimbus.dto.LoginResponse;
import com.nimbus.dto.RegisterRequest;
import com.nimbus.model.User;
import com.nimbus.service.UserService;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.*;

/**
 * AuthController — Phase 4 REST API for authentication.
 *
 * Endpoints (matching TCS Sprint 3 requirements):
 *   POST /api/auth/login            → validateLogin
 *   POST /api/auth/register         → registerCustomer
 *   POST /api/auth/register-admin   → registerAdmin
 *   GET  /api/auth/user/{userId}    → getUserDetails
 */
@RestController
@RequestMapping("/api/auth")
public class AuthController {

    private final UserService userService;

    public AuthController(UserService userService) {
        this.userService = userService;
    }

    /**
     * POST /api/auth/login
     * TCS Sprint 3: validateLogin
     *
     * Request:  { "userId": "260401-AK01", "password": "Amit@123" }
     * Response: { "success": true, "message": "...", "user": {...} }
     */
    @PostMapping("/login")
    public ResponseEntity<LoginResponse> login(@RequestBody LoginRequest request) {
        try {
            User user = userService.validateLogin(request.getUserId(), request.getPassword());

            LoginResponse.UserDTO dto = mapToDTO(user);
            LoginResponse response = new LoginResponse(true, "Login successful.", dto);
            return ResponseEntity.ok(response);

        } catch (IllegalArgumentException e) {
            LoginResponse response = new LoginResponse(false, e.getMessage(), null);
            return ResponseEntity.status(401).body(response);
        }
    }

    /**
     * POST /api/auth/register
     * TCS Sprint 3: registerCustomer
     */
    @PostMapping("/register")
    public ResponseEntity<ApiResponse> register(@RequestBody RegisterRequest request) {
        try {
            User user = userService.register(
                    request.getFirstName(), request.getMiddleName(), request.getLastName(),
                    request.getEmail(), request.getCountryCode(), request.getMobile(),
                    request.getStreet(), request.getZipCode(), request.getCity(),
                    request.getState(), request.getCountry(),
                    request.getPassword(), request.getConfirmPassword(),
                    request.getPreferences());

            LoginResponse.UserDTO dto = mapToDTO(user);
            return ResponseEntity.ok(ApiResponse.ok("Registration successful. User ID: " + user.getUserId(), dto));

        } catch (IllegalArgumentException e) {
            return ResponseEntity.badRequest().body(ApiResponse.error(e.getMessage()));
        }
    }

    /**
     * POST /api/auth/register-admin
     * TCS Sprint 3: registerAdmin
     */
    @PostMapping("/register-admin")
    public ResponseEntity<ApiResponse> registerAdmin(@RequestBody RegisterRequest request) {
        try {
            String role = request.getRole() != null ? request.getRole() : "officer";
            User user = userService.registerAdmin(
                    request.getFirstName(), request.getLastName(),
                    request.getEmail(), request.getMobile(),
                    request.getPassword(), role);

            return ResponseEntity.ok(ApiResponse.ok("Admin registered. User ID: " + user.getUserId(), mapToDTO(user)));

        } catch (IllegalArgumentException e) {
            return ResponseEntity.badRequest().body(ApiResponse.error(e.getMessage()));
        }
    }

    /**
     * GET /api/auth/users — Get all users.
     */
    @GetMapping("/users")
    public ResponseEntity<ApiResponse> getAllUsers() {
        return ResponseEntity.ok(ApiResponse.ok(userService.getAllUsers()));
    }

    /**
     * GET /api/auth/user/{userId}
     */
    @GetMapping("/user/{userId}")
    public ResponseEntity<ApiResponse> getUser(@PathVariable String userId) {
        try {
            User user = userService.getUserById(userId);
            return ResponseEntity.ok(ApiResponse.ok(mapToDTO(user)));
        } catch (IllegalArgumentException e) {
            return ResponseEntity.status(404).body(ApiResponse.error(e.getMessage()));
        }
    }

    // --- Map User entity to DTO (strips password) ---
    private LoginResponse.UserDTO mapToDTO(User user) {
        LoginResponse.UserDTO dto = new LoginResponse.UserDTO();
        dto.setUserId(user.getUserId());
        dto.setFirstName(user.getFirstName());
        dto.setMiddleName(user.getMiddleName());
        dto.setLastName(user.getLastName());
        dto.setFullName(user.getFullName());
        dto.setEmail(user.getEmail());
        dto.setCountryCode(user.getCountryCode());
        dto.setMobile(user.getMobile());
        dto.setStreet(user.getStreet());
        dto.setZipCode(user.getZipCode());
        dto.setCity(user.getCity());
        dto.setState(user.getState());
        dto.setCountry(user.getCountry());
        dto.setRole(user.getRole());
        dto.setPreferences(user.getPreferences());
        dto.setCreatedAt(user.getCreatedAt() != null ? user.getCreatedAt().toString() : null);
        return dto;
    }
}
Key parts explained
1
@RestController + @RequestMapping("/api/auth")
This class handles REST requests. All endpoints start with /api/auth.
2
private final UserService userService;
Dependency injection. Spring creates the UserService and passes it to this controller's constructor. The controller never creates it with new.
3
@PostMapping("/login") ... @RequestBody LoginRequest request
When a POST request hits /api/auth/login, Spring parses the JSON body into a LoginRequest object and passes it to this method.
4
ResponseEntity.ok(response) vs ResponseEntity.status(401).body(response)
On success: return HTTP 200 with the login data. On failure: return HTTP 401 (Unauthorized) with the error message.
5
mapToDTO(user)
Converts the User entity (which has a password) into a UserDTO (which does NOT have a password). This is the security layer that prevents password leaks.

Key parts explained
1
@PostMapping (no path)
No path means it handles POST to the base URL: /api/bookings. Combined with @RequestMapping("/api/bookings") on the class.
2
@RequestParam(required = false) String userId
Reads from the URL query string: /api/bookings/history?userId=260401-AK01. required = false means it is optional.
3
@RequestBody Map<String, String> request
Instead of creating a DTO for simple requests, we use a Map. request.get("status") reads the "status" field from the JSON body. Good for small, simple payloads.
4
LocalDateTime.parse(request.get("pickupTime"))
Converts the string "2026-04-02T10:00:00" into a Java LocalDateTime object. The "T" in the middle separates date from time. This is the ISO-8601 format.

CorsConfig.java

One small class that tells Spring Boot to allow cross-origin requests.

src/main/java/com/nimbus/config/CorsConfig.java 
package com.nimbus.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

/**
 * CorsConfig — Cross-Origin Resource Sharing configuration.
 *
 * This allows the frontend (running on a different port or file://) to
 * make API calls to the Spring Boot backend at http://localhost:8080.
 *
 * Without CORS, the browser blocks requests from different origins.
 * Example: nimbus/login.html (file://) calling http://localhost:8080/api/auth/login
 */
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("*")
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*");
    }
}
Line-by-line explanation
1
@Configuration
Tells Spring: "This class contains configuration settings. Read it at startup."
2
implements WebMvcConfigurer
An interface from Spring MVC. By implementing it, we can override methods to customize Spring's web behavior.
3
registry.addMapping("/api/**")
Apply CORS rules to all URLs that start with /api/. The ** means "any path underneath."
4
.allowedOrigins("*")
Allow requests from ANY origin. In production, you would list specific domains instead of "*".
5
.allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
Allow these HTTP methods. OPTIONS is used by browsers as a "preflight check" before the actual request.
FRONTEND INTEGRATION

What Changes?

Almost nothing. That is the beauty of good architecture.

Your nimbus/ frontend already works with LocalStorage via storage.js. Every page calls methods like NimbusDB.validateLogin(), NimbusDB.addBooking(), NimbusDB.getBookings(), etc.

storage-api.js is a drop-in replacement for storage.js. It has the exact same NimbusDB interface — same method names, same return format. But instead of reading/writing LocalStorage, it calls REST APIs using fetch().

storage.js (LocalStorage)

NimbusDB.validateLogin()

→ reads from LocalStorage

→ compares password in JavaScript

→ returns { success, user }

storage-api.js (REST API)

NimbusDB.validateLogin()

→ calls POST /api/auth/login

→ Java validates on the server

→ returns { success, user }

KEY INSIGHT

The rest of your HTML, CSS, and JavaScript code does not change at all. The dashboard pages, the booking form, the tracking page — they all keep calling NimbusDB.addBooking(), NimbusDB.getBookings(), etc. They do not know (or care) whether the data comes from LocalStorage or a REST API. This is called abstraction — hiding the implementation details behind a consistent interface.

THE SWITCH

How to Switch

One line change per HTML file. That is it.

To switch your frontend from LocalStorage to REST APIs, you literally just change ONE line in each HTML file:

Before (LocalStorage)
<script src="js/storage.js"></script>
After (REST API)
<script src="js/storage-api.js"></script>

That is it. Every HTML page continues to call NimbusDB.validateLogin(), NimbusDB.addBooking(), etc. — but now those calls go to your Java backend instead of LocalStorage.

ANALOGY

Imagine your car's steering wheel is connected to the front wheels (LocalStorage). You swap the steering mechanism to power steering (REST API). You still turn the steering wheel the same way. Your driving does not change. Only the mechanism underneath changed. That is what storage-api.js does — it swaps the engine without changing the controls.

Common issues and how to fix them.

ProblemSolution

What You Have Built

Four sprints. From zero to a full-stack application.

Sprint 1: Static Frontend
HTML + CSS + JavaScript with LocalStorage. Three dashboards (Customer, Officer, Support). All the UI, no backend.
Sprint 2: Java Console App
Java models, services, repositories. JDBC connection to H2 database. Collections, exceptions, generics. The brain of the app, running in a terminal.
Sprint 3: Servlets + Spring
HTTP basics, Servlets, JSP pages, Spring MVC. The console app becomes a web app. Server-rendered HTML pages.
Sprint 4: REST APIs + Frontend Integration
REST controllers, DTOs, CORS, JSON. The same frontend, now powered by a real Java backend. Modern architecture. Full-stack.
THE FULL PICTURE

You started with HTML pages that stored data in the browser. You built a Java backend that stores data in a real database. You connected the two with REST APIs. Your frontend calls NimbusDB.addBooking() and a Java method runs on the server, saves to a database, calculates the cost, and sends back JSON. That is a full-stack application. That is how real companies build software.
