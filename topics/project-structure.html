Section 01

Why Project Structure Matters

Before you write a single class, you need to know where everything goes. A well-organized project is the difference between a codebase you can navigate and one that makes you want to quit.

The Toolbox Analogy

Imagine two mechanics. Mechanic A has a toolbox where every wrench, screwdriver, and socket has its own labeled slot. He reaches in and grabs exactly what he needs in two seconds. Mechanic B has a cardboard box where everything is thrown in together. He spends five minutes digging through the pile every single time.

Both mechanics have the same tools. But Mechanic A finishes jobs in half the time because he never wastes time searching.

Your project structure is your toolbox. Every file has a designated place. When you need to find the code that connects to the database, you know exactly where it is. When you need to change how users are validated, you know which folder to open. You never search. You navigate.

What Happens Without Structure?

No Structure (Chaos)
  • 20 Java files in one folder
  • Database code mixed with display logic
  • "Where's the login code?" — could be anywhere
  • One change breaks three unrelated things
  • New team member takes weeks to understand
  • You forget your own code in two weeks
Clean Structure (Professional)
  • Files grouped by responsibility
  • Database code in dao/, display in app/
  • "Where's login?" — service/UserService.java
  • Changes to one layer don't ripple to others
  • New team member productive in hours
  • Code documents itself through organization
Think About It

Professional projects can have hundreds or thousands of files. Google's monorepo has 2 billion lines of code. Without a strict, agreed-upon structure, it would be impossible for 25,000+ engineers to work in the same codebase. Structure isn't optional — it's how teams scale.

Section 02

Eclipse Setup Step-by-Step

Setting up Eclipse for Java development. We'll create a Maven project from scratch and configure it properly.

Creating a Java Maven Project

  1. Launch Eclipse. When prompted for a workspace location, choose a dedicated folder like C:\eclipse-workspace. This is where all your Eclipse projects live. Keep it clean — don't use your Desktop.
  2. Switch to Java EE perspective. Go to Window → Perspective → Open Perspective → Java EE. This gives you the right panels for web development: Package Explorer, Servers view, etc.
  3. Create the project. File → New → Maven Project. Check "Create a simple project" and click Next.
  4. Enter project details:
    • Group Id: com.nimbus
    • Artifact Id: nimbus-parcel
    • Packaging: war
    Click Finish. Eclipse creates the Maven directory structure.
  5. Create webapp directory. In Package Explorer, right-click src/main → New → Folder → name: webapp. Inside it, create another folder: WEB-INF.
  6. Configure the pom.xml. Double-click pom.xml. Switch to the "pom.xml" tab (not the Overview tab) to see raw XML. Paste in your dependencies, properties, and plugins. Save (Ctrl+S).
  7. Update Maven. Right-click the project name → Maven → Update Project (Alt+F5). Check "Force Update of Snapshots/Releases" on your first setup. Click OK. Wait for downloads to finish (watch the progress bar at bottom-right).
  8. Verify. Expand "Maven Dependencies" in Package Explorer. You should see all your JARs listed. No red X marks means everything resolved correctly.

Adding Tomcat Server

  1. Open the Servers view. Window → Show View → Servers. If the Servers tab is empty, click "No servers are available. Click this link to create a new server."
  2. Select Apache Tomcat v9.0. Under "Apache," choose Tomcat v9.0 Server. Click Next.
  3. Browse to Tomcat installation. Click "Browse" and navigate to where you extracted Tomcat (e.g., C:\apache-tomcat-9.0.80). Click Finish.
  4. Add your project. Right-click the Tomcat server in the Servers view → "Add and Remove" → move nimbus-parcel to the right side (Configured) → Finish.
  5. Start the server. Right-click Tomcat → Start. The Console should show "Server startup in [xxx] ms." Open a browser and go to http://localhost:8080/nimbus-parcel/.
Port 8080 Already In Use?

If Tomcat fails to start with "Address already in use," another program is using port 8080. Double-click the Tomcat server in Eclipse → change the port from 8080 to 8081 or 9090. Or find and kill whatever process is using 8080.

Section 03

Package Naming Convention

Java packages are how you organize your classes into logical groups. The naming convention isn't arbitrary — it follows a universal standard.

The Reverse Domain Convention

Java packages follow the reverse domain name convention. If your company's website is nimbus.com, your base package is com.nimbus. This guarantees uniqueness worldwide — no two organizations will accidentally use the same package name.

Package naming examples 
// Google's code
com.google.maps.api
com.google.cloud.storage

// Apache's code
org.apache.commons.lang3
org.apache.maven.plugins

// Our Nimbus project
com.nimbus.app         // Servlets (presentation layer)
com.nimbus.service     // Business logic
com.nimbus.dao         // Database access
com.nimbus.model       // Data classes
com.nimbus.util        // Utility/helper classes
com.nimbus.exception   // Custom exceptions

Package Rules

RuleExampleWhy
All lowercase com.nimbus.dao not com.Nimbus.DAO Convention. Packages are always lowercase to distinguish from class names (which use PascalCase).
No special characters com.nimbus not com.nimbus-app Hyphens, spaces, and numbers at the start are invalid in package names.
Singular nouns com.nimbus.model not com.nimbus.models Convention. The package represents a category, not a collection.
Folder path = package name com.nimbus.dao = com/nimbus/dao/ Java requires this. The file system path must match the package declaration exactly.
Common Mistake

Creating a class in src/main/java/com/nimbus/dao/ but writing package com.nimbus.service; at the top of the file. Java will refuse to compile. The package declaration must match the folder path. Eclipse creates the correct declaration automatically when you create a class inside a package.

Section 04

Layered Architecture

Nimbus follows a layered architecture where each layer has one specific job and only talks to the layer directly below it.

The Four Layers

Think of it like a restaurant kitchen. The customer (browser) talks to the waiter (presentation layer). The waiter relays the order to the head chef (service layer). The head chef tells the prep cook what ingredients to get from the pantry (DAO layer). The pantry is the database.

The customer never walks into the kitchen. The waiter never opens the pantry. Each person has one job and talks to one other person.

Presentation
Package: com.nimbus.app
What: Servlets and JSPs. Receives HTTP requests, calls the Service layer, sends responses back to the browser.
Talks to: Service layer only. Never touches the database directly.
Service
Package: com.nimbus.service
What: Business logic and rules. "Is this user allowed to book?" "Calculate delivery fee." Orchestrates DAO calls.
Talks to: DAO layer. Receives calls from Presentation layer.
DAO
Package: com.nimbus.dao
What: Data Access Objects. The only layer that writes SQL. All database reads and writes go through here.
Talks to: Database only. Receives calls from Service layer.
Database
System: MySQL
What: Stores all persistent data: users, bookings, issues. Tables, rows, columns.
Talks to: Nobody. It just stores data and answers SQL queries from DAO.

The Flow: User Login Example

Let's trace exactly what happens when a user clicks "Login" on the Nimbus website:

BrowserPOST /login
→
LoginServletapp/ layer
→
UserServiceservice/ layer
→
UserDAOdao/ layer
→
MySQLdatabase
The flow in code 
// 1. LoginServlet (Presentation) — receives the HTTP request
String email = request.getParameter("email");
String password = request.getParameter("password");
User user = userService.validateLogin(email, password);  // calls Service

// 2. UserService (Service) — applies business rules
public User validateLogin(String email, String password) {
    User user = userDAO.findByEmail(email);               // calls DAO
    if (user == null) throw new UserNotFoundException("No account found");
    if (!user.getPassword().equals(password)) throw new InvalidCredentialsException();
    return user;
}

// 3. UserDAO (DAO) — runs the SQL
public User findByEmail(String email) {
    String sql = "SELECT * FROM tbl_UserProfile WHERE email = ?";
    // Execute query, map ResultSet to User object, return it
}

Why Layers?

Without layers, your servlet would contain the HTTP handling, the business rules, AND the SQL query all in one method. That's 100+ lines of tangled code. With layers:

  • Change the database? Only rewrite DAO files. Service and Presentation don't change.
  • Change the business rule? Only edit the Service class. DAO and Presentation untouched.
  • Switch from servlets to REST API? Only rewrite the Presentation layer. Service and DAO stay identical.
  • Testing? Test each layer independently. Mock the DAO when testing Service. Mock the Service when testing Servlet.
In Nimbus

Every feature in Nimbus follows this exact pattern. User registration: RegisterServlet → UserService → UserDAO → MySQL. Booking a parcel: BookingServlet → BookingService → BookingDAO → MySQL. Once you understand one flow, you understand all of them.

Full Nimbus Package Structure

Every file in the Nimbus project, organized by package. This is your project map — memorize it.

Complete Nimbus project structure 
src/main/java/com/nimbus/
│
├── model/                          // Data classes (POJOs)
│   ├── User.java                   // userId, name, email, password, role
│   ├── Booking.java                // bookingId, userId, senderAddr, receiverAddr, status, date
│   └── SupportIssue.java           // issueId, userId, bookingId, description, status
│
├── dao/                            // Database access — SQL lives here
│   ├── UserDAO.java                // findByEmail(), save(), findById()
│   ├── BookingDAO.java             // save(), findByUserId(), updateStatus(), findById()
│   └── SupportIssueDAO.java        // save(), findByUserId(), updateStatus()
│
├── service/                        // Business logic — rules live here
│   ├── UserService.java            // validateLogin(), registerUser()
│   ├── BookingService.java         // createBooking(), getBookingsByUser(), updateDeliveryStatus()
│   └── SupportService.java         // createIssue(), getIssuesByUser()
│
├── app/                            // Servlets — HTTP handling
│   ├── LoginServlet.java           // POST /login
│   ├── RegisterServlet.java        // POST /register
│   ├── BookingServlet.java         // POST /book, GET /track
│   ├── DashboardServlet.java       // GET /dashboard
│   └── SupportServlet.java         // POST /support, GET /support
│
├── util/                           // Helpers
│   └── DBConnection.java           // getConnection() — reads db.properties, returns Connection
│
└── exception/                      // Custom exceptions
    ├── UserNotFoundException.java
    ├── InvalidCredentialsException.java
    ├── DuplicateEmailException.java
    └── BookingNotFoundException.java

Every File Explained

model/ — Data Classes (POJOs)

POJO = Plain Old Java Object. These classes have only fields, getters, setters, and constructors. No logic, no SQL, no HTTP. They represent database tables as Java objects.

com/nimbus/model/User.java 
package com.nimbus.model;

public class User {
    private int userId;
    private String name;
    private String email;
    private String password;
    private String role;      // "customer", "officer", "support"

    // Default constructor
    public User() {}

    // Parameterized constructor
    public User(int userId, String name, String email, String password, String role) {
        this.userId = userId;
        this.name = name;
        this.email = email;
        this.password = password;
        this.role = role;
    }

    // Getters and Setters
    public int getUserId() { return userId; }
    public void setUserId(int userId) { this.userId = userId; }

    public String getName() { return name; }
    public void setName(String name) { this.name = name; }

    public String getEmail() { return email; }
    public void setEmail(String email) { this.email = email; }

    public String getPassword() { return password; }
    public void setPassword(String password) { this.password = password; }

    public String getRole() { return role; }
    public void setRole(String role) { this.role = role; }
}
Why each part exists
1
package com.nimbus.model;
Declares this class belongs to the model package. Must match folder path: src/main/java/com/nimbus/model/
2
private fields
Each field maps to a column in tbl_UserProfile. Private = encapsulation. No one can directly access user.password; they must use getPassword().
3
Default constructor User()
Required by frameworks and when you want to create an empty User then fill fields with setters. JDBC ResultSet mapping uses this: create User(), then set each field.
4
Parameterized constructor
Convenience: create a fully-populated User in one line. Used when you already have all the values (e.g., from registration form).
5
Getters and Setters
Public access methods. JSP EL expressions like ${user.name} automatically call getName(). Without getters, JSP can't read your data.

dao/ — Database Access Objects

DAO classes are the only place where SQL queries exist. They take Java objects in, talk to the database, and return Java objects out. No business logic, no HTTP awareness.

service/ — Business Logic

Service classes contain the rules of your application. "Is the password correct?" "Does this user already exist?" "Can this officer update this booking?" The service layer orchestrates DAO calls and applies logic between them.

app/ — Servlets (Presentation)

Servlets handle HTTP. They read request parameters, call service methods, and forward results to JSP pages for display. A servlet should be thin — mostly just glue between HTTP and the service layer.

util/ — Utilities

DBConnection.java is the most important utility. It reads database connection settings from db.properties and provides a getConnection() method that all DAOs use. One central place to configure the database.

exception/ — Custom Exceptions

Instead of throwing generic Exception everywhere, custom exceptions tell you exactly what went wrong. UserNotFoundException is a lot more helpful than Exception: something happened.

Section 06

Build Path Configuration

The build path tells Eclipse where to find the libraries your code needs. With Maven, most of this is automatic, but you need to understand it.

What is the Build Path?

When you write import java.sql.Connection;, Java needs to know where java.sql lives. The build path is a list of all places Java should look for classes: the JDK itself, your project's source folders, and external libraries (JARs).

Adding JARs (Non-Maven Projects)

In older projects without Maven, you add JARs manually:

  1. Right-click the project → Properties → Java Build Path
  2. Go to the Libraries tab
  3. Click Add External JARs
  4. Browse to the JAR file (e.g., mysql-connector-j-8.0.33.jar)
  5. Click Apply and Close

With Maven, you never do this. Maven handles the build path automatically. When you add a dependency to pom.xml and run Alt+F5, Eclipse's m2e plugin updates the build path for you. All Maven JARs appear under "Maven Dependencies" in Package Explorer.

Configuring JDK Version

  1. Right-click project → Properties → Java Build Path
  2. Click JRE System Library → Edit
  3. Select "Workspace default JRE" (if it's JDK 17) or "Alternate JRE" and pick JDK 17
  4. Click Finish → Apply and Close
JDK vs JRE

Eclipse needs a JDK (Java Development Kit), not just a JRE (Java Runtime Environment). The JDK includes the compiler (javac). If Eclipse says "No compiler is provided," you have a JRE instead of a JDK. Download JDK 17 from adoptium.net and configure it in Eclipse: Window → Preferences → Java → Installed JREs → Add → Standard VM.

Section 07

db.properties File

Configuration that shouldn't be hardcoded. Database URL, username, password — all in one external file that's easy to change without recompiling.

Why a Properties File?

Imagine hardcoding your database password in UserDAO.java:

BAD: Hardcoded credentials 
// DON'T DO THIS
Connection conn = DriverManager.getConnection(
    "jdbc:mysql://localhost:3306/nimbus_db",
    "root",
    "MyP@ssw0rd123"
);

Problems: The password is in source code (goes to Git = everyone sees it). Changing the database means editing Java files and recompiling. Different developers have different MySQL passwords — they'd each need to change the code.

The Properties File

src/main/resources/db.properties 
db.driver=com.mysql.cj.jdbc.Driver
db.url=jdbc:mysql://localhost:3306/nimbus_db
db.username=root
db.password=your_password_here

Reading It in Java

com/nimbus/util/DBConnection.java 
package com.nimbus.util;

import java.io.InputStream;
import java.sql.Connection;
import java.sql.DriverManager;
import java.util.Properties;

public class DBConnection {

    private static String url;
    private static String username;
    private static String password;

    static {
        try {
            Properties props = new Properties();
            InputStream input = DBConnection.class
                .getClassLoader()
                .getResourceAsStream("db.properties");
            props.load(input);

            Class.forName(props.getProperty("db.driver"));
            url = props.getProperty("db.url");
            username = props.getProperty("db.username");
            password = props.getProperty("db.password");
        } catch (Exception e) {
            throw new RuntimeException("Failed to load db.properties", e);
        }
    }

    public static Connection getConnection() throws Exception {
        return DriverManager.getConnection(url, username, password);
    }
}
Line-by-line explanation
1
static { ... }
Static initializer block. Runs ONCE when the class is first loaded. Reads db.properties and stores values. Every subsequent call to getConnection() uses these cached values.
2
getClassLoader().getResourceAsStream("db.properties")
Reads the file from the classpath (target/classes/). Since Maven copies src/main/resources/ to target/classes/ during build, the file is available at runtime.
3
Class.forName(props.getProperty("db.driver"))
Loads the MySQL JDBC driver class into memory. This registers the driver with DriverManager so it knows how to talk to MySQL.
4
DriverManager.getConnection(url, username, password)
Creates and returns a new database connection. Every DAO method calls this, uses the connection, then closes it.
In Nimbus

Every DAO class starts with Connection conn = DBConnection.getConnection();. If you ever need to change the database URL (like switching to a production server), you edit db.properties — not a single Java file. Add db.properties to your .gitignore so passwords don't get pushed to Git.

Section 08

.classpath and .project Files

Eclipse creates hidden configuration files in your project root. You rarely edit them manually, but you should know what they are.

.project

This file tells Eclipse what kind of project this is. It contains the project name and which "natures" (capabilities) the project has — like Java, Maven, and Web.

.project 
<projectDescription>
    <name>nimbus-parcel</name>
    <buildSpec>
        <buildCommand>
            <name>org.eclipse.jdt.core.javabuilder</name>
        </buildCommand>
        <buildCommand>
            <name>org.eclipse.m2e.core.maven2Builder</name>
        </buildCommand>
    </buildSpec>
    <natures>
        <nature>org.eclipse.jdt.core.javanature</nature>
        <nature>org.eclipse.m2e.core.maven2Nature</nature>
    </natures>
</projectDescription>

javabuilder = "this is a Java project, compile .java files." maven2Builder = "use Maven to manage dependencies." These are set automatically when you create a Maven project.

.classpath

This file lists all the source folders and libraries Eclipse should use when compiling. For Maven projects, it includes the Maven Dependencies container.

.classpath 
<classpath>
    <classpathentry kind="src" path="src/main/java"/>
    <classpathentry kind="src" path="src/main/resources"/>
    <classpathentry kind="src" path="src/test/java"/>
    <classpathentry kind="con" path="org.eclipse.jdt.launching.JRE_CONTAINER"/>
    <classpathentry kind="con" path="org.eclipse.m2e.MAVEN2_CLASSPATH_CONTAINER"/>
    <classpathentry kind="output" path="target/classes"/>
</classpath>
  • kind="src" — Source folders. Eclipse compiles files from these locations.
  • kind="con" — Containers. JRE_CONTAINER = the Java runtime. MAVEN2_CLASSPATH_CONTAINER = all Maven dependencies.
  • kind="output" — Where compiled .class files go.
Git and Eclipse Files

Some teams include .project and .classpath in Git so everyone has the same Eclipse config. Others add them to .gitignore because different IDEs generate different files. For our Nimbus project: include them if your whole team uses Eclipse. Gitignore them if anyone uses IntelliJ.

Section 10

Creating Nimbus Classes in Eclipse

The step-by-step flow of creating each Nimbus class in Eclipse. Follow this exact process for every file.

Creating a Model Class (User.java)

  1. Right-click the com.nimbus.model package in Package Explorer → New → Class.
  2. Name: Type User (PascalCase, no .java extension — Eclipse adds it). Leave "Superclass" as java.lang.Object. Don't check any interfaces. Click Finish.
  3. Eclipse creates User.java with the package declaration and an empty class body. The cursor is inside the class, ready for you to type.
  4. Add fields. Type your private fields: private int userId;, private String name;, etc.
  5. Generate getters/setters. Right-click inside the class → Source → Generate Getters and Setters (or Alt+Shift+S, R). Select All → Generate. Eclipse writes every getter and setter for you.
  6. Generate constructors. Alt+Shift+S, O → select all fields → Generate. Then create a default constructor manually: public User() {}.
  7. Save (Ctrl+S). Eclipse compiles automatically. Green check = no errors.

Creating a DAO Class (UserDAO.java)

  1. Right-click com.nimbus.dao → New → Class. Name: UserDAO. Finish.
  2. Add method stubs. Start with the method signatures you need: public User findByEmail(String email) { ... } public void save(User user) { ... }
  3. Write SQL and JDBC code inside each method. Use DBConnection.getConnection() to get the connection.
  4. Organize imports. Press Ctrl+Shift+O. Eclipse adds imports for Connection, PreparedStatement, ResultSet, etc. If there are multiple options (like java.sql.Connection vs com.sun.Connection), Eclipse shows a dialog — always pick the java.sql one.

Creating a Service Class (UserService.java)

  1. Right-click com.nimbus.service → New → Class. Name: UserService. Finish.
  2. Add a DAO field: private UserDAO userDAO = new UserDAO();. The service layer calls the DAO, so it needs a reference to it.
  3. Add methods that use the DAO. Example: validateLogin() calls userDAO.findByEmail(), checks the password, throws exceptions if invalid.

Creating a Servlet (LoginServlet.java)

  1. Right-click com.nimbus.app → New → Class. Name: LoginServlet. In the "Superclass" field, type javax.servlet.http.HttpServlet (or browse for it). Click Finish.
  2. Add the @WebServlet annotation above the class: @WebServlet("/login"). This maps the URL /login to this servlet.
  3. Override doPost method. Right-click inside the class → Source → Override/Implement Methods → check doPost(HttpServletRequest, HttpServletResponse) → OK. Eclipse generates the method with the correct signature.
  4. Add servlet logic: Read form parameters with request.getParameter(), call userService.validateLogin(), set session attributes, and forward/redirect to the appropriate page.
  5. Import resolution. Ctrl+Shift+O to add javax.servlet imports. These work because servlet-api is in your pom.xml with scope provided.
In Nimbus

You'll repeat this pattern for every feature: create the Model class first (data structure), then the DAO (database access), then the Service (business logic), then the Servlet (HTTP handling). Always build bottom-up. Model → DAO → Service → Servlet. Each layer depends on the one below it, so you can't write a Service without a working DAO.

The Golden Rule

Every file has exactly one responsibility. UserDAO does user database operations — not booking operations. BookingService handles booking business logic — not user authentication. If a class is doing two different things, it needs to be split into two classes. This is the Single Responsibility Principle and it's non-negotiable in professional code.
