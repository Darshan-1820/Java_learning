Section 01

What is JDBC?

JDBC (Java Database Connectivity) is the bridge between your Java code and a database. Without it, your Java program has no way to store, retrieve, or modify data in MySQL, PostgreSQL, or any other database.

Analogy

Imagine two people who speak different languages. Java speaks "Java." MySQL speaks "SQL." They cannot talk to each other directly. JDBC is the translator sitting between them — it takes Java's requests, translates them into SQL, sends them to the database, gets the response back, and translates it into Java objects.

The Architecture

Every JDBC interaction follows this chain:

Your Java CodeUserDAO, BookingDAO
→
JDBC APIjava.sql package
→
JDBC DriverMySQL Connector/J
→
MySQL Databasenimbus_db
Java Code → JDBC API → Driver (translates) → Database

Why does this matter? Because the JDBC API is the same no matter which database you use. If tomorrow Nimbus switches from MySQL to PostgreSQL, you just swap the driver JAR file and change the connection URL. Your DAO code stays the same. That is the power of having a standard API.

In Nimbus

Every time UserDAO saves a new user or BookingDAO fetches a booking, the call goes through JDBC. The connection string jdbc:mysql://localhost:3306/nimbus_db in your DBConnection class is the address that JDBC uses to find the database.

Why Not Just Write SQL Directly?

You cannot send a raw SQL string to MySQL from Java. MySQL runs as a separate process (a server), usually on port 3306. To communicate with it, your Java program needs to:

  1. Open a network connection to the MySQL server
  2. Authenticate with a username and password
  3. Send a SQL query over that connection using the MySQL wire protocol
  4. Read the binary response and parse it into rows and columns
  5. Close the connection when done

JDBC handles all of this. You just write connection.prepareStatement("SELECT * FROM users") and JDBC does the rest.

Section 02

JDBC Components

JDBC has six core components. Each one plays a specific role in the chain from Java to the database.

Driver
The translator. Converts JDBC calls into the specific protocol the database understands. For MySQL, this is com.mysql.cj.jdbc.Driver (inside the MySQL Connector/J JAR).
Connection
An open session with the database. Created via DriverManager.getConnection(url, user, pass). Think of it as a phone call — you dial the database, it picks up, and you have a live line.
Statement
Sends a static SQL string to the database. Simple but dangerous — vulnerable to SQL injection. Used for one-off queries where no user input is involved.
PreparedStatement
Sends a parameterized SQL string with ? placeholders. Safe from SQL injection. Used in all Nimbus DAO code. Pre-compiled for better performance.
CallableStatement
Calls stored procedures in the database. Syntax: "{call procedureName(?, ?)}". Not used in Nimbus but common in enterprise apps.
ResultSet
The result of a SELECT query. A cursor that starts before the first row. You call next() to move to each row, then getString(), getInt(), etc. to read columns.
Think About It

Statement, PreparedStatement, and CallableStatement are all in the java.sql package. PreparedStatement extends Statement, and CallableStatement extends PreparedStatement. So a CallableStatement can do everything a PreparedStatement can do, plus call stored procedures.

Section 03

Setting Up JDBC

Three things need to happen before you can talk to a database from Java: add the driver JAR, load the driver class, and open a connection.

Step 1: Add MySQL Connector JAR

The MySQL JDBC driver is not part of Java. It is a separate library (mysql-connector-j-8.x.x.jar). You add it to your project in one of two ways:

Manual (no build tool)

Download the JAR file from dev.mysql.com and add it to your project's /lib folder. Then add it to the classpath when compiling:

javac -cp lib/mysql-connector-j-8.3.0.jar src/*.java
Maven (Nimbus uses this)

Add this dependency to pom.xml and Maven downloads it automatically:

<dependency>
  <groupId>com.mysql</groupId>
  <artifactId>mysql-connector-j</artifactId>
</dependency>

Step 2: The DBConnection Utility

In Nimbus, all database connections go through a single utility class. This is the one place where the URL, username, and password live. Every DAO calls DBConnection.getConnection() instead of writing connection code themselves.

com/nimbus/util/DBConnection.java 
package com.nimbus.util;

import java.sql.Connection;
import java.sql.DriverManager;
import java.sql.SQLException;

public class DBConnection {

    private static final String URL = "jdbc:mysql://localhost:3306/nimbus_db";
    private static final String USER = "root";
    private static final String PASSWORD = "password";

    static {
        try {
            Class.forName("com.mysql.cj.jdbc.Driver");
        } catch (ClassNotFoundException e) {
            throw new RuntimeException("MySQL Driver not found!", e);
        }
    }

    public static Connection getConnection() throws SQLException {
        return DriverManager.getConnection(URL, USER, PASSWORD);
    }
}
Line-by-line explanation
1
package com.nimbus.util;
This class lives in the util (utility) package. Utilities are helper classes shared across the project.
2
import java.sql.Connection;
Imports the Connection interface from JDBC. This represents an open session with the database.
3
import java.sql.DriverManager;
DriverManager is the factory that creates Connection objects. You give it a URL + credentials, it gives you a connection.
4
private static final String URL = "jdbc:mysql://localhost:3306/nimbus_db";
jdbc: protocol | mysql: database type | localhost: server address | 3306: MySQL default port | nimbus_db: database name. This is the "address" of our database.
5
private static final String USER = "root";
MySQL username. "root" is the default admin user. In production, you would use a less privileged user.
6
static { ... Class.forName("com.mysql.cj.jdbc.Driver"); }
A static initializer block — runs ONCE when the class is first loaded. Class.forName() loads the MySQL driver class into memory and registers it with DriverManager. Without this, DriverManager does not know MySQL exists.
7
public static Connection getConnection() throws SQLException
static so you can call DBConnection.getConnection() without creating an object. throws SQLException because the connection might fail (wrong password, database down, etc.).
8
return DriverManager.getConnection(URL, USER, PASSWORD);
DriverManager looks at the URL, finds the registered MySQL driver, and asks it to open a real TCP connection to MySQL on port 3306. Returns a Connection object you can use to send queries.

The Connection URL Anatomy

jdbc:mysql://localhost:3306/nimbus_db
 |    |        |       |       |
 |    |        |       |       +-- Database name
 |    |        |       +---------- Port number (MySQL default: 3306)
 |    |        +------------------ Host (localhost = this machine)
 |    +--------------------------- Database type (mysql, postgresql, oracle)
 +-------------------------------- Protocol (always "jdbc")
Every part of the URL has meaning. Change "localhost" to an IP address for a remote database.
Section 04

Statement vs PreparedStatement

This is the most important security concept in JDBC. Statement is dangerous. PreparedStatement is safe. Here is why.

Statement — The Dangerous Way

With Statement, you build the SQL string by concatenating user input directly:

BAD: SQL String Concatenation 
// NEVER DO THIS in real code
String email = request.getParameter("email");
String password = request.getParameter("password");

Statement stmt = connection.createStatement();
String sql = "SELECT * FROM users WHERE email = '" + email
           + "' AND password = '" + password + "'";
ResultSet rs = stmt.executeQuery(sql);
SQL Injection Attack — Step by Step

If a user types this into the email field:

' OR '1'='1' --

The SQL becomes:

SELECT * FROM users WHERE email = '' OR '1'='1' --' AND password = ''

Breaking it down:

  • ' OR '1'='1' — The '1'='1' is always true, so the WHERE clause matches every row
  • -- — This is a SQL comment. Everything after it is ignored, including the password check
  • Result: The attacker logs in as the first user in the database (usually an admin) without knowing any password

An attacker could also type: '; DROP TABLE users; -- and your entire users table is deleted.

PreparedStatement — The Safe Way

With PreparedStatement, you use ? placeholders. The database treats user input as data, never as SQL code:

GOOD: PreparedStatement with Placeholders 
String email = request.getParameter("email");
String password = request.getParameter("password");

String sql = "SELECT * FROM users WHERE email = ? AND password = ?";
PreparedStatement pstmt = connection.prepareStatement(sql);
pstmt.setString(1, email);    // First ? gets the email value
pstmt.setString(2, password); // Second ? gets the password value
ResultSet rs = pstmt.executeQuery();

Now if the user types ' OR '1'='1' --, the database sees it as a literal string value, not SQL code. It searches for an email that is literally ' OR '1'='1' --. No match. No injection. Safe.

How PreparedStatement Works Internally

Step 1: CompileSQL structure sent to DB
→
Step 2: BindValues fill ? slots
→
Step 3: ExecuteDB runs pre-compiled plan
The SQL structure is locked before values are inserted. Values cannot change the structure.
Feature Statement PreparedStatement
SQL Injection Vulnerable Protected
Performance (repeated queries) Re-compiled every time Compiled once, executed many times
Readability Messy string concatenation Clean ? placeholders
Type safety Everything is a string setInt(), setString(), setTimestamp()
Used in Nimbus? No Yes — everywhere
In Nimbus

Every single query in UserDAO and BookingDAO uses PreparedStatement. You will never see string concatenation in our SQL. This is a hard rule, not a suggestion.

Section 05

CRUD Operations

CRUD stands for Create, Read, Update, Delete. Every database operation falls into one of these four categories. Each uses a specific SQL command and a specific JDBC method.

Operation SQL JDBC Method Returns
Create INSERT INTO executeUpdate() int — number of rows inserted
Read SELECT executeQuery() ResultSet — the matching rows
Update UPDATE executeUpdate() int — number of rows changed
Delete DELETE FROM executeUpdate() int — number of rows deleted
Think About It

Notice that INSERT, UPDATE, and DELETE all use executeUpdate() — they modify data and return a row count. Only SELECT uses executeQuery() because it returns data (a ResultSet).

CREATE — Inserting a User

Nimbus: addUser() 
public boolean addUser(User user) {
    String sql = "INSERT INTO users (name, email, password, phone, role) VALUES (?, ?, ?, ?, ?)";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        pstmt.setString(1, user.getName());
        pstmt.setString(2, user.getEmail());
        pstmt.setString(3, user.getPassword());
        pstmt.setString(4, user.getPhone());
        pstmt.setString(5, user.getRole());

        int rowsInserted = pstmt.executeUpdate();
        return rowsInserted > 0;  // true if at least 1 row was inserted

    } catch (SQLException e) {
        e.printStackTrace();
        return false;
    }
}
Line-by-line explanation
1
String sql = "INSERT INTO users (...) VALUES (?, ?, ?, ?, ?)";
The SQL template. Five ? placeholders for the five columns. The actual values are set below.
2
try (Connection conn = DBConnection.getConnection(); PreparedStatement pstmt = ...)
try-with-resources. Both conn and pstmt are automatically closed when the try block ends, even if an exception occurs.
3
pstmt.setString(1, user.getName());
Fills the 1st ? with the user's name. JDBC placeholders are 1-indexed (starts at 1, not 0).
4
int rowsInserted = pstmt.executeUpdate();
executeUpdate() sends the INSERT to MySQL and returns how many rows were inserted. For a single INSERT, this is 1 on success, 0 on failure.
5
return rowsInserted > 0;
Returns true if the insert worked, false if not. The caller (e.g. a servlet) uses this to show a success/error message.

READ — Fetching a User by ID

Nimbus: getUserById() 
public User getUserById(int id) {
    String sql = "SELECT * FROM users WHERE user_id = ?";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        pstmt.setInt(1, id);
        ResultSet rs = pstmt.executeQuery();

        if (rs.next()) {
            User user = new User();
            user.setUserId(rs.getInt("user_id"));
            user.setName(rs.getString("name"));
            user.setEmail(rs.getString("email"));
            user.setPhone(rs.getString("phone"));
            user.setRole(rs.getString("role"));
            return user;
        }

    } catch (SQLException e) {
        e.printStackTrace();
    }
    return null;  // No user found with that ID
}
Line-by-line explanation
1
pstmt.setInt(1, id);
setInt() instead of setString() because user_id is an integer column. JDBC ensures type safety.
2
ResultSet rs = pstmt.executeQuery();
executeQuery() for SELECT. Returns a ResultSet (cursor pointing to the results).
3
if (rs.next()) { ... }
rs.next() moves the cursor to the first row. If it returns false, there are no results. We use if (not while) because we expect at most one row for a primary key lookup.
4
rs.getInt("user_id"), rs.getString("name")
Extracts values from the current row by column name. You can also use column index: rs.getInt(1), but names are more readable.
5
return null;
If no row matched the ID, or if an exception occurred, return null. The caller must check for null before using the result.

UPDATE — Changing Booking Status

Nimbus: updateStatus() 
public boolean updateStatus(int bookingId, String newStatus) {
    String sql = "UPDATE bookings SET status = ? WHERE booking_id = ?";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        pstmt.setString(1, newStatus);
        pstmt.setInt(2, bookingId);

        int rowsUpdated = pstmt.executeUpdate();
        return rowsUpdated > 0;

    } catch (SQLException e) {
        e.printStackTrace();
        return false;
    }
}

DELETE — Removing a Booking

Nimbus: deleteBooking() 
public boolean deleteBooking(int bookingId) {
    String sql = "DELETE FROM bookings WHERE booking_id = ?";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        pstmt.setInt(1, bookingId);
        int rowsDeleted = pstmt.executeUpdate();
        return rowsDeleted > 0;

    } catch (SQLException e) {
        e.printStackTrace();
        return false;
    }
}
Important

Notice the pattern? INSERT, UPDATE, and DELETE all follow the same structure: prepare the statement, set the parameters, call executeUpdate(), check the row count. The only thing that changes is the SQL string.

Section 06

ResultSet Deep Dive

A ResultSet is like a cursor pointing at the results of a SELECT query. It starts before the first row — you must call next() to move to each row.

ResultSet cursor positions:

  [Before first]  ←  rs starts here
       |
  Row 1: { id=1, name="Darshan", email="darshan@mail.com" }
  Row 2: { id=2, name="Ravi",    email="ravi@mail.com"    }
  Row 3: { id=3, name="Priya",   email="priya@mail.com"   }
       |
  [After last]

  rs.next() → moves cursor to Row 1 (returns true)
  rs.next() → moves cursor to Row 2 (returns true)
  rs.next() → moves cursor to Row 3 (returns true)
  rs.next() → moves to After Last (returns false)

Getting Values from a Row

Method Java Type SQL Type Example
getString("col") String VARCHAR, TEXT rs.getString("name")
getInt("col") int INT, TINYINT rs.getInt("user_id")
getDouble("col") double DOUBLE, DECIMAL rs.getDouble("weight")
getTimestamp("col") Timestamp DATETIME, TIMESTAMP rs.getTimestamp("created_at")
getBoolean("col") boolean BOOLEAN, TINYINT(1) rs.getBoolean("is_active")

Column Name vs Column Index

By Name (Recommended)
rs.getString("email")

Readable. Works even if you rearrange columns in the SELECT.

By Index (Fragile)
rs.getString(3)

Which column is 3? If you add a column to the SELECT, all indexes shift.

Navigating Multiple Rows

When a SELECT returns multiple rows, use a while loop:

Nimbus: getBookingsByUser() 
public List<Booking> getBookingsByUser(int userId) {
    List<Booking> bookings = new ArrayList<>();
    String sql = "SELECT * FROM bookings WHERE user_id = ? ORDER BY created_at DESC";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        pstmt.setInt(1, userId);
        ResultSet rs = pstmt.executeQuery();

        while (rs.next()) {
            bookings.add(extractBooking(rs));  // Helper method builds Booking object
        }

    } catch (SQLException e) {
        e.printStackTrace();
    }
    return bookings;
}
Think About It

if (rs.next()) = expecting 0 or 1 row (e.g. findById). while (rs.next()) = expecting 0 or many rows (e.g. findAll, search). Use the right one for the right situation.

Section 07

Transaction Management

A transaction groups multiple SQL operations into a single all-or-nothing unit. Either everything succeeds, or everything rolls back as if nothing happened.

ACID Properties

Analogy — Bank Transfer

You transfer Rs. 500 from Account A to Account B. That requires two operations: debit A, credit B. What if the system crashes after debiting A but before crediting B? You just lost Rs. 500 into thin air. Transactions prevent this.

Atomicity
All or nothing. If the debit succeeds but the credit fails, the debit is undone (rolled back). Both operations succeed together, or neither happens.
Consistency
Valid state to valid state. Before the transfer, total money in A+B = Rs. 10,000. After the transfer, total must still be Rs. 10,000. No money created or destroyed.
Isolation
Concurrent transactions do not interfere. If two people transfer money simultaneously, each transaction sees a consistent snapshot. No partial reads.
Durability
Once committed, it is permanent. Even if the server crashes 1 millisecond after commit, the data is safe on disk.

AutoCommit and Manual Transactions

By default, JDBC runs in auto-commit mode: every single SQL statement is its own transaction. It is committed immediately. To group multiple statements, you turn off auto-commit:

Nimbus: Booking + Payment Transaction 
public boolean createBookingWithPayment(Booking booking, Payment payment) {
    Connection conn = null;
    try {
        conn = DBConnection.getConnection();
        conn.setAutoCommit(false);  // START TRANSACTION

        // Step 1: Insert booking
        String bookingSql = "INSERT INTO bookings (user_id, pickup, dropoff, weight, status) VALUES (?, ?, ?, ?, ?)";
        PreparedStatement bookingStmt = conn.prepareStatement(bookingSql);
        bookingStmt.setInt(1, booking.getUserId());
        bookingStmt.setString(2, booking.getPickup());
        bookingStmt.setString(3, booking.getDropoff());
        bookingStmt.setDouble(4, booking.getWeight());
        bookingStmt.setString(5, "PENDING");
        bookingStmt.executeUpdate();

        // Step 2: Insert payment
        String paymentSql = "INSERT INTO payments (booking_id, amount, method, status) VALUES (LAST_INSERT_ID(), ?, ?, ?)";
        PreparedStatement paymentStmt = conn.prepareStatement(paymentSql);
        paymentStmt.setDouble(1, payment.getAmount());
        paymentStmt.setString(2, payment.getMethod());
        paymentStmt.setString(3, "COMPLETED");
        paymentStmt.executeUpdate();

        conn.commit();  // BOTH succeeded — make it permanent
        return true;

    } catch (SQLException e) {
        if (conn != null) {
            try { conn.rollback(); } catch (SQLException ex) { ex.printStackTrace(); }
        }
        e.printStackTrace();
        return false;

    } finally {
        if (conn != null) {
            try {
                conn.setAutoCommit(true);  // Reset for next use
                conn.close();
            } catch (SQLException e) { e.printStackTrace(); }
        }
    }
}
Transaction flow
1
conn.setAutoCommit(false);
Turns off auto-commit. From now on, nothing is saved to the database until you explicitly call commit(). This starts the transaction.
2
bookingStmt.executeUpdate();
Inserts the booking. But because auto-commit is off, this change is not permanent yet. It is in a temporary state.
3
paymentStmt.executeUpdate();
Inserts the payment. Also not permanent yet. LAST_INSERT_ID() gets the auto-generated booking_id from Step 1.
4
conn.commit();
Both inserts succeeded. commit() makes them permanent. If the server crashes after this line, both rows are safe.
5
conn.rollback();
If anything fails (exception), rollback() undoes everything. The booking insert is erased. No orphan booking without payment.
In Nimbus

When a user creates a booking and pays, both operations must succeed together. If payment insertion fails but the booking was already inserted, you would have a booking with no payment record — a data inconsistency. Transactions prevent this.

Section 08

try-with-resources

Database connections, statements, and result sets are system resources that must be closed when you are done. Forgetting to close them causes connection leaks — eventually your application runs out of connections and crashes.

Before: Manual Closing (Old Way)

The Ugly Old Way 
Connection conn = null;
PreparedStatement pstmt = null;
ResultSet rs = null;

try {
    conn = DBConnection.getConnection();
    pstmt = conn.prepareStatement("SELECT * FROM users WHERE user_id = ?");
    pstmt.setInt(1, id);
    rs = pstmt.executeQuery();
    // ... process results ...

} catch (SQLException e) {
    e.printStackTrace();
} finally {
    // MUST close in reverse order, each with its own try-catch
    if (rs != null) { try { rs.close(); } catch (SQLException e) { } }
    if (pstmt != null) { try { pstmt.close(); } catch (SQLException e) { } }
    if (conn != null) { try { conn.close(); } catch (SQLException e) { } }
}

Look at that mess. Three null checks, three try-catch blocks, just to close resources. And if you forget one, you have a leak.

After: try-with-resources (Java 7+)

The Clean Way (Nimbus uses this) 
try (Connection conn = DBConnection.getConnection();
     PreparedStatement pstmt = conn.prepareStatement("SELECT * FROM users WHERE user_id = ?")) {

    pstmt.setInt(1, id);
    ResultSet rs = pstmt.executeQuery();
    // ... process results ...

} catch (SQLException e) {
    e.printStackTrace();
}
// conn, pstmt, and rs are ALL automatically closed here
Think About It

Any object that implements AutoCloseable can be used in try-with-resources. Connection, PreparedStatement, and ResultSet all implement it. When the try block ends (normally or by exception), Java calls .close() on each resource in reverse order. No leaks. No forgotten close calls.

What Happens When You Leak Connections

Request 1Opens connection, forgets to close
→
Request 2Opens another connection
→
Request 100Opens another...
→
CRASHMySQL: "Too many connections"
MySQL has a connection limit (default: 151). Leaked connections pile up until the limit is hit.
In Nimbus

Every DAO method in Nimbus uses try-with-resources. This is non-negotiable. You will never see a finally block with manual .close() calls in Nimbus code.

Section 09

Connection Pooling

Opening a new database connection is expensive. It involves a TCP handshake, authentication, and memory allocation. Connection pooling solves this by reusing connections.

The Problem

Without pooling, every request follows this cycle:

Request arrives
→
Open new connection~50-100ms
→
Execute query~5ms
→
Close connection~10ms
Opening a connection takes 10-20x longer than the actual query. That is wasted time.

The Solution: Connection Pool

A pool maintains a set of pre-opened connections. When your code needs a connection, it borrows one from the pool. When done, it returns it. The connection stays open for the next request.

  Connection Pool (e.g. HikariCP, pool size = 5)
  ┌────────────────────────────────────────────┐
  │                                            │
  │   [Conn 1] ←── borrowed by Request A       │
  │   [Conn 2] ←── borrowed by Request B       │
  │   [Conn 3]     idle (waiting)              │
  │   [Conn 4]     idle (waiting)              │
  │   [Conn 5] ←── borrowed by Request C       │
  │                                            │
  └────────────────────────────────────────────┘
  Request D arrives → borrows Conn 3
  Request A finishes → returns Conn 1 to pool
Connections are reused, not created and destroyed for every request.

HikariCP — The Industry Standard

HikariCP is the most popular connection pool in Java. It is the default in Spring Boot. Here is what it would look like in Nimbus:

DBConnection.java with HikariCP (production version) 
import com.zaxxer.hikari.HikariConfig;
import com.zaxxer.hikari.HikariDataSource;
import java.sql.Connection;
import java.sql.SQLException;

public class DBConnection {
    private static final HikariDataSource dataSource;

    static {
        HikariConfig config = new HikariConfig();
        config.setJdbcUrl("jdbc:mysql://localhost:3306/nimbus_db");
        config.setUsername("root");
        config.setPassword("password");
        config.setMaximumPoolSize(10);       // Max 10 connections in pool
        config.setMinimumIdle(2);            // Keep at least 2 idle
        config.setConnectionTimeout(30000);  // Wait 30s before giving up
        dataSource = new HikariDataSource(config);
    }

    public static Connection getConnection() throws SQLException {
        return dataSource.getConnection();  // Borrows from pool, not new
    }
}
In Nimbus

The current Nimbus DBConnection uses DriverManager.getConnection() directly (creates a new connection each time). This is fine for learning. In production, you would swap it with HikariCP as shown above. The beautiful part: no DAO code changes. They all call DBConnection.getConnection() and do not care how the connection was obtained.

Section 10

The DAO Pattern

DAO stands for Data Access Object. It is a design pattern that separates database code from business logic. Each table gets its own DAO class.

Why Separate Database Code?

Without the DAO pattern, your servlet (or controller) would contain SQL queries, connection handling, result processing, AND business logic all mixed together. That is a maintenance disaster.

Without DAO

The servlet has SQL mixed with HTTP handling:

// In the servlet
Connection conn = ...;
PreparedStatement pstmt = conn.prepareStatement("SELECT...");
ResultSet rs = ...;
// parse results
// also handle HTTP request/response
// also validate input
// 200 lines of mixed code
With DAO

Servlet calls the DAO. Clean separation:

// In the servlet
UserDAO userDAO = new UserDAO();
User user = userDAO.getUserById(id);
// handle HTTP request/response
// ONE line for DB access

What Does a DAO Do?

  • Encapsulates all SQL queries for one table
  • Translates between database rows and Java objects
  • Handles connections, exceptions, and resource cleanup
  • Exposes clean methods like addUser(), getUserById()

Nimbus DAO Structure

ServletHTTP handling
→
DAOUserDAO, BookingDAO
→
DBConnectionConnection utility
→
MySQLnimbus_db
Each layer has one job. Servlets never touch SQL. DAOs never touch HTTP.

Why This Makes Code Better

Benefit How
Testable You can test UserDAO independently without running a servlet. Just call its methods directly.
Swappable If you switch from MySQL to PostgreSQL, only the DAO changes. Servlets do not care.
Readable A servlet calling userDAO.addUser(user) is instantly clear. A servlet with 30 lines of SQL is not.
Reusable Multiple servlets can call the same getUserById(). No duplicate SQL.
Section 11

Batch Operations

When you need to execute the same query many times (e.g. inserting 100 bookings), sending them one by one is slow. Batch operations group them and send everything at once.

Without Batch: 100 Round Trips

Slow: One Insert Per Round Trip 
// 100 bookings = 100 round trips to the database
for (Booking b : bookings) {
    pstmt.setInt(1, b.getUserId());
    pstmt.setString(2, b.getPickup());
    pstmt.setString(3, b.getDropoff());
    pstmt.executeUpdate();  // Network round trip each time!
}

With Batch: 1 Round Trip

Fast: Batch Insert 
public void addBookingsBatch(List<Booking> bookings) {
    String sql = "INSERT INTO bookings (user_id, pickup, dropoff, weight, status) VALUES (?, ?, ?, ?, ?)";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement pstmt = conn.prepareStatement(sql)) {

        conn.setAutoCommit(false);  // Wrap in transaction for safety

        for (Booking b : bookings) {
            pstmt.setInt(1, b.getUserId());
            pstmt.setString(2, b.getPickup());
            pstmt.setString(3, b.getDropoff());
            pstmt.setDouble(4, b.getWeight());
            pstmt.setString(5, "PENDING");
            pstmt.addBatch();  // Queue it, don't send yet
        }

        int[] results = pstmt.executeBatch();  // Send ALL at once
        conn.commit();

        System.out.println("Inserted " + results.length + " bookings");

    } catch (SQLException e) {
        e.printStackTrace();
    }
}
Line-by-line explanation
1
pstmt.addBatch();
Adds the current set of parameter values to the batch queue. Does NOT send to the database yet. You can call this hundreds of times.
2
int[] results = pstmt.executeBatch();
Sends all queued statements to the database in ONE network trip. Returns an array where each element is the row count for that statement (usually 1 for each INSERT).
3
conn.setAutoCommit(false) / conn.commit();
Wrapping the batch in a transaction means if row 50 fails, all 49 previous rows are rolled back. All or nothing.
Think About It

Performance difference: 100 individual inserts might take 2-3 seconds. The same 100 inserts as a batch typically take 50-100 milliseconds. That is a 20-30x improvement.

Full Nimbus UserDAO

The complete UserDAO class with every method Nimbus uses. This is real production code — study every method.

com/nimbus/dao/UserDAO.java 
package com.nimbus.dao;

import com.nimbus.model.User;
import com.nimbus.util.DBConnection;
import java.sql.*;
import java.util.ArrayList;
import java.util.List;

public class UserDAO {

    // ==================== CREATE ====================
    public boolean addUser(User user) {
        String sql = "INSERT INTO users (name, email, password, phone, role) VALUES (?, ?, ?, ?, ?)";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, user.getName());
            pstmt.setString(2, user.getEmail());
            pstmt.setString(3, user.getPassword());
            pstmt.setString(4, user.getPhone());
            pstmt.setString(5, user.getRole());

            return pstmt.executeUpdate() > 0;

        } catch (SQLException e) {
            e.printStackTrace();
            return false;
        }
    }

    // ==================== READ (Login) ====================
    public User validateLogin(String email, String password) {
        String sql = "SELECT * FROM users WHERE email = ? AND password = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, email);
            pstmt.setString(2, password);
            ResultSet rs = pstmt.executeQuery();

            if (rs.next()) {
                return extractUser(rs);
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return null;
    }

    // ==================== READ (by ID) ====================
    public User getUserById(int id) {
        String sql = "SELECT * FROM users WHERE user_id = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setInt(1, id);
            ResultSet rs = pstmt.executeQuery();

            if (rs.next()) {
                return extractUser(rs);
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return null;
    }

    // ==================== READ (unique check) ====================
    public boolean isEmailUnique(String email) {
        String sql = "SELECT COUNT(*) FROM users WHERE email = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, email);
            ResultSet rs = pstmt.executeQuery();

            if (rs.next()) {
                return rs.getInt(1) == 0;  // 0 means no user has this email
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return false;
    }

    // ==================== READ (support users) ====================
    public List<User> getSupportUsers() {
        List<User> users = new ArrayList<>();
        String sql = "SELECT * FROM users WHERE role = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, "SUPPORT");
            ResultSet rs = pstmt.executeQuery();

            while (rs.next()) {
                users.add(extractUser(rs));
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return users;
    }

    // ==================== HELPER ====================
    private User extractUser(ResultSet rs) throws SQLException {
        User user = new User();
        user.setUserId(rs.getInt("user_id"));
        user.setName(rs.getString("name"));
        user.setEmail(rs.getString("email"));
        user.setPassword(rs.getString("password"));
        user.setPhone(rs.getString("phone"));
        user.setRole(rs.getString("role"));
        return user;
    }
}
Method-by-method explanation
1
addUser(User user) → boolean
Takes a User object, inserts it into the users table. Returns true if insertion succeeded (row count > 0). Used during registration.
2
validateLogin(String email, String password) → User or null
Checks if email + password combination exists. Returns the User object if match found (login success), null if not (login failure). Uses if (rs.next()) because there is at most one matching user.
3
getUserById(int id) → User or null
Fetches a single user by primary key. Used when you need to display user details (e.g. profile page, booking owner info).
4
isEmailUnique(String email) → boolean
Counts users with this email. If count is 0, email is unique (returns true). Used before registration to prevent duplicate accounts. Note: uses rs.getInt(1) (column index) because COUNT(*) has no column name.
5
getSupportUsers() → List<User>
Returns all users with role "SUPPORT." Uses while (rs.next()) because there can be multiple support staff. Used by admin dashboards to assign agents to tickets.
6
extractUser(ResultSet rs) → User
A private helper that builds a User object from a ResultSet row. Called by every READ method. This avoids repeating rs.getString("name"), rs.getString("email"), etc. in every method. DRY principle in action.

package com.nimbus.dao;

import com.nimbus.model.Booking;

public class BookingDAO {

    // ==================== CREATE ====================
    public boolean addBooking(Booking booking) {
        String sql = "INSERT INTO bookings (user_id, sender_name, sender_phone, sender_address, "
                   + "receiver_name, receiver_phone, receiver_address, weight, parcel_type, "
                   + "pickup_date, status) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setInt(1, booking.getUserId());
            pstmt.setString(2, booking.getSenderName());
            pstmt.setString(3, booking.getSenderPhone());
            pstmt.setString(4, booking.getSenderAddress());
            pstmt.setString(5, booking.getReceiverName());
            pstmt.setString(6, booking.getReceiverPhone());
            pstmt.setString(7, booking.getReceiverAddress());
            pstmt.setDouble(8, booking.getWeight());
            pstmt.setString(9, booking.getParcelType());
            pstmt.setTimestamp(10, booking.getPickupDate());
            pstmt.setString(11, "PENDING");

            return pstmt.executeUpdate() > 0;

        } catch (SQLException e) {
            e.printStackTrace();

        }
    }

    // ==================== READ (single) ====================
    public Booking getBookingById(int bookingId) {

            }

            e.printStackTrace();
        }
        return bookings;
    }

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setTimestamp(1, start);
            pstmt.setTimestamp(2, end);
            ResultSet rs = pstmt.executeQuery();

            while (rs.next()) {
                bookings.add(extractBooking(rs));
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return bookings;
    }

    public List<Booking> getAllBookings() {
        List<Booking> bookings = new ArrayList<>();
        String sql = "SELECT * FROM bookings ORDER BY created_at DESC";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            ResultSet rs = pstmt.executeQuery();

            while (rs.next()) {
                bookings.add(extractBooking(rs));
            }

        } catch (SQLException e) {
            e.printStackTrace();
        }
        return bookings;
    }

    public boolean updateStatus(int bookingId, String newStatus) {
        String sql = "UPDATE bookings SET status = ? WHERE booking_id = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, newStatus);
            pstmt.setInt(2, bookingId);
            return pstmt.executeUpdate() > 0;

        } catch (SQLException e) {
            e.printStackTrace();
            return false;
        }
    }

        String sql = "UPDATE bookings SET sender_address = ?, receiver_address = ? WHERE booking_id = ?";

        try (Connection conn = DBConnection.getConnection();
             PreparedStatement pstmt = conn.prepareStatement(sql)) {

            pstmt.setString(1, pickup);
            pstmt.setString(2, dropoff);
            pstmt.setInt(3, bookingId);
            return pstmt.executeUpdate() > 0;

        } catch (SQLException e) {
            e.printStackTrace();
            return false;
        }
    }

    private Booking extractBooking(ResultSet rs) throws SQLException {
        Booking booking = new Booking();
        booking.setBookingId(rs.getInt("booking_id"));
        booking.setUserId(rs.getInt("user_id"));

        booking.setSenderAddress(rs.getString("sender_address"));
        booking.setReceiverName(rs.getString("receiver_name"));
        booking.setReceiverPhone(rs.getString("receiver_phone"));
        booking.setReceiverAddress(rs.getString("receiver_address"));
        booking.setWeight(rs.getDouble("weight"));
        booking.setParcelType(rs.getString("parcel_type"));
        booking.setPickupDate(rs.getTimestamp("pickup_date"));
        booking.setStatus(rs.getString("status"));
        booking.setCreatedAt(rs.getTimestamp("created_at"));
        return booking;
    }

1

Inserts a new booking with 11 parameters. Sets status to "PENDING" by default. Returns true/false for success/failure. Used when a customer submits the booking form.
2
getBookingById(int bookingId) → Booking or null
Fetches a single booking by its primary key. Used for the booking detail page, tracking page, and status updates.
3
getBookingsByUser(int userId) → List<Booking>
Gets all bookings for a specific user, ordered by newest first. Used on the "My Bookings" dashboard. Uses while loop because a user can have many bookings.
4
getBookingsByDateRange(Timestamp start, Timestamp end) → List<Booking>
Uses SQL BETWEEN to find bookings within a date range. Used for admin reports ("show all bookings from March 1-31"). Uses setTimestamp() for date parameters.
5
getAllBookings() → List<Booking>
Returns every booking in the system. No WHERE clause, no parameters. Used for the admin "All Bookings" view. Note: in production, you would add pagination (LIMIT/OFFSET).
6
updateStatus(int bookingId, String newStatus) → boolean
Changes a booking's status (PENDING → PICKED_UP → IN_TRANSIT → DELIVERED). Used by delivery agents and admins to track parcel progress.

updatePickupDrop(int bookingId, String pickup, String dropoff) → boolean
Updates the sender and receiver addresses. Used when a customer or admin corrects the address after booking.

Private helper that reads one row from the ResultSet and builds a Booking object. Called by every READ method. Centralizes the column → field mapping. If you add a column to the table, you update it in ONE place.

Common JDBC Errors

These are the errors you will encounter when working with JDBC. Know what they mean and how to fix them.

1. ClassNotFoundException: com.mysql.cj.jdbc.Driver

Error

java.lang.ClassNotFoundException: com.mysql.cj.jdbc.Driver

What it means: Java cannot find the MySQL driver class. The MySQL Connector JAR is not on the classpath.

Fix:

  • If using Maven: Add the mysql-connector-j dependency to pom.xml and run mvn clean install
  • If manual: Make sure the JAR file is in /lib and included in the classpath (-cp flag)
  • If in an IDE: Right-click the JAR → Add to Build Path

2. SQLException: Access denied for user 'root'@'localhost'

Error

java.sql.SQLException: Access denied for user 'root'@'localhost' (using password: YES)

What it means: Wrong username or password in your connection URL.

Fix:

  • Double-check the password in DBConnection.java
  • Test it manually: open MySQL terminal and login with the same credentials
  • If you reset your MySQL password, update it in DBConnection.java

3. CommunicationsException: Connection Refused

Error

com.mysql.cj.jdbc.exceptions.CommunicationsException: Communications link failure

What it means: Java cannot reach the MySQL server. Either MySQL is not running, or the port/host is wrong.

Fix:

  • Check if MySQL is running: sudo systemctl status mysql (Linux) or check Services on Windows
  • Verify the port: mysql --port=3306 (default). If you changed it, update the URL
  • If using a remote server, check firewall rules

4. SQLSyntaxErrorException: Table doesn't exist

Error

java.sql.SQLSyntaxErrorException: Table 'nimbus_db.bookingz' doesn't exist

What it means: The table name in your SQL does not match any table in the database. Typo or the table was never created.

Fix:

  • Check the exact table name: SHOW TABLES; in MySQL
  • SQL table names are case-sensitive on Linux, case-insensitive on Windows
  • Make sure you selected the right database: USE nimbus_db;

5. SQLIntegrityConstraintViolationException

Error

java.sql.SQLIntegrityConstraintViolationException: Duplicate entry 'user@email.com' for key 'users.email'

What it means: You tried to insert a row that violates a UNIQUE constraint. The email already exists in the table.

Fix:

  • Check before inserting: use isEmailUnique() in UserDAO
  • Or handle the exception gracefully and show "Email already registered" to the user

Quick Debugging Checklist

Symptom Check
ClassNotFoundException Is the MySQL JAR on the classpath?
Access denied Are username/password correct?
Connection refused Is MySQL running? Correct port?
Table doesn't exist Typo? Did you run CREATE TABLE?
Duplicate entry UNIQUE constraint violated. Check before inserting.
ResultSet is closed Are you using the ResultSet outside the try block?
Too many connections Are you closing connections? Use try-with-resources.
