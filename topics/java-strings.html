Section 01

String Creation — Literal vs new

Strings are the most used data type in any Java application. Every user ID, booking ID, email address, and status message in Nimbus is a String. Understanding how Java creates and stores them is the foundation for everything else on this page.

Two Ways to Create a String

Java gives you two ways to create a String, and they behave very differently in memory:

StringCreation.java 
// Way 1: String literal (uses the String Pool)
String status1 = "BOOKED";
String status2 = "BOOKED";

// Way 2: new keyword (creates a NEW object on the heap)
String status3 = new String("BOOKED");
String status4 = new String("BOOKED");
Line-by-line explanation
1
String status1 = "BOOKED";
Java checks the String Pool. "BOOKED" doesn't exist yet, so it creates it there. status1 points to that pool entry.
2
String status2 = "BOOKED";
Java checks the pool again. "BOOKED" already exists! So status2 points to the SAME object. No new object created.
3
String status3 = new String("BOOKED");
The new keyword FORCES Java to create a brand new object on the heap, even though "BOOKED" already exists in the pool. status3 points to this new copy.
4
String status4 = new String("BOOKED");
Another new = another separate object on the heap. status4 points to yet another copy. Three objects with the same content now exist.

The String Pool (String Intern Pool)

Java keeps a special area inside the Heap memory called the String Pool (also called the String Intern Pool). When you create a String using a literal (double quotes), Java first checks this pool. If the same text already exists, it reuses it. If not, it creates a new entry.

This is a performance optimization. Think about how many times your code uses strings like "BOOKED", "DELIVERED", "CANCELLED". Without the pool, every usage would create a new object. With the pool, all of them share one object.

                         HEAP MEMORY
    +---------------------------------------------------------+
    |                                                         |
    |    STRING POOL                                          |
    |    +-------------------------------------------+        |
    |    |                                           |        |
    |    |   [ "BOOKED" ]  <--- status1              |        |
    |    |        ^                                  |        |
    |    |        +-------- status2                  |        |
    |    |                                           |        |
    |    +-------------------------------------------+        |
    |                                                         |
    |    REGULAR HEAP                                         |
    |    +-------------------------------------------+        |
    |    |                                           |        |
    |    |   [ "BOOKED" ]  <--- status3              |        |
    |    |                                           |        |
    |    |   [ "BOOKED" ]  <--- status4              |        |
    |    |                                           |        |
    |    +-------------------------------------------+        |
    |                                                         |
    +---------------------------------------------------------+

    STACK
    +-------------------+
    | status1 ---> pool |      status1 and status2 point to
    | status2 ---> pool |      the SAME object in the pool.
    | status3 ---> heap |
    | status4 ---> heap |      status3 and status4 each point
    +-------------------+      to their OWN separate objects.
String Pool vs Heap — literal strings share pool entries, new String() always creates a separate heap object

This is why:

PoolDemo.java 
String a = "hello";
String b = "hello";
String c = new String("hello");
String d = new String("hello");

System.out.println(a == b);   // true  — same pool object
System.out.println(a == c);   // false — pool vs heap = different objects
System.out.println(c == d);   // false — two different heap objects
System.out.println(a.equals(c)); // true — same CONTENT
Key Takeaway

Always use string literals (double quotes) unless you have a specific reason to use new String(). Literals are faster (pool reuse) and use less memory. You will almost never need new String() in real code.

The intern() Method

If you have a String created with new, you can manually put it into the pool using .intern():

InternDemo.java 
String a = "BOOKED";
String b = new String("BOOKED");
String c = b.intern(); // puts "BOOKED" into pool, returns pool reference

System.out.println(a == b); // false
System.out.println(a == c); // true — c now points to the pool entry
Section 02

Immutability — Strings Cannot Be Changed

Once a String object is created, its content can never be modified. Every operation that appears to "change" a String actually creates a brand new String object. This is one of Java's most important design decisions.

What Immutability Means

When you do this:

ImmutabilityDemo.java 
String name = "Darshan";
name = name.toUpperCase(); // "DARSHAN"

You are not modifying the original "Darshan" object. You are creating a new String "DARSHAN" and making the name variable point to it. The original "Darshan" still exists in memory (until garbage collected).

    BEFORE:  name ---> [ "Darshan" ]

    AFTER:   name ---> [ "DARSHAN" ]   (NEW object)
                       [ "Darshan" ]   (still exists, no variable points to it)
The variable changed which object it points to. The object itself was never modified.

Every method that seems to modify a String — toUpperCase(), trim(), replace(), substring(), concat() — returns a new String. The original is untouched.

ImmutabilityProof.java 
String original = "  Nimbus Parcel  ";

// None of these change 'original'
original.trim();          // returns "Nimbus Parcel" but original is unchanged
original.toUpperCase();   // returns "  NIMBUS PARCEL  " but original is unchanged
original.replace("Nimbus", "Thunder"); // returns "  Thunder Parcel  "

System.out.println(original); // "  Nimbus Parcel  " — UNCHANGED!

// You have to ASSIGN the result to use it
String trimmed = original.trim();
System.out.println(trimmed); // "Nimbus Parcel"
Common Mistake

Beginners write str.trim(); or str.toUpperCase(); and expect the variable to change. It does not. You must assign the result: str = str.trim();

Why Did Java Make Strings Immutable?

Three critical reasons:

  1. Security — Strings are used for passwords, file paths, database connection URLs, class names. If someone could change a String after it was validated, they could bypass security checks. Imagine validating a file path as "/safe/folder/file.txt" then someone changes it to "/etc/passwd" before the file is actually opened.
  2. Thread Safety — Immutable objects are inherently thread-safe. Multiple threads can read the same String without locks or synchronization. No thread can change it, so no thread can corrupt it for another.
  3. Caching / String Pool — The String Pool only works because strings are immutable. If you could change "BOOKED" to "CANCELLED", every variable pointing to that pool entry would suddenly see "CANCELLED". Immutability makes pool sharing safe.
In Nimbus

When Nimbus stores a booking status like "BOOKED", it relies on immutability to guarantee that status value cannot be tampered with by another part of the code. The Booking class stores a String status field, and no external code can reach in and mutate that String object. They can only assign a new one through the setter, which triggers validation.

Section 03

== vs .equals() — The #1 Java Trap

This is the single most common mistake in Java. Every beginner makes it. Many experienced developers still slip up. Understand this section deeply and you will avoid a category of bugs that wastes hours of debugging time.

The Rule

Operator What It Checks Use For
== Reference equality — are these two variables pointing to the same object in memory? Primitives (int, char, boolean)
.equals() Content equality — do these two objects contain the same text? Strings and all objects
EqualsDemo.java 
String a = "BOOKED";
String b = "BOOKED";
String c = new String("BOOKED");

// == checks: are they the SAME OBJECT?
System.out.println(a == b);       // true  (same pool object)
System.out.println(a == c);       // false (different objects!)

// .equals() checks: do they have the SAME TEXT?
System.out.println(a.equals(b));  // true
System.out.println(a.equals(c)); // true  (same content, different objects)

Memory Diagram — Why == Fails

    STACK                          HEAP
    +-----------+
    |  a -------+---> [ "BOOKED" ]  (String Pool, address 0x100)
    |  b -------+---/
    |  c -------+---> [ "BOOKED" ]  (Regular Heap, address 0x200)
    +-----------+

    a == b   →  0x100 == 0x100  →  true   (same address)
    a == c   →  0x100 == 0x200  →  false  (different address!)

    a.equals(b) → compare characters: B-O-O-K-E-D == B-O-O-K-E-D → true
    a.equals(c) → compare characters: B-O-O-K-E-D == B-O-O-K-E-D → true
== compares arrows (references). .equals() compares the boxes (content).
In Nimbus

When checking if a booking status is "DELIVERED", you must use .equals(). The status might come from user input, a database query, or string concatenation — all of which can create new String objects. Using == would randomly fail.

NimbusStatusCheck.java 
// WRONG — will randomly fail depending on how status was created
if (booking.getStatus() == "DELIVERED") {
    System.out.println("Package delivered!");
}

// CORRECT — always works
if (booking.getStatus().equals("DELIVERED")) {
    System.out.println("Package delivered!");
}

// SAFEST — avoids NullPointerException if status is null
if ("DELIVERED".equals(booking.getStatus())) {
    System.out.println("Package delivered!");
}
Line-by-line explanation
1
booking.getStatus() == "DELIVERED"
Compares memory addresses. If getStatus() returns a String built at runtime (e.g., from DB), it will be a different object than the literal "DELIVERED", so this returns false even when the status IS "DELIVERED".
2
booking.getStatus().equals("DELIVERED")
Compares the actual characters. Works correctly. But if getStatus() returns null, calling .equals() on null throws NullPointerException.
3
"DELIVERED".equals(booking.getStatus())
The literal "DELIVERED" is never null, so .equals() is called on a non-null object. If getStatus() returns null, .equals() simply returns false instead of crashing. This is the defensive pattern.

equalsIgnoreCase()

When comparing strings where capitalization might vary (user input, for example):

IgnoreCaseDemo.java 
String userInput = "booked";
String expected = "BOOKED";

userInput.equals(expected);            // false
userInput.equalsIgnoreCase(expected);  // true

Java's String class has dozens of methods. These are the ones you will actually use. Every example uses Nimbus data so you can see exactly when each method matters.

Returns the character at the given index (0-based).

substring(int begin) and substring(int begin, int end)

Extracts a portion of the string. begin is inclusive, end is exclusive.

SubstringDemo.java 
String bookingId = "NMB-260401-0001";

// Extract the prefix
String prefix = bookingId.substring(0, 3);  // "NMB"

In Nimbus

The booking ID "NMB-260401-0001" is structured: prefix (NMB) + date (260401) + sequence (0001). Parsing these parts with substring() is how Nimbus extracts the booking date or sequence number from an ID without querying the database.

replace() takes literal strings. replaceAll() takes a regex pattern.

Section 05

String Concatenation

Joining strings together is something you do constantly. But how you do it matters for performance, especially in loops.

The + Operator

The simplest way. Works great for one-off concatenations.

ConcatPlus.java 
String firstName = "Darshan";
String lastName = "Mohan";

// Simple concatenation
String fullName = firstName + " " + lastName; // "Darshan Mohan"

// Concatenation with non-strings (auto-converted)
int age = 25;
String message = "Age: " + age; // "Age: 25" — int auto-converted to String

The concat() Method

ConcatMethod.java 
String greeting = "Hello, ".concat("Darshan"); // "Hello, Darshan"

// Less common than + operator. No advantage. Doesn't auto-convert types.
// "Age: ".concat(25); // COMPILE ERROR — concat only takes String

Why + in Loops is Bad

Because strings are immutable, every + creates a new String object. In a loop, this means creating and throwing away hundreds or thousands of objects:

BadLoop.java 
// BAD — creates 1000 intermediate String objects
String result = "";
for (int i = 0; i < 1000; i++) {
    result = result + i + ", ";  // Each + creates a NEW string
}
// Iteration 0: "" + "0" + ", " = "0, " (new object #1)
// Iteration 1: "0, " + "1" + ", " = "0, 1, " (new object #2)
// Iteration 2: "0, 1, " + "2" + ", " = "0, 1, 2, " (new object #3)
// ... 1000 objects created and thrown away!
    Loop iteration 0:  result = "" + "0" + ", "
                       Creates: ["0"], ["0, "]          2 objects

    Loop iteration 1:  result = "0, " + "1" + ", "
                       Creates: ["0, 1"], ["0, 1, "]    2 objects

    Loop iteration 2:  result = "0, 1, " + "2" + ", "
                       Creates: ["0, 1, 2"], ["0, 1, 2, "]  2 objects

    ...after 1000 iterations: ~2000 temporary String objects in memory!
    All discarded except the final one. Massive waste.
Each + operation creates a new String. Old ones become garbage. Use StringBuilder instead.
In Nimbus

Name concatenation: firstName + " " + lastName is fine — it is a one-time operation. But if you are building a report string for all 500 bookings in a loop, use StringBuilder (next section).

Section 06

StringBuilder — The Mutable String

StringBuilder is a mutable (changeable) sequence of characters. Unlike String, modifications happen in-place without creating new objects. Use it whenever you are building strings in loops or assembling strings from many parts.

Basic Usage

StringBuilderDemo.java 
StringBuilder sb = new StringBuilder();

sb.append("Booking: ");
sb.append("NMB-260401-0001");
sb.append(" | Status: ");
sb.append("BOOKED");

String result = sb.toString();
// "Booking: NMB-260401-0001 | Status: BOOKED"

Key Methods

StringBuilderMethods.java 
StringBuilder sb = new StringBuilder("Nimbus");

// append() — adds to the end
sb.append(" Parcel");            // "Nimbus Parcel"

// insert() — adds at a specific position
sb.insert(6, " Express");       // "Nimbus Express Parcel"

// delete() — removes a range (begin inclusive, end exclusive)
sb.delete(6, 14);               // "Nimbus Parcel"

// replace() — replaces a range with new text
sb.replace(7, 13, "Courier");   // "Nimbus Courier"

// reverse() — reverses the entire sequence
sb.reverse();                    // "reiruoC submiN"

// toString() — converts back to an immutable String
String final = sb.toString();

StringBuilder in Loops — The Right Way

GoodLoop.java 
// GOOD — ONE object, modified in-place
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i).append(", ");
}
String result = sb.toString();
// Only 1 StringBuilder object used, no waste

Performance Comparison

String + in Loop

1000 iterations with +:

  • ~2000 temporary objects
  • Each iteration copies all previous characters
  • Time complexity: O(n²)
  • Benchmark: ~50ms for 10,000 iterations
StringBuilder

1000 iterations with append():

  • 1 object, resized as needed
  • Each append only copies the new characters
  • Time complexity: O(n)
  • Benchmark: ~1ms for 10,000 iterations
In Nimbus

Building an invoice output for a booking with multiple items? StringBuilder. Assembling a formatted address from street, city, state, pin code? StringBuilder. Any time you are joining more than 3-4 parts or doing it in a loop, reach for StringBuilder.

Section 07

StringBuffer — Thread-Safe StringBuilder

StringBuffer has the exact same API as StringBuilder (append, insert, delete, reverse, toString) but every method is synchronized, meaning only one thread can use it at a time.

Feature StringBuilder StringBuffer
Mutable?YesYes
Thread-safe?NoYes (synchronized)
PerformanceFasterSlower (lock overhead)
When to use99% of the timeOnly when multiple threads share the same builder
IntroducedJava 5Java 1.0
Key Takeaway

Use StringBuilder by default. Use StringBuffer only if multiple threads will modify the same buffer simultaneously, which is rare. In Nimbus and most applications, StringBuilder is what you want.

Section 08

String.format() — Template Strings

Instead of messy concatenation with +, use String.format() to create formatted strings using placeholders. Cleaner, more readable, and supports number/date formatting.

Format Specifiers

Specifier Type Example Output
%sStringString.format("Hi %s", "Darshan")Hi Darshan
%dIntegerString.format("Count: %d", 42)Count: 42
%fFloat/DoubleString.format("Price: %.2f", 99.5)Price: 99.50
%04dZero-padded intString.format("%04d", 1)0001
%nNewline(platform-specific line break)
%%Literal %String.format("100%%")100%

Nimbus Examples

FormatNimbus.java 
// Generate user ID: YYMMDD-XX## format
String userId = String.format("%s-%s%02d", "260401", "DM", 1);
// "260401-DM01"

// Generate booking ID: NMB-YYMMDD-#### format
int sequenceNumber = 1;
String bookingId = String.format("NMB-%s-%04d", "260401", sequenceNumber);
// "NMB-260401-0001"

// Format invoice line
String invoiceLine = String.format(
    "%-30s %5d x Rs. %8.2f = Rs. %10.2f",
    "Express Delivery", 2, 150.00, 300.00
);
// "Express Delivery                  2 x Rs.   150.00 = Rs.     300.00"

// Format booking summary
String summary = String.format(
    "Booking %s | From: %s | To: %s | Status: %s",
    "NMB-260401-0001", "Mumbai", "Delhi", "BOOKED"
);
// "Booking NMB-260401-0001 | From: Mumbai | To: Delhi | Status: BOOKED"
Format specifier breakdown
1
%02d
%d = integer, 02 = pad with zeros to at least 2 digits. So 1 becomes "01", 9 becomes "09", 12 stays "12".
2
%04d
Pad with zeros to at least 4 digits. 1 becomes "0001", 42 becomes "0042", 1234 stays "1234".
3
%-30s
%s = string, 30 = minimum width, - = left-align. If the string is shorter than 30 chars, it pads with spaces on the right. Good for table formatting.
4
%8.2f
%f = float/double, 8 = minimum total width (including decimal point), .2 = exactly 2 decimal places. 150.0 becomes " 150.00".
In Nimbus

Nimbus uses String.format() heavily for ID generation. The booking ID format NMB-YYMMDD-#### needs zero-padded sequence numbers (%04d), and the user ID format YYMMDD-XX## needs zero-padded user numbers (%02d). Without format(), you would have to manually check digit count and prepend zeros.

Section 09

String Conversion — Parsing and Converting

Converting between Strings and other types (int, double, boolean) is something you do constantly when reading user input, parsing IDs, or formatting output.

String to Number

ParseDemo.java 
// String to int
String seqStr = "0001";
int seqNum = Integer.parseInt(seqStr);   // 1

// String to double
String priceStr = "149.99";
double price = Double.parseDouble(priceStr); // 149.99

// String to long
String trackingNum = "9876543210";
long tracking = Long.parseLong(trackingNum);

// String to boolean
String flag = "true";
boolean isActive = Boolean.parseBoolean(flag); // true

Number to String

ToStringDemo.java 
int count = 42;
double price = 149.99;

// Way 1: String.valueOf() (preferred)
String s1 = String.valueOf(count);     // "42"
String s2 = String.valueOf(price);     // "149.99"

// Way 2: .toString() on wrapper class
String s3 = Integer.toString(count);   // "42"
String s4 = Double.toString(price);    // "149.99"

// Way 3: Concatenation trick (works but less clear)
String s5 = "" + count;               // "42"

NumberFormatException — When Parsing Fails

ParseError.java 
// These all throw NumberFormatException:
Integer.parseInt("abc");     // not a number
Integer.parseInt("12.5");    // parseInt expects integer, not decimal
Integer.parseInt("");        // empty string
Integer.parseInt(null);      // null

// Always handle it:
String input = "abc";
try {
    int value = Integer.parseInt(input);
} catch (NumberFormatException e) {
    System.out.println("Invalid number: " + input);
}

// Nimbus example: parsing sequence number from booking ID
String bookingId = "NMB-260401-0001";
String seqPart = bookingId.substring(11); // "0001"
try {
    int seqNum = Integer.parseInt(seqPart);
    System.out.println("Sequence: " + seqNum); // 1
} catch (NumberFormatException e) {
    System.out.println("Invalid booking ID format");
}

Regular Expressions — Pattern Matching

Regular expressions (regex) let you define patterns to match, validate, search, and replace text. Java provides the Pattern and Matcher classes, plus built-in regex support in String methods like matches(), replaceAll(), and split().

Common Regex Syntax

Pattern Matches Example
.Any single character"c.t" matches "cat", "cut"
\dAny digit (0-9)"\d{4}" matches "0001"
\wWord character (letter, digit, _)"\w+" matches "hello_123"
\sWhitespace (space, tab, newline)"\s+" matches " "
[abc]Any character in the set"[aeiou]" matches any vowel
[^abc]Any character NOT in the set"[^0-9]" matches non-digits
*0 or more"ab*" matches "a", "ab", "abb"
+1 or more"ab+" matches "ab", "abb" (not "a")
?0 or 1"colou?r" matches "color", "colour"
{n}Exactly n times"\d{6}" matches "260401"
{n,m}Between n and m times"\d{8,10}" matches 8-10 digit numbers
^Start of string"^NMB" matches if starts with NMB
$End of string"com$" matches if ends with com

Email Validation

EmailValidation.java 
import java.util.regex.Pattern;
import java.util.regex.Matcher;

public class EmailValidation {

    // Regex: one or more word chars/dots/hyphens, then @,
    // then one or more word chars/dots, then . and 2-6 letters
    private static final String EMAIL_REGEX =
        "^[\\w.-]+@[\\w.-]+\\.[a-zA-Z]{2,6}$";

    private static final Pattern EMAIL_PATTERN =
        Pattern.compile(EMAIL_REGEX);

    public static boolean isValidEmail(String email) {
        if (email == null) return false;
        Matcher matcher = EMAIL_PATTERN.matcher(email);
        return matcher.matches();
    }

    public static void main(String[] args) {
        System.out.println(isValidEmail("darshan@nimbus.com"));  // true
        System.out.println(isValidEmail("user.name@mail.co.in")); // true
        System.out.println(isValidEmail("invalid@"));             // false
        System.out.println(isValidEmail("no-at-sign.com"));       // false
        System.out.println(isValidEmail(null));                    // false
    }
}

Password Validation

PasswordValidation.java 
public class PasswordValidation {

    public static boolean isStrongPassword(String password) {
        if (password == null || password.length() < 8) return false;

        // Check each requirement with regex
        boolean hasUpper = password.matches(".*[A-Z].*");
        boolean hasLower = password.matches(".*[a-z].*");
        boolean hasDigit = password.matches(".*\\d.*");
        boolean hasSpecial = password.matches(".*[!@#$%^&*()_+\\-=].*");

        return hasUpper && hasLower && hasDigit && hasSpecial;
    }

    public static void main(String[] args) {
        System.out.println(isStrongPassword("Nimbus@123"));  // true
        System.out.println(isStrongPassword("weakpass"));    // false
        System.out.println(isStrongPassword("NoDigit!"));    // false
    }
}

Pattern and Matcher (Full API)

PatternMatcherDemo.java 
import java.util.regex.*;

// Compile pattern once, reuse many times (better performance)
Pattern bookingPattern = Pattern.compile("NMB-(\\d{6})-(\\d{4})");
//                                       ^     ^group1    ^group2

String id = "NMB-260401-0001";
Matcher m = bookingPattern.matcher(id);

if (m.matches()) {
    System.out.println("Full match: " + m.group(0));  // "NMB-260401-0001"
    System.out.println("Date: " + m.group(1));         // "260401"
    System.out.println("Sequence: " + m.group(2));     // "0001"
}

// find() — searches anywhere in the string (not just full match)
String log = "Bookings: NMB-260401-0001, NMB-260401-0002, NMB-260402-0001";
Matcher finder = bookingPattern.matcher(log);
while (finder.find()) {
    System.out.println("Found: " + finder.group(0));
}
// Found: NMB-260401-0001
// Found: NMB-260401-0002
// Found: NMB-260402-0001

User ID Generation

NimbusUserIdGenerator.java 
/**
 * Generates a user ID in format: YYMMDD-XX##
 * Example: 260401-DM01
 *
 * YY     = last 2 digits of year (26 for 2026)
 * MM     = month (04 for April)
 * DD     = day (01)
 * XX     = first letters of first and last name
 * ##     = zero-padded sequence number
 */
public static String generateUserId(String firstName, String lastName,
                                     String joinDate, int sequenceNum) {
    // Extract date part: "2026-04-01" → "260401"
    String datePart = joinDate.substring(2, 4)   // "26" (year)
                    + joinDate.substring(5, 7)   // "04" (month)
                    + joinDate.substring(8, 10); // "01" (day)

    // Extract initials: "Darshan" → "D", "Mohan" → "M"
    String initials = firstName.substring(0, 1).toUpperCase()
                    + lastName.substring(0, 1).toUpperCase();

    // Format: YYMMDD-XX## with zero-padded sequence
    return String.format("%s-%s%02d", datePart, initials, sequenceNum);
    // "260401-DM01"
}
Line-by-line explanation
1
joinDate.substring(2, 4)
joinDate is "2026-04-01". Index 2 to 4 extracts "26" (the last two digits of the year).
2
joinDate.substring(5, 7)
Index 5 to 7 extracts "04" (the month). The dash at index 4 is skipped.
3
firstName.substring(0, 1).toUpperCase()
Takes the first character of the first name and uppercases it. "darshan" → "d" → "D".
4
String.format("%s-%s%02d", datePart, initials, sequenceNum)
%s inserts datePart ("260401"), %s inserts initials ("DM"), %02d formats sequenceNum (1 → "01"). Result: "260401-DM01".

Booking ID Parsing

NimbusBookingParser.java 
/**
 * Parses a Nimbus booking ID: "NMB-260401-0001"
 * Returns: prefix, date, and sequence number
 */
public static void parseBookingId(String bookingId) {
    // Validate format first
    if (bookingId == null || !bookingId.matches("NMB-\\d{6}-\\d{4}")) {
        throw new IllegalArgumentException("Invalid booking ID: " + bookingId);
    }

    // Split by dash
    String[] parts = bookingId.split("-");
    String prefix = parts[0];          // "NMB"
    String datePart = parts[1];        // "260401"
    int seqNum = Integer.parseInt(parts[2]); // 1

    // Parse date components
    String year = "20" + datePart.substring(0, 2);  // "2026"
    String month = datePart.substring(2, 4);         // "04"
    String day = datePart.substring(4, 6);           // "01"

    System.out.println("Prefix: " + prefix);
    System.out.println("Booked on: " + year + "-" + month + "-" + day);
    System.out.println("Sequence: " + seqNum);
}

Status Comparison (The Right Way)

NimbusStatusHandler.java 
public class NimbusStatusHandler {

    // Define status constants to avoid typos
    public static final String STATUS_BOOKED = "BOOKED";
    public static final String STATUS_IN_TRANSIT = "IN_TRANSIT";
    public static final String STATUS_DELIVERED = "DELIVERED";
    public static final String STATUS_CANCELLED = "CANCELLED";

    public static String getStatusMessage(String status) {
        // ALWAYS use .equals(), NEVER ==
        // Put the constant FIRST to avoid NullPointerException
        if (STATUS_BOOKED.equals(status)) {
            return "Your parcel has been booked and is awaiting pickup.";
        } else if (STATUS_IN_TRANSIT.equals(status)) {
            return "Your parcel is on its way!";
        } else if (STATUS_DELIVERED.equals(status)) {
            return "Your parcel has been delivered.";
        } else if (STATUS_CANCELLED.equals(status)) {
            return "This booking has been cancelled.";
        } else {
            return "Unknown status: " + status;
        }
    }
}

Password Sequential Character Check

NimbusPasswordValidator.java 
/**
 * Checks if password contains 3+ sequential characters.
 * "abc", "123", "xyz" are sequential — weak passwords.
 * Uses charAt() to compare consecutive character codes.
 */
public static boolean hasSequentialChars(String password) {
    if (password == null || password.length() < 3) return false;

    String lower = password.toLowerCase();
    for (int i = 0; i < lower.length() - 2; i++) {
        char c1 = lower.charAt(i);
        char c2 = lower.charAt(i + 1);
        char c3 = lower.charAt(i + 2);

        // Check ascending: a,b,c or 1,2,3
        if (c2 == c1 + 1 && c3 == c2 + 1) return true;

        // Check descending: c,b,a or 3,2,1
        if (c2 == c1 - 1 && c3 == c2 - 1) return true;
    }
    return false;
}

public static void main(String[] args) {
    System.out.println(hasSequentialChars("Nabc@99"));   // true  (a,b,c)
    System.out.println(hasSequentialChars("N123@bc"));    // true  (1,2,3)
    System.out.println(hasSequentialChars("Nxyz!01"));    // true  (x,y,z)
    System.out.println(hasSequentialChars("Nimbus@99"));  // false (no sequence)
    System.out.println(hasSequentialChars("Ncba!01"));    // true  (c,b,a descending)
}
How sequential detection works
1
char c1 = lower.charAt(i);
Characters have numeric values (ASCII). 'a'=97, 'b'=98, 'c'=99. So 'b' == 'a'+1 is true.
2
c2 == c1 + 1 && c3 == c2 + 1
If three consecutive chars are each +1 of the previous (a,b,c or 1,2,3), that is an ascending sequence. The password is weak.
3
c2 == c1 - 1 && c3 == c2 - 1
Also checks descending sequences like c,b,a or 9,8,7. Same logic, just subtract instead of add.

Address Formatting

NimbusAddressFormatter.java 
/**
 * Formats a full delivery address from individual components.
 * Handles null/blank fields gracefully.
 */
public static String formatAddress(String street, String city,
                                    String state, String pinCode) {
    StringBuilder sb = new StringBuilder();

    if (street != null && !street.isBlank()) {
        sb.append(street.trim());
    }
    if (city != null && !city.isBlank()) {
        if (sb.length() > 0) sb.append(", ");
        sb.append(city.trim());
    }
    if (state != null && !state.isBlank()) {
        if (sb.length() > 0) sb.append(", ");
        sb.append(state.trim());
    }
    if (pinCode != null && !pinCode.isBlank()) {
        if (sb.length() > 0) sb.append(" - ");
        sb.append(pinCode.trim());
    }

    return sb.toString();
}

// Usage:
String addr = formatAddress("12 MG Road", "Mumbai", "Maharashtra", "400001");
// "12 MG Road, Mumbai, Maharashtra - 400001"

String partial = formatAddress("  Sector 5  ", "Noida", null, "201301");
// "Sector 5, Noida - 201301"

Common Mistakes

These are the bugs that will eat your time if you do not know about them. Every single one of these has bitten real developers on real projects.

Mistake 1: Using == Instead of .equals()

Wrong
if (status == "BOOKED") {
    // May work with literals
    // FAILS with runtime strings
}
Correct
if ("BOOKED".equals(status)) {
    // Always works
    // Null-safe
}

Mistake 2: NullPointerException on null String

Crashes
String name = null;
name.equals("Darshan");
// NullPointerException!
Safe
String name = null;
"Darshan".equals(name);
// false (no crash)

Mistake 3: StringIndexOutOfBoundsException

IndexOutOfBounds.java 
String id = "NMB";

// CRASH: index 5 does not exist (max index = 2)
char c = id.charAt(5); // StringIndexOutOfBoundsException

// CRASH: substring end beyond length
String sub = id.substring(0, 10); // StringIndexOutOfBoundsException

// FIX: always check length first
if (id.length() > 5) {
    char c = id.charAt(5);
}

Mistake 4: Forgetting Strings Are Immutable

Does Nothing
String name = "  darshan  ";
name.trim();       // result discarded
name.toUpperCase(); // result discarded
// name is STILL "  darshan  "
Assign the Result
String name = "  darshan  ";
name = name.trim();
name = name.toUpperCase();
// name is now "DARSHAN"

Mistake 5: Concatenation in Loops

O(n²) Performance
String csv = "";
for (String s : list) {
    csv += s + ",";
}
O(n) Performance
StringBuilder sb = new StringBuilder();
for (String s : list) {
    sb.append(s).append(",");
}
String csv = sb.toString();
