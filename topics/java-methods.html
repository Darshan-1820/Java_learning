Section 01

What is a Method?

A method is a named, reusable block of code that performs a specific task. You define it once, then call it whenever you need that task done — without rewriting the same logic.

Analogy

A method is like a recipe. You write the recipe once (definition). Every time you want that dish, you follow the recipe (call the method). You can even adjust ingredients (pass different arguments). The recipe always gives you a result — the dish (return value).

Why Methods Exist

Without methods, your entire program would be one massive block of code in main(). Imagine writing Nimbus with 500 lines inside main — booking logic, login logic, history display, cost calculation, all tangled together. Every time you need to calculate cost in two places, you copy-paste the same 15 lines.

Methods solve this by letting you:

  • Reuse code — write once, call many times
  • Organize code — each method does one thing
  • Debug easier — if cost calculation is wrong, you fix one method, not 5 copies
  • Read easier — calculateCost(weight, distance) tells you what happens without reading the logic
In Nimbus

Every action in Nimbus is a method: calculateCost() computes shipping price, generateBookingId() creates unique IDs, validateLogin() checks credentials, addUser() saves a new user, getBookingsByUser() fetches bookings, displayHistory() prints them, extractBooking() reads from a ResultSet. All methods. All reusable.

Every method in Java has a precise structure. Each part tells the compiler something specific about what this method is and what it does.

MethodAnatomy.java 
// A complete method with every part labeled
public static double calculateCost(double weight, int distance) {

    double perKg = 10.0;
    double perKm = 5.0;
    double total = baseCost + (weight * perKg) + (distance * perKm);
    return total;  // sends the result back to whoever called this method

Method Signature vs Method Declaration

These terms get mixed up. The method signature is just the name + parameter types. The method declaration is the full first line including return type and modifiers.

Signature vs Declaration 
// Method declaration (full line):
public static double calculateCost(double weight, int distance)

// Method signature (what Java uses to tell methods apart):
// calculateCost(double, int)

// Two methods can have the same name if their signatures differ.
// This is called method overloading (covered in Section 05).
Section 03

Parameters vs Arguments

These two words sound similar but mean different things. Mixing them up is one of the most common beginner mistakes.

Parameter

The placeholder variable in the method definition. It is a name that will receive a value later.

Think of it as a blank on a form: "Name: ______"

void greet(String name) — name is a parameter.

Argument

The actual value you pass when calling the method. It fills the blank.

Think of it as the answer on the form: "Name: Darshan"

greet("Darshan") — "Darshan" is an argument.

ParamVsArg.java 
// DEFINITION — "weight" and "distance" are PARAMETERS
public static double calculateCost(double weight, int distance) {
    return 50 + (weight * 10) + (distance * 5);
}

// CALLING — 2.5 and 120 are ARGUMENTS
double cost = calculateCost(2.5, 120);
//                          ^^^  ^^^
//                      argument  argument

// The argument 2.5 gets copied into the parameter "weight"
// The argument 120 gets copied into the parameter "distance"
Remember It

Parameter = Placeholder (in the definition).
Argument = Actual value (when you call it).

Multiple Parameters & Order Matters

Arguments must match parameters in order, type, and count. If the method expects (String, int) and you pass (int, String), the compiler rejects it.

OrderMatters.java 
public static void addUser(String userId, String name, String password) {
    // ...
}

// Correct — matches order: String, String, String
addUser("USR001", "Darshan", "secret123");

// Wrong — even though all are Strings, the ORDER gives them meaning
addUser("secret123", "USR001", "Darshan");
// This compiles! But now the password is treated as userId. Silent bug.
Watch Out

When multiple parameters share the same type (like three Strings), the compiler cannot catch wrong order. The code compiles, but it does the wrong thing. This is why meaningful variable names and careful ordering matter so much.

Section 04

Return Types

Every method must declare what type of value it gives back. This is the return type — the contract between the method and its caller.

void — Returns Nothing

Some methods perform an action but do not give back a value. They print something, save to a database, or update a variable. These use void.

VoidMethod.java 
// void — this method does something but gives nothing back
public static void displayHistory(List<Booking> bookings) {
    System.out.println("=== Booking History ===");
    for (Booking b : bookings) {
        System.out.println(b.getBookingId() + " | " + b.getStatus());
    }
    // No return statement needed (but you CAN use "return;" to exit early)
}

// Calling a void method — you cannot store its result
displayHistory(myBookings);  // correct
// String x = displayHistory(myBookings);  // COMPILE ERROR — void has no value

Primitive Returns

Methods can return any primitive type: int, double, boolean, char, etc.

PrimitiveReturns.java 
// Returns double
public static double calculateCost(double weight, int distance) {
    return 50 + (weight * 10) + (distance * 5);
}

// Returns boolean
public static boolean validateLogin(String userId, String password) {
    // Check credentials against database
    User user = UserDAO.findById(userId);
    if (user == null) return false;
    return user.getPassword().equals(password);
}

// Returns int
public static int countBookings(String userId) {
    List<Booking> bookings = BookingDAO.getByUser(userId);
    return bookings.size();
}

Object Returns

Methods can return any object — String, array, your own custom class, even a List.

ObjectReturns.java 
// Returns String
public static String generateBookingId() {
    return "NB-" + System.currentTimeMillis();
}

// Returns a Booking object
public static Booking extractBooking(ResultSet rs) throws SQLException {
    Booking b = new Booking();
    b.setBookingId(rs.getString("booking_id"));
    b.setSenderId(rs.getString("sender_id"));
    b.setWeight(rs.getDouble("weight"));
    b.setStatus(rs.getString("status"));
    return b;  // returns the entire object
}

// Returns a List
public static List<Booking> getBookingsByUser(String userId) {
    List<Booking> bookings = new ArrayList<>();
    // ... fetch from database, add to list ...
    return bookings;
}

return Exits Immediately

The moment Java hits a return statement, the method stops. Any code after it is unreachable and causes a compile error.

ReturnExits.java 
public static boolean validateLogin(String userId, String password) {
    if (userId == null || password == null) {
        return false;  // exits HERE — nothing below runs
    }

    User user = UserDAO.findById(userId);
    if (user == null) {
        return false;  // exits HERE if user not found
    }

    return user.getPassword().equals(password);  // final return
}

// This is a common pattern: "early return" or "guard clause"
// It handles error cases first, so the main logic stays clean and unindented.
Key Rule

If a method declares a return type (not void), every possible code path must end with a return statement. If Java finds a path that might reach the end without returning, it gives a compile error: "missing return statement".

Section 05

Method Overloading

Java lets you create multiple methods with the same name as long as their parameter lists differ. This is called overloading — the compiler picks the right version based on the arguments you pass.

The Rules

  • Same method name
  • Different parameter list (different number, types, or order of parameters)
  • Return type alone is NOT enough to distinguish overloaded methods
In Nimbus

calculateCost is a perfect candidate for overloading. Sometimes you have just the weight. Sometimes weight + distance. Sometimes weight + distance + parcel type. Instead of three different method names, you use one name with different parameter combinations.

CostCalculator.java — Overloaded calculateCost 
public class CostCalculator {

    // Version 1: weight only — assumes default distance
    public static double calculateCost(double weight) {
        return 50 + (weight * 10);
    }

    // Version 2: weight + distance
    public static double calculateCost(double weight, int distance) {
        return 50 + (weight * 10) + (distance * 5);
    }

    // Version 3: weight + distance + parcel type
    public static double calculateCost(double weight, int distance, String type) {
        double base = 50 + (weight * 10) + (distance * 5);
        if ("EXPRESS".equals(type)) {
            base *= 1.5;  // 50% extra for express
        }
        return base;
    }
}

// Calling — Java picks the right version based on arguments
double c1 = CostCalculator.calculateCost(2.5);              // calls Version 1
double c2 = CostCalculator.calculateCost(2.5, 120);         // calls Version 2
double c3 = CostCalculator.calculateCost(2.5, 120, "EXPRESS"); // calls Version 3
How the compiler chooses
1
calculateCost(2.5)
One argument (double) → matches calculateCost(double weight). Only one method has exactly one double parameter.
2
calculateCost(2.5, 120)
Two arguments (double, int) → matches calculateCost(double weight, int distance). The compiler looks at parameter count AND types.
3
calculateCost(2.5, 120, "EXPRESS")
Three arguments (double, int, String) → matches the third version. All resolved at compile time — no guessing at runtime.

Return Type Alone is NOT Enough

InvalidOverload.java 
// THIS DOES NOT COMPILE
public static int getCount() { return 5; }
public static double getCount() { return 5.0; }

// Why? Because when you call getCount(), Java has no way to know
// which version you want. The parameters are identical (none).
// Return type is not part of the method signature.

Constructor Overloading

The same overloading concept applies to constructors. A class can have multiple constructors with different parameter lists.

Booking.java — Overloaded Constructors 
public class Booking {
    private String bookingId;
    private String senderId;
    private double weight;
    private String status;

    // Constructor 1: empty — for when you'll set fields later
    public Booking() {
        this.status = "PENDING";
    }

    // Constructor 2: with core fields
    public Booking(String bookingId, String senderId, double weight) {
        this.bookingId = bookingId;
        this.senderId = senderId;
        this.weight = weight;
        this.status = "PENDING";
    }

    // Constructor 3: all fields — used when reading from database
    public Booking(String bookingId, String senderId, double weight, String status) {
        this.bookingId = bookingId;
        this.senderId = senderId;
        this.weight = weight;
        this.status = status;
    }
}

// All valid:
Booking b1 = new Booking();
Booking b2 = new Booking("NB-001", "USR-001", 2.5);
Booking b3 = new Booking("NB-001", "USR-001", 2.5, "DELIVERED");
Section 06

Pass by Value

This is one of the most confusing topics in Java. The rule is simple but the implications trip up even experienced developers: Java ALWAYS passes by value. Always. No exceptions.

With Primitives: Copy of the Value

When you pass a primitive (int, double, etc.) to a method, Java copies the value. The method works on its own copy. Changing it inside the method has zero effect on the original.

PrimitivePassByValue.java 
public static void doubleIt(int x) {
    x = x * 2;   // modifies the LOCAL copy only
    System.out.println("Inside method: " + x);  // 20
}

public static void main(String[] args) {
    int num = 10;
    doubleIt(num);
    System.out.println("After method: " + num); // still 10!
}
  MEMORY (Primitives)

  main() stack frame:          doubleIt() stack frame:
  ┌─────────────────┐          ┌──────────────────┐
  │  num = 10       │ ──copy──>│  x = 10          │
  └─────────────────┘          │  x = 20  (changed)│
                               └──────────────────┘
                               (destroyed when method ends)

  After doubleIt() returns:
  ┌─────────────────┐
  │  num = 10       │  <── unchanged, because x was a COPY
  └─────────────────┘
Primitives: method gets its own copy, original untouched

With Objects: Copy of the REFERENCE

This is where it gets tricky. When you pass an object, Java does NOT pass the object itself. It passes a copy of the reference (the memory address). Both the original and the copy point to the same object in heap memory.

ObjectPassByValue.java 
public static void updateStatus(Booking b) {
    b.setStatus("DELIVERED");  // modifies the ACTUAL object via the reference copy
}

public static void replaceBooking(Booking b) {
    b = new Booking();  // reassigns the LOCAL copy of the reference
    b.setStatus("CANCELLED");
}

public static void main(String[] args) {
    Booking booking = new Booking("NB-001", "USR-001", 2.5);
    booking.setStatus("PENDING");

    updateStatus(booking);
    System.out.println(booking.getStatus()); // "DELIVERED" — changed!

    replaceBooking(booking);
    System.out.println(booking.getStatus()); // still "DELIVERED" — NOT cancelled!
}
  MEMORY (Objects)

  main() stack:                HEAP:
  ┌──────────────┐            ┌──────────────────────┐
  │ booking: 0x42│────────┐   │  Booking@0x42         │
  └──────────────┘        ├──>│  status = "PENDING"   │
                          │   └──────────────────────┘
  updateStatus() stack:   │
  ┌──────────────┐        │
  │ b: 0x42      │────────┘   Same object! b.setStatus("DELIVERED")
  └──────────────┘            changes the actual object.

  replaceBooking() stack:     HEAP:
  ┌──────────────┐            ┌──────────────────────┐
  │ b: 0x42      │──(was)──> │  Booking@0x42         │
  │ b: 0x99      │──(now)──> │  status = "DELIVERED" │
  └──────────────┘            └──────────────────────┘
  b = new Booking() makes    ┌──────────────────────┐
  b point to a NEW object    │  Booking@0x99         │
  in heap. The ORIGINAL      │  status = "CANCELLED" │
  booking variable in main   └──────────────────────┘
  still points to 0x42.       (this new object is lost
                               when replaceBooking ends)
Objects: reference is copied, but both point to the same heap object
The Bottom Line

You can modify an object's fields through a copied reference (because both references point to the same object). But you cannot make the caller's variable point to a different object (because reassigning the local reference only changes the copy).

Section 07

static Methods

A static method belongs to the class itself, not to any particular object. You call it using the class name, and you do not need to create an object first.

Calling static Methods

StaticMethods.java 
// Called via ClassName.methodName() — no object needed
double cost = CostCalculator.calculateCost(2.5, 120);
String id = BookingIdGenerator.generateBookingId();
Connection conn = DBConnection.getConnection();

// You do NOT do this:
// CostCalculator calc = new CostCalculator();
// calc.calculateCost(2.5, 120);  // works but unnecessary and misleading

Why Utility Methods Are static

Some methods do not need any object-level data. They take input, compute, and return a result. These are utility methods — pure functions that live on the class.

In Nimbus

DBConnection.getConnection() is static because you do not need a DBConnection object — you just need the connection. CostCalculator.calculateCost() is static because the formula does not depend on any object's state. InputValidator.isValidEmail() would be static — it just checks a string.

static Cannot Access Instance Members

A static method has no this — there is no object. So it cannot access instance variables or instance methods directly.

StaticRestrictions.java 
public class BookingService {
    private String serviceName = "Nimbus Express";  // instance variable

    // STATIC method
    public static double calculateCost(double weight) {
        // System.out.println(serviceName);  // COMPILE ERROR!
        // Static method cannot access instance variable.
        // There is no "this" — no object — so which serviceName?
        return 50 + (weight * 10);
    }

    // INSTANCE method — can access both static and instance members
    public void printServiceInfo() {
        System.out.println(serviceName);  // fine — this is an instance method
        double cost = calculateCost(2.5); // fine — instance can call static
    }
}

Why main() is static

When your program starts, no objects exist yet. The JVM needs a way to call your starting method without creating an object first. That is why main() must be public static void main(String[] args) — it is the entry point that runs before any objects are created.

Can a static method... Answer
Call another static method? Yes
Access a static variable? Yes
Access an instance variable? No (no this)
Call an instance method? No (unless it has an object reference)
Create an object and call its methods? Yes (that is how main works)
Section 08

Instance Methods

Instance methods belong to a specific object. They can access this — the object they were called on — and use its instance variables.

Booking.java — Instance Methods 
public class Booking {
    private String bookingId;
    private String status;
    private double weight;

    // Instance method — operates on THIS booking's data
    public String getSummary() {
        return this.bookingId + " | " + this.weight + "kg | " + this.status;
    }

    // Instance method — modifies THIS booking's state
    public void markDelivered() {
        this.status = "DELIVERED";
    }

    // Getters and setters are also instance methods
    public String getBookingId() { return this.bookingId; }
    public void setBookingId(String bookingId) { this.bookingId = bookingId; }
}

// Usage — you MUST create an object first
Booking b = new Booking("NB-001", "USR-001", 2.5);
String summary = b.getSummary();    // calls getSummary on THIS specific booking
b.markDelivered();                  // changes THIS booking's status

// Booking.getSummary();  // COMPILE ERROR — instance method needs an object

static vs Instance — Quick Comparison

static Method
  • Belongs to the class
  • Called via ClassName.method()
  • No this keyword
  • Cannot access instance variables
  • Use for: utility, helpers, factory methods
Instance Method
  • Belongs to an object
  • Called via object.method()
  • Has this — the current object
  • Can access everything (instance + static)
  • Use for: getters, setters, business logic on object data
Section 09

Variable Scope

Scope defines where a variable can be seen and used. A variable declared inside a method cannot be used outside it. A variable declared inside an if-block cannot be used outside that block. Understanding scope prevents countless bugs.

The Four Levels of Scope

Class / Static
Shared across all objects. Declared with static at the class level. Exists for the entire program. Example: static int totalBookings = 0;
Instance
One copy per object. Declared at the class level without static. Lives as long as the object. Example: private String bookingId;
Local
Inside a method. Born when the method is called, dies when the method returns. Not accessible from other methods. Example: double total = weight * rate;
Block
Inside { } of if, for, while, try. Only visible within those braces. Example: for (int i = 0; ...) — i does not exist outside the loop.
ScopeDemo.java 
public class BookingService {

    // CLASS/STATIC scope — shared by ALL BookingService objects
    private static int totalBookings = 0;

    // INSTANCE scope — each BookingService object has its own copy
    private String serviceName;

    public void processBooking(String userId) {
        // LOCAL scope — exists only during this method call
        double cost = CostCalculator.calculateCost(2.5, 120);

        if (cost > 100) {
            // BLOCK scope — exists only inside this if-block
            String message = "Premium booking!";
            System.out.println(message);  // OK
        }

        // System.out.println(message);  // COMPILE ERROR — message is block-scoped
        System.out.println(cost);         // OK — cost is local to this method

        totalBookings++;  // OK — static variable, accessible everywhere in the class
    }

    // System.out.println(cost);  // COMPILE ERROR — cost died when processBooking ended
}
Variable lifetimes
1
static int totalBookings = 0;
Class scope. Created when the class is loaded. Destroyed when the JVM shuts down. Same value shared by every object and every method in this class.
2
private String serviceName;
Instance scope. Created when new BookingService() is called. Destroyed when the object is garbage-collected. Each object has its own serviceName.
3
double cost = ...
Local scope. Created when processBooking() starts. Destroyed when processBooking() returns. Another method cannot see this variable.
4
String message = "Premium booking!";
Block scope. Created when the if block executes. Destroyed when the } of the if is reached. Cannot be used even one line after the closing brace.
Section 10

Variable Shadowing

When a local variable has the same name as an instance variable, the local variable "shadows" (hides) the instance variable. Java uses the nearest declaration.

Shadowing.java 
public class User {
    private String name;  // instance variable

    // The parameter "name" SHADOWS the instance variable "name"
    public void setName(String name) {
        // "name" here refers to the PARAMETER (nearest scope)
        // "this.name" refers to the INSTANCE variable
        this.name = name;
    }

    // Without this, you get a silent bug:
    public void setNameBuggy(String name) {
        name = name;  // assigns the parameter to itself!
        // The instance variable "this.name" is never updated.
        // No error. No warning. Just a bug that is hard to find.
    }
}
Common Beginner Bug

Forgetting this. in setters is one of the most common bugs in Java. The code compiles, runs, and appears to work — but the field never gets set. Always use this. when a parameter has the same name as an instance variable.

Shadowing in Constructors

ConstructorShadowing.java 
public class Booking {
    private String bookingId;
    private double weight;
    private String status;

    // Constructor — all parameters shadow instance variables
    public Booking(String bookingId, double weight, String status) {
        this.bookingId = bookingId;  // instance = parameter
        this.weight = weight;        // instance = parameter
        this.status = status;        // instance = parameter
    }
    // Without "this.", all three fields would remain null/0.0
}
Section 11

Varargs (Variable Arguments)

Sometimes you do not know how many arguments a method will receive. Varargs lets a method accept zero or more arguments of the same type, using the ... syntax.

VarargsDemo.java 
// String... names means: accept any number of String arguments
public static void printNames(String... names) {
    // Internally, "names" is a String[] array
    System.out.println("Received " + names.length + " names:");
    for (String name : names) {
        System.out.println("  - " + name);
    }
}

// All of these are valid calls:
printNames();                              // 0 arguments — empty array
printNames("Darshan");                     // 1 argument
printNames("Darshan", "Ravi", "Priya");    // 3 arguments
printNames(new String[]{"A", "B", "C"});   // pass an array directly

Rules for Varargs

  • Only one varargs parameter per method
  • Must be the LAST parameter
  • Inside the method, it behaves exactly like an array
VarargsRules.java 
// VALID — varargs is the last parameter
public static double totalWeight(String label, double... weights) {
    double total = 0;
    for (double w : weights) {
        total += w;
    }
    System.out.println(label + ": " + total + " kg");
    return total;
}

totalWeight("Batch A", 2.5, 3.0, 1.2);  // label="Batch A", weights=[2.5, 3.0, 1.2]

// INVALID — varargs not last
// public static void bad(double... weights, String label) {}  // COMPILE ERROR

// INVALID — two varargs
// public static void bad(int... a, String... b) {}  // COMPILE ERROR
Under the Hood

When you write method(String... names), the compiler converts it to method(String[] names). Varargs is syntactic sugar. At the call site, the compiler wraps your arguments into an array automatically. That is why you can also pass an actual array instead of individual arguments.

Section 12

Recursion

A recursive method is one that calls itself. It sounds circular, but it works because each call handles a smaller piece of the problem until it hits a base case — the stopping condition.

Analogy

Imagine you are standing in a line and want to know your position. You ask the person in front: "What is your position?" They ask the person in front of them. This keeps going until the person at the front says "I am #1." Then each answer comes back: "I am #2", "I am #3", etc. That is recursion — delegating to a smaller problem until you reach a known answer.

Factorial — The Classic Example

5! = 5 × 4 × 3 × 2 × 1 = 120. Recursively: 5! = 5 × 4!

Factorial.java 
public static int factorial(int n) {
    // BASE CASE — the stopping condition
    if (n <= 1) {
        return 1;
    }
    // RECURSIVE CASE — calls itself with a smaller problem
    return n * factorial(n - 1);
}

System.out.println(factorial(5));  // 120

// How it executes:
// factorial(5) = 5 * factorial(4)
// factorial(4) = 4 * factorial(3)
// factorial(3) = 3 * factorial(2)
// factorial(2) = 2 * factorial(1)
// factorial(1) = 1              <-- base case hit, start returning
// factorial(2) = 2 * 1 = 2
// factorial(3) = 3 * 2 = 6
// factorial(4) = 4 * 6 = 24
// factorial(5) = 5 * 24 = 120

The Two Essential Parts

Part What It Does What Happens Without It
Base Case Stops the recursion — returns a known value Infinite recursion → StackOverflowError
Recursive Case Calls itself with a smaller/simpler input Method never calls itself — it is just a normal method
StackOverflowError

Every method call gets a stack frame in memory. If a recursive method never hits its base case, it keeps creating frames until the stack runs out of space. Java throws StackOverflowError — the program crashes. Always make sure the input moves toward the base case on every recursive call.

When to Use Recursion vs Loops

  • Use loops when the problem is straightforward iteration (summing numbers, traversing an array)
  • Use recursion when the problem naturally breaks into smaller sub-problems (tree traversal, directory walking, parsing nested structures)
  • Loops are generally faster (no overhead of creating stack frames)
  • In business applications like Nimbus, you will rarely need recursion — but understanding it is essential for interviews and data structures

Full Nimbus Methods

Let us look at three real methods from Nimbus in detail — the kind of code you will write and read every day in the project.

calculateCost()

CostCalculator.java — Nimbus 
public class CostCalculator {
    private static final double BASE_CHARGE = 50.0;
    private static final double RATE_PER_KG = 10.0;
    private static final double RATE_PER_KM = 5.0;
    private static final double EXPRESS_MULTIPLIER = 1.5;

    /**
     * Calculates shipping cost based on weight, distance, and type.
     * @param weight  Parcel weight in kilograms
     * @param distance Distance in kilometers
     * @param type    "STANDARD" or "EXPRESS"
     * @return Total cost in rupees
     */
    public static double calculateCost(double weight, int distance, String type) {
        if (weight <= 0 || distance <= 0) {
            throw new IllegalArgumentException("Weight and distance must be positive");
        }

        double cost = BASE_CHARGE + (weight * RATE_PER_KG) + (distance * RATE_PER_KM);

        if ("EXPRESS".equalsIgnoreCase(type)) {
            cost *= EXPRESS_MULTIPLIER;
        }

        return Math.round(cost * 100.0) / 100.0;  // round to 2 decimal places
    }
}
calculateCost — line by line
1-4
private static final double BASE_CHARGE = 50.0;
Constants (static final) — values that never change. Named in UPPER_SNAKE_CASE by convention. Stored at class level, not inside the method.
5
public static double calculateCost(...)
public = callable from anywhere. static = no object needed. double = returns a decimal. Three parameters with clear names and types.
6-8
if (weight <= 0 || distance <= 0) throw ...
Input validation. Defensive programming — catch bad data immediately instead of silently computing garbage.
9
"EXPRESS".equalsIgnoreCase(type)
Calling equals on the constant string avoids NullPointerException. If type is null, type.equalsIgnoreCase("EXPRESS") would crash. This way is safe.
10
Math.round(cost * 100.0) / 100.0
Rounds to 2 decimal places. Multiply by 100, round to nearest integer, divide by 100. Standard technique for money calculations.

generateBookingId()

BookingIdGenerator.java — Nimbus 
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;

public class BookingIdGenerator {

    private static int counter = 0;
    private static final DateTimeFormatter FMT = DateTimeFormatter.ofPattern("yyyyMMdd");

    /**
     * Generates a unique booking ID in the format: NB-20260423-001
     * Thread-safety note: in a real app, use AtomicInteger or UUID.
     * @return A unique booking ID string
     */
    public static String generateBookingId() {
        counter++;
        String date = LocalDateTime.now().format(FMT);
        return String.format("NB-%s-%03d", date, counter);
    }
}
generateBookingId — line by line
1
private static int counter = 0;
static variable — shared across all calls. Each call increments it, so IDs are unique within one program run.
2
DateTimeFormatter.ofPattern("yyyyMMdd")
Formats today's date as "20260423". static final because the pattern never changes — no need to recreate it every call.
3
String.format("NB-%s-%03d", date, counter)
%s = string (the date). %03d = integer padded to 3 digits with leading zeros. Result: "NB-20260423-001".

extractBooking()

BookingDAO.java — extractBooking method 
/**
 * Converts one row of a ResultSet into a Booking object.
 * Called inside a while(rs.next()) loop — once per database row.
 * @param rs The ResultSet positioned at the current row
 * @return A fully populated Booking object
 * @throws SQLException if a column name is wrong or DB error occurs
 */
private static Booking extractBooking(ResultSet rs) throws SQLException {
    Booking booking = new Booking();
    booking.setBookingId(rs.getString("booking_id"));
    booking.setSenderId(rs.getString("sender_id"));
    booking.setReceiverName(rs.getString("receiver_name"));
    booking.setReceiverAddress(rs.getString("receiver_address"));
    booking.setWeight(rs.getDouble("weight"));
    booking.setDistance(rs.getInt("distance"));
    booking.setCost(rs.getDouble("cost"));
    booking.setStatus(rs.getString("status"));
    booking.setBookingDate(rs.getTimestamp("booking_date").toLocalDateTime());
    return booking;
}

// Usage — inside getBookingsByUser()
public static List<Booking> getBookingsByUser(String userId) throws SQLException {
    List<Booking> bookings = new ArrayList<>();
    String sql = "SELECT * FROM bookings WHERE sender_id = ?";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {

        ps.setString(1, userId);
        ResultSet rs = ps.executeQuery();

        while (rs.next()) {
            bookings.add(extractBooking(rs));  // reuse for every row
        }
    }
    return bookings;
}
extractBooking — why it exists
1
private static Booking extractBooking(ResultSet rs)
private = only used inside BookingDAO. static = no object needed. Returns a Booking object. Takes a ResultSet (database cursor) as input.
2
rs.getString("booking_id")
Reads the "booking_id" column from the current row. Different rs.getXxx() methods for different types: getString, getDouble, getInt, getTimestamp.
3
bookings.add(extractBooking(rs));
Called inside a loop. Without this method, the 10 lines of setter calls would be duplicated everywhere you read bookings from the database. Single Responsibility — extractBooking does one thing: convert a row to an object.
Section 14

Best Practices

Writing methods that work is the first step. Writing methods that are readable, maintainable, and debuggable is what separates professional code from hobby code.

1. Single Responsibility

A method should do one thing. If you can describe what it does and you use the word "and", it probably does too much.

Bad — Does Too Much
void processBooking(Booking b) {
    // validates
    // calculates cost
    // saves to DB
    // sends email
    // updates dashboard
}
Good — Each Does One Thing
void processBooking(Booking b) {
    validate(b);
    b.setCost(calculateCost(b));
    save(b);
    sendConfirmation(b);
}

2. Meaningful Names

Method names should be verbs or verb phrases that describe what they do. A reader should understand the method's purpose without reading its body.

Bad Name Good Name Why
doStuff() calculateShippingCost() Says exactly what it does
check() validateLogin() Check what? Validate is specific
get() getBookingsByUser() Get what? From where? For whom?
process() markAsDelivered() Process how? Mark as delivered is clear

3. Keep Methods Short

If a method is longer than 20-30 lines, it is probably doing too much. Extract sub-tasks into their own methods. Short methods are easier to test, easier to name, and easier to reuse.

4. Minimize Parameters

Three or fewer parameters is ideal. If you need more, consider grouping related parameters into an object.

TooManyParams.java 
// Bad — too many parameters, easy to mix up order
void createBooking(String senderId, String receiverName, String address,
                   double weight, int distance, String type, double cost) { ... }

// Better — group into an object
void createBooking(Booking booking) { ... }

// The Booking object carries all the data. Cleaner, safer, extensible.

5. Validate Inputs Early

Check parameters at the top of the method. Return or throw immediately if invalid. This is the guard clause pattern you saw earlier.

6. Prefer Returning Values Over Side Effects

A method that returns a value is easier to test than one that modifies external state. double cost = calculateCost(w, d) is safer than booking.cost = calculateAndSetCost(w, d, booking).
