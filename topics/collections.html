Section 01

The Collections Hierarchy

Java arrays have a fatal flaw: fixed size. You declare User[] users = new User[10] and you are stuck with 10 slots forever. What if 15 users register? What if only 2 do? The Java Collections Framework solves this — it gives you resizable, flexible, powerful data structures out of the box.

Analogy

Think of an array as a row of fixed chairs in a classroom — you set up exactly 30 chairs before anyone arrives. If 35 students show up, some stand. If only 10 show up, 20 chairs sit empty. A Collection is like a conference hall with stackable chairs — you add chairs as people arrive, remove them as people leave. The room adapts to the crowd.

The Full Family Tree

Every collection class in Java descends from one of two root interfaces: Collection (for groups of elements) and Map (for key-value pairs). Here is the complete hierarchy you need to know:

                            Iterable<E>
                                |
                          Collection<E>
                         /      |       \
                        /       |        \
                   List<E>   Set<E>    Queue<E>
                  /  |  \     /  |  \
                 /   |   \   /   |   \
          ArrayList  |  Vector  |  TreeSet
                     |       HashSet  |
                 LinkedList      LinkedHashSet


                           Map<K,V>
                          /    |    \
                         /     |     \
                   HashMap  TreeMap  LinkedHashMap
Java Collections Framework hierarchy — the two family trees

Let us break this down:

Interface What It Stores Key Property Common Implementations
List Ordered elements Duplicates allowed, index-based access ArrayList, LinkedList, Vector
Set Unique elements No duplicates, no index HashSet, TreeSet, LinkedHashSet
Queue Elements in FIFO order First-in, first-out processing LinkedList, PriorityQueue
Map Key-value pairs Unique keys, each maps to one value HashMap, TreeMap, LinkedHashMap
In Nimbus

Right now, when you query all users from the database, you get them one at a time from ResultSet. You need somewhere to put them. An array? You do not know how many users exist. That is why you use ArrayList<User> — it grows as you add each user from the ResultSet. When you search bookings, the result goes into ArrayList<Booking>. When you want to look up a booking instantly by ID, you use HashMap<String, Booking>.

Why Not Just Use Arrays?

Arrays
  • Fixed size — cannot grow or shrink
  • No built-in search, sort, or remove
  • Manual index management
  • No type safety without generics
  • Inserting in the middle requires shifting everything
Collections
  • Dynamic size — grows and shrinks automatically
  • Built-in search, sort, remove, contains
  • Iterator for safe traversal
  • Full generics support: ArrayList<User>
  • Specialized structures for every use case
Section 02

ArrayList — The Workhorse

ArrayList is the collection you will use 80% of the time. It is a resizable array — it acts like an array but grows automatically when you add elements.

How ArrayList Works Internally

Under the hood, ArrayList is just a regular array. When you create new ArrayList<>(), Java creates an internal array of size 10 (the default capacity). As you add elements:

  Step 1: new ArrayList<String>()
  Internal array: [null, null, null, null, null, null, null, null, null, null]
  Capacity: 10    Size: 0

  Step 2: add("Alice"), add("Bob"), add("Charlie") ... add 10 elements
  Internal array: [Alice, Bob, Charlie, Dave, Eve, Frank, Grace, Hank, Ivy, Jack]
  Capacity: 10    Size: 10    <-- FULL!

  Step 3: add("Kate") — no room!
  Java creates a NEW array with capacity = old * 1.5 = 15
  Copies all 10 elements to the new array, then adds "Kate"
  Internal array: [Alice, Bob, Charlie, Dave, Eve, Frank, Grace, Hank, Ivy, Jack, Kate, null, null, null, null]
  Capacity: 15    Size: 11

  This "grow and copy" is called RESIZING.
  It is O(n) when it happens, but it happens rarely, so
  on average (amortized), add() is still O(1).
ArrayList internal resizing: capacity grows by 50% each time it fills up
Analogy

Imagine a notebook with 10 pages. You fill all 10. To add more notes, you buy a bigger notebook (15 pages), copy everything from the old one, throw the old one away, and continue writing. That "copy to a bigger notebook" is expensive — but you do not do it every single time you write a note. You only do it when the notebook is full.

Every ArrayList Method You Need

NimbusUserList.java 
import java.util.ArrayList;

public class NimbusUserList {
    public static void main(String[] args) {
        // CREATE — type-safe list of Users
        ArrayList<User> users = new ArrayList<>();

        // ADD — append to end (O(1) amortized)
        users.add(new User("U001", "Darshan", "darshan@nimbus.com"));
        users.add(new User("U002", "Priya", "priya@nimbus.com"));
        users.add(new User("U003", "Rahul", "rahul@nimbus.com"));

        // ADD at index — insert at position (O(n), shifts elements right)
        users.add(1, new User("U004", "Meera", "meera@nimbus.com"));
        // List is now: [Darshan, Meera, Priya, Rahul]

        // GET — access by index (O(1), instant)
        User firstUser = users.get(0);  // Darshan
        System.out.println("First user: " + firstUser.getName());

        // SET — replace element at index (O(1))
        users.set(2, new User("U005", "Anita", "anita@nimbus.com"));
        // Priya replaced with Anita at index 2

        // SIZE — how many elements
        System.out.println("Total users: " + users.size());  // 4

        // CONTAINS — check if element exists (O(n), uses equals())
        boolean hasDarshan = users.contains(new User("U001", "Darshan", "darshan@nimbus.com"));

        // INDEXOF — find position of element (O(n))
        int pos = users.indexOf(new User("U001", "Darshan", "darshan@nimbus.com"));
        // Returns -1 if not found

        // REMOVE by index (O(n), shifts elements left)
        users.remove(3);  // Removes Rahul at index 3

        // REMOVE by object (O(n), finds and removes first match)
        users.remove(new User("U004", "Meera", "meera@nimbus.com"));

        // ISEMPTY — check if list has zero elements
        if (!users.isEmpty()) {
            System.out.println("We have " + users.size() + " users");
        }

        // CLEAR — remove ALL elements
        // users.clear();  // List is now empty, size = 0

        // LOOP through all users
        for (User u : users) {
            System.out.println(u.getUserId() + ": " + u.getName());
        }
    }
}
Line-by-line explanation
1
ArrayList<User> users = new ArrayList<>();
Creates an empty ArrayList that can ONLY hold User objects. The <> on the right is the diamond operator — Java infers the type from the left side.
2
users.add(new User("U001", "Darshan", "darshan@nimbus.com"));
Creates a new User and appends it to the end of the list. ArrayList grows automatically if needed. This is O(1) amortized.
3
users.add(1, new User("U004", "Meera", ...));
Inserts Meera at index 1. Everything from index 1 onward shifts right by one position. This is O(n) because of the shifting.
4
User firstUser = users.get(0);
Direct index access, just like an array. O(1) — instant, no matter how big the list is. This is ArrayList's biggest advantage.
5
users.set(2, new User("U005", "Anita", ...));
Replaces the element at index 2 (Priya) with Anita. Does NOT shift anything — just overwrites that slot. O(1).
6
users.contains(new User("U001", ...));
Walks through the entire list calling equals() on each element. Returns true if any match. O(n). Requires User to override equals().
7
users.remove(3);
Removes the element at index 3. Everything after index 3 shifts left. O(n) in the worst case.

ArrayList Performance Summary

Operation Method Time Complexity Why
Access by index get(i) O(1) Direct array index lookup
Add to end add(e) O(1)* Append to end (amortized, occasional resize)
Add at beginning add(0, e) O(n) Must shift every element right by one
Remove from end remove(size-1) O(1) No shifting needed
Remove from beginning remove(0) O(n) Must shift every element left by one
Search contains(e) O(n) Must check each element sequentially
Set/replace set(i, e) O(1) Direct overwrite at index
In Nimbus

Your UserDAO.getAllUsers() method returns an ArrayList<User>. Inside the method, you loop through the ResultSet and call users.add(user) for each row. Since you are always adding to the end, this is O(1) per add. Later, when you display the user list, you use users.get(i) which is also O(1). ArrayList is perfect for this — build a list by appending, then access by index.

Section 03

LinkedList — The Chain

A LinkedList stores elements as nodes. Each node holds the data plus pointers to the previous and next node. No internal array, no resizing, no shifting.

How LinkedList Works Internally

  ArrayList (under the hood):
  [Darshan | Priya | Rahul | Meera | null | null | null]
   index 0   index 1  index 2  index 3
   Continuous block of memory. Access any index instantly.

  LinkedList (under the hood):
  HEAD
   |
   v
  [prev:null | "Darshan" | next] --> [prev | "Priya" | next] --> [prev | "Rahul" | next] --> [prev | "Meera" | next:null]
                                                                                                         ^
                                                                                                        TAIL

  Each box is a NODE. Each node knows its neighbors but NOT its position.
  To reach index 2, you must walk: HEAD → node 0 → node 1 → node 2.
ArrayList = contiguous array vs LinkedList = chain of nodes
Analogy

Imagine a paper chain decoration. Each link is connected to the next by hooking through it. To add a new link at the start, you just hook it in — instant, nothing else moves. But to find the 50th link, you have to count from the beginning: 1, 2, 3, ... 50. Compare that to a bookshelf (ArrayList) where you can grab book #50 directly because they are numbered and in order.

ArrayList vs LinkedList — When to Use Which

ArrayList
  • Fast random access — get(50) is O(1)
  • Slow insert/remove at beginning — O(n) shifting
  • Memory efficient — just stores data, no pointers
  • Use when: you read more than you insert/delete
  • Great for: displaying user lists, search results
LinkedList
  • Slow random access — get(50) is O(n)
  • Fast insert/remove at ends — O(1)
  • More memory — each node stores 2 extra pointers
  • Use when: you insert/delete at ends frequently
  • Great for: queues, deques, undo histories
Operation ArrayList LinkedList Winner
get(index) O(1) O(n) ArrayList
add(element) — to end O(1)* O(1) Tie
add(0, element) — to beginning O(n) O(1) LinkedList
remove(0) — from beginning O(n) O(1) LinkedList
remove(size-1) — from end O(1) O(1) Tie
contains(element) O(n) O(n) Tie
Memory overhead Low High (2 pointers per node) ArrayList
In Nimbus

For Nimbus, you almost always want ArrayList. Why? Because you mostly read data from the database and display it. You call get(i) far more than you insert at the beginning. LinkedList would be useful if Nimbus had an undo feature (last-in, first-out) or a parcel processing queue (first-in, first-out). For user lists and booking results — ArrayList wins.

LinkedListAsQueue.java 
import java.util.LinkedList;

// LinkedList as a parcel processing queue
LinkedList<String> processingQueue = new LinkedList<>();

// Parcels arrive (add to end)
processingQueue.addLast("PKG-001");  // O(1)
processingQueue.addLast("PKG-002");  // O(1)
processingQueue.addLast("PKG-003");  // O(1)

// Process parcels in order (remove from front)
String next = processingQueue.removeFirst();  // "PKG-001" — O(1)
// No shifting! Just update the HEAD pointer.

// Peek at next without removing
String upcoming = processingQueue.peekFirst();  // "PKG-002"
Section 04

HashSet — Uniqueness Enforced

A HashSet stores elements with no duplicates and no ordering. It uses hashing internally to give you O(1) add, remove, and contains operations.

How Hashing Works (Simplified)

  When you add "darshan@nimbus.com" to a HashSet:

  Step 1: Java calls "darshan@nimbus.com".hashCode()
          Returns a number like 184927633

  Step 2: Java computes: 184927633 % 16 = 1
          (16 is the default number of internal "buckets")

  Step 3: Java puts the element in bucket #1

  BUCKET ARRAY:
  [0] --> empty
  [1] --> "darshan@nimbus.com"
  [2] --> empty
  [3] --> "priya@nimbus.com"
  [4] --> empty
  ...
  [15] --> "rahul@nimbus.com"

  When you call contains("darshan@nimbus.com"):
  Step 1: hashCode() → 184927633
  Step 2: 184927633 % 16 = 1
  Step 3: Look in bucket #1. Found it! O(1).

  No need to search through the entire collection.
Hashing: convert the element to a number, use it as an index into buckets
Analogy

Imagine a library with 26 shelves labeled A through Z. When a book arrives, you look at the author's last name and put it on the corresponding shelf — "Sharma" goes on shelf S. To find a book by Sharma, you go straight to shelf S. You do not search all 26 shelves. That is hashing — a formula that tells you exactly where to look.

HashSet in Action — Tracking Unique Emails

UniqueEmailCheck.java 
import java.util.HashSet;

public class RegistrationValidator {
    // Track all registered emails to prevent duplicates
    private HashSet<String> registeredEmails = new HashSet<>();

    public boolean registerUser(String email) {
        // add() returns false if element already exists
        if (!registeredEmails.add(email.toLowerCase())) {
            System.out.println("Email already registered: " + email);
            return false;
        }
        System.out.println("Registration successful: " + email);
        return true;
    }

    public boolean isEmailTaken(String email) {
        // contains() is O(1) with hashing!
        return registeredEmails.contains(email.toLowerCase());
    }

    public int totalRegistered() {
        return registeredEmails.size();
    }

    public static void main(String[] args) {
        RegistrationValidator validator = new RegistrationValidator();

        validator.registerUser("darshan@nimbus.com");   // Success
        validator.registerUser("priya@nimbus.com");     // Success
        validator.registerUser("darshan@nimbus.com");   // FAILS — duplicate!

        System.out.println("Total: " + validator.totalRegistered());  // 2, not 3

        // Iterating — order is NOT guaranteed
        for (String email : validator.registeredEmails) {
            System.out.println(email);
        }
        // Could print in ANY order — that is HashSet.
    }
}

HashSet vs TreeSet vs LinkedHashSet

Set Type Ordering Performance When to Use
HashSet No order guaranteed O(1) add/remove/contains Default choice for uniqueness checks
TreeSet Sorted (natural or custom) O(log n) add/remove/contains When you need elements sorted (e.g., sorted city names)
LinkedHashSet Insertion order preserved O(1) add/remove/contains When you need uniqueness AND insertion order
In Nimbus

During user registration, you need to check if an email is already taken. With an ArrayList, you would call contains(email) which loops through every element — O(n). With 10,000 users, that means checking up to 10,000 entries. With a HashSet, the same contains(email) call is O(1) — it hashes the email, goes straight to the right bucket, done. For duplicate-checking, HashSet is orders of magnitude faster.

Section 05

HashMap — Instant Lookup by Key

A HashMap stores key-value pairs. You give it a key, it gives you back the value — in O(1) time. Think of it as a dictionary: you look up a word (key) and instantly find its definition (value).

How HashMap Works

  HashMap<String, Booking> bookingMap

  PUT: bookingMap.put("BK-1001", booking1)
  Step 1: "BK-1001".hashCode() → 7482910
  Step 2: 7482910 % 16 = 14
  Step 3: Store {key:"BK-1001", value:booking1} in bucket #14

  BUCKET ARRAY:
  [0]  --> empty
  [1]  --> empty
  [2]  --> {key:"BK-1003", value:booking3}
  ...
  [7]  --> {key:"BK-1002", value:booking2}
  ...
  [14] --> {key:"BK-1001", value:booking1}
  [15] --> empty

  GET: bookingMap.get("BK-1001")
  Step 1: Hash "BK-1001" → bucket #14
  Step 2: Found! Return booking1.     O(1)

  COLLISION: Two keys hash to same bucket? Java chains them in a list:
  [14] --> {BK-1001, booking1} --> {BK-2045, booking45}
  Then it uses equals() to find the right one.
HashMap: hash the key to find which bucket, then store key-value pair there

Complete HashMap Example — Booking Lookup

BookingLookupService.java 
import java.util.HashMap;
import java.util.Map;

public class BookingLookupService {
    // Key = booking ID, Value = Booking object
    private HashMap<String, Booking> bookingCache = new HashMap<>();

    // Load bookings into the map for instant lookup
    public void loadBookings(ArrayList<Booking> bookingsFromDB) {
        for (Booking b : bookingsFromDB) {
            bookingCache.put(b.getBookingId(), b);
        }
    }

    // O(1) lookup by booking ID — instant!
    public Booking findBooking(String bookingId) {
        return bookingCache.get(bookingId);  // null if not found
    }

    // Check if a booking exists
    public boolean bookingExists(String bookingId) {
        return bookingCache.containsKey(bookingId);
    }

    // ALL keys
    public void printAllBookingIds() {
        for (String id : bookingCache.keySet()) {
            System.out.println("Booking ID: " + id);
        }
    }

    // ALL values
    public void printAllBookings() {
        for (Booking b : bookingCache.values()) {
            System.out.println(b.getBookingId() + " | " + b.getSenderName());
        }
    }

    // ALL key-value pairs (entrySet is the most efficient way to iterate)
    public void printFullDetails() {
        for (Map.Entry<String, Booking> entry : bookingCache.entrySet()) {
            String id = entry.getKey();
            Booking b = entry.getValue();
            System.out.println(id + " → " + b.getSenderName()
                + " | Rs. " + b.getCost() + " | " + b.getStatus());
        }
    }

    // Remove a cancelled booking
    public void cancelBooking(String bookingId) {
        Booking removed = bookingCache.remove(bookingId);
        if (removed != null) {
            System.out.println("Cancelled: " + bookingId);
        }
    }

    // Total bookings
    public int totalBookings() {
        return bookingCache.size();
    }

    // Check if cache is empty
    public boolean isEmpty() {
        return bookingCache.isEmpty();
    }
}
Key methods explained
1
put(key, value)
Stores a key-value pair. If the key already exists, the old value is REPLACED and returned. O(1).
2
get(key)
Returns the value associated with the key. Returns null if the key does not exist. O(1).
3
containsKey(key)
Returns true if this map has an entry for the given key. Faster than checking get() != null because some values CAN be null. O(1).
4
keySet()
Returns a Set of all keys. Useful when you only need the keys (like all booking IDs).
5
values()
Returns a Collection of all values. Useful when you only need the values (like all Booking objects).
6
entrySet()
Returns a Set of Map.Entry<K,V> pairs. MOST EFFICIENT way to iterate when you need both key and value — only one lookup per entry.
In Nimbus

When a customer enters booking ID "BK-1001" to track their parcel, you need to find that specific booking. With an ArrayList<Booking>, you would loop through ALL bookings comparing IDs — O(n). With 50,000 bookings, that is 50,000 comparisons. With a HashMap<String, Booking>, you hash "BK-1001" and jump straight to the answer — O(1). One lookup. Done. This is why real systems use HashMaps for lookups.

Section 06

TreeMap — Sorted Keys

A TreeMap keeps its keys in sorted order automatically. Every time you put a new entry, it slots it into the right position. It uses a Red-Black Tree internally (a self-balancing binary search tree).

HashMap vs TreeMap

HashMap
  • O(1) put/get/remove
  • No ordering of keys
  • Allows one null key
  • Use for: fast lookup by key
TreeMap
  • O(log n) put/get/remove
  • Keys always sorted
  • No null keys allowed
  • Use for: sorted display, range queries
SortedBookingDisplay.java 
import java.util.TreeMap;
import java.time.LocalDate;

// TreeMap sorts by key automatically
// Key = booking date, Value = list of bookings on that date
TreeMap<LocalDate, ArrayList<Booking>> bookingsByDate = new TreeMap<>();

// Add bookings (inserted in sorted order automatically)
bookingsByDate.put(LocalDate.of(2026, 4, 23), aprilBookings);
bookingsByDate.put(LocalDate.of(2026, 3, 15), marchBookings);
bookingsByDate.put(LocalDate.of(2026, 5, 1), mayBookings);

// Iterating — always in date order (earliest first)
for (Map.Entry<LocalDate, ArrayList<Booking>> entry : bookingsByDate.entrySet()) {
    System.out.println("Date: " + entry.getKey());
    for (Booking b : entry.getValue()) {
        System.out.println("  " + b.getBookingId());
    }
}
// Output: March 15, then April 23, then May 1

// NavigableMap methods:
LocalDate firstDate = bookingsByDate.firstKey();     // 2026-03-15
LocalDate lastDate  = bookingsByDate.lastKey();      // 2026-05-01

// Get all bookings between two dates (inclusive)
Map<LocalDate, ArrayList<Booking>> aprilBookingsRange =
    bookingsByDate.subMap(
        LocalDate.of(2026, 4, 1), true,   // from (inclusive)
        LocalDate.of(2026, 4, 30), true    // to (inclusive)
    );

// Get all bookings before a date
Map<LocalDate, ArrayList<Booking>> beforeApril =
    bookingsByDate.headMap(LocalDate.of(2026, 4, 1), false);

// Get the booking date just before a given date
LocalDate previousDate = bookingsByDate.lowerKey(LocalDate.of(2026, 4, 23));
// Returns 2026-03-15
In Nimbus

When displaying booking history for a customer, you want bookings sorted by date — most recent first. A TreeMap<LocalDate, List<Booking>> gives you this automatically. You can also answer questions like "show me all bookings in April" using subMap(), or "what was the last booking before today" using lowerKey(). HashMap cannot do any of this — it has no concept of order.

Section 07

Iterator — Safe Traversal

An Iterator is a cursor that walks through a collection one element at a time. Its superpower: it lets you remove elements safely while iterating — something a regular for-each loop cannot do.

The ConcurrentModificationException Problem

DangerousRemoval.java 
ArrayList<Booking> bookings = getBookings();

// THIS CRASHES with ConcurrentModificationException!
for (Booking b : bookings) {
    if (b.getStatus().equals("CANCELLED")) {
        bookings.remove(b);  // Modifying the list WHILE iterating
    }
}
// Java detects that the list changed during iteration and throws an exception.
// The for-each loop uses an iterator internally, and the iterator's
// expected modification count no longer matches the list's actual count.
Why This Crashes

A for-each loop secretly creates an Iterator behind the scenes. That iterator keeps a count of how many times the list has been modified. When you call bookings.remove(b) directly, the list's modification count increases but the iterator does not know. On the next hasNext() call, the iterator sees the mismatch and throws ConcurrentModificationException. It is a safety mechanism to prevent corrupted iteration.

The Solution: Use Iterator.remove()

SafeRemoval.java 
import java.util.Iterator;
import java.util.ArrayList;

ArrayList<Booking> bookings = getBookings();

// SAFE removal using Iterator
Iterator<Booking> iterator = bookings.iterator();

while (iterator.hasNext()) {
    Booking b = iterator.next();       // Move to next element

    if (b.getStatus().equals("CANCELLED")) {
        iterator.remove();             // Remove via iterator — SAFE!
        System.out.println("Removed cancelled booking: " + b.getBookingId());
    }
}

// The iterator updates its internal state when YOU remove through it,
// so there is no mismatch. No exception. Clean removal.
Iterator methods
1
bookings.iterator()
Creates an Iterator positioned BEFORE the first element. Think of it as a cursor sitting at position -1.
2
iterator.hasNext()
Returns true if there are more elements after the cursor's current position. Used as the loop condition.
3
iterator.next()
Advances the cursor by one position and returns the element it just passed. You MUST call next() before remove().
4
iterator.remove()
Removes the element that was last returned by next(). This is the ONLY safe way to remove during iteration. You can only call it once per next() call.

For-Each vs Iterator — When to Use Which

Situation Use Why
Just reading all elements For-each loop Cleaner syntax, fewer lines of code
Need to remove during iteration Iterator Only Iterator.remove() is safe
Need the index while iterating Regular for loop for (int i = 0; i < list.size(); i++)
In Nimbus

When cleaning up expired bookings from a list, you need to iterate through all bookings and remove the expired ones. Use an Iterator for this. When simply displaying booking history to the user, a plain for-each loop is cleaner. Know both, use the right one.

Section 08

Comparable — Natural Ordering

When you call Collections.sort(bookings), Java needs to know: how do you compare two Bookings? By date? By cost? By ID? The Comparable interface lets YOUR class define its own natural ordering.

Implementing Comparable<Booking>

Booking.java 
import java.time.LocalDate;

public class Booking implements Comparable<Booking> {
    private String bookingId;
    private String senderName;
    private LocalDate bookingDate;
    private double cost;
    private String status;

    // Constructor, getters, setters...

    @Override
    public int compareTo(Booking other) {
        // Natural ordering: by booking date (earliest first)
        return this.bookingDate.compareTo(other.bookingDate);
    }

    // compareTo() contract:
    // Returns NEGATIVE if this < other  (this comes first)
    // Returns ZERO     if this == other (same position)
    // Returns POSITIVE if this > other  (other comes first)
}
SortBookings.java 
import java.util.ArrayList;
import java.util.Collections;

ArrayList<Booking> bookings = bookingDAO.getAllBookings();

// This works because Booking implements Comparable<Booking>
Collections.sort(bookings);
// Bookings are now sorted by date (earliest first)

// Without Comparable, this line would cause a compile error:
// "Booking cannot be cast to java.lang.Comparable"
Analogy

Think of Comparable as giving your class a built-in ruler. When someone asks "which Booking comes first?" the Booking itself knows how to answer that question because compareTo() is baked into the class. It is like alphabetical order for Strings — Strings already implement Comparable, so "Apple".compareTo("Banana") returns a negative number (Apple comes before Banana).

Section 09

Comparator — Custom Sorting

Comparable defines ONE natural ordering. But what if you want to sort bookings by cost sometimes and by date other times? Comparator lets you define external, interchangeable sorting strategies.

Comparable vs Comparator

Comparable
  • Defined INSIDE the class
  • Only ONE natural ordering
  • Implements compareTo()
  • Collections.sort(list)
  • "I know how I should be sorted"
Comparator
  • Defined OUTSIDE the class
  • Unlimited sort strategies
  • Implements compare(a, b)
  • Collections.sort(list, comparator)
  • "Someone else decides how to sort me"

Multiple Sort Strategies for Bookings

BookingSortStrategies.java 
import java.util.Comparator;
import java.util.ArrayList;
import java.util.Collections;

ArrayList<Booking> bookings = bookingDAO.getAllBookings();

// --- Sort by COST (cheapest first) ---
Comparator<Booking> byCost = Comparator.comparing(Booking::getCost);
Collections.sort(bookings, byCost);

// --- Sort by COST (most expensive first) ---
Comparator<Booking> byCostDesc = Comparator.comparing(Booking::getCost).reversed();
Collections.sort(bookings, byCostDesc);

// --- Sort by DATE (most recent first) ---
Comparator<Booking> byDateDesc = Comparator.comparing(Booking::getBookingDate).reversed();
Collections.sort(bookings, byDateDesc);

// --- Sort by STATUS alphabetically ---
Comparator<Booking> byStatus = Comparator.comparing(Booking::getStatus);
Collections.sort(bookings, byStatus);

// --- Sort by STATUS first, then by DATE within each status ---
Comparator<Booking> byStatusThenDate = Comparator
    .comparing(Booking::getStatus)
    .thenComparing(Booking::getBookingDate);
Collections.sort(bookings, byStatusThenDate);
// Result: All "BOOKED" sorted by date, then all "DELIVERED" sorted by date, etc.

// --- Sort by COST, but put nulls last ---
Comparator<Booking> byCostNullSafe = Comparator.comparing(
    Booking::getCost, Comparator.nullsLast(Comparator.naturalOrder())
);

// --- Inline with lambda (same as byCost above) ---
bookings.sort((b1, b2) -> Double.compare(b1.getCost(), b2.getCost()));
Comparator method chain explained
1
Comparator.comparing(Booking::getCost)
Creates a Comparator that extracts the cost from each Booking and compares those values. Booking::getCost is a method reference — shorthand for (b) -> b.getCost().
2
.reversed()
Flips the sort order. If the original was ascending (cheapest first), reversed() makes it descending (most expensive first).
3
.thenComparing(Booking::getBookingDate)
When two bookings have the same status, this secondary comparator breaks the tie using booking date. You can chain as many thenComparing() calls as you need.
4
bookings.sort((b1, b2) -> Double.compare(b1.getCost(), b2.getCost()))
Lambda version. b1 and b2 are two Bookings being compared. Double.compare returns negative, zero, or positive. The list sorts itself in-place.
In Nimbus

In the booking history display, a customer might want to see bookings sorted differently: "Show me most recent first" (sort by date descending), "Show me cheapest first" (sort by cost ascending), "Group by status" (sort by status then date). With Comparator, you create one list and sort it multiple ways without changing the Booking class. The admin dashboard could sort by status + date, while the customer view sorts by date only.

Section 10

Collections Utility Class

The java.util.Collections class (note the 's' — plural) is a utility class full of static helper methods for working with collections. Think of it as a toolbox that operates ON collections.

Do Not Confuse

Collection (singular, no 's') = the interface that List, Set, Queue extend.
Collections (plural, with 's') = the utility class with static helper methods like sort(), reverse(), min().

CollectionsUtilDemo.java 
import java.util.ArrayList;
import java.util.Collections;

ArrayList<Booking> bookings = bookingDAO.getAllBookings();

// SORT — natural ordering (uses compareTo)
Collections.sort(bookings);

// SORT with Comparator
Collections.sort(bookings, Comparator.comparing(Booking::getCost));

// REVERSE — flip the current order
Collections.reverse(bookings);

// MIN — smallest element (by natural ordering)
Booking cheapest = Collections.min(bookings, Comparator.comparing(Booking::getCost));
System.out.println("Cheapest booking: Rs. " + cheapest.getCost());

// MAX — largest element
Booking mostExpensive = Collections.max(bookings, Comparator.comparing(Booking::getCost));

// FREQUENCY — count occurrences
ArrayList<String> statuses = new ArrayList<>();
for (Booking b : bookings) statuses.add(b.getStatus());
int deliveredCount = Collections.frequency(statuses, "DELIVERED");
System.out.println("Delivered bookings: " + deliveredCount);

// UNMODIFIABLE LIST — read-only wrapper
List<String> validStatuses = Collections.unmodifiableList(
    new ArrayList<>(List.of("BOOKED", "IN_TRANSIT", "DELIVERED", "CANCELLED"))
);
// validStatuses.add("LOST");  // Throws UnsupportedOperationException!

// SHUFFLE — randomize order (useful for testing)
Collections.shuffle(bookings);

// SWAP — swap two elements
Collections.swap(bookings, 0, 3);  // Swap element at index 0 with index 3
In Nimbus

Collections.unmodifiableList() is useful for things that should never change — like the list of valid booking statuses. If someone accidentally tries to add to it, Java throws an exception immediately instead of letting corrupted data sneak through. Defensive programming.

Section 11

Immutable Collections (Java 9+)

Java 9 introduced factory methods to create immutable (unmodifiable) collections in a single line. These collections cannot be changed after creation — no add, no remove, no set.

ImmutableCollections.java 
import java.util.List;
import java.util.Set;
import java.util.Map;

// IMMUTABLE LIST — fixed content, cannot change
List<String> parcelTypes = List.of("Document", "Package", "Fragile", "Hazardous");
// parcelTypes.add("Oversize");  // UnsupportedOperationException!
// parcelTypes.set(0, "Letter"); // UnsupportedOperationException!

// IMMUTABLE SET — unique, fixed elements
Set<String> validStatuses = Set.of("BOOKED", "IN_TRANSIT", "DELIVERED", "CANCELLED");
boolean valid = validStatuses.contains("BOOKED");  // true — reading is fine

// IMMUTABLE MAP — fixed key-value pairs
Map<String, Double> zoneRates = Map.of(
    "LOCAL", 50.0,
    "REGIONAL", 100.0,
    "NATIONAL", 200.0,
    "INTERNATIONAL", 500.0
);
double rate = zoneRates.get("NATIONAL");  // 200.0

// For maps with more than 10 entries, use Map.ofEntries()
Map<String, String> statusLabels = Map.ofEntries(
    Map.entry("BOOKED", "Parcel Booked"),
    Map.entry("PICKED_UP", "Picked Up by Courier"),
    Map.entry("IN_TRANSIT", "In Transit"),
    Map.entry("OUT_FOR_DELIVERY", "Out for Delivery"),
    Map.entry("DELIVERED", "Delivered"),
    Map.entry("CANCELLED", "Cancelled"),
    Map.entry("RETURNED", "Returned to Sender")
);

// IMPORTANT: List.of() does NOT allow null elements
// List.of("A", null, "B");  // NullPointerException!
When to Use Immutable Collections

Use immutable collections for data that is fixed at compile time and should never change: parcel types, status codes, zone mappings, pricing tiers. This prevents bugs where someone accidentally modifies a shared reference. If you need a collection you can modify later, use regular new ArrayList<>() / new HashMap<>().

Section 12

Generics — Why ArrayList<User> Not ArrayList

Before Java 5, collections had no type parameter. You used raw ArrayList and it stored Objects. You could put a User, a String, and an Integer into the same list. The compiler would not warn you, but at runtime — ClassCastException.

Without Generics (pre-Java 5)
// Raw ArrayList — accepts ANYTHING
ArrayList users = new ArrayList();
users.add(new User("U001", "Darshan"));
users.add("This is a String?!");  // No error!
users.add(42);                     // No error!

// Retrieving requires dangerous casting
User u = (User) users.get(0);  // Works
User u2 = (User) users.get(1); // CRASH!
// ClassCastException: String cannot
// be cast to User
With Generics (Java 5+)
// Type-safe ArrayList — ONLY Users
ArrayList<User> users = new ArrayList<>();
users.add(new User("U001", "Darshan"));
users.add("String?!");  // COMPILE ERROR!
users.add(42);           // COMPILE ERROR!

// No casting needed
User u = users.get(0);  // Just works.
// Compiler guarantees every element
// is a User.

The <User> in ArrayList<User> is a type parameter. It tells the compiler: "this list will ONLY hold User objects." Any attempt to add something else is caught at compile time — before the program even runs. This is type safety, and it eliminates an entire category of bugs.

In Nimbus

Every collection in Nimbus uses generics: ArrayList<User> for user lists, ArrayList<Booking> for booking results, HashMap<String, Booking> for booking lookups. This means the compiler catches mistakes like accidentally putting a User into a Booking list. Never use raw types. Always specify the type parameter.

Section 13

Choosing the Right Collection

The hardest part of collections is not learning the API — it is knowing which one to reach for. This decision flowchart covers every common scenario.

  START: What do you need?
  |
  +-- Need KEY-VALUE pairs?
  |   |
  |   YES --> Need keys sorted?
  |   |       |
  |   |       YES --> TreeMap
  |   |       NO  --> Need insertion order?
  |   |               |
  |   |               YES --> LinkedHashMap
  |   |               NO  --> HashMap  (DEFAULT CHOICE)
  |
  +-- Need UNIQUENESS (no duplicates)?
  |   |
  |   YES --> Need elements sorted?
  |   |       |
  |   |       YES --> TreeSet
  |   |       NO  --> Need insertion order?
  |   |               |
  |   |               YES --> LinkedHashSet
  |   |               NO  --> HashSet  (DEFAULT CHOICE)
  |
  +-- Need an ORDERED list (duplicates OK)?
      |
      YES --> Mostly reading/accessing by index?
              |
              YES --> ArrayList  (DEFAULT CHOICE)
              NO  --> Frequent insert/remove at ends?
                      |
                      YES --> LinkedList
                      NO  --> ArrayList  (still usually wins)

Section 14

Performance Comparison — The Big Table

This table is your reference card. When choosing a collection, check the operation you will use most — pick the one where that operation is fastest.

Operation ArrayList LinkedList HashSet HashMap TreeMap
Add (end) O(1)* O(1) O(1) O(1) O(log n)
Add (beginning) O(n) O(1) N/A N/A N/A
Get by index O(1) O(n) N/A N/A N/A
Get by key N/A N/A N/A O(1) O(log n)
Contains/Search O(n) O(n) O(1) O(1) O(log n)
Remove O(n) O(1)** O(1) O(1) O(log n)
Sorted? No No No No Yes
Duplicates? Yes Yes No Keys: No Keys: No
Null allowed? Yes Yes 1 null 1 null key No null keys
Thread-safe? No No No No No

* Amortized O(1) — occasionally O(n) during resize.
** O(1) only if you already have a reference to the node. Finding the node first is O(n).

Rule of Thumb

When in doubt, start with ArrayList for lists and HashMap for key-value lookups. They cover 90% of use cases. Only switch to LinkedList, TreeMap, or other structures when you have a specific performance need (frequent front-insertions, sorted keys, etc.). Premature optimization is the root of all evil — pick the simple one first, optimize later if needed.
