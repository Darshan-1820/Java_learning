Section 01

What is a Database?

Before you write SQL, you need to understand what a database actually is, why it exists, and why you cannot just store everything in files or spreadsheets.

The Spreadsheet Analogy

You already know spreadsheets. Open Google Sheets, make columns for Name, Email, Phone — each row is a person. A database table works exactly the same way:

Spreadsheet ConceptDatabase Concept
Entire spreadsheet fileDatabase
One sheet/tabTable
Column header (Name, Email)Column / Field
One row of dataRow / Record / Tuple
One cell valueField value

So when someone says "the tbl_UserProfile table," picture a sheet tab called tbl_UserProfile with columns like User_ID, FullName, Email — and each row is one registered Nimbus user.

Why Not Just Use Files?

You could store data in a text file or CSV. Nimbus Phase 1 used LocalStorage (basically a file in the browser). So why bother with a database?

ProblemFile/LocalStorageDatabase (MySQL)
Multiple users accessing at same timeCorrupts data — two people writing the file simultaneouslyHandles thousands of concurrent users safely
Find one record among 100,000Read entire file, scan line by line. Slow.Uses indexes. Finds it in milliseconds.
Ensure email is uniqueYou write the code yourself and hope it worksAdd UNIQUE constraint. Database enforces it for you.
Two related things (user + their bookings)Separate files, you manually match IDsForeign keys link tables. Database keeps them in sync.
Crash recoveryData goneTransaction log restores to last safe state
In Nimbus

Phase 1 stored bookings in localStorage — browser-only, one user at a time, no relationships, data lost if cache is cleared. Phase 2 moves to MySQL because a real parcel management system needs multiple officers tracking packages simultaneously, enforced data integrity, and persistent storage that survives browser crashes.

What is an RDBMS?

RDBMS stands for Relational Database Management System. Break it down:

  • Relational — Data is stored in tables that can relate to each other (a booking relates to a user)
  • Database — An organized collection of data
  • Management System — The software that manages storing, retrieving, updating, and deleting that data

Popular RDBMS options:

RDBMSUsed ByCost
MySQLFacebook, Twitter, Airbnb, UberFree (open-source)
PostgreSQLInstagram, Spotify, AppleFree (open-source)
Oracle DatabaseBanks, telecom, governmentVery expensive
Microsoft SQL ServerEnterprise Windows shopsPaid (free Express edition)
SQLiteAndroid apps, small toolsFree (file-based, no server)

Nimbus uses MySQL because it is free, widely used in the industry, and the standard for Java web applications learning.

SQL — The Language of Databases

SQL (Structured Query Language, pronounced "sequel" or "S-Q-L") is how you talk to a database. You do not click buttons. You write commands:

  • DDL (Data Definition Language) — Define structure: CREATE TABLE, ALTER TABLE, DROP TABLE
  • DML (Data Manipulation Language) — Work with data: SELECT, INSERT, UPDATE, DELETE
  • DCL (Data Control Language) — Permissions: GRANT, REVOKE

This page covers DDL and DML — the parts you will use every single day.

Section 02

MySQL Data Types

Every column in a table has a data type — it tells MySQL what kind of value it should store. Using the right type saves storage, improves performance, and prevents bad data.

Types You Will Actually Use

TypeWhat It StoresExampleWhen to Use
INTWhole numbers (-2.1 billion to 2.1 billion)42, -7, 0IDs, counts, quantities, weight in grams
BIGINTVery large whole numbers9223372036854775807Phone numbers (10+ digits), huge counters
VARCHAR(n)Variable-length text, up to n characters'Darshan'Names, emails, short text. Only uses as much space as the actual value.
TEXTLong text (up to 65,535 characters)A full description or commentLong descriptions, notes. Cannot be indexed efficiently.
DECIMAL(p,s)Exact decimal numbers149.50Money, costs. DECIMAL(10,2) = up to 10 digits, 2 after the decimal.
DATETIMEDate and time combined'2026-04-23 14:30:00'Timestamps for when something happened (booking date, pickup time)
DATEDate only (no time)'2026-04-23'Birth dates, event dates
BOOLEANTrue or false (stored as 0 or 1 in MySQL)TRUE, FALSEFlags: is_active, is_verified
ENUM(...)One value from a fixed list'Express'Status values, categories with known options

VARCHAR vs TEXT — When to Use Which

Both store text. The difference:

  • VARCHAR(100) — You know the max length (names, emails). Can be indexed. Enforces a limit.
  • TEXT — You do not know how long the text will be (descriptions, comments). Cannot be used as a primary key or have a default value easily.

Rule of thumb: If you can put a reasonable max length on it, use VARCHAR. Otherwise, TEXT.

INT vs BIGINT — The Phone Number Trap

INT maxes out at about 2.1 billion. An Indian phone number like 9876543210 is 9.8 billion — too big for INT. Use BIGINT for phone numbers, or better yet, store them as VARCHAR(15) since phone numbers can have leading zeros and country codes like +91.

In Nimbus — Column Types Explained

Here is why each Nimbus column uses the type it does:

  • User_ID INT — Auto-incrementing integer. Users get IDs 1, 2, 3... INT handles billions.
  • FullName VARCHAR(100) — No name exceeds 100 chars. VARCHAR only uses space for the actual name length.
  • Email VARCHAR(100) — RFC standard says emails max out at 254 chars, but 100 covers 99.9% of cases.
  • MobileNumber VARCHAR(15) — Stored as text, not number. Preserves leading zeros, allows +91 prefix.
  • Password VARCHAR(255) — Plain text for now (Phase 2). When hashed with bcrypt later, hashes are 60 chars. 255 gives room.
  • Role VARCHAR(20) — 'Customer', 'Officer', or 'Support'. Could use ENUM, but VARCHAR is more flexible if roles change.
  • Par_Weight_Gram INT — Weight in grams as a whole number. 5000 grams = 5kg. No decimals needed.
  • Par_ServiceCost DECIMAL(10,2) — Money should NEVER use FLOAT (rounding errors). DECIMAL is exact.
  • Par_PickupTime DATETIME — Stores both date and time of scheduled pickup.
  • Par_Status VARCHAR(20) — 'Pending', 'PickedUp', 'InTransit', 'Delivered', 'Cancelled'.
  • Booking_Date DATETIME — When the booking was created. Defaults to CURRENT_TIMESTAMP.
Never Use FLOAT or DOUBLE for Money

FLOAT and DOUBLE are approximate types. 0.1 + 0.2 might give you 0.30000000000000004. When you are dealing with money (service costs, payments), always use DECIMAL. Nimbus uses DECIMAL(10,2) for Par_ServiceCost — exact to the paisa.

Section 03

CREATE TABLE

This is where you define the structure of your data — what columns exist, what types they use, and what rules (constraints) the database enforces automatically.

Basic Syntax

CREATE TABLE syntax 
CREATE TABLE table_name (
    column1  datatype  constraints,
    column2  datatype  constraints,
    ...
);

Constraints — Rules the Database Enforces

ConstraintWhat It DoesNimbus Example
PRIMARY KEYUniquely identifies each row. No duplicates, no NULLs. Every table must have one.User_ID in tbl_UserProfile, Booking_ID in Booking
AUTO_INCREMENTMySQL auto-generates the next number (1, 2, 3...). You do not supply the value.User_ID INT AUTO_INCREMENT
NOT NULLColumn cannot be empty. Insert fails if you try.FullName — every user must have a name
UNIQUENo two rows can have the same value in this column.Email — prevents duplicate registrations
DEFAULTIf you do not provide a value, use this fallback.Booking_Date DEFAULT CURRENT_TIMESTAMP
CHECKValue must satisfy a condition.Role CHECK (Role IN ('Customer','Officer','Support'))
FOREIGN KEYLinks a column to another table's primary key. Ensures the referenced row exists.User_ID in Booking references User_ID in tbl_UserProfile

Nimbus: tbl_UserProfile

nimbus_db — tbl_UserProfile 
CREATE DATABASE IF NOT EXISTS nimbus_db;
USE nimbus_db;

CREATE TABLE tbl_UserProfile (
    User_ID       INT           AUTO_INCREMENT PRIMARY KEY,
    FullName      VARCHAR(100)  NOT NULL,
    Rec_Address   VARCHAR(255)  NOT NULL,
    Email         VARCHAR(100)  NOT NULL UNIQUE,
    MobileNumber  VARCHAR(15)   NOT NULL,
    Password      VARCHAR(255)  NOT NULL,
    Role          VARCHAR(20)   NOT NULL CHECK (Role IN ('Customer', 'Officer', 'Support'))
);
Line-by-line explanation
1
CREATE DATABASE IF NOT EXISTS nimbus_db;
Creates the database only if it does not already exist. Prevents errors on re-run.
2
USE nimbus_db;
Switches to the nimbus_db database. All following commands run inside this database.
3
User_ID INT AUTO_INCREMENT PRIMARY KEY,
Integer column. MySQL auto-generates 1, 2, 3... for each new user. PRIMARY KEY means this uniquely identifies each row — no two users can have the same ID.
4
FullName VARCHAR(100) NOT NULL,
Text up to 100 characters. NOT NULL means you cannot register without providing a name — the INSERT will fail.
5
Email VARCHAR(100) NOT NULL UNIQUE,
NOT NULL + UNIQUE = every user must have an email AND no two users can share the same email. This prevents duplicate registrations at the database level.
6
Role VARCHAR(20) NOT NULL CHECK (Role IN ('Customer', 'Officer', 'Support'))
CHECK constraint limits the allowed values. If someone tries to insert Role = 'Admin', the database rejects it. This is your safety net — even if the application code has a bug, bad data cannot get in.

Nimbus: Booking Table

nimbus_db — Booking 
CREATE TABLE Booking (
    Booking_ID                INT           AUTO_INCREMENT PRIMARY KEY,
    User_ID                   INT           NOT NULL,
    Rec_Name                  VARCHAR(100)  NOT NULL,
    Rec_Address               VARCHAR(255)  NOT NULL,
    Rec_Pin                   VARCHAR(10)   NOT NULL,
    Rec_Mobile                VARCHAR(15)   NOT NULL,
    Par_Weight_Gram           INT           NOT NULL,
    Par_Contents_Description  TEXT,
    Par_Delivery_Type         VARCHAR(20)   NOT NULL,
    Par_Packing_Preference    VARCHAR(30),
    Par_PickupTime            DATETIME,
    Par_DropoffTime           DATETIME,
    Par_ServiceCost           DECIMAL(10,2) NOT NULL,
    Par_PaymentTime           DATETIME,
    Par_Status                VARCHAR(20)   NOT NULL DEFAULT 'Pending',
    Booking_Date              DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,

    FOREIGN KEY (User_ID) REFERENCES tbl_UserProfile(User_ID)
);
Line-by-line explanation
1
User_ID INT NOT NULL,
Which user created this booking. NOT NULL because every booking must belong to someone.
2
Rec_Pin VARCHAR(10) NOT NULL,
PIN code as VARCHAR, not INT. PIN codes like '400001' can start with 0, and you never do math on them.
3
Par_Contents_Description TEXT,
No NOT NULL — description is optional. User can leave it blank. TEXT because descriptions can be long.
4
Par_Status VARCHAR(20) NOT NULL DEFAULT 'Pending',
Every new booking starts as 'Pending'. If the INSERT does not specify a status, MySQL fills in 'Pending' automatically.
5
FOREIGN KEY (User_ID) REFERENCES tbl_UserProfile(User_ID)
This is the relationship. It says: "The User_ID in this Booking table MUST exist in tbl_UserProfile." You cannot create a booking for user 999 if user 999 does not exist. The database blocks it.
Why Separate Tables?

Why not put user info and booking info in one giant table? Because a user creates many bookings. If you store the user's name, email, and phone in every booking row, you duplicate data hundreds of times. If the user changes their phone number, you have to update hundreds of rows instead of one. Separate tables with a FOREIGN KEY link solve this — store user info once, reference it by ID.

Order Matters

You must insert into tbl_UserProfile BEFORE Booking. The FOREIGN KEY in Booking says "this User_ID must exist in tbl_UserProfile." If you insert a booking first, the referenced user does not exist yet, and MySQL throws an error.

Section 05

SELECT — Reading Data

SELECT is the most-used SQL command. You will write it hundreds of times. It reads data from tables without changing anything.

Basic SELECT

Get all users 
-- Get everything from every user
SELECT * FROM tbl_UserProfile;

-- Get only names and emails (better practice — don't select * in production)
SELECT FullName, Email FROM tbl_UserProfile;

* means "all columns." It works for quick checks, but in production code you should list the columns you need. Fetching columns you do not use wastes memory and bandwidth.

WHERE — Filtering Rows

WHERE examples 
-- Find a specific user by email
SELECT * FROM tbl_UserProfile
WHERE Email = 'arjun.mehta@email.com';

-- All customers (not officers or support)
SELECT FullName, Email FROM tbl_UserProfile
WHERE Role = 'Customer';

-- Bookings heavier than 2 kg (2000 grams)
SELECT Booking_ID, Rec_Name, Par_Weight_Gram FROM Booking
WHERE Par_Weight_Gram > 2000;

AND / OR — Combining Conditions

AND and OR 
-- Express deliveries that are still pending
SELECT * FROM Booking
WHERE Par_Delivery_Type = 'Express' AND Par_Status = 'Pending';

-- Bookings that are either delivered or cancelled
SELECT * FROM Booking
WHERE Par_Status = 'Delivered' OR Par_Status = 'Cancelled';

LIKE — Pattern Matching

LIKE with wildcards 
-- Users whose name starts with 'P'
SELECT * FROM tbl_UserProfile
WHERE FullName LIKE 'P%';

-- Users with @nimbus.com emails
SELECT * FROM tbl_UserProfile
WHERE Email LIKE '%@nimbus.com';

-- Names containing 'sh' anywhere
SELECT * FROM tbl_UserProfile
WHERE FullName LIKE '%sh%';

% = any number of characters. _ = exactly one character. So 'P%' matches Priya, Patil, Prakash. '_riya' would match Priya but not Shriya.

IN — Matching Multiple Values

IN clause 
-- Bookings that are pending OR in transit (cleaner than OR OR OR)
SELECT * FROM Booking
WHERE Par_Status IN ('Pending', 'InTransit', 'PickedUp');

-- Users with IDs 1, 3, or 5
SELECT FullName FROM tbl_UserProfile
WHERE User_ID IN (1, 3, 5);

BETWEEN — Range Filtering

BETWEEN for ranges 
-- Bookings between April 20 and April 23
SELECT * FROM Booking
WHERE Booking_Date BETWEEN '2026-04-20' AND '2026-04-23';

-- Parcels weighing between 1kg and 3kg
SELECT Booking_ID, Par_Weight_Gram FROM Booking
WHERE Par_Weight_Gram BETWEEN 1000 AND 3000;

ORDER BY — Sorting Results

ORDER BY 
-- Bookings sorted by date (newest first)
SELECT * FROM Booking
ORDER BY Booking_Date DESC;

-- Users sorted alphabetically
SELECT FullName, Email FROM tbl_UserProfile
ORDER BY FullName ASC;

-- Sort by status, then by date within each status
SELECT * FROM Booking
ORDER BY Par_Status ASC, Booking_Date DESC;

ASC = ascending (A-Z, 1-9, oldest first). DESC = descending (Z-A, 9-1, newest first). Default is ASC.

LIMIT — Controlling Result Count

LIMIT 
-- Get the 5 most recent bookings
SELECT * FROM Booking
ORDER BY Booking_Date DESC
LIMIT 5;

-- Pagination: skip first 10, get next 10
SELECT * FROM Booking
ORDER BY Booking_ID
LIMIT 10 OFFSET 10;

DISTINCT — Removing Duplicates

DISTINCT 
-- What delivery types exist in our bookings?
SELECT DISTINCT Par_Delivery_Type FROM Booking;

-- Which cities have we delivered to?
SELECT DISTINCT Rec_Pin FROM Booking;
Section 06

JOINs — Combining Tables

Data lives in separate tables. JOINs connect them. This is the core power of relational databases — and the concept that trips up most beginners.

Why Nimbus Needs JOINs

The Booking table has User_ID = 1, but it does not have the user's name or email. That information is in tbl_UserProfile. To show "Arjun Mehta's bookings" in one result, you need to JOIN the two tables.

  tbl_UserProfile                        Booking
  ┌──────────┬─────────────┐             ┌────────────┬──────────┬───────────┐
  │ User_ID  │ FullName    │             │ Booking_ID │ User_ID  │ Par_Status│
  ├──────────┼─────────────┤             ├────────────┼──────────┼───────────┤
  │ 1        │ Arjun Mehta │─────┐       │ 1          │ 1        │ Delivered │
  │ 2        │ Priya Sharma│──┐  └──────▶│ 3          │ 1        │ Pending   │
  │ 3        │ Rahul Desai │  └────────▶ │ 2          │ 2        │ InTransit │
  │ 4        │ Sneha Patil │             └────────────┴──────────┴───────────┘
  │ 5        │ Vikram Joshi│
  └──────────┴─────────────┘
         User_ID links the two tables
Foreign key relationship: Booking.User_ID references tbl_UserProfile.User_ID

INNER JOIN — Only Matching Rows

Returns rows where there is a match in both tables. If a user has no bookings, they do not appear. If a booking somehow had no matching user, it would not appear either.

      ┌───────────────────────────────────┐
      │         INNER JOIN                │
      │                                   │
      │    ┌─────────┐  ┌─────────┐       │
      │    │  User   │  │ Booking │       │
      │    │  Table  │▓▓│  Table  │       │
      │    │         │▓▓│         │       │
      │    └─────────┘  └─────────┘       │
      │                                   │
      │    ▓▓ = Only the overlap          │
      │    Users WITH bookings            │
      │    Bookings WITH valid users      │
      └───────────────────────────────────┘
INNER JOIN returns only rows where both sides match
INNER JOIN 
-- Show booking details WITH the customer's name
SELECT u.FullName, b.Booking_ID, b.Rec_Name, b.Par_Status, b.Booking_Date
FROM Booking b
INNER JOIN tbl_UserProfile u ON b.User_ID = u.User_ID;

-- Result:
-- Arjun Mehta  | 1 | Deepak Verma | Delivered | 2026-04-19
-- Priya Sharma | 2 | Kiran Rao    | InTransit | 2026-04-21
-- Arjun Mehta  | 3 | Meena Iyer   | Pending   | 2026-04-22
Line-by-line explanation
1
FROM Booking b
b is an alias — a short nickname for the Booking table. Instead of writing Booking.Par_Status, you write b.Par_Status.
2
INNER JOIN tbl_UserProfile u
Join with the user table, aliased as u.
3
ON b.User_ID = u.User_ID
The join condition. "Match each booking's User_ID with the user who has the same User_ID." This is how the database knows which user goes with which booking.

LEFT JOIN — All From Left, Matches From Right

Returns all rows from the left table, even if there is no match in the right table. Unmatched right-side columns show as NULL.

      ┌───────────────────────────────────┐
      │         LEFT JOIN                 │
      │                                   │
      │    ┌─────────┐  ┌─────────┐       │
      │    │▓▓▓▓▓▓▓▓▓│  │         │       │
      │    │▓▓ User ▓│▓▓│ Booking │       │
      │    │▓▓ Table ▓│▓▓│  Table  │       │
      │    │▓▓▓▓▓▓▓▓▓│  │         │       │
      │    └─────────┘  └─────────┘       │
      │                                   │
      │    ▓▓ = ALL users returned        │
      │    Even those with zero bookings  │
      └───────────────────────────────────┘
LEFT JOIN: all rows from left table, NULLs where right has no match
LEFT JOIN 
-- Show ALL users and their booking count (even if 0)
SELECT u.FullName, COUNT(b.Booking_ID) AS total_bookings
FROM tbl_UserProfile u
LEFT JOIN Booking b ON u.User_ID = b.User_ID
GROUP BY u.FullName;

-- Result:
-- Arjun Mehta   | 2
-- Priya Sharma  | 1
-- Rahul Desai   | 0    <-- shows up even with no bookings
-- Sneha Patil   | 0
-- Vikram Joshi  | 0
-- Amit Kumar    | 0
-- Neha Singh    | 0
-- Ravi Support  | 0
In Nimbus — When to Use Which JOIN
  • INNER JOIN when you only want bookings that have valid users (the normal case for displaying booking lists)
  • LEFT JOIN when you want all users even if they have not booked yet (useful for admin dashboards showing inactive users)

RIGHT JOIN & FULL JOIN

      RIGHT JOIN                          FULL OUTER JOIN
      ┌─────────┐  ┌─────────┐           ┌─────────┐  ┌─────────┐
      │         │  │▓▓▓▓▓▓▓▓▓│           │▓▓▓▓▓▓▓▓▓│  │▓▓▓▓▓▓▓▓▓│
      │  User   │▓▓│▓▓Booking│           │▓▓ User ▓│▓▓│▓▓Booking│
      │  Table  │▓▓│▓▓ Table ▓│           │▓▓ Table ▓│▓▓│▓▓ Table ▓│
      │         │  │▓▓▓▓▓▓▓▓▓│           │▓▓▓▓▓▓▓▓▓│  │▓▓▓▓▓▓▓▓▓│
      └─────────┘  └─────────┘           └─────────┘  └─────────┘

      All bookings, even                  Everything from both
      orphaned ones                       sides, NULLs where
                                          no match on either side
RIGHT JOIN and FULL OUTER JOIN visual comparison
RIGHT JOIN 
-- All bookings, even if the user was somehow deleted
-- (Rarely used — you can always just swap the table order and use LEFT JOIN)
SELECT u.FullName, b.Booking_ID, b.Par_Status
FROM tbl_UserProfile u
RIGHT JOIN Booking b ON u.User_ID = b.User_ID;

FULL OUTER JOIN returns everything from both tables. MySQL does not support it directly — you simulate it by combining a LEFT JOIN and a RIGHT JOIN with UNION. In practice, you rarely need it.

Quick Rule

90% of the time, you use INNER JOIN. 9% of the time, LEFT JOIN. The remaining 1% is RIGHT/FULL. Master INNER and LEFT first.

Section 07

GROUP BY & Aggregate Functions

Aggregate functions crunch multiple rows into a single summary value. GROUP BY splits data into groups first, then aggregates each group separately.

The Five Aggregate Functions

FunctionWhat It DoesNimbus Example
COUNT(*)Counts rowsHow many bookings total?
SUM(col)Adds up valuesTotal revenue from all bookings?
AVG(col)Average valueAverage parcel weight?
MIN(col)Smallest valueCheapest booking cost?
MAX(col)Largest valueHeaviest parcel?
Aggregates without GROUP BY 
-- Total number of bookings
SELECT COUNT(*) AS total_bookings FROM Booking;
-- Result: 3

-- Total revenue
SELECT SUM(Par_ServiceCost) AS total_revenue FROM Booking;
-- Result: 587.50

-- Average parcel weight
SELECT AVG(Par_Weight_Gram) AS avg_weight FROM Booking;
-- Result: 1833.33

-- Heaviest and lightest parcels
SELECT MIN(Par_Weight_Gram) AS lightest, MAX(Par_Weight_Gram) AS heaviest
FROM Booking;
-- Result: 800 | 3200

GROUP BY — Aggregating Per Group

GROUP BY examples 
-- Count bookings per status
SELECT Par_Status, COUNT(*) AS count
FROM Booking
GROUP BY Par_Status;
-- Result:
-- Delivered | 1
-- InTransit | 1
-- Pending   | 1

-- Revenue per delivery type
SELECT Par_Delivery_Type, SUM(Par_ServiceCost) AS revenue
FROM Booking
GROUP BY Par_Delivery_Type;
-- Result:
-- Express  | 398.50
-- Standard | 189.00

-- Bookings per user
SELECT User_ID, COUNT(*) AS bookings
FROM Booking
GROUP BY User_ID;
-- Result:
-- 1 | 2
-- 2 | 1

HAVING — Filtering Groups

WHERE filters individual rows BEFORE grouping. HAVING filters groups AFTER grouping. You cannot use aggregate functions in WHERE — that is what HAVING is for.

HAVING 
-- Users who have made more than 1 booking
SELECT User_ID, COUNT(*) AS bookings
FROM Booking
GROUP BY User_ID
HAVING COUNT(*) > 1;
-- Result: User_ID 1 | 2 bookings

-- Delivery types with revenue over 200
SELECT Par_Delivery_Type, SUM(Par_ServiceCost) AS revenue
FROM Booking
GROUP BY Par_Delivery_Type
HAVING SUM(Par_ServiceCost) > 200;
Analogy — WHERE vs HAVING

Think of a school exam. WHERE is like saying "only look at students who submitted the exam" (filter before counting). HAVING is like saying "after counting scores, only show classes where the average score is above 80" (filter after aggregating).

Execution Order of a SELECT

SQL does not run in the order you write it. Here is the actual execution sequence:

FROMPick the tables
→
WHEREFilter rows
→
GROUP BYMake groups
→
HAVINGFilter groups
→
SELECTPick columns
→
ORDER BYSort

This is why you cannot use a column alias from SELECT inside WHERE — WHERE runs before SELECT.

Section 08

UPDATE — Modifying Data

Change existing values in rows. The most dangerous command after DELETE — because forgetting the WHERE clause affects every single row.

UPDATE syntax 
UPDATE table_name
SET column1 = value1, column2 = value2
WHERE condition;

Nimbus Examples

Nimbus UPDATE operations 
-- Officer picks up a parcel
UPDATE Booking
SET Par_Status = 'PickedUp',
    Par_PickupTime = '2026-04-23 10:30:00'
WHERE Booking_ID = 3;

-- Officer delivers a parcel
UPDATE Booking
SET Par_Status = 'Delivered',
    Par_DropoffTime = NOW()
WHERE Booking_ID = 2;

-- User updates their phone number
UPDATE tbl_UserProfile
SET MobileNumber = '9876500000'
WHERE User_ID = 1;
The WHERE Clause Is Not Optional

Look at this disaster:

WRONG — No WHERE
UPDATE Booking SET Par_Status = 'Delivered';

This marks ALL bookings as delivered. Every single one. Even ones that have not been picked up yet. There is no "undo" button in SQL.

CORRECT — With WHERE
UPDATE Booking SET Par_Status = 'Delivered' WHERE Booking_ID = 2;

Only updates booking #2. Exactly what you intended.

Safety Rule

Before running any UPDATE, first run a SELECT with the same WHERE clause. If the SELECT returns the rows you expect, then run the UPDATE. This prevents accidental mass updates.

Section 09

DELETE — Removing Data

Removes rows from a table. Same danger as UPDATE — always use a WHERE clause.

DELETE syntax and examples 
-- Delete a specific booking
DELETE FROM Booking WHERE Booking_ID = 3;

-- Delete all cancelled bookings
DELETE FROM Booking WHERE Par_Status = 'Cancelled';

-- DANGER: This deletes ALL bookings. Every single one.
-- DELETE FROM Booking;

CASCADE — Deleting Related Data

What happens if you delete a user who has bookings? The FOREIGN KEY constraint blocks it — you would have orphaned bookings pointing to a user that no longer exists.

You have three options:

OptionSQLWhat Happens
RESTRICT (default)FOREIGN KEY (User_ID) REFERENCES tbl_UserProfile(User_ID)Blocks the delete. "You cannot delete this user because they have bookings."
CASCADEFOREIGN KEY (User_ID) REFERENCES tbl_UserProfile(User_ID) ON DELETE CASCADEDeleting user 1 also deletes ALL of user 1's bookings. Powerful but dangerous.
SET NULLFOREIGN KEY (User_ID) REFERENCES tbl_UserProfile(User_ID) ON DELETE SET NULLDeleting user 1 sets User_ID = NULL in their bookings. Bookings survive but become "orphaned."
In Nimbus

Nimbus uses the default (RESTRICT). If you try to delete a user who has bookings, MySQL says no. This is intentional — we do not want to accidentally wipe out booking history. If a user needs to be "removed," we would add an is_active column and set it to FALSE instead of actually deleting the row. This is called a soft delete.

Section 10

Subqueries

A subquery is a query inside another query. Think of it as solving a problem in two steps, where the inner query provides data for the outer query.

Subquery in WHERE

Subquery examples 
-- Find users who have made at least one booking
SELECT FullName, Email FROM tbl_UserProfile
WHERE User_ID IN (SELECT DISTINCT User_ID FROM Booking);
-- Arjun Mehta, Priya Sharma

-- Find bookings with above-average weight
SELECT Booking_ID, Par_Weight_Gram FROM Booking
WHERE Par_Weight_Gram > (SELECT AVG(Par_Weight_Gram) FROM Booking);
-- Booking 2: 3200g (average is 1833g)

-- Find the user who made the most expensive booking
SELECT FullName FROM tbl_UserProfile
WHERE User_ID = (
    SELECT User_ID FROM Booking
    ORDER BY Par_ServiceCost DESC
    LIMIT 1
);

Subquery vs JOIN — When to Use Which

Use Subquery WhenUse JOIN When
You need a single value (average, max, count) to compare againstYou need columns from both tables in the output
The inner query is independent (can run on its own)Tables are directly related by a key
You are checking existence (IN, EXISTS)Performance matters (JOINs are usually faster)
Same result: subquery vs JOIN 
-- Using subquery
SELECT FullName FROM tbl_UserProfile
WHERE User_ID IN (SELECT User_ID FROM Booking WHERE Par_Status = 'Pending');

-- Same result using JOIN (generally faster for large data)
SELECT DISTINCT u.FullName
FROM tbl_UserProfile u
INNER JOIN Booking b ON u.User_ID = b.User_ID
WHERE b.Par_Status = 'Pending';
Section 11

Normalization

Normalization is the process of organizing tables to eliminate redundancy and prevent anomalies. It is why you split data into separate tables instead of dumping everything into one giant spreadsheet.

The Problem: One Giant Table

What if Nimbus stored everything in one table?

BAD: Everything in one table
Booking_IDFullNameEmailPhoneRec_NameStatus
1Arjun Mehtaarjun@email.com987654DeepakDelivered
2Priya Sharmapriya@email.com912345KiranInTransit
3Arjun Mehtaarjun@email.com987654MeenaPending

Arjun's name, email, and phone are stored twice. If he changes his phone number, you must update EVERY booking row — miss one and you have inconsistent data.

1NF — First Normal Form

Rule: Every column holds a single value (no lists, no repeating groups).

Violates 1NF
Phone: "9876543210, 9123456789"

Two phone numbers in one cell. How do you search for one? How do you add a third?

Follows 1NF
MobileNumber: "9876543210"

One value per column. If you need multiple phones, create a separate UserPhones table.

In Nimbus

Nimbus follows 1NF. Each column stores exactly one value. MobileNumber holds one number. Rec_Address holds one address. No comma-separated lists anywhere.

2NF — Second Normal Form

Rule: 1NF + every non-key column depends on the entire primary key (no partial dependencies).

This matters when you have a composite primary key (a key made of multiple columns). If a column depends on only part of the key, move it to its own table.

Nimbus does not use composite keys (User_ID and Booking_ID are single-column primary keys), so 2NF is automatically satisfied. But the concept matters when designing more complex systems.

3NF — Third Normal Form

Rule: 2NF + no column depends on another non-key column (no transitive dependencies).

Violates 3NF

If the Booking table had columns: User_ID, User_FullName, User_Email

User_FullName depends on User_ID, not on Booking_ID. That is a transitive dependency — it belongs in the users table.

Follows 3NF

Booking table only has User_ID. When you need the user's name, you JOIN to get it.

User info is stored once, in tbl_UserProfile. Change it once, it is updated everywhere.

In Nimbus

Nimbus is in 3NF. The Booking table does not store any user details except User_ID. It does store Rec_Name, Rec_Address, Rec_Mobile — but these belong to the recipient, who is a different person from the sender. That is not redundancy; that is different data.

Section 13

Indexes

Indexes make searches fast. Without them, MySQL reads every single row to find what you want (a "full table scan"). With an index, it jumps directly to the right row.

The Phone Book Analogy

Analogy

Imagine finding "Sharma" in a phone book. You do not start from page 1 and read every name. You jump to the "S" section, then narrow down. That is what an index does for a database — it creates a sorted lookup structure so MySQL can jump directly to matching rows instead of scanning everything.

Automatic Indexes

MySQL automatically creates indexes on:

  • PRIMARY KEY columns — User_ID and Booking_ID are automatically indexed
  • UNIQUE columns — Email is automatically indexed
  • FOREIGN KEY columns — Booking.User_ID is automatically indexed

This is why WHERE User_ID = 1 and WHERE Email = 'arjun@email.com' are fast even with millions of rows — MySQL uses the index to jump directly to the matching row.

When to Add Your Own Index

Creating indexes 
-- If you frequently search/filter by Par_Status, add an index
CREATE INDEX idx_booking_status ON Booking(Par_Status);

-- If you frequently search by date range
CREATE INDEX idx_booking_date ON Booking(Booking_Date);

-- Composite index: if you often filter by BOTH status AND date
CREATE INDEX idx_status_date ON Booking(Par_Status, Booking_Date);

Index Trade-offs

BenefitCost
SELECT queries become much fasterINSERT/UPDATE/DELETE become slightly slower (must update the index)
Sorting is faster (ORDER BY)Uses extra disk space to store the index

Rule of thumb: Add indexes on columns you frequently use in WHERE, JOIN ON, and ORDER BY clauses. Do not add indexes on every column — that wastes space and slows writes.

In Nimbus

Nimbus already has automatic indexes on User_ID, Booking_ID, Email, and the foreign key Booking.User_ID. As the app grows, we would add an index on Par_Status (because officers constantly filter by status) and Booking_Date (for date range queries in reports).
