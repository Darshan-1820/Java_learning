Section 01

What is Java?

Java is the language you will use to build Nimbus's entire backend. Before writing a single line of it, you need to understand what it is, where it came from, and why half the world's enterprise software runs on it.

A Brief History

Java was created by James Gosling at Sun Microsystems in 1995. The original name was "Oak" (after a tree outside Gosling's office), but that was already trademarked, so they renamed it to Java — after the Indonesian island famous for coffee. That is why the Java logo is a coffee cup.

The problem Java was trying to solve: in the 1990s, if you wrote a program in C or C++, you had to compile it separately for every operating system. Write it for Windows? It will not run on Mac. Compile it for Linux? Useless on Solaris. Every time you wanted to support a new machine, you had to rewrite parts of your code.

Java's answer was revolutionary: "Write Once, Run Anywhere" (WORA). Write your code once, and it runs on any machine that has a Java Virtual Machine installed — Windows, Mac, Linux, servers, ATMs, Android phones, even smart cards.

Analogy

Think of Java like a Bollywood movie with subtitles. The movie is made once (in Hindi). But because subtitles exist for English, Tamil, Telugu, and more, people everywhere can watch it. The movie does not change — the subtitle system adapts it for each audience. Java code is the movie. The JVM is the subtitle system.

Why Companies Use Java

Java is not the "coolest" or "newest" language. So why do banks, airlines, governments, and companies like Google, Amazon, Netflix, and TCS use it everywhere?

ReasonWhat It Means
Platform IndependentSame code runs on Windows servers, Linux servers, Mac dev machines — no changes needed
Extremely ReliableStrong type system catches errors at compile time, not at 3 AM in production
Massive EcosystemThousands of libraries, frameworks (Spring, Hibernate), tools (Maven, Eclipse) — you never build from scratch
Backward CompatibleCode written in Java 8 (2014) still runs on Java 21 (2023). Companies do not have to rewrite their systems every few years
Talent PoolMillions of Java developers worldwide. Easy to hire, easy to train

In Nimbus

You are building Nimbus in Java because that is what TCS projects use. The backend will use JDBC (database connections), Servlets (handling HTTP requests), and eventually Spring (enterprise framework). All of these are Java. Your team of 8 will write Java in Eclipse on Windows machines — but the same code can run on TCS's Linux servers without changing a single line. That is WORA in action.

Java by Numbers

  • 3+ billion devices run Java worldwide
  • #1 or #2 on every major programming language ranking (TIOBE, Stack Overflow)
  • 35 million+ Java developers globally
  • 90% of Fortune 500 companies use Java in their backend systems
  • Android apps (until Kotlin), banking systems, airline booking, government portals — all Java

Section 03

The Compilation Process

When you click "Run" in Eclipse, a multi-step process happens. Understanding this process is the difference between blindly pressing buttons and knowing what your tools actually do.

The Full Pipeline

Hello.java Source Code (human-readable)
→
javac Java Compiler
→
Hello.class Bytecode (intermediate)
→
JVM Executes on your OS
→
Output Hello, World!
The complete journey of a Java program from source to output

Step-by-Step

  1. You write source code — a .java file containing human-readable Java code. This is what you type in Eclipse. The file name must match the class name exactly: Hello.java for public class Hello.
  2. The compiler (javac) compiles it — the Java compiler reads your .java file, checks for syntax errors, type errors, and other problems. If everything is valid, it produces a .class file containing bytecode.
  3. Bytecode is generated — the .class file contains bytecode, which is neither human-readable source code nor machine-specific binary. It is an intermediate format that any JVM on any operating system can understand.
  4. The JVM executes the bytecode — when you run the program, the JVM loads the .class file, verifies it is safe, and executes the bytecode instructions. The JVM translates bytecode into native machine code for your specific CPU and OS.
  5. You see the output — System.out.println() writes text to the console. Your program has completed its journey from text file to running software.

What is Bytecode?

Bytecode is the key to everything. It sits between two extremes:

Source Code (.java)

Human-readable. You write it. Contains keywords like public, class, if, for.

Cannot be executed directly — the computer does not understand English-like syntax.

Machine Code (.exe)

CPU-readable. Pure binary (0s and 1s). Different for every OS and processor.

A Windows .exe will not run on Mac or Linux. It is locked to one platform.

Bytecode (.class) is in the middle. It is not human-readable and it is not machine-specific. It is a compact, standardized set of instructions that any JVM can interpret. Think of it as a "universal instruction set" that the JVM translates to the right machine code at runtime.

Viewing bytecode (you will never need to write this) 
// Your source code:
System.out.println("Hello");

// What the compiler turns it into (bytecode, shown with javap -c):
// getstatic     #2    // Field java/lang/System.out:Ljava/io/PrintStream;
// ldc           #3    // String Hello
// invokevirtual #4    // Method java/io/PrintStream.println:(Ljava/lang/String;)V

You will never write bytecode. But knowing it exists explains why Java is platform-independent and why you need to compile before running.

In Nimbus

When you save a .java file in Eclipse, it automatically compiles it (Eclipse has "Build Automatically" on by default). The .class files go into a bin/ folder inside your project. If you see a red X on a file in Eclipse, that means javac found errors and could not produce the .class file. No .class file = you cannot run the program.

Section 04

Why Bytecode Matters

This is where Java's superpower becomes clear. The same .class file runs on any operating system, because each OS has its own JVM that knows how to translate bytecode into that OS's native instructions.

The Platform Independence Model

One .class file, three operating systems, identical results — Write Once, Run Anywhere

Contrast with C/C++

In C or C++, the compiler produces native machine code directly. That machine code only works on the specific OS + CPU combination it was compiled for. If you want to support 3 platforms, you compile 3 times and potentially change platform-specific code.

In Java, you compile once to bytecode. The JVM on each platform handles the translation. You ship one .class file (or a .jar package of many .class files), and it works everywhere a JVM exists.

Analogy

Imagine you wrote a recipe in a universal language that no human speaks directly. In India, a Hindi translator reads it and cooks the dish. In Japan, a Japanese translator reads the same recipe and cooks the same dish. In Brazil, a Portuguese translator does the same. The recipe never changes — only the translator is different for each country. Your .class file is the recipe. The JVM is the translator.

Reality Check

"Write Once, Run Anywhere" is mostly true, but not 100%. If your code uses OS-specific features (like Windows file paths with backslashes, or Linux-only system calls), it might behave differently across platforms. But for standard Java — which is what Nimbus uses — WORA works perfectly.

Section 05

JVM Internals (Simplified)

The JVM is not a simple box that "just runs code." It has several components working together. You do not need to memorize all of this — but understanding the major parts will help you debug issues and write better code.

What Happens When You Run java Hello

Class Loader Finds & loads .class files
→
Bytecode Verifier Checks code is safe
→
Interpreter Executes line by line
→
JIT Compiler Optimizes hot code
The four major stages inside the JVM

1. Class Loader

The Class Loader is responsible for finding and loading .class files into memory. When your program says new User(), the Class Loader finds User.class, loads it, and makes it available for the JVM to use.

There are three built-in class loaders that work in a chain:

  • Bootstrap Class Loader — loads core Java classes (java.lang.String, java.util.List, etc.) from the JRE's rt.jar
  • Extension Class Loader — loads classes from Java's extension directories
  • Application Class Loader — loads your classes (the ones you write) from the classpath

2. Bytecode Verifier

Before executing any bytecode, the JVM checks it for safety. Is it well-formed? Does it try to access memory it should not? Does it violate access rules (like calling a private method from outside the class)? This is a security feature — it prevents corrupted or malicious .class files from crashing your system or stealing data.

3. Interpreter

The Interpreter reads bytecode instructions one by one and executes them. It is simple and works immediately, but it is slow because it translates the same instruction every time it encounters it. If a loop runs 10,000 times, the Interpreter translates the same bytecode 10,000 times.

4. JIT (Just-In-Time) Compiler

This is Java's performance secret. The JIT Compiler watches which code runs frequently (called "hot" code — like a loop that runs thousands of times). When it identifies hot code, it compiles that bytecode into native machine code and caches it. Next time that code runs, the JVM uses the pre-compiled native code instead of interpreting bytecode again.

This is why Java programs start slightly slower (the JIT has not warmed up yet) but get faster over time — the hot paths get compiled to native speed.

Analogy

The Interpreter is like a translator who reads a foreign recipe step by step every time you cook. The JIT Compiler is like that translator eventually memorizing the recipe. After cooking the same dish 50 times, the translator does not even look at the original — they just cook from memory. Much faster.

In Nimbus

When you run Nimbus, the Class Loader loads your classes (User.class, Parcel.class, UserDAO.class, etc.) from the bin/ folder. The Bytecode Verifier checks them. The Interpreter runs them. If you have a loop that processes 100 parcels, the JIT Compiler will optimize that loop after it runs enough times. You will not "see" any of this — it all happens invisibly inside the JVM.

Section 06

Memory Model — Stack vs Heap

When your Java program runs, the JVM divides memory into two main areas: the Stack and the Heap. Understanding this division is fundamental to understanding how variables, objects, and method calls actually work.

Stack Memory

The Stack is where method calls and local variables live. Every time you call a method, the JVM creates a "stack frame" — a block of memory that holds:

  • The method's local variables (primitives like int, boolean, double)
  • References (pointers) to objects on the Heap
  • The return address — where to go back to when this method finishes

When the method finishes (returns), its stack frame is removed. All local variables in that frame are gone forever. This is why local variables do not persist between method calls — they literally do not exist after the method returns.

Heap Memory

The Heap is where objects and arrays live. When you write new User("Darshan"), the actual User object is created on the Heap. The variable that "holds" the object is actually just a reference (address) stored on the Stack, pointing to the real object on the Heap.

Objects on the Heap live until no references point to them, at which point the Garbage Collector cleans them up (more on that in the next section).

Stack vs Heap — Side by Side

FeatureStackHeap
StoresPrimitives, references, method infoObjects and arrays
LifetimeUntil the method returnsUntil no references remain (GC cleans up)
SpeedVery fast (LIFO, simple push/pop)Slower (complex allocation, GC overhead)
SizeSmall (usually 512KB – 1MB per thread)Large (can be gigabytes)
ThreadEach thread has its own stackShared across all threads
ErrorStackOverflowError (too deep recursion)OutOfMemoryError (too many objects)

Visual Example

MemoryExample.java 
public class MemoryExample {
    public static void main(String[] args) {
        int age = 25;                        // primitive on Stack
        String name = "Darshan";             // reference on Stack, object on Heap
        User user = new User("Darshan", 25); // reference on Stack, object on Heap

        printUser(user);                     // new stack frame created
    }   // main() returns → its stack frame is removed

    static void printUser(User u) {          // 'u' is a copy of the reference
        System.out.println(u.getName());
    }   // printUser() returns → its stack frame is removed
}
     STACK                                    HEAP
  (per thread)                        (shared, managed by GC)
  +------------------+
  | printUser() frame |
  |   u = 0xABC ------+---+
  +------------------+    |
  | main() frame      |    |
  |   age = 25         |    |      +---------------------------+
  |   name = 0x123 ----+---+-->   | String "Darshan"          |
  |   user = 0xABC ----+---+->   | address: 0x123            |
  +------------------+    |      +---------------------------+
                           |
                           |      +---------------------------+
                           +----> | User object               |
                                  | name = "Darshan"          |
                                  | age = 25                  |
                                  | address: 0xABC            |

age (25) lives directly on the stack. The User object lives on the heap. The variable "user" holds a reference (address 0xABC) pointing to it.

What Happens When a Method Returns

When printUser() finishes, its stack frame is popped (removed). The local variable u is gone. But the User object on the Heap is still alive because main() still has a reference (user) pointing to it.

When main() finishes, all references to the User object are gone. The object is now "unreachable" and eligible for garbage collection.

In Nimbus

Every time your Nimbus code creates an object — new User(), new Parcel(), new ArrayList<Parcel>() — those objects go on the Heap. Local variables like int parcelId = 101 in a method stay on the Stack. When your createParcel() method returns, the local variables are gone, but the Parcel object remains on the Heap as long as it is stored in a list or referenced somewhere. If you had a method that creates 10,000 Parcel objects in a loop and stores them in a List<Parcel>, that list and all 10,000 objects live on the Heap.

Section 07

Garbage Collection

In C and C++, you manually allocate and free memory. Forget to free it? Memory leak. Free it twice? Program crash. Java eliminates this entire category of bugs with automatic garbage collection.

How It Works

The Garbage Collector (GC) is a background process inside the JVM that periodically scans the Heap looking for objects that no longer have any references pointing to them. When it finds such objects, it reclaims their memory.

GarbageExample.java 
public class GarbageExample {
    public static void main(String[] args) {
        User user1 = new User("Alice");   // User("Alice") created on Heap
        User user2 = new User("Bob");     // User("Bob") created on Heap

        user1 = user2;  // user1 now points to Bob
                         // No one points to Alice anymore!
                         // Alice is eligible for garbage collection

        user2 = null;    // user2 no longer points to Bob
                         // But user1 still points to Bob → Bob is safe

        // At this point:
        // Alice → no references → WILL be collected
        // Bob   → user1 points to it → SAFE
    }
}
Line-by-line explanation
1
User user1 = new User("Alice");
Creates a User object with name "Alice" on the Heap. The reference user1 on the Stack points to it.
2
User user2 = new User("Bob");
Creates another User object with name "Bob" on the Heap. user2 points to it.
3
user1 = user2;
Reassigns user1 to point to the same object as user2 (Bob). The Alice object now has zero references — it becomes garbage.
4
user2 = null;
Sets user2 to null (points to nothing). But Bob is still referenced by user1, so Bob is not garbage.

Key Rules of Garbage Collection

  • You cannot force GC. System.gc() is a hint — the JVM may ignore it completely. You are asking, not commanding.
  • GC only cleans the Heap. Stack memory is cleaned automatically when methods return (stack frames are popped).
  • An object is eligible for GC when no live reference points to it. If even one reference exists, the object stays alive.
  • GC runs in the background. You do not call it. You do not schedule it. The JVM decides when to run it based on memory pressure.
  • You can still get memory leaks — if you accidentally keep references to objects you no longer need (e.g., adding to a list but never removing), the GC cannot collect them.
Analogy

Think of the Heap as a parking lot. Every car (object) is parked by someone who has a ticket (reference). The garbage collector is a tow truck that drives around looking for cars with no tickets. No ticket? The car gets towed (memory reclaimed). But you cannot call the tow truck directly — it comes on its own schedule. And if someone keeps holding a ticket to a car they never drive, that car stays forever, wasting a parking spot. That is a memory leak.

In Nimbus

When a customer logs in and you create new User("Darshan"), that object lives on the Heap for the session. When the user logs out and no code holds a reference to that User object anymore, the GC will eventually clean it up. You never write free(user) or delete user like you would in C/C++. Java handles it. In Phase 3, when you use Servlets, each HTTP request might create temporary objects (parcels, lists, DTOs) — the GC cleans them all up after the response is sent.

Section 08

Your First Java Program

The classic "Hello, World!" program. But instead of just copying it, we will dissect every single keyword so you understand exactly what each part does and why it is there.

Hello.java 
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, World!");
    }
}
Every keyword explained
1
public
Access modifier. This class is visible to everyone, everywhere. Other options are private (visible only inside the same class) and protected (visible in the same package + subclasses). The main class of a file must be public.
2
class
Declares a class. In Java, ALL code lives inside a class. You cannot write a standalone function like in Python or JavaScript. A class is a blueprint for creating objects (or in this case, a container for the main method).
3
Hello
Class name. Must match the filename exactly (Hello.java). Convention: PascalCase (first letter of each word capitalized). User, ParcelService, DatabaseConnection — not user or parcel_service.
4
public static void main(String[] args)
The entry point. When you run java Hello, the JVM looks for this exact method signature. It must be public (JVM needs access), static (no object needed to call it), void (returns nothing), named main, and take String[] args.
5
static
Belongs to the class, not an object. The JVM needs to call main() without creating a Hello object first. static means "you can call this directly on the class: Hello.main(args)."
6
void
Return type. This method returns nothing. Other methods might return int, String, User, etc. main() always returns void — there is nothing to return to (the program is ending).
7
String[] args
Command-line arguments. An array of Strings passed when you run the program. java Hello John 25 makes args[0]="John" and args[1]="25". Usually empty, but the parameter must be there.
8
System.out.println("Hello, World!");
Print to console. System is a built-in class. out is a static field of type PrintStream representing standard output. println() prints the text and adds a newline. print() does the same without a newline.

Compiling and Running from Command Line

Terminal 
// Step 1: Compile (creates Hello.class)
javac Hello.java

// Step 2: Run (JVM executes Hello.class)
java Hello

// Output:
// Hello, World!

Notice: when compiling you use javac Hello.java (with the .java extension). When running you use java Hello (without any extension — you are giving the class name, not the file name).

Common Mistakes
  • File name does not match class name — hello.java containing public class Hello will not compile. The file must be Hello.java.
  • Running with .class extension — java Hello.class will fail. Use java Hello.
  • Missing semicolon — every statement in Java ends with ;. Forget one and the compiler will throw an error.
  • Wrong capitalization — String not string. System not system. Java is case-sensitive.
In Nimbus

In Eclipse, you never manually type javac or java. Eclipse auto-compiles on save and has a "Run" button. But knowing the underlying commands matters because: (1) you will use them in the TCS Unix terminal, (2) you will need them for Maven builds, and (3) if something goes wrong in Eclipse, understanding what javac does helps you diagnose the problem. Your first Nimbus Java file will be something like NimbusApp.java with a main() method that starts the console menu system.

Section 09

Packages

As your project grows beyond a handful of files, you need a way to organize classes into groups and avoid name conflicts. That is what packages do.

What is a Package?

A package is a folder that holds related Java classes. It is also a namespace — it gives each class a unique full name so two classes called User in different parts of a system do not conflict.

com/nimbus/model/User.java 
package com.nimbus.model;  // This file belongs to the "com.nimbus.model" package

public class User {
    private String name;
    private String userId;

    // The full name of this class is: com.nimbus.model.User
}

Naming Convention

Java packages follow a reverse domain name convention:

Company/ProjectPackage PrefixExample Full Package
Googlecom.googlecom.google.maps.api
Apacheorg.apacheorg.apache.commons.lang
Nimbus (our project)com.nimbuscom.nimbus.model, com.nimbus.dao, com.nimbus.service

Why reverse domain? Because domain names are globally unique. com.google can never conflict with com.nimbus. This guarantees that even if two companies both have a class called User, their full names (com.google.auth.User vs com.nimbus.model.User) are different.

Import Statements

When you want to use a class from a different package, you import it:

com/nimbus/service/UserService.java 
package com.nimbus.service;

import com.nimbus.model.User;        // Import specific class
import com.nimbus.dao.UserDAO;        // Import another class
import java.util.List;               // Import from Java standard library
import java.util.*;                  // Import ALL classes from java.util (wildcard)

public class UserService {
    private UserDAO userDAO = new UserDAO();

    public User findUser(String userId) {
        return userDAO.getById(userId);
    }

    public List<User> getAllUsers() {
        return userDAO.getAll();
    }
}
Import rules
1
import com.nimbus.model.User;
Specific import. Imports only the User class. Best practice — you know exactly what you are importing.
2
import java.util.*;
Wildcard import. Imports ALL classes from java.util (List, Map, ArrayList, etc.). Convenient but makes it harder to see what you are actually using. Eclipse prefers specific imports.
3
No import needed for java.lang
Classes in java.lang (String, System, Integer, Math) are automatically imported. You never write import java.lang.String;.

Package = Folder

A critical rule: the package declaration must match the folder structure. If your class says package com.nimbus.model;, the file must be at com/nimbus/model/User.java. Mismatch = compilation error.

  src/
  └── com/
      └── nimbus/
          ├── model/
          │   ├── User.java          → package com.nimbus.model;
          │   ├── Parcel.java        → package com.nimbus.model;
          │   └── Address.java       → package com.nimbus.model;

Nimbus project package structure — package names map directly to folder paths
In Nimbus

Nimbus uses four main packages: com.nimbus.model (data classes like User, Parcel), com.nimbus.dao (database access), com.nimbus.service (business logic), and com.nimbus.util (helpers). When UserService needs to use User, it writes import com.nimbus.model.User; at the top. Eclipse generates these imports automatically when you press Ctrl+Shift+O (Organize Imports). This is one of the most useful Eclipse shortcuts — memorize it.

Section 10

Java Versions

Java releases new versions every 6 months. But not all versions are created equal. Some are Long-Term Support (LTS) versions that get years of updates. Companies stick to LTS versions for stability.

Timeline of Key Versions

Java 1.0 (1996)
The beginning. Basic OOP, AWT for GUIs, applets for browsers. "Write Once, Run Anywhere" was born.
Java 5 (2004)
Massive update: Generics (List<String>), Enhanced for-loop (for (String s : list)), Enums, Autoboxing, Annotations (@Override). Modern Java starts here.
Java 8 (2014) — LTS
The most important release. Lambdas (x -> x * 2), Streams API (list.stream().filter().map()), Optional, new Date/Time API (LocalDate). Still the most widely used version in production. Many TCS projects run on Java 8.
Java 9 (2017)
Module system (Project Jigsaw) — organize code into modules with explicit dependencies. JShell (REPL for trying out Java code interactively). List.of(), Map.of() factory methods.
Java 11 (2018) — LTS
var for local variable type inference (var name = "Darshan"), HTTP Client (built-in, replaces old HttpURLConnection), String new methods (isBlank(), strip(), lines()). JRE no longer distributed separately — just use the JDK.
Java 17 (2021) — LTS
Sealed classes (restrict which classes can extend yours), Pattern matching for instanceof (if (obj instanceof String s)), Records (record User(String name, int age) {} — immutable data classes in one line), Text blocks (multi-line strings with """).
Java 21 (2023) — LTS
Virtual threads (Project Loom — lightweight threads for massive concurrency), Pattern matching for switch, Record patterns, Sequenced collections. The latest LTS and the future of Java.

In Nimbus

Nimbus will use Java 8 features (since that is what most TCS machines have). But we will call out when something could be done differently in Java 11+ or 17+. For example, in Java 8 you write List<User> users = new ArrayList<User>();. In Java 11+ you can write var users = new ArrayList<User>(); — shorter, same meaning. When you see these notes, you will know the modern way even if you cannot use it today.

Section 11

In Nimbus — Full Mapping

Every concept you just learned appears in the Nimbus project. Here is where each one shows up.

JDK
Eclipse uses the JDK to compile your .java files. The javac compiler runs every time you save a file. Check your JDK version: Window > Preferences > Java > Installed JREs in Eclipse.
JVM
Runs your Nimbus app when you click "Run." It loads classes from bin/, manages Stack and Heap memory, and runs the Garbage Collector. If you see StackOverflowError, your recursion is too deep. If you see OutOfMemoryError, you are creating too many objects.
Bytecode
The bin/ folder in Eclipse contains all your .class files. The same .class files can be copied to a Linux server and run there with java com.nimbus.NimbusApp. That is exactly what happens when you deploy to TCS servers.
Packages
com.nimbus.model (User, Parcel, Address), com.nimbus.dao (UserDAO, ParcelDAO — database CRUD), com.nimbus.service (UserService, ParcelService — business logic), com.nimbus.util (InputValidator, DateUtils — helpers). Every import statement connects these packages.
Memory
When you write User user = new User("Darshan"), the reference user goes on the Stack, the User object goes on the Heap. When ParcelDAO.getAllParcels() returns a List<Parcel>, that list and all its Parcel objects are on the Heap. When the method calling it finishes, the local reference is gone, but the list survives if something else holds it.

Section 12

Practice Exercises

Do not just read — type these out. Every one of these exercises maps to something you will do in Nimbus. If you can do these, you are ready for the next topic.

Exercise 1 — Hello Nimbus

Create a file called NimbusApp.java. Write a main() method that prints:

=============================
   NIMBUS Parcel Management
=============================
Welcome! System initialized.

Use System.out.println() for each line. Compile with javac and run with java from the command line. Then run it in Eclipse.

Exercise 2 — Command Line Arguments

Modify your NimbusApp.java to accept a user name from the command line:

java NimbusApp Darshan

It should print: Welcome, Darshan! You are logged into Nimbus.

If no argument is provided (args.length == 0), print: Welcome, Guest!

Hint: Use args[0] to access the first argument. Check args.length first to avoid ArrayIndexOutOfBoundsException.

Exercise 3 — Multiple Classes

Create two files:

  1. Greeter.java — a class with a static method greet(String name) that returns "Hello, " + name + "!"
  2. Main.java — a class with main() that calls Greeter.greet("Nimbus") and prints the result

Compile both: javac Greeter.java Main.java. Run: java Main. This teaches you that one class can use another — the foundation of every real project.

Exercise 4 — Package Practice

Create the following folder structure and files:

src/com/nimbus/model/User.java
src/com/nimbus/app/Main.java

User.java should have package com.nimbus.model; and a field String name with a constructor and getName() method.

Main.java should have package com.nimbus.app;, import com.nimbus.model.User, create a new User("Darshan"), and print their name.

Compile from the src/ directory: javac com/nimbus/model/User.java com/nimbus/app/Main.java

Run: java com.nimbus.app.Main

Exercise 5 — Memory Tracing

Read this code and answer the questions WITHOUT running it:

public class MemoryQuiz {
    public static void main(String[] args) {
        String a = new String("Hello");
        String b = a;
        String c = new String("Hello");
        a = null;
        b = null;
    }
}
  1. How many String objects are created on the Heap?
  2. After a = null;, is the first "Hello" eligible for GC?
  3. After b = null;, is the first "Hello" eligible for GC?
  4. Is c's "Hello" ever eligible for GC before main() ends?

Answers: (1) Two objects (the two new String("Hello") calls). (2) No — b still references it. (3) Yes — both a and b are null, no references remain. (4) No — c still holds a reference until main() returns.

Exercise 6 — JDK Exploration

Open your terminal and run these commands. Write down the output:

  1. javac -version — What JDK version do you have?
  2. java -version — What JRE/JVM version?
  3. Create Hello.java, compile it, then look for the Hello.class file. Open it in a text editor — what do you see? (Hint: unreadable binary, because it is bytecode, not source code)
  4. Run javap -c Hello — this disassembles the bytecode into human-readable instructions. Find the println call in the output.

This exercise proves that .class files are not source code — they are compiled bytecode that only the JVM can execute.
