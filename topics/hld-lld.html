Section 01

What is System Design?

Before you write a single line of code, you need a plan. System design is that plan. It answers: what are we building, how do the pieces fit together, and what does each piece do internally?

Why We Design Before We Code

Imagine you want to build a house. You wouldn't just start laying bricks randomly and hope it turns into something livable. You'd get blueprints first. You'd decide how many rooms, where the kitchen goes, how the plumbing works, where the electrical wiring runs. System design is the blueprint for software.

Without design, you get:

  • Code that works for one feature but breaks when you add another
  • No clear separation — everything tangled together like a bowl of spaghetti
  • Impossible to debug because nobody knows where anything lives
  • Can't add new team members because there's no map of the system

With design, you get:

  • A clear picture of every component before writing code
  • Each piece has one job and does it well
  • Easy to change one part without breaking others
  • New developers can read the design docs and understand the system in hours, not weeks
Analogy: The Restaurant

You want to open a restaurant. Before you start cooking, you need:

  • The Menu — what dishes will you serve? (features)
  • Kitchen Layout — where's the stove, prep table, fridge? (architecture)
  • Staffing Plan — who's the chef, who's the waiter, who washes dishes? (components/roles)
  • Workflow — customer orders → waiter writes ticket → chef cooks → waiter serves (data flow)
  • Recipes — exact steps, ingredients, temperatures for each dish (low-level logic)

The menu + kitchen layout + staffing plan = High Level Design.
The recipes with exact steps = Low Level Design.

How The Architecture Evolves: 4 Phases

Nimbus doesn't jump straight to the full architecture. It evolves through phases, each adding a new layer of complexity. This is exactly how real-world projects grow.

  PHASE 1: Frontend Only (LocalStorage)
  ======================================
  +-------------------+         +-------------------+
  |   Browser          | -----> |   LocalStorage     |
  |   HTML/CSS/JS      | <----- |   (browser memory) |
  +-------------------+         +-------------------+

  - No server. No database. Everything in the browser.
  - Data disappears when you clear browser storage.
  - Good for prototyping the UI.


  PHASE 2: Java Console + MySQL (JDBC)
  ======================================
  +-------------------+         +-------------------+         +-------------------+
  |   Console App      | -----> |   Java Classes     | -----> |   MySQL Database   |
  |   (Terminal/CMD)   | <----- |   Service + DAO    | <----- |   tbl_UserProfile  |
  |   NimbusApp.java   |         |   + JDBC           |         |   Booking          |
  +-------------------+         +-------------------+         +-------------------+

  - No browser. User interacts via terminal menus.
  - Data is permanent — stored in MySQL.
  - This is where we learn Java, OOP, JDBC.


  PHASE 3: Servlets + JSP (Web App)
  ======================================
  +-------------------+  HTTP   +-------------------+  calls  +-------------------+  JDBC   +----------+
  |   Browser          | -----> |   Tomcat Server    | -----> |   Service + DAO    | -----> |  MySQL   |
  |   HTML Forms       | <----- |   Servlets + JSP   | <----- |   (same classes)   | <----- |          |
  +-------------------+         +-------------------+         +-------------------+         +----------+

  - Now it's a real web application.
  - The SAME Service and DAO classes from Phase 2 are reused!
  - Servlets replace the console menus.


  PHASE 4: REST API + Angular (Modern)
  ======================================
  +-------------------+  HTTP   +-------------------+  calls  +-------------------+  JDBC   +----------+
  |   Angular App      | -----> |   REST Controllers | -----> |   Service + DAO    | -----> |  MySQL   |
  |   (TypeScript)     | <----- |   (JSON responses) | <----- |   (same classes)   | <----- |          |
  |   Components       |  JSON  |   Spring Boot      |         |                    |         |          |
  +-------------------+         +-------------------+         +-------------------+         +----------+

  - Frontend is now a separate Angular application.
  - Backend sends JSON, not HTML.
  - This is how modern apps (Instagram, Twitter, etc.) work.
Nimbus evolves through 4 phases — each phase reuses the Service and DAO layers from the previous phase.
Key Insight

Notice how the Service layer and DAO layer stay the same across Phases 2, 3, and 4? That's the power of layered architecture. The business logic doesn't care whether the user is typing in a terminal, clicking a web form, or using an Angular app. UserService.registerCustomer() works the same way regardless.

Client-Server Architecture

At the highest level, Nimbus uses client-server architecture. This is the most fundamental pattern in web development.

Analogy: Restaurant Again

Client = the customer sitting at the table. They make requests ("I want butter chicken").
Server = the kitchen. It receives requests, processes them, and sends back results (the dish).
Waiter = HTTP protocol. Carries requests and responses back and forth.

The customer never goes into the kitchen. The kitchen never comes to the table. The waiter is the middleman. That's client-server.

  CLIENT-SERVER MODEL
  ====================

  +------------------+                              +------------------+
  |                  |   1. "Give me booking #42"   |                  |
  |   CLIENT         |  --------------------------> |   SERVER         |
  |   (Browser)      |                              |   (Tomcat+Java)  |
  |                  |   2. "Here's booking #42:    |                  |
  |   - Sends        |       Status: In Transit"    |   - Receives     |
  |     requests     |  <------------------------   |     requests     |
  |   - Displays     |                              |   - Processes    |
  |     responses    |                              |   - Queries DB   |
  |   - No business  |                              |   - Sends back   |
  |     logic here   |       (HTTP Protocol)        |     responses    |
  +------------------+                              +------------------+

  KEY RULES:
  - Client NEVER talks to the database directly
  - Server NEVER shows UI to the user
  - They communicate ONLY through HTTP (request/response)
  - Multiple clients can talk to the same server simultaneously
Client-server: the browser asks, the server answers.

3-Tier Architecture

Nimbus uses 3-tier architecture, which splits the system into three separate concerns:

  3-TIER ARCHITECTURE
  ====================

  TIER 1: PRESENTATION            TIER 2: BUSINESS LOGIC           TIER 3: DATA
  (What the user sees)             (The brain)                      (Where data lives)
  +-------------------+           +-------------------+            +-------------------+
  |                   |           |                   |            |                   |
  |  - HTML pages     |           |  - Validate input |            |  - MySQL tables   |
  |  - CSS styling    |  ------> |  - Apply rules    |  -------> |  - Store data     |
  |  - JavaScript     |           |  - Calculate cost |            |  - Run queries    |
  |  - Forms          |  <------ |  - Check email    |  <------- |  - Return results |
  |  - Buttons        |           |    uniqueness     |            |                   |
  |  - Displays data  |           |  - Handle errors  |            |  tbl_UserProfile  |
  |                   |           |                   |            |  Booking          |
  +-------------------+           +-------------------+            +-------------------+
       NimbusApp.java                UserService.java                  UserDAO.java
       (or JSP/Angular)              TrackingService.java              BookingDAO.java
                                     BookingThread.java                DBConnection.java

  WHY 3 TIERS?
  - Change the UI without touching business logic
  - Change database from MySQL to PostgreSQL without touching UI
  - Test business logic without a browser or database
  - Different developers can work on different tiers
Each tier has a specific job and only talks to its immediate neighbor.
In Nimbus

Tier 1 (Presentation): In Phase 2, this is NimbusApp.java with its console menus. In Phase 3, this becomes JSP pages. In Phase 4, it's Angular components.

Tier 2 (Business Logic): UserService, TrackingService, BookingThread. These don't change between phases.

Tier 3 (Data): UserDAO, BookingDAO, DBConnection, and the MySQL database. These also stay the same.

The DAO Pattern: How It Works

DAO stands for Data Access Object. It's a design pattern that separates the database operations from the business logic. Think of it as a translator between your Java objects and your database tables.

Analogy: The Translator

You speak English. Your database speaks SQL. The DAO is a translator who converts your English instructions ("save this user") into SQL ("INSERT INTO tbl_UserProfile ...") and translates the database's SQL responses back into Java objects you can use.

UserDAO.java — insertUser method 
public boolean insertUser(User user) {
    String sql = "INSERT INTO tbl_UserProfile (Name, Email, Password, Role, Phone) VALUES (?, ?, ?, ?, ?)";

    try (Connection conn = DBConnection.getConnection();
         PreparedStatement ps = conn.prepareStatement(sql)) {

        ps.setString(1, user.getName());
        ps.setString(2, user.getEmail());
        ps.setString(3, user.getPassword());
        ps.setString(4, user.getRole());
        ps.setString(5, user.getPhone());

        int rowsAffected = ps.executeUpdate();
        return rowsAffected > 0;

    } catch (SQLException e) {
        e.printStackTrace();
        return false;
    }
}

Why Each Layer Exists: Separation of Concerns

Separation of concerns means each part of your code is responsible for one thing and one thing only. It's like a hospital: the surgeon doesn't also do the billing, and the receptionist doesn't perform surgery.

Presentation Show things to user, read input Service method names SQL, database tables, business rules
Service Apply business rules DAO method names, domain rules SQL, how the UI looks, HTTP
DAO Read/write database SQL, table structure, JDBC Business rules, what the user sees
Database Store data permanently Tables, columns, constraints Java, classes, methods

ER Diagram: Entity Relationship

An ER diagram shows the entities (tables) in your database and how they relate to each other. An entity is just a "thing" you want to store data about.

What is Cardinality?

Cardinality describes how many instances of one entity can be associated with instances of another entity. There are three types:

  CARDINALITY TYPES
  ==================

  1:1 (One-to-One)
  =================
  One person has ONE passport. One passport belongs to ONE person.

  +--------+     1:1     +----------+
  | Person | ----------- | Passport |
  +--------+             +----------+

  Example: User has one Profile. (Not used in Nimbus, but common.)


  1:N (One-to-Many)
  ==================
  One user has MANY bookings. Each booking belongs to ONE user.

  +--------+     1:N     +----------+
  | User   | ----------- | Booking  |
  +--------+             +----------+

  This is what Nimbus uses! One customer can book 100 parcels.
  Each of those 100 parcels has only ONE owner (the customer).


  M:N (Many-to-Many)
  ====================
  One student enrolls in MANY courses. One course has MANY students.

  +---------+     M:N     +----------+
  | Student | ----------- | Course   |
  +---------+             +----------+
                  |
                  v
          Requires a JUNCTION TABLE:
          +-------------------+
          | student_course     |
          | student_id (FK)   |
          | course_id  (FK)   |
          +-------------------+

  Not used in Nimbus, but very common in real apps.
Three types of relationships between tables.
In Nimbus

Nimbus uses 1:N (one-to-many) between User and Booking. The User_ID column in the Booking table is a foreign key that points back to tbl_UserProfile. When you query SELECT * FROM Booking WHERE User_ID = 5, you get ALL bookings for user #5.

Normalization: 1NF, 2NF, 3NF

Normalization is the process of organizing database tables to reduce redundancy (repeated data) and ensure data integrity. There are "normal forms" — rules your tables should follow.

First Normal Form (1NF)

Rule: Every cell must contain a single value (no lists, no arrays, no comma-separated values).

  VIOLATES 1NF (BAD):
  +-----+----------+---------------------------+
  | ID  | Name     | Phones                    |
  +-----+----------+---------------------------+
  | 1   | Darshan  | 9876543210, 9123456789    |  <-- TWO values in one cell!
  +-----+----------+---------------------------+

  FOLLOWS 1NF (GOOD):
  +-----+----------+--------------+
  | ID  | Name     | Phone        |
  +-----+----------+--------------+
  | 1   | Darshan  | 9876543210   |  <-- ONE value per cell
  | 1   | Darshan  | 9123456789   |  <-- separate row for second phone
  +-----+----------+--------------+
1NF: no multi-valued cells. One value per cell.

Second Normal Form (2NF)

Rule: Already in 1NF + every non-key column depends on the ENTIRE primary key (not just part of it). This only matters when you have a composite primary key (two columns as PK).

  VIOLATES 2NF (BAD):
  Composite PK: (Student_ID, Course_ID)
  +-----------+-----------+-----------+-------------+
  | Student_ID| Course_ID | Grade     | CourseName  |
  +-----------+-----------+-----------+-------------+
  | 1         | 101       | A         | Java        |
  | 1         | 102       | B         | Python      |
  | 2         | 101       | B+        | Java        |  <-- "Java" repeated!
  +-----------+-----------+-----------+-------------+

  CourseName depends ONLY on Course_ID, not on the full PK (Student_ID + Course_ID).
  That's a partial dependency = violates 2NF.

  FOLLOWS 2NF (GOOD):
  Table 1: Enrollment                  Table 2: Course
  +-----------+-----------+--------+   +-----------+------------+
  | Student_ID| Course_ID | Grade  |   | Course_ID | CourseName |
  +-----------+-----------+--------+   +-----------+------------+
  | 1         | 101       | A      |   | 101       | Java       |
  | 1         | 102       | B      |   | 102       | Python     |
  | 2         | 101       | B+     |   +-----------+------------+
  +-----------+-----------+--------+
  Now CourseName is in its own table, dependent on its own PK.
2NF: every non-key column depends on the FULL primary key.

Third Normal Form (3NF)

Rule: Already in 2NF + no non-key column depends on another non-key column (no transitive dependencies).

  VIOLATES 3NF (BAD):
  +-----+--------+-----------+-------------+
  | ID  | Name   | City      | State       |
  +-----+--------+-----------+-------------+
  | 1   | Darshan| Mumbai    | Maharashtra |
  | 2   | Ravi   | Pune      | Maharashtra |  <-- "Maharashtra" repeated!
  | 3   | Kiran  | Bangalore | Karnataka   |
  +-----+--------+-----------+-------------+

  State depends on City (not on ID directly).
  City --> State is a transitive dependency. Violates 3NF.

  FOLLOWS 3NF (GOOD):
  Table 1: User                     Table 2: City
  +-----+--------+-----------+      +-----------+-------------+
  | ID  | Name   | City      |      | City      | State       |
  +-----+--------+-----------+      +-----------+-------------+
  | 1   | Darshan| Mumbai    |      | Mumbai    | Maharashtra |
  | 2   | Ravi   | Pune      |      | Pune      | Maharashtra |
  | 3   | Kiran  | Bangalore |      | Bangalore | Karnataka   |
  +-----+--------+-----------+      +-----------+-------------+
3NF: no column depends on another non-key column. Eliminate transitive dependencies.
In Nimbus

Nimbus tables are already in 3NF:

  • 1NF: Every cell has a single value. No lists anywhere.
  • 2NF: Primary keys are single columns (User_ID, Booking_ID), so partial dependency can't happen.
  • 3NF: No non-key column depends on another non-key column. Status doesn't determine Cost. SenderAddr doesn't determine Weight.

  ONE GIANT TABLE (BAD):
  +-----+--------+-------+------+------+-----------+-----------+--------+------+--------+--------+
  | UID | Name   | Email | Pass | Role | BookingID | SenderAddr| RecAddr| Weight| Cost  | Status |

Indexes: Why Lookups Are Fast

An index is like the index at the back of a textbook. Instead of reading every page to find "polymorphism," you check the index, find "polymorphism: page 245," and go directly there.

Analogy: Library Card Catalog

Without an index, finding a book means walking through every shelf and checking every spine. That's a full table scan — MySQL reads every single row. With an index (like a card catalog sorted by title), you jump straight to the right shelf. MySQL uses B-tree indexes that let it find any row in milliseconds, even in tables with millions of rows.

  WITHOUT INDEX (Full Table Scan):
  SELECT * FROM Booking WHERE Booking_ID = 42;

  MySQL checks: Row 1... no. Row 2... no. Row 3... no. ... Row 41... no. Row 42... YES!
  Time: O(n) — checks every row. Slow for large tables.


  WITH INDEX on Booking_ID (default for PRIMARY KEY):
  SELECT * FROM Booking WHERE Booking_ID = 42;

  MySQL uses B-tree: 42 > 25? Go right. 42 < 50? Go left. 42 = 42? Found!
  Time: O(log n) — jumps directly. Fast even for millions of rows.


  INDEXES IN NIMBUS:
  +-------------------+---------------------+---------------------------+
  | Table             | Indexed Column      | Why                       |
  +-------------------+---------------------+---------------------------+
  | tbl_UserProfile   | User_ID (PK)        | Fast user lookups by ID   |
  | tbl_UserProfile   | Email (UNIQUE)      | Fast login + email check  |
  | Booking           | Booking_ID (PK)     | Fast booking lookups      |
  | Booking           | User_ID (FK)        | Fast "all bookings for    |
  |                   |                     | user X" queries           |
  +-------------------+---------------------+---------------------------+

  PRIMARY KEY columns are automatically indexed.
  UNIQUE columns are automatically indexed.
  FOREIGN KEY columns should be indexed (MySQL does this automatically).
Indexes turn slow full-table scans into fast direct lookups.
In Nimbus

When BookingDAO runs SELECT * FROM Booking WHERE Booking_ID = ?, MySQL doesn't scan every row. It uses the primary key index on Booking_ID to jump directly to the right row. Similarly, SELECT * FROM tbl_UserProfile WHERE Email = ? uses the unique index on Email for instant lookup during login.

What is an API?

Analogy: The Waiter

You're at a restaurant. You want food, but you can't walk into the kitchen and cook it yourself. Instead, you tell the waiter what you want. The waiter goes to the kitchen, gives your order to the chef, waits for the food, and brings it back to you.

An API (Application Programming Interface) is the waiter. It's the middleman between the frontend (you, the customer) and the backend (the kitchen). You send a request ("I want booking #42"), and the API sends back a response (the booking data).

You never touch the database directly. Just like you never touch the stove. The API handles it.

  WHAT IS AN API?
  ================

  Without API (bad):
  +--------+                              +----------+
  | Browser| ------- SQL queries -------> | Database |    DANGEROUS!
  +--------+                              +----------+    Anyone can delete data.

  With API (good):
  +--------+         +--------+          +----------+
  | Browser| --API-->| Server |---JDBC-->| Database |    SAFE!
  +--------+  (HTTP) +--------+  (SQL)   +----------+    Server controls access.

  The API is a CONTROLLED GATEWAY.
  It decides:
  - WHO can access what (authentication)
  - WHAT operations are allowed (authorization)
  - HOW data is formatted (validation)
  - WHAT data is returned (filtering)
APIs are controlled gateways — the frontend never touches the database directly.

REST Basics: GET, POST, PUT, DELETE

REST (Representational State Transfer) is a set of rules for how APIs should work. It uses HTTP methods to indicate what action you want to perform.
