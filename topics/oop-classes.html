Section 01

What is a Class vs Object?

Everything in Java revolves around classes and objects. Understanding the difference between the two is the foundation of object-oriented programming.

Blueprint vs Actual Thing

A class is a blueprint. It describes what something will look like and what it can do. An object is an actual instance of that blueprint — a real thing in memory with real values.

Analogy

A class is a cookie cutter. An object is an actual cookie. One cookie cutter can make 100 cookies. Each cookie is a separate thing — you can eat one without affecting the others. Same shape (same class), different existence (different objects).

Another way to think about it:

Class (Blueprint)Object (Instance)
Defines the structureLives in memory with actual data
Written once in a .java fileCreated at runtime with new
Has no data itselfEach object holds its own data
User — the concept of a userdarshan — a specific user with userId=1
One classMany objects from same class
Multiple objects from one class 
// One class definition
public class User {
    String fullName;
    String email;
}

// Three separate objects from the SAME class
User user1 = new User();
user1.fullName = "Darshan";
user1.email = "darshan@nimbus.com";

User user2 = new User();
user2.fullName = "Rahul";
user2.email = "rahul@nimbus.com";

User user3 = new User();
user3.fullName = "Sneha";
user3.email = "sneha@nimbus.com";

// Each object is independent. Changing user1 does NOT affect user2.
In Nimbus

The User class is the blueprint for every person who uses the parcel management system — senders, receivers, admins. When someone registers, we create a new User object with their details. The system might have 500 users at runtime, but all of them were created from the same User class definition.

Section 02

Anatomy of a Class

A class is made of four main parts: fields, constructors, methods, and access modifiers that control visibility.

  ┌─────────────────────────────────────────────────────┐
  │               public class User                      │
  ├─────────────────────────────────────────────────────┤
  │  FIELDS (Instance Variables)                         │
  │  ─────────────────────────                           │
  │  private int userId;           // data the object    │
  │  private String fullName;      // holds — each       │
  │  private String email;         // object gets its    │
  │  private String mobile;        // own copy            │
  │  private String address;                              │
  │  private String password;                             │
  │  private String role;                                 │
  ├─────────────────────────────────────────────────────┤
  │  CONSTRUCTORS                                        │
  │  ────────────                                        │
  │  public User() { }             // empty (no-arg)     │
  │  public User(all params) { }   // parameterized      │
  ├─────────────────────────────────────────────────────┤
  │  METHODS                                             │
  │  ───────                                             │
  │  public int getUserId() { }    // getters — read     │
  │  public void setUserId() { }   // setters — write    │
  │  public String toString() { }  // behavior           │
  ├─────────────────────────────────────────────────────┤
  │  ACCESS MODIFIERS                                    │
  │  ────────────────                                    │
  │  public  → class itself, constructors, methods       │
  │  private → fields (encapsulation)                    │
  └─────────────────────────────────────────────────────┘
Full anatomy of the Nimbus User class

Fields (Instance Variables)

Fields are the data that each object carries. Every User object gets its own copy of userId, fullName, email, etc. When you write private String fullName;, you are saying: "every User will have a fullName, and nobody outside this class can touch it directly."

Constructors

Constructors are special methods that run when you create a new object with new. They set up the object's initial state. A class can have multiple constructors (overloading).

Methods

Methods define what the object can do. Getters read field values. Setters change field values. Other methods perform operations — like toString() which converts the object to a readable string.

Access Modifiers

Keywords like public and private control who can see and use each part of the class. This is the enforcement layer of encapsulation. More on this in Section 6.

Section 03

Creating Objects — The new Keyword

When you write new User(), Java does three things: allocates memory on the heap, runs the constructor, and returns a reference to the object.

Object creation 
User darshan = new User();
//  │     │       │    │
//  │     │       │    └── () calls the constructor
//  │     │       └─────── new allocates memory on the HEAP
//  │     └─────────────── darshan is a REFERENCE variable (on the STACK)
//  └───────────────────── User is the type (which class)

What Happens in Memory

Java splits memory into two areas: the stack (fast, for method calls and local variables) and the heap (larger, for objects). When you create an object:

  1. Heap allocation: Java finds space on the heap and creates the actual User object there with all its fields (userId, fullName, email, etc.), initialized to defaults (0 for int, null for String).
  2. Constructor runs: The constructor code executes, setting up field values if any were passed in.
  3. Reference returned: The new expression returns a reference — basically a pointer/address to where the object lives on the heap. This reference is stored in the variable darshan on the stack.
    STACK (method frame)              HEAP (object storage)
  ┌──────────────────┐           ┌──────────────────────────┐
  │                  │           │  User object @0x7A3F     │
  │  darshan ──────────────────► │  userId   = 0            │
  │  [ref: 0x7A3F]   │           │  fullName = null         │
  │                  │           │  email    = null         │
  │  rahul ────────────────┐     │  mobile   = null         │
  │  [ref: 0x8B2C]   │    │     │  address  = null         │
  │                  │    │     │  password = null         │
  └──────────────────┘    │     │  role     = null         │
                          │     └──────────────────────────┘
                          │     ┌──────────────────────────┐
                          └───► │  User object @0x8B2C     │
                                │  userId   = 0            │
                                │  fullName = null         │
                                │  ...                     │
                                └──────────────────────────┘
Stack holds references, Heap holds actual objects
Key Insight

The variable darshan does NOT hold the User object itself. It holds a reference (memory address) that points to the object on the heap. This is why when you do User copy = darshan;, both copy and darshan point to the same object. Changing copy.fullName also changes darshan.fullName because they are the same object.

Reference vs Object 
User darshan = new User();       // 1 object on heap, darshan points to it
User copy = darshan;              // copy points to the SAME object

copy.setFullName("Changed!");
System.out.println(darshan.getFullName()); // "Changed!" — same object!

User separate = new User();      // THIS creates a second, separate object
separate.setFullName("Different");
System.out.println(darshan.getFullName()); // Still "Changed!" — different object
Section 04

Constructors

Constructors are special methods that initialize objects when they are created. They have the same name as the class and no return type (not even void).

Default (No-Arg) Constructor

A no-arg constructor takes zero parameters. It creates an object with default/empty values. This is useful when you want to build up the object piece by piece after creation.

No-arg constructor 
public User() {
    // All fields stay at defaults: int=0, String=null, boolean=false
}

// Usage:
User user = new User();
user.setFullName("Darshan");  // set values one at a time
user.setEmail("darshan@nimbus.com");

Parameterized Constructor

A parameterized constructor takes arguments and uses them to set field values immediately during creation. One line creates a fully initialized object.

Parameterized constructor 
public User(int userId, String fullName, String email,
            String mobile, String address, String password, String role) {
    this.userId = userId;
    this.fullName = fullName;
    this.email = email;
    this.mobile = mobile;
    this.address = address;
    this.password = password;
    this.role = role;
}

// Usage — fully initialized in one line:
User darshan = new User(1, "Darshan", "darshan@nimbus.com",
                        "9876543210", "Mumbai", "hashed_pw", "admin");

Constructor Overloading

A class can have multiple constructors with different parameter lists. Java picks the right one based on what arguments you pass. This is called overloading.

Overloaded constructors 
public class User {
    // Constructor 1: No-arg
    public User() { }

    // Constructor 2: All fields
    public User(int userId, String fullName, String email,
                String mobile, String address, String password, String role) {
        this.userId = userId;
        this.fullName = fullName;
        this.email = email;
        this.mobile = mobile;
        this.address = address;
        this.password = password;
        this.role = role;
    }
}

// Java figures out which constructor to call:
User empty = new User();                              // calls Constructor 1
User full = new User(1, "Darshan", "d@n.com",
                     "987", "Mumbai", "pw", "admin"); // calls Constructor 2

Constructor Chaining with this()

One constructor can call another constructor in the same class using this(). This avoids duplicating initialization code. The this() call must be the first statement in the constructor.

Constructor chaining 
public class User {
    public User() {
        this(0, "Unknown", "none@none.com", "", "", "", "user");
        // Calls the full constructor with default values
        // this() MUST be the first statement
    }

    public User(int userId, String fullName, String email,
                String mobile, String address, String password, String role) {
        this.userId = userId;
        this.fullName = fullName;
        this.email = email;
        this.mobile = mobile;
        this.address = address;
        this.password = password;
        this.role = role;
    }
}
In Nimbus

The Nimbus User class has both an empty constructor and a full constructor. Why? The empty constructor is needed when building a User from a database ResultSet. You create an empty User, then loop through the ResultSet columns and call setters one by one: user.setUserId(rs.getInt("user_id")), user.setFullName(rs.getString("full_name")), etc. You cannot use the parameterized constructor because the values come from the ResultSet row by row, not all at once in code.

What If You Write No Constructor?

If you do not write any constructor, Java silently provides a default no-arg constructor for you. But the moment you write any constructor (even a parameterized one), Java stops providing the default. If you then try new User() without writing your own no-arg constructor, you will get a compile error. This is a common trap.

Section 05

The this Keyword

this refers to the current object — the specific instance that the method or constructor is running on.

Disambiguating Parameters from Fields

The most common use of this is when a parameter has the same name as a field. Without this, Java thinks you are assigning the parameter to itself.

Without this (BUG)
public void setFullName(String fullName) {
    fullName = fullName;
    // Assigns parameter to itself!
    // Field stays null
}
With this (CORRECT)
public void setFullName(String fullName) {
    this.fullName = fullName;
    // this.fullName = the field
    // fullName = the parameter
}

this() to Call Another Constructor

As shown in the constructor chaining section, this() (with parentheses) calls another constructor of the same class. this (without parentheses) refers to the current object. Different things.

When You MUST Use this

  • Parameter shadows field: When constructor/method parameter has the same name as a field — you MUST use this.fieldName to refer to the field.
  • Constructor chaining: this(args) to call another constructor. Must be the first statement.
  • Passing current object: When a method needs to pass the current object as an argument: someMethod(this).
  • Returning current object (fluent API): return this; to enable method chaining like user.setName("X").setEmail("Y").
Rule of Thumb

If the parameter name is different from the field name (e.g., parameter name, field fullName), you do not need this. But most professional code uses the same names for clarity, which makes this essential in every setter and constructor.

Section 06

Access Modifiers

Access modifiers control who can see and use a class, field, method, or constructor. Java has four levels of access.

Modifier Same Class Same Package Subclass (other pkg) Everywhere
public Yes Yes Yes Yes
protected Yes Yes Yes No
default (no keyword) Yes Yes No No
private Yes No No No

Why Each Exists

  • public — Use for things the outside world needs to interact with: the class itself, getters, setters, service methods. The API surface.
  • private — Use for internal data and helper methods. Nobody outside the class should know or care about these. This is the default choice for fields.
  • protected — Use when subclasses (child classes) need access but the general public should not. Useful for inheritance hierarchies.
  • default (package-private) — Use when classes within the same package need to collaborate closely but outsiders should not see the details. You get this by writing no modifier at all.
Access modifiers in practice 
public class User {                    // public — anyone can use this class
    private int userId;                // private — only User class can touch
    private String fullName;           // private — only User class can touch
    private String password;           // private — DEFINITELY only internal

    public String getFullName() {      // public — external code reads the name
        return this.fullName;
    }

    public void setPassword(String password) {  // public — external code sets pw
        this.password = hashPassword(password); // but we control HOW it's set
    }

    private String hashPassword(String raw) {   // private — internal helper
        // nobody outside needs to know about hashing
        return BCrypt.hash(raw);
    }
}
In Nimbus

Every field in the Nimbus User class is private: userId, fullName, email, mobile, address, password, role. All access goes through public getters and setters. The class itself is public because other packages (com.nimbus.dao, com.nimbus.service) need to create and use User objects.

Section 07

Encapsulation

Encapsulation is the practice of hiding internal data and only exposing controlled access through methods. It is the #1 reason for private fields + public getters/setters.

Why Not Just Make Everything Public?

If every field is public, anyone can set any value directly. There are no guardrails.

Public Fields (Dangerous)
public class User {
    public String email; // anyone can set
}

User u = new User();
u.email = "not-an-email"; // no validation!
u.email = "";             // empty? sure!
u.email = null;           // null? why not!

No control. Any garbage goes in. Bugs everywhere.

Private + Setter (Safe)
public class User {
    private String email;

    public void setEmail(String email) {
        if (!email.contains("@")) {
            throw new IllegalArgumentException(
                "Invalid email");
        }
        this.email = email;
    }
}

Invalid data gets rejected. Object is always valid.

Future-Proofing: The Real Power

The greatest benefit of encapsulation is future-proofing. Today your setter just assigns the value. Tomorrow you add validation, logging, formatting, or notifications — and no external code has to change.

Evolution of a setter over time 
// Version 1: Simple assignment
public void setEmail(String email) {
    this.email = email;
}

// Version 2: Added validation (3 months later)
public void setEmail(String email) {
    if (email == null || !email.contains("@")) {
        throw new IllegalArgumentException("Invalid email format");
    }
    this.email = email;
}

// Version 3: Added normalization (6 months later)
public void setEmail(String email) {
    if (email == null || !email.contains("@")) {
        throw new IllegalArgumentException("Invalid email format");
    }
    this.email = email.trim().toLowerCase(); // normalize
}

// The external code NEVER changed. It always just called:
// user.setEmail("Darshan@Nimbus.COM");
// Encapsulation let us evolve the internals without breaking anything.
In Nimbus

Right now, Nimbus setEmail() does a simple assignment. But imagine later we want to: (1) validate email format, (2) check for duplicate emails in the database, (3) convert to lowercase. With encapsulation, we just update the setter. Every place in the codebase that calls user.setEmail() — the registration servlet, the profile update, the admin panel — automatically gets the new behavior. Zero refactoring.

The Three Rules of Encapsulation

  1. Make fields private. Always. No exceptions for model classes.
  2. Provide public getters for fields that external code needs to read.
  3. Provide public setters for fields that external code needs to write — with validation logic inside.
Section 08

Getters and Setters

Getters read field values. Setters write field values. They follow strict naming conventions that Java frameworks rely on.

Naming Convention

FieldGetterSetter
String fullNamegetFullName()setFullName(String fullName)
int userIdgetUserId()setUserId(int userId)
String emailgetEmail()setEmail(String email)
boolean activeisActive()setActive(boolean active)
Boolean Getters

For boolean fields, the getter convention is isFieldName(), not getFieldName(). So boolean active has isActive(), not getActive(). Java frameworks like Jackson and Hibernate rely on this convention to auto-detect boolean properties.

Pattern: Every Getter and Setter Looks the Same

Standard getter/setter pattern 
// Getter: returns the field value
public String getFullName() {
    return this.fullName;       // or just: return fullName;
}

// Setter: takes a parameter, assigns to field
public void setFullName(String fullName) {
    this.fullName = fullName;   // this.field = parameter
}

// Boolean getter: uses "is" instead of "get"
public boolean isActive() {
    return this.active;
}

Eclipse Auto-Generation Shortcut

You do not have to type getters and setters by hand. In Eclipse:

  1. Write your private fields.
  2. Right-click in the editor → Source → Generate Getters and Setters.
  3. Select which fields you want getters/setters for.
  4. Click Generate. Done in 3 seconds.

Keyboard shortcut: Alt + Shift + S, then R.

Why Bother?

If getters and setters are so boilerplate, why have them? Because they give you a hook point. Today the getter just returns the field. Tomorrow you might add logging, caching, lazy loading, or computed values — and external code never knows. The contract (method signature) stays the same. The implementation evolves.

Section 09

The static Keyword

static means "belongs to the class itself, not to any specific object." Static members are shared across all instances.

Static Fields

A static field exists once per class, not once per object. All objects share the same static field.

Static field example 
public class User {
    private static int userCount = 0;  // shared across ALL User objects
    private int userId;                // each object has its own copy
    private String fullName;

    public User(String fullName) {
        userCount++;                   // increment the shared counter
        this.userId = userCount;       // assign unique ID
        this.fullName = fullName;
    }

    public static int getUserCount() {
        return userCount;              // access without creating an object
    }
}

User u1 = new User("Darshan");    // userCount = 1
User u2 = new User("Rahul");     // userCount = 2
User u3 = new User("Sneha");     // userCount = 3

System.out.println(User.getUserCount()); // 3 — called on the CLASS, not an object
  ┌─────────────────────────────────────────────────┐
  │       User CLASS (shared static memory)          │
  │                                                  │
  │  static userCount = 3                            │
  └─────────────────────────────────────────────────┘
            │              │              │
     ┌──────┴──────┐ ┌────┴──────┐ ┌────┴──────┐
     │  u1 object  │ │ u2 object │ │ u3 object │
     │  userId = 1 │ │ userId = 2│ │ userId = 3│
     │  "Darshan"  │ │  "Rahul"  │ │  "Sneha"  │
     └─────────────┘ └───────────┘ └───────────┘
     Each object has        All objects share
     its OWN userId         the SAME userCount
Static field (shared) vs Instance field (per-object)

Static Methods

A static method belongs to the class, not an object. You call it on the class name: User.getUserCount(), not user1.getUserCount(). Static methods cannot access instance fields (because there is no specific object to read from).

Static method rules 
public class MathUtils {
    // Static utility method — no object needed
    public static double calculateDistance(double x1, double y1, double x2, double y2) {
        return Math.sqrt(Math.pow(x2 - x1, 2) + Math.pow(y2 - y1, 2));
    }
}

// Call directly on the class:
double dist = MathUtils.calculateDistance(0, 0, 3, 4); // 5.0

// Common examples you already use:
Math.sqrt(16);           // Math is a class, sqrt is static
Integer.parseInt("42");  // Integer is a class, parseInt is static
Arrays.sort(myArray);    // Arrays is a class, sort is static

Why is main() Static?

The JVM needs to start your program without creating any objects first. The entry point main() must be callable on the class directly: App.main(args). If main() were an instance method, the JVM would need to do new App().main(args) — but who decides which constructor to call? What arguments? Static avoids this chicken-and-egg problem entirely.

Static Block

A static block runs once when the class is first loaded into memory, before any objects are created or static methods are called. Used for one-time initialization.

Static block 
public class DatabaseConfig {
    private static String dbUrl;

    static {
        // Runs once when class is loaded
        dbUrl = System.getenv("DATABASE_URL");
        if (dbUrl == null) {
            dbUrl = "jdbc:mysql://localhost:3306/nimbus";
        }
        System.out.println("Database URL configured: " + dbUrl);
    }
}

When to Use Static vs Instance

Use StaticUse Instance
Utility/helper methods (Math.sqrt)Methods that use object data (user.getEmail())
Constants (MAX_LOGIN_ATTEMPTS)Data unique to each object (fullName)
Counters/trackers shared across all objectsState that varies per object
Factory methods (User.createAdmin())Behavior that depends on the object's state
Section 10

The final Keyword

final means "this cannot be changed." It applies to variables, methods, and classes — each with a different meaning.

Final Variable (Constant)

A final variable can only be assigned once. After that, any attempt to change it causes a compile error. Constants use the naming convention ALL_CAPS_WITH_UNDERSCORES.

Final variables 
// Class-level constant: static + final
public static final int MAX_LOGIN_ATTEMPTS = 5;
public static final String DEFAULT_ROLE = "user";
public static final double TAX_RATE = 0.18;

// Local constant:
final String greeting = "Hello";
greeting = "Hi"; // COMPILE ERROR — cannot reassign final variable

// Usage:
if (attempts > MAX_LOGIN_ATTEMPTS) {
    lockAccount();
}

Final Method

A final method cannot be overridden by subclasses. Use this when the method's behavior must stay exactly as written.

Final method 
public class User {
    public final String getRole() {
        return this.role;  // no subclass can override this
    }
}

Final Class

A final class cannot be extended (no subclasses allowed). String is a final class in Java — you cannot write class MyString extends String.

Final class 
public final class SecurityUtils {
    // Nobody can extend this class and override its security methods
    public static String hashPassword(String password) {
        return BCrypt.hashpw(password, BCrypt.gensalt());
    }
}

// This would cause a COMPILE ERROR:
// public class MySecurityUtils extends SecurityUtils { }  // NOPE
final OnMeaningNaming
VariableCannot be reassignedMAX_LOGIN_ATTEMPTS
MethodCannot be overridden by subclassesNormal camelCase
ClassCannot be extended (no child classes)Normal PascalCase
Section 11

toString()

Every Java object inherits a toString() method. The default one prints garbage like User@7a81197d. Override it for meaningful output.

Why Override?

When you print an object with System.out.println(user), Java automatically calls user.toString(). Without an override, you get the class name + memory hash — completely useless for debugging.

Default toString()
System.out.println(user);
// Output: User@7a81197d
// What does that tell you? Nothing.
Overridden toString()
System.out.println(user);
// Output: User{userId=1,
//   fullName='Darshan',
//   email='d@n.com', role='admin'}
// Instantly useful for debugging.

Nimbus User toString()

User.java — toString() 
@Override
public String toString() {
    return "User{" +
            "userId=" + userId +
            ", fullName='" + fullName + '\'' +
            ", email='" + email + '\'' +
            ", mobile='" + mobile + '\'' +
            ", address='" + address + '\'' +
            ", role='" + role + '\'' +
            '}';
    // Note: password is intentionally EXCLUDED — never print passwords
}

Nimbus Booking toString()

Booking.java — toString() 
@Override
public String toString() {
    return "Booking{" +
            "bookingId=" + bookingId +
            ", trackingNumber='" + trackingNumber + '\'' +
            ", senderId=" + senderId +
            ", senderName='" + senderName + '\'' +
            ", receiverName='" + receiverName + '\'' +
            ", receiverMobile='" + receiverMobile + '\'' +
            ", receiverAddress='" + receiverAddress + '\'' +
            ", parcelType='" + parcelType + '\'' +
            ", weight=" + weight +
            ", price=" + price +
            ", status='" + status + '\'' +

Key Detail

The @Override annotation tells the compiler "I am intentionally replacing the parent class version of this method." If you misspell toString (e.g., tostring), the annotation will throw a compile error instead of silently creating a new method that never gets called. Always use @Override.

Section 12

equals() and hashCode()

== checks if two references point to the same object in memory. equals() checks if two objects have the same content. This difference matters enormously.

The Problem with ==

== vs equals() 
User user1 = new User(1, "Darshan", "d@n.com", "987", "Mumbai", "pw", "admin");
User user2 = new User(1, "Darshan", "d@n.com", "987", "Mumbai", "pw", "admin");

// Same data, but:
System.out.println(user1 == user2);      // false — different objects in memory!
System.out.println(user1.equals(user2)); // also false... by DEFAULT

// Default equals() from Object class does the same thing as == (checks reference)
// You must OVERRIDE equals() to compare content

Overriding equals()

Proper equals() for User 
@Override
public boolean equals(Object obj) {
    // 1. Same reference? Same object.
    if (this == obj) return true;

    // 2. Null or different class? Not equal.
    if (obj == null || getClass() != obj.getClass()) return false;

    // 3. Cast and compare the field(s) that define identity
    User other = (User) obj;
    return this.userId == other.userId;  // users are equal if same userId
}

// Now:
System.out.println(user1.equals(user2)); // true — same userId!

The hashCode() Contract

Rule: If equals() returns true for two objects, their hashCode() must return the same value. If you override equals(), you must override hashCode(). Breaking this contract causes bugs in HashMap, HashSet, etc.

hashCode() matching equals() 
@Override
public int hashCode() {
    return Integer.hashCode(userId); // based on same field as equals()
}

// Or using Objects utility (for multiple fields):
@Override
public int hashCode() {
    return java.util.Objects.hash(userId);
}

When You NEED to Override

You need equals() and hashCode() when you use objects in:

  • HashSet — to detect duplicates. Without proper equals/hashCode, a HashSet will treat two User objects with userId=1 as different entries.
  • HashMap — as keys. The key lookup uses hashCode() to find the bucket and equals() to match the exact key.
  • List.contains() — calls equals() to check if an element exists.
  • Any deduplication logic in your code.
In Nimbus

If you store User objects in a HashSet<User> (e.g., to track logged-in users), two User objects with userId=1 would be treated as two separate entries without proper equals() and hashCode(). The same user could appear twice in the set. Override both methods, base them on userId, and the set correctly treats them as one.

Garbage Collection

When an object has zero references pointing to it, Java's garbage collector eventually reclaims that memory. You do not have to free memory manually (unlike C/C++). But creating too many objects in a tight loop can still cause performance issues because the garbage collector has to work harder.
