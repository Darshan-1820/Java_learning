Section 01

What is Inheritance?

Inheritance is the mechanism where one class acquires all the fields and methods of another class. The child class gets everything from the parent — for free — and can add its own stuff on top.

The Family Analogy

Think about a real family. A child inherits features from their parents: eye color, hair type, height tendencies. The child doesn't have to "build" those features from scratch — they come built-in. But the child also develops their own unique traits: maybe they learn guitar, or they're left-handed. They are their parents + something extra.

That is exactly how Java inheritance works:

  • Parent class (superclass) — the base class that has common fields and methods
  • Child class (subclass) — inherits everything from the parent, then adds or customizes
  • The child IS-A type of the parent. A CustomerTrackingService IS-A TrackingService.
Analogy

A smartphone inherits from phone. Every smartphone can make calls and send texts (inherited). But a smartphone also has apps, a camera, GPS (its own additions). A smartphone IS-A phone. But a phone is NOT necessarily a smartphone.

The "is-a" Relationship

Inheritance models an "is-a" relationship. Before using inheritance, ask: "Is [Child] a type of [Parent]?" If yes, inheritance makes sense. If not, you probably want composition (covered later).

Relationship Is-a test Inheritance?
CustomerTrackingService → TrackingService A CustomerTrackingService IS-A TrackingService Yes
BookingThread → Thread A BookingThread IS-A Thread Yes
DuplicateEmailException → Exception A DuplicateEmailException IS-A Exception Yes
Car → Engine A Car IS-A Engine? No. Car HAS-A Engine. No — use composition
In Nimbus

Every extends relationship in Nimbus passes the is-a test. CustomerTrackingService IS-A TrackingService. BookingNotFoundException IS-A Exception. BookingThread IS-A Thread. These are not random — each one means the child is a specialized version of the parent.

Section 02

The extends Keyword

In Java, you create an inheritance relationship using the extends keyword. The child class says: "I am everything my parent is, plus more."

Basic Syntax

Inheritance Syntax 
// Parent class
public class TrackingService {
    protected String serviceName;

    public void trackParcel(String trackingId) {
        System.out.println("Tracking parcel: " + trackingId);
    }
}

// Child class — inherits from TrackingService
public class CustomerTrackingService extends TrackingService {
    // This class automatically has:
    //   - serviceName field (inherited)
    //   - trackParcel() method (inherited)

    // Plus its own additions:
    public void viewDeliveryEstimate(String trackingId) {
        System.out.println("Estimated delivery for " + trackingId + ": 3 days");
    }
}
Line-by-line explanation
1
public class TrackingService {
This is the parent (superclass). It defines the common behavior shared by all tracking services.
2
protected String serviceName;
protected means child classes CAN access this directly. If it were private, child classes could not see it (they'd need getters).
3
public void trackParcel(String trackingId)
A public method. Any child class inherits this and can call it as if it were its own.
4
public class CustomerTrackingService extends TrackingService {
The extends keyword creates the inheritance link. CustomerTrackingService now has everything TrackingService has.
5
public void viewDeliveryEstimate(String trackingId)
A new method that only CustomerTrackingService has. The parent TrackingService does NOT have this method.

What Gets Inherited (and What Doesn't)

Inherited NOT Inherited
public methods Constructors (must call via super())
protected methods and fields private fields (exist but not directly accessible)
public fields private methods
Package-private members (same package only) —
Important

private fields still exist in the child object — they are allocated in memory. The child just cannot access them directly. It must use the parent's public/protected getters and setters. Think of it as: you inherited your parent's safe deposit box, but you need the key (getter) to open it.

Section 03

Types of Inheritance

Java supports three types of class inheritance. It deliberately blocks a fourth (multiple inheritance) for a very specific reason.

1. Single Inheritance

One parent, one child. The simplest and most common form.

  ┌─────────────────────┐
  │   TrackingService    │  ← Parent
  │─────────────────────│
  │  trackParcel()       │
  └──────────┬──────────┘
             │ extends
  ┌──────────▼──────────┐
  │ CustomerTracking     │  ← Child
  │      Service         │
  │─────────────────────│
  │  viewEstimate()      │
  └─────────────────────┘
Single inheritance: TrackingService → CustomerTrackingService

2. Multilevel Inheritance

A chain: A → B → C. The grandchild inherits from both parent and grandparent.

  ┌─────────────────────┐
  │      Vehicle         │  ← Grandparent
  │─────────────────────│
  │  start(), stop()     │
  └──────────┬──────────┘
             │ extends
  ┌──────────▼──────────┐
  │        Car           │  ← Parent
  │─────────────────────│
  │  openTrunk()         │
  └──────────┬──────────┘
             │ extends
  ┌──────────▼──────────┐
  │     ElectricCar      │  ← Child
  │─────────────────────│
  │  chargeBattery()     │
  └─────────────────────┘
Multilevel: ElectricCar has start(), stop(), openTrunk(), AND chargeBattery()

3. Hierarchical Inheritance

One parent, multiple children. Each child gets the parent's stuff independently.

              ┌─────────────────────┐
              │   TrackingService    │  ← Parent
              │─────────────────────│
              │  trackParcel()       │
              └──────────┬──────────┘
              ┌──────────┴──────────┐
              │ extends              │ extends
  ┌───────────▼──────────┐  ┌───────▼──────────────┐
  │ CustomerTracking      │  │ OfficerTracking       │
  │      Service          │  │      Service          │
  │──────────────────────│  │──────────────────────│
  │ viewEstimate()        │  │ updateStatus()        │
  └──────────────────────┘  └──────────────────────┘
Hierarchical: Both children inherit trackParcel() but add their own methods
In Nimbus

Nimbus uses hierarchical inheritance for tracking: TrackingService is the parent, with CustomerTrackingService and OfficerTrackingService as children. Customers can view estimates; officers can update statuses. Both can track parcels (inherited). And single inheritance for history: BaseHistoryService → CustomerHistoryService.

4. Multiple Inheritance — NOT Allowed in Java

What if a class could extend two parents? Java says no. Here is why:

  ┌──────────────┐     ┌──────────────┐
  │   ClassA      │     │   ClassB      │
  │──────────────│     │──────────────│
  │  display()    │     │  display()    │
  └──────┬───────┘     └──────┬───────┘
         │ extends?            │ extends?
         └──────────┬─────────┘
            ┌───────▼────────┐
            │     ClassC      │
            │────────────────│
            │  display() ??   │  ← Which parent's display()?
            └────────────────┘
The Diamond Problem: ClassC inherits display() from BOTH parents. Which one wins?

This is called the Diamond Problem. If both parents have a method with the same signature, the compiler has no way to decide which one the child should use. Java avoids this ambiguity entirely by forbidding multiple class inheritance.

This WILL NOT compile 
// COMPILER ERROR: Java does not support multiple inheritance
public class ClassC extends ClassA, ClassB {  // WRONG
    // Which display() do I inherit?
}
What about interfaces?

Java allows a class to implement multiple interfaces — that is different. Interfaces only define method signatures (contracts), not implementations (before default methods). So there is no ambiguity about "which code to run." You will learn this in the Abstraction topic.

Section 04

The super Keyword

When a child needs to reference its parent, it uses super. It is the bridge between child and parent.

Three Uses of super

1. super.method() — Call parent's version of a method

Calling parent method 
public class TrackingService {
    public String getTrackingInfo(String id) {
        return "Parcel " + id + " is in transit";
    }
}

public class CustomerTrackingService extends TrackingService {
    @Override
    public String getTrackingInfo(String id) {
        // Call parent's version first, then add to it
        String basic = super.getTrackingInfo(id);
        return basic + " | Estimated delivery: 3 days";
    }
}

Without super.getTrackingInfo(id), calling getTrackingInfo() would call the child's own version (infinite recursion!). super explicitly says: "call the parent's version."

2. super() — Call parent's constructor

Calling parent constructor 
public class TrackingService {
    protected String serviceName;

    public TrackingService(String serviceName) {
        this.serviceName = serviceName;
    }
}

public class CustomerTrackingService extends TrackingService {
    private String customerId;

    public CustomerTrackingService(String serviceName, String customerId) {
        super(serviceName);         // MUST be the first line
        this.customerId = customerId;
    }
}

3. super.field — Access parent's field

Accessing parent field 
public class TrackingService {
    protected String serviceName = "General Tracking";
}

public class CustomerTrackingService extends TrackingService {
    protected String serviceName = "Customer Tracking";  // hides parent's field

    public void printNames() {
        System.out.println(serviceName);        // "Customer Tracking"
        System.out.println(super.serviceName);  // "General Tracking"
    }
}
When super() is MANDATORY

If the parent class has no default (no-arg) constructor — meaning it only has constructors that take parameters — the child must explicitly call super(args). Otherwise the compiler will try to insert super() automatically, fail to find a no-arg constructor, and throw a compilation error.

Section 05

Constructor Chaining

When you create a child object, the parent's constructor always runs first. Always. No exceptions. This is called constructor chaining.

The Rule

Every child constructor calls a parent constructor — either you write super(...) explicitly, or the compiler inserts super() (no-arg) automatically. The parent initializes its part of the object before the child initializes its own part.

Analogy

Building a house: you pour the foundation first (parent constructor), then build the walls on top (child constructor). You cannot build walls without a foundation. Java enforces this order automatically.

Full Example with Output

ConstructorChainDemo.java 
class Grandparent {
    Grandparent() {
        System.out.println("1. Grandparent constructor");
    }
}

class Parent extends Grandparent {
    Parent() {
        // super() is inserted automatically by compiler
        System.out.println("2. Parent constructor");
    }
}

class Child extends Parent {
    Child() {
        // super() is inserted automatically by compiler
        System.out.println("3. Child constructor");
    }
}

public class ConstructorChainDemo {
    public static void main(String[] args) {
        Child c = new Child();
    }
}

// OUTPUT:
// 1. Grandparent constructor
// 2. Parent constructor
// 3. Child constructor
Execution order explained
1
new Child()
You call the Child constructor. But before it can run its body, it calls Parent's constructor via implicit super().
2
Parent() calls super()
Parent's constructor is called, but before IT runs, it calls Grandparent's constructor via implicit super().
3
Grandparent() runs first
Grandparent has no parent (well, Object, which has an empty constructor). So it prints "1. Grandparent constructor" first.
4
Parent() body runs
After Grandparent finishes, control returns to Parent. It prints "2. Parent constructor".
5
Child() body runs
After Parent finishes, control returns to Child. It prints "3. Child constructor". Object is fully built.
Key Takeaway

Constructor order is top-down (grandparent → parent → child). The topmost ancestor always initializes first. This guarantees that by the time a child constructor runs, all inherited fields from the parent are already initialized and ready to use.

Section 06

Method Overriding

When a child class provides its own version of a method that already exists in the parent, that is method overriding. The child says: "I know you gave me this behavior, but I want to do it differently."

How It Works

Method Overriding 
public class TrackingService {
    public String getStatus(String trackingId) {
        return "Status: In Transit";
    }
}

public class OfficerTrackingService extends TrackingService {
    @Override
    public String getStatus(String trackingId) {
        // Officer gets detailed internal status
        return "Status: In Transit | Warehouse: Mumbai-Central | Bay: 14";
    }
}

The @Override Annotation

@Override is optional but you should always use it. It tells the compiler: "I intend to override a parent method." If you misspell the method name or get the parameters wrong, the compiler will catch your mistake.

With @Override (safe)
@Override
public String getStatus(String id) {
    // Compiler verifies this
    // method exists in parent
}
Without @Override (risky)
// Typo: getSatus instead of getStatus
public String getSatus(String id) {
    // No error! This is a NEW method,
    // not an override. Bug is silent.
}

Rules of Overriding

Rule Explanation
Same method signature Name, parameter types, and order must match exactly
Same or covariant return type Return type must be the same or a subclass of the parent's return type
Cannot reduce visibility If parent is public, child cannot make it protected or private
Cannot throw broader checked exceptions Child can throw the same or narrower exceptions, not broader ones
Cannot override final methods A final method is locked — the parent says "nobody changes this"
Cannot override static methods Static methods belong to the class, not instances. They are hidden, not overridden (see next section)
Visibility rule violation 
public class TrackingService {
    public String getStatus(String id) { ... }
}

public class CustomerTrackingService extends TrackingService {
    // COMPILER ERROR: Cannot reduce visibility from public to protected
    @Override
    protected String getStatus(String id) { ... }
}
Section 07

Method Hiding (Static Methods)

When a child class defines a static method with the same signature as a parent's static method, it hides the parent's version — it does NOT override it. The difference is subtle but important.

Overriding vs Hiding

StaticHidingDemo.java 
class Parent {
    // Instance method — can be overridden
    public String greet() { return "Hello from Parent"; }

    // Static method — can be hidden, NOT overridden
    public static String info() { return "Parent info"; }
}

class Child extends Parent {
    @Override
    public String greet() { return "Hello from Child"; }

    // This HIDES Parent.info(), it does NOT override it
    public static String info() { return "Child info"; }
}

public class StaticHidingDemo {
    public static void main(String[] args) {
        Parent obj = new Child();

        // Instance method: calls CHILD's version (runtime polymorphism)
        System.out.println(obj.greet());   // "Hello from Child"

        // Static method: calls PARENT's version (compile-time, based on reference type)
        System.out.println(obj.info());    // "Parent info"  ← SURPRISE!
    }
}
Why static behaves differently
1
Parent obj = new Child();
The reference type is Parent, but the actual object is Child.
2
obj.greet() → "Hello from Child"
Instance methods use runtime polymorphism. Java looks at the actual object type (Child) and calls Child's greet().
3
obj.info() → "Parent info"
Static methods use compile-time binding. The compiler sees the reference type is Parent, so it calls Parent.info(). The actual object type is irrelevant for static calls.
Rule of Thumb

Overriding = instance methods, resolved at runtime (which object am I?).
Hiding = static methods, resolved at compile time (what type is the reference?).
This is why you should always call static methods using the class name (Parent.info()), not an object reference.

Section 08

The Object Class

Every class in Java — whether you write extends or not — implicitly extends java.lang.Object. It is the root of the entire class hierarchy. Every object you ever create IS-A Object.

Implicit Inheritance

Implicit Object Extension 
// These two are IDENTICAL:
public class TrackingService { }
public class TrackingService extends Object { }

// This means TrackingService automatically has:
// toString(), equals(), hashCode(), getClass(), clone(), finalize(), etc.

Key Methods Inherited from Object

Method Default Behavior Usually Override?
toString() Returns class name + @ + hashcode (e.g., Parcel@1a2b3c) Yes — to get readable output
equals(Object) Same as == (checks if same reference in memory) Yes — to compare by field values
hashCode() Returns a memory-based integer Yes — must override with equals()
getClass() Returns the runtime class of the object No — it is final
clone() Creates a shallow copy (must implement Cloneable) Sometimes
finalize() Called by garbage collector before destroying object No — deprecated since Java 9
Overriding toString() 
public class Parcel {
    private String trackingId;
    private String destination;

    @Override
    public String toString() {
        return "Parcel[id=" + trackingId + ", dest=" + destination + "]";
    }
}

// Without override: System.out.println(parcel) → "Parcel@1a2b3c"
// With override:    System.out.println(parcel) → "Parcel[id=NMB-1001, dest=Mumbai]"
In Nimbus

Every Nimbus class — TrackingService, CustomerTrackingService, BookingThread, DuplicateEmailException — implicitly extends Object. When CustomerTrackingService extends TrackingService, the chain is: Object → TrackingService → CustomerTrackingService. Object is always at the top.

Section 09

Composition vs Inheritance

Not every relationship should be modeled with extends. Sometimes a class should contain another class rather than be another class. This is the "has-a" vs "is-a" distinction.

"is-a" vs "has-a"

Inheritance (is-a)

CustomerTrackingService IS-A TrackingService

The child is a specialized version of the parent. They share an identity.

class CustomerTrackingService
    extends TrackingService {
    // inherits all tracking behavior
}
Composition (has-a)

UserService HAS-A UserDAO

The outer class uses the inner class as a tool. They are separate things.

class UserService {
    private UserDAO userDao; // has-a
    // uses userDao to do work
}

When to Use Which

Use Inheritance When Use Composition When
There is a true is-a relationship There is a has-a or "uses-a" relationship
Child needs to be used wherever parent is expected (polymorphism) You want to reuse behavior without tight coupling
You want to override specific behaviors The relationship could change at runtime
BookingThread extends Thread BookingService has a ParcelDAO
The Golden Rule

"Favor composition over inheritance." This is one of the most quoted rules in software design. Inheritance creates tight coupling — changing the parent can break all children. Composition is more flexible: you can swap out the composed object without touching the outer class. Use inheritance only when there is a genuine is-a relationship AND you need polymorphism.

In Nimbus

Nimbus uses both correctly:
Inheritance: CustomerTrackingService extends TrackingService (is-a, polymorphic use)
Composition: BookingService has a ParcelDAO field (uses-a, not a type relationship). A BookingService is NOT a ParcelDAO — it uses one to access the database.

Section 11

TrackingService → CustomerTrackingService

A full implementation showing how the customer tracking service inherits from the base tracking service and adds its own behavior.

TrackingService.java 
package com.nimbus.service;

public class TrackingService {
    protected String serviceName;
    protected String serviceRegion;

    public TrackingService(String serviceName, String serviceRegion) {
        this.serviceName = serviceName;
        this.serviceRegion = serviceRegion;
    }

    public String trackParcel(String trackingId) {
        return "Tracking " + trackingId + " via " + serviceName;
    }

    public String getServiceInfo() {
        return serviceName + " [" + serviceRegion + "]";
    }

    @Override
    public String toString() {
        return "TrackingService{name='" + serviceName + "', region='" + serviceRegion + "'}";
    }
}
CustomerTrackingService.java 
package com.nimbus.service;

public class CustomerTrackingService extends TrackingService {
    private String customerId;

    public CustomerTrackingService(String serviceRegion, String customerId) {
        super("Customer Tracking", serviceRegion);  // calls parent constructor
        this.customerId = customerId;
    }

    // Inherited: trackParcel(), getServiceInfo(), toString()

    // New method: only customers need delivery estimates
    public String getDeliveryEstimate(String trackingId) {
        return "Parcel " + trackingId + " | Estimated delivery: 2-3 business days";
    }

    // Override: customize tracking output for customers
    @Override
    public String trackParcel(String trackingId) {
        String base = super.trackParcel(trackingId);  // reuse parent logic
        return base + " | Customer: " + customerId;
    }

    @Override
    public String toString() {
        return "CustomerTrackingService{customer='" + customerId
            + "', region='" + serviceRegion + "'}";
    }
}
CustomerTrackingService breakdown
1
extends TrackingService
Inherits all protected/public fields (serviceName, serviceRegion) and methods (trackParcel, getServiceInfo, toString).
2
private String customerId;
New field that only the child has. The parent TrackingService has no concept of a customer ID.
3
super("Customer Tracking", serviceRegion);
Calls the parent's constructor to initialize serviceName and serviceRegion. Must be the first line. Parent has no default constructor, so this is mandatory.
4
getDeliveryEstimate()
A brand new method. Only CustomerTrackingService has it. The parent cannot see or call this method.
5
super.trackParcel(trackingId)
Overrides trackParcel() but reuses the parent's implementation via super. This is the "extend, don't replace" pattern: keep what the parent does and add more.
Section 12

BaseHistoryService → CustomerHistoryService

Another real Nimbus inheritance example: a base history service that provides generic history retrieval, and a customer-specific child that adds customer context.

BaseHistoryService.java 
package com.nimbus.service;

import java.util.List;
import java.util.ArrayList;

public class BaseHistoryService {
    protected List<String> historyLog;

    public BaseHistoryService() {
        this.historyLog = new ArrayList<>();
    }

    public void addEntry(String entry) {
        historyLog.add(entry);
    }

    public List<String> getFullHistory() {
        return new ArrayList<>(historyLog);  // return a copy, not the original
    }

    public int getHistoryCount() {
        return historyLog.size();
    }
}
CustomerHistoryService.java 
package com.nimbus.service;

import java.util.List;
import java.util.stream.Collectors;

public class CustomerHistoryService extends BaseHistoryService {
    private String customerId;

    public CustomerHistoryService(String customerId) {
        super();  // calls BaseHistoryService() — initializes historyLog
        this.customerId = customerId;
    }

    // New: add entry with customer prefix
    public void addCustomerEntry(String action) {
        String entry = "[Customer:" + customerId + "] " + action;
        super.addEntry(entry);  // reuse parent's addEntry
    }

    // New: filter history for bookings only
    public List<String> getBookingHistory() {
        return historyLog.stream()
            .filter(e -> e.contains("BOOKED") || e.contains("CANCELLED"))
            .collect(Collectors.toList());
    }

    // Override: add customer context to full history
    @Override
    public List<String> getFullHistory() {
        List<String> history = super.getFullHistory();
        history.add(0, "=== History for Customer " + customerId + " ===");
        return history;
    }
}
CustomerHistoryService breakdown
1
extends BaseHistoryService
Inherits historyLog field, addEntry(), getFullHistory(), getHistoryCount().
2
super()
Calls the parent's no-arg constructor, which creates the empty ArrayList. Even though the parent has a default constructor, making the call explicit is clearer.
3
addCustomerEntry()
New method that prefixes entries with customer ID, then delegates to super.addEntry(). This is composition of behavior: child builds on parent's method.
4
getBookingHistory()
New method. Uses Java streams to filter the inherited historyLog. Only available on CustomerHistoryService, not the parent.
5
@Override getFullHistory()
Overrides parent method. Calls super.getFullHistory() to get the base list, then inserts a header line at position 0. Extends parent behavior rather than replacing it.
Section 13

BookingThread extends Thread

When Nimbus needs to process a booking without blocking the main thread, it creates a BookingThread. This is inheritance from a Java standard library class.

BookingThread.java 
package com.nimbus.thread;

public class BookingThread extends Thread {
    private String bookingId;
    private String customerName;

    public BookingThread(String bookingId, String customerName) {
        super("BookingThread-" + bookingId);  // sets the thread name
        this.bookingId = bookingId;
        this.customerName = customerName;
    }

    @Override
    public void run() {
        // This method is inherited from Thread and overridden
        System.out.println("Processing booking " + bookingId + " for " + customerName);

        try {
            // Simulate booking steps
            System.out.println("Step 1: Validating address...");
            Thread.sleep(1000);

            System.out.println("Step 2: Calculating shipping cost...");
            Thread.sleep(500);

            System.out.println("Step 3: Booking confirmed!");
        } catch (InterruptedException e) {
            System.out.println("Booking " + bookingId + " was interrupted!");
        }
    }
}

// Usage:
// BookingThread bt = new BookingThread("NMB-1001", "Darshan");
// bt.start();  // start() is inherited from Thread — it calls run() in a new thread
BookingThread breakdown
1
extends Thread
Inherits from java.lang.Thread. BookingThread IS-A Thread. It gets start(), sleep(), interrupt(), getName(), and dozens of other thread methods.
2
super("BookingThread-" + bookingId)
Thread has a constructor that accepts a name. By calling super with a descriptive name, this thread shows up in debuggers as "BookingThread-NMB-1001" instead of "Thread-0".
3
@Override public void run()
Thread.run() does nothing by default. We override it to define what this thread actually does when started. This is the core pattern of extending Thread.
4
bt.start()
start() is inherited from Thread. You do NOT call run() directly. start() creates a new OS-level thread and calls run() inside it. Calling run() directly would execute on the same thread (no parallelism).
In Nimbus

BookingThread lets Nimbus process multiple bookings simultaneously. When a customer submits a booking, a new BookingThread is created and started. The main application thread can continue accepting new requests while the booking is processed in the background. The custom exceptions (DuplicateEmailException, BookingNotFoundException) follow the same pattern — they extend Exception to create domain-specific error types.

Section 14

Common Mistakes

These are the mistakes beginners make with inheritance. Learn them now so you can spot them in your own code.

Mistake 1: Confusing Overriding with Overloading

Overriding (same signature)
class Parent {
    void display(String msg) { ... }
}
class Child extends Parent {
    @Override
    void display(String msg) { ... }
    // SAME name, SAME params
    // Replaces parent's version
}
Overloading (different params)
class Parent {
    void display(String msg) { ... }
}
class Child extends Parent {
    void display(String msg, int count) { ... }
    // SAME name, DIFFERENT params
    // This is a NEW method!
}

The trap: You think you overrode display() but you actually created a new overloaded method. The parent's display(String) still runs when called polymorphically. Always use @Override to catch this.

Mistake 2: Forgetting super()

Missing super() call 
public class TrackingService {
    // Only parameterized constructor — no default constructor
    public TrackingService(String name) {
        this.serviceName = name;
    }
}

public class CustomerTrackingService extends TrackingService {
    public CustomerTrackingService() {
        // COMPILER ERROR: There is no default constructor in TrackingService
        // The compiler tries to insert super() automatically,
        // but TrackingService has no no-arg constructor!

        // FIX: super("Customer Tracking");
    }
}

Mistake 3: Trying Circular Inheritance

Circular inheritance attempt 
// COMPILER ERROR: Cyclic inheritance
class A extends B { }
class B extends A { }

// A needs B to exist first, B needs A to exist first.
// This is an impossible loop. Java catches it immediately.

Mistake 4: Calling run() Instead of start()

Thread mistake 
BookingThread bt = new BookingThread("NMB-1001", "Darshan");

bt.run();    // WRONG: runs on the current thread, no parallelism
bt.start();  // RIGHT: creates a new thread, then calls run() inside it

Mistake 5: Overriding a final or static Method

Final method restriction 
class Parent {
    public final void validate() { ... }
}

class Child extends Parent {
    @Override
    public void validate() { ... }  // COMPILER ERROR: Cannot override final method
}
