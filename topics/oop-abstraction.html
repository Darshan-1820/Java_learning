Section 01

What is Abstraction?

Abstraction is the OOP principle of hiding internal complexity and exposing only what the user needs. You define what something does, not how it does it.

Analogy — Driving a Car

When you drive a car, you interact with the steering wheel, accelerator, brake, and gear shift. You do not need to know how the engine combustion cycle works, how the transmission converts RPM to wheel torque, or how the ABS controller pulses the brake calipers 15 times per second. All of that complexity is abstracted away. You get a simple interface: turn the wheel, press the pedal, the car moves.

Software abstraction works the same way. You expose simple methods like trackParcel(id). The caller does not care if you query a database, call a REST API, or read from a cache internally. They just want the result.

Why Abstraction Matters

Without abstraction, every class that uses your tracking system would need to know about database connections, SQL queries, result set parsing, error handling, caching logic, and retry mechanisms. Change any internal detail and every caller breaks.

With abstraction, callers depend on a contract (the method signature), not the implementation. You can completely rewrite the internals — switch from MySQL to PostgreSQL, add Redis caching, change the algorithm — and no caller needs to change a single line of code.

In Nimbus

When a customer calls trackParcel("NB-20260423-0001"), they do not care whether the system queries a local database, calls the courier partner's API, or checks a Redis cache. They just want the parcel status. The tracking service abstracts all of that complexity behind a single method call.

Two Tools for Abstraction in Java

Java gives you two mechanisms to achieve abstraction:

Mechanism Abstraction Level Use When
Abstract Class Partial (0–100%) Related classes share common code + some methods must be customized
Interface Full (100% before Java 8) Unrelated classes need to follow the same contract

We will cover both in depth. By the end of this page you will know exactly when to use each one and why.

           Abstraction in Java
          ________|________
         |                 |
   Abstract Class       Interface
         |                 |
  - Partial contract   - Full contract
  - Has constructors   - No constructors
  - Has fields         - Only constants
  - Single inherit     - Multiple inherit
  - "is-a" base        - "can-do" capability
Two paths to abstraction — different tools for different problems
Section 02

Abstract Classes

An abstract class is a class declared with the abstract keyword. It cannot be instantiated directly — it exists only to be extended by subclasses.

The Rules

  • Declared with the abstract keyword before class
  • Cannot create objects with new AbstractClass() — compile error
  • Can have abstract methods (no body) and concrete methods (with body)
  • Can have instance fields, constructors, and static methods
  • A subclass must override all abstract methods, or it must also be declared abstract
  • Can have any access modifier on methods (public, protected, private)

Nimbus Example: Abstract TrackingService

In Nimbus, tracking a parcel involves common steps (validate booking ID, check if parcel exists, log the request) and role-specific steps (what data to display). The common logic belongs in the abstract class. The role-specific logic is forced onto subclasses.

TrackingService.java 
public abstract class TrackingService {

    // Instance field - shared by all subclasses
    protected BookingDAO bookingDAO;

    // Constructor - subclasses call this via super()
    public TrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    // CONCRETE method - shared logic, same for everyone
    public void trackParcel(String bookingId) {
        // Step 1: Validate (common)
        if (bookingId == null || bookingId.isEmpty()) {
            System.out.println("Error: Booking ID cannot be empty.");
            return;
        }

        // Step 2: Fetch parcel (common)
        Parcel parcel = bookingDAO.getParcelByBookingId(bookingId);
        if (parcel == null) {
            System.out.println("No parcel found for: " + bookingId);
            return;
        }

        // Step 3: Display result (DIFFERENT per role)
        displayResult(parcel);
    }

    // ABSTRACT method - each subclass MUST implement this
    public abstract void displayResult(Parcel parcel);

    // Another concrete method - shared utility
    protected String formatDate(long timestamp) {
        return new java.text.SimpleDateFormat("dd-MMM-yyyy HH:mm")
                   .format(new java.util.Date(timestamp));
    }
}
Line-by-line explanation
1
public abstract class TrackingService {
The abstract keyword means: this class cannot be instantiated directly. You cannot write new TrackingService(). It exists to be a parent.
2
protected BookingDAO bookingDAO;
Abstract classes CAN have fields. protected means subclasses can access this directly. Every subclass inherits this DAO reference.
3
public TrackingService(BookingDAO bookingDAO)
Abstract classes CAN have constructors. They run when a subclass calls super(bookingDAO). This ensures every subclass has a properly initialized DAO.
4
public void trackParcel(String bookingId)
A concrete method (has a body). All subclasses inherit this exact logic. Validation and fetching are the same regardless of user role.
5
public abstract void displayResult(Parcel parcel);
An abstract method — no body, just a semicolon. Every subclass MUST provide its own implementation. This is the "contract" that forces customization.
6
protected String formatDate(long timestamp)
Another concrete method. Shared utility that all subclasses can use without rewriting. protected means only subclasses and same-package classes can call it.

Subclasses Implement the Abstract Method

CustomerTrackingService.java 
public class CustomerTrackingService extends TrackingService {

    public CustomerTrackingService(BookingDAO bookingDAO) {
        super(bookingDAO);  // Calls abstract class constructor
    }

    @Override
    public void displayResult(Parcel parcel) {
        // Customer sees: status + ETA only
        System.out.println("=== Your Parcel Status ===");
        System.out.println("Booking ID : " + parcel.getBookingId());
        System.out.println("Status     : " + parcel.getStatus());
        System.out.println("ETA        : " + formatDate(parcel.getEta()));
    }
}
OfficerTrackingService.java 
public class OfficerTrackingService extends TrackingService {

    public OfficerTrackingService(BookingDAO bookingDAO) {
        super(bookingDAO);
    }

    @Override
    public void displayResult(Parcel parcel) {
        // Officer sees: everything including route + driver
        System.out.println("=== Officer Tracking View ===");
        System.out.println("Booking ID : " + parcel.getBookingId());
        System.out.println("Status     : " + parcel.getStatus());
        System.out.println("Origin     : " + parcel.getOriginCity());
        System.out.println("Destination: " + parcel.getDestCity());
        System.out.println("Route      : " + parcel.getRouteDetails());
        System.out.println("Driver     : " + parcel.getAssignedDriver());
        System.out.println("ETA        : " + formatDate(parcel.getEta()));
        System.out.println("Last Scan  : " + formatDate(parcel.getLastScanTime()));
    }
}
Key Insight

Notice the Template Method pattern at work. The abstract class defines the algorithm skeleton in trackParcel(): validate → fetch → display. Steps 1 and 2 are fixed. Step 3 is delegated to subclasses via the abstract method. The overall flow is locked; only the variable part is customizable.

What You Cannot Do

Compile Errors with Abstract Classes 
// ERROR 1: Cannot instantiate abstract class
TrackingService ts = new TrackingService(dao);
// Compiler: "TrackingService is abstract; cannot be instantiated"

// ERROR 2: Subclass that does NOT override abstract method
public class LazyTracker extends TrackingService {
    public LazyTracker(BookingDAO dao) { super(dao); }
    // Missing displayResult() override!
}
// Compiler: "LazyTracker must implement abstract method displayResult()"

// FIX: Either implement it OR declare LazyTracker as abstract too
public abstract class LazyTracker extends TrackingService {
    // OK - but now LazyTracker itself cannot be instantiated
}
Section 03

Abstract Methods

An abstract method is a method signature with no body. It declares what must be done but leaves how to the subclass.

Syntax

Abstract Method Syntax 
// Abstract method - note the semicolon, NO curly braces
public abstract void displayResult(Parcel parcel);

// Concrete method - has curly braces with a body
public void trackParcel(String bookingId) {
    // ... actual code here ...
}

Rules for Abstract Methods

Rule Why
Can only exist inside an abstract class or interface A non-abstract class must have all methods implemented
Cannot be private Subclasses need to see and override the method
Cannot be final final prevents overriding — contradicts the purpose
Cannot be static Static methods belong to the class, not instances — cannot be overridden polymorphically
Ends with a semicolon ; No body, so no curly braces — just the signature
First concrete subclass must implement it Otherwise that subclass must also be abstract

Chain of Abstraction

If an abstract class extends another abstract class, it does NOT have to implement the parent's abstract methods. The obligation passes down until a concrete class appears.

Abstract Chain Example 
// Level 1: defines the contract
public abstract class TrackingService {
    public abstract void displayResult(Parcel parcel);
    public abstract void logAccess(String userId);
}

// Level 2: implements ONE, leaves the other abstract
public abstract class RoleBasedTracker extends TrackingService {
    @Override
    public void logAccess(String userId) {
        System.out.println("Access logged for: " + userId);
    }
    // displayResult() is still abstract - passes to next level
}

// Level 3: MUST implement displayResult() - first concrete class
public class CustomerTracker extends RoleBasedTracker {
    @Override
    public void displayResult(Parcel parcel) {
        System.out.println("Status: " + parcel.getStatus());
    }
}
  TrackingService (abstract)
  ├── abstract displayResult()
  └── abstract logAccess()
         │
         ▼
  RoleBasedTracker (abstract)
  ├── abstract displayResult()   ← still abstract, passed down
  └── logAccess()                ← implemented here
         │
         ▼
  CustomerTracker (concrete)
  └── displayResult()            ← MUST implement here
Abstract methods flow down until a concrete class handles them
Section 04

Interfaces

An interface is a pure contract. It defines what a class can do without saying how. Before Java 8, every method in an interface was implicitly abstract. Think of it as a job description — it lists required skills, but each employee brings their own way of doing the work.

Basic Syntax

Trackable.java 
public interface Trackable {

    // Abstract method (public abstract by default)
    void trackParcel(String bookingId);

    // Another abstract method
    void displayResult(Parcel parcel);

    // Constant (public static final by default)
    String TRACKING_PREFIX = "NB-";
}
Line-by-line explanation
1
public interface Trackable {
Uses interface keyword instead of class. Naming convention: often an adjective ending in "-able" (Trackable, Serializable, Comparable) or a noun describing a role (Runnable, Iterator).
2
void trackParcel(String bookingId);
No abstract keyword needed — interface methods are abstract by default. No public keyword needed — interface methods are public by default. Just the signature.
3
String TRACKING_PREFIX = "NB-";
Interface fields are public static final by default — always constants. You cannot have instance variables in an interface. This constant is shared across all implementing classes.

Implementing an Interface

CustomerTrackingService.java 
// "implements" keyword - the class signs the contract
public class CustomerTrackingService implements Trackable {

    private BookingDAO bookingDAO;

    public CustomerTrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    @Override
    public void trackParcel(String bookingId) {
        Parcel parcel = bookingDAO.getParcelByBookingId(bookingId);
        if (parcel != null) {
            displayResult(parcel);
        } else {
            System.out.println("Parcel not found: " + bookingId);
        }
    }

    @Override
    public void displayResult(Parcel parcel) {
        System.out.println("Status: " + parcel.getStatus());
        System.out.println("ETA   : " + parcel.getEta());
    }
}

Interface Rules

Feature Allowed? Details
Instance fields No Only public static final constants
Constructors No Interfaces cannot be instantiated
Abstract methods Yes Implicitly public abstract
Default methods Yes (Java 8+) Concrete method with default keyword
Static methods Yes (Java 8+) Called on interface: Trackable.helper()
Private methods Yes (Java 9+) Helper for default methods, not visible to implementors
Multiple inheritance Yes A class can implements A, B, C
Section 05

Interface Methods — Java 8, 9, and Beyond

Interfaces evolved significantly starting with Java 8. What was once a pure contract can now include shared behavior through default, static, and private methods.

Abstract Methods (Original — Pre-Java 8)

The classic interface method. No body. Every implementing class must provide the implementation.

Abstract Method in Interface 
public interface Trackable {
    void trackParcel(String bookingId);  // must be implemented
}

Default Methods (Java 8+)

The default keyword lets you add a method with a body to an interface. Implementing classes inherit this method automatically. They can override it if they want, but they do not have to.

Why this was added: Before Java 8, adding a new method to an interface would break every class that implements it. Imagine the List interface in Java — thousands of classes implement it. When Java wanted to add forEach(), they could not make it abstract (every existing class would break). So they added it as a default method with a provided body.

Default Method Example 
public interface Trackable {

    // Abstract - must implement
    void trackParcel(String bookingId);

    // Default - already has a body, optional to override
    default void printTrackingHeader() {
        System.out.println("============================");
        System.out.println("   NIMBUS PARCEL TRACKING   ");
        System.out.println("============================");
    }

    // Default with logic
    default boolean isValidBookingId(String id) {
        return id != null
            && id.startsWith("NB-")
            && id.length() == 18;
    }
}

// Implementing class - gets printTrackingHeader() for free
public class CustomerTracker implements Trackable {

    @Override
    public void trackParcel(String bookingId) {
        printTrackingHeader();  // Inherited from interface!
        System.out.println("Tracking: " + bookingId);
    }
    // No need to implement printTrackingHeader or isValidBookingId
}

Static Methods (Java 8+)

Static methods in interfaces work like static methods in classes. They belong to the interface itself, not to implementing classes. Called as InterfaceName.methodName().

Static Method in Interface 
public interface Trackable {

    void trackParcel(String bookingId);

    // Static utility method
    static String generateBookingId(String city) {
        String date = java.time.LocalDate.now()
                          .format(java.time.format.DateTimeFormatter.BASIC_ISO_DATE);
        int random = (int)(Math.random() * 10000);
        return "NB-" + date + "-" + String.format("%04d", random);
    }
}

// Usage - called on the INTERFACE, not on an object
String id = Trackable.generateBookingId("Mumbai");
// "NB-20260423-0583"

Private Methods (Java 9+)

When multiple default methods share common logic, you extract it into a private method. These are not visible to implementing classes — purely internal to the interface.

Private Method in Interface (Java 9+) 
public interface Trackable {

    void trackParcel(String bookingId);

    default void printSuccess(String bookingId) {
        printMessage("Parcel " + bookingId + " found successfully.");
    }

    default void printError(String bookingId) {
        printMessage("ERROR: Parcel " + bookingId + " not found.");
    }

    // Private helper - shared by both default methods above
    // NOT visible to implementing classes
    private void printMessage(String msg) {
        System.out.println("[NIMBUS] " + msg);
    }
}

Evolution Summary

Pre-Java 8 Abstract only
→
Java 8 + default + static
→
Java 9+ + private methods
Interfaces went from pure contracts to contracts with shared behavior

Abstract Class vs Interface — The Big Comparison

This is the most-asked interview question in Java OOP. Know every difference cold.

The 15+ Differences Table

Feature Abstract Class Interface
Keyword abstract class interface
Method types Abstract + concrete (any mix) Abstract + default + static + private
Constructors Yes No
Instance fields Yes (any type, any modifier) No (only public static final constants)
Access modifiers on methods Any (public, protected, private) public only (abstract/default), private (Java 9+)
Multiple inheritance No (single extends) Yes (multiple implements)
Can extend One class (abstract or concrete) Multiple interfaces
Instantiation No No
State (mutable data) Yes (instance fields change) No (constants only)
final methods Yes No
static blocks Yes No
Speed Slightly faster (direct method dispatch) Slightly slower (virtual table lookup)
Relationship "is-a" (Dog is-a Animal) "can-do" (Dog can-do Trainable)
Use case Shared base for closely related classes Common capability for unrelated classes
Evolution Add concrete methods freely Add default methods (Java 8+) without breaking implementors
Example abstract class TrackingService interface Trackable

Decision Flowchart

  Do the classes share common CODE (not just a contract)?
      │
      ├── YES ──> Does the shared code need instance fields?
      │               │
      │               ├── YES ──> Use ABSTRACT CLASS
      │               │
      │               └── NO ──> Could different class hierarchies
      │                           need this capability?
      │                               │
      │                               ├── YES ──> Use INTERFACE
      │                               │           (with default methods)
      │                               │
      │                               └── NO ──> Use ABSTRACT CLASS
      │
      └── NO ──> Do the classes need to "promise" they can do something?
                      │
                      ├── YES ──> Use INTERFACE
                      │
                      └── NO ──> You might not need abstraction at all
When in doubt, prefer interface. You can always add an abstract class later.

Side-by-Side Comparison

Abstract Class

When to use:

  • Classes are closely related (all are "tracking services")
  • You need shared state (fields like bookingDAO)
  • You want to provide a constructor that initializes shared state
  • You need protected methods (only for subclasses)
  • You want to enforce a template (Template Method pattern)

Nimbus: abstract class TrackingService — all tracker types share the same DAO, same validation, same flow.

Interface

When to use:

  • Unrelated classes need the same capability
  • A class needs to "wear multiple hats" (implement several interfaces)
  • You want to define a contract without forcing an inheritance hierarchy
  • You are designing an API that others will implement
  • You need multiple inheritance of type

Nimbus: interface Trackable — both CustomerTrackingService and a future ThirdPartyTracker can be Trackable, even if they have no common parent class.

Common Mistake

Do not use an abstract class just because you want to add one shared method. If the classes are unrelated, use an interface with a default method instead. Abstract class = "these classes are fundamentally the same thing." Interface = "these classes can do the same thing."

Section 07

Multiple Interface Implementation

A class in Java can extend only one class, but it can implement multiple interfaces. This is Java's answer to multiple inheritance.

One Class, Multiple Interfaces

Multiple Interfaces in Nimbus 
public interface Trackable {
    void trackParcel(String bookingId);
    void displayResult(Parcel parcel);
}

public interface Bookable {
    Booking createBooking(BookingRequest request);
}

public interface HistoryViewable {
    List<Parcel> getHistory(String customerId);
    void displayHistory(List<Parcel> parcels);
}

// One class implements ALL three interfaces
public class CustomerService implements Trackable, Bookable, HistoryViewable {

    @Override
    public void trackParcel(String bookingId) {
        System.out.println("Tracking: " + bookingId);
    }

    @Override
    public void displayResult(Parcel parcel) {
        System.out.println("Status: " + parcel.getStatus());
    }

    @Override
    public Booking createBooking(BookingRequest request) {
        System.out.println("Booking created for: " + request.getSenderName());
        return new Booking(request);
    }

    @Override
    public List<Parcel> getHistory(String customerId) {
        return bookingDAO.getParcelsByCustomer(customerId);
    }

    @Override
    public void displayHistory(List<Parcel> parcels) {
        for (Parcel p : parcels) {
            System.out.println(p.getBookingId() + " - " + p.getStatus());
        }
    }
}
In Nimbus

CustomerService wears three hats: it can track parcels (Trackable), create bookings (Bookable), and show booking history (HistoryViewable). With interfaces, you can pass a CustomerService object anywhere a Trackable is expected, anywhere a Bookable is expected, or anywhere a HistoryViewable is expected. One object, multiple types.

The Diamond Problem with Default Methods

What happens when two interfaces have a default method with the same signature?

Diamond Problem 
public interface Trackable {
    default void printHeader() {
        System.out.println("=== TRACKING ===");
    }
}

public interface Bookable {
    default void printHeader() {
        System.out.println("=== BOOKING ===");
    }
}

// COMPILE ERROR: class inherits unrelated defaults for printHeader()
public class CustomerService implements Trackable, Bookable {
    // Must resolve the conflict!
}

// SOLUTION: Override and choose (or combine)
public class CustomerService implements Trackable, Bookable {

    @Override
    public void printHeader() {
        // Option 1: Pick one
        Trackable.super.printHeader();

        // Option 2: Combine both
        // Trackable.super.printHeader();
        // Bookable.super.printHeader();

        // Option 3: Write your own
        // System.out.println("=== CUSTOMER SERVICE ===");
    }
}
Resolving the diamond problem
1
Trackable.super.printHeader();
InterfaceName.super.method() lets you explicitly call a specific interface's default method. This is the resolution syntax Java requires when two interfaces collide.
2
The class MUST override the conflicting method
Java will not guess which default to use. If you do not override, you get a compile error. The implementing class is always in control.
       Trackable          Bookable
    printHeader()       printHeader()
          \                 /
           \               /
            ▼             ▼
         CustomerService
      MUST override printHeader()
      to resolve the conflict
Diamond problem: two interfaces, same default method, one implementing class
Section 08

Functional Interfaces

A functional interface has exactly one abstract method. It is the foundation of lambda expressions in Java 8+.

Definition and Rules

  • Exactly one abstract method (called SAM — Single Abstract Method)
  • Can have any number of default and static methods
  • Annotated with @FunctionalInterface (optional but recommended — compiler enforces the rule)
  • Can be used as the target for a lambda expression or method reference
Custom Functional Interface for Nimbus 
@FunctionalInterface
public interface ParcelFilter {
    boolean test(Parcel parcel);  // The ONE abstract method

    // Default methods are fine - don't count
    default ParcelFilter and(ParcelFilter other) {
        return (parcel) -> this.test(parcel) && other.test(parcel);
    }
}

// Usage with lambda expression (preview - covered in depth later)
ParcelFilter inTransit = parcel -> parcel.getStatus().equals("In Transit");
ParcelFilter toMumbai  = parcel -> parcel.getDestCity().equals("Mumbai");

// Combine filters using the default method
ParcelFilter inTransitToMumbai = inTransit.and(toMumbai);

// Apply to a list
for (Parcel p : allParcels) {
    if (inTransitToMumbai.test(p)) {
        System.out.println(p.getBookingId());
    }
}

Built-in Functional Interfaces You Will See

Interface Abstract Method Purpose Example
Comparable<T> compareTo(T o) Natural ordering of objects Sort parcels by date
Runnable run() Task to execute in a thread Background parcel status check
Comparator<T> compare(T a, T b) Custom ordering Sort parcels by weight, by city
Predicate<T> test(T t) Test a condition Filter delivered parcels
Function<T,R> apply(T t) Transform one type to another Convert Parcel to ParcelDTO
Consumer<T> accept(T t) Perform action, return nothing Print parcel details
Supplier<T> get() Provide a value, take nothing Generate booking ID
Lambda Preview

Functional interfaces are the type that lambda expressions target. Instead of creating an anonymous class with one overridden method, you write a compact arrow expression: parcel -> parcel.getStatus().equals("Delivered"). Lambdas are covered fully in a later topic, but functional interfaces are the prerequisite.

Comparable in Nimbus - Sorting Parcels 
public class Parcel implements Comparable<Parcel> {

    private String bookingId;
    private double weight;
    private long bookingDate;

    // Natural ordering: by booking date (most recent first)
    @Override
    public int compareTo(Parcel other) {
        return Long.compare(other.bookingDate, this.bookingDate);
    }
}

// Now you can sort a list of parcels
List<Parcel> parcels = bookingDAO.getAllParcels();
Collections.sort(parcels);  // Uses compareTo automatically
Section 09

Marker Interfaces

A marker interface is an empty interface with zero methods. It "marks" or "tags" a class as having a certain property.

Why Empty Interfaces Exist

Sometimes you do not need the class to do anything new. You just need to tell the JVM or a framework: "this class is special in this way." The interface acts as a flag.

Built-in Marker Interfaces 
// java.io.Serializable - marks a class as safe to convert to bytes
public interface Serializable {
    // EMPTY - no methods
}

// java.lang.Cloneable - marks a class as safe to clone
public interface Cloneable {
    // EMPTY - no methods
}

// Usage: Nimbus Parcel can be serialized (saved to file/sent over network)
public class Parcel implements Serializable {
    private String bookingId;
    private String status;
    private double weight;
    // ... all fields automatically become serializable
}

// Now you can do this:
ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("parcel.dat"));
out.writeObject(parcel);  // Works because Parcel implements Serializable

// Without Serializable:
// out.writeObject(parcel);  // Throws NotSerializableException at runtime

Custom Marker Interface for Nimbus

Custom Marker Interface 
// Marker: classes that can be audited (logged for compliance)
public interface Auditable {
    // Empty - just a tag
}

// Mark specific classes
public class Booking implements Auditable {
    // ... booking fields and methods
}

public class Payment implements Auditable {
    // ... payment fields and methods
}

public class TempCalculation {
    // NOT auditable - internal temporary data
}

// Audit system checks the marker
public class AuditLogger {
    public void log(Object entity) {
        if (entity instanceof Auditable) {
            System.out.println("AUDIT: " + entity.getClass().getSimpleName()
                             + " at " + System.currentTimeMillis());
        } else {
            System.out.println("Skipped: " + entity.getClass().getSimpleName()
                             + " is not auditable");
        }
    }
}
Modern Alternative

In modern Java, annotations (@Auditable) are often preferred over marker interfaces because they do not affect the type hierarchy. But marker interfaces still have an advantage: they can be used as a type. You can write void process(Auditable entity) — the compiler enforces that only Auditable objects are passed. Annotations cannot do that.

Section 10

Nimbus Redesign with Interfaces

Let us redesign key parts of Nimbus using interfaces to see how abstraction makes the system flexible, testable, and extensible.

Three Core Interfaces

Trackable.java 
public interface Trackable {

    /**
     * Tracks a parcel by its booking ID.
     * @param bookingId the unique booking identifier (e.g., "NB-20260423-0001")
     */
    void trackParcel(String bookingId);

    /**
     * Displays the tracking result to the user.
     * Each role shows different information.
     */
    void displayResult(Parcel parcel);

    /**
     * Validates a booking ID format.
     * Default implementation - can be overridden.
     */
    default boolean isValidId(String bookingId) {
        return bookingId != null
            && bookingId.startsWith("NB-")
            && bookingId.length() >= 10;
    }
}

public interface Bookable {

    /**

     */

                         String originCity, String destCity,
                         double weight);

    /**
     * Calculates the cost of a booking.
     */
    double calculateCost(double weight, String originCity, String destCity);
}
HistoryViewable.java 
public interface HistoryViewable {

    /**
     * Retrieves parcel history for a customer.
     */
    List<Parcel> getHistory(String customerId);

    /**
     * Displays formatted history to the console.
     */
    void displayHistory(List<Parcel> parcels);

    /**
     * Default: display count summary
     */
    default void displaySummary(List<Parcel> parcels) {
        long delivered = parcels.stream()
            .filter(p -> "Delivered".equals(p.getStatus()))
            .count();
        System.out.println("Total: " + parcels.size()
                         + " | Delivered: " + delivered
                         + " | In Transit: " + (parcels.size() - delivered));
    }
}

Full Implementation — Combining Abstract Class + Interfaces

CustomerService.java — The Full Picture 
// Abstract class provides shared infrastructure
// Interfaces provide capabilities
public class CustomerService extends TrackingService
                             implements Bookable, HistoryViewable {

    private BookingDAO bookingDAO;

    public CustomerService(BookingDAO bookingDAO) {
        super(bookingDAO);
        this.bookingDAO = bookingDAO;
    }

    // From TrackingService (abstract class) - role-specific display
    @Override
    public void displayResult(Parcel parcel) {
        System.out.println("=== Your Parcel ===");
        System.out.println("ID     : " + parcel.getBookingId());
        System.out.println("Status : " + parcel.getStatus());
        System.out.println("ETA    : " + formatDate(parcel.getEta()));
    }

    // From Bookable interface
    @Override
    public String createBooking(String senderName, String receiverName,
                                String originCity, String destCity,
                                double weight) {
        double cost = calculateCost(weight, originCity, destCity);
        Parcel parcel = new Parcel(senderName, receiverName,
                                   originCity, destCity, weight, cost);
        bookingDAO.save(parcel);
        System.out.println("Booking created! Cost: Rs. " + cost);
        return parcel.getBookingId();
    }

    @Override
    public double calculateCost(double weight, String origin, String dest) {
        double baseCost = 50.0;
        double perKg = 15.0;
        double distanceFactor = origin.equals(dest) ? 1.0 : 2.5;
        return (baseCost + (weight * perKg)) * distanceFactor;
    }

    // From HistoryViewable interface
    @Override
    public List<Parcel> getHistory(String customerId) {
        return bookingDAO.getParcelsByCustomer(customerId);
    }

    @Override
    public void displayHistory(List<Parcel> parcels) {
        System.out.println("=== Your Booking History ===");
        displaySummary(parcels);  // Uses default method from interface!
        System.out.println("---");
        for (Parcel p : parcels) {
            System.out.println(p.getBookingId() + " | "
                + p.getStatus() + " | " + p.getDestCity());
        }
    }
}
Architecture breakdown
1
extends TrackingService implements Bookable, HistoryViewable
extends one abstract class (shared state + template). implements multiple interfaces (capabilities). This is the standard Java pattern: single class inheritance + multiple interface implementation.
2
displayResult() from TrackingService
The abstract method from TrackingService. This class fulfills the contract by showing customer-appropriate data (status + ETA only, no internal route details).
3
createBooking() + calculateCost() from Bookable
Both methods from the Bookable interface. This class can now be used anywhere a Bookable is expected.
4
displaySummary(parcels) - default method
Called inside displayHistory() but never explicitly implemented. It comes from the HistoryViewable interface's default method. Free shared logic.

Section 11

Interface Segregation Principle

No class should be forced to implement methods it does not use. Keep interfaces small and focused.

The Problem: Fat Interfaces

BAD — One Fat Interface
interface ParcelService {
  void trackParcel(String id);
  void displayResult(Parcel p);
  String createBooking(...);
  double calculateCost(...);
  List<Parcel> getHistory(String id);
  void displayHistory(List<Parcel> p);
  void updateStatus(String id, String s);
  void assignDriver(String id, String d);
  void generateReport();
}

Problem: A CustomerService must implement assignDriver() and generateReport() even though customers never do those things. Empty methods or UnsupportedOperationException everywhere.

GOOD — Small Focused Interfaces
interface Trackable {
  void trackParcel(String id);
  void displayResult(Parcel p);
}

interface Bookable {
  String createBooking(...);
  double calculateCost(...);
}

interface HistoryViewable {
  List<Parcel> getHistory(String id);
  void displayHistory(List<Parcel> p);
}

interface Assignable {
  void assignDriver(String id, String d);
}

Result: Each class implements only what it needs. CustomerService implements Trackable, Bookable, HistoryViewable. OfficerService implements Trackable, Assignable.

SOLID Connection

Interface Segregation is the "I" in SOLID. The five SOLID principles are: Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion. You covered S, O, and L in the Software Engineering topic. Here is I. D comes with dependency injection patterns.

How to Know If Your Interface Is Too Big

  • Any implementing class has methods that throw UnsupportedOperationException
  • Any implementing class has empty method bodies { }
  • The interface has methods that only some implementors need
  • You cannot describe the interface's purpose in one sentence
  • The interface has more than 5-6 methods (not a hard rule, but a smell)
In Nimbus

We split the original monolithic ParcelService into four focused interfaces: Trackable, Bookable, HistoryViewable, and Assignable. Now CustomerService only implements Trackable + Bookable + HistoryViewable. OfficerService implements Trackable + Assignable. Neither class is forced to implement methods it does not use. Clean. Focused. SOLID.

Trackable.java — The Interface Contract 
public interface Trackable {
    void trackParcel(String bookingId);
    void displayResult(Parcel parcel);

    default boolean isValidId(String id) {
        return id != null && id.startsWith("NB-") && id.length() >= 10;
    }
}
TrackingService.java — Abstract Base Class 
public abstract class TrackingService implements Trackable {

    protected BookingDAO bookingDAO;

    public TrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    // Template method - defines the algorithm skeleton
    @Override
    public void trackParcel(String bookingId) {
        if (!isValidId(bookingId)) {
            System.out.println("Invalid booking ID format.");
            return;
        }

        Parcel parcel = bookingDAO.getParcelByBookingId(bookingId);
        if (parcel == null) {
            System.out.println("No parcel found for: " + bookingId);
            return;
        }

        displayResult(parcel);  // Abstract - subclass decides
    }

    // displayResult is still abstract - from Trackable interface
    // Subclasses MUST implement it

    protected String formatDate(long timestamp) {
        return new java.text.SimpleDateFormat("dd-MMM-yyyy HH:mm")
                   .format(new java.util.Date(timestamp));
    }
}

Exercise 1 — Basic Abstract Class

Create an abstract class NotificationService with:

  • A protected String recipientId field
  • A constructor that takes recipientId
  • An abstract method void send(String message)
  • A concrete method String formatMessage(String msg) that prepends "[NIMBUS] "

Exercise 2 — Interface Implementation

Create an interface Cancellable with:

  • Abstract method: boolean cancelBooking(String bookingId)
  • Abstract method: double calculateRefund(String bookingId)
  • Default method: void printCancellationReceipt(String bookingId, double refund) that prints a formatted receipt

Implement this interface in a CustomerCancellationService class. The refund should be 80% of the booking cost if cancelled within 24 hours, 50% otherwise.

Exercise 3 — Multiple Interfaces

Create three interfaces: Printable (with void printReceipt()), Emailable (with void emailReceipt(String email)), and Exportable (with String exportToCSV()). Exercise 4 — Abstract + Interface Combo

Exercise 5 — Diamond Problem

Exercise 6 — Functional Interface

Exercise 7 — Marker Interface

Create a marker interface PriorityHandling. Create two parcel classes: RegularParcel and VIPParcel implements PriorityHandling. Write a DispatchService with a method void dispatch(Object parcel) that checks instanceof PriorityHandling and prints "PRIORITY QUEUE" or "STANDARD QUEUE" accordingly.
