Section 01

What Are Threads?

A thread is the smallest unit of execution inside a program. Every Java program starts with one thread — the main thread. Multithreading lets you run multiple tasks at the same time within a single program.

Single-Threaded vs Multi-Threaded

A single-threaded program does one thing at a time. It finishes Task A, then starts Task B, then Task C. If Task A takes 5 seconds, everything else waits. The program is sequential — one line of execution from top to bottom.

A multi-threaded program can do Task A, Task B, and Task C at the same time. Each task runs on its own thread. The CPU switches between them so fast it looks simultaneous. On multi-core processors, they can truly run in parallel.

🍴 Analogy: Restaurant Waiters

Single-threaded = 1 waiter. One customer orders, the waiter goes to the kitchen, waits for the food, brings it back, THEN takes the next customer's order. Everyone waits in a long queue.

Multi-threaded = 10 waiters. Ten customers can order at the same time. Each waiter handles one customer independently. The kitchen (CPU) processes all orders, and food comes out much faster overall.

In Nimbus: when 5 parcels arrive for booking, single-threaded = process one booking at a time. Multi-threaded = process all 5 bookings simultaneously with 5 threads.

Single-Threaded
  • One task at a time
  • Simple to write and debug
  • No data sharing issues
  • Wastes CPU on waiting
  • Slow for I/O-heavy work
Multi-Threaded
  • Multiple tasks simultaneously
  • Complex to write and debug
  • Must handle shared data carefully
  • Uses CPU efficiently
  • Fast for I/O and parallel work

Where Threads Are Used in Real Life

  • Web servers — each incoming HTTP request gets its own thread (Tomcat uses a thread pool)
  • Android apps — UI runs on main thread, network calls run on background threads
  • Games — rendering, physics, audio, and input each run on separate threads
  • Batch processing — process 10,000 records in parallel instead of one by one
📦 In Nimbus

Nimbus uses BookingThread extends Thread for parallel booking processing. When multiple parcels need to be booked simultaneously — say during a peak hour rush — each booking runs on its own thread. The main thread doesn't block waiting for each booking to finish. It dispatches all of them and collects results.

Section 03

Creating Threads

Java gives you two ways to create threads. Both work, but one is better practice. Understanding why is key.

Way 1: Extending Thread

Your class extends the Thread class and overrides the run() method. This is what Nimbus does with BookingThread.

ExtendingThread.java 
class MyThread extends Thread {
    @Override
    public void run() {
        // This code runs on a separate thread
        System.out.println("Running on: " + getName());
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        MyThread t1 = new MyThread();
        MyThread t2 = new MyThread();
        t1.start();  // Starts a new thread, calls run() internally
        t2.start();  // Another new thread
    }
}
🔎 Line-by-line explanation
1
class MyThread extends Thread {
Our class inherits from Thread. Gets all thread capabilities (start, sleep, join, etc.)
2-3
@Override public void run() {
Override the run() method — this is the code that executes when the thread starts
4
System.out.println("Running on: " + getName());
getName() is inherited from Thread — returns "Thread-0", "Thread-1", etc.
10-11
t1.start(); t2.start();
start() creates a new OS thread and calls run() on it. Both threads run concurrently.

Way 2: Implementing Runnable

Your class implements the Runnable interface and passes itself to a Thread constructor.

ImplementingRunnable.java 
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("Running on: " + Thread.currentThread().getName());
    }
}

// Usage
public class Main {
    public static void main(String[] args) {
        MyTask task = new MyTask();
        Thread t1 = new Thread(task);
        Thread t2 = new Thread(task);
        t1.start();
        t2.start();

        // Lambda shortcut (Java 8+):
        Thread t3 = new Thread(() -> {
            System.out.println("Lambda thread: " + Thread.currentThread().getName());
        });
        t3.start();
    }
}

extends Thread vs implements Runnable

Criteriaextends Threadimplements Runnable
Inheritance Uses up your only extends slot Can still extend another class
Reusability Task tied to Thread object Same task can be shared by multiple threads
Thread pool Cannot use with ExecutorService Works perfectly with thread pools
Design IS-A Thread (tight coupling) HAS-A task (loose coupling, better design)
Best for Simple standalone threads Production code, enterprise apps
💡 Why Runnable is Preferred

Java only allows single inheritance. If your class extends Thread, it cannot extend anything else. With Runnable, your class can implement Runnable AND extend another class. Plus, Runnable separates "what to do" (the task) from "how to run it" (the thread), which is cleaner design.

📦 In Nimbus

Nimbus uses BookingThread extends Thread — which is fine for a learning project. In a production system, you would refactor it to BookingTask implements Runnable and use a thread pool. We will see both approaches.

Section 04

start() vs run()

This is THE most common threading mistake. You will see it in interviews, in exams, and in buggy code at work. Learn this once and never forget it.

The Difference

start() — CORRECT
  • Creates a new OS thread
  • JVM calls run() on that new thread
  • Code runs concurrently with main
  • Can only be called once per thread
run() — WRONG (usually)
  • No new thread created
  • Just a normal method call on main thread
  • Code runs sequentially on main
  • Can be called multiple times (but why?)
StartVsRun.java 
class DemoThread extends Thread {
    public void run() {
        System.out.println("Thread: " + Thread.currentThread().getName());
    }
}

public class Main {
    public static void main(String[] args) {
        DemoThread t = new DemoThread();

        // WRONG: Calling run() directly
        t.run();   // Prints: Thread: main   <-- runs on MAIN thread!

        // CORRECT: Calling start()
        t.start(); // Prints: Thread: Thread-0  <-- runs on NEW thread!
    }
}
⚠️ Interview Trap

"What happens if you call run() directly instead of start()?"

Answer: No new thread is created. run() executes as a regular method on the calling thread (main). The program behaves as single-threaded. This defeats the entire purpose of threading.

Calling start():                     Calling run():

main thread ──┬──────────────►       main thread ──── run() ──── continues
              │                      (everything on one thread, sequential)
              └── new thread ──►
              (two threads, concurrent)
start() forks execution; run() stays on the same thread

Thread Methods

The Thread class comes loaded with methods for controlling thread behavior. Here are the ones you will actually use.

Essential Methods

sleep() — Pausing a Thread

SleepExample.java 
public class Countdown extends Thread {
    public void run() {
        for (int i = 5; i >= 1; i--) {
            System.out.println("Launch in " + i + "...");
            try {
                Thread.sleep(1000); // Pause 1 second
            } catch (InterruptedException e) {
                System.out.println("Countdown interrupted!");
                return;
            }
        }
        System.out.println("LAUNCH!");
    }
}
💡 Why the try-catch?

Thread.sleep() throws InterruptedException — a checked exception. If another thread calls interrupt() on a sleeping thread, it wakes up by throwing this exception. You must handle it.

join() — Waiting for a Thread

join() is how you say "don't continue until this thread is done." Without it, the main thread races ahead while worker threads are still running.

JoinExample.java 
public class Main {
    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            System.out.println("Downloading file...");
            try { Thread.sleep(3000); } catch (InterruptedException e) {}
            System.out.println("Download complete.");
        });

        t1.start();
        System.out.println("Waiting for download...");

        t1.join();  // main thread BLOCKS here until t1 finishes

        System.out.println("File ready. Processing...");
        // Without join(), this would print BEFORE download finishes!
    }
}
main:   start t1 ──── join() ────────────── BLOCKED ────── "File ready" ──►
                         │                     ▲
t1:                      └─── downloading ─────┘ (t1 finishes, main unblocks)
join() makes the calling thread wait until the target thread completes

interrupt() — Signaling a Thread

InterruptExample.java 
Thread worker = new Thread(() -> {
    while (!Thread.currentThread().isInterrupted()) {
        System.out.println("Working...");
        try {
            Thread.sleep(500);
        } catch (InterruptedException e) {
            System.out.println("Got interrupt signal. Cleaning up...");
            break;  // Exit the loop gracefully
        }
    }
    System.out.println("Worker stopped.");
});

worker.start();
Thread.sleep(2000);    // Let it work for 2 seconds
worker.interrupt();     // Send stop signal
Section 06

Synchronization & Race Conditions

When multiple threads access the same data, things break in subtle and terrifying ways. Synchronization is how you prevent that chaos.

What is a Race Condition?

A race condition happens when two or more threads read and write the same variable at the same time, and the result depends on which thread gets there first. The "race" is unpredictable — your program gives different results on different runs.

RaceCondition.java — THE BUG 
class BankAccount {
    private int balance = 1000;

    public void withdraw(int amount) {
        if (balance >= amount) {          // Thread A checks: 1000 >= 800? YES
            // Thread A gets paused here by CPU scheduler...
            // Thread B checks: 1000 >= 800? YES (still 1000!)
            balance -= amount;             // Both threads subtract 800
        }
    }

    public int getBalance() { return balance; }
}

// Two threads withdrawing 800 from an account with 1000
// Expected: one succeeds, one fails (not enough balance)
// Reality: BOTH succeed! balance = 1000 - 800 - 800 = -600 !!!
⚠️ This is a real-world bug

This exact race condition has caused banks to lose money, e-commerce sites to oversell inventory, and airlines to double-book seats. In Nimbus, without synchronization, two threads could assign the same tracking number to different parcels.

The Fix: synchronized Keyword

The synchronized keyword creates a lock (also called a monitor). Only one thread can hold the lock at a time. Other threads must wait.

Synchronized Method

SynchronizedMethod.java 
class BankAccount {
    private int balance = 1000;

    // The ENTIRE method is locked. Only one thread at a time can execute it.
    public synchronized void withdraw(int amount) {
        if (balance >= amount) {
            balance -= amount;
            System.out.println(Thread.currentThread().getName()
                + " withdrew " + amount + ". Balance: " + balance);
        } else {
            System.out.println(Thread.currentThread().getName()
                + " failed. Insufficient balance: " + balance);
        }
    }

    public synchronized int getBalance() { return balance; }
}

Synchronized Block (more precise)

SynchronizedBlock.java 
class BankAccount {
    private int balance = 1000;
    private final Object lock = new Object(); // Explicit lock object

    public void withdraw(int amount) {
        // Only the critical section is locked, not the entire method
        synchronized (lock) {
            if (balance >= amount) {
                balance -= amount;
            }
        }
        // Code outside the block runs without lock — better performance
        System.out.println("Withdrawal processed.");
    }
}
Synchronized Method
  • Locks the entire method
  • Simpler to write
  • Lock object = this
  • Less granular — may lock too much
Synchronized Block
  • Locks only the critical section
  • Slightly more code
  • Lock object = any Object you choose
  • Better performance — locks only what's needed

How the Monitor/Lock Works

Thread A arrives ─── acquires lock ─── enters critical section ─── releases lock
                                          │
Thread B arrives ─── lock taken! ─── WAITS ─── lock released ─── acquires ─── enters
                                          │
Thread C arrives ─── lock taken! ─── WAITS ─── ─── ─── ─── ─── WAITS ─── acquires ─── enters
Only one thread can hold the monitor at a time. Others queue up.
Section 07

Thread Safety

Beyond synchronized, Java provides several tools for writing code that works correctly when multiple threads access it.

volatile Keyword

Each thread has its own CPU cache. When thread A writes a variable, thread B might read the old cached value. volatile forces every read/write to go directly to main memory — no caching.

VolatileExample.java 
class StoppableWorker extends Thread {
    // Without volatile, the worker thread might NEVER see the change
    private volatile boolean running = true;

    public void run() {
        while (running) {   // Always reads from main memory
            System.out.println("Working...");
        }
        System.out.println("Stopped.");
    }

    public void stopWorker() {
        running = false;    // Writes directly to main memory
    }
}

// Usage:
StoppableWorker w = new StoppableWorker();
w.start();
Thread.sleep(2000);
w.stopWorker();  // Without volatile, this might not work!
💡 volatile vs synchronized

volatile guarantees visibility — changes are seen by all threads. But it does NOT guarantee atomicity. count++ is NOT thread-safe even with volatile because count++ is actually three operations: read, increment, write. Use synchronized or Atomic classes for compound operations.

Atomic Classes

The java.util.concurrent.atomic package provides lock-free, thread-safe variables for single operations.

AtomicExample.java 
import java.util.concurrent.atomic.AtomicInteger;

class Counter {
    // AtomicInteger: thread-safe increment without synchronized
    private AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet();  // Atomic: read + increment + write in one step
    }

    public int getCount() {
        return count.get();
    }
}

// Other atomic classes:
// AtomicLong, AtomicBoolean, AtomicReference<T>

Thread-Safe Collections

Not Thread-SafeThread-Safe AlternativeNotes

HashMap ConcurrentHashMap Segments locked independently — high concurrency
LinkedList (as queue) ConcurrentLinkedQueue Lock-free queue for producer-consumer

Section 08

wait(), notify(), notifyAll()

These three methods enable threads to communicate with each other. They form the foundation of the producer-consumer pattern — one of the most common patterns in concurrent programming.

How It Works

  • wait() — releases the lock and puts the thread to sleep until notified
  • notify() — wakes up ONE waiting thread (chosen randomly)
  • notifyAll() — wakes up ALL waiting threads (they compete for the lock)

All three must be called inside a synchronized block. They are methods of Object, not Thread.

Producer-Consumer Pattern

One thread produces data, another consumes it. They share a buffer. The producer waits when the buffer is full. The consumer waits when the buffer is empty.

ProducerConsumer.java 
import java.util.LinkedList;
import java.util.Queue;

class ParcelBuffer {
    private final Queue<String> buffer = new LinkedList<>();
    private final int MAX_SIZE = 5;

    public synchronized void addParcel(String parcel) throws InterruptedException {
        while (buffer.size() == MAX_SIZE) {
            System.out.println("Buffer full! Producer waiting...");
            wait();  // Release lock, sleep until notified
        }
        buffer.add(parcel);
        System.out.println("Produced: " + parcel + " | Buffer: " + buffer.size());
        notifyAll();  // Wake up consumers
    }

    public synchronized String takeParcel() throws InterruptedException {
        while (buffer.isEmpty()) {
            System.out.println("Buffer empty! Consumer waiting...");
            wait();  // Release lock, sleep until notified
        }
        String parcel = buffer.poll();
        System.out.println("Consumed: " + parcel + " | Buffer: " + buffer.size());
        notifyAll();  // Wake up producers
        return parcel;
    }
}

// Producer thread
class Producer extends Thread {
    private final ParcelBuffer buffer;
    Producer(ParcelBuffer buffer) { this.buffer = buffer; }

    public void run() {
        for (int i = 1; i <= 10; i++) {
            try {
                buffer.addParcel("PKG-" + i);
                Thread.sleep(200);
            } catch (InterruptedException e) { break; }
        }
    }
}

// Consumer thread
class Consumer extends Thread {
    private final ParcelBuffer buffer;
    Consumer(ParcelBuffer buffer) { this.buffer = buffer; }

    public void run() {
        for (int i = 0; i < 10; i++) {
            try {
                buffer.takeParcel();
                Thread.sleep(500);  // Consumer is slower
            } catch (InterruptedException e) { break; }
        }
    }
}
🔎 Producer-Consumer breakdown
1
while (buffer.size() == MAX_SIZE) { wait(); }
Why while not if? Because when a thread wakes up, the condition might have changed again (spurious wakeup). Always re-check in a loop.
2
wait()
Releases the synchronized lock AND puts the thread to sleep. Other threads can now enter the synchronized block.
3
notifyAll()
Wakes up all waiting threads. They re-acquire the lock one by one and re-check their while condition.
⚠️ Common Mistake

Always use while with wait(), never if. Java allows spurious wakeups — a thread can wake up even without being notified. The while loop re-checks the condition after waking up.

Section 09

ExecutorService & Thread Pools

Manually creating threads is like hiring and firing employees for every task. Thread pools keep a reusable pool of threads — much more efficient. This is how real-world Java applications manage threads.

Why Thread Pools?

  • Creating threads is expensive — each thread needs memory for its stack (~512KB-1MB). Creating 1000 threads = ~1GB wasted.
  • No control over how many — without pools, 10,000 requests = 10,000 threads = system crash.
  • Thread pools reuse — a fixed number of threads pick up tasks from a queue. Finished? Take the next task. No creating/destroying.

Creating Thread Pools

ThreadPoolBasics.java 
import java.util.concurrent.*;

public class Main {
    public static void main(String[] args) throws Exception {
        // Create a pool with 3 threads
        ExecutorService pool = Executors.newFixedThreadPool(3);

        // Submit 5 tasks — only 3 run at a time, 2 wait in queue
        for (int i = 1; i <= 5; i++) {
            final int taskId = i;
            pool.submit(() -> {
                System.out.println("Task " + taskId
                    + " on " + Thread.currentThread().getName());
                try { Thread.sleep(1000); } catch (InterruptedException e) {}
            });
        }

        pool.shutdown();  // No new tasks accepted. Existing tasks finish.
        pool.awaitTermination(10, TimeUnit.SECONDS);  // Wait for all to finish
        System.out.println("All tasks complete.");
    }
}
🔎 Thread pool breakdown
1
Executors.newFixedThreadPool(3)
Creates a pool of exactly 3 threads. Tasks beyond 3 wait in an internal queue.
2
pool.submit(() -> { ... })
Submits a Runnable (lambda) to the pool. The pool assigns it to a free thread.
3
pool.shutdown()
Stops accepting new tasks. Already submitted tasks will complete. Without this, the program hangs forever.
4
pool.awaitTermination(10, TimeUnit.SECONDS)
Blocks until all tasks finish or 10 seconds pass, whichever comes first.

Types of Thread Pools

Factory MethodThreadsUse Case
newFixedThreadPool(n) Exactly n threads Known workload. E.g., process 3 bookings at a time.
newCachedThreadPool() Creates as needed, reuses idle Short-lived tasks with unpredictable volume.
newSingleThreadExecutor() Exactly 1 thread Tasks must run sequentially. E.g., logging.
newScheduledThreadPool(n) n threads with scheduling Periodic tasks. E.g., check for delayed parcels every 5 minutes.

Future — Getting Results from Threads

Runnable returns void. If you need a result from a thread, use Callable and Future.

FutureExample.java 
import java.util.concurrent.*;

public class Main {
    public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(2);

        // Callable returns a value (unlike Runnable)
        Callable<String> bookingTask = () -> {
            Thread.sleep(2000);
            return "Booking confirmed: PKG-" + (int)(Math.random() * 1000);
        };

        // submit() returns a Future — a promise of the result
        Future<String> future1 = pool.submit(bookingTask);
        Future<String> future2 = pool.submit(bookingTask);

        System.out.println("Tasks submitted. Doing other work...");

        // get() blocks until the result is ready
        String result1 = future1.get();  // Blocks until done
        String result2 = future2.get();

        System.out.println(result1);
        System.out.println(result2);

        pool.shutdown();
    }
}
💡 Runnable vs Callable

Runnable.run() returns void and cannot throw checked exceptions. Callable.call() returns a value and can throw exceptions. Use Callable when you need a result back from the thread.

Section 10

Nimbus BookingThread

This is the thread that powers parallel booking processing in Nimbus. Let's build it, break it down, and understand every line.

📦 In Nimbus

When a batch of parcels arrives at a Nimbus hub, each parcel needs booking — weight validation, cost calculation, tracking number generation, database insert. Instead of processing sequentially (slow), Nimbus spawns a BookingThread per parcel for parallel processing.

BookingThread.java 
package com.nimbus.threads;

import com.nimbus.model.Parcel;
import com.nimbus.service.BookingService;
import java.time.LocalDateTime;

public class BookingThread extends Thread {

    private final Parcel parcel;
    private final BookingService bookingService;
    private String result;
    private boolean success;

    public BookingThread(Parcel parcel, BookingService bookingService) {
        this.parcel = parcel;
        this.bookingService = bookingService;
        this.success = false;
        // Set a meaningful thread name for debugging
        setName("Booking-" + parcel.getTrackingId());
    }

    @Override
    public void run() {
        try {
            System.out.println("[" + getName() + "] Starting booking at "
                + LocalDateTime.now());

            // Step 1: Validate parcel weight and dimensions
            if (!bookingService.validateParcel(parcel)) {
                result = "FAILED: Validation failed for " + parcel.getTrackingId();
                return;
            }

            // Step 2: Calculate shipping cost
            double cost = bookingService.calculateCost(parcel);
            parcel.setCost(cost);

            // Step 3: Generate tracking number (synchronized internally)
            String trackingNumber = bookingService.generateTrackingNumber();
            parcel.setTrackingNumber(trackingNumber);

            // Step 4: Save to database
            bookingService.saveBooking(parcel);

            result = "SUCCESS: " + parcel.getTrackingId()
                + " | Cost: Rs." + cost
                + " | Tracking: " + trackingNumber;
            success = true;

            System.out.println("[" + getName() + "] Booking complete.");

        } catch (Exception e) {
            result = "ERROR: " + parcel.getTrackingId()
                + " - " + e.getMessage();
            System.err.println("[" + getName() + "] " + result);
        }
    }

    public String getResult() { return result; }
    public boolean isSuccess() { return success; }
}
🔎 BookingThread breakdown
1
public class BookingThread extends Thread
BookingThread IS-A Thread. It inherits start(), join(), getName(), sleep(), etc.
2
private final Parcel parcel;
Each thread processes one specific parcel. Marked final — cannot be reassigned after construction.
3
private final BookingService bookingService;
Shared service object. Multiple threads use the same BookingService — so its methods must be thread-safe (synchronized internally).
4
setName("Booking-" + parcel.getTrackingId());
Custom thread name for debugging. Instead of "Thread-0", logs show "Booking-PKG001". Makes logs readable.
5
bookingService.generateTrackingNumber()
This method MUST be synchronized. If two threads call it at the same time without synchronization, they could get the same tracking number.
6
private String result; private boolean success;
These store the outcome. The main thread calls getResult() and isSuccess() AFTER join() to read the results safely.
Section 11

Processing 5 Bookings in Parallel

Here is the complete example: creating 5 parcels, processing them on separate threads simultaneously, and collecting results.

ParallelBookingProcessor.java 
package com.nimbus.threads;

import com.nimbus.model.Parcel;
import com.nimbus.service.BookingService;
import java.util.ArrayList;
import java.util.List;

public class ParallelBookingProcessor {

    public static void main(String[] args) {
        BookingService service = new BookingService();

        // Create 5 parcels to book
        List<Parcel> parcels = List.of(
            new Parcel("PKG-001", "Mumbai", "Delhi", 2.5),
            new Parcel("PKG-002", "Chennai", "Kolkata", 1.8),
            new Parcel("PKG-003", "Pune", "Bangalore", 5.0),
            new Parcel("PKG-004", "Hyderabad", "Jaipur", 0.3),
            new Parcel("PKG-005", "Delhi", "Mumbai", 12.0)  // Overweight?
        );

        // Step 1: Create and start all threads
        List<BookingThread> threads = new ArrayList<>();
        long startTime = System.currentTimeMillis();

        for (Parcel parcel : parcels) {
            BookingThread thread = new BookingThread(parcel, service);
            threads.add(thread);
            thread.start();  // Each booking runs on its own thread
        }

        // Step 2: Wait for ALL threads to finish
        for (BookingThread thread : threads) {
            try {
                thread.join();  // Main thread waits for each one
            } catch (InterruptedException e) {
                System.err.println("Interrupted while waiting for " + thread.getName());
            }
        }

        long elapsed = System.currentTimeMillis() - startTime;

        // Step 3: Collect and display results
        System.out.println("\n===== BOOKING RESULTS =====");
        int successCount = 0;
        for (BookingThread thread : threads) {
            System.out.println(thread.getResult());
            if (thread.isSuccess()) successCount++;
        }

        System.out.println("\nTotal: " + parcels.size()
            + " | Success: " + successCount
            + " | Failed: " + (parcels.size() - successCount));
        System.out.println("Time: " + elapsed + "ms");
        System.out.println("(Sequential would take ~" + (parcels.size() * 1000) + "ms)");
    }
}

BookingPoolProcessor.java 
import java.util.concurrent.*;
import java.util.List;
import java.util.ArrayList;

public class BookingPoolProcessor {

    public static void main(String[] args) throws Exception {
        BookingService service = new BookingService();

        // Fixed pool of 3 threads — at most 3 bookings process at a time
        ExecutorService pool = Executors.newFixedThreadPool(3);

        List<Future<String>> futures = new ArrayList<>();

        for (int i = 1; i <= 5; i++) {
            final int id = i;
            Future<String> future = pool.submit(() -> {
                // Simulate booking work
                Thread.sleep(1000);
                return "Booking-" + id + " confirmed on "
                    + Thread.currentThread().getName();
            });
            futures.add(future);
        }

        // Collect results
        for (Future<String> f : futures) {
            System.out.println(f.get());  // Blocks until result is ready
        }

        pool.shutdown();
    }
}
Section 12

Deadlock

Deadlock is when two or more threads are stuck forever, each waiting for the other to release a lock. Nobody moves. The program freezes. No error. No exception. Just silence.

How Deadlock Happens

Thread A:  holds Lock-1  ───►  waiting for Lock-2
                                     ▲
                                     │ (held by Thread B)
                                     │
Thread B:  holds Lock-2  ───►  waiting for Lock-1
                                     ▲
                                     │ (held by Thread A)

Result: BOTH threads wait forever. Program frozen.
Classic deadlock: circular dependency on locks
DeadlockExample.java — DO NOT do this 
public class DeadlockDemo {
    private static final Object lockA = new Object();
    private static final Object lockB = new Object();

    public static void main(String[] args) {
        Thread t1 = new Thread(() -> {
            synchronized (lockA) {
                System.out.println("Thread 1: holds lockA, waiting for lockB...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockB) {  // STUCK! Thread 2 holds lockB
                    System.out.println("Thread 1: got both locks!");
                }
            }
        });

        Thread t2 = new Thread(() -> {
            synchronized (lockB) {  // Lock order is REVERSED!
                System.out.println("Thread 2: holds lockB, waiting for lockA...");
                try { Thread.sleep(100); } catch (InterruptedException e) {}
                synchronized (lockA) {  // STUCK! Thread 1 holds lockA
                    System.out.println("Thread 2: got both locks!");
                }
            }
        });

        t1.start();
        t2.start();
        // Program hangs forever. No exception. No crash. Just frozen.
    }
}

How to Prevent Deadlock

  1. Lock ordering: Always acquire locks in the same order. If Thread A locks A then B, Thread B must also lock A then B — never B then A.
  2. Timeout: Use tryLock(timeout) from ReentrantLock instead of synchronized. If the lock isn't available within the timeout, give up and retry.
  3. Minimize lock scope: Hold locks for the shortest time possible. Don't do I/O inside a synchronized block.
  4. Avoid nested locks: If you can avoid locking inside a lock, do it. Flatten your lock structure.
  5. Use higher-level concurrency: ExecutorService, ConcurrentHashMap, and BlockingQueue handle locking internally so you don't have to.
DeadlockFix.java — Same lock order 
// FIX: Both threads lock in the SAME order: A first, then B
Thread t1 = new Thread(() -> {
    synchronized (lockA) {
        synchronized (lockB) {
            System.out.println("Thread 1: got both locks!");
        }
    }
});

Thread t2 = new Thread(() -> {
    synchronized (lockA) {  // Same order as t1: A first!
        synchronized (lockB) {
            System.out.println("Thread 2: got both locks!");
        }
    }
});
// No deadlock possible. Threads compete for lockA, loser waits,
// winner gets both locks, finishes, releases, then loser proceeds.
⚠️ Deadlock in Real Systems

Deadlocks are some of the hardest bugs to find. They don't crash the program — it just stops responding. In production, monitoring tools detect "thread not progressing" and alert the team. Prevention is always better than detection.

Practice Exercises

Thread questions are tricky because output order is non-deterministic. These exercises test your understanding of lifecycle, scheduling, and synchronization.

Exercise 1: What Will This Print?

Exercise1.java 
class MyThread extends Thread {
    public void run() {
        System.out.println("A");
    }
}

public class Main {
    public static void main(String[] args) {
        MyThread t = new MyThread();
        t.run();
        t.run();
        t.start();
        System.out.println("B");
    }
}
🎯 Answer

The first two t.run() calls are normal method calls on the main thread. They print "A", "A" sequentially. Then t.start() creates a new thread. After that, "B" and "A" can print in ANY order because the new thread and main thread race. Possible outputs:

  • A A B A (main prints B before thread runs)
  • A A A B (thread runs before main prints B)

The first two A's are always first and always in order. The last A and B are non-deterministic.

Exercise 2: Thread Interleaving

Exercise2.java 
Thread t1 = new Thread(() -> {
    for (int i = 1; i <= 3; i++) System.out.print(i + " ");
});
Thread t2 = new Thread(() -> {
    for (int i = 1; i <= 3; i++) System.out.print(i + " ");
});
t1.start();
t2.start();
🎯 Answer

Both threads print 1, 2, 3 — but interleaved in unpredictable ways. All of these are possible:

  • 1 2 3 1 2 3 (t1 finishes before t2 starts)
  • 1 1 2 2 3 3 (perfectly interleaved)
  • 1 2 1 3 2 3 (random interleaving)

The CPU scheduler decides. You will get different output each time you run it. This is the nature of concurrency.

Exercise 3: join() Behavior

Exercise3.java 
Thread t1 = new Thread(() -> System.out.println("First"));
Thread t2 = new Thread(() -> System.out.println("Second"));

t1.start();
t1.join();   // main waits until t1 finishes
t2.start();
t2.join();   // main waits until t2 finishes
System.out.println("Done");
🎯 Answer

Output is always: First, Second, Done — in that exact order. The join() calls force sequential execution. t2 doesn't start until t1 is finished. "Done" doesn't print until t2 is finished. This effectively makes it single-threaded (which defeats the purpose, but it's for understanding join).

Exercise 4: Race Condition Detection

Exercise4.java 
class Counter {
    int count = 0;
    void increment() { count++; }
}

Counter c = new Counter();
Thread t1 = new Thread(() -> { for (int i = 0; i < 10000; i++) c.increment(); });
Thread t2 = new Thread(() -> { for (int i = 0; i < 10000; i++) c.increment(); });
t1.start(); t2.start();
t1.join(); t2.join();
System.out.println(c.count);  // Expected: 20000. Actual: ???
🎯 Answer

The answer is almost always less than 20000. Could be 18432, 19567, or any random number. Why? count++ is not atomic. It is three operations: read count, add 1, write count. Two threads doing this simultaneously lose updates. Fix: make increment() synchronized or use AtomicInteger.

Exercise 5: Starting a Thread Twice

Exercise5.java 
Thread t = new Thread(() -> System.out.println("Hello"));
t.start();
t.start();  // What happens here?
🎯 Answer

IllegalThreadStateException on the second start(). A thread can only be started once. After it finishes (or even while running), calling start() again throws this exception. To run the same task again, create a new Thread object.
