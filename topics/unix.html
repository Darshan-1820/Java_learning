Section 01

What is Unix/Shell?

Before GUIs existed, every computer was operated by typing text commands. Unix is the operating system that started it all — and its command line is still the fastest way to process files, automate tasks, and manage servers.

GUI vs Command Line

You already know the GUI way: click a folder, double-click a file, drag to delete. The command line interface (CLI) does all of this with typed commands. It looks scary at first, but it is far more powerful.

GUI (Graphical)
  • Click, drag, scroll
  • Easy to learn
  • Slow for repetitive tasks
  • Cannot automate easily
CLI (Command Line)
  • Type commands
  • Small learning curve
  • Lightning fast for bulk operations
  • Fully scriptable & automatable

Why Developers Use It

In Nimbus, the parcel data lives in text files (Parcel.txt, booking.txt, customer.txt). Instead of writing a full Java program to search, filter, or update these files, you can do it in a single shell command. That is the power of Unix.

In Nimbus

Nimbus requires 6 shell commands that search, count, sort, and update data across three text files. Every command you learn in this page builds directly toward those 6.

The Shell

The shell is the program that reads your commands and executes them. Think of it as a translator between you and the operating system. The most common shell is Bash (Bourne Again Shell). When you open a terminal, you are inside a shell.

  You type a command
        |
        v
  +-----------+      +-----------------+      +------------+
  |   Shell   | -->  | Operating System | -->  |   Result   |
  |  (Bash)   |      |   (Unix/Linux)   |      |  (Output)  |
  +-----------+      +-----------------+      +------------+
The shell interprets your commands and talks to the OS

The Prompt

When the shell is ready, it shows a prompt — usually ending with $. This is where you type.

Terminal 
user@machine:~/nimbus$ ls
Parcel.txt  booking.txt  customer.txt

Everything before $ is info (user, machine, current directory). Everything after $ is what you type.

Section 03

File Permissions

Unix controls who can read, write, or execute every file. Understanding permissions is essential for managing files on servers.

The rwx System

Every file has three permission types for three groups of users:

PermissionLetterNumberMeaning
Readr4Can view the file contents
Writew2Can modify the file
Executex1Can run the file as a program

Three groups: Owner (u), Group (g), Others (o).

  ls -l Parcel.txt
  -rw-r--r--  1  user  staff  2048  Apr 23  Parcel.txt
   |__|__|__
    |   |  |
    |   |  +-- Others: r-- (read only)
    |   +----- Group:  r-- (read only)
    +--------- Owner:  rw- (read + write)
Breaking down the permission string from ls -l

chmod — Change Permissions

Use chmod with numbers (add up r=4, w=2, x=1 for each group):

Permission Examples 
# Owner: rwx (7), Group: r-x (5), Others: r-- (4)
chmod 754 script.sh

# Make a file readable and writable by everyone
chmod 666 Parcel.txt

# Make a script executable
chmod +x deploy.sh
Think About It

chmod 755 means 7 (rwx) for owner, 5 (r-x) for group, 5 (r-x) for others. The owner can do everything; others can read and run the file but not modify it. This is the most common permission for scripts.

Section 04

The Pipe Operator |

The pipe is Unix's superpower. It lets you chain commands together — the output of one command flows directly as input to the next.

The Assembly Line Analogy

Imagine a factory assembly line. Raw material enters station 1, gets processed, moves to station 2 for more processing, then station 3 for finishing. Each station does one thing well, and the product flows between them.

That is exactly what | (pipe) does with commands:

cat Parcel.txtRead the file
→
grep "Delivered"Filter rows
→
wc -lCount lines
→
Result: 5Final output
Each command processes data and passes it to the next via the pipe (|)
Pipe in action 
# Without pipe: two separate steps
grep "Delivered" Parcel.txt > temp.txt
wc -l temp.txt

# With pipe: one clean command
grep "Delivered" Parcel.txt | wc -l

Key rules of pipes:

  • The | symbol connects two commands
  • stdout (standard output) of the left command becomes stdin (standard input) of the right command
  • You can chain as many pipes as you need: cmd1 | cmd2 | cmd3 | cmd4
  • Each command in the chain does one specific job — the Unix philosophy
Analogy

Think of pipe as a conveyor belt. cat Parcel.txt dumps all parcels onto the belt. grep "Delivered" picks out only delivered ones. wc -l counts how many came through. No temporary files, no mess.

grep — Pattern Matching

grep searches through text and prints every line that matches your pattern. It is the most-used command in Unix.

Basic Syntax

grep syntax 
grep "pattern" filename

grep reads each line of the file. If the line contains the pattern, it prints it. If not, it skips it.

Essential Flags

Section 06

awk — Column Processing

awk is a mini programming language built for processing structured text. It reads a file line by line, splits each line into columns, and lets you pick, filter, and transform specific columns.

How awk Thinks

awk sees every line as a row of columns separated by a delimiter. By default, it splits on whitespace. For Nimbus data files (pipe-delimited), you tell it to split on | with -F'|'.

  Line: C101|B001|2024-01-15|Ravi|Mumbai|500|Delivered

  awk -F'|' splits this into:
  $1 = C101           (Customer ID)
  $2 = B001           (Booking ID)
  $3 = 2024-01-15     (Booking Date)
  $4 = Ravi           (Receiver Name)
  $5 = Mumbai         (Delivered Address)
  $6 = 500            (Amount)
  $7 = Delivered       (Status)

  $0 = the entire line
  NR = current line number
  NF = number of fields in current line
awk splits each line into numbered columns ($1, $2, $3...)

Basic Syntax

awk syntax 
# Print specific columns
awk -F'|' '{print $1, $2}' Parcel.txt

# With condition: only print if column 7 is "Delivered"
awk -F'|' '$7 == "Delivered" {print $1, $2, $6}' Parcel.txt

# Custom output separator
awk -F'|' '{print $1 " | " $4 " | " $7}' Parcel.txt

Conditions in awk

awk can filter rows using conditions before the {action} block:

awk conditions 
# Only rows where amount > 300
awk -F'|' '$6 > 300 {print $1, $4, $6}' Parcel.txt

# Pattern matching with ~ (contains)
awk -F'|' '$7 ~ /Deliver/ {print $0}' Parcel.txt

# Multiple conditions with && (AND) and || (OR)
awk -F'|' '$6 > 200 && $7 == "Delivered" {print $2, $6}' Parcel.txt

BEGIN and END Blocks

BEGIN runs once before processing any line. END runs once after all lines are processed. Useful for headers, totals, and counters.

BEGIN and END 
# Print header, then data, then total
awk -F'|' '
  BEGIN { print "BookingID | Customer | Amount" }
  { total += $6; print $2 " | " $4 " | " $6 }
  END { print "---"; print "Total: " total }
' Parcel.txt

Associative Arrays (Counting)

awk has built-in associative arrays (like Java's HashMap). This is how you count things by group.

Counting with arrays 
# Count bookings per customer
awk -F'|' '{count[$2]++} END {for (c in count) print c, count[c]}' booking.txt
In Nimbus — Command 2

Search booking by BookingID, display BookingId | CustomerName | Par_Status:

awk -F'|' -v id="B001" '$1 == id {found=1; print $1 " | " $2 " | " $7} END {if (!found) print "Booking details Not Found"}' booking.txt

The -v id="B001" passes the search value as a variable. If no match is found after processing all lines, the END block prints the error message.

Line-by-line: Booking search command
1
awk -F'|'
Start awk, set field separator to pipe character
2
-v id="B001"
Pass a variable called id with value "B001" into the awk program

Section 07

sed — Stream Editor

sed reads text line by line, applies transformations (find & replace, delete, insert), and outputs the result. Think of it as "Find & Replace" for the command line.

Substitute Command

The most common sed operation is substitution:

sed syntax 
# Basic: replace first occurrence on each line
sed 's/old/new/' filename

# Replace ALL occurrences on each line (g = global)
sed 's/old/new/g' filename

# Case insensitive replace
sed 's/old/new/gi' filename

In-Place Editing with -i

By default, sed prints the modified text to screen but does NOT change the file. To actually modify the file, use -i:

In-place editing 
# Preview the change (no -i, just prints to screen)
sed 's/old@email.com/new@email.com/' customer.txt

# Actually modify the file (-i = in-place)
sed -i 's/old@email.com/new@email.com/' customer.txt

# Create a backup before modifying (-i.bak saves original as customer.txt.bak)
sed -i.bak 's/old@email.com/new@email.com/' customer.txt
Warning

sed -i modifies the file permanently. Always preview without -i first, or use -i.bak to keep a backup.

Other sed Operations

More sed tricks 
# Delete lines containing "Cancelled"
sed '/Cancelled/d' Parcel.txt

# Delete line 3
sed '3d' Parcel.txt

# Print only lines 2-5
sed -n '2,5p' Parcel.txt

# Replace only on line 4
sed '4s/old/new/' filename
In Nimbus — Command 5

Update email of customer in customer.txt:

sed -i 's/ravi@old.com/ravi@new.com/' customer.txt

This finds the exact old email and replaces it with the new one, modifying the file directly. Since customer.txt is comma-delimited, the email is just another field in the line — sed does not care about delimiters, it matches text anywhere in the line.

Section 08

sort — Ordering Data

sort arranges lines of text in a specific order — alphabetical, numerical, ascending, descending, or by a specific column.

Basic Usage

sort basics 
# Alphabetical sort (default)
sort names.txt

# Reverse (descending) sort
sort -r names.txt

# Numeric sort (treats content as numbers, not text)
sort -n numbers.txt

# Numeric descending
sort -rn numbers.txt

Sorting by Column

For structured files, you sort by a specific column using -k (key) and -t (delimiter):

FlagMeaningExample
-t'|'Set field delimiter to pipesort -t'|' ...
-k2Sort by column 2sort -t'|' -k2 booking.txt
-k3,3Sort by column 3 only (start and end at 3)sort -t'|' -k3,3 Parcel.txt
-rReverse (descending)sort -t'|' -k1,1r booking.txt
-nNumeric sortsort -t'|' -k6,6nr Parcel.txt
In Nimbus — Command 6

Display Booking ID, Customer Name, Date of Booking in descending order of Booking ID:

awk -F'|' '{print $1 "|" $2 "|" $6}' booking.txt | sort -t'|' -k1,1r

First, awk extracts columns 1, 2, and 6 from booking.txt. Then sort orders them by column 1 (Booking ID) in reverse (descending). The pipe connects the two commands.

Line-by-line: Sort by Booking ID descending
1
awk -F'|'
Start awk with pipe as the field separator
2
'{print $1 "|" $2 "|" $6}'
For each line, print Booking ID ($1), Customer Name ($2), Date of Booking ($6), separated by pipes

booking.txt
Read from the booking.txt file

| sort -t'|' -k1,1r
Pipe output to sort: -t'|' sets delimiter, -k1,1 sorts by column 1, r makes it descending

Section 10

uniq — Removing Duplicates

uniq removes consecutive duplicate lines. The critical rule: the data MUST be sorted first, because uniq only compares each line with the one directly above it.

Critical Rule

uniq only removes adjacent duplicates. If duplicates are scattered across the file, they will NOT be removed. Always sort first!

uniq examples 
# WRONG: uniq without sort (will miss non-adjacent duplicates)
cut -d'|' -f7 Parcel.txt | uniq

# CORRECT: sort first, then uniq
cut -d'|' -f7 Parcel.txt | sort | uniq

# Count occurrences of each unique value
cut -d'|' -f7 Parcel.txt | sort | uniq -c

# Show only duplicate lines
cut -d'|' -f7 Parcel.txt | sort | uniq -d

# Show only unique (non-duplicate) lines
cut -d'|' -f7 Parcel.txt | sort | uniq -u
FlagMeaning
-cPrefix lines with count of occurrences
-dOnly show duplicated lines
-uOnly show unique (non-repeated) lines
-iCase insensitive comparison

Section 13

Chaining Commands with Pipes

Real-world data processing often needs 3, 4, or even 5 commands chained together. Here are practical examples using Nimbus data.

Multi-Step Pipeline

Chaining examples 
# List all unique statuses from Parcel.txt, sorted
cut -d'|' -f7 Parcel.txt | sort | uniq

# Count how many parcels each status has, sorted by count
cut -d'|' -f7 Parcel.txt | sort | uniq -c | sort -rn

# Top 3 highest-amount parcels (columns: CustomerID, Amount, Status)
awk -F'|' '{print $1 "|" $6 "|" $7}' Parcel.txt | sort -t'|' -k2,2nr | head -n 3

# All delivered parcels, show only receiver and address, sorted alphabetically
grep "Delivered" Parcel.txt | awk -F'|' '{print $4 " - " $5}' | sort

# Customer IDs with the most bookings
cut -d'|' -f1 Parcel.txt | sort | uniq -c | sort -rn

Reading a Pipeline Left to Right

Every pipeline is a story. Read it left to right, one command at a time:

cut -d'|' -f7Extract status column
→
sortAlphabetize
→
uniq -cCount each unique value
→
sort -rnSort by count descending
Each step transforms the data and passes it to the next
Building Pipelines

When building a complex pipeline, start with just the first command and run it. Then add the next pipe and run again. Keep adding one step at a time until you get the result you want. Never write the full pipeline at once — build it piece by piece.

Redirection: Saving Output to Files

Instead of printing to screen, you can save output to a file:

Output redirection 
# Save to a new file (overwrites if exists)
grep "Delivered" Parcel.txt > delivered_parcels.txt

# Append to a file (does not overwrite)
grep "Pending" Parcel.txt >> status_report.txt

# Redirect errors to a file
grep "xyz" Parcel.txt 2> errors.txt

# Redirect both output and errors
grep "Delivered" Parcel.txt > output.txt 2>&1
OperatorMeaning
>Redirect output to file (overwrite)
>>Redirect output to file (append)
2>Redirect errors to file
2>&1Redirect errors to same place as output
<Read input from a file
