Section 01

Why Generics Exist

Before Java 5, collections stored everything as Object. You had to cast every time you pulled something out, and the compiler could not help you if you put the wrong type in. Generics fix this by letting you declare what type a collection holds — so the compiler catches mistakes before your code ever runs.

The Problem: Life Without Generics

Imagine you have an ArrayList that is supposed to hold User objects. Without generics, Java has no way to enforce that. Anyone can add anything — a String, a Booking, an Integer — and the compiler will not complain.

WithoutGenerics.java 
// Without generics — raw ArrayList
ArrayList users = new ArrayList();
users.add(new User("Darshan", "darshan@email.com"));
users.add("This is not a User!");  // No error! Compiler allows it.

// Later, when you try to use it...
User u = (User) users.get(1);  // BOOM! ClassCastException at runtime
// You put a String in, but tried to cast it to User

The problem is clear: the compiler let you put a String into a list that was supposed to hold User objects. You only find the bug when the program crashes at runtime — possibly in production, in front of real users.

The Solution: Generics

WithGenerics.java 
// With generics — type-safe ArrayList
ArrayList<User> users = new ArrayList<>();
users.add(new User("Darshan", "darshan@email.com"));
users.add("This is not a User!");  // COMPILE ERROR! Won't even run.

// No casting needed — Java already knows it's a User
User u = users.get(0);  // Clean. Safe. No cast.
Line-by-line explanation
1
ArrayList<User> users = new ArrayList<>();
<User> is the type parameter. It tells Java: "this ArrayList will ONLY hold User objects." The <> on the right (diamond operator) lets Java infer the type from the left side.
2
users.add(new User("Darshan", "darshan@email.com"));
Adding a User — matches the declared type. Compiler says OK.
3
users.add("This is not a User!");
Adding a String to an ArrayList<User> — compiler immediately flags this as an error. The bug is caught BEFORE the program runs.
4
User u = users.get(0);
No cast needed. The compiler knows get() returns a User because the list was declared as ArrayList<User>.

Three Benefits of Generics

Benefit Without Generics With Generics
Type Safety Wrong types discovered at runtime (crash) Wrong types caught at compile time (red underline)
No Casting (User) list.get(0) — manual cast every time list.get(0) — clean, no cast
Code Reuse Write separate classes for UserList, BookingList, etc. One ArrayList<T> works for any type
In Nimbus

Every collection in Nimbus uses generics. ArrayList<User> in UserDAO, ArrayList<Booking> in BookingDAO, HashMap<String, Booking> in BookingService. Without generics, every call to get() would need a manual cast, and one wrong type would crash the parcel management system in production.

Section 02

Generic Classes

A generic class is a class that takes a type parameter. Instead of hardcoding a specific type, you use a placeholder like T that gets replaced with a real type when someone creates an instance.

Writing Your First Generic Class

Box.java 
public class Box<T> {
    private T item;

    public void put(T item) {
        this.item = item;
    }

    public T get() {
        return item;
    }
}
Line-by-line explanation
1
public class Box<T> {
T is a type parameter — a placeholder. It does not exist as a real type. When someone creates Box<String>, every T in this class becomes String. When someone creates Box<User>, every T becomes User.
2
private T item;
The field's type is whatever T is. For Box<String>, this is private String item;. For Box<Booking>, this is private Booking item;.
4
public void put(T item)
The method only accepts an argument of type T. If this is Box<User>, you can only put a User in.
8
public T get()
Returns whatever type T is. No casting needed by the caller.

Using the Generic Class

BoxUsage.java 
// A box that holds Strings
Box<String> nameBox = new Box<>();
nameBox.put("Darshan");
String name = nameBox.get();  // Returns String, no cast

// A box that holds Users
Box<User> userBox = new Box<>();
userBox.put(new User("Darshan", "darshan@email.com"));
User u = userBox.get();  // Returns User, no cast

// A box that holds Integers
Box<Integer> ageBox = new Box<>();
ageBox.put(24);
int age = ageBox.get();  // Auto-unboxing from Integer to int

One class definition. Three completely different uses. The compiler enforces type safety for each one independently.

Multiple Type Parameters

A class can take more than one type parameter. Java's HashMap<K, V> is the most common example — K for key type, V for value type.

Pair.java 
public class Pair<A, B> {
    private A first;
    private B second;

    public Pair(A first, B second) {
        this.first = first;
        this.second = second;
    }

    public A getFirst()  { return first; }
    public B getSecond() { return second; }
}

// Usage
Pair<String, Integer> entry = new Pair<>("Darshan", 24);
String name = entry.getFirst();   // String
int age = entry.getSecond();      // Integer (auto-unboxed)

Common Type Parameter Names

Letter Convention Example
T Type (general purpose) Box<T>, ArrayList<T>
E Element (collections) List<E>, Set<E>
K Key (maps) HashMap<K, V>
V Value (maps) HashMap<K, V>
N Number Calculator<N>
Key Insight

These are just conventions, not rules. You could write class Box<PIZZA> and Java would compile it fine. But following conventions makes your code readable. When other developers see T, they instantly know "this is a type parameter."

Section 03

Generic Methods

You do not need an entire generic class to use generics. A single method can have its own type parameter, independent of its class. The type parameter is declared before the return type.

Syntax

GenericMethod.java 
public class Util {

    // Generic method — <T> declares the type parameter
    public static <T> T findFirst(ArrayList<T> list) {
        if (list.isEmpty()) {
            return null;
        }
        return list.get(0);
    }
}
Line-by-line explanation
1
public static <T> T findFirst(ArrayList<T> list)
Three parts: <T> declares the type parameter. The second T is the return type (returns whatever type the list holds). ArrayList<T> list is the parameter (a list of that same type). Java infers T from whatever you pass in.

Calling Generic Methods

GenericMethodUsage.java 
ArrayList<User> users = new ArrayList<>();
users.add(new User("Darshan", "d@email.com"));

// Java infers T = User from the argument
User first = Util.findFirst(users);  // Returns User

ArrayList<String> names = new ArrayList<>();
names.add("Nimbus");

// Java infers T = String
String firstName = Util.findFirst(names);  // Returns String

You never had to tell Java what T is. It figured it out from the argument type. This is called type inference.

A More Practical Example: Print Any Array

PrintArray.java 
public static <T> void printAll(T[] items) {
    for (T item : items) {
        System.out.println(item);
    }
}

// Works with any type
String[] names = {"Darshan", "Nimbus", "Parcel"};
Integer[] ids = {101, 102, 103};

printAll(names);  // T = String
printAll(ids);    // T = Integer
Analogy: The Template Letter

A generic method is like a template letter with blanks: "Dear ___, your order #___ is ready." The blanks (T) get filled in with real values when you actually use the letter. One template, unlimited uses.

Section 04

Bounded Types

Sometimes T being "anything" is too broad. What if you need T to be a number? Or something that can be compared? Bounded types let you restrict what T can be.

Upper Bounded: <T extends Something>

The keyword extends means "T must be this type or any subclass of it." It sets an upper limit.

NumberBox.java 
// T must be Number or a subclass (Integer, Double, Float, etc.)
public class NumberBox<T extends Number> {
    private T value;

    public NumberBox(T value) {
        this.value = value;
    }

    public double toDouble() {
        // We can call Number methods because T is guaranteed to be a Number
        return value.doubleValue();
    }
}

// OK — Integer extends Number
NumberBox<Integer> intBox = new NumberBox<>(42);

// OK — Double extends Number
NumberBox<Double> doubleBox = new NumberBox<>(3.14);

// COMPILE ERROR — String does NOT extend Number
NumberBox<String> strBox = new NumberBox<>("hello");

Bounded Type in Methods: Comparable

The most common bounded type in real Java code is <T extends Comparable<T>>. This means "T must be something that knows how to compare itself to other T's."

FindMax.java 
public static <T extends Comparable<T>> T findMax(ArrayList<T> list) {
    T max = list.get(0);
    for (T item : list) {
        if (item.compareTo(max) > 0) {
            max = item;
        }
    }
    return max;
}

// Works with String (String implements Comparable<String>)
ArrayList<String> names = new ArrayList<>(List.of("Darshan", "Nimbus", "Alpha"));
String maxName = findMax(names);  // "Nimbus" (alphabetically last)

// Works with Integer (Integer implements Comparable<Integer>)
ArrayList<Integer> nums = new ArrayList<>(List.of(5, 12, 3, 8));
int maxNum = findMax(nums);  // 12
Line-by-line explanation
1
<T extends Comparable<T>>
T must implement the Comparable interface for its own type. This guarantees we can call .compareTo() on any two T objects. Without this bound, Java would not let us call compareTo because it does not know T has that method.
4
item.compareTo(max) > 0
compareTo returns a positive number if item is greater than max. We can call this ONLY because of the extends Comparable<T> bound.

Multiple Bounds

A type parameter can have multiple bounds using &. The class must come first, then interfaces.

MultipleBounds.java 
// T must extend Number AND implement Comparable
public static <T extends Number & Comparable<T>> T findMax(ArrayList<T> list) {
    T max = list.get(0);
    for (T item : list) {
        if (item.compareTo(max) > 0) {
            max = item;
        }
    }
    return max;
}
Watch Out

In generics, extends means both "extends a class" AND "implements an interface." You always write extends, never implements, even when the bound is an interface. <T implements Comparable> is a syntax error.

Section 05

Wildcards

Wildcards (?) are used when you do not care about the exact type, or when you want to accept a range of types. They appear in method parameters and variable declarations — never in class definitions.

The Problem Wildcards Solve

You might expect this to work, but it does not:

WildcardProblem.java 
public static void printList(ArrayList<Object> list) {
    for (Object item : list) {
        System.out.println(item);
    }
}

ArrayList<String> names = new ArrayList<>();
printList(names);  // COMPILE ERROR!
// ArrayList<String> is NOT a subtype of ArrayList<Object>

This is counterintuitive. String is a subtype of Object, so why is ArrayList<String> not a subtype of ArrayList<Object>? Because if it were, you could add an Integer to a list of Strings through the Object reference, which would break type safety. Java prevents this at compile time.

Unbounded Wildcard: ?

The ? wildcard means "any type." Use it when you only need to read from a collection and do not care about the specific type.

UnboundedWildcard.java 
// Accepts ArrayList of ANY type
public static void printList(ArrayList<?> list) {
    for (Object item : list) {
        System.out.println(item);
    }
}

ArrayList<String> names = new ArrayList<>();
ArrayList<Integer> nums = new ArrayList<>();
ArrayList<User> users = new ArrayList<>();

printList(names);  // OK
printList(nums);   // OK
printList(users);  // OK

Upper Bounded Wildcard: ? extends T

"Accept this type or any subtype." Used when you want to read from a collection.

UpperBoundedWildcard.java 
// Accepts ArrayList of Number or any subclass (Integer, Double, Float...)
public static double sumAll(ArrayList<? extends Number> numbers) {
    double total = 0;
    for (Number n : numbers) {
        total += n.doubleValue();
    }
    return total;
}

ArrayList<Integer> ints = new ArrayList<>(List.of(1, 2, 3));
ArrayList<Double> doubles = new ArrayList<>(List.of(1.5, 2.5));

sumAll(ints);     // OK — Integer extends Number
sumAll(doubles);  // OK — Double extends Number

Lower Bounded Wildcard: ? super T

"Accept this type or any supertype." Used when you want to write to a collection.

LowerBoundedWildcard.java 
// Accepts ArrayList of Integer or any supertype (Number, Object)
public static void addNumbers(ArrayList<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
}

ArrayList<Integer> ints = new ArrayList<>();
ArrayList<Number> nums = new ArrayList<>();
ArrayList<Object> objs = new ArrayList<>();

addNumbers(ints);  // OK — Integer is Integer
addNumbers(nums);  // OK — Number is a supertype of Integer
addNumbers(objs);  // OK — Object is a supertype of Integer

PECS Rule: Producer Extends, Consumer Super

This is the golden rule for remembering which wildcard to use:

Situation Wildcard Mnemonic Why
You READ from the collection (it produces values) ? extends T Producer Extends You know everything is at least T, so reading is safe
You WRITE to the collection (it consumes values) ? super T Consumer Super You know the collection can hold T or bigger, so adding T is safe
You both read AND write No wildcard, use T — You need the exact type for both operations
Key Insight

With ? extends Number you can GET items out (as Number) but you CANNOT put items in (because Java does not know the exact subtype). With ? super Integer you can PUT Integer items in but you cannot safely GET specific types out (you only get Object).

Section 06

Type Erasure

Here is a fact that surprises most Java developers: generics only exist at compile time. The JVM has no idea about them at runtime. This design decision is called type erasure, and it explains many of the "weird" limitations of generics.

What Actually Happens

When you compile your code, the Java compiler:

  1. Checks all your generic types for correctness (type safety)
  2. Replaces every type parameter T with Object (or the bound, if there is one)
  3. Inserts casts where needed
  4. Produces bytecode that has NO generic information at all
What You Write
ArrayList<User> users = new ArrayList<>();
users.add(new User("Darshan"));
User u = users.get(0);
What JVM Sees (After Erasure)
ArrayList users = new ArrayList();
users.add(new User("Darshan"));
User u = (User) users.get(0);

At runtime, ArrayList<User> and ArrayList<Booking> are the exact same class. The JVM sees just ArrayList.

What You CANNOT Do Because of Type Erasure

ErasureLimitations.java 
public class Box<T> {

    // CANNOT create an instance of T
    T item = new T();          // ERROR: Type parameter T cannot be instantiated

    // CANNOT create an array of T
    T[] items = new T[10];     // ERROR: Cannot create a generic array

    // CANNOT use instanceof with T
    if (obj instanceof T) {}   // ERROR: Cannot perform instanceof check against T

    // CANNOT use T in a static context
    static T value;            // ERROR: Cannot make a static reference to T
}

All of these fail because T does not exist at runtime. The JVM cannot create an instance of something it does not know about. It cannot check instanceof against a type it has never heard of. These are compile-time constructs that get erased.

Bounded Type Erasure

If you have <T extends Number>, the compiler erases T to Number (the bound) instead of Object. This is why bounded types can call methods of their bound.

Common Gotcha

Because of type erasure, you cannot overload methods based on generic types. void process(ArrayList<User> list) and void process(ArrayList<Booking> list) have the same erasure — void process(ArrayList list) — so Java treats them as duplicate methods.

Section 07

Generic Interfaces

Many of Java's most important interfaces are generic. When you implement them, you specify the type parameter, and the compiler enforces type safety on all the methods you must implement.

Comparable<T>

Comparable<T> has one method: compareTo(T other). It defines the natural ordering of objects.

User.java 
public class User implements Comparable<User> {
    private String name;
    private String email;

    @Override
    public int compareTo(User other) {
        return this.name.compareTo(other.name);  // Sort by name
    }
}

// Now you can sort users
ArrayList<User> users = new ArrayList<>();
Collections.sort(users);  // Uses compareTo internally

Without the generic parameter, compareTo would take an Object and you would need to cast it. With Comparable<User>, the parameter is already typed as User.

Iterable<T>

If a class implements Iterable<T>, it can be used in a for-each loop.

BookingList.java 
public class BookingList implements Iterable<Booking> {
    private ArrayList<Booking> bookings = new ArrayList<>();

    public void add(Booking b) {
        bookings.add(b);
    }

    @Override
    public Iterator<Booking> iterator() {
        return bookings.iterator();
    }
}

// Now you can use for-each
BookingList list = new BookingList();
for (Booking b : list) {
    System.out.println(b.getTrackingId());
}

Comparator<T>

Comparator<T> defines an external ordering. Unlike Comparable (which is built into the class), a Comparator is a separate object that knows how to compare two T's.

BookingComparators.java 
// Sort bookings by cost
Comparator<Booking> byCost = new Comparator<Booking>() {
    @Override
    public int compare(Booking a, Booking b) {
        return Double.compare(a.getCost(), b.getCost());
    }
};

// Sort bookings by date
Comparator<Booking> byDate = new Comparator<Booking>() {
    @Override
    public int compare(Booking a, Booking b) {
        return a.getBookingDate().compareTo(b.getBookingDate());
    }
};

ArrayList<Booking> bookings = new ArrayList<>();
Collections.sort(bookings, byCost);   // Sort by cost
Collections.sort(bookings, byDate);   // Sort by date
In Nimbus

Nimbus uses Comparator<Booking> to sort bookings by cost, date, or status in the admin console. Without generics, the compare method would take Object parameters and you would need to cast inside every comparator — more code, more risk of ClassCastException.

Section 08

Generics in Nimbus

Every layer of Nimbus uses generics. Here is exactly where and why.

UserDAO — ArrayList<User>

UserDAO.java (simplified) 
public class UserDAO {

    public ArrayList<User> getAllUsers() throws SQLException {
        ArrayList<User> users = new ArrayList<>();
        // ... JDBC query ...
        while (rs.next()) {
            User u = new User();
            u.setName(rs.getString("name"));
            u.setEmail(rs.getString("email"));
            users.add(u);  // Only User objects allowed
        }
        return users;
    }
}

The return type ArrayList<User> guarantees that callers get a list of User objects. No casting needed on the receiving end.

BookingService — HashMap<String, Booking>

BookingService.java (simplified) 
public class BookingService {

    // Key = tracking ID (String), Value = Booking object
    private HashMap<String, Booking> activeBookings = new HashMap<>();

    public void trackBooking(Booking booking) {
        activeBookings.put(booking.getTrackingId(), booking);
    }

    public Booking findByTrackingId(String trackingId) {
        return activeBookings.get(trackingId);  // Returns Booking, not Object
    }
}

Sorting Bookings — Comparator<Booking>

BookingReports.java (simplified) 
public class BookingReports {

    public ArrayList<Booking> sortByCost(ArrayList<Booking> bookings) {
        Comparator<Booking> byCost = new Comparator<Booking>() {
            @Override
            public int compare(Booking a, Booking b) {
                return Double.compare(a.getCost(), b.getCost());
            }
        };
        Collections.sort(bookings, byCost);
        return bookings;
    }
}
In Nimbus

Generics are not something you "add" to Nimbus — they are already woven into every data structure. ArrayList<User> in the DAO layer, HashMap<String, Booking> in the service layer, Comparator<Booking> in the reports layer. Removing generics would mean adding casts everywhere and losing all compile-time safety.

Section 09

Why You Cannot Use Primitives

Generics only work with reference types (objects). Primitive types like int, double, boolean, char are not objects, so they cannot be used as type parameters.

PrimitivesFail.java 
// These will NOT compile
ArrayList<int> numbers = new ArrayList<>();       // ERROR
HashMap<int, double> map = new HashMap<>();       // ERROR
Box<boolean> flag = new Box<>();                  // ERROR

// Use wrapper classes instead
ArrayList<Integer> numbers = new ArrayList<>();   // OK
HashMap<Integer, Double> map = new HashMap<>();   // OK
Box<Boolean> flag = new Box<>();                  // OK

Wrapper Classes

Each primitive has a corresponding wrapper class that is a real object:

Primitive Wrapper Class Example
int Integer ArrayList<Integer>
double Double ArrayList<Double>
boolean Boolean ArrayList<Boolean>
char Character ArrayList<Character>
long Long ArrayList<Long>
float Float ArrayList<Float>

Autoboxing and Unboxing

Java automatically converts between primitives and wrappers. This is called autoboxing (primitive → wrapper) and unboxing (wrapper → primitive).

Autoboxing.java 
ArrayList<Integer> numbers = new ArrayList<>();

// Autoboxing: int 42 → Integer.valueOf(42)
numbers.add(42);

// Unboxing: Integer → int
int first = numbers.get(0);

// Behind the scenes, Java did:
// numbers.add(Integer.valueOf(42));
// int first = numbers.get(0).intValue();
Performance Warning

Autoboxing creates objects. If you are doing millions of operations, ArrayList<Integer> is significantly slower than a primitive int[] array. For Nimbus-scale data (hundreds of bookings), this does not matter. For high-performance computing, it does.

Section 10

Common Mistakes

These are the errors every Java developer hits when learning generics. Know them so you do not repeat them.

Mistake 1: Using Raw Types

Wrong
// Raw type — no type parameter
ArrayList users = new ArrayList();
users.add(new User("Darshan"));
users.add("Not a user");  // No error!

User u = (User) users.get(1);
// ClassCastException at runtime!
Right
// Parameterized type
ArrayList<User> users = new ArrayList<>();
users.add(new User("Darshan"));
users.add("Not a user");
// COMPILE ERROR — caught immediately

User u = users.get(0);  // No cast

Mistake 2: Using ArrayList<int>

Wrong
ArrayList<int> ids = new ArrayList<>();
// COMPILE ERROR: type argument
// cannot be a primitive type
Right
ArrayList<Integer> ids = new ArrayList<>();
ids.add(101);  // Autoboxing handles it

Mistake 3: Creating Instances of Type Parameters

Wrong
public <T> T create() {
    return new T();  // ERROR!
    // T is erased at runtime
}
Right
// Pass the class object
public <T> T create(Class<T> type)
    throws Exception {
    return type.getDeclaredConstructor()
               .newInstance();
}

Mistake 4: Assuming Generic Subtyping

Wrong Assumption
// String extends Object, so...
ArrayList<Object> list
    = new ArrayList<String>();
// COMPILE ERROR!
// Generic types are invariant
Right Approach
// Use wildcards instead
ArrayList<? extends Object> list
    = new ArrayList<String>();
// OK — wildcard allows it

Mistake 5: Overloading with Different Generic Types

Wrong
// These two methods have the SAME
// erasure (both become ArrayList)
void process(ArrayList<User> u) {}
void process(ArrayList<Booking> b) {}
// COMPILE ERROR: duplicate method
Right
// Give them different names
void processUsers(ArrayList<User> u) {}
void processBookings(
    ArrayList<Booking> b) {}
// No ambiguity
Section 11

Practice Exercises

Write these yourself. Do not copy-paste. The goal is muscle memory.

Exercise 1: Generic Stack

Create a generic class Stack<T> that uses an ArrayList<T> internally. Implement push(T item), pop() (returns and removes top), peek() (returns top without removing), and isEmpty(). Test it with both Stack<String> and Stack<Integer>.

Solution
1
public class Stack<T> {
    private ArrayList<T> items = new ArrayList<>();

    public void push(T item) {
        items.add(item);
    }

    public T pop() {
        if (isEmpty()) throw new RuntimeException("Stack is empty");
        return items.remove(items.size() - 1);
    }

    public T peek() {
        if (isEmpty()) throw new RuntimeException("Stack is empty");
        return items.get(items.size() - 1);
    }

    public boolean isEmpty() {
        return items.isEmpty();
    }
}

// Test
Stack<String> names = new Stack<>();
names.push("Darshan");
names.push("Nimbus");
System.out.println(names.pop());   // Nimbus
System.out.println(names.peek());  // Darshan
The class uses T everywhere — the internal list, the push parameter, and the pop/peek return types. One class works for any type.

Exercise 2: Generic findAll Method

Write a generic method <T> ArrayList<T> findAll(ArrayList<T> list, T target) that returns a new list containing all elements that are .equals() to the target. Test with ArrayList<String> and ArrayList<Integer>.

Solution
1
public static <T> ArrayList<T> findAll(ArrayList<T> list, T target) {
    ArrayList<T> result = new ArrayList<>();
    for (T item : list) {
        if (item.equals(target)) {
            result.add(item);
        }
    }
    return result;
}

// Test
ArrayList<String> words = new ArrayList<>(List.of("hi", "hello", "hi", "hey", "hi"));
ArrayList<String> matches = findAll(words, "hi");
System.out.println(matches.size());  // 3
Java infers T = String from the arguments. The method is type-safe: you cannot pass a list of Strings and search for an Integer.

Exercise 3: Bounded Type — findMin

Write a method <T extends Comparable<T>> T findMin(ArrayList<T> list) that returns the smallest element. Test with a list of Integer and a list of String.

Solution
1
public static <T extends Comparable<T>> T findMin(ArrayList<T> list) {
    if (list.isEmpty()) throw new RuntimeException("List is empty");
    T min = list.get(0);
    for (T item : list) {
        if (item.compareTo(min) < 0) {
            min = item;
        }
    }
    return min;
}

ArrayList<Integer> nums = new ArrayList<>(List.of(8, 3, 12, 1, 5));
System.out.println(findMin(nums));  // 1

ArrayList<String> names = new ArrayList<>(List.of("Nimbus", "Alpha", "Zeta"));
System.out.println(findMin(names));  // Alpha
The bound extends Comparable<T> is required so we can call compareTo. Without it, Java does not know T has that method.

Exercise 4: Generic Pair with Swap

Create a Pair<A, B> class with getFirst(), getSecond(), and a method swap() that returns a new Pair<B, A> with the values reversed. Note the return type carefully.

Solution
1
public class Pair<A, B> {
    private A first;
    private B second;

    public Pair(A first, B second) {
        this.first = first;
        this.second = second;
    }

    public A getFirst()  { return first; }
    public B getSecond() { return second; }

    public Pair<B, A> swap() {
        return new Pair<>(second, first);
    }
}

Pair<String, Integer> p1 = new Pair<>("Darshan", 24);
Pair<Integer, String> p2 = p1.swap();
System.out.println(p2.getFirst());   // 24
System.out.println(p2.getSecond());  // Darshan
The swap() method returns Pair<B, A> — the type parameters are reversed. The compiler tracks this, so p2.getFirst() returns Integer and p2.getSecond() returns String.

Exercise 5: Wildcard Method — Count Elements

Write a method int countElements(ArrayList<?> list) that returns the number of non-null elements. It should accept ArrayList of ANY type. Think about which wildcard to use and why.

Solution
1
public static int countElements(ArrayList<?> list) {
    int count = 0;
    for (Object item : list) {
        if (item != null) {
            count++;
        }
    }
    return count;
}

ArrayList<String> names = new ArrayList<>(List.of("A", "B"));
ArrayList<User> users = new ArrayList<>();
System.out.println(countElements(names));  // 2
System.out.println(countElements(users));  // 0
Unbounded wildcard <?> is correct here because we do not need to know the type — we only need to check for null. We read items as Object since that is all we need.

Exercise 6: Nimbus — Generic DAO Interface

Design a generic interface DAO<T> with methods: void save(T entity), T findById(int id), ArrayList<T> findAll(), void delete(int id). Then write class headers (no implementation) for UserDAO implements DAO<User> and BookingDAO implements DAO<Booking>.

1
public interface DAO<T> {
    void save(T entity) throws SQLException;
    T findById(int id) throws SQLException;
    ArrayList<T> findAll() throws SQLException;
    void delete(int id) throws SQLException;
}

public class UserDAO implements DAO<User> {

    public ArrayList<User> findAll() throws SQLException { /* ... */ }
    @Override
    public void delete(int id) throws SQLException { /* ... */ }
}

public class BookingDAO implements DAO<Booking> {

One interface, two implementations. When UserDAO says implements DAO<User>, every T in the interface becomes User. The compiler enforces that findById returns a User, not an Object. This is the real power of generic interfaces.
