Section 01

What is Software Engineering?

Before writing a single line of Java, you need to understand how professional software is built. Software engineering is not just coding — it is the discipline of building software that works, scales, and can be maintained by a team over months or years.

Building Software vs Writing Code

Many people think software engineering is just "writing code." That is like saying building a house is just "stacking bricks." Sure, bricks are involved. But if you just start stacking bricks without a plan, without a foundation, without plumbing and electrical design — your "house" will collapse the first time it rains.

Writing code means you open an editor, type some logic, run it, and it works. Great. That is programming. You solved a problem.

Software engineering means you figure out what problem to solve (requirements), plan how to solve it (design), build it in an organized way (implementation), make sure it actually works (testing), put it somewhere people can use it (deployment), and keep fixing things as the world changes (maintenance).

Here is the core difference:

Just Coding Software Engineering
One person writes code A team collaborates on code
Works on your machine Works everywhere, for everyone
You remember how it works Anyone can read and understand it
Fix bugs when you notice them Automated tests catch bugs before users do
One file with everything Organized packages, classes, layers
"It works, ship it" "It works, it's tested, it's documented, ship it"
In Nimbus

Nimbus started as "just code" in Phase 1 — HTML files with inline JavaScript, LocalStorage as the database. That was fine for a prototype. But now, in Phase 2+, you are doing real software engineering: separating into packages (com.nimbus.model, com.nimbus.dao, com.nimbus.service), designing database schemas, writing reusable classes, and building in layers. That transition — from "it works" to "it's engineered" — is exactly what this section is about.

Why Processes Matter

Imagine 8 people trying to cook dinner together in a kitchen. No recipe. No one knows who is making what. Two people are both boiling rice. No one is making the curry. Someone accidentally uses sugar instead of salt because nothing is labeled.

That is what happens when a software team writes code without a process. You get:

  • Duplicate work — two people build the same feature
  • Conflicting code — person A assumes the user ID is an integer, person B assumes it is a string
  • Missing features — everyone assumed someone else was building it
  • Bugs that take hours to find — because code is tangled together
  • Impossible to change — touching one thing breaks five other things

Processes are the recipe. They tell each team member what to build, when to build it, how to build it, and how to check if it is correct. Processes are not bureaucracy — they are survival.

In Nimbus

Your team of 8 freshers is building Nimbus together. Without a process, person A might create a User class with username while person B creates one with userId. Your team avoids this by following the Programming File Structure document — it defines exactly what each class looks like, what fields it has, and who builds what. That document IS your process.

In software, this is called refactoring — rewriting code because the original design was wrong. Good engineering minimizes how much refactoring you need.

Section 02

SDLC — Software Development Life Cycle

The SDLC is the step-by-step journey every piece of software goes through from idea to retirement. Think of it as the complete lifecycle of a building, from the first conversation with the architect to eventually demolishing it decades later.

Every software product — Instagram, Google Maps, Nimbus — goes through these six phases. The only difference is how fast they move through them and whether they do them in order (Waterfall) or in loops (Agile).

   Requirements ──> Design ──> Implementation ──> Testing ──> Deployment ──> Maintenance
       |                                                                        |
       |                                                                        |
       +────────────────────── feedback loop ──────────────────────────────────+
The six phases of the Software Development Life Cycle

Phase 1: Requirements Gathering

Analogy: You are planning a wedding. Before booking a venue, a caterer, or a DJ, you ask: "How many guests? Vegetarian or non-veg? Indoor or outdoor? What is the budget?" If you skip this step, you might book a venue for 50 people when 300 are coming.

In software, requirements gathering means sitting down with the people who will USE the software and asking: "What do you need this to do? Who will use it? How many users? What are the most important features?"

There are two types of requirements:

  • Functional Requirements — What the system DOES. "Users can create bookings." "Officers can update delivery status." "System calculates shipping cost based on weight."
  • Non-Functional Requirements — How well the system does it. "The system should handle 1000 simultaneous users." "Pages should load in under 2 seconds." "Data must be encrypted."
In Nimbus

Your requirements came from the TCS PBL sprint documents (.xlsx files). Each sprint defines User Stories like US1 (Main Menu), US2 (Registration), US3 (Login), and so on. These user stories ARE your requirements. When the document says "Customer can track parcels by booking ID," that is a functional requirement. When it says "The application should work in Eclipse," that is a non-functional requirement.

Phase 2: Design

Analogy: The architect draws blueprints. Not one blueprint — multiple. A floor plan (top-down view), an elevation (front view), plumbing diagrams, wiring diagrams, structural diagrams. Each shows the same house from a different angle. Together, they give the builder complete information about what to construct.

In software design, you create similar documents:

Design Artifact What It Shows House Equivalent
High-Level Architecture (HLA) Major components and how they connect Floor plan — shows rooms and hallways
Low-Level Architecture (LLA) Detailed class design, methods, data flow Plumbing/wiring diagrams — internal details
Database Design Tables, columns, relationships Storage plan — where things go
Flow Diagrams Step-by-step logic for key operations Fire escape route — what happens in each situation
In Nimbus

You already have these design documents in the docs/ folder: High_Level_Architecture.md, Low_Level_Architecture.md, DB_Design_Document.md, Flow_Diagrams.md, Logical_Connection_Map.md. These were created BEFORE writing Java code. That is professional software engineering — design first, code second.

Phase 3: Implementation (Coding)

Analogy: The construction workers arrive. They follow the blueprints. The plumber does plumbing, the electrician does wiring, the mason does walls. Each specialist works on their part, following the architectural plan.

This is the phase most people think IS software engineering. But as you can see, it is just one of six phases. Implementation means writing the actual code — creating classes, methods, database queries, and connecting everything together.

Key principles during implementation:

  • Follow the design — do not invent new classes that are not in the design document
  • Write clean code — meaningful variable names, comments where needed, consistent formatting
  • Build in layers — database layer first, then service layer, then UI layer
  • Test as you go — do not write 500 lines and then try to run it
In Nimbus

Implementation is split across phases. Phase 1 = HTML/CSS/JS frontend. Phase 2 = Java backend with JDBC. Phase 3 = Servlets/JSP. Phase 4 = REST APIs. Each phase implements more of the design. Your team of 8 splits work using the Team_Task_Breakdown.md document — each person implements specific classes and features.

Phase 4: Testing

Analogy: Before the family moves into their new house, an inspector checks everything. Turn on every faucet — does water flow? Flip every switch — do lights turn on? Open every door — does it lock? Check the foundation — any cracks? You do NOT move in and THEN discover the bathroom floods.

In software, testing means systematically checking that every feature works correctly. This includes:

  • Does the login work with correct credentials?
  • Does the login FAIL with wrong credentials? (This is equally important!)
  • What happens if someone enters a negative weight for a parcel?
  • What happens if the database is down?
  • Can a customer see another customer's bookings? (Security testing)
In Nimbus

In Phase 1, testing was manual — you opened the browser, clicked buttons, and checked if things worked. In Phase 2+, you will write JUnit test cases — code that automatically tests your code. For example, a test that creates a booking with weight 2.5 kg and checks if the price is calculated correctly. We will cover JUnit in detail in Section 8.

Phase 5: Deployment

Analogy: The house is built. Inspected. Approved. The family gets the keys and moves in. But the builder does not just dump them at the door. They do a walkthrough — "Here is how the heating works. Here is where the breaker panel is. Call us if anything breaks."

Deployment means putting your software where users can actually use it. For a web app, that means uploading it to a server. For a mobile app, that means publishing it to the App Store or Play Store. For a console application like Nimbus Phase 2, it means packaging the .jar file so it can run on any machine with Java installed.

In Nimbus

Phase 1 (frontend) was "deployed" by simply opening login.html in a browser — static files need no server. Phase 2 (Java console app) runs inside Eclipse. In Phase 3, you will deploy to a Tomcat server. In Phase 4, your REST API will be a deployable .war or .jar file. Each phase teaches a different deployment model.

Phase 6: Maintenance

Analogy: Three months after moving in, a tap starts leaking. A year later, the family wants to add a balcony. Two years later, the paint is peeling and needs redoing. The house is NEVER "done" — it needs ongoing care.

Software maintenance is the longest phase. It includes:

  • Bug fixes — users find problems you missed during testing
  • Feature additions — users want new capabilities
  • Security patches — new vulnerabilities are discovered
  • Performance improvements — system slows down as more users join
  • Technology updates — libraries get updated, old ones get deprecated

Here is a shocking statistic: 60-80% of a software project's total cost is maintenance. That means if building the software costs 20 lakh, maintaining it over its lifetime costs 60-80 lakh. This is why writing clean, well-structured code matters — because someone (maybe you) will be reading and changing it for years.

In Nimbus

Even within the project timeline, you are already doing maintenance. When the pricing formula in Phase 1 (JavaScript) needs to match the formula in Phase 2 (Java), that is maintenance. When you discover that the user ID generation has a bug with same-day registrations, fixing it across all files is maintenance. Good engineering makes maintenance easier.

Waterfall vs Agile

Now that you know the 6 phases, the big question is: in what order do you do them?

Waterfall Model

Analogy: A waterfall flows in one direction — down. Water does not flow back up. Similarly, in Waterfall, you complete each phase 100% before moving to the next. No going back.

  WATERFALL MODEL
  ===============

  Requirements  ====>  DONE (sign off, move on)
                           |
                           v
                       Design  ====>  DONE (sign off, move on)
                                          |
                                          v
                                      Implementation  ====>  DONE
                                                                |
                                                                v
                                                            Testing  ====>  DONE
                                                                               |
                                                                               v
                                                                           Deployment
                                                                               |
                                                                               v
                                                                           Maintenance

  Problem: What if you discover during Testing that
  the Requirements were wrong? You have to go ALL
  the way back. Expensive. Time-consuming. Painful.
Waterfall: each phase completes fully before the next begins

When Waterfall works: Building a bridge. Launching a satellite. You CANNOT go back and change the foundation of a bridge after it is built. Requirements must be 100% correct upfront.

When Waterfall fails: Most software projects. Because users change their minds. Markets shift. You discover problems late. By the time you finish, the requirements are outdated.

Agile Model

Analogy: Instead of building the entire house at once, you build one room at a time. Build the kitchen. Let the family use it. Get feedback. "Oh, we need more counter space." Adjust. Now build the bedroom. Use feedback. Adjust. Each room is a small, complete, usable unit.

  AGILE MODEL
  ===========

  Sprint 1 (2 weeks):   Requirements > Design > Code > Test > Deploy  =>  Working Login
                                                                             |
                                                                             v  (feedback)
  Sprint 2 (2 weeks):   Requirements > Design > Code > Test > Deploy  =>  Working Booking
                                                                             |
                                                                             v  (feedback)
  Sprint 3 (2 weeks):   Requirements > Design > Code > Test > Deploy  =>  Working Tracking
                                                                             |
                                                                             v  (feedback)
  Sprint 4 (2 weeks):   Requirements > Design > Code > Test > Deploy  =>  Full System

  Each sprint delivers something USABLE.
  Feedback from each sprint improves the next.
Agile: small cycles, each delivering working software
Aspect Waterfall Agile
Planning Everything planned upfront Plan one sprint at a time
Feedback Only at the end After every 2-week sprint
Changes Expensive and difficult Welcome and expected
Working software Only at the very end After every sprint
Risk High — problems found late Low — problems found early
Documentation Heavy (every detail documented first) Light (just enough to build)
Best for Fixed requirements (military, medical devices) Evolving requirements (apps, websites, SaaS)
In Nimbus

Nimbus uses a hybrid approach. Your TCS PBL structure is like Waterfall — Sprint 1 (UI), Sprint 2 (JDBC), Sprint 3 (Servlets), Sprint 4 (APIs) must be completed in order. But within each sprint, you work in an Agile way — build one feature, test it, move to the next. The sprint documents define WHAT to build, but your team decides HOW to break it into smaller pieces and what order to tackle them.

Agile & Scrum

Agile is the philosophy. Scrum is the framework that implements it. Think of Agile as "I believe in eating healthy" and Scrum as "Here is my exact meal plan for the week."

Scrum Framework Overview

Scrum breaks work into sprints — fixed time periods (usually 2 weeks) where the team builds a specific set of features. At the end of each sprint, you have working software that you can show to someone.

  ===============

  | - Support      |                                                             v
  | - Reports      |       <========= feedback from review goes back ============+
  +----------------+
How Scrum flows: Backlog to Sprint to Working Software to Feedback

Key Scrum Concepts

1. Sprint

A sprint is a fixed time period (1-4 weeks, usually 2 weeks) where the team picks a set of features from the backlog and builds them. The sprint has a clear goal ("By end of this sprint, users can log in and register") and a fixed deadline. You do NOT add more work mid-sprint. If something new comes up, it goes into the backlog for the NEXT sprint.

In Nimbus

Each TCS PBL phase maps to a sprint. Phase 1 sprint goal: "Working frontend with all pages and LocalStorage." Phase 2 sprint goal: "Java console app with JDBC database connection." Each phase has a clear deliverable and deadline.

2. Product Backlog

The product backlog is the master list of EVERYTHING the software needs to do. It is ordered by priority. The most important features are at the top. It is a living document — new items get added, old ones get removed or reprioritized.

In Nimbus

Your product backlog is all the User Stories across all sprints: US1 (Main Menu), US2 (Registration), US3 (Login), US4 (Customer Menu), US5 (Officer Menu), US6 (Booking), US7 (Payment), US8 (Invoice), US9-US10 (Tracking), US11-US12 (History), US13-US14 (Support). All of these, across all 4 phases, form the complete product backlog.

3. Sprint Backlog

The sprint backlog is the subset of the product backlog that the team commits to completing in THIS sprint. Once the sprint starts, this list is locked. No adding, no removing. Focus on what you committed to.

4. User Stories

A user story is a feature described from the user's perspective. It follows a specific format:

5. Daily Standup

A daily standup (also called "daily scrum") is a 15-minute meeting every morning where each team member answers three questions:

  1. What did I do yesterday?
  2. What will I do today?
  3. Are there any blockers? (things stopping me from working)

The standup is NOT a status report to the boss. It is a coordination meeting among the team. If person A says "I'm blocked because the database is not set up yet," person B might say "I can help with that today."

In Nimbus

Even if your team does not have formal daily standups, the concept applies. When you share updates in your WhatsApp group — "I finished the User model, starting on BookingService today, but I'm stuck on the pricing formula" — that IS a standup. Three questions: done, doing, blocked.

6. Sprint Review / Demo

At the end of each sprint, the team demos what they built. They show working software to the stakeholders (in your case, the TCS mentors). The stakeholders give feedback, which goes into the backlog for the next sprint.

In Nimbus

Each TCS PBL phase ends with a review. Phase 1 review: demo the working HTML/CSS/JS frontend with all pages. Phase 2 review: demo the Java console app with database operations. The mentors' feedback becomes your improvement backlog for the next phase.

Section 04

Design Principles — SOLID

SOLID is an acronym for five design principles that make software easier to understand, maintain, and extend. These principles were identified by Robert C. Martin ("Uncle Bob") and are considered fundamental to good object-oriented design. If you follow them, your code will not turn into a tangled mess as it grows.

Analogy: Think of SOLID principles like the rules of good cooking. You CAN throw all ingredients into one pot and heat it. The food might be edible. But if you follow rules — prep vegetables separately, season at the right time, use the right heat — you get a dish that tastes great, looks great, and can be reproduced consistently. SOLID principles are the "rules of good cooking" for code.

S — Single Responsibility Principle (SRP)

Definition

A class should have one, and only one, reason to change.

In simpler terms: each class should do ONE thing. If a class handles user login AND sends emails AND calculates pricing AND writes to the database, it is doing four things. When you need to change the pricing formula, you are editing the same class that handles login. You might accidentally break login while fixing pricing. That is a nightmare.

Analogy: In a restaurant, the chef cooks. The waiter serves. The cashier handles payment. The dishwasher cleans. If one person does all four jobs, the restaurant is chaos. If the cashier is also cooking, they cannot give change to a customer while a dish is burning. Each person has a SINGLE responsibility.

The Wrong Way

UserManager.java (BAD - does too many things) 
// BAD: This class has FOUR responsibilities
public class UserManager {

    // Responsibility 1: Database operations
    public void saveUserToDatabase(User user) {
        Connection conn = DriverManager.getConnection(DB_URL);
        PreparedStatement ps = conn.prepareStatement(
            "INSERT INTO users VALUES (?, ?, ?, ?)"
        );
        ps.setString(1, user.getUserId());
        ps.setString(2, user.getName());
        // ... execute
    }

    // Responsibility 2: Validation
    public boolean validateEmail(String email) {
        return email.matches("^[\\w.-]+@[\\w.-]+\\.\\w{2,}$");
    }

    // Responsibility 3: Password hashing
    public String hashPassword(String password) {
        return BCrypt.hashpw(password, BCrypt.gensalt());
    }

    // Responsibility 4: Sending welcome emails
    public void sendWelcomeEmail(String email) {
        // SMTP setup, compose email, send...
    }
}
// If the email provider changes, you edit UserManager.
// If the DB schema changes, you edit UserManager.
// If validation rules change, you edit UserManager.
// ONE class changing for FOUR different reasons = violation of SRP.

The Right Way

Separated classes (GOOD - each does ONE thing) 
// GOOD: Each class has ONE responsibility

// Class 1: Only handles database operations for users
public class UserDAO {
    public void save(User user) { /* DB insert */ }
    public User findById(String userId) { /* DB query */ }
    public void update(User user) { /* DB update */ }
    public void delete(String userId) { /* DB delete */ }
}

// Class 2: Only handles input validation
public class InputValidator {
    public boolean isValidEmail(String email) { /* regex check */ }
    public boolean isValidPhone(String phone) { /* format check */ }
    public boolean isValidPassword(String pwd) { /* strength check */ }
}

// Class 3: Only handles password security
public class PasswordService {
    public String hash(String password) { /* hashing logic */ }
    public boolean verify(String plain, String hashed) { /* comparison */ }
}

// Class 4: Only handles email sending
public class EmailService {
    public void sendWelcomeEmail(String email) { /* email logic */ }
    public void sendPasswordReset(String email) { /* email logic */ }
}

// Now if the email provider changes, ONLY EmailService changes.
// Database schema changes? ONLY UserDAO changes.
// Validation rules change? ONLY InputValidator changes.
// Each class has ONE reason to change.
In Nimbus

Nimbus follows SRP with its package structure. UserDAO in com.nimbus.dao ONLY handles user database operations — save(), findById(), findByEmail(), update(). It does not validate input (that is InputValidator's job). It does not calculate pricing (that is PricingService's job). It does not display menus (that is NimbusApp's job). Each class has exactly one reason to exist.

O — Open/Closed Principle (OCP)

Definition

Software entities should be open for extension, but closed for modification.

This means: you should be able to ADD new behavior without CHANGING existing code. Why? Because every time you modify existing code, you risk breaking things that already work. If you can add new features by writing NEW code instead of changing old code, existing features stay safe.

Analogy: Think of a power strip. When you need to plug in a new device, you do not rewire the power strip. You just plug into an available socket. The power strip is CLOSED for modification (you do not change its internals) but OPEN for extension (you can plug in new devices).

The Wrong Way

TrackingService.java (BAD - must modify for each new role) 
// BAD: Every time a new role is added, you MODIFY this class
public class TrackingService {

    public void showTracking(String role, String bookingId) {
        if (role.equals("customer")) {
            // Show limited info: status + estimated delivery
            System.out.println("Status: " + getStatus(bookingId));
            System.out.println("ETA: " + getETA(bookingId));
        }
        else if (role.equals("officer")) {
            // Show detailed info: status + location + route + driver
            System.out.println("Status: " + getStatus(bookingId));
            System.out.println("Location: " + getLocation(bookingId));
            System.out.println("Route: " + getRoute(bookingId));
            System.out.println("Driver: " + getDriver(bookingId));
        }
        // What if we add "admin" role? Or "warehouse" role?
        // We have to come back here and ADD more if-else blocks.
        // This class is MODIFIED every time we extend the system.
    }
}

The Right Way

TrackingService.java (GOOD - extend without modifying) 
// GOOD: Define a base class/interface, extend with new classes

// Base class - CLOSED for modification
public abstract class TrackingService {
    public abstract void showTracking(String bookingId);
}

// Extension for Customer - no modification to TrackingService
public class CustomerTrackingService extends TrackingService {
    @Override
    public void showTracking(String bookingId) {
        System.out.println("Status: " + getStatus(bookingId));
        System.out.println("ETA: " + getETA(bookingId));
    }
}

// Extension for Officer - no modification to TrackingService
public class OfficerTrackingService extends TrackingService {
    @Override
    public void showTracking(String bookingId) {
        System.out.println("Status: " + getStatus(bookingId));
        System.out.println("Location: " + getLocation(bookingId));
        System.out.println("Route: " + getRoute(bookingId));
        System.out.println("Driver: " + getDriver(bookingId));
    }
}

// NEW: Add Admin role WITHOUT touching ANY existing code

Nimbus has three roles: Customer, Officer, and Support. Each role sees different information when tracking a parcel. L — Liskov Substitution Principle (LSP)

Definition

Objects of a superclass should be replaceable with objects of a subclass without breaking the program.

In simpler terms: if your code works with a TrackingService, it should work equally well with CustomerTrackingService or OfficerTrackingService — without the code even knowing which one it is using.

Analogy: A TV remote control works with any TV — Samsung, LG, Sony. If you replace your Samsung TV with an LG TV, the remote still works the same way. The "interface" (power button, volume button, channel button) is the same. The internal implementation is different, but the user does not care. If a new TV required you to press power twice because it works differently, that VIOLATES the Liskov principle — the replacement is not seamless.

The Wrong Way

Violation of LSP 
// BAD: Subclass changes the behavior in an unexpected way
public class TrackingService {
    public String getStatus(String bookingId) {
        // Returns status like "In Transit", "Delivered", etc.
        return bookingDAO.getStatus(bookingId);
    }
}

public class GuestTrackingService extends TrackingService {

    }

// Code that uses TrackingService:

    String status = service.getStatus(id);  // CRASHES with GuestTrackingService!
    System.out.println("Your parcel is: " + status);
}

The Right Way

Correct LSP implementation

    public String getStatus(String bookingId) {
        return bookingDAO.getStatus(bookingId);
    }
}

public class CustomerTrackingService extends TrackingService {
    @Override
    public String getStatus(String bookingId) {

If a method accepts TrackingService as a parameter, you can pass CustomerTrackingService or OfficerTrackingService and it will work correctly. The calling code does not need to know which specific implementation it is talking to — it just calls getStatus() and gets a valid status string back.

I — Interface Segregation Principle (ISP)

Definition

No client should be forced to depend on methods it does not use.

In simpler terms: do not create one giant interface with 20 methods and force every class to implement all 20. Instead, create small, focused interfaces. A class can implement multiple small interfaces if it needs to.

Analogy: Imagine a Swiss Army Knife vs separate tools. A Swiss Army Knife has a blade, scissors, screwdriver, can opener, corkscrew, and magnifying glass. But if you just need to open a can, you are carrying 5 unnecessary tools. The Interface Segregation Principle says: give the can-opener user just a can opener, not the whole Swiss Army Knife.

The Wrong Way

Fat Interface (BAD) 
// BAD: One fat interface forces all implementations to have everything
public interface UserOperations {

    User login(String id, String password);
    void createBooking(Booking booking);

    // These make NO SENSE for a customer:
    public void updateDeliveryStatus(String id, String s) {
        throw new UnsupportedOperationException(); // Ugly!
    }
    public void assignIssue(String iid, String oid) {
        throw new UnsupportedOperationException(); // Ugly!

        throw new UnsupportedOperationException(); // Ugly!

    }

D — Dependency Inversion Principle (DIP)

Definition

High-level modules should not depend on low-level modules. Both should depend on abstractions.

In simpler terms: your business logic should not directly depend on a specific database, a specific email provider, or a specific file format. Instead, it should depend on an INTERFACE (abstraction), and the specific implementation can be swapped.

Analogy: When you plug your phone charger into a wall socket, your phone does not care whether the electricity comes from a coal plant, a nuclear plant, or a solar panel. Your phone depends on the SOCKET INTERFACE (220V, Type C plug). The power source can be changed without affecting your phone. If your phone was hardwired directly to a coal plant, you could never switch to solar.

The Wrong Way

Direct dependency (BAD) 
// BAD: BookingService directly depends on MySQLBookingDAO
public class BookingService {
    // Hardcoded dependency on MySQL implementation
    private MySQLBookingDAO bookingDAO = new MySQLBookingDAO();

    public void createBooking(Booking booking) {
        bookingDAO.save(booking);
    }
}

// Problem: What if you want to switch from MySQL to PostgreSQL?
// You have to CHANGE BookingService code.
// What if you want to use an in-memory store for testing?
// You have to CHANGE BookingService code.
// BookingService is TIGHTLY COUPLED to MySQLBookingDAO.

The Right Way

Depend on abstraction (GOOD) 
// GOOD: Define an interface (abstraction)
public interface BookingDAO {
    void save(Booking booking);
    Booking findById(String bookingId);
    List<Booking> findByUserId(String userId);
}

// Implementation 1: MySQL

}

    private List<Booking> bookings = new ArrayList<>();
    public void save(Booking b) { bookings.add(b); }
    public Booking findById(String id) { /* search list */ }
    public List<Booking> findByUserId(String uid) { /* filter list */ }
}

// BookingService depends on the INTERFACE, not the implementation
public class BookingService {
    private BookingDAO bookingDAO;  // Interface type!

    // Implementation is INJECTED from outside
    public BookingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;

In Phase 2 (Sprint 1), you use ArrayListBookingDAO (in-memory storage). In Phase 2 (Sprint 2), you swap to JDBCBookingDAO (MySQL). BookingService code does not change at all. That is the power of Dependency Inversion.

Section 05

Other Important Principles

Beyond SOLID, there are several widely-used principles that every developer should know. These are simpler than SOLID but equally important in day-to-day coding.

DRY — Don't Repeat Yourself

Definition: Every piece of knowledge should have a single, unambiguous, authoritative representation in the system.

In simpler terms: if you find yourself copy-pasting the same code in three places, STOP. Extract it into a method or class and call it from those three places. When you need to change that logic, you change it in ONE place instead of hunting for all the copies.

Analogy: Imagine you write your phone number on 10 different forms. If you change your phone number, you have to update all 10 forms. But if all 10 forms say "phone number: see your profile card," you only update the profile card once. DRY = write it once, reference it everywhere.

The Wrong Way

KISS — Keep It Simple, Stupid

Definition: The simplest solution that works is usually the best one.

Analogy: You need to hang a picture on the wall. You COULD use a laser level, stud finder, industrial-grade anchors, and a power drill. OR you could use a nail and a hammer. The picture hangs either way. Do not over-engineer.

In code, KISS means: do not write a 50-line algorithm when a 5-line one does the same thing. Do not use design patterns where simple code works. Do not create 10 classes when 3 would suffice.

KISS example 
// Over-engineered (violates KISS):
public class StatusMapper {
    private static final Map<Integer, String> STATUS_MAP = new HashMap<>();
    static {
        STATUS_MAP.put(1, "Booked");
        STATUS_MAP.put(2, "Picked Up");
        STATUS_MAP.put(3, "In Transit");
        STATUS_MAP.put(4, "Delivered");
    }
    public static String getStatusName(int code) {
        return STATUS_MAP.getOrDefault(code, "Unknown");
    }
}

// Simple and clear (follows KISS):
public static String getStatusName(int code) {
    switch (code) {
        case 1: return "Booked";
        case 2: return "Picked Up";
        case 3: return "In Transit";
        case 4: return "Delivered";
        default: return "Unknown";
    }
}
// Same result. Half the code. Instantly readable.
// Use the HashMap approach when you have 50+ statuses or load them from DB.
// For 4 statuses, a switch is perfectly fine.
In Nimbus

Nimbus has exactly 4 delivery statuses: Booked, Picked Up, In Transit, Delivered. A simple switch statement is KISS-compliant. You do not need an Enum, a StatusFactory, a StatusStrategy, and a StatusRegistry for 4 values. Use the simplest solution that works. When the system grows to 20 statuses pulled from a config file, THEN add complexity.

YAGNI — You Ain't Gonna Need It

Definition: Do not build features or abstractions until you actually need them.

Analogy: You are packing for a 2-day trip. Do you pack for a 2-day trip, or do you pack a suitcase with clothes for 14 days "just in case" you decide to stay longer? YAGNI says: pack for 2 days. If you decide to stay longer, you can buy clothes or do laundry. Do not carry the weight of imaginary future needs.

In code, YAGNI means: do not build a notification system before anyone has asked for notifications. Do not add multi-language support before anyone has asked for it. Do not build a plugin architecture when you have zero plugins. Each of these adds complexity NOW for a benefit that might NEVER come.

YAGNI example 
// YAGNI violation: Building for imaginary future needs
public class PricingService {
    private CurrencyConverter converter;        // "We might go international"
    private DiscountEngine discountEngine;      // "We might have coupons"
    private TaxCalculator taxCalculator;        // "We might add tax"
    private InsuranceCalculator insurance;      // "We might insure parcels"
    private LoyaltyPointsService loyalty;       // "We might have rewards"

    public double calculate(double weight, String type) {
        // 5 extra dependencies for features nobody asked for
        // Each one needs maintenance, testing, and documentation
        // None of them are in the requirements
    }
}

// YAGNI-compliant: Build what is needed NOW
public class PricingService {
    // Requirements say: base rate + weight charge + type multiplier
    public double calculate(double weight, String type, String packaging) {
        double baseCost = 50.0;
        double weightCharge = weight * 15.0;
        double typeMultiplier = getTypeMultiplier(type);
        double packagingCharge = getPackagingCharge(packaging);
        return (baseCost + weightCharge) * typeMultiplier + packagingCharge;
    }
    // Clean. Simple. Does exactly what the requirements ask for.
    // Add insurance WHEN (and IF) the client asks for it.
}
In Nimbus

The Nimbus pricing formula is: base rate + weight charge + type multiplier + packaging charge. That is what the TCS PBL requirements specify. You do NOT add currency conversion, discount codes, tax calculation, or insurance — because none of those are in the requirements. If Phase 4 adds a "discount code" feature, you add it THEN. Not now.

Separation of Concerns (SoC)

Definition: Different concerns (responsibilities) should be handled by different parts of the system.

Analogy: In a hospital, the reception handles patient registration. The lab handles blood tests. The pharmacy handles medicines. The surgeon handles operations. They are all part of the same hospital, but each department handles its own concern. The surgeon does not dispense medicines. The pharmacist does not do surgeries.

In software, separation of concerns means organizing code into layers where each layer handles one concern:

  NIMBUS PACKAGE STRUCTURE = SEPARATION OF CONCERNS
  ==================================================

  com.nimbus.model/          <-- CONCERN: Data shapes (what does a User look like?)
    User.java                    Just fields, constructors, getters, setters
    Booking.java                 No business logic. No DB code. No display code.

  com.nimbus.dao/            <-- CONCERN: Database operations (how to read/write data)
    UserDAO.java                 Only SQL queries. No business logic. No display code.
    BookingDAO.java              Does not know what a "menu" is. Does not validate input.

  com.nimbus.service/        <-- CONCERN: Business logic (rules and calculations)
    UserService.java             Login rules, registration rules
    BookingService.java          Pricing formula, status transitions
    PricingService.java          Cost calculation logic

  com.nimbus.util/           <-- CONCERN: Reusable utilities
    InputValidator.java          Email regex, phone format, password strength
    IdGenerator.java             User ID and Booking ID generation
    MenuDisplay.java             Formatted console output

  com.nimbus.app/            <-- CONCERN: User interaction (menus, input/output)
    NimbusApp.java               Scanner input, System.out.println output
                                 Does not know SQL. Does not know pricing formula.
Each package handles exactly one concern in Nimbus

Why does this matter? Because when the pricing formula changes, you ONLY look at PricingService.java. You do not search through 20 files. When the database switches from MySQL to PostgreSQL, you ONLY change files in the dao package. The service layer and the app layer do not even know the database changed.

In Nimbus

This is perhaps the most important principle for your team. In Phase 1 (JavaScript), everything was in one storage.js file — database operations, business logic, and utilities all mixed together. In Phase 2 (Java), the code is properly separated: model, dao, service, util, app. Each person on the team can work on their package without stepping on anyone else's toes. Person A works on UserDAO, person B works on BookingService, and they do not conflict because the concerns are separated.

Section 06

Design Patterns

Design patterns are proven solutions to common problems. They are not code you copy-paste — they are strategies and structures that experienced developers have found work well, documented, and shared with the community. Think of them as recipes in a cookbook. You do not invent how to make butter chicken every time — you follow a known recipe and adapt it.

MVC Pattern — Model-View-Controller

Analogy: Think of a restaurant again. The Model is the kitchen (data and recipes). The View is the dining table presentation (what the customer sees). The Controller is the waiter (takes the order from the customer, gives it to the kitchen, and brings the food back). The customer never goes into the kitchen. The chef never comes to the dining table. The waiter coordinates between them.

  |             |  <------ |                |  <------ |             |

Singleton Pattern

Analogy: India has ONE Prime Minister at a time. You cannot create a second Prime Minister. No matter how many people ask "who is the PM?", they all get the same person. That is the Singleton pattern — a class that guarantees only ONE instance exists.

When to use it: When you need exactly ONE of something in your entire application. Database connections are the classic example — you want ONE connection pool, not a new pool every time someone needs to run a query.

DBConnection.java (Singleton Pattern) 
public class DBConnection {
    // Step 1: Private static instance (the ONE object)
    private static DBConnection instance;
    private Connection connection;

    // Step 2: Private constructor (nobody can call "new DBConnection()")
    private DBConnection() {
        try {
            Class.forName("com.mysql.cj.jdbc.Driver");
            this.connection = DriverManager.getConnection(
                "jdbc:mysql://localhost:3306/nimbus_db",
                "root",
                "password"
            );
        } catch (Exception e) {
            e.printStackTrace();
        }
    }

    // Step 3: Public static method to get the ONE instance
    public static DBConnection getInstance() {
        if (instance == null) {
            instance = new DBConnection();  // Created ONLY once
        }
        return instance;
    }

    public Connection getConnection() {
        return this.connection;
    }
}

// Usage everywhere in the app:
Connection conn = DBConnection.getInstance().getConnection();
// No matter how many times you call getInstance(),
// you get the SAME DBConnection object.
// ONE connection. Shared across the entire application.
In Nimbus

DBConnection is the perfect Singleton candidate in Nimbus. You do not want UserDAO creating its own connection, BookingDAO creating another connection, and NimbusApp creating yet another. That wastes memory and can exceed MySQL's connection limit. One connection, shared by all DAOs. That is Singleton.

Version Control Concepts

Version control is the system that tracks every change made to your code, who made it, when, and why. Even if you cannot use Git at the TCS office, understanding version control is a must-know for any software engineer.

What is Git and Why It Matters

Analogy: Imagine writing an essay in a notebook. You write 3 pages. Then you realize page 2 is wrong and rewrite it. But now you cannot see what page 2 originally said. What if the original version was actually better?

Now imagine Google Docs. It saves every version automatically. You can go back to "April 15, 3:47 PM" and see exactly what the document looked like at that moment. You can compare any two versions. You can undo any change. Git does this for code.

Git is the world's most popular version control system. Here are the core concepts:

Repository (Repo)

A repository is a folder that Git tracks. Every file, every change, every version is stored in the repository's history. Nimbus is a repository.

Commit

A commit is a snapshot of your code at a specific point in time. It has a message describing what changed, a timestamp, and the author's name. Think of it as a save point in a video game — you can always go back to it.

Git commit examples 
// Each commit is a labeled save point:
//
// Commit 1: "Add User.java model with fields and getters"
//   - Created User.java with userId, firstName, lastName, etc.
//
// Commit 2: "Add UserDAO with save() and findById()"
//   - Created UserDAO.java
//   - Added JDBC code for INSERT and SELECT
//
// Commit 3: "Fix login bug: password comparison was case-sensitive"
//   - Changed line 47 in UserService.java
//   - equals() replaced with equalsIgnoreCase() -- wait, no,
//     passwords SHOULD be case-sensitive! Revert this commit.
//
// Commit 4: "Revert commit 3: passwords must be case-sensitive"
//   - Undone the change from commit 3
//
// Without Git, commit 3's mistake would be permanent.
// With Git, you go back to any previous state instantly.

Branch

A branch is a parallel version of your code. Think of it like a fork in the road. The main road (main branch) continues, and a side road (feature branch) goes off to build a specific feature. Once the feature is done, the side road merges back into the main road.

  =================

A merge combines two branches together. When your feature is done and tested, you merge it into the main branch. If two developers changed the SAME line in the SAME file, Git raises a merge conflict — a human must decide which version to keep.

Testing Concepts

Testing is how you prove your code works. Not by staring at it and thinking "this looks right" — but by running it with specific inputs and checking if the outputs match expectations. Professional software has automated tests that run in seconds and catch bugs before they reach users.

Why Testing Matters

Analogy: A car manufacturer does not ship a car after building it and saying "it looks fine." They crash-test it. They test the brakes at 200 km/h. They test the wipers in a rainstorm simulation. They test the airbags with dummies. They test the engine at extreme temperatures. They test everything that COULD go wrong BEFORE the customer drives off the lot.

Software testing is the same. You test every feature, every edge case, every error condition. The goal: catch bugs before users catch them.

Unit Testing

A unit test tests a single method in isolation. "Does PricingService.calculate(2.5, "standard", "basic") return 102.5?" That is a unit test. You are testing one UNIT (method) with specific inputs and checking the output.

Analogy: Testing one bulb in a string of lights. You take one bulb out, put it in a tester, and check if it lights up. You do not test the entire string — just one bulb. If it works, put it back. Move to the next bulb.

Integration Testing

An integration test tests multiple components working together. "When I call UserService.login(), does it correctly call UserDAO.findById(), get the user from the database, compare the password, and return the right result?" That is an integration test — multiple layers working together.

Analogy: Testing the entire string of lights. Plug it in and see if the whole string lights up. Individual bulbs might work, but the wiring between them might be bad. Integration testing checks the wiring.

  UNIT vs INTEGRATION TESTING
  ============================

  UNIT TEST (isolated):
  +--------------------+
  | PricingService     |
  |   calculate()      |   Input: weight=2.5, type="standard"
  |                    |   Expected output: 102.5
  +--------------------+   Does not touch DB. Does not touch UI.


  INTEGRATION TEST (connected):
  +------------+     +-----------+     +--------+
  | UserService| --> | UserDAO   | --> | MySQL  |
  | login()    |     | findById()|     | SELECT |
  +------------+     +-----------+     +--------+
                                           |
  Input: userId="260401-DM01", pwd="Test@123"
  Expected: User object with role="customer"
  Tests the ENTIRE chain from service to DB and back.
Unit tests check one piece. Integration tests check the whole flow.

Security Basics

Security is not a feature you add later — it is a way of thinking from day one. Every input is a potential attack. Every database query is a potential vulnerability. Understanding the basics protects your users and your reputation.

SQL Injection

What it is: SQL Injection is the #1 web vulnerability in the world. It happens when user input is directly inserted into a SQL query without protection. An attacker types SQL code into a login form, and your application executes it as if it were a legitimate query.

Analogy: Imagine a bank teller who follows instructions literally. You walk up and say "My name is Robert; DROP ALL ACCOUNTS." The teller writes your name on the form, but the system interprets "DROP ALL ACCOUNTS" as a command and deletes everything. That is SQL injection — the system cannot tell the difference between data and commands.

The Attack

SQL Injection Attack (VULNERABLE CODE) 
// VULNERABLE: User input directly concatenated into SQL
public User login(String userId, String password) {
    String sql = "SELECT * FROM users WHERE user_id = '" + userId +
                 "' AND password = '" + password + "'";
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery(sql);
    // ...
}

// Normal usage - works fine:
// userId = "260401-DM01"
// password = "Test@123"
// SQL becomes: SELECT * FROM users WHERE user_id = '260401-DM01'
//              AND password = 'Test@123'
// This is correct.

// ATTACK: Attacker types this as the password:
// password = "anything' OR '1'='1"
// SQL becomes: SELECT * FROM users WHERE user_id = '260401-DM01'
//              AND password = 'anything' OR '1'='1'
//
// Since '1'='1' is ALWAYS true, this returns ALL users!
// The attacker just logged in without knowing any password.

// WORSE ATTACK: Attacker types this as userId:
// userId = "'; DROP TABLE users; --"
// SQL becomes: SELECT * FROM users WHERE user_id = '';
//              DROP TABLE users; --' AND password = 'x'
//
// This DELETES the entire users table!
// The -- makes everything after it a comment (ignored).

The Defense: PreparedStatement

Safe code with PreparedStatement 
// SAFE: PreparedStatement separates SQL structure from data
public User login(String userId, String password) {
    String sql = "SELECT * FROM users WHERE user_id = ? AND password = ?";
    //                                               ^                ^
    //                                     placeholders (not user input)

    PreparedStatement ps = conn.prepareStatement(sql);
    ps.setString(1, userId);    // Safely inserts userId as DATA
    ps.setString(2, password);  // Safely inserts password as DATA
    ResultSet rs = ps.executeQuery();
    // ...
}

// Now if attacker types: password = "anything' OR '1'='1"
// PreparedStatement treats it as a LITERAL STRING, not SQL code.
// It searches for a password that literally equals: anything' OR '1'='1
// No results found. Attack fails.

// If attacker types: userId = "'; DROP TABLE users; --"
// PreparedStatement treats it as a literal user ID string.
// It searches for a user_id that literally equals: '; DROP TABLE users; --
// No results found. Table is safe. Attack fails.

// WHY IT WORKS:
// PreparedStatement sends the SQL structure and data SEPARATELY to the DB.
// The DB knows: "This is the query template. These are the data values."
// Data can NEVER be interpreted as SQL commands. Ever.
In Nimbus

Every SQL query in Nimbus uses PreparedStatement with ? placeholders. In UserDAO.findById(), BookingDAO.save(), BookingDAO.updateStatus() — every single query uses parameterized queries. This is a hard rule: NEVER concatenate user input into SQL strings. Always use ? placeholders and setString()/setInt()/setDouble().

Password Hashing

What it is: When a user registers with password "Test@123", you do NOT store "Test@123" in the database. If a hacker breaks into your database, they now have everyone's passwords. Instead, you store a hash — a one-way scrambled version of the password that cannot be reversed.

Analogy: A meat grinder. You put in a steak (password), and out comes ground meat (hash). You can easily go from steak to ground meat, but you CANNOT go from ground meat back to a steak. Even if someone steals the ground meat, they cannot reconstruct the original steak.

  PASSWORD HASHING
  =================

  Registration:
    User types:     "Test@123"
    Hash function:  bcrypt("Test@123")
    Stored in DB:   "$2a$10$N9qo8uLOickgx2ZMRZoMy.MQDq3lRzqM3Cs6vHh8pY3.T7YL7VqOi"

  Login:
    User types:     "Test@123"
    Hash function:  bcrypt("Test@123")
    Result:         "$2a$10$N9qo8uLOickgx2ZMRZoMy.MQDq3lRzqM3Cs6vHh8pY3.T7YL7VqOi"
    Compare:        stored hash == new hash?  YES --> login success!

  If hacker steals database:
    They see:       "$2a$10$N9qo8uLOickgx2ZMRZoMy.MQDq3lRzqM3Cs6vHh8pY3.T7YL7VqOi"
    They CANNOT reverse this to get "Test@123"
    The password is safe even though the database was compromised.
Hashing is a one-way function: easy to hash, impossible to reverse
Password hashing with bcrypt 
// Using jBCrypt library (add to Maven: org.mindrot:jbcrypt:0.4)

import org.mindrot.jbcrypt.BCrypt;

public class PasswordService {

    // During registration: hash the password before storing
    public String hashPassword(String plainPassword) {
        // BCrypt.gensalt() generates a random salt
        // Salt ensures same password produces different hashes each time
        return BCrypt.hashpw(plainPassword, BCrypt.gensalt());
    }

    // During login: compare plain password with stored hash
    public boolean verifyPassword(String plainPassword, String storedHash) {
        return BCrypt.checkpw(plainPassword, storedHash);
    }
}

// Usage:
PasswordService pwdService = new PasswordService();

// Registration:
String hash = pwdService.hashPassword("Test@123");
// hash = "$2a$10$N9qo8uLO..."  (60 characters, different each time!)
user.setPassword(hash);
userDAO.save(user);  // Store the HASH, not the plain password

// Login:
User user = userDAO.findById(userId);
boolean valid = pwdService.verifyPassword("Test@123", user.getPassword());
// valid = true if password matches, false if not

Authentication vs Authorization

These two words sound similar but mean very different things:

Authentication = WHO are you? Proving your identity. Login with user ID and password. If credentials match, you are authenticated — the system knows who you are.

Authorization = WHAT can you do? After knowing who you are, the system checks your permissions. A customer can create bookings but cannot update delivery status. An officer can update delivery status but cannot raise support issues.

Analogy: At an office building, the security guard checks your ID card (authentication — proving who you are). Then your ID card has a color code that determines which floors you can access (authorization — what you are allowed to do). The CEO's card opens all floors. The intern's card opens only floor 1.

  ==========================================

  Step 1: AUTHENTICATION (Login)
  +----------------------------------+

  |         NO  = "Invalid login"    |
  +----------------------------------+
              |
              v (authenticated)

  |                                  |

  Customer:  book, track, pay, history, support

  Support:   view issues, assign, resolve

In Nimbus

Authorization: After login, the system checks user.getRole() and shows the corresponding menu — Customer Menu (US4), Officer Menu (US5), or Support Menu. A customer cannot access officer functions because the menu does not show those options, AND the backend services check the role before executing operations. Both layers of protection matter.

Input Validation

What it is: Checking that user input is safe and valid BEFORE processing it. Never trust user input. Ever. Users make typos. Users enter garbage. And attackers deliberately send malicious data.

Analogy: Airport security. Every bag is scanned. Every person walks through a metal detector. You do not let people through and THEN check if they have something dangerous. You check BEFORE they enter.

There are two types of validation:

Type Where Why Both Matter
Client-side In the browser (JavaScript), before the form is submitted Good user experience — instant feedback ("Email format is wrong"). But can be bypassed by attackers (they can disable JavaScript or send requests directly).
Server-side In the Java backend, after the form is submitted Security — CANNOT be bypassed by the user. Even if client-side is disabled, server catches bad input. This is the REAL validation.

Rule: Client-side validation is for user convenience. Server-side validation is for security. You need BOTH. Never rely on only one.

InputValidator.java (Server-side validation) 
public class InputValidator {

    // Email: must match pattern user@domain.tld
    public boolean isValidEmail(String email) {
        if (email == null || email.isEmpty()) return false;
        String regex = "^[\\w.-]+@[\\w.-]+\\.\\w{2,}$";
        return email.matches(regex);
    }

    // Phone: country code + 10 digits (e.g., +919876543210)
    public boolean isValidPhone(String phone) {
        if (phone == null) return false;
        return phone.matches("^\\+\\d{1,3}\\d{10}$");
    }

    // Password: min 8 chars, at least 1 uppercase, 1 lowercase,
    //           1 digit, 1 special character
    public boolean isValidPassword(String password) {
        if (password == null || password.length() < 8) return false;
        boolean hasUpper = password.matches(".*[A-Z].*");
        boolean hasLower = password.matches(".*[a-z].*");
        boolean hasDigit = password.matches(".*\\d.*");
        boolean hasSpecial = password.matches(".*[!@#$%^&*()].*");
        return hasUpper && hasLower && hasDigit && hasSpecial;
    }

    // Name: 1-50 characters, letters and spaces only
    public boolean isValidName(String name) {
        if (name == null || name.isEmpty()) return false;
        return name.length() <= 50 && name.matches("^[a-zA-Z ]+$");
    }

    // Weight: must be positive number
    public boolean isValidWeight(double weight) {
        return weight > 0 && weight <= 100; // max 100 kg
    }
}
In Nimbus

Current state: Phase 1 has client-side validation in JavaScript (email regex, phone format, password strength checks in the HTML forms). Phase 2 must duplicate ALL of these checks in the Java InputValidator class. Why duplicate? Because in Phase 1 (browser-only), JavaScript is the only option. In Phase 2+ (with a backend), an attacker could bypass the browser entirely and send a malicious request directly to the server. The server-side InputValidator is the last line of defense. What should improve: Add length limits on ALL text inputs to prevent buffer overflow attacks. Add sanitization to strip HTML/script tags from names and addresses.

Remember

Security is not about making things unhackable — nothing is. It is about making attacks so difficult and time-consuming that attackers give up and move to easier targets. PreparedStatement, bcrypt, input validation, and role-based access — these four alone eliminate 90% of common web application vulnerabilities. Learn them. Use them. Every time. No exceptions.
