Section 01

What Are Exceptions?

When something goes wrong while your program is running, Java does not just crash silently. It creates an exception object that describes exactly what went wrong, where it happened, and why. Your job as a developer is to handle these exceptions so the program can recover gracefully instead of dying.

Error vs Exception

People often use "error" and "exception" interchangeably. In Java, they mean very different things.

An Error is something catastrophic that your program cannot recover from. The JVM itself is in trouble. You ran out of memory. The stack overflowed. A class file is corrupted. There is nothing your code can do about it — the JVM is dying.

An Exception is something that went wrong during normal operation, but your program can recover from it. A file was not found. A database connection failed. A user entered "abc" when you expected a number. These are situations you can anticipate and handle.

Error Exception
JVM-level problem Application-level problem
Cannot recover Can recover
OutOfMemoryError, StackOverflowError IOException, SQLException, NullPointerException
You should NOT catch these You SHOULD catch these
Extends Error Extends Exception
Analogy: The Fire Alarm System

Think of your Java program as a building with a fire alarm system.

An Exception is like a small fire in the kitchen. The alarm goes off, the sprinklers activate, someone grabs a fire extinguisher, and the situation is handled. The building stays standing. People are safe. Life goes on. That is what try-catch does — it detects the fire (exception), activates the right response (catch block), and lets the building (program) keep running.

An Error is like the entire building collapsing from an earthquake. No alarm system can save you. No fire extinguisher helps. The foundation is gone. That is what OutOfMemoryError or StackOverflowError is — the JVM itself is failing, and no amount of try-catch can fix it.

What Happens Without Exception Handling?

If you do not catch an exception, Java prints a stack trace to the console and kills your program. Every line of code after the exception is skipped. The program just stops.

NoCatch.java 
public class NoCatch {
    public static void main(String[] args) {
        System.out.println("Starting program...");

        int result = 10 / 0;  // ArithmeticException!

        // This line NEVER runs
        System.out.println("Result: " + result);
        System.out.println("Program finished.");
    }
}

Output:

Console Output
Starting program...
Exception in thread "main" java.lang.ArithmeticException: / by zero
    at NoCatch.main(NoCatch.java:5)

Notice: "Result: ..." and "Program finished." never print. The program crashed at line 5 and everything after it was abandoned. In a real application like Nimbus, this would mean a user clicks "Book Parcel" and the entire system goes down. That is unacceptable.

In Nimbus

Without exception handling, if the database is temporarily down when a user tries to register, the entire Nimbus application would crash. With exception handling, UserDAO catches the SQLException, UserService translates it into a meaningful message, and NimbusApp shows the user "Registration failed. Please try again." — the system stays alive.

Section 02

Exception Hierarchy

Every exception in Java is an object. They are all connected through a class hierarchy. Understanding this tree is the key to understanding why Java treats different exceptions differently.

                        java.lang.Object
                              |
                    java.lang.Throwable
                         /            \
                        /              \
              java.lang.Error     java.lang.Exception
              (DON'T catch)        /                \
                  |               /                  \
          StackOverflowError    RuntimeException     (Checked Exceptions)
          OutOfMemoryError      (Unchecked)            IOException
          VirtualMachineError       |                  SQLException
                              NullPointerException     FileNotFoundException
                              ArrayIndexOutOfBounds    ClassNotFoundException
                              ArithmeticException      ParseException
                              ClassCastException
                              NumberFormatException
                              IllegalArgumentException
The Java Exception Hierarchy — everything extends Throwable

Breaking Down the Tree

Throwable is the root. Only classes that extend Throwable (directly or indirectly) can be thrown with the throw keyword or caught with catch. You cannot throw a String or an int.

Error (left branch) represents JVM-level failures. You should almost never catch these. If the JVM is out of memory, catching it will not give you more memory.

Exception (right branch) splits into two sub-branches:

  • RuntimeException and its subclasses — these are unchecked. The compiler does NOT force you to catch them. They represent programmer mistakes like accessing a null reference or dividing by zero.
  • Everything else under Exception (IOException, SQLException, etc.) — these are checked. The compiler FORCES you to either catch them or declare them with throws. They represent external problems that can happen even in perfect code.
Key Insight

The rule is simple: if a class extends RuntimeException, it is unchecked. If it extends Exception but NOT RuntimeException, it is checked. If it extends Error, leave it alone.

In Nimbus

Nimbus has two custom exceptions. DuplicateEmailException extends Exception (checked) because duplicate emails are a foreseeable condition the caller MUST handle. BookingNotFoundException extends RuntimeException (unchecked) because it represents a logic error — asking for history of a non-existent customer is a programming mistake in the calling code.

Section 03

try-catch-finally

The try-catch-finally block is Java's mechanism for detecting and handling exceptions. It is the fire alarm + sprinkler + emergency exit — all in one construct.

The Basic Structure

BasicTryCatch.java 
try {
    // Code that MIGHT throw an exception
    int result = 10 / 0;
    System.out.println("This won't print");
} catch (ArithmeticException e) {
    // Code that runs ONLY if an ArithmeticException is thrown
    System.out.println("Cannot divide by zero!");
    System.out.println("Error: " + e.getMessage());
} finally {
    // Code that ALWAYS runs, no matter what
    System.out.println("Cleanup done.");
}
Line-by-line explanation
1
try {
Start a "danger zone." Any code inside here might fail. Java is now watching for exceptions.
2-3
int result = 10 / 0;
This line throws ArithmeticException. Java immediately STOPS executing the try block. Line 4 is skipped entirely. Java jumps to the catch block.
5
catch (ArithmeticException e) {
Java checks: "Is the thrown exception an ArithmeticException (or subclass)?" Yes, so this catch block runs. The variable e holds the exception object with details about what went wrong.
7
e.getMessage()
Returns the error message string. For ArithmeticException with division by zero, this returns "/ by zero".
9
finally {
This block runs NO MATTER WHAT — whether the try succeeded, whether catch ran, even if someone called return inside try or catch. It is guaranteed execution.

The Flow Step by Step

There are three possible scenarios when you use try-catch-finally:

Scenario 1: No Exception (happy path)

try blockruns completely
→
catch blockSKIPPED
→
finally blockALWAYS runs
→
code aftercontinues

Scenario 2: Exception is caught

try blockfails at line X
→
catch blockhandles it
→
finally blockALWAYS runs
→
code aftercontinues

Scenario 3: Exception is NOT caught (wrong type)

try blockfails at line X
→
catch blockwrong type, SKIPPED
→
finally blockSTILL runs
→
program CRASHESexception propagates up

finally ALWAYS Runs

This is one of the most important guarantees in Java. The finally block executes no matter what happens — even if there is a return statement inside the try or catch block.

FinallyWithReturn.java 
public static String test() {
    try {
        System.out.println("In try");
        return "from try";       // return is queued, but...
    } catch (Exception e) {
        System.out.println("In catch");
        return "from catch";
    } finally {
        System.out.println("In finally");  // ...this STILL runs!
        // If you put a return here, it OVERRIDES the try/catch return
    }
}

// Output when called:
// In try
// In finally
// Return value: "from try"
Warning

Never put a return inside a finally block. It will override the return value from try or catch, leading to confusing bugs. The compiler will even warn you about it.

What If the catch Block Also Throws?

If an exception is thrown inside the catch block, the finally block still runs. After finally completes, the new exception from catch propagates up to the caller.

CatchThrows.java 
try {
    int[] arr = new int[3];
    arr[10] = 5;  // ArrayIndexOutOfBoundsException
} catch (ArrayIndexOutOfBoundsException e) {
    System.out.println("Caught: " + e.getMessage());
    String s = null;
    s.length();   // NullPointerException inside catch!
} finally {
    System.out.println("Finally still runs!");  // YES, this prints
}
// After finally, the NullPointerException from catch propagates up
In Nimbus

UserDAO.findByEmail() uses try-catch-finally. The try opens a database connection and runs a query. The catch handles SQLException if the database is down. The finally always closes the connection — because an unclosed connection leaks resources and eventually crashes the server. (In modern Nimbus code, we use try-with-resources instead, covered in Section 08.)

Section 04

Multiple Catch Blocks

A single try block can throw different types of exceptions depending on what goes wrong. You can write multiple catch blocks to handle each type differently.

Order Matters: Specific First, General Last

Java checks catch blocks top to bottom. The first matching catch wins. If you put a general exception type (like Exception) before a specific one (like FileNotFoundException), the specific one will never execute — and the compiler will give you an error.

Correct Order
try {
    // risky code
} catch (FileNotFoundException e) {
    // specific handler
} catch (IOException e) {
    // broader handler
} catch (Exception e) {
    // catch-all
}
Wrong Order (won't compile)
try {
    // risky code
} catch (Exception e) {
    // catches EVERYTHING
} catch (IOException e) {
    // UNREACHABLE! Compiler error
} catch (FileNotFoundException e) {
    // UNREACHABLE! Compiler error
}
Analogy: Hospital Triage

Think of multiple catch blocks like a hospital emergency room. You want the specialist (specific catch) to see patients first. If a heart attack patient (FileNotFoundException) walks in, they should go to the cardiologist (specific catch), not the general practitioner (Exception). If you put the GP first, they take everyone, and the specialists sit idle. Java prevents this because it is always a bug.

Multi-catch with the Pipe Operator

If you want to handle two different exception types with the same code, use the pipe | operator (Java 7+). This avoids duplicating your error handling logic.

MultiCatch.java 
try {
    String input = "abc";
    int number = Integer.parseInt(input);      // NumberFormatException
    int[] arr = new int[2];
    arr[5] = number;                           // ArrayIndexOutOfBoundsException
} catch (NumberFormatException | ArrayIndexOutOfBoundsException e) {
    // Same handler for both
    System.out.println("Invalid input or index: " + e.getMessage());
} catch (Exception e) {
    // Catch-all for anything else
    System.out.println("Unexpected error: " + e.getMessage());
}
Multi-catch Rule

The exception types in a multi-catch cannot have a parent-child relationship. You cannot write catch (IOException | FileNotFoundException e) because FileNotFoundException is a subclass of IOException. Java will give a compiler error — just use catch (IOException e) instead, which already covers both.

In Nimbus

NimbusApp.java uses multiple catch blocks in its main operation loop. When a user tries to register, the try block can throw DuplicateEmailException (email already exists) or SQLException (database problem). Each gets its own catch block with a different user-facing message — "Email already registered" vs "System error, please try again."

Section 05

Checked vs Unchecked Exceptions

This is the concept that confuses beginners the most. Why does the compiler force you to catch some exceptions but not others? The answer is about whose fault it is and whether you can prevent it.

Checked Exceptions: Things That Can Go Wrong Despite Perfect Code

A checked exception represents a foreseeable external problem that can happen even if your code is 100% correct. Your code is perfect, but the file does not exist. Your code is perfect, but the database is down. Your code is perfect, but the network timed out.

Java says: "I know this can fail. You MUST plan for it. Either catch it or declare that your method throws it." If you do neither, the code will not compile.

Unchecked Exceptions: Programmer Mistakes

An unchecked exception (RuntimeException and its subclasses) represents a bug in your code. You accessed a null reference. You went past the array boundary. You divided by zero. These are things you could have prevented with an if-check.

Java says: "These are your fault. I trust you to write correct code. I won't force you to catch them because the fix is to write better code, not to wrap everything in try-catch."

Analogy: Driving Rules

Checked exceptions are like mandatory car insurance. Even if you are a perfect driver, another car can hit you. You MUST have insurance (must handle the exception). The law (compiler) requires it.

Unchecked exceptions are like running a red light. It is YOUR fault. The law does not require you to carry "red light insurance." The fix is to not run red lights (fix your code), not to plan for running them.

In Nimbus

SQLException is checked — so every method in UserDAO and BookingDAO that runs database queries MUST handle it. The compiler won't let you write connection.prepareStatement(sql) without either a try-catch or a throws SQLException declaration. This is Java protecting you: database calls are inherently unreliable, and you MUST have a plan for when they fail.

Section 06

throw vs throws

Two keywords that look almost identical but do completely different things. throw creates and fires an exception. throws is a warning label on a method.

throw — Creating and Firing an Exception

The throw keyword creates a new exception object and immediately stops the current method. Control is handed to the nearest matching catch block, or if there is none, the program crashes.

ThrowExample.java 
public void setAge(int age) {
    if (age < 0) {
        throw new IllegalArgumentException("Age cannot be negative: " + age);
        // Code after throw NEVER executes
    }
    if (age > 150) {
        throw new IllegalArgumentException("Age cannot exceed 150: " + age);
    }
    this.age = age;
}
Line-by-line explanation
1
throw new IllegalArgumentException("Age cannot be negative: " + age);
throw is the verb — it is an action. new IllegalArgumentException(...) creates the exception object. Together, this line fires the exception. The method stops immediately. The string message explains what went wrong.
2
// Code after throw NEVER executes
Any code after a throw statement is unreachable. The compiler knows this — if you write code directly after throw (not in another branch), it will give you an "unreachable statement" error.

throws — Declaring That a Method Might Fail

The throws keyword goes in the method signature. It is a warning label that says "this method might throw this exception, and I am NOT handling it here — YOU (the caller) must handle it."

ThrowsExample.java 
// throws goes in the method SIGNATURE (declaration)
public User findByEmail(String email) throws SQLException {
    Connection conn = DriverManager.getConnection(DB_URL);
    PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE email = ?");
    ps.setString(1, email);
    ResultSet rs = ps.executeQuery();
    // ... build User object
    return user;
}

// Now the CALLER must handle it:
public void registerUser(String email) {
    try {
        User existing = findByEmail(email);   // must be in try-catch
    } catch (SQLException e) {
        System.out.println("Database error: " + e.getMessage());
    }
}

Side-by-Side Comparison

Feature throw throws
What it does Creates and fires an exception NOW Declares that a method MIGHT throw later
Where it goes Inside a method body In the method signature
How many? Throws one exception at a time Can declare multiple: throws A, B, C
Followed by An exception object: throw new X() Exception class names: throws X, Y
Required? Only when you want to signal an error Required for checked exceptions not caught locally

Using Both Together

BothTogether.java 
// throws in signature: "I might throw DuplicateEmailException"
// throw in body: "I AM throwing DuplicateEmailException right now"
public void registerUser(String name, String email)
        throws DuplicateEmailException {          // <-- throws (declaration)

    User existing = userDAO.findByEmail(email);
    if (existing != null) {
        throw new DuplicateEmailException(email); // <-- throw (action)
    }
    userDAO.save(new User(name, email));
}
In Nimbus

UserService.registerUser() both throws and throws. The method signature declares throws DuplicateEmailException to warn callers. Inside, it uses throw new DuplicateEmailException(email) to actually fire the exception when a duplicate is found. Then in NimbusApp.java, the caller catches it: catch (DuplicateEmailException e) { System.out.println(e.getMessage()); }.

Section 07

Custom Exceptions

Java comes with dozens of built-in exceptions, but sometimes none of them describe your specific business problem. That is when you create your own exception class. It is just a class that extends Exception or RuntimeException.

Checked Custom Exception: DuplicateEmailException

This exception is checked (extends Exception) because a duplicate email is a foreseeable business condition. The caller MUST handle it — either show the user a message or try a different email.

DuplicateEmailException.java 
package com.nimbus.exception;

public class DuplicateEmailException extends Exception {

    private final String email;

    public DuplicateEmailException(String email) {
        super("Email already registered: " + email);
        this.email = email;
    }

    public String getEmail() {
        return email;
    }
}
Line-by-line explanation
1
extends Exception
This makes it a checked exception. Any method that throws this MUST declare it in the signature, and any caller MUST handle it. Java forces this at compile time.
2
private final String email;
Custom exceptions can carry extra data. Here we store the duplicate email so the caller can use it — maybe to pre-fill a "login instead?" form.
3
super("Email already registered: " + email);
Calls the parent Exception constructor with a descriptive message. This message is what e.getMessage() returns when the exception is caught.
4
public String getEmail()
Getter for the extra data. The catch block can call e.getEmail() to get the specific email that was duplicated.

Unchecked Custom Exception: BookingNotFoundException

This exception is unchecked (extends RuntimeException) because looking up bookings for a non-existent customer is a programming mistake in the calling code. The caller should have verified the customer exists first.

BookingNotFoundException.java 
package com.nimbus.exception;

public class BookingNotFoundException extends RuntimeException {

    private final String bookingId;

    public BookingNotFoundException(String bookingId) {
        super("No booking found with ID: " + bookingId);
        this.bookingId = bookingId;
    }

    public BookingNotFoundException(String bookingId, Throwable cause) {
        super("No booking found with ID: " + bookingId, cause);
        this.bookingId = bookingId;
    }

    public String getBookingId() {
        return bookingId;
    }
}
Line-by-line explanation
1
extends RuntimeException
This makes it unchecked. Callers are NOT forced to catch it. The compiler won't complain. But if it is thrown and not caught, the program still crashes.
2
public BookingNotFoundException(String bookingId, Throwable cause)
This is the "chaining" constructor. It takes the original exception (cause) so you don't lose the original error. For example, if a SQLException caused this failure, you can pass it as the cause and the full stack trace is preserved.

When to Use Which?

Scenario Extend Reasoning
Duplicate email at registration Exception (checked) Foreseeable business condition. Caller MUST handle it — show user a message.
Booking ID not found RuntimeException (unchecked) Programming error. Caller should have validated the ID first.
Invalid parcel weight (negative) RuntimeException (unchecked) Validation bug. The UI should prevent negative weight.
Payment gateway timeout Exception (checked) External service failure. Can happen anytime. Must have retry/fallback logic.
In Nimbus

UserService throws DuplicateEmailException when a user tries to register with an email that already exists. Because it is checked, NimbusApp.java is FORCED to have a catch block for it. CustomerHistoryService throws BookingNotFoundException when no bookings are found. Because it is unchecked, the caller CAN choose to catch it (and NimbusApp does), but the compiler does not force it.

Section 08

try-with-resources

Before Java 7, closing resources (database connections, file streams, etc.) was painful. You had to write ugly finally blocks with null checks. try-with-resources fixes all of this — resources are automatically closed when the block ends.

The Problem: Manual Resource Cleanup

ManualCleanup.java (the old, ugly way) 
Connection conn = null;
PreparedStatement ps = null;
ResultSet rs = null;

try {
    conn = DriverManager.getConnection(DB_URL);
    ps = conn.prepareStatement("SELECT * FROM users WHERE email = ?");
    ps.setString(1, email);
    rs = ps.executeQuery();
    // ... process results
} catch (SQLException e) {
    System.out.println("DB error: " + e.getMessage());
} finally {
    // Three separate null checks and close calls. Ugly!
    try { if (rs != null) rs.close(); } catch (SQLException e) { /* ignore */ }
    try { if (ps != null) ps.close(); } catch (SQLException e) { /* ignore */ }
    try { if (conn != null) conn.close(); } catch (SQLException e) { /* ignore */ }
}

That finally block is 3 lines of boilerplate just to close resources. And each close() can itself throw an exception, so you need try-catch inside try-catch. It is a mess.

The Solution: try-with-resources

AutoCleanup.java (the modern way) 
try (
    Connection conn = DriverManager.getConnection(DB_URL);
    PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE email = ?");
) {
    ps.setString(1, email);
    ResultSet rs = ps.executeQuery();
    // ... process results
} catch (SQLException e) {
    System.out.println("DB error: " + e.getMessage());
}
// conn, ps, and rs are ALL automatically closed here!
// No finally block needed. No null checks. No nested try-catch.
Line-by-line explanation
1
try (Connection conn = ...; PreparedStatement ps = ...) {
Resources declared inside the parentheses ( ) after try are auto-closed. They must implement AutoCloseable (which Connection, PreparedStatement, and ResultSet all do). Java calls .close() on each one in reverse order when the block ends.
2
// No finally block needed
Java generates the cleanup code for you. It handles null checks, exception suppression, and reverse-order closing — all invisibly. Your code stays clean and focused on business logic.

How It Works Behind the Scenes

Any class that implements the AutoCloseable interface (which has a single method: close()) can be used in a try-with-resources block. When the block ends — whether normally or via exception — Java calls close() on each resource in reverse declaration order.

Declare resourcesin try( )
→
Execute try bodyuse the resources
→
Auto-close ALLreverse order, guaranteed
→
Run catch (if any)after resources closed
Analogy: Hotel Checkout

Think of try-with-resources like a hotel that auto-checks you out. With the old way (finally), you had to remember to return the key, close the minibar tab, and check out at reception — and if you forgot any step, the hotel (JVM) loses resources. With try-with-resources, the hotel automatically handles checkout when you leave your room. You can't forget because you don't have to do it manually.

File I/O Example

ReadFile.java 
try (BufferedReader reader = new BufferedReader(new FileReader("parcels.txt"))) {
    String line;
    while ((line = reader.readLine()) != null) {
        System.out.println(line);
    }
} catch (FileNotFoundException e) {
    System.out.println("File not found: parcels.txt");
} catch (IOException e) {
    System.out.println("Error reading file: " + e.getMessage());
}
// reader is automatically closed here, even if an exception was thrown
In Nimbus

Every method in UserDAO and BookingDAO uses try-with-resources for Connection, PreparedStatement, and ResultSet. This ensures database connections are ALWAYS returned to the connection pool, even when exceptions occur. Without this, a busy system would run out of connections and grind to a halt.

Section 09

Exception Best Practices

Knowing the syntax of try-catch is easy. Using it well separates beginners from professionals. These are the rules every experienced Java developer follows.

1. Never Catch Exception or Throwable (Unless You Are at the Top)

Catching Exception means you catch everything — including runtime bugs like NullPointerException that you should fix, not hide. Only catch the specific exception you know how to handle.

Bad
try {
    registerUser(name, email);
} catch (Exception e) {
    // Which exception? NPE? SQL?
    // DuplicateEmail? Who knows!
    System.out.println("Error");
}
Good
try {
    registerUser(name, email);
} catch (DuplicateEmailException e) {
    System.out.println(e.getMessage());
} catch (SQLException e) {
    logger.error("DB failed", e);
}

Exception: At the very top level of your application (like main() in NimbusApp.java), a broad catch (Exception e) is acceptable as a last-resort safety net to prevent complete crashes.

2. Never Swallow Exceptions

An empty catch block is the worst thing you can do. The exception happened, but you threw away all evidence. When something breaks later, you have zero clues why.

Terrible (swallowed)
try {
    conn = getConnection();
} catch (SQLException e) {
    // silently ignore
}
// conn is null here. NullPointerException
// incoming, and you won't know why.
Good (logged)
try {
    conn = getConnection();
} catch (SQLException e) {
    logger.error("Failed to connect "
      + "to database", e);
    throw new ServiceException(
      "Registration unavailable", e);
}

3. Log Meaningful Messages

When logging an exception, include WHAT was being attempted, the input values (if safe to log), and the full exception (not just the message).

Useless log
catch (SQLException e) {
    logger.error("Error occurred");
    // Which error? Where? What input?
}
Useful log
catch (SQLException e) {
    logger.error("Failed to find user "
      + "by email: " + email, e);
    // Logs the full stack trace too
}

4. Do Not Use Exceptions for Flow Control

Exceptions are for exceptional situations. Do not use them as glorified if-else statements. They are expensive (creating the stack trace takes time) and make code hard to follow.

Exception as flow control
// Using exception to check if
// a number is valid
try {
    int n = Integer.parseInt(input);
    processNumber(n);
} catch (NumberFormatException e) {
    askForInputAgain();
}
Validate first
// Check BEFORE attempting
if (input.matches("-?\\d+")) {
    int n = Integer.parseInt(input);
    processNumber(n);
} else {
    askForInputAgain();
}

5. Catch Late, Throw Early

Throw the exception as soon as you detect the problem (throw early). Let it propagate up until it reaches a layer that knows how to handle it meaningfully (catch late). Do not catch an exception in the DAO if only the UI knows what message to show the user.

6. Always Close Resources (use try-with-resources)

Never leave a Connection, InputStream, or OutputStream unclosed. Use try-with-resources (Section 08) for everything that implements AutoCloseable.

7. Preserve the Original Exception (Exception Chaining)

When you catch one exception and throw a different one, pass the original as the cause. This preserves the full chain of what went wrong.

ExceptionChaining.java 
// BAD: original exception lost
catch (SQLException e) {
    throw new ServiceException("Registration failed");  // WHERE did SQL fail? Lost forever.
}

// GOOD: original exception preserved
catch (SQLException e) {
    throw new ServiceException("Registration failed", e);  // Pass 'e' as cause
    // Now the stack trace shows both ServiceException AND the original SQLException
}
In Nimbus

Nimbus follows these practices across all layers. UserDAO catches SQLException and either logs it and re-throws, or wraps it in a custom exception while preserving the cause. NimbusApp.java (the top-level UI) catches specific exceptions and shows user-friendly messages — never raw stack traces. Resources (Connection, PreparedStatement) always use try-with-resources.

Section 10

Common Exceptions Every Beginner Hits

These are the exceptions you WILL see in your first month of Java. Every single beginner encounters them. Knowing what causes each one and how to fix it will save you hours of debugging.

NullPointerException (NPE)

The #1 most common exception in all of Java. It means you tried to use an object reference that points to null — nothing is there.

NPE.java 
String name = null;
int length = name.length();  // NullPointerException!
// You called .length() on null. There is no String object to ask.

// FIX: Check before using
if (name != null) {
    int length = name.length();
}

Common causes: method returns null and you don't check, uninitialized object fields, getting null from a Map/List, calling methods on the result of another method that returned null.

ArrayIndexOutOfBoundsException

You tried to access an array element at an index that does not exist. Arrays are 0-indexed, so an array of size 5 has valid indices 0-4.

ArrayOutOfBounds.java 
int[] scores = {90, 85, 78};        // indices 0, 1, 2
System.out.println(scores[3]);       // ArrayIndexOutOfBoundsException!
System.out.println(scores[-1]);      // Also ArrayIndexOutOfBoundsException!

// FIX: Always check bounds
for (int i = 0; i < scores.length; i++) {
    System.out.println(scores[i]);   // safe
}

ClassCastException

You tried to cast an object to a type it is not. Java checked at runtime and said no.

ClassCast.java 
Object obj = "Hello";
Integer num = (Integer) obj;  // ClassCastException! String is not Integer.

// FIX: Use instanceof before casting
if (obj instanceof Integer) {
    Integer num = (Integer) obj;
}

NumberFormatException

You tried to parse a string into a number, but the string is not a valid number.

NumberFormat.java 
String input = "12.5kg";
int weight = Integer.parseInt(input);  // NumberFormatException!

// FIX: Validate or use try-catch
try {
    int weight = Integer.parseInt(input.replaceAll("[^0-9]", ""));
} catch (NumberFormatException e) {
    System.out.println("Please enter a valid number.");
}

StackOverflowError

A method calls itself recursively without a proper base case, creating an infinite loop of method calls until the stack memory runs out. Technically an Error, not an Exception, but every beginner sees it.

StackOverflow.java 
// Infinite recursion — no base case!
public int factorial(int n) {
    return n * factorial(n - 1);  // Never stops calling itself
}

// FIX: Add a base case
public int factorial(int n) {
    if (n <= 1) return 1;        // Base case — stops recursion
    return n * factorial(n - 1);
}

Quick Reference: What Causes Each

Exception Cause One-line Fix
NullPointerException Calling method on null Check if (x != null) before use
ArrayIndexOutOfBoundsException Index < 0 or >= array.length Use for (i < arr.length)
ClassCastException Invalid type cast Check with instanceof first
NumberFormatException Non-numeric string parsed Validate input or catch the exception
StackOverflowError Infinite recursion Add a base case
ArithmeticException Integer division by zero Check if (divisor != 0)
StringIndexOutOfBoundsException charAt/substring with bad index Check if (i < str.length())
ConcurrentModificationException Modifying collection during for-each Use Iterator.remove()
Section 11

Full Nimbus Exception Flow

This section shows how exceptions flow through all three layers of Nimbus: from the database (DAO), through business logic (Service), to the user interface (App). This is how real applications handle errors.

Layer Architecture

NimbusApp.javaUI Layer — catches & shows messages
→
UserService.javaBusiness Logic — validates & throws
→
UserDAO.javaDatabase Layer — catches SQLException

DAO Layer: UserDAO

The DAO layer talks to the database. It catches SQLException and either handles it or re-throws it wrapped in a custom exception.

UserDAO.java 
package com.nimbus.dao;

import com.nimbus.model.User;
import java.sql.*;

public class UserDAO {

    private static final String DB_URL = "jdbc:mysql://localhost:3306/nimbus";

    public User findByEmail(String email) {
        String sql = "SELECT * FROM users WHERE email = ?";

        // try-with-resources: Connection and PreparedStatement auto-close
        try (
            Connection conn = DriverManager.getConnection(DB_URL, "root", "password");
            PreparedStatement ps = conn.prepareStatement(sql);
        ) {
            ps.setString(1, email);
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    return new User(
                        rs.getInt("id"),
                        rs.getString("name"),
                        rs.getString("email")
                    );
                }
            }
        } catch (SQLException e) {
            // Log the full error for developers
            System.err.println("DB error in findByEmail for: " + email);
            e.printStackTrace();
            // Return null to indicate "not found" vs throwing
        }
        return null;
    }

    public boolean save(User user) {
        String sql = "INSERT INTO users (name, email) VALUES (?, ?)";

        try (
            Connection conn = DriverManager.getConnection(DB_URL, "root", "password");
            PreparedStatement ps = conn.prepareStatement(sql);
        ) {
            ps.setString(1, user.getName());
            ps.setString(2, user.getEmail());
            return ps.executeUpdate() > 0;
        } catch (SQLException e) {
            System.err.println("DB error saving user: " + user.getEmail());
            e.printStackTrace();
            return false;
        }
    }
}

Service Layer: UserService

The service layer contains business logic. It uses the DAO and throws custom exceptions for business rule violations.

UserService.java 
package com.nimbus.service;

import com.nimbus.dao.UserDAO;
import com.nimbus.exception.DuplicateEmailException;
import com.nimbus.model.User;

public class UserService {

    private final UserDAO userDAO = new UserDAO();

    // throws DuplicateEmailException — forces caller to handle it
    public User registerUser(String name, String email)
            throws DuplicateEmailException {

        // Business rule: email must be unique
        User existing = userDAO.findByEmail(email);
        if (existing != null) {
            // throw — fires the exception NOW
            throw new DuplicateEmailException(email);
        }

        // Business rule: name must not be empty
        if (name == null || name.trim().isEmpty()) {
            throw new IllegalArgumentException("Name cannot be empty");
        }

        User newUser = new User(name, email);
        boolean saved = userDAO.save(newUser);

        if (!saved) {
            throw new RuntimeException("Failed to save user — database error");
        }

        return newUser;
    }
}

App Layer: NimbusApp (UI)

The top-level application catches all exceptions and translates them into user-friendly messages. The user never sees a stack trace.

NimbusApp.java 
package com.nimbus.app;

import com.nimbus.exception.DuplicateEmailException;
import com.nimbus.exception.BookingNotFoundException;
import com.nimbus.service.UserService;
import com.nimbus.service.CustomerHistoryService;
import java.util.Scanner;

public class NimbusApp {

    private final UserService userService = new UserService();
    private final CustomerHistoryService historyService = new CustomerHistoryService();

    public void handleRegistration() {
        Scanner scanner = new Scanner(System.in);
        System.out.print("Enter name: ");
        String name = scanner.nextLine();
        System.out.print("Enter email: ");
        String email = scanner.nextLine();

        try {
            userService.registerUser(name, email);
            System.out.println("Registration successful!");

        } catch (DuplicateEmailException e) {
            // Specific: email already exists
            System.out.println("This email is already registered: " + e.getEmail());
            System.out.println("Please use a different email or login instead.");

        } catch (IllegalArgumentException e) {
            // Specific: invalid input
            System.out.println("Invalid input: " + e.getMessage());

        } catch (RuntimeException e) {
            // Broad catch for unexpected errors
            System.out.println("Something went wrong. Please try again later.");
            System.err.println("Unexpected error: " + e.getMessage());
        }
    }

    public void handleViewHistory(String customerId) {
        try {
            historyService.getHistory(customerId);

        } catch (BookingNotFoundException e) {
            System.out.println("No bookings found for customer: " + e.getBookingId());
            System.out.println("Have you made any bookings yet?");
        }
    }
}
Exception flow: User registers with duplicate email
1
NimbusApp calls userService.registerUser("John", "john@email.com")
The app layer calls the service layer. It has a try-catch ready.
2
UserService calls userDAO.findByEmail("john@email.com")
The service layer checks if the email already exists by querying the database through the DAO.
3
UserDAO runs SQL query → finds existing user → returns User object
The DAO uses try-with-resources for the Connection. The query succeeds, finds a match, and returns the User.
4
UserService: existing != null → throw new DuplicateEmailException("john@email.com")
The service detects the business rule violation and fires the exception. The service method STOPS here. The userDAO.save() line never runs.
5
NimbusApp: catch (DuplicateEmailException e) runs
The exception propagates up from Service to App. The matching catch block runs and prints a friendly message: "This email is already registered."
6
Program continues normally
The exception was handled. The program does NOT crash. It can ask for a different email or show a menu.
The Pattern

DAO catches database exceptions (SQLException) and handles resource cleanup. Service throws business exceptions (DuplicateEmailException) when rules are violated. App catches everything and translates it to user-friendly messages. Each layer has a specific responsibility in the exception chain.

Exercise 1: What Will This Print?

Exercise1.java
public static void main(String[] args) {
    try {
        System.out.println("A");
        int x = 10 / 0;
        System.out.println("B");
    } catch (ArithmeticException e) {
        System.out.println("C");
    } finally {
        System.out.println("D");
    }
    System.out.println("E");
}
Answer
1
Output: A C D E
"A" prints. Then 10/0 throws ArithmeticException — "B" is skipped. Catch runs, prints "C". Finally always runs, prints "D". Exception was handled, so code after try-catch continues, prints "E".

Exercise 2: What Will This Print?

Exercise2.java
public static void main(String[] args) {
    try {
        System.out.println("A");
        String s = null;
        s.length();
        System.out.println("B");
    } catch (ArithmeticException e) {

Exercise 3: What Will This Return?

Exercise3.java
public static int mystery() {
    try {
        return 1;
    } catch (Exception e) {
        return 2;
    } finally {
        return 3;
    }
}
Answer
1
Returns: 3
The try block queues return 1. No exception, so catch is skipped. But finally ALWAYS runs, and its return 3 overrides the queued return. This is why you should NEVER put a return in a finally block — it causes exactly this kind of confusing behavior.

Exercise 4: Will This Compile?

Exercise4.java
public void readFile() {
    BufferedReader reader = new BufferedReader(new FileReader("data.txt"));
    String line = reader.readLine();
    System.out.println(line);
    reader.close();
}
Answer
1
No, it will NOT compile.
FileReader throws FileNotFoundException (checked) and readLine() throws IOException (checked). Since these are checked exceptions, the compiler forces you to either wrap them in try-catch or add throws IOException to the method signature. The fix is to use try-with-resources.

Exercise 5: Find the Bug

Exercise5.java
try {
    Connection conn = DriverManager.getConnection(DB_URL);
    PreparedStatement ps = conn.prepareStatement("SELECT * FROM users");
    ResultSet rs = ps.executeQuery();
    while (rs.next()) {
        System.out.println(rs.getString("name"));
    }
} catch (SQLException e) {
    System.out.println("Database error");
}
Answer
1
Bug: Resources are never closed.
Connection, PreparedStatement, and ResultSet are opened but never closed. If an exception occurs (or even if it doesn't), these resources leak. Fix: use try-with-resources: try (Connection conn = ...; PreparedStatement ps = ...) { ... }

Exercise 6: Find the Bug

Exercise6.java
try {
    processPayment(order);
} catch (Exception e) {
    // handle all errors
} catch (PaymentTimeoutException e) {
    retryPayment(order);
} catch (InsufficientFundsException e) {
    notifyUser("Not enough balance");
}
Answer
1
Bug: Won't compile. Catch order is wrong.
catch (Exception e) is the broadest type and catches everything. The two catches after it are unreachable — they can never run because Exception already caught everything. Fix: put specific exceptions first (PaymentTimeoutException, InsufficientFundsException), then the broad Exception last.

public class InvalidTrackingNumberException extends Exception {

Exercise 8: What Will This Print?

Exercise8.java
public static void main(String[] args) {
    System.out.println("Start");
    try {
        System.out.println("Try 1");
        try {
            System.out.println("Try 2");
            int[] a = new int[2];
            a[5] = 10;
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Inner catch");
        }
        System.out.println("After inner try");
    } catch (Exception e) {
        System.out.println("Outer catch");
    } finally {
        System.out.println("Finally");
    }
    System.out.println("End");
}
Answer
1
Output: Start, Try 1, Try 2, Inner catch, After inner try, Finally, End
The exception is caught by the INNER catch (most specific match). After the inner try-catch completes, execution continues normally inside the outer try — "After inner try" prints. The outer catch is NOT triggered because the inner catch already handled it. Finally runs as always. "End" prints because no uncaught exception remains.
