Section 01

What is Maven?

Maven is a build tool that manages your Java project's dependencies, compilation, testing, and packaging — all from a single configuration file.

The Contractor Analogy

Imagine you want to build a house. You could go to every supplier yourself — one shop for bricks, another for cement, another for pipes, another for wiring. You'd have to know the exact brand, size, and quantity for each item. You'd carry everything to the construction site yourself. Then you'd need to know the right order to build: foundation first, then walls, then roof, then plumbing.

Or you could hire a contractor. You give the contractor a shopping list (what materials you need), and the contractor:

  • Goes to the right suppliers and gets everything for you
  • Makes sure the versions are compatible (you don't get pipes that don't fit your faucets)
  • Knows the right order to build things (can't put the roof before the walls)
  • Handles the entire build process from start to finish

Maven is that contractor. Your shopping list is a file called pom.xml. You write down what libraries (JARs) you need, and Maven downloads them, sets them up, compiles your code, runs your tests, and packages everything into a deployable file.

Analogy: Maven = Contractor

You = the architect. You decide what the house looks like.
pom.xml = the shopping list. What materials you need.
Maven = the contractor. Goes and gets everything, builds it in the right order.
Maven Central = the hardware store. Where all the materials are available.
.m2 folder = your local storage shed. Once Maven buys something, it stores a copy here so it doesn't have to buy it again.

What Did We Do Before Maven?

Before Maven, Java developers had to:

  1. Manually download JARs — Go to each library's website, find the right version, download the .jar file
  2. Copy JARs into the project — Drop them into a lib/ folder
  3. Configure the classpath — Tell Java exactly where every JAR is
  4. Hope versions are compatible — If Library A needs version 2 of something and Library B needs version 3, you're in trouble
  5. Repeat for every team member — Everyone has to manually download the same JARs

This was a nightmare for teams. One developer's project works, another's doesn't, because they have slightly different JARs. Maven solved all of this with one configuration file that everyone shares.

Think About It

When you clone a Maven project from Git, you don't get any JAR files. You get the pom.xml. Maven reads it and downloads everything automatically. Every developer gets the exact same dependencies. No "works on my machine" problems.

Section 02

pom.xml Anatomy

The pom.xml (Project Object Model) is the single file that describes everything about your Maven project. Let's break down every piece.

The Minimal pom.xml

Every Maven project needs at least this:

pom.xml (minimal) 
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
         http://maven.apache.org/xsd/maven-4.0.0.xsd">

    <modelVersion>4.0.0</modelVersion>

    <groupId>com.nimbus</groupId>
    <artifactId>nimbus-parcel</artifactId>
    <version>1.0-SNAPSHOT</version>
    <packaging>war</packaging>

</project>
Line-by-line explanation
1-4
<project xmlns="...">
Root element. The xmlns stuff is just XML boilerplate telling the parser "this is a Maven POM file." You never change this.
6
<modelVersion>4.0.0</modelVersion>
Always 4.0.0. This is the POM schema version. Never changes.
8
<groupId>com.nimbus</groupId>
Your organization's reverse domain name. Like a family name. All Nimbus projects share this.
9
<artifactId>nimbus-parcel</artifactId>
The project's unique name within the group. Like a first name. This is what your built file will be called.
10
<version>1.0-SNAPSHOT</version>
Current version. SNAPSHOT means "in development, not final." When you release, you remove -SNAPSHOT.
11
<packaging>war</packaging>
What type of output to create. "war" = Web Application Archive (for web apps deployed to Tomcat). "jar" = Java Archive (for standalone apps).

The GAV Coordinates

Every Maven artifact (library, project, plugin) in the world is uniquely identified by three values — groupId, artifactId, and version. Together they're called GAV coordinates, like GPS coordinates for a library.

CoordinateWhat It IsExample
groupId Organization/company identifier. Reverse domain convention. com.mysql, org.springframework, com.nimbus
artifactId Specific project/library name within that organization. mysql-connector-j, spring-core, nimbus-parcel
version Which release of that library. Semantic versioning: major.minor.patch. 8.0.33, 5.3.30, 1.0-SNAPSHOT

Think of it like a postal address: groupId is the city, artifactId is the street, version is the house number. With all three, Maven can find exactly the right JAR file on Maven Central.

Properties Section

Properties are like variables for your pom.xml. Instead of writing the same version number in 10 places, you define it once and reference it everywhere.

pom.xml (properties) 
<properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    <spring.version>5.3.30</spring.version>
</properties>
  • maven.compiler.source and maven.compiler.target — Tell Maven what Java version to compile with. We use Java 17.
  • project.build.sourceEncoding — Character encoding. UTF-8 is the standard. Without this, Maven throws warnings.
  • spring.version — A custom property. Reference it later with ${spring.version} instead of typing 5.3.30 everywhere.
Section 03

Dependencies

Dependencies are the external libraries your project needs. Maven downloads them automatically so you never have to manually hunt for JAR files.

How Maven Downloads JARs

When you add a dependency to pom.xml and run any Maven command, here's what happens:

Maven reads pom.xmlfinds <dependencies>
→
Checks local .m2~/.m2/repository/
→
Found locally?Yes = use it
↓ No
Downloads from Maven Centralrepo.maven.apache.org
→
Saves to .m2cached for next time
→
Added to classpathavailable in your code

Maven Central Repository

Maven Central (repo.maven.apache.org) is the world's largest repository of Java libraries. It's like npm for JavaScript or pip for Python, but for Java. When you search for a library, go to search.maven.org or mvnrepository.com — both show you the exact XML to copy into your pom.xml.

Dependency Coordinates

Every dependency uses the same GAV coordinates we discussed. Here's how you add MySQL connector:

pom.xml (single dependency) 
<dependencies>
    <dependency>
        <groupId>com.mysql</groupId>
        <artifactId>mysql-connector-j</artifactId>
        <version>8.0.33</version>
    </dependency>
</dependencies>

This tells Maven: "Go find the library made by com.mysql, named mysql-connector-j, version 8.0.33, and download it." Maven translates these coordinates into a folder path inside .m2:

Where Maven stores it 
~/.m2/repository/com/mysql/mysql-connector-j/8.0.33/mysql-connector-j-8.0.33.jar
//               ^groupId^  ^artifactId^      ^ver^  ^actual JAR file^

Transitive Dependencies

This is where Maven gets really powerful. Say you add Spring MVC as a dependency. But Spring MVC itself needs Spring Core, Spring Beans, and Spring Context to work. Do you have to list all of those too? No.

Maven handles transitive dependencies automatically. If Library A depends on Library B, and Library B depends on Library C — you only add A. Maven downloads B and C automatically.

You add:        spring-webmvc (5.3.30)
                    |
Maven also      +---+--- spring-web (5.3.30)
downloads:      |   +--- spring-context (5.3.30)
                |   +--- spring-beans (5.3.30)
                |   +--- spring-core (5.3.30)
                |   +--- spring-expression (5.3.30)
                |
You write 1 dependency. Maven downloads 6.
Transitive dependency resolution: Maven walks the entire tree.
Watch Out

Transitive dependencies can cause version conflicts. If Library A needs commons-logging 1.2 and Library B needs commons-logging 1.3, Maven picks the one "closest" to your project in the dependency tree. You can override this with <exclusions> or by declaring the specific version yourself.

Section 04

Maven Project Structure

Maven enforces a standard directory layout. Every Maven project on Earth follows this same structure. Once you learn it, you can navigate any project.

Standard Maven Web Project Structure 
nimbus-parcel/
├── pom.xml                          // The brain of the project
├── src/
│   ├── main/
│   │   ├── java/                    // Your Java source code
│   │   │   └── com/nimbus/
│   │   │       ├── app/             // Servlets (presentation layer)
│   │   │       ├── service/         // Business logic
│   │   │       ├── dao/             // Database access
│   │   │       ├── model/           // Data classes (User, Booking)
│   │   │       ├── util/            // Helpers (DBConnection)
│   │   │       └── exception/       // Custom exceptions
│   │   ├── resources/               // Config files (db.properties)
│   │   └── webapp/                  // Web files (JSP, CSS, JS)
│   │       ├── WEB-INF/
│   │       │   └── web.xml          // Deployment descriptor
│   │       ├── css/
│   │       ├── js/
│   │       └── index.jsp
│   └── test/
│       └── java/                    // Unit tests (JUnit)
│           └── com/nimbus/
│               └── dao/
│                   └── UserDAOTest.java
└── target/                          // Maven's output (compiled code, WAR file)
    ├── classes/                     // Compiled .class files
    ├── test-classes/                // Compiled test classes
    └── nimbus-parcel-1.0-SNAPSHOT.war

Why This Structure?

DirectoryPurposeWhy It Exists
src/main/java All your application Java code Separates source from output. Maven knows exactly where to find code to compile.
src/main/resources Config files like db.properties These get copied to target/classes during build, so they're on the classpath at runtime.
src/main/webapp Web assets: JSP, HTML, CSS, JS, web.xml Only for war projects. This becomes the root of your web application.
src/test/java Unit test classes Tests are separate from production code. They compile and run during mvn test but don't go into the final WAR.
target/ Everything Maven generates Never edit files here. mvn clean deletes this entire folder. It's regenerated on every build.
In Nimbus

Our Nimbus parcel management project follows this exact structure. Your servlets go in src/main/java/com/nimbus/app/, your DAO classes in dao/, your business logic in service/, and all JSP pages in src/main/webapp/. The db.properties file with MySQL connection details goes in src/main/resources/.

Common Mistake

Putting Java files in the wrong directory. If your class is in src/main/java but your package declaration says package com.nimbus.dao;, then the file MUST be at src/main/java/com/nimbus/dao/YourClass.java. The folder path must match the package. Maven will fail to compile if they don't match.

Section 05

Build Lifecycle

Maven doesn't just compile your code. It runs a sequence of phases, each building on the last. This sequence is called the build lifecycle.

The Default Lifecycle

When you run a Maven command like mvn package, Maven doesn't just run the "package" step. It runs every step before it, in order. Think of it like a recipe — you can't frost a cake before baking it.

validate
Checks if the project structure is correct and pom.xml is valid. "Do we have everything we need to start?"
compile
Compiles all .java files in src/main/java into .class files in target/classes.
test
Runs all unit tests in src/test/java using JUnit/TestNG. If any test fails, the build stops.
package
Takes compiled code and packages it into a .war or .jar file in target/.
install
Copies the built .war/.jar into your local .m2 repository so other local projects can use it as a dependency.
deploy
Uploads the artifact to a remote repository (like a company's private Maven repo) for other developers to use.
Key Insight

If you run mvn package, Maven automatically runs validate → compile → test → package. You don't need to run each step manually. Each phase includes all phases before it.

Maven Commands You'll Actually Use

CommandWhat It DoesWhen You Use It
mvn clean Deletes the entire target/ folder. Fresh start. When something feels stale or broken. "Turn it off and on again."
mvn compile Runs validate → compile. Produces .class files. Quick check: "Does my code even compile?"
mvn test Runs validate → compile → test. Before committing: "Do all my tests pass?"
mvn package Runs validate → compile → test → package. Creates the WAR file for deployment.
mvn install Runs everything through install. When other local projects need this as a dependency.
mvn clean package Deletes target/ first, then does a full build. Most common command. Clean build every time.
In Nimbus

For Nimbus, your most-used command will be mvn clean package. This creates target/nimbus-parcel-1.0-SNAPSHOT.war, which you then deploy to Tomcat. During development in Eclipse, the IDE runs mvn compile automatically when you save a file.

Section 06

Plugins

Plugins are what actually do the work in Maven. Every lifecycle phase is executed by a plugin. Think of phases as job titles and plugins as the people who do those jobs.

How Plugins Work

Maven itself is just a framework. It doesn't know how to compile Java, run tests, or create WAR files. It delegates everything to plugins. When you run mvn compile, Maven calls the maven-compiler-plugin to do the actual compilation.

mvn compileyou type this
→
Maven Frameworklooks up "compile" phase
→
maven-compiler-plugindoes the actual work

Key Plugins for Nimbus

1. maven-compiler-plugin

Controls how Java source code is compiled. You configure the Java version here.

pom.xml (compiler plugin) 
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.11.0</version>
    <configuration>
        <source>17</source>
        <target>17</target>
    </configuration>
</plugin>

<source>17</source> means "accept Java 17 syntax." <target>17</target> means "produce bytecode that runs on JRE 17+."

2. maven-war-plugin

Creates the .war file from your web application. Packages compiled classes, dependencies, JSPs, and web.xml into a single deployable archive.

pom.xml (war plugin) 
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-war-plugin</artifactId>
    <version>3.4.0</version>
    <configuration>
        <failOnMissingWebXml>false</failOnMissingWebXml>
    </configuration>
</plugin>

failOnMissingWebXml set to false means Maven won't fail if you don't have a web.xml — useful when using annotations like @WebServlet instead.

3. maven-surefire-plugin

Runs your unit tests during the test phase. It finds all classes ending in Test or Tests in src/test/java and executes them.

pom.xml (surefire plugin) 
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <version>3.1.2</version>
</plugin>
Plugin vs Dependency

A dependency is a library your code uses at runtime (like MySQL connector). A plugin is a tool Maven uses during the build process (like the compiler). Dependencies go in <dependencies>. Plugins go in <build><plugins>.

Section 07

Dependency Scopes

Not every dependency is needed at every stage. Scopes control when a dependency is available and whether it gets packaged into the final WAR/JAR.

The Six Scopes

ScopeCompileTestRuntimeIn WAR?Example
compile Yes Yes Yes Yes spring-webmvc — needed everywhere
provided Yes Yes No No servlet-api — Tomcat provides it
runtime No Yes Yes Yes mysql-connector-j — not needed at compile time, loaded by JDBC at runtime
test No Yes No No junit — only for testing, never in production
system Yes Yes No No Local JAR from a specific path. Avoid this scope.
import Special: only for <dependencyManagement> to import another POM's dependency list. Spring BOM

Why "provided" Matters

This is the most confusing scope for beginners, so let's be clear. servlet-api is a library you need to compile your servlets. But at runtime, Tomcat already has its own copy of servlet-api. If you package your copy into the WAR too, you get two copies and they conflict.

The provided scope says: "I need this to compile, but don't put it in the WAR — the server will provide it."

pom.xml (provided scope) 
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>javax.servlet-api</artifactId>
    <version>4.0.1</version>
    <scope>provided</scope>   <!-- Tomcat has this -->
</dependency>
In Nimbus

Our Nimbus project uses three scopes: compile for Spring and JSTL (packaged in WAR), provided for servlet-api (Tomcat provides it), and runtime for MySQL connector (loaded by JDBC driver manager, not referenced directly in code imports).

Dependency-by-dependency explanation
1
javax.servlet-api (provided)
The Servlet API gives you HttpServletRequest, HttpServletResponse, HttpServlet, etc. Scope is "provided" because Tomcat already has this JAR — we only need it to compile against.
2
javax.servlet.jsp-api (provided)
The JSP API for writing JSP pages with embedded Java. Also provided by Tomcat.
3
jstl 1.2 (compile — default)
JSP Standard Tag Library. Gives you <c:forEach>, <c:if>, <fmt:formatDate> etc. This IS packaged in the WAR because Tomcat does NOT include it.
4
mysql-connector-j (runtime)
JDBC driver for MySQL. Your code never directly imports MySQL classes — you use java.sql.Connection. The driver is loaded at runtime via Class.forName() or DriverManager.
5
spring-core
Foundation of Spring Framework. Contains core utilities, resource loading, and the base abstractions that all other Spring modules depend on.
6
spring-context
The dependency injection (DI) container. This is where @Component, @Autowired, @Service, @Repository live. It manages object creation and wiring.
7
spring-webmvc
Spring's web framework. Provides DispatcherServlet, @Controller, @RequestMapping, view resolvers. This is what turns Spring into a web application framework.
8
spring-jdbc
Spring's JDBC abstraction. Provides JdbcTemplate which eliminates the boilerplate try-catch-finally of raw JDBC. Handles connection management, exception translation.
9
junit-jupiter (test)
JUnit 5 testing framework. Only available in src/test/java, never packaged in the WAR. Used for unit testing DAO, Service, and utility classes.

Notice how Spring dependencies all use ${spring.version}. This references the property we defined in <properties>. When Spring releases a new version, you change one number and all four update together. No chance of version mismatch.

Section 09

Creating a Maven Project in Eclipse

Step-by-step guide to creating the Nimbus project as a Maven web application in Eclipse IDE.

Step-by-Step Process

  1. Open Eclipse and go to File → New → Maven Project. If you don't see "Maven Project," go to File → New → Other and search for "Maven."
  2. Skip archetype selection. Check "Create a simple project (skip archetype selection)" and click Next. Archetypes are templates, but they often create outdated structures. We'll set up everything ourselves.
  3. Fill in project coordinates:
    • Group Id: com.nimbus
    • Artifact Id: nimbus-parcel
    • Version: 1.0-SNAPSHOT (default, keep it)
    • Packaging: war (dropdown — select "war" not "jar")
    Click Finish.
  4. Eclipse creates the project with the standard Maven structure. You'll see src/main/java, src/main/resources, src/test/java, and pom.xml in the Package Explorer.
  5. Create the webapp folder. Right-click src/main → New → Folder → name it webapp. Inside webapp, create a folder called WEB-INF.
  6. Open pom.xml and paste the complete Nimbus pom.xml from Section 8 above. Save the file. Eclipse will show a notification.
  7. Update Maven project. Right-click the project → Maven → Update Project (or press Alt+F5). This tells Eclipse to re-read the pom.xml and download all dependencies. Watch the bottom-right progress bar — Maven is downloading JARs to your .m2 folder.
  8. Verify dependencies. Expand "Maven Dependencies" in the Package Explorer. You should see all the JARs listed: mysql-connector, spring jars, jstl, junit. If you see red X marks, right-click → Maven → Update Project again.
  9. Set the JDK. Right-click project → Properties → Java Build Path → Libraries tab. Make sure "JRE System Library" shows Java 17 (or JDK 17). If not, click it → Edit → select "Workspace default JRE" or choose the installed JDK 17.
  10. Create packages. Right-click src/main/java → New → Package. Create:
    • com.nimbus.app
    • com.nimbus.service
    • com.nimbus.dao
    • com.nimbus.model
    • com.nimbus.util
    • com.nimbus.exception
If Dependencies Don't Download

Check your internet connection first. Then try: Right-click project → Maven → Update Project → check "Force Update of Snapshots/Releases" → OK. If still stuck, delete the ~/.m2/repository folder and try again. Maven will re-download everything fresh.

Eclipse + Maven Integration

Eclipse has a built-in Maven plugin called m2e (Maven 2 Eclipse). It reads your pom.xml and automatically configures your project's classpath, Java version, and build settings. When you change pom.xml, do Alt+F5 to sync. Eclipse also runs mvn compile automatically every time you save a file (incremental compilation).

Section 10

Common Maven Errors

Every Java developer hits these errors eventually. Here's what they mean and how to fix them.

1. Dependency Not Found

Error output 
Could not find artifact com.mysql:mysql-connector:jar:8.0.33
in central (https://repo.maven.apache.org/maven2)

Cause: Wrong GAV coordinates. Maybe you typed mysql-connector instead of mysql-connector-j. Maven can't find something that doesn't exist at that exact address.

Fix: Go to mvnrepository.com, search for the library, and copy the exact XML. Don't type it from memory.

2. Version Conflict

Error output 
java.lang.NoSuchMethodError: org.springframework.core.ResolvableType.forClass()
// or
java.lang.ClassNotFoundException: org.springframework.beans.factory.BeanFactory

Cause: You have Spring Core 5.3.30 but Spring Web 6.0.0. They're incompatible. Different major versions of Spring don't work together.

Fix: Use the same version for all Spring modules. That's why we use ${spring.version} property — one value, all in sync.

3. Cannot Resolve Symbol / Import Not Found

Error in IDE 
import javax.servlet.http.HttpServlet;  // Red underline: "cannot resolve"

Cause: The dependency isn't in your pom.xml, or you haven't done Maven → Update Project after adding it.

Fix: Add javax.servlet-api to pom.xml, save, then Alt+F5.

4. Compilation Error: Source/Target Version

Error output 
error: source option 5 is no longer supported. Use 7 or later.

Cause: You didn't set maven.compiler.source and maven.compiler.target in properties. Maven defaults to Java 5, which modern JDKs refuse.

Fix: Add the <properties> section with source and target set to 17.

5. WAR Build Fails: web.xml Missing

Error output 
Error assembling WAR: webxml attribute is required

Cause: Maven expects a web.xml in src/main/webapp/WEB-INF/, but you're using @WebServlet annotations instead.

Fix: Add maven-war-plugin with <failOnMissingWebXml>false</failOnMissingWebXml>, or create a minimal web.xml.

Nuclear Option: Clean .m2

If nothing else works and Maven is behaving strangely, delete the entire ~/.m2/repository folder. This forces Maven to re-download every dependency from scratch. It takes a few minutes but fixes most corruption issues. On Windows: C:\Users\YourName\.m2\repository.

Practice Exercises

Hands-on exercises to solidify your Maven understanding. Do these in Eclipse.

Exercise 1: Create from Scratch

Create a new Maven project in Eclipse called hello-maven with groupId com.practice, packaging jar. Add a single class com.practice.App with a main method that prints "Hello Maven!" Run mvn clean package from the terminal. Find the generated JAR in target/.

Success criteria: You see BUILD SUCCESS and hello-maven-1.0-SNAPSHOT.jar exists in target.

Exercise 2: Add a Dependency

To your hello-maven project, add the gson dependency (Google's JSON library). GAV: com.google.code.gson : gson : 2.10.1. Create a class that converts a Java object to JSON string using new Gson().toJson(object). Run mvn compile.

Success criteria: Code compiles without errors. Check .m2/repository/com/google/code/gson/ — the JAR should be there.

Exercise 3: Understand Scopes

Add JUnit 5 with <scope>test</scope>. Try importing org.junit.jupiter.api.Test in a class inside src/main/java. What happens? Now move that class to src/test/java. What changes?

Success criteria: Import fails in main, works in test. You understand why: test-scoped dependencies are only on the classpath for test sources.

Exercise 4: Break and Fix

In your pom.xml, deliberately change the MySQL version to 99.99.99 (doesn't exist). Run mvn compile. Read the error message carefully. Now fix it back to 8.0.33. Remove the <properties> section entirely. Run mvn compile again. What error do you get? Fix it.

Success criteria: You can read Maven error messages and know which part of pom.xml to fix.

Exercise 5: Nimbus Setup

Follow Section 9 to create the full Nimbus project. Use the complete pom.xml from Section 8. Create all six packages. Add a simple User.java model class in com.nimbus.model with fields userId, name, email, password. Run mvn clean compile.

Success criteria: BUILD SUCCESS. All packages visible in Eclipse. User.java compiles. Maven Dependencies shows all JARs.
