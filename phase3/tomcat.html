Section 01

What is a Web Server?

A web server is a program that sits on a machine, listens for incoming requests from browsers, and sends back the right files or data. Every website you visit is served by one.

The Post Office Analogy

Analogy: The Post Office

Think of a web server as a post office. People (browsers) send letters (HTTP requests) to a specific address (URL). The post office receives the letter, looks at where it needs to go, fetches the right package (HTML page, image, data), and sends it back to the person who asked.

It doesn't write the letters. It doesn't create the packages. It just receives, routes, and delivers. That's exactly what a web server does.

What Happens When You Visit a Website

You type URLhttp://nimbus.com/track
→
DNS Lookupnimbus.com → 192.168.1.50
→
Web ServerListening on port 8080
→

→
Send ResponseHTML back to browser
Every website visit follows this exact flow

How a Web Server Works

  1. Listen on a port. The server binds to a port number (usually 80 for HTTP, 443 for HTTPS, 8080 for development). It waits for connections.
  2. Receive an HTTP request. A browser connects and sends a request: GET /track?bookingId=42 HTTP/1.1. The server reads the method (GET), the path (/track), and any parameters.
  3. Process the request. For static files (HTML, CSS, images), the server finds the file on disk and reads it. For dynamic content, the server passes the request to application code (like a Servlet).
  4. Send the response. The server sends back an HTTP response with a status code (200 OK, 404 Not Found), headers (content type, length), and the body (the HTML page, JSON data, or image).
Static vs Dynamic Content

Static: The server returns a file exactly as it is on disk. An about.html page looks the same for every visitor. No processing needed.

Dynamic: The server runs code to generate the response. A tracking page shows different data for each booking ID. The server calls your Java code, which queries the database, and builds the HTML on the fly.

Section 02

Apache Tomcat — A Servlet Container

Tomcat is the most popular Java web server. But it's not a full web server like Apache HTTP or Nginx — it's specifically a Servlet container that runs your Java web applications.

Web Server vs Servlet Container

AspectWeb Server (Apache HTTP, Nginx)Servlet Container (Tomcat)
Primary job Serve static files (HTML, CSS, images) very fast Run Java Servlets and JSPs
Dynamic content Needs plugins (mod_php, FastCGI) for server-side code Built-in — runs Java code natively
Languages PHP, Python, etc. via modules Java only
Performance (static) Extremely fast, handles 10K+ concurrent connections Decent, but not optimized for static files
Used for Static websites, reverse proxy, load balancing Java web applications (Servlets, JSP, Spring)

In production, companies often put Nginx in front of Tomcat. Nginx handles static files and SSL, and forwards dynamic requests to Tomcat. For development, Tomcat alone is enough.

Analogy: Hotel vs Kitchen

A full web server (Nginx) is like the entire hotel — reception, hallways, elevators, room service. It handles everything.

Tomcat is like the kitchen. It doesn't handle the front desk or the elevator. But when a guest orders food (dynamic Java request), the kitchen is where the actual cooking (Java code execution) happens. In production, the hotel (Nginx) takes the order and sends it to the kitchen (Tomcat).

Tomcat Architecture

Catalina
The Servlet container engine. Manages the lifecycle of Servlets (init, service, destroy). This is the core of Tomcat.
Coyote
The HTTP connector. Listens on port 8080, parses incoming HTTP requests, and passes them to Catalina for processing.
Jasper
The JSP engine. Converts .jsp files into Servlet Java classes, compiles them, and runs them. JSPs are actually Servlets under the hood.
Cluster
Handles session replication across multiple Tomcat instances. Used in production for high availability.
Section 03

Setting Up Tomcat in Eclipse

Eclipse has built-in support for Tomcat. You add it once, and Eclipse can start/stop the server, deploy your app, and show console output — all from the IDE.

Step-by-Step Setup

  1. Download Tomcat. Go to tomcat.apache.org → Download → Tomcat 10.x (for Jakarta Servlet API). Extract the zip to a folder like C:\apache-tomcat-10.
  2. Open Eclipse. Go to Window → Preferences → Server → Runtime Environments → Add. Select Apache Tomcat v10.x. Point to the folder where you extracted Tomcat.
  3. Create a Server. Go to the Servers tab at the bottom of Eclipse. Right-click → New → Server. Select Tomcat v10.x. Click Finish.
  4. Add your project. Right-click the server in the Servers tab → Add and Remove → move your project to the "Configured" side. Click Finish.
  5. Start the server. Right-click server → Start. The console shows Tomcat startup logs. Visit http://localhost:8080/nimbus/ to see your app.
Common Setup Issue: Port 8080 Already in Use

If another program is using port 8080, Tomcat won't start. Fix: Double-click the server in the Servers tab. Change the HTTP/1.1 port from 8080 to 8081 or 9090. Save. Restart.

In Nimbus — Your First Server Start

When you start Tomcat with the Nimbus project, the console shows something like:

INFO: Starting Servlet engine: [Apache Tomcat/10.1.x]

INFO: Deploying web application directory [nimbus]

Your app is now live at http://localhost:8080/nimbus/. Every Servlet you write is accessible from a browser.

Section 04

WAR Files — Packaging Your Web App

A WAR (Web Application Archive) file is a single .war file that contains your entire web application — compiled classes, JSPs, config files, libraries — ready to be dropped into any Tomcat server.

What's Inside a WAR?

A WAR file is just a ZIP file with a specific structure. When you rename nimbus.war to nimbus.zip and extract it, you see the same directory structure as your project's deployed form.

Analogy: Moving Box

Building a WAR is like packing a moving box. You put all your belongings (classes, JSPs, config) into one box (WAR file), label it (nimbus.war), and ship it to the new apartment (production server). The movers (Tomcat) unpack it and set everything up in the right place.

Creating a WAR in Eclipse

  1. Right-click your project → Export → Web → WAR file.
  2. Choose destination. Pick a folder to save the WAR. Name it nimbus.war.
  3. Click Finish. Eclipse compiles all your Java code, bundles everything into the WAR structure, and creates the file.

Creating a WAR with Maven

Terminal command 
// In your project root directory:
// mvn clean package
//
// This compiles code, runs tests, and creates:
// target/nimbus.war
//
// The <packaging>war</packaging> in pom.xml tells Maven
// to build a WAR instead of a JAR.

Deploying the WAR

  1. Copy the WAR file to Tomcat's webapps/ folder: C:\apache-tomcat-10\webapps\nimbus.war
  2. Start Tomcat. It detects the new WAR file, extracts it automatically into a nimbus/ folder, and deploys the app.
  3. Access the app at http://localhost:8080/nimbus/. The folder name becomes the context path.
Hot Deploy

If Tomcat is already running, you can drop a WAR into webapps/ and Tomcat will detect and deploy it automatically — no restart needed. This is called hot deployment. In development with Eclipse, this happens every time you save a file (Eclipse pushes the change to the running Tomcat).

Section 05

web.xml — The Deployment Descriptor

The web.xml file is Tomcat's instruction manual for your app. It tells Tomcat which Servlets exist, what URLs they handle, what the welcome page is, and how to handle errors.

Location

The file lives at WEB-INF/web.xml inside your project. Tomcat reads it when the app starts up. If it's missing or malformed, the app won't deploy.

Full Nimbus web.xml

WEB-INF/web.xml 
<?xml version="1.0" encoding="UTF-8"?>
<web-app xmlns="https://jakarta.ee/xml/ns/jakartaee"
         version="6.0">

    <display-name>Nimbus Parcel Management</display-name>

    <!-- ======= WELCOME FILE ======= -->
    <welcome-file-list>
        <welcome-file>index.jsp</welcome-file>
    </welcome-file-list>

    <!-- ======= SERVLET REGISTRATION ======= -->
    <servlet>
        <servlet-name>LoginServlet</servlet-name>
        <servlet-class>com.nimbus.servlet.LoginServlet</servlet-class>
        <load-on-startup>1</load-on-startup>
    </servlet>
    <servlet-mapping>
        <servlet-name>LoginServlet</servlet-name>
        <url-pattern>/login</url-pattern>
    </servlet-mapping>

    <servlet>
        <servlet-name>BookingServlet</servlet-name>
        <servlet-class>com.nimbus.servlet.BookingServlet</servlet-class>
    </servlet>
    <servlet-mapping>
        <servlet-name>BookingServlet</servlet-name>
        <url-pattern>/book</url-pattern>
    </servlet-mapping>

    <!-- ======= ERROR PAGES ======= -->
    <error-page>
        <error-code>404</error-code>
        <location>/WEB-INF/error/404.jsp</location>
    </error-page>
    <error-page>
        <error-code>500</error-code>
        <location>/WEB-INF/error/500.jsp</location>
    </error-page>

    <!-- ======= SESSION CONFIG ======= -->
    <session-config>
        <session-timeout>30</session-timeout>
    </session-config>

</web-app>
Key sections explained
1
<web-app>
Root element. The version="6.0" tells Tomcat which Servlet specification this app uses. Version 6.0 is the Jakarta EE version used with Tomcat 10+.
2
<welcome-file-list>
When someone visits /nimbus/ without a specific page, Tomcat serves index.jsp. Like a default homepage.
3
<servlet> + <servlet-mapping>
Two-part registration. <servlet> gives the Servlet a name and points to its Java class. <servlet-mapping> links that name to a URL pattern. The name is the connector between them.
4
<load-on-startup>1</load-on-startup>
Normally, Servlets are created on first request. This tag forces Tomcat to create the Servlet when the server starts. Lower numbers load first. Good for Servlets that need initialization time.
5
<error-page>
Custom error pages. Instead of Tomcat's ugly default error screen, users see your styled error page. You can map by HTTP status code or by Java exception type.
6
<session-config>
Sets the default session timeout (in minutes). After 30 minutes of inactivity, the user's session is destroyed and they need to log in again.

Key Elements Reference

ElementPurposeExample
<welcome-file> Default page when no file specified index.jsp
<servlet> Register a Servlet with a name Name + fully qualified class
<servlet-mapping> Map a Servlet name to a URL /login, /api/*
<error-page> Custom error page for status codes or exceptions 404 → /WEB-INF/error/404.jsp
<session-config> Session timeout in minutes 30 minutes
<context-param> App-wide configuration values Database URL, app name
Section 06

Web App Directory Structure

Every Java web application follows a strict directory structure. Tomcat expects files in specific locations. Put them in the wrong place and they won't work.

The Standard Structure

nimbus/                          ← Context root (app name)
│
├── index.jsp                   ← Welcome page (public)
├── css/
│   └── style.css                ← Stylesheets (public)
├── js/
│   └── app.js                   ← JavaScript (public)
├── images/
│   └── logo.png                 ← Images (public)
│
├── WEB-INF/                     ← PROTECTED (not accessible from browser)
│   ├── web.xml                  ← Deployment descriptor
│   ├── classes/                 ← Compiled .class files
│   │   └── com/nimbus/
│   │       ├── servlet/
│   │       │   ├── LoginServlet.class
│   │       │   └── BookingServlet.class
│   │       ├── dao/
│   │       │   └── UserDAO.class
│   │       ├── model/
│   │       │   └── User.class
│   │       └── service/
│   │           └── BookingService.class
│   ├── lib/                     ← JAR dependencies
│   │   ├── mysql-connector-j-8.x.jar
│   │   └── jstl-2.0.jar
│   └── login.jsp                ← JSP pages (protected)
│   └── dashboard.jsp
│   └── error/
│       ├── 404.jsp
│       └── 500.jsp
│
└── META-INF/
    └── MANIFEST.MF              ← JAR/WAR metadata
Standard Java web application directory layout

Key Folders Explained

FolderWhat Goes HereAccessible from Browser?
/ (root) Public files: CSS, JS, images, public JSPs Yes — /nimbus/css/style.css
/WEB-INF/ Protected files: web.xml, classes, libraries, private JSPs No — 404 if accessed directly
/WEB-INF/classes/ Compiled .class files (your Servlets, DAOs, models) No
/WEB-INF/lib/ JAR files (MySQL driver, JSTL, any library your app uses) No
/META-INF/ Manifest file, context configuration No
WEB-INF is Your Security Wall

Anything inside /WEB-INF/ is invisible to the browser. A user cannot type http://localhost:8080/nimbus/WEB-INF/web.xml and see your config. Tomcat blocks it. This is why you put JSP pages inside /WEB-INF/ — it forces users to go through your Servlets first, where you can validate input and check login status.

Section 07

Context Path — How URLs Map to Your App

The context path is the first part of the URL after the domain. It tells Tomcat which web application to route the request to.

URL Anatomy

http://localhost:8080/nimbus/login?email=john@test.com
────────────────────── ─────── ────── ───────────────────
       Host + Port        Context  Servlet   Query Parameters
                           Path    Mapping
Each part of the URL routes to a different layer
URL PartHandled ByValue
localhost:8080 Tomcat (Coyote connector) Which server and port
/nimbus Tomcat (context path) Which web app
/login web.xml or @WebServlet Which Servlet
?email=john@test.com Your Servlet code request.getParameter("email")

Where the Context Path Comes From

By default, the context path equals the WAR filename or the folder name in webapps/:

  • webapps/nimbus.war → context path = /nimbus
  • webapps/parcel-tracker.war → context path = /parcel-tracker
  • webapps/ROOT.war → context path = / (no prefix, the default app)
In Nimbus — Context Path = /nimbus

When you deploy nimbus.war to Tomcat, all your URLs start with /nimbus:

  • http://localhost:8080/nimbus/ → welcome page (index.jsp)
  • http://localhost:8080/nimbus/login → LoginServlet
  • http://localhost:8080/nimbus/book → BookingServlet
  • http://localhost:8080/nimbus/track?bookingId=42 → TrackingServlet

Inside your Servlets, you never write /nimbus in URLs. When you do response.sendRedirect("login"), Tomcat automatically prepends the context path.

Multiple Apps on One Tomcat

Tomcat can run many apps simultaneously, each with its own context path:

webapps/
├── nimbus/         →  /nimbus         (parcel management)
├── shopify-clone/  →  /shopify-clone  (e-commerce)
├── blog/           →  /blog           (blog engine)
└── ROOT/           →  /               (default landing page)
Each folder = a separate web application with its own context path

Putting it all together — how to take your Nimbus project, build a WAR, and deploy it to a Tomcat server.

Production Deployment

  1. Build the WAR. In Eclipse: Export → WAR file. Or with Maven: mvn clean package. This creates nimbus.war.
  2. Stop Tomcat on the production server (optional but safer): ./shutdown.sh (Linux) or shutdown.bat (Windows).
  3. Copy WAR to webapps/ folder. Remove the old nimbus/ folder if it exists.
  4. Start Tomcat: ./startup.sh or startup.bat. Tomcat extracts and deploys the WAR.
  5. Verify by visiting http://yourserver:8080/nimbus/.
Don't Forget the Database

Your WAR contains the application code, but not the database. On the production server, you also need MySQL installed, the nimbus_db database created, and the JDBC connection URL in your code pointing to the correct host. In Phase 4, you'll learn about application.properties for environment-specific config.

Section 09

Tomcat Configuration Files

Tomcat has its own configuration files (separate from your app's web.xml) that control server-level behavior like ports, connection limits, and virtual hosts.

Key Config Files

FileLocationWhat It Controls
server.xml conf/server.xml Ports, connectors, virtual hosts, thread pools. The main Tomcat config.
web.xml conf/web.xml Default config for ALL apps. Your app's web.xml overrides these.
context.xml conf/context.xml Default context settings for all apps: session management, JNDI resources.
tomcat-users.xml conf/tomcat-users.xml Users and roles for Tomcat's built-in manager app. Used to deploy WARs via browser.

Changing the Port

conf/server.xml — Connector port 
<!-- Change 8080 to any available port -->
<Connector port="9090"
           protocol="HTTP/1.1"
           connectionTimeout="20000"
           redirectPort="8443"
           maxThreads="200" />

After changing server.xml, you must restart Tomcat for the change to take effect.

Section 10

Practice Exercises

Hands-on tasks to solidify your understanding of web servers, Tomcat, and deployment.

Exercise 1: Setup Tomcat

Download Tomcat 10, configure it in Eclipse, create a new Dynamic Web Project called "nimbus", and verify you can access http://localhost:8080/nimbus/ showing a welcome page.

Goal: Verify your development environment works end-to-end.

Exercise 2: Write a Complete web.xml

Write a web.xml for Nimbus with:

  • Welcome file: index.jsp
  • 5 Servlets registered: Login, Register, Booking, Tracking, History
  • Error pages for 404 and 500
  • Session timeout of 30 minutes
  • A context parameter for the app name: nimbus-parcel-management

Goal: Practice XML syntax and understand every element.

Exercise 3: Build and Deploy a WAR

Export your Nimbus project as a WAR file. Stop the Eclipse Tomcat server. Copy the WAR to a standalone Tomcat's webapps/ folder. Start the standalone Tomcat from the command line. Access your app and verify it works outside Eclipse.

Goal: Understand that your app isn't tied to Eclipse — it runs on any Tomcat.

Exercise 4: Change the Port

Edit server.xml to change Tomcat's port from 8080 to 9090. Restart Tomcat. Access your app at http://localhost:9090/nimbus/. Then change it back to 8080.

Goal: Learn that the port is just config — your app code doesn't change.
