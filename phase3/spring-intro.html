Section 01

Why Spring Exists — The Problem

Before we learn what Spring is, we need to understand the problem it solves. Look at the Nimbus code you already know and see what's broken about it.

How Nimbus Works Right Now

In your current Nimbus application, the NimbusApp main class does everything manually. It creates every object by hand, wires them together by hand, and passes them around by hand:

NimbusApp.java — Current Code (No Spring) 
public class NimbusApp {
    public static void main(String[] args) {
        // Step 1: Manually create DAO objects
        UserDAO userDAO = new UserDAO();
        BookingDAO bookingDAO = new BookingDAO();

        // Step 2: Manually create Service objects, passing DAOs
        UserService userService = new UserService(userDAO);
        TrackingService trackingService = new TrackingService(bookingDAO);
        CustomerTrackingService customerTracking =
            new CustomerTrackingService(bookingDAO);
        OfficerTrackingService officerTracking =
            new OfficerTrackingService(bookingDAO);

        // Step 3: Now use them...
        userService.registerUser("John", "john@mail.com");
        bookingDAO.createBooking(new Booking(...));
    }
}
Line-by-line explanation
3
UserDAO userDAO = new UserDAO();
YOU are creating the UserDAO object. YOU decide exactly which class to use. This is "tight coupling" — your code is married to this specific class.
7
UserService userService = new UserService(userDAO);
YOU are manually wiring the dependency. UserService needs a DAO, and YOU are responsible for giving it one. If UserDAO changes its constructor, YOU fix it here.
8-11
TrackingService, CustomerTrackingService, OfficerTrackingService...
Every single service gets manually created and manually connected. Imagine doing this for 50 services in a real app. Every change = update this file.

What's Wrong With This?

This code works, but it has serious problems that become nightmares as the app grows:

Problem 1: Tight Coupling

new UserDAO() means your code is permanently glued to this exact class. Want to swap it for a UserDAOMongoDB? You'd change every file that says new UserDAO(). In a real project, that could be 10, 20, or 50 files.

Problem 2: Hard to Test

Want to test UserService without a real database? You can't. UserService directly creates or receives a real UserDAO that hits MySQL. In testing, you want a fake (mock) DAO that returns dummy data — but the current design doesn't allow that easily.

Problem 3: Single Point of Explosion

All object creation lives in NimbusApp.main(). As you add more services, more DAOs, more features — this file grows into an unmanageable monster. One change breaks everything.

Problem 4: Database Switch = Rewrite

If your client says "switch from MySQL to PostgreSQL" — you'd need to change DBConnection, every DAO, and every place that creates those DAOs. With Spring, you change one configuration and everything else stays the same.

Analogy: Restaurant vs. Cooking at Home

Without Spring = Cooking at home. YOU go to the store, YOU buy ingredients, YOU prepare everything, YOU cook, YOU serve. If you want a different dish, YOU do everything from scratch.

With Spring = Ordering at a restaurant. You just say "I want pasta." The kitchen (Spring) handles buying ingredients, preparing, cooking, and serving. You just consume the result. Want a different dish? Just change your order — the kitchen handles everything.

In Nimbus

Right now, NimbusApp.java is doing both jobs — it's the "customer" AND the "kitchen." It creates UserDAO, BookingDAO, UserService, TrackingService, connects them all, and then uses them. Spring will take over the "kitchen" part so your main class just says "give me a UserService" and Spring delivers it, fully wired and ready to use.

Section 02

What is Spring Framework?

Spring is an application framework and Inversion of Control container for the Java platform. In plain English: it's a system that creates and manages your objects for you.

The One-Line Definition

Spring is a framework that takes over object creation and wiring so you can focus on business logic.

Instead of you writing new UserDAO() and new UserService(userDAO), you tell Spring "I have a UserDAO class and a UserService class, and UserService needs a UserDAO." Spring creates both objects and connects them automatically.

Spring's Core Concepts

Everything in Spring revolves around three ideas:

Concept What It Means Analogy
IoC (Inversion of Control) Spring controls the creation of objects, not you The restaurant kitchen decides how to cook, not the customer
DI (Dependency Injection) Spring "injects" (gives) objects the other objects they need The waiter brings your sides with the main dish automatically
Beans Objects that Spring manages are called "beans" Menu items — things the kitchen knows how to make

Spring Modules — The Big Picture

Spring is not one thing. It's a collection of modules, each handling a different part of your application. You pick only the modules you need:

Core
IoC container, dependency injection, bean management. The foundation everything else builds on. This is what we're learning today.
Spring MVC
Build web applications. Handles HTTP requests, routes them to controllers, returns HTML/JSON. Replaces raw servlets.
Spring JDBC
Simplifies database access. No more writing try-catch-finally for every query. Replaces raw JDBC boilerplate.
Spring Security
Authentication (login) and authorization (permissions). Handles passwords, sessions, roles. Replaces manual login checks.
Spring Boot
Auto-configuration, embedded server, zero XML. Makes Spring easy to start with. Built ON TOP of Spring Framework.

Spring Framework vs. Spring Boot

This confuses everyone. They are NOT the same thing:

Spring Framework
  • The core framework — IoC, DI, MVC, JDBC, etc.
  • Requires manual configuration
  • You set up the server (Tomcat) yourself
  • You add dependencies manually in pom.xml
  • More control, more setup work
  • Exists since 2003
Spring Boot
  • A tool built on top of Spring Framework
  • Auto-configures everything with smart defaults
  • Embedded Tomcat — just run the JAR
  • "Starter" dependencies bundle everything
  • Less control, way less setup
  • Exists since 2014
Think of it this way

Spring Framework = a fully equipped kitchen. You have all the tools, but you set everything up yourself.

Spring Boot = a pre-arranged kitchen where the pots are already on the stove, ingredients are pre-measured, and the oven is preheated. Same kitchen, just ready to go.

We learn Spring Framework first because you need to understand what's happening underneath before using Spring Boot's shortcuts.

In Nimbus

We'll start by adding Spring Core to Nimbus (IoC + DI) to manage our DAOs and Services. Then we'll add Spring MVC to replace servlets. Then Spring JDBC to simplify database code. Each module solves one problem.

Section 03

Inversion of Control (IoC)

The single most important concept in Spring. Once you get this, everything else makes sense.

Normal Control: You Create Everything

In normal Java programming, your code is in control. Your code decides when to create objects, which objects to create, and how to connect them:

Normal Control — YOUR code creates objects 
// YOU are the boss. YOU create everything.
UserDAO userDAO = new UserDAO();              // YOU create it
UserService userService = new UserService();  // YOU create it
userService.setUserDAO(userDAO);              // YOU wire them together

Inverted Control: Spring Creates Everything

With IoC, control is inverted. A framework (Spring) takes over object creation. You don't create objects — Spring creates them and gives them to you.

Inverted Control — SPRING creates objects 
// SPRING is the boss. SPRING creates everything.
// You just ASK for what you need:
ApplicationContext context = new AnnotationConfigApplicationContext(AppConfig.class);
UserService userService = context.getBean(UserService.class);
// Spring already created UserDAO and injected it into UserService for you!

Visual: Without IoC vs. With IoC

Without IoC (Current Nimbus)
NimbusApp creates everything
↓
new UserDAO()
 
new BookingDAO()
↓
new UserService(dao)
Your code creates + wires every object
With IoC (Spring Nimbus)
Spring Container creates & wires everything
↓
UserDAO
→
UserService
↓
NimbusApp just uses them
Spring handles creation + wiring automatically

The Container: ApplicationContext

The "Spring Container" is an object called ApplicationContext. Think of it as Spring's object factory. It:

  1. Reads your configuration (annotations or XML) to learn what classes exist
  2. Creates instances of those classes (called beans)
  3. Figures out which beans depend on which other beans
  4. Injects (passes) dependencies automatically
  5. Stores all beans and gives them to you when you ask
Your Configuration @Component, @Service, @Repository
→
ApplicationContext Spring's IoC Container
→
Ready-to-use Beans fully wired objects
You describe your classes → Spring creates & wires them → You use them
Analogy: Grocery Shopping vs. Meal Kit Delivery

Without IoC: You drive to the store, find every ingredient, come home, prep everything, cook everything. You control the whole process.

With IoC: A meal kit arrives at your door with pre-measured, pre-prepped ingredients and instructions. You just assemble and eat. The meal kit company (Spring) decided what goes in the box and how it's organized. Control is inverted — you gave up control of sourcing and prepping.

In Nimbus

The ApplicationContext will replace the top of your NimbusApp.main() method. Instead of 6 lines of new statements, you'll have one line: context.getBean(UserService.class). Spring figures out that UserService needs a UserDAO, creates both, connects them, and gives you the fully-working UserService.

Section 04

Dependency Injection (DI)

IoC is the principle. Dependency Injection is HOW Spring implements that principle. It's the mechanism Spring uses to give objects what they need.

What is a "Dependency"?

A dependency is an object that another object needs to do its job. Simple as that.

Understanding Dependencies 
public class UserService {
    private UserDAO userDAO;  // UserService DEPENDS on UserDAO

    // Without UserDAO, UserService can't save users,
    // can't find users, can't do anything useful.
    // UserDAO is a DEPENDENCY of UserService.

    public void registerUser(String name, String email) {
        User user = new User(name, email);
        userDAO.save(user);  // Needs UserDAO to work!
    }
}

"Dependency Injection" means: instead of UserService creating its own UserDAO, someone else (Spring) creates the UserDAO and "injects" (passes) it into UserService.

Three Types of Dependency Injection

Spring can inject dependencies in three ways. Here they are, from best to worst:

1. Constructor Injection (RECOMMENDED)

Dependencies are passed through the constructor. This is the best practice because:

  • The object is never in an invalid state — it gets all dependencies when created
  • Dependencies can be final — they can't be changed after creation
  • Easy to test — just pass mock objects in the constructor
  • Makes it obvious what dependencies a class needs
Constructor Injection — The Right Way 
@Service
public class UserService {
    private final UserDAO userDAO;  // final = can't change after creation

    @Autowired  // Spring: "inject a UserDAO here when creating this object"
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }

    public void registerUser(String name, String email) {
        User user = new User(name, email);
        userDAO.save(user);
    }
}
Line-by-line explanation
1
@Service
Tells Spring: "This class is a service bean. Create an instance and manage it."
3
private final UserDAO userDAO;
final means once the constructor sets this value, it can never change. This guarantees the dependency stays the same for the life of the object. Safe and predictable.
5
@Autowired
Tells Spring: "When you create UserService, look for a UserDAO bean and pass it into this constructor." Spring finds the UserDAO, creates it first (if not already created), then passes it here.
6
public UserService(UserDAO userDAO)
The constructor receives the dependency. Spring calls this constructor automatically — you never write new UserService(dao) yourself.
Note

If a class has only one constructor, Spring 4.3+ doesn't even require @Autowired on it. Spring automatically uses the single constructor. But it's good practice to add it for clarity when learning.

2. Setter Injection

Dependencies are passed through setter methods. Used when dependencies are optional or might change at runtime.

Setter Injection 
@Service
public class UserService {
    private UserDAO userDAO;  // NOT final — can change later

    @Autowired
    public void setUserDAO(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
    // Spring calls this setter AFTER creating the UserService object
}

Downside: The object exists in an incomplete state between creation and setter call. Someone could call a method before the dependency is set — NullPointerException.

3. Field Injection (NOT RECOMMENDED)

Dependencies are injected directly into the field. No constructor, no setter.

Field Injection — Easy but Bad 
@Service
public class UserService {
    @Autowired                // Spring injects directly into the field
    private UserDAO userDAO;  // No constructor, no setter needed

    // Looks clean, but...
}
Why Field Injection is Bad
  • Can't use final — the field must be mutable
  • Hard to test — in unit tests, you can't easily pass a mock because there's no constructor or setter
  • Hides dependencies — looking at the class, it's not obvious what it needs without reading every field
  • Spring-specific — ties your code to Spring's internals (uses reflection)

You'll see @Autowired on fields in tutorials and legacy code. Always use constructor injection in new code.

Before vs. After: Converting Nimbus

Before Spring (Manual DI)
// NimbusApp.java
UserDAO dao = new UserDAO();
UserService svc = new UserService(dao);
svc.registerUser("John", "john@mail.com");

// Problems:
// - NimbusApp knows about UserDAO
// - NimbusApp creates everything
// - Change UserDAO = change NimbusApp
// - Can't swap implementations
After Spring (Automatic DI)
// NimbusApp.java
ApplicationContext ctx =
  new AnnotationConfigApplicationContext(
    AppConfig.class);
UserService svc = ctx.getBean(UserService.class);
svc.registerUser("John", "john@mail.com");

// Benefits:
// - NimbusApp doesn't know about UserDAO
// - Spring creates + wires everything
// - Change UserDAO = change nothing here
// - Swap implementations with config
In Nimbus

Every service in Nimbus uses constructor injection. UserService gets UserDAO through its constructor. TrackingService gets BookingDAO through its constructor. CustomerTrackingService and OfficerTrackingService each get BookingDAO through their constructors. Spring creates one BookingDAO instance and shares it with all three services (singleton scope).

Section 05

Beans

A "bean" is just an object that Spring creates and manages. That's it. Any object managed by Spring's container = bean.

What Makes an Object a Bean?

Any regular Java object becomes a bean when Spring manages it. The difference:

Regular Java Object Spring Bean
Created by You (using new) Spring Container
Managed by You Spring Container
Lifecycle You handle creation & cleanup Spring handles creation, init, & cleanup
Dependencies You wire them manually Spring injects them automatically
Instance count New object every time you call new One shared instance by default (singleton)

Bean Lifecycle

When Spring creates a bean, it follows a specific lifecycle. Understanding this helps you hook into the right stage:

1. Instantiate Spring calls constructor
→
2. Populate Inject dependencies
→
3. Init @PostConstruct runs
↓
5. Destroy @PreDestroy runs
←
4. In Use Application runs
Full bean lifecycle from birth to death
Bean Lifecycle Hooks 
@Repository
public class UserDAO {
    private Connection connection;

    @PostConstruct   // Runs AFTER the bean is created and dependencies injected
    public void init() {
        System.out.println("UserDAO initialized — opening DB connection");
        this.connection = DBConnection.getConnection();
    }

    @PreDestroy      // Runs BEFORE the bean is destroyed (app shutdown)
    public void cleanup() {
        System.out.println("UserDAO shutting down — closing DB connection");
        if (connection != null) connection.close();
    }
}
Line-by-line explanation
1
@Repository
Marks this as a data-access bean. Spring creates and manages it.
5-8
@PostConstruct + init()
This method runs right AFTER Spring creates the bean and injects all dependencies. Perfect for setup tasks like opening database connections, loading config files, or warming caches.
10-13
@PreDestroy + cleanup()
This method runs just BEFORE Spring destroys the bean (when the app shuts down). Perfect for cleanup: closing connections, flushing buffers, releasing resources.

Bean Scopes

Scope controls how many instances of a bean Spring creates. The default is singleton (one shared instance).

Scope How Many Instances? When to Use Nimbus Example
singleton ONE instance for the entire app (default) Services, DAOs, configs — anything stateless UserService, UserDAO, BookingDAO
prototype NEW instance every time you ask for it Objects with state, temporary objects Booking object if it were a bean
request ONE per HTTP request Web apps only — per-request data A request-specific tracking context
session ONE per user session Web apps only — per-user data Shopping cart, logged-in user info
Setting Bean Scope 
@Service
@Scope("singleton")  // Default — one instance shared everywhere
public class UserService { ... }

@Component
@Scope("prototype")  // New instance every time you call getBean()
public class BookingForm { ... }
In Nimbus

Almost everything in Nimbus is singleton. You only need one UserDAO, one BookingDAO, one UserService. They don't hold user-specific state — they're reusable by everyone. The User and Booking model classes are NOT beans — they're plain data objects you create with new as needed.

Rule of Thumb

If a class provides functionality (services, DAOs, controllers) → make it a singleton bean.

If a class holds data (User, Booking, DTO) → keep it as a regular object, don't make it a bean.

Section 06

Configuration

Spring needs to know which classes are beans, what their dependencies are, and how to wire them. You tell Spring this through configuration. There are three ways — we'll cover all three.

1. XML Configuration (Old Way — Legacy)

The original way to configure Spring. You write an XML file that lists every bean. You'll see this in older projects and tutorials, but don't use it in new code.

applicationContext.xml — XML Configuration 
<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
       http://www.springframework.org/schema/beans/spring-beans.xsd">

    <!-- Define UserDAO bean -->
    <bean id="userDAO" class="com.nimbus.dao.UserDAO" />

    <!-- Define UserService bean, inject UserDAO via constructor -->
    <bean id="userService" class="com.nimbus.service.UserService">
        <constructor-arg ref="userDAO" />
    </bean>

</beans>
Why XML Configuration is Obsolete
  • Verbose — every bean requires 2-3 lines of XML
  • No type safety — typos in class names only fail at runtime
  • No IDE autocomplete — it's just a text file
  • Separated from code — have to switch between Java files and XML files

Modern Spring uses annotations and Java config classes instead.

2. Annotation-Based Configuration (Modern Way)

The modern approach: put annotations directly on your Java classes. Spring scans your packages, finds annotated classes, and creates beans automatically.

Annotation-Based — Just annotate your classes 
// Just add @Repository and Spring discovers it automatically
@Repository
public class UserDAO {
    public void save(User user) { /* ... */ }
    public User findById(int id) { /* ... */ }
}

// Just add @Service and Spring discovers it automatically
@Service
public class UserService {
    private final UserDAO userDAO;

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
}

That's it. No XML. No separate config file for these beans. Spring sees @Repository and @Service and knows these are beans.

3. Java-Based Configuration (@Configuration + @Bean)

Sometimes you need more control than annotations give you. Maybe you want to create a bean from a third-party class you can't annotate. Use a @Configuration class with @Bean methods:

AppConfig.java — Java Configuration Class 
@Configuration
@ComponentScan("com.nimbus")  // "Scan this package for @Component, @Service, etc."
public class AppConfig {

    // You can also define beans manually here
    @Bean
    public DataSource dataSource() {
        // This creates a bean for a class you can't annotate
        // (because it's from a library, not your code)
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setUrl("jdbc:mysql://localhost:3306/nimbus");
        ds.setUsername("root");
        ds.setPassword("password");
        return ds;
    }
}
Line-by-line explanation
1
@Configuration
Tells Spring: "This is a config class. Look for @Bean methods inside."
2
@ComponentScan("com.nimbus")
Tells Spring: "Scan the com.nimbus package and all sub-packages. Any class with @Component, @Service, @Repository, or @Controller — make it a bean." This is how Spring auto-discovers your annotated classes.
5-6
@Bean public DataSource dataSource()
The @Bean annotation says: "The return value of this method is a bean. Register it in the container." The method name (dataSource) becomes the bean's name. Use this for third-party classes you can't add @Component to.

Component Scanning: How Spring Finds Your Beans

When you write @ComponentScan("com.nimbus"), here's what happens at startup:

Spring starts reads @ComponentScan
→
Scans com.nimbus.* all sub-packages
→
Finds annotations @Component, @Service, etc.
↓
Ready to use! all wired up
←
Creates & wires beans resolves dependencies
Spring's startup process — scan → discover → create → wire
In Nimbus

We use annotation-based config + @ComponentScan as the primary approach. Classes in com.nimbus.dao get @Repository, classes in com.nimbus.service get @Service. The AppConfig class has @ComponentScan("com.nimbus") which discovers everything automatically. The only @Bean method we might need is for DataSource since it's a library class.

Section 07

Core Annotations

These are the annotations you'll use every day. For each one: what it does, where to put it, and how it maps to Nimbus.

@Component — Generic Bean

The base annotation. Tells Spring: "This class is a bean. Create and manage an instance."

@Component 
@Component
public class EmailNotifier {
    public void send(String to, String message) {
        // Send notification email
    }
}

When to use: When a class doesn't fit into service, DAO, or controller categories. It's the generic "make this a bean" annotation.

@Service — Business Logic Layer

A specialized @Component for business logic classes. Functionally identical to @Component but communicates intent: "this class contains business rules."

@Service — Nimbus Services 
@Service
public class UserService {
    private final UserDAO userDAO;

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }

    public User registerUser(String name, String email) {
        // Business logic: validate, create, save
        if (name == null || name.isEmpty()) {
            throw new InvalidInputException("Name is required");
        }
        User user = new User(name, email);
        userDAO.save(user);
        return user;
    }
}

@Service
public class TrackingService {
    private final BookingDAO bookingDAO;

    @Autowired
    public TrackingService(BookingDAO bookingDAO) {
        this.bookingDAO = bookingDAO;
    }

    public String getStatus(int bookingId) {
        Booking booking = bookingDAO.findById(bookingId);
        return booking.getStatus();
    }
}
In Nimbus

All of these get @Service: UserService, TrackingService, CustomerTrackingService, OfficerTrackingService. They contain the business rules for each role.

@Repository — Data Access Layer

A specialized @Component for database/data-access classes. Does something extra: translates database exceptions into Spring's DataAccessException hierarchy, so you get consistent error handling.

@Repository — Nimbus DAOs 
@Repository
public class UserDAO {
    public void save(User user) {
        // JDBC code to INSERT into users table
    }

    public User findById(int id) {
        // JDBC code to SELECT from users table
    }

    public List<User> findAll() {
        // JDBC code to SELECT all users
    }
}

@Repository
public class BookingDAO {
    public void createBooking(Booking booking) {
        // JDBC code to INSERT into bookings table
    }

    public Booking findById(int id) {
        // JDBC code to SELECT from bookings table
    }

    public List<Booking> findByUserId(int userId) {
        // JDBC code to SELECT bookings for a user
    }
}
In Nimbus

UserDAO and BookingDAO both get @Repository. They talk to the MySQL database. The @Repository annotation also catches JDBC exceptions (like SQLException) and wraps them in Spring's exception hierarchy, making error handling cleaner.

@Controller — Web Layer

A specialized @Component for Spring MVC controllers that handle HTTP requests. We'll cover this properly in the Spring MVC topic.

@Controller — Preview (covered in Spring MVC)

public class BookingController {
    private final UserService userService;

The Stereotype Annotation Hierarchy

These four annotations form a hierarchy. @Component is the parent. The other three are specializations:

@Component Generic bean
↓ specializes into
@Service Business logic
 
@Repository Data access
 
@Controller Web layer
All four are beans. Use the specific one for your layer.
Why Not Just Use @Component Everywhere?

Technically you could. @Service, @Repository, and @Controller are all @Component under the hood. But using the specific annotation:

  • Communicates intent — anyone reading the code knows the class's role
  • Enables special behavior — @Repository translates exceptions, @Controller enables request mapping
  • Follows convention — every Spring developer expects these patterns

@Autowired — Inject Dependency

Tells Spring to find a matching bean and inject it. Can be placed on constructors, setters, or fields (constructor is best).

@Autowired — All Three Placements 
// 1. Constructor (BEST)
@Service
public class UserService {
    private final UserDAO userDAO;

    @Autowired
    public UserService(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
}

// 2. Setter (for optional dependencies)
@Service
public class UserService {
    private UserDAO userDAO;

    @Autowired
    public void setUserDAO(UserDAO userDAO) {
        this.userDAO = userDAO;
    }
}

// 3. Field (DON'T use in new code)
@Service
public class UserService {
    @Autowired
    private UserDAO userDAO;
}

@Configuration — Config Class

Marks a class as a configuration source. Spring reads this class for @Bean definitions and @ComponentScan directives.

@Configuration 
@Configuration
@ComponentScan("com.nimbus")
public class AppConfig {
    // This class itself is a bean too!
    // Spring treats it specially because of @Configuration
}

@Bean — Manual Bean Definition

Used inside @Configuration classes. The method's return value becomes a bean. The method name becomes the bean's name.

@Bean — For third-party classes 
@Configuration
public class AppConfig {

    @Bean  // Method name "dataSource" = bean name
    public DataSource dataSource() {
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setUrl("jdbc:mysql://localhost:3306/nimbus");
        ds.setUsername("root");
        ds.setPassword("password");
        return ds;  // This returned object is the bean
    }

    @Bean
    public JdbcTemplate jdbcTemplate(DataSource dataSource) {
        // Spring sees this method needs a DataSource parameter
        // It automatically passes the dataSource bean from above!
        return new JdbcTemplate(dataSource);
    }
}
@Component vs. @Bean

@Component goes on YOUR classes — classes you wrote, where you can add annotations.

@Bean goes in a @Configuration class — for third-party/library classes you CAN'T annotate.

Example: You can add @Repository to your UserDAO because you wrote it. But you can't add @Component to DriverManagerDataSource because it's from Spring's library. So you use @Bean in a config class.

@Scope — Bean Scope

Controls how many instances Spring creates. Covered in the Beans section above. Quick reference:

@Scope 
@Service
@Scope("singleton")   // Default. One instance for the whole app.
public class UserService { }

@Component
@Scope("prototype")   // New instance each time it's requested.
public class ReportGenerator { }

@PostConstruct / @PreDestroy — Lifecycle Hooks

Run code after bean creation or before bean destruction. Covered in the Beans lifecycle section above.

Lifecycle Annotations 
import jakarta.annotation.PostConstruct;
import jakarta.annotation.PreDestroy;

@Repository

@Qualifier — Choosing Between Multiple Beans

What if you have two classes that implement the same interface? Spring doesn't know which one to inject. @Qualifier tells it exactly which one:

@Qualifier — Resolving Ambiguity 
// Two implementations of the same interface
public interface NotificationService {
    void send(String message);
}

@Service("emailNotifier")
public class EmailNotificationService implements NotificationService {
    public void send(String message) { /* send email */ }
}

@Service("smsNotifier")
public class SmsNotificationService implements NotificationService {
    public void send(String message) { /* send SMS */ }
}

// Now, which one should Spring inject?
@Service
public class UserService {
    private final NotificationService notifier;

    @Autowired
    public UserService(@Qualifier("emailNotifier") NotificationService notifier) {
        // Spring now knows: use the EmailNotificationService bean
        this.notifier = notifier;
    }
}
Line-by-line explanation
6
@Service("emailNotifier")
Gives this bean a specific name: "emailNotifier". Without the name, the default would be the class name in camelCase: "emailNotificationService".
21
@Qualifier("emailNotifier")
Spring sees two NotificationService beans and doesn't know which to inject. @Qualifier says: "I want the one named emailNotifier." Without this, Spring throws a NoUniqueBeanDefinitionException.
In Nimbus

You'll use @Qualifier if you create multiple tracking service implementations. For example, if CustomerTrackingService and OfficerTrackingService both implement a common TrackingService interface, a controller would need @Qualifier to specify which one it wants.

@Value — Inject Values from Properties

Injects values from application.properties (or environment variables) into your beans. Useful for database URLs, API keys, feature flags — anything that changes between environments.

application.properties 
nimbus.db.url=jdbc:mysql://localhost:3306/nimbus
nimbus.db.username=root
nimbus.app.name=Nimbus Parcel Management
nimbus.max-bookings-per-day=50
Using @Value in a Bean 
@Configuration
public class AppConfig {

    @Value("${nimbus.db.url}")
    private String dbUrl;

    @Value("${nimbus.db.username}")
    private String dbUsername;

    @Value("${nimbus.max-bookings-per-day}")
    private int maxBookings;  // Spring converts String "50" to int

    @Bean
    public DataSource dataSource() {
        DriverManagerDataSource ds = new DriverManagerDataSource();
        ds.setUrl(dbUrl);       // Uses value from properties file
        ds.setUsername(dbUsername);
        return ds;
    }
}
In Nimbus

The current DBConnection class has the database URL, username, and password hardcoded. With @Value, those move to application.properties. Deploy to production? Just change the properties file — no code changes.

Annotation Where Purpose
@Component Class Generic bean
@Service Class Business logic bean
@Repository Class Data access bean + exception translation
@Controller Class Web request handler
@Autowired Constructor / Setter / Field Inject a dependency
@Configuration Class Declares config class with @Bean methods
@Bean Method (in @Configuration) Register return value as a bean
@ComponentScan Class (in @Configuration) Tell Spring which packages to scan
@Scope Class or Method Set bean scope (singleton, prototype, etc.)
@PostConstruct Method Run after bean creation
@PreDestroy Method Run before bean destruction
@Qualifier Parameter / Field Choose between multiple beans of same type
@Value Field / Parameter Inject value from properties file
Section 08

ApplicationContext

The ApplicationContext is the heart of Spring. It's the container that creates, stores, and serves all your beans.

What is ApplicationContext?

Think of it as a giant HashMap where:

  • Keys = bean names or types (like "userService" or UserService.class)
  • Values = fully-created, fully-wired bean instances

When you ask context.getBean(UserService.class), Spring looks up UserService in its map and returns the ready-to-use instance.

How to Create an ApplicationContext

For annotation-based configuration (which we use), the implementation is AnnotationConfigApplicationContext:

Creating the Container 
// Pass your @Configuration class to the constructor
ApplicationContext context =
    new AnnotationConfigApplicationContext(AppConfig.class);

// Spring now:
// 1. Reads AppConfig.class
// 2. Finds @ComponentScan("com.nimbus")
// 3. Scans all classes in com.nimbus.*
// 4. Creates beans for @Component, @Service, @Repository classes
// 5. Resolves and injects all @Autowired dependencies
// 6. Calls all @PostConstruct methods
// 7. All beans are now ready to use!

Getting Beans from the Context

Getting Beans — Three Ways 
// 1. By type (RECOMMENDED)
UserService userService = context.getBean(UserService.class);

// 2. By name (returns Object, needs casting)
UserService userService = (UserService) context.getBean("userService");

// 3. By name + type (name + no casting needed)
UserService userService = context.getBean("userService", UserService.class);

Always use method 1 (by type) unless you have multiple beans of the same type and need to pick one by name.

Full Working Example with Nimbus

NimbusApp.java — Spring Version 
import org.springframework.context.ApplicationContext;
import org.springframework.context.annotation.AnnotationConfigApplicationContext;

public class NimbusApp {
    public static void main(String[] args) {
        // 1. Start the Spring container
        ApplicationContext context =
            new AnnotationConfigApplicationContext(AppConfig.class);

        // 2. Get beans from the container (no 'new' keyword!)
        UserService userService = context.getBean(UserService.class);
        TrackingService trackingService = context.getBean(TrackingService.class);

        // 3. Use them — they're fully wired with their dependencies
        userService.registerUser("John", "john@mail.com");

        String status = trackingService.getStatus(1001);
        System.out.println("Booking status: " + status);

        // 4. Close the container (triggers @PreDestroy methods)
        ((AnnotationConfigApplicationContext) context).close();
    }
}
Line-by-line explanation
7-8
new AnnotationConfigApplicationContext(AppConfig.class)
This single line triggers the entire Spring boot-up: scans packages, creates ALL beans, injects ALL dependencies, calls ALL @PostConstruct methods. When this line finishes, every bean is ready to use.
11
context.getBean(UserService.class)
Asks the container: "Give me the UserService bean." Spring returns the singleton instance, which already has UserDAO injected into it. No new, no manual wiring.
15
userService.registerUser("John", "john@mail.com")
Using the bean normally. It works exactly like before — the only difference is who created it (Spring) and who wired its dependencies (Spring).
21
context.close()
Shuts down the container. This triggers every bean's @PreDestroy method (close DB connections, flush caches, etc.). In a web app, the server handles this automatically. In a console app like Nimbus, you call it manually.
In Nimbus

This is the only file that changes significantly. The main() method goes from 6+ lines of new statements to 2 lines: create the context, get the beans. All the DAO and Service classes just get annotations added — their internal logic stays exactly the same.

Key Takeaway

Adopting Spring does NOT mean rewriting your app. Your business logic and database code stay exactly the same. You add a few annotations, create one config class, and change how the main class starts up. That's it. The real power shows when you start adding features — swapping implementations, adding AOP, using Spring MVC — because the foundation is now flexible.

What Spring Does at Startup (Visual)

When new AnnotationConfigApplicationContext(AppConfig.class) runs, here's the complete sequence:

1. Read AppConfig finds @ComponentScan
→
2. Scan com.nimbus.* finds all annotations
↓
3. Create Beans UserDAO, BookingDAO first
→
4. Inject Dependencies pass DAOs into Services
↓
5. Run @PostConstruct initialization logic
→
6. Container Ready! all beans available
Spring resolves dependency order automatically — DAOs are created before Services that need them

Self-Check Questions

After completing the exercises, you should be able to answer:

  1. What problem does IoC solve? (Answer: tight coupling + manual object management)
  2. What's the difference between @Component and @Service? (Answer: @Service is a specialized @Component that indicates business logic)
