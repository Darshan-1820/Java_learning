Section 01

What is JSP?

JSP (JavaServer Pages) is an HTML page that can contain Java code. Tomcat takes this page and converts it into a Servlet behind the scenes. The result: you write HTML like normal, but you can insert dynamic data wherever you need it.

Analogy

HTML is a letter template — it looks the same every time. JSP lets you fill in the blanks dynamically. The template says "Dear ____," and JSP fills in the customer's name from the database. Same template, different result for every person.

Why JSP Exists

In the Servlets topic, you saw how painful it is to generate HTML from Java code. Every line of HTML needs out.println(). Imagine writing an entire booking page that way — hundreds of out.println("<tr><td>" + booking.getId() + "</td></tr>") lines. Unreadable. Unmaintainable.

JSP flips the approach. Instead of writing Java that outputs HTML, you write HTML that contains bits of Java. The result looks like a normal web page with a few special tags mixed in.

Servlet approach (painful)
out.println("<h1>Welcome</h1>");
out.println("<p>Hello, " + name + "</p>");
out.println("<table>");
for (Booking b : list) {
  out.println("<tr><td>" + b.getId() + "</td></tr>");
}
JSP approach (natural)
<h1>Welcome</h1>
<p>Hello, ${user.fullName}</p>
<table>
  <c:forEach items="${bookings}" var="b">
    <tr><td>${b.bookingId}</td></tr>
  </c:forEach>
</table>

JSP = Servlet (Under the Hood)

This is the key fact: every JSP file gets compiled into a Servlet by Tomcat. When you create login.jsp, Tomcat converts it into login_jsp.java, compiles it into login_jsp.class, and runs it as a regular Servlet. JSP is not a different technology — it is a convenient way to write Servlets.

In Nimbus

We are converting our console application into a web application. Every System.out.println() that showed information to the user becomes a JSP page. The login menu becomes login.jsp, the booking flow becomes booking.jsp, the tracking display becomes tracking.jsp. The Servlet handles the logic (validate user, query database), and the JSP handles the display (show the form, show the results).

Section 02

JSP Lifecycle

When a browser requests a JSP page for the first time, Tomcat goes through a multi-step process to turn that .jsp file into executable code. After the first request, the compiled version is reused until you modify the file.

login.jsp .jsp file
→
Translation JSP → .java
→
Compilation .java → .class
→
Loading Class loaded into memory
↓
jspInit() One-time setup
→
_jspService() Handles every request
→
jspDestroy() Cleanup on shutdown
JSP Lifecycle: Translation → Compilation → Loading → Init → Service → Destroy

Each Phase Explained

  1. Translation — Tomcat reads login.jsp and generates a Java file called login_jsp.java. All your HTML becomes out.write() statements. All your Java snippets stay as Java code. This happens only once (or when the JSP file changes).
  2. Compilation — The generated .java file is compiled into a .class file, just like any Java program. This is why the first request to a JSP page is slightly slower — Tomcat is compiling it on the fly.
  3. Loading — The compiled class is loaded into memory by Tomcat's class loader. The class extends HttpJspBase (which itself extends HttpServlet). This proves JSP IS a servlet.
  4. jspInit() — Called once when the servlet is first created. You can override this to set up resources. Rarely used in practice.
  5. _jspService(request, response) — Called for every request. This is where all your HTML and Java code ends up. You do NOT override this — Tomcat generates it from your JSP file.
  6. jspDestroy() — Called when Tomcat shuts down or the JSP is removed. Used for cleanup. Also rarely used.
Key Insight

The first request to a JSP is slower because Tomcat has to translate and compile it. Every subsequent request reuses the compiled class and is fast. If you edit the .jsp file, Tomcat detects the change and retranslates it on the next request.

Where Does Tomcat Put the Generated Files?

Tomcat stores the generated Java and class files in its work directory. For a project called nimbus, you would find them at:

Tomcat work directory 
TOMCAT_HOME/work/Catalina/localhost/nimbus/org/apache/jsp/
  login_jsp.java      <-- Generated servlet source
  login_jsp.class     <-- Compiled servlet

You never need to touch these files. But if you are curious what your JSP looks like as a Servlet, open the .java file — you will see every line of your HTML wrapped in out.write() calls.

Section 03

JSP Directives

Directives are instructions to Tomcat about how to process the JSP page. They go at the top of the file and do not produce any output. Think of them as configuration settings for the page.

All directives use the syntax: <%@ directive attribute="value" %>

1. Page Directive — <%@ page %>

Controls page-level settings. This is the most common directive and almost every JSP file starts with one.

login.jsp — page directive 
<%@ page contentType="text/html;charset=UTF-8" language="java" %>
<%@ page import="java.util.List, com.nimbus.model.Booking" %>
<%@ page errorPage="error.jsp" %>
Page Directive Attributes
1
contentType="text/html;charset=UTF-8"
Tells the browser this page is HTML with UTF-8 encoding. Same as calling response.setContentType() in a Servlet.
2
language="java"
Specifies that Java is the scripting language. This is the default, so you can omit it. No other language is supported in practice.
3
import="java.util.List, com.nimbus.model.Booking"
Imports Java classes so you can use them in the page. Same as import statements in a Java file. Comma-separated for multiple imports.
4
errorPage="error.jsp"
If an exception occurs on this page, Tomcat automatically forwards to error.jsp instead of showing a stack trace to the user.
AttributePurposeExample
contentTypeMIME type of the response"text/html;charset=UTF-8"
importImport Java classes"java.util.List"
errorPagePage to redirect to on error"error.jsp"
isErrorPageMarks this page as an error handler"true"
sessionWhether page participates in session"true" (default)
isELIgnoredDisable Expression Language"false" (default)

2. Include Directive — <%@ include %>

Inserts the contents of another file into this JSP at compile time. The included file becomes part of this page — as if you copy-pasted it in.

customer-dashboard.jsp — include directive 
<%@ include file="header.jsp" %>

<h1>Customer Dashboard</h1>
<p>Welcome, ${sessionScope.user.fullName}</p>

<%@ include file="footer.jsp" %>

Static include means the content is merged at translation time (when JSP becomes Java). If you change header.jsp, you need to retranslate all pages that include it.

3. Taglib Directive — <%@ taglib %>

Declares a tag library so you can use custom tags in your JSP. The most important one is JSTL (covered in detail in Section 6).

history.jsp — taglib directive 
<%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>
<%@ taglib prefix="fmt" uri="http://java.sun.com/jsp/jstl/fmt" %>
Taglib Directive
1
prefix="c"
The shorthand name you use in your JSP. With prefix "c", you write <c:forEach>, <c:if>, etc. You could use any prefix, but "c" is the convention for JSTL core.
2
uri="http://java.sun.com/jsp/jstl/core"
A unique identifier for the tag library. This is NOT a URL the browser visits — it is just an ID that Tomcat uses to find the right JAR file. The JSTL core library provides <c:if>, <c:forEach>, <c:choose>, etc.
3
prefix="fmt"
The JSTL formatting library. Provides <fmt:formatDate> and <fmt:formatNumber> for displaying dates and currencies properly.
Section 04

Scripting Elements

Scripting elements let you embed raw Java code inside a JSP page. There are three types. You need to know them, but you should NOT use them in real projects (we will explain why).

1. Scriptlet — <% %>

A block of Java code that runs when the page is requested. The code goes directly into the _jspService() method of the generated servlet.

Scriptlet example — tracking.jsp 
<%
  String bookingId = request.getParameter("bookingId");
  List<Booking> results = (List<Booking>) request.getAttribute("results");
  if (results != null && !results.isEmpty()) {
%>
    <table>
      <tr><th>Booking ID</th><th>Status</th></tr>
      <% for (Booking b : results) { %>
        <tr>
          <td><%= b.getBookingId() %></td>
          <td><%= b.getParStatus() %></td>
        </tr>
      <% } %>
    </table>
<% } else { %>
    <p>No results found.</p>
<% } %>

2. Expression — <%= %>

Outputs the value of a Java expression directly into the HTML. Whatever the expression evaluates to gets converted to a String and written to the page.

Expression examples 
<!-- Output a variable value -->
<p>Welcome, <%= user.getFullName() %></p>

<!-- Output a calculation -->
<p>Total: Rs. <%= booking.getParWeight() * ratePerKg %></p>

<!-- Output current date -->
<p>Date: <%= new java.util.Date() %></p>

Notice there is no semicolon inside <%= %>. It is an expression, not a statement. The value is what gets printed.

3. Declaration — <%! %>

Declares methods or instance variables that become part of the servlet class (not the _jspService method). These exist for the entire life of the servlet.

Declaration example 
<%!
  private int visitCount = 0;

  private String formatStatus(String status) {
    switch(status) {
      case "BOOKED": return "Booked";
      case "IN_TRANSIT": return "In Transit";
      case "DELIVERED": return "Delivered";
      default: return status;
    }
  }
%>

<p>Page visited: <%= ++visitCount %> times</p>
<p>Status: <%= formatStatus(booking.getParStatus()) %></p>
Declarations Are Dangerous

Variables declared with <%! %> are shared across all users because they are instance variables of the servlet. That visitCount above counts visits from ALL users combined, not per user. This creates thread-safety issues. Avoid declarations in real code.

Quick Reference: Scripting Elements

ElementSyntaxWhere It Goes in ServletPurpose
Scriptlet<% %>Inside _jspService()Execute Java code
Expression<%= %>Inside out.print()Output a value
Declaration<%! %>Class level (outside any method)Define methods/variables

Why Scriptlets Are Bad Practice

You just learned scriptlets. Now here is why you should never use them in real projects:

  • Mixing logic with view. A JSP file is supposed to display data. The moment you put if statements, loops, and database queries inside a JSP, the file becomes impossible to maintain. Is it HTML? Is it Java? Nobody knows.
  • Not testable. You cannot unit test Java code that is embedded in an HTML file. Servlets and service classes can be tested. Scriptlets cannot.
  • Designers cannot work with it. If a designer needs to change the HTML layout, they have to navigate around Java code blocks. One wrong keystroke breaks the Java, and now a designer is debugging a NullPointerException.
  • Duplication. If three JSP pages need the same logic, you copy-paste the scriptlet into all three. Change the logic? Update all three. Miss one? Bug.
Old way (scriptlets)
<% if (user != null) { %>
  <p>Hello, <%= user.getFullName() %></p>
<% } else { %>
  <p>Please log in</p>
<% } %>
Modern way (EL + JSTL)
<c:choose>
  <c:when test="${not empty user}">
    <p>Hello, ${user.fullName}</p>
  </c:when>
  <c:otherwise>
    <p>Please log in</p>
  </c:otherwise>
</c:choose>
Rule

In Nimbus and in professional Java web development: zero scriptlets in JSP files. Use Expression Language (EL) and JSTL instead. The only reason we teach scriptlets is so you understand them when reading old code or when exam questions ask about them.

Section 05

Expression Language (EL)

Expression Language is the modern, clean way to access data in JSP pages. Instead of writing <%= user.getFullName() %>, you write ${user.fullName}. Shorter, cleaner, and no Java syntax needed.

Basic Syntax: ${}

Everything inside ${ } is an EL expression. Tomcat evaluates it and outputs the result. The EL engine automatically calls getter methods — ${user.fullName} calls user.getFullName() behind the scenes.

EL basics in Nimbus JSP pages 
<!-- Access request attributes (set by servlet) -->
<p>Welcome, ${user.fullName}</p>
<p>Booking ID: ${booking.bookingId}</p>
<p>Status: ${booking.parStatus}</p>
<p>Weight: ${booking.parWeight} kg</p>

<!-- Access session attributes -->
<p>Logged in as: ${sessionScope.user.userId}</p>

<!-- Access request parameters (from URL or form) -->
<p>Search for: ${param.bookingId}</p>

<!-- Access list size -->
<p>Total bookings: ${bookings.size()}</p>

How EL Finds Data

When you write ${user}, EL searches for an attribute named "user" in this order:

  1. Page scope — attributes set with pageContext.setAttribute("user", ...). Only visible within this JSP page.
  2. Request scope — attributes set with request.setAttribute("user", ...). This is the most common one. Servlets set request attributes before forwarding to JSP.
  3. Session scope — attributes set with session.setAttribute("user", ...). Survives across multiple requests for the same user.
  4. Application scope — attributes set with application.setAttribute("user", ...). Shared across all users. Rarely used.

If you want to be explicit about which scope, use the scope prefix:

EL scope prefixes 
<!-- Explicitly specify the scope -->
${requestScope.user.fullName}     <!-- from request.setAttribute() -->
${sessionScope.user.userId}       <!-- from session.setAttribute() -->
${applicationScope.appName}       <!-- from application.setAttribute() -->
${pageScope.tempVar}              <!-- from pageContext.setAttribute() -->

<!-- Access request parameters -->
${param.bookingId}                <!-- from URL: ?bookingId=NB001 -->
${param.userId}                   <!-- from form: <input name="userId"> -->

<!-- Access HTTP headers -->
${header["User-Agent"]}           <!-- browser info -->

<!-- Access cookies -->
${cookie.JSESSIONID.value}        <!-- session cookie value -->
In Nimbus

The Servlet handles the logic and sets data as request attributes. The JSP reads them with EL. For example, CustomerDashboardServlet does request.setAttribute("bookedCount", 5) and then customer-dashboard.jsp displays ${bookedCount}. The JSP never touches the database or runs any business logic.

EL with Java Bean Properties

EL uses the JavaBean convention: ${booking.parStatus} calls booking.getParStatus(). The property name matches the getter name minus "get" with the first letter lowercased.

EL ExpressionJava Method Called
${user.fullName}user.getFullName()
${booking.bookingId}booking.getBookingId()
${booking.parWeight}booking.getParWeight()
${user.admin}user.isAdmin() (boolean getter)

EL Operators

You can do comparisons and arithmetic directly in EL:

OperationSymbolWord FormExample
Equal==eq${booking.parStatus == 'DELIVERED'}
Not equal!=ne${booking.parStatus != 'CANCELLED'}
Less than<lt${booking.parWeight lt 5}
Greater than>gt${booking.parWeight gt 10}
Less or equal<=le${pageNum le totalPages}
Greater or equal>=ge${pageNum ge 1}
And&&and${not empty user and user.admin}
Or||or${empty user or not user.active}

Word Operators

In JSP/HTML, using < and > can cause parsing problems because they look like HTML tags. That is why EL provides word-based alternatives: lt, gt, le, ge, eq, ne. Use the word forms inside JSP attributes to avoid issues.

The empty Operator

One of the most useful EL operators. It returns true if a value is:

  • null
  • An empty String ""
  • An empty Collection (List with 0 elements)
  • An empty Map
  • An empty array
Using empty in Nimbus JSPs 
<!-- Show results only if the list is not empty -->
<c:if test="${not empty bookings}">
  <table>...</table>
</c:if>

<!-- Show error message only if it exists -->
<c:if test="${not empty errorMessage}">
  <div class="error">${errorMessage}</div>
</c:if>

<!-- Show "no results" message -->
<c:if test="${empty results}">
  <p>No tracking results found for this booking ID.</p>
</c:if>
Section 06

JSTL (JSP Standard Tag Library)

JSTL provides ready-made tags for common tasks: loops, conditionals, formatting. Combined with EL, JSTL eliminates the need for any Java scriptlets in your JSP pages.

Adding JSTL to Your Project

JSTL is not built into Tomcat. You need to add it as a dependency.

pom.xml — Maven dependency 
<!-- JSTL for Tomcat 9 (Jakarta EE 8 / javax) -->
<dependency>
    <groupId>javax.servlet</groupId>
    <artifactId>jstl</artifactId>
    <version>1.2</version>
</dependency>

If not using Maven, download the jstl-1.2.jar and place it in WEB-INF/lib/.

At the top of every JSP that uses JSTL tags, add the taglib directive:

Required taglib declarations 
<%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>
<%@ taglib prefix="fmt" uri="http://java.sun.com/jsp/jstl/fmt" %>

<c:out> — Safe Output

Outputs a value with automatic HTML escaping, preventing XSS attacks.

c:out example 
<!-- UNSAFE: If user.fullName contains <script>alert('hack')</script>, it RUNS -->
<p>${user.fullName}</p>

<!-- SAFE: c:out escapes HTML characters -->
<p><c:out value="${user.fullName}"/></p>

<!-- With default value if null -->
<p><c:out value="${user.phone}" default="Not provided"/></p>

<c:set> — Set Variables

Creates or sets a variable within the page.

c:set example 
<!-- Set a simple variable -->
<c:set var="pageTitle" value="Booking History"/>
<title>${pageTitle} - Nimbus</title>

<!-- Calculate a value -->
<c:set var="totalCost" value="${booking.parWeight * 50}"/>
<p>Shipping cost: Rs. ${totalCost}</p>

<!-- Set in a specific scope -->
<c:set var="currentPage" value="3" scope="request"/>

<c:if> — Simple Conditional

Renders its body only if the test condition is true. There is no else — use <c:choose> for if/else.

tracking.jsp — conditional display 
<!-- Show error message if login failed -->
<c:if test="${not empty errorMessage}">
  <div class="alert alert-danger">${errorMessage}</div>
</c:if>

<!-- Show results table only if there are results -->
<c:if test="${not empty trackingResults}">
  <h3>Tracking Results</h3>
  <table>...</table>
</c:if>

<!-- Show admin link only for officers -->
<c:if test="${sessionScope.user.role == 'OFFICER'}">
  <a href="/officer-dashboard">Admin Panel</a>
</c:if>

<c:choose> / <c:when> / <c:otherwise> — If/Else If/Else

Works like a Java switch or if/else-if/else chain.

delivery-status.jsp — status badge colors 
<c:choose>
  <c:when test="${booking.parStatus == 'BOOKED'}">
    <span class="badge badge-info">Booked</span>
  </c:when>
  <c:when test="${booking.parStatus == 'IN_TRANSIT'}">
    <span class="badge badge-warning">In Transit</span>
  </c:when>
  <c:when test="${booking.parStatus == 'DELIVERED'}">
    <span class="badge badge-success">Delivered</span>
  </c:when>
  <c:when test="${booking.parStatus == 'CANCELLED'}">
    <span class="badge badge-danger">Cancelled</span>
  </c:when>
  <c:otherwise>
    <span class="badge">${booking.parStatus}</span>
  </c:otherwise>
</c:choose>
c:choose structure
1
<c:choose>
Container tag. Like the opening of a switch block. Only ONE of the inner <c:when> blocks will execute.
2
<c:when test="${booking.parStatus == 'BOOKED'}">
Like "if" or "case". The test attribute takes an EL expression that evaluates to true/false. If true, this block's content is rendered. Tomcat stops checking remaining <c:when> tags.
3
<c:otherwise>
Like "else" or "default". Runs only if none of the <c:when> conditions matched. Optional but recommended.

<c:forEach> — Loop

Iterates over a collection. This is the JSTL tag you will use the most — every table of bookings, every list of users, every history page uses this.

history.jsp — booking list table 
<table class="table">
  <thead>
    <tr>
      <th>#</th>
      <th>Booking ID</th>
      <th>Sender</th>
      <th>Receiver</th>
      <th>Status</th>
      <th>Date</th>
    </tr>
  </thead>
  <tbody>
    <c:forEach items="${bookings}" var="b" varStatus="loop">
      <tr>
        <td>${loop.count}</td>
        <td>${b.bookingId}</td>
        <td>${b.senderName}</td>
        <td>${b.receiverName}</td>
        <td>${b.parStatus}</td>
        <td><fmt:formatDate value="${b.bookingDate}" pattern="dd-MMM-yyyy"/></td>
      </tr>
    </c:forEach>
  </tbody>
</table>
c:forEach attributes
1
items="${bookings}"
The collection to loop over. This reads the "bookings" attribute from the request (set by the Servlet with request.setAttribute("bookings", bookingList)).
2
var="b"
The loop variable name. Each iteration, "b" holds the current Booking object. You access its properties with ${b.bookingId}, ${b.senderName}, etc.
3
varStatus="loop"
Gives you loop metadata. loop.count = 1-based index. loop.index = 0-based index. loop.first = is this the first item? loop.last = is this the last item?

<c:forEach> with Numbers (Counter Loop)

You can also use <c:forEach> as a counter loop, like a Java for (int i = 1; i <= 10; i++):

history.jsp — pagination links 
<!-- Generate page links: 1 2 3 4 5 -->
<nav class="pagination">
  <c:forEach begin="1" end="${totalPages}" var="pageNum">
    <c:choose>
      <c:when test="${pageNum == currentPage}">
        <span class="page-link active">${pageNum}</span>
      </c:when>
      <c:otherwise>
        <a href="/history?page=${pageNum}" class="page-link">${pageNum}</a>
      </c:otherwise>
    </c:choose>
  </c:forEach>
</nav>

<fmt:formatDate> — Format Dates

Formats a java.util.Date object for display.

invoice.jsp — date formatting 
<!-- Simple date -->
<fmt:formatDate value="${booking.bookingDate}" pattern="dd-MMM-yyyy"/>
<!-- Output: 23-Apr-2026 -->

<!-- Date with time -->
<fmt:formatDate value="${booking.bookingDate}" pattern="dd/MM/yyyy hh:mm a"/>
<!-- Output: 23/04/2026 02:30 PM -->

<!-- Full date -->
<fmt:formatDate value="${booking.bookingDate}" type="both" dateStyle="long"/>
<!-- Output: April 23, 2026 2:30:00 PM -->

<fmt:formatNumber> — Format Numbers and Currency

invoice.jsp — currency formatting 
<!-- Format as currency -->
<fmt:formatNumber value="${totalCost}" type="currency" currencySymbol="Rs. "/>
<!-- Output: Rs. 1,250.00 -->

<!-- Format with decimal places -->
<fmt:formatNumber value="${booking.parWeight}" maxFractionDigits="2"/>
<!-- Output: 3.50 -->

<!-- Format as percentage -->
<fmt:formatNumber value="${discount}" type="percent"/>
<!-- Output: 15% -->

Section 07

Implicit Objects

JSP gives you 9 objects automatically — you do not create them. They are available in every JSP page, ready to use in scriptlets. In modern JSP, you access most of these through EL instead.

ObjectTypeWhat It Gives You
request HttpServletRequest The incoming HTTP request. Access parameters (request.getParameter()), attributes (request.getAttribute()), headers, URL info.
response HttpServletResponse The HTTP response being sent back. Set headers, cookies, status codes, redirects. Rarely used directly in JSP.
session HttpSession The user's session. Store and retrieve data that persists across requests: session.getAttribute("user").
application ServletContext The web application context. Shared across all users and all pages. Used for app-wide settings.
out JspWriter The output stream. What <%= %> writes to. You can call out.println() in scriptlets, but you should not need to.
pageContext PageContext Access to all other implicit objects and page-scoped attributes. Acts as a gateway to everything.
config ServletConfig Configuration for this servlet (JSP). Access init parameters from web.xml. Rarely used.
page Object Reference to the current JSP page instance itself. Equivalent to this. Almost never used.
exception Throwable The exception that occurred. Only available in error pages (where isErrorPage="true").
In Nimbus

The ones you use daily: request (to get data from servlets), session (to check logged-in user), and out (to write output, though EL/JSTL is preferred). The others are there for special cases.

EL Implicit Objects (Different from JSP Implicit Objects)

When you use ${} syntax, EL has its own set of implicit objects. These are not the same as the JSP ones above, but they provide access to similar data:

EL ObjectMaps ToExample

sessionScopeSession attributes${sessionScope.user}
applicationScopeApplication attributes${applicationScope.appName}
pageScopePage attributes${pageScope.tempVar}
paramRequest parameters (single value)${param.bookingId}
paramValuesRequest parameters (multiple)${paramValues.status[0]}

Context Path

${pageContext.request.contextPath} is extremely useful. It gives you the base URL of your application (e.g., /nimbus). Use it in links and form actions so your URLs work regardless of deployment path: <a href="${pageContext.request.contextPath}/booking">

Section 08

How Servlet + JSP Work Together

This is the most important concept. The Servlet handles the logic (Controller). The JSP displays the data (View). The model classes carry the data between them. This is the MVC (Model-View-Controller) pattern.

Browser User clicks "View History"
→
HistoryServlet Controller (doGet)
→
BookingDAO Database query
↓
Browser Sees the HTML table
←
history.jsp View (renders HTML)
←
List<Booking> Model (data objects)
MVC Flow: Browser → Servlet (logic) → DAO (data) → Model (objects) → JSP (display) → Browser

The Complete Flow, Step by Step

  1. Browser sends request. User clicks "View History" which sends a GET request to /history.
  2. Servlet receives request. HistoryServlet.doGet() runs. It gets the logged-in user from the session.
  3. Servlet calls the DAO. BookingDAO.getBookingsByUser(userId) queries the database and returns a List<Booking>.
  4. Servlet sets request attributes. request.setAttribute("bookings", bookingList) attaches the data to the request.
  5. Servlet forwards to JSP. request.getRequestDispatcher("/WEB-INF/views/history.jsp").forward(request, response) passes control to the JSP.
  6. JSP renders HTML. The JSP reads ${bookings} using EL, loops through them with <c:forEach>, and generates an HTML table.
  7. Browser receives HTML. The user sees a table of their bookings. They have no idea a Servlet or JSP was involved — they just see a web page.

Servlet Code (Controller)

HistoryServlet.java 
@WebServlet("/history")
public class HistoryServlet extends HttpServlet {

    private BookingDAO bookingDAO = new BookingDAO();

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // 1. Get logged-in user from session
        User user = (User) request.getSession().getAttribute("user");
        if (user == null) {
            response.sendRedirect("/login");
            return;
        }

        // 2. Get pagination parameters
        int page = 1;
        if (request.getParameter("page") != null) {
            page = Integer.parseInt(request.getParameter("page"));
        }
        int pageSize = 10;

        // 3. Query database through DAO
        List<Booking> bookings = bookingDAO
            .getBookingsByUser(user.getUserId(), page, pageSize);
        int totalBookings = bookingDAO
            .countBookingsByUser(user.getUserId());
        int totalPages = (int) Math.ceil((double) totalBookings / pageSize);

        // 4. Set data as request attributes
        request.setAttribute("bookings", bookings);
        request.setAttribute("currentPage", page);
        request.setAttribute("totalPages", totalPages);

        // 5. Forward to JSP (view)
        request.getRequestDispatcher("/WEB-INF/views/history.jsp")
               .forward(request, response);
    }
}
Servlet → JSP flow
1
request.getSession().getAttribute("user")
Gets the User object stored in the session during login. If null, the user is not logged in — redirect to login page.
2
request.getParameter("page")
Reads the "page" parameter from the URL (e.g., /history?page=2). If not present, defaults to page 1.
3
request.setAttribute("bookings", bookings)
THIS is the bridge between Servlet and JSP. The Servlet attaches data to the request object. The JSP reads it using ${bookings} in EL.
4
request.getRequestDispatcher("/WEB-INF/views/history.jsp").forward(request, response)
Forwards the request to history.jsp. The JSP receives the same request object with all the attributes set. The URL in the browser stays as /history (not /WEB-INF/views/history.jsp). Placing JSP in WEB-INF prevents direct browser access.
Golden Rule

JSP files go in /WEB-INF/views/. Files inside WEB-INF cannot be accessed directly by the browser (typing the URL gives 404). The only way to reach them is through a Servlet forward. This enforces MVC — all requests go through the Servlet first.

JSP Rules (What JSP Should Never Do)

  • Never create database connections
  • Never call DAO methods
  • Never run SQL queries
  • Never contain business logic (validation, calculations)
  • Never import service classes
  • Never use scriptlets

JSP does one thing: take data from request attributes and render HTML. Everything else belongs in the Servlet or service layer.

Form techniques
1
method="POST"
Forms that create or modify data use POST. Search forms use GET (so the search term appears in the URL and can be bookmarked).
2
<input type="hidden" name="step" value="1"/>

value="<c:out value='${param.senderName}'/>"

<c:if test="${not empty errors.senderName}">

2
${pageContext.exception.message}
In EL, the exception object is accessed through pageContext.exception (not directly as "exception"). This gives you the error message without exposing the full stack trace to the user.

Global Error Pages in web.xml

Instead of setting errorPage on every single JSP, configure error pages globally in web.xml:

web.xml — global error pages 
<!-- Handle 404 Not Found -->
<error-page>
    <error-code>404</error-code>
    <location>/WEB-INF/views/404.jsp</location>
</error-page>

<!-- Handle 500 Internal Server Error -->
<error-page>
    <error-code>500</error-code>
    <location>/WEB-INF/views/500.jsp</location>
</error-page>

<!-- Handle all Java exceptions -->
<error-page>
    <exception-type>java.lang.Exception</exception-type>
    <location>/WEB-INF/views/error.jsp</location>
</error-page>
Section 12

Including Other JSPs

Real web applications do not copy the same header and footer HTML into every page. Instead, you create reusable fragments and include them. There are two ways to include, and they work differently.

Static Include (Directive)

<%@ include file="header.jsp" %>

  • Content inserted at compile time
  • Becomes part of the same servlet
  • Faster (no overhead at runtime)
  • Changes to included file require recompilation of the parent
  • Best for: header, footer, navigation that rarely changes
Dynamic Include (Action)

<jsp:include page="header.jsp"/>

  • Content included at runtime
  • Each file is a separate servlet
  • Slightly slower (separate request processing)
  • Changes to included file are picked up automatically
  • Best for: content that changes often, or includes with parameters

Common Layout Pattern

header.jsp — reusable header fragment 
<%@ taglib prefix="c" uri="http://java.sun.com/jsp/jstl/core" %>
<nav class="navbar">
    <a href="${pageContext.request.contextPath}/" class="brand">Nimbus</a>
    <div class="nav-links">
        <c:choose>
            <c:when test="${not empty sessionScope.user}">
                <a href="${pageContext.request.contextPath}/dashboard">Dashboard</a>
                <a href="${pageContext.request.contextPath}/booking">New Booking</a>
                <a href="${pageContext.request.contextPath}/tracking">Track</a>
                <span>Hello, ${sessionScope.user.fullName}</span>
                <a href="${pageContext.request.contextPath}/logout">Logout</a>
            </c:when>
            <c:otherwise>
                <a href="${pageContext.request.contextPath}/login">Login</a>
                <a href="${pageContext.request.contextPath}/register">Register</a>
            </c:otherwise>
        </c:choose>
    </div>
</nav>
footer.jsp — reusable footer fragment 
<footer class="footer">
    <p>&copy; 2026 Nimbus Parcel Management. All rights reserved.</p>
</footer>
Any page using the layout 
<%@ page contentType="text/html;charset=UTF-8" %>
<!DOCTYPE html>
<html>
<head>
    <title>My Page - Nimbus</title>
</head>
<body>
    <%@ include file="header.jsp" %>

    <div class="container">
        <!-- Page-specific content here -->
    </div>

    <%@ include file="footer.jsp" %>
</body>
</html>

Dynamic Include with Parameters

Passing parameters to included page 
<!-- Include with parameters -->
<jsp:include page="sidebar.jsp">
    <jsp:param name="activePage" value="history"/>
</jsp:include>

<!-- In sidebar.jsp, access the parameter: -->
<c:if test="${param.activePage == 'history'}">
    <a href="/history" class="active">History</a>
</c:if>
Section 13

Common Mistakes

Mistakes every beginner makes with JSP — and how to avoid them.

Mistake 1: Using Scriptlets Instead of EL/JSTL

Wrong
<% String name = (String)
   request.getAttribute("name"); %>
<p>Hello, <%= name %></p>
Right
<p>Hello, ${name}</p>

Mistake 2: Not Escaping Output (XSS Vulnerability)

Unsafe
<!-- If name = "<script>alert('hacked')</script>"
     the script RUNS in the browser -->
<p>${user.fullName}</p>
Safe
<!-- c:out escapes HTML characters.
     <script> becomes &lt;script&gt;
     and does NOT execute -->
<p><c:out value="${user.fullName}"/></p>

Mistake 3: Putting Business Logic in JSP

Wrong
<!-- NEVER do this in JSP -->
<%
  Connection conn = DriverManager
    .getConnection("jdbc:mysql://...");
  PreparedStatement ps = conn
    .prepareStatement("SELECT * FROM bookings");
  ResultSet rs = ps.executeQuery();
  while (rs.next()) {
%>
  <tr><td><%= rs.getString("id") %></td></tr>
<% } %>
Right
<!-- Servlet does the DB query,
     JSP only displays results -->
<c:forEach items="${bookings}" var="b">
  <tr><td>${b.bookingId}</td></tr>
</c:forEach>

Mistake 4: Forgetting the Taglib Directive

If you use <c:forEach> or <c:if> but forget to add the taglib directive at the top of the page, the tags are treated as plain text and rendered as-is in the HTML output. Nothing breaks, but nothing works either — you just see the raw tag text on the page.

Mistake 5: Using == for String Comparison in Scriptlets

Wrong (scriptlet)
<!-- == compares references, not values -->
<% if (status == "DELIVERED") { %>
Correct (EL handles it right)
<!-- EL == correctly compares String values -->
<c:if test="${status == 'DELIVERED'}">

In Java scriptlets, == on Strings compares memory references (wrong). You need .equals(). But in EL, == correctly compares String values. One more reason to use EL instead of scriptlets.

Mistake 6: JSP Accessible Directly by Browser

If JSP files are placed in the root of the web application (e.g., /history.jsp), users can access them directly by typing the URL. This bypasses the Servlet, which means no data is loaded and the page shows empty or throws errors. Always place JSPs inside /WEB-INF/.

Section 14

Practice Exercises

Build these to solidify your understanding. Each exercise focuses on a different JSP concept.

Exercise 1 — Registration Form

Create register.jsp with fields for: userId, fullName, email, phone, password, confirmPassword. Use ${param.fieldName} to pre-fill fields on validation failure. Show field-specific errors using a ${errors} map. The form should POST to /register.

Exercise 2 — Officer Dashboard

Create officer-dashboard.jsp that displays ALL bookings (not just the logged-in user's). Use <c:forEach> to build a table showing bookingId, senderName, receiverName, status, and date. Add a <c:choose> block to color-code the status column. Include a "View" link in each row that goes to /booking?id=${b.bookingId}.

Exercise 3 — Conditional Navigation

Create header.jsp that shows different navigation links based on the user's role. If ${sessionScope.user.role == 'CUSTOMER'}, show: Dashboard, New Booking, Track, History. If ${sessionScope.user.role == 'OFFICER'}, show: All Bookings, Update Status, Pickup Management. If no user is logged in, show: Login, Register.

Exercise 4 — Invoice with Formatting

Create invoice.jsp that displays a booking's full details. Use <fmt:formatDate> to show the booking date in "dd MMMM yyyy" format. Use <fmt:formatNumber> to show the total cost as currency with Rs. symbol and two decimal places. Add a print button that calls window.print(). Add a @media print CSS rule to hide the button when printing.

Exercise 5 — Error Page

Create error.jsp with isErrorPage="true". Display a user-friendly message and the exception message (but not the stack trace). Add a "Back to Home" link. Then create a 404.jsp that says "Page not found" and configure both in web.xml using <error-page> elements.

Exercise 6 — Full Servlet + JSP Pair

Write TrackingServlet.java that reads a bookingId parameter, calls BookingDAO.findByBookingId(), and either sets a booking attribute (if found) or an errorMessage attribute (if not found). Then forward to tracking.jsp. In the JSP, show the search form, the error message (if any), and the booking details (if found) using <c:if> to conditionally display each section.
