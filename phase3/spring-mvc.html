Section 01

What is Spring MVC?

Spring MVC is Spring's web framework. It implements the Model-View-Controller pattern with Spring managing everything — routing, parameter binding, view resolution, sessions, validation. You write clean controller methods instead of messy servlet code.

The Problem with Raw Servlets

With raw servlets, you do everything manually:

  • Parse request parameters with request.getParameter()
  • Manually route URLs to different servlets in web.xml
  • Cast parameters to correct types yourself
  • Forward to JSP pages with RequestDispatcher
  • Handle sessions, validation, errors — all by hand

For one servlet, this is fine. For Nimbus with 6+ servlets, it becomes a mess of boilerplate code that’s painful to maintain.

Analogy

Servlets = building a car from scratch. You weld the chassis, build the engine, wire the electronics, stitch the seats. You control everything, but it takes forever and you’ll make mistakes.

Spring MVC = buying a car with features built in. Engine, transmission, power steering, ABS — all included. You just drive. Spring handles the plumbing; you handle the business logic.

What Spring MVC Gives You

FeatureRaw ServletSpring MVC
URL routingweb.xml — one servlet per URL@GetMapping("/track") on method
Parameter bindingrequest.getParameter("id") + parse@RequestParam int id — auto-parsed
Form bindingGet each field manually, create object@ModelAttribute Booking b — auto-filled
View forwardingRequestDispatcher.forward()return "booking"; — view name
Dependency injectionnew UserDAO() everywhere@Autowired UserService — injected
ValidationManual if/else checks@Valid + annotations on fields

In Nimbus

Our 6 servlets (LoginServlet, RegisterServlet, BookingServlet, TrackingServlet, DeliveryServlet, HistoryServlet) each become one or two methods inside a @Controller class. No web.xml URL mappings. No manual parameter parsing. Spring does it all.

The MVC Pattern in Spring

  • Model — the data. Java objects (User, Booking, Parcel) passed to the view via Model or ModelAndView
  • View — the presentation. JSP pages that render HTML using the model data
  • Controller — the handler. Java class annotated with @Controller that processes requests, calls service layer, and returns a view name

Spring adds one critical piece: the DispatcherServlet — a front controller that sits between the browser and your controllers, routing every request to the right place.

Section 02

DispatcherServlet — The Front Controller

Every single HTTP request in a Spring MVC app goes through one servlet: the DispatcherServlet. It’s the traffic cop that routes requests to the correct controller method.

How It Works

In raw servlets, each URL is mapped to a different servlet. You could have 20 servlets for 20 URLs. With Spring MVC, there’s only one servlet — the DispatcherServlet. It receives all requests and delegates to the right @Controller method based on the URL and HTTP method.

Analogy

Raw servlets: A company where customers walk directly to different desks — one for billing, one for complaints, one for returns. No central reception. Chaos.

DispatcherServlet: A receptionist at the front desk. Every visitor checks in, the receptionist looks at what they need, and sends them to the right department. Organized, centralized, controlled.

Step-by-Step Breakdown

  1. Browser sends request — GET /nimbus/tracking?bookingId=101
  2. DispatcherServlet receives it — it’s mapped to / or /* in web.xml, so it catches everything
  3. HandlerMapping finds the right method — scans all @Controller classes, finds @GetMapping("/tracking") in TrackingController
  4. Controller method executes — receives bookingId as @RequestParam, calls service, adds result to Model
  5. Controller returns view name — return "tracking";
  6. ViewResolver resolves JSP path — prefix /WEB-INF/views/ + name tracking + suffix .jsp = /WEB-INF/views/tracking.jsp
  7. JSP renders HTML — uses model data (booking object) to build the page
  8. Response sent to browser — user sees the tracking page with booking details

DispatcherServlet Configuration

Option 1: web.xml (Traditional)

web.xml 
<web-app>

    <!-- Register DispatcherServlet -->
    <servlet>
        <servlet-name>dispatcher</servlet-name>
        <servlet-class>
            org.springframework.web.servlet.DispatcherServlet
        </servlet-class>
        <init-param>
            <param-name>contextConfigLocation</param-name>
            <param-value>/WEB-INF/spring-mvc.xml</param-value>
        </init-param>
        <load-on-startup>1</load-on-startup>
    </servlet>

    <!-- Map ALL requests to DispatcherServlet -->
    <servlet-mapping>
        <servlet-name>dispatcher</servlet-name>
        <url-pattern>/</url-pattern>
    </servlet-mapping>

</web-app>
Line-by-line explanation
1
<servlet-name>dispatcher</servlet-name>
Names this servlet "dispatcher" — Spring will look for dispatcher-servlet.xml by default (unless you override with init-param)
2
<servlet-class>...DispatcherServlet</servlet-class>
This IS a regular servlet — it extends HttpServlet. Spring provides it; you don't write it. It handles doGet/doPost internally and delegates to your controllers.
3
<param-name>contextConfigLocation</param-name>
Tells DispatcherServlet where to find Spring's config file. This file defines component scanning, view resolver, etc.
4
<load-on-startup>1</load-on-startup>
Load this servlet when Tomcat starts (not on first request). 1 = highest priority. Ensures Spring context is ready before any request comes in.
5
<url-pattern>/</url-pattern>
Map ALL URLs to DispatcherServlet. Every request (GET /login, POST /booking, GET /tracking) goes through this one servlet. It handles the routing internally using @RequestMapping annotations.

Option 2: Java Config (Modern — No web.xml)

WebAppInitializer.java 
public class WebAppInitializer
        extends AbstractAnnotationConfigDispatcherServletInitializer {

    @Override
    protected Class<?>[] getRootConfigClasses() {
        return null;  // No root config needed for simple apps
    }

    @Override
    protected Class<?>[] getServletConfigClasses() {
        return new Class[] { WebConfig.class };
    }

    @Override
    protected String[] getServletMappings() {
        return new String[] { "/" };  // Map all URLs to DispatcherServlet
    }
}
Key Insight

The DispatcherServlet is just a regular servlet (extends HttpServlet). The difference is that instead of you writing doGet()/doPost(), Spring's DispatcherServlet does that internally and calls YOUR controller methods. You never subclass it — you configure it.

Section 03

@Controller

The annotation that marks a Java class as a web controller. Spring scans for it, registers the class, and maps its methods to URLs.

Basic Controller

HomeController.java 
@Controller
public class HomeController {

    @GetMapping("/")
    public String home() {
        return "home";  // Returns view name "home"
                        // ViewResolver finds /WEB-INF/views/home.jsp
    }
}

That’s it. No extending HttpServlet. No overriding doGet(). No web.xml mapping. Just a POJO (plain old Java object) with annotations.

@Controller vs @RestController

Use caseTraditional web app (HTML pages)REST API (mobile/frontend consumes data)
Equivalent to@Component + web awareness@Controller + @ResponseBody on every method
Nimbus Phase 3We use this — JSP pagesPhase 4 — REST APIs

@Controller

    @GetMapping("/booking")
    public String showForm() {

    }

@RestController

    @GetMapping("/api/bookings")
    public List<Booking> getAll() {
        return bookingService.findAll();  // → JSON: [{...}, {...}]

In Nimbus

Phase 3 uses @Controller because we render JSP pages. In Phase 4, when we build REST APIs for a mobile app or frontend framework, we’ll switch to @RestController.

The Annotation Hierarchy

@Controller is a specialization of @Component. Spring's component scan finds it the same way it finds @Service and @Repository. The difference is semantic — it tells Spring "this class handles HTTP requests."

@Component
Generic Spring bean. Base annotation. "Register this class in the container."
@Controller
Web layer. Handles HTTP requests, returns view names. DispatcherServlet looks for these.
@Service
Business logic layer. No special behavior, but marks intent: "this is a service class."
@Repository
Data access layer. Adds automatic exception translation (SQL exceptions → Spring's DataAccessException).
Section 04

@RequestMapping

Maps a URL (and optionally an HTTP method) to a controller method. Can go on the class level (base URL) and method level (specific endpoint).

Basic Usage

BookingController.java 
@Controller
@RequestMapping("/booking")   // Class level — base URL
public class BookingController {

    @RequestMapping(value = "/new", method = RequestMethod.GET)
    public String showForm() {
        return "booking-form";
    }
    // Handles: GET /booking/new

    @RequestMapping(value = "/new", method = RequestMethod.POST)
    public String processForm(@ModelAttribute Booking booking) {
        bookingService.save(booking);
        return "redirect:/booking/confirmation";
    }
    // Handles: POST /booking/new
}

@RequestMapping Attributes

AttributePurposeExample
value / pathURL path(s) to mapvalue = "/login"
methodHTTP method(s) allowedmethod = RequestMethod.POST
producesResponse content typeproduces = "application/json"
consumesRequest content type requiredconsumes = "application/json"
paramsRequired request parametersparams = "action=delete"
headersRequired headersheaders = "X-API-Key"

Class-Level + Method-Level

When you put @RequestMapping on the class, it becomes the base URL. All method-level mappings are appended to it.

URL Composition 
@Controller
@RequestMapping("/delivery")         // Base: /delivery
public class DeliveryController {

    @RequestMapping("/status")       // Full: /delivery/status
    public String statusPage() { ... }

    @RequestMapping("/history")      // Full: /delivery/history
    public String historyPage() { ... }

    @RequestMapping("/update/{id}")  // Full: /delivery/update/42
    public String update(@PathVariable int id) { ... }
}
Common Mistake

Don’t put @RequestMapping on the class if all your methods have completely different base paths. Class-level mapping is for grouping related endpoints: /booking/new, /booking/cancel, /booking/track.

Shortcut Annotations

Spring 4.3 introduced shorthand annotations so you don’t need to write @RequestMapping(method = RequestMethod.GET) every time.

// BEFORE (verbose)

// AFTER (clean)
@PostMapping("/login")

  • @PostMapping("/register") — RegisterController creates user
  • @GetMapping("/booking") — BookingController shows the form
  • @PostMapping("/booking") — BookingController processes form
  • @GetMapping("/tracking") — TrackingController shows tracking info
  • @PostMapping("/delivery/status") — DeliveryController updates status
  • @GetMapping("/history") — HistoryController shows booking history
Section 06

Method Parameters

Spring MVC can inject almost anything into your controller methods — request parameters, path variables, form objects, session, model. You just declare them as method arguments.

@RequestParam — Query Parameters

Extracts parameters from the URL query string. /tracking?bookingId=101 → bookingId = 101.

TrackingController.java 
@GetMapping("/tracking")
public String track(
    @RequestParam("bookingId") int bookingId,  // Required by default
    Model model
) {
    Booking booking = bookingService.findById(bookingId);
    model.addAttribute("booking", booking);
    return "tracking";
}
// URL: /tracking?bookingId=101
// Spring automatically parses "101" → int 101

Optional and Default Values

HistoryController.java 
@GetMapping("/history")
public String history(
    @RequestParam(required = false) String startDate,  // Optional
    @RequestParam(required = false) String endDate,
    @RequestParam(defaultValue = "1") int page,        // Default value
    Model model
) {
    List<Booking> bookings = bookingService.findByDateRange(startDate, endDate, page);
    model.addAttribute("bookings", bookings);
    return "history";
}
// /history                         → startDate=null, endDate=null, page=1
// /history?startDate=2026-01-01    → startDate="2026-01-01", endDate=null, page=1
// /history?page=3                  → startDate=null, endDate=null, page=3

@PathVariable — URL Parts

Extracts values from the URL path itself. /booking/101 → id = 101.

PathVariable examples 
// Single path variable
@GetMapping("/booking/{id}")
public String viewBooking(@PathVariable int id, Model model) {
    Booking booking = bookingService.findById(id);
    model.addAttribute("booking", booking);
    return "booking-detail";
}
// URL: /booking/101    → id = 101
// URL: /booking/205    → id = 205

// Multiple path variables
@GetMapping("/delivery/{city}/{date}")
public String deliveries(
    @PathVariable String city,
    @PathVariable String date,
    Model model
) {
    List<Booking> deliveries = deliveryService.findByCityAndDate(city, date);
    model.addAttribute("deliveries", deliveries);
    return "delivery-list";
}
// URL: /delivery/Mumbai/2026-04-23

Automatically maps form fields to a Java object. If the form has fields named senderName, senderCity, weight, Spring creates a Booking object and calls the matching setters.

HttpSession — Session Access

Session injection 
@PostMapping("/login")
public String login(
    @RequestParam String email,
    @RequestParam String password,
    HttpSession session   // Spring injects the session automatically
) {
    User user = userService.authenticate(email, password);
    if (user != null) {
        session.setAttribute("loggedInUser", user);
        return "redirect:/dashboard";
    }
    return "redirect:/login?error=true";
}

Model — Passing Data to View

Model usage 
@GetMapping("/dashboard")
public String dashboard(Model model, HttpSession session) {
    User user = (User) session.getAttribute("loggedInUser");

    model.addAttribute("userName", user.getName());          // Single value
    model.addAttribute("bookings", bookingService.findByUser(user.getId())); // List
    model.addAttribute("totalBookings", 42);                 // Primitive

    return "dashboard";
    // In dashboard.jsp: ${userName}, ${bookings}, ${totalBookings}
}
Key Insight

Model is a container that carries data from controller to view. Whatever you put in with addAttribute() becomes available in the JSP via EL expressions (${key}). Think of it as a map that Spring passes along to the JSP for rendering.

Section 07

Returning Views

When a @Controller method returns a string, Spring treats it as a view name and uses the ViewResolver to find the actual JSP file.

Three Ways to Return Views

1. Return String (Most Common)

String return 
@GetMapping("/login")
public String showLogin() {
    return "login";  // → /WEB-INF/views/login.jsp
}

@GetMapping("/booking")
public String showBookingForm(Model model) {
    model.addAttribute("booking", new Booking());
    return "booking-form";  // → /WEB-INF/views/booking-form.jsp
}

2. ModelAndView (View + Data in One Object)

ModelAndView return 
@GetMapping("/tracking")
public ModelAndView track(@RequestParam int bookingId) {
    ModelAndView mav = new ModelAndView("tracking");  // View name
    Booking booking = bookingService.findById(bookingId);
    mav.addObject("booking", booking);                // Data
    mav.addObject("status", booking.getStatus());
    return mav;
}

3. Redirect

Redirect 
@PostMapping("/login")
public String login(@RequestParam String email,
                    @RequestParam String password,
                    HttpSession session) {
    User user = userService.authenticate(email, password);
    if (user != null) {
        session.setAttribute("loggedInUser", user);
        return "redirect:/dashboard";  // HTTP 302 redirect
    }
    return "redirect:/login?error=true";  // Redirect with query param
}

// "redirect:" prefix tells Spring: DON'T look for a JSP called "redirect".
// Instead, send an HTTP 302 redirect response to the browser.

How ViewResolver Works

The ViewResolver takes the view name you return and resolves it to an actual file path. The most common one is InternalResourceViewResolver:

spring-mvc.xml or WebConfig.java 
// Java Config
@Bean
public InternalResourceViewResolver viewResolver() {
    InternalResourceViewResolver resolver = new InternalResourceViewResolver();
    resolver.setPrefix("/WEB-INF/views/");   // Folder
    resolver.setSuffix(".jsp");              // Extension
    return resolver;
}

// When you return "tracking":
//   prefix + viewName + suffix
//   /WEB-INF/views/ + tracking + .jsp
//   = /WEB-INF/views/tracking.jsp
Why /WEB-INF/views/?

Files inside /WEB-INF/ are not directly accessible from the browser. A user can’t type /WEB-INF/views/admin.jsp in their browser and see it. They MUST go through a controller. This is a security feature — controllers check authentication before showing the page.

Form Handling

Spring MVC handles forms elegantly: @ModelAttribute binds form fields to Java objects, BindingResult captures errors, and the PRG pattern prevents duplicate submissions.

Complete Form Flow

→

Form flow: GET shows form → POST processes → Redirect prevents duplicate submit → GET shows confirmation

Controller Code

BookingController.java 
@Controller
@RequestMapping("/booking")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    // Step 1: Show the empty form
    @GetMapping
    public String showForm(Model model) {
        model.addAttribute("booking", new Booking()); // Empty object for form binding
        return "booking-form";
    }

    // Step 2: Process the submitted form
    @PostMapping
    public String processForm(
        @Valid @ModelAttribute("booking") Booking booking,
        BindingResult result,      // MUST be right after @ModelAttribute
        HttpSession session
    ) {
        // If validation errors, re-show the form
        if (result.hasErrors()) {
            return "booking-form";   // Form will show error messages
        }

        // Get logged-in user from session
        User user = (User) session.getAttribute("loggedInUser");
        booking.setUserId(user.getId());

        bookingService.createBooking(booking);
        return "redirect:/booking/confirmation";  // PRG pattern
    }

    // Step 3: Show confirmation (after redirect)
    @GetMapping("/confirmation")
    public String confirmation() {
        return "booking-confirmation";
    }
}
Line-by-line explanation
1
model.addAttribute("booking", new Booking());
Creates an empty Booking object and puts it in the model. The JSP form binds to this object. Without it, Spring form tags would throw an error.
2
@Valid @ModelAttribute("booking") Booking booking
@ModelAttribute tells Spring to map form fields to Booking properties. @Valid triggers validation annotations on the Booking class. "booking" matches the model attribute name.
3
BindingResult result
Holds validation errors. MUST come immediately after @ModelAttribute — if you put another parameter between them, Spring throws an error. Check result.hasErrors() before processing.
4
return "booking-form";
If validation fails, show the same form again. The form will still have the user's entered data (because the booking object is in the model) plus error messages.
5
return "redirect:/booking/confirmation";
PRG Pattern (Post-Redirect-Get). After successful POST, redirect to a GET endpoint. If user refreshes the page, it's a GET to /confirmation — not a re-POST of the form. Prevents duplicate bookings.

The JSP Form (Spring Form Tags)

booking-form.jsp 
<%@ taglib prefix="form" uri="http://www.springframework.org/tags/form" %>

Key Insight — PRG Pattern

Post-Redirect-Get prevents the "do you want to resubmit?" browser popup. Without it, if a user refreshes after booking, the browser re-sends the POST and creates a duplicate booking. With PRG: POST → 302 Redirect → GET confirmation. Refresh just re-GETs the confirmation page. Safe.

Section 09

Session Handling

Sessions track logged-in users across requests. Spring gives you two approaches: inject HttpSession directly, or use @SessionAttributes.

Approach 1: HttpSession Injection (Simpler)

LoginController.java 
@Controller
public class LoginController {

    @Autowired
    private UserService userService;

    @GetMapping("/login")
    public String showLoginPage() {
        return "login";
    }

    @PostMapping("/login")
    public String login(
        @RequestParam String email,
        @RequestParam String password,
        HttpSession session,
        Model model
    ) {
        User user = userService.authenticate(email, password);

        if (user == null) {
            model.addAttribute("error", "Invalid email or password");
            return "login";
        }

        // Store user in session — available across all requests
        session.setAttribute("loggedInUser", user);
        return "redirect:/dashboard";
    }

    @GetMapping("/logout")
    public String logout(HttpSession session) {
        session.invalidate();   // Destroy session, clear all data
        return "redirect:/login";
    }
}

Reading Session in Other Controllers

Any controller that needs logged-in user 
@GetMapping("/booking")
public String showBookingForm(HttpSession session, Model model) {
    // Check if user is logged in
    User user = (User) session.getAttribute("loggedInUser");
    if (user == null) {
        return "redirect:/login";  // Not logged in → send to login
    }

    model.addAttribute("userName", user.getName());
    model.addAttribute("booking", new Booking());
    return "booking-form";
}

Approach 2: @SessionAttributes (Spring-Managed)

@SessionAttributes example 
@Controller
@SessionAttributes("booking")  // Keep "booking" in session across requests
public class MultiStepBookingController {

    @GetMapping("/booking/step1")
    public String step1(Model model) {
        model.addAttribute("booking", new Booking());
        return "booking-step1";
    }

    @PostMapping("/booking/step2")
    public String step2(@ModelAttribute Booking booking) {
        // booking from step1 is preserved in session
        return "booking-step2";
    }

    @PostMapping("/booking/confirm")
    public String confirm(@ModelAttribute Booking booking,
                          SessionStatus status) {
        bookingService.save(booking);
        status.setComplete();  // Clear "booking" from session
        return "redirect:/booking/success";
    }
}
In Nimbus

We use Approach 1 (HttpSession injection) because it’s simpler and our login flow is straightforward. LoginController sets session.setAttribute("loggedInUser", user), and every other controller checks it. The authentication filter (from Phase 3 topic 7) catches unauthenticated requests before they reach the controller.

Section 10

Validation

Instead of writing if (name == null || name.isEmpty()) for every field, Spring lets you put validation annotations directly on your model class. One annotation per rule.

Step 1: Add Annotations to Model

Booking.java 
import jakarta.validation.constraints.*;

public class Booking {

    @NotBlank(message = "Sender name is required")
    @Size(min = 2, max = 50, message = "Name must be 2-50 characters")
    private String senderName;

    @NotBlank(message = "Sender city is required")
    private String senderCity;

    @NotBlank(message = "Receiver name is required")
    private String receiverName;

    @NotBlank(message = "Receiver city is required")
    private String receiverCity;

    @NotNull(message = "Weight is required")
    @Min(value = 1, message = "Minimum weight is 0.1 kg")
    @Max(value = 50, message = "Maximum weight is 50 kg")
    private Double weight;

    // getters + setters
}

        <!-- Shows: "Maximum weight is 50 kg" if > 50 -->
    </div>

    <!-- Show ALL errors at top -->
    <form:errors path="*" cssClass="error-summary" />

    <button type="submit">Book</button>
</form:form>
In Nimbus

User registration uses @Email on the email field, @Size(min=6) on password. Booking uses @NotBlank on sender/receiver names and @Min(1) on weight. All validation errors display inline next to the form field.

Dependency Required

Validation annotations come from jakarta.validation:jakarta.validation-api and the implementation org.hibernate.validator:hibernate-validator. Add both to your pom.xml. Without the implementation, @Valid silently does nothing.

Section 11

Exception Handling

Instead of try/catch in every controller method, Spring lets you handle exceptions centrally with @ExceptionHandler and @ControllerAdvice.

Approach 1: @ExceptionHandler (Per Controller)

BookingController.java 
@Controller
public class BookingController {

    @Autowired
    private BookingService bookingService;

    @GetMapping("/tracking")
    public String track(@RequestParam int bookingId, Model model) {
        // If booking doesn't exist, service throws BookingNotFoundException
        Booking booking = bookingService.findById(bookingId);
        model.addAttribute("booking", booking);
        return "tracking";
    }

    // Handles BookingNotFoundException ONLY in this controller
    @ExceptionHandler(BookingNotFoundException.class)
    public String handleBookingNotFound(BookingNotFoundException ex, Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error/booking-not-found";  // → booking-not-found.jsp
    }
}

Approach 2: @ControllerAdvice (Global)

A single class that handles exceptions from ALL controllers. Much better than putting @ExceptionHandler in every controller.

GlobalExceptionHandler.java 
@ControllerAdvice
public class GlobalExceptionHandler {

    // Handle specific business exceptions
    @ExceptionHandler(BookingNotFoundException.class)
    public String handleBookingNotFound(BookingNotFoundException ex, Model model) {
        model.addAttribute("errorMessage", "Booking not found: " + ex.getMessage());
        return "error/not-found";
    }

    @ExceptionHandler(DuplicateEmailException.class)
    public String handleDuplicateEmail(DuplicateEmailException ex, Model model) {
        model.addAttribute("errorMessage", "Email already registered");
        return "error/duplicate";
    }

    @ExceptionHandler(UnauthorizedException.class)
    public String handleUnauthorized() {
        return "redirect:/login?error=unauthorized";
    }

    // Catch-all for anything unexpected
    @ExceptionHandler(Exception.class)
    public String handleGeneral(Exception ex, Model model) {
        model.addAttribute("errorMessage", "Something went wrong. Please try again.");
        // Log the actual error for debugging
        ex.printStackTrace();
        return "error/general";
    }
}
Line-by-line explanation
1
@ControllerAdvice
Marks this class as a global exception handler. Spring scans all controllers, and any unhandled exception gets routed here. "Advice" is AOP terminology — it advises (wraps around) controller methods.
2
@ExceptionHandler(BookingNotFoundException.class)
This method handles BookingNotFoundException from ANY controller. Spring matches the exception type. More specific handlers (BookingNotFoundException) are tried before generic ones (Exception).
3
@ExceptionHandler(Exception.class)
The catch-all. If no other handler matches, this one catches it. Always put this last. Log the real error for debugging, but show a user-friendly message on the page.
In Nimbus

We use @ControllerAdvice with handlers for:

  • BookingNotFoundException → "Booking #xyz not found" page
  • DuplicateEmailException → redirect to register with error
  • UnauthorizedException → redirect to login
  • Exception (catch-all) → generic error page

Full Nimbus Controllers

The actual controllers for Nimbus — converting raw servlets to Spring MVC. Each servlet becomes one or two annotated methods.

LoginController

LoginController.java 
@Controller
public class LoginController {

    @Autowired
    private UserService userService;

    @GetMapping("/login")
    public String showLoginPage() {
        return "login";  // → /WEB-INF/views/login.jsp
    }

    @PostMapping("/login")
    public String processLogin(
        @RequestParam String email,
        @RequestParam String password,
        HttpSession session,
        Model model
    ) {
        User user = userService.authenticate(email, password);

        if (user == null) {
            model.addAttribute("error", "Invalid email or password");
            return "login";  // Re-show login page with error
        }

        session.setAttribute("loggedInUser", user);

        // Redirect based on role
        if ("ADMIN".equals(user.getRole())) {
            return "redirect:/admin/dashboard";
        }
        return "redirect:/dashboard";
    }

    @GetMapping("/logout")
    public String logout(HttpSession session) {
        session.invalidate();
        return "redirect:/login?logout=true";
    }
}
Line-by-line explanation
1
@Autowired private UserService userService;
Spring injects the UserService bean. No "new UserService()" — Spring's IoC container creates it and gives it to us. The service handles authentication logic.
2
@GetMapping("/login") → return "login"
When user visits /login in browser (GET request), show the login.jsp page. No data needed, just the form.
3
@PostMapping("/login") with @RequestParam
When the login form is submitted (POST), Spring extracts email and password from the form data. No request.getParameter() needed.
4
session.setAttribute("loggedInUser", user);
Stores the authenticated user in the HTTP session. Every subsequent request from this browser will have access to this user object via session.getAttribute().
5
return "redirect:/dashboard";
After successful login, redirect (HTTP 302) to the dashboard. PRG pattern — if user refreshes, it's a GET to /dashboard, not a re-POST of credentials.

BookingController

BookingController.java 
@Controller
@RequestMapping("/booking")
public class BookingController {

    @Autowired
    private BookingService bookingService;

    // Show booking form
    @GetMapping
    public String showForm(Model model, HttpSession session) {
        User user = (User) session.getAttribute("loggedInUser");
        if (user == null) {
            return "redirect:/login";
        }

        model.addAttribute("booking", new Booking());
        model.addAttribute("userName", user.getName());
        return "booking-form";
    }

    // Process booking form submission
    @PostMapping
    public String processBooking(
        @Valid @ModelAttribute("booking") Booking booking,
        BindingResult result,
        HttpSession session,
        RedirectAttributes redirectAttrs
    ) {
        if (result.hasErrors()) {
            return "booking-form";
        }

        User user = (User) session.getAttribute("loggedInUser");
        booking.setUserId(user.getId());
        booking.setStatus("BOOKED");
        booking.setBookingDate(LocalDate.now());

        String bookingId = bookingService.createBooking(booking);

        // Flash attribute — available after redirect, then auto-deleted
        redirectAttrs.addFlashAttribute("successMessage",
            "Booking created! ID: " + bookingId);

        return "redirect:/booking/confirmation";
    }

    // Confirmation page
    @GetMapping("/confirmation")
    public String confirmation() {
        return "booking-confirmation";
    }
}

        @RequestParam(required = false) String endDate,

        Model model
    ) {

        model.addAttribute("startDate", startDate);
        model.addAttribute("endDate", endDate);
        return "history";
    }
}

Spring MVC Configuration

Three things to configure: component scanning (find controllers), view resolver (find JSPs), and the DispatcherServlet itself.

Java Config (Modern Approach)

WebConfig.java 
@Configuration
@EnableWebMvc
@ComponentScan(basePackages = "com.nimbus")
public class WebConfig implements WebMvcConfigurer {

    // View Resolver — tells Spring where JSPs live
    @Bean
    public InternalResourceViewResolver viewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setPrefix("/WEB-INF/views/");
        resolver.setSuffix(".jsp");
        return resolver;
    }

    // Static resources (CSS, JS, images)
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("/static/");
    }

    // Message source for validation messages (optional)
    @Bean
    public MessageSource messageSource() {
        ResourceBundleMessageSource source = new ResourceBundleMessageSource();
        source.setBasename("messages");
        return source;
    }
}
Line-by-line explanation
1
@Configuration
Marks this as a Spring config class. Replaces XML config files. Spring reads this at startup.
2
@EnableWebMvc
Enables Spring MVC features: argument resolvers, message converters, etc. Without this, @RequestParam, @ModelAttribute won't work.
3
@ComponentScan(basePackages = "com.nimbus")
Tells Spring to scan the com.nimbus package (and sub-packages) for @Controller, @Service, @Repository, @Component classes and register them as beans.
4
implements WebMvcConfigurer
Interface that lets you customize Spring MVC behavior. Override methods to add interceptors, resource handlers, CORS config, etc.
5
resolver.setPrefix("/WEB-INF/views/") + setSuffix(".jsp")
When controller returns "login", ViewResolver builds: /WEB-INF/views/ + login + .jsp = /WEB-INF/views/login.jsp. This way controllers only return short names.

Raw Servlet vs Spring MVC — Side by Side

The same feature implemented both ways. See exactly how Spring MVC eliminates boilerplate.

Login Feature

Raw Servlet
// LoginServlet.java
@WebServlet("/login")
public class LoginServlet
    extends HttpServlet {

  private UserDAO userDAO
    = new UserDAO();

  @Override
  protected void doGet(
    HttpServletRequest req,
    HttpServletResponse resp)
    throws ServletException,
           IOException {
    req.getRequestDispatcher(
      "/WEB-INF/login.jsp")
      .forward(req, resp);
  }

  @Override
  protected void doPost(
    HttpServletRequest req,
    HttpServletResponse resp)
    throws ServletException,
           IOException {
    String email = req
      .getParameter("email");
    String password = req
      .getParameter("password");

    User user = userDAO
      .authenticate(email, password);

    if (user != null) {
      HttpSession session =
        req.getSession();
      session.setAttribute(
        "loggedInUser", user);
      resp.sendRedirect(
        "/nimbus/dashboard");
    } else {
      req.setAttribute(
        "error", "Invalid login");
      req.getRequestDispatcher(
        "/WEB-INF/login.jsp")
        .forward(req, resp);
    }
  }
}

Tracking Feature

Raw Servlet
@WebServlet("/tracking")
public class TrackingServlet
    extends HttpServlet {

  private BookingDAO bookingDAO
    = new BookingDAO();

  @Override
  protected void doGet(
    HttpServletRequest req,
    HttpServletResponse resp)
    throws ServletException,
           IOException {
    String idStr = req
      .getParameter("bookingId");

    if (idStr == null
      || idStr.isEmpty()) {
      req.setAttribute("error",
        "Booking ID required");
      req.getRequestDispatcher(
        "/WEB-INF/tracking.jsp")
        .forward(req, resp);
      return;
    }

    try {
      int id = Integer
        .parseInt(idStr);
      Booking b = bookingDAO
        .findById(id);
      req.setAttribute(
        "booking", b);
    } catch (NumberFormatException e) {
      req.setAttribute("error",
        "Invalid booking ID");
    }

    req.getRequestDispatcher(
      "/WEB-INF/tracking.jsp")
      .forward(req, resp);
  }
}

    bookingService;

Key Insight

Spring MVC doesn’t do anything magical. It does the same work the servlet does (getParameter, parseInt, forward) — but it does it for you, behind the scenes. @RequestParam int bookingId means Spring calls getParameter("bookingId"), parses it to int, and if it fails, throws a TypeMismatchException that your @ControllerAdvice catches. You just write the business logic.
