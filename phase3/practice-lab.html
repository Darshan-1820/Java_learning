Servlet Exercises

Exercise 1: Hello Servlet

Task

Create a servlet mapped to /hello that responds with an HTML page showing "Hello, [name]!" where name comes from a query parameter. If no name is provided, show "Hello, World!".

What will happen if...

  • You visit /hello with no parameters? → Shows "Hello, World!" because request.getParameter("name") returns null, and your null check defaults to "World".
  • You visit /hello?name=Rahul? → Shows "Hello, Rahul!" because the query parameter is extracted.
  • You forget to set content type? → Browser shows raw HTML tags as text instead of rendering the page, because the default content type is text/plain.
HelloServlet.java 
@WebServlet("/hello")
public class HelloServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        String name = request.getParameter("name");
        if (name == null || name.isEmpty()) {
            name = "World";
        }

        response.setContentType("text/html");
        PrintWriter out = response.getWriter();
        out.println("<html><body>");
        out.println("<h1>Hello, " + name + "!</h1>");
        out.println("</body></html>");
    }
}

Exercise 2: Form Processing Servlet

Task

Create two files: (1) an HTML form at parcel-form.html with fields for sender name, recipient name, and weight, and (2) a servlet at /process-booking that reads the form data via POST and displays a confirmation page with all the details.

What will happen if...

  • You use method="get" on the form? → Data appears in the URL as query parameters (e.g., ?sender=Rahul&recipient=Priya). Your doPost() method won't be called — you'll get a 405 error unless you also implement doGet().
  • You type letters in the weight field and parse with Double.parseDouble()? → NumberFormatException at runtime. You need validation or a try/catch.

Exercise 3: Servlet Forwarding

Task

Create a CalculatorServlet at /calculate that takes weight and type parameters, calculates a price (Document=50, Small=100, Medium=200, Large=350, plus weight*15), stores the price as a request attribute, and forwards to result.jsp which displays it.

What will happen if...

  • You use response.sendRedirect("result.jsp") instead of RequestDispatcher.forward()? → The request attribute is lost. Redirect creates a brand new request. The JSP won't have access to the price attribute. Forward shares the same request object.

Exercise 4: Servlet Context & Init Parameters

Task

Configure an init parameter company-name with value "Nimbus Logistics" in web.xml. Write a servlet that reads this parameter using getServletConfig().getInitParameter() and displays it on the page.

Then add a context parameter (not init parameter) called version with value "1.0". Read it with getServletContext().getInitParameter().

What will happen if...

  • You try to read a context parameter using getServletConfig()? → Returns null. Servlet config only has servlet-level params. Context params are application-wide and need getServletContext().

Exercise 5: Servlet Lifecycle Logger

Task

Create a servlet that overrides init(), doGet(), and destroy(). In each method, print a timestamp and the method name to the console using System.out.println(). Deploy it, hit it 3 times, then stop Tomcat. Check the console output.

What will happen?

  • init() prints once — on the first request (or at startup if load-on-startup is set).
  • doGet() prints 3 times — once per request.
  • destroy() prints once — when Tomcat shuts down.
  • This proves servlets are singletons. One instance handles all requests.

    </c:otherwise>
</c:choose>

Exercise 9: JSP Include Directive

Task

Create a header.jsp with navigation links and a footer.jsp with copyright text. Create a home.jsp that includes both using <%@ include %> directive.

Question: What is the difference between <%@ include file="header.jsp" %> and <jsp:include page="header.jsp" />?

Answer

<%@ include %> = static include. The content of header.jsp is copied into home.jsp at compile time. It becomes one big servlet. Changes to header.jsp require recompiling home.jsp.

<jsp:include> = dynamic include. At runtime, the server processes header.jsp separately and merges the output. Changes to header.jsp are picked up immediately without recompiling. Use this for content that changes frequently.

Exercise 11: Dependency Injection

Task

Create a NotificationService interface with a send(String message) method. Implement two classes: EmailNotification and SmsNotification. Use @Service and @Primary to make EmailNotification the default. Inject it into a controller and call send().

What will happen if...

  • You have two @Service implementations without @Primary? → Spring throws NoUniqueBeanDefinitionException at startup. It doesn't know which one to inject. Fix with @Primary on one, or @Qualifier("emailNotification") at the injection point.
  • You use @Autowired on a field that has no matching bean? → NoSuchBeanDefinitionException at startup. Spring can't find anything to inject.

Exercise 12: @RequestParam vs @PathVariable

Task

Create a controller with two endpoints:

  1. GET /search?city=Mumbai using @RequestParam
  2. GET /parcel/NIM123ABC using @PathVariable

Both should return a JSP view. For the search, pass the city name to the view. For the parcel, look it up and pass the Parcel object.

SearchController.java (solution) 
@Controller
public class SearchController {

    @Autowired
    private ParcelDAO parcelDAO;

    // /search?city=Mumbai
    @GetMapping("/search")
    public String searchByCity(
            @RequestParam String city,
            Model model) {
        model.addAttribute("city", city);
        return "search-results";
    }

    // /parcel/NIM123ABC
    @GetMapping("/parcel/{trackingId}")
    public String getParcel(
            @PathVariable String trackingId,
            Model model) {
        Parcel p = parcelDAO.findByTrackingId(trackingId);
        model.addAttribute("parcel", p);
        return "parcel-detail";
    }
}

Exercise 13: @ModelAttribute Form Binding

Task

Create a "Contact Us" form with fields: name, email, subject, message. Create a ContactForm POJO matching these fields. Write a controller that uses @ModelAttribute ContactForm form to automatically bind all fields from the form. Display a confirmation page showing the submitted data.

What will happen if...

  • The form field name is email but the POJO field is emailAddress? → The field will be null. Spring matches form field names to setter methods. email looks for setEmail(). If the setter is setEmailAddress(), there's no match.
  • Your POJO has no default constructor? → Spring cannot instantiate it. @ModelAttribute calls the no-arg constructor first, then calls setters. No constructor = exception.

Exercise 14: Exception Handling

Task

Create a ParcelNotFoundException custom exception. In your tracking controller, throw it when no parcel matches the tracking ID. Write an @ExceptionHandler method in the controller that catches this exception and returns an error view with the message.

Exception handling (solution) 
// Custom exception
public class ParcelNotFoundException extends RuntimeException {
    public ParcelNotFoundException(String trackingId) {
        super("No parcel found with ID: " + trackingId);
    }
}

// In the controller
@Controller
@RequestMapping("/tracking")
public class TrackingController {

    @Autowired
    private ParcelDAO parcelDAO;

    @PostMapping
    public String track(@RequestParam String trackingId,
                        Model model) {
        Parcel p = parcelDAO.findByTrackingId(trackingId);
        if (p == null) {
            throw new ParcelNotFoundException(trackingId);
        }
        model.addAttribute("parcel", p);
        return "tracking";
    }

    @ExceptionHandler(ParcelNotFoundException.class)
    public String handleNotFound(ParcelNotFoundException ex,
                                  Model model) {
        model.addAttribute("errorMessage", ex.getMessage());
        return "error";  // error.jsp
    }
}

Exercise 15: Spring MVC Redirect vs Forward

Task

Create two endpoints in a controller:

  1. POST /register that saves a user and does return "redirect:/login?registered=true";
  2. POST /search that finds results and does return "search-results"; (forward)

After each POST, try refreshing the browser. What happens in each case?

Answer

Redirect (return "redirect:/login"): After register, the browser's URL changes to /login. Refreshing the page just reloads /login — the form is NOT resubmitted. This is the Post-Redirect-Get (PRG) pattern. Always use this after a successful form submission.

Forward (return "search-results"): The URL stays as /search. Refreshing the page shows a browser warning "Resubmit form data?" and sends the POST again. This can cause duplicate submissions (double bookings!). That's why we use redirect after booking in Nimbus.

Exercise 16: Session-Based Cart

Task

Build a "parcel cart" where users can add multiple parcels before submitting. Store the list in the session. Create three endpoints:

  1. POST /cart/add — adds a parcel to the session list
  2. GET /cart — shows all parcels in the cart
  3. POST /cart/submit — saves all parcels to the database and clears the session

What will happen if...

  • The user opens a second browser tab? → Same session, same cart. Sessions are per-browser, not per-tab. Both tabs see the same cart data.
  • The session expires while the user is filling the cart? → Cart is lost. Session data lives in server memory. When the session expires (default 30 min), all attributes are gone.

Exercise 17: Login Filter + Role Check

Task

Modify the AuthFilter to not only check if the user is logged in, but also check their role. The /delivery/* URLs should only be accessible by users with role "OFFICER". If a customer tries to access delivery pages, redirect them to /dashboard with an error message.

RoleFilter.java (solution) 
public class RoleFilter implements Filter {

    @Override
    public void doFilter(ServletRequest request,
                         ServletResponse response,
                         FilterChain chain)
            throws IOException, ServletException {

        HttpServletRequest req = (HttpServletRequest) request;
        HttpServletResponse res = (HttpServletResponse) response;
        HttpSession session = req.getSession(false);

        if (session == null) {
            res.sendRedirect(req.getContextPath() + "/login");
            return;
        }

        User user = (User) session.getAttribute("loggedInUser");
        if (user == null) {
            res.sendRedirect(req.getContextPath() + "/login");
            return;
        }

        String uri = req.getRequestURI();
        if (uri.contains("/delivery")
                && !"OFFICER".equals(user.getRole())) {
            res.sendRedirect(req.getContextPath()
                + "/dashboard?error=unauthorized");
            return;
        }

        chain.doFilter(request, response);
    }
}

Prediction Challenges

Read each code snippet and predict the output BEFORE running it. Test your understanding.

Challenge 1: Servlet Output

What does this print? 
@WebServlet("/test")
public class TestServlet extends HttpServlet {
    private int count = 0;

    protected void doGet(HttpServletRequest req,
                         HttpServletResponse res)
            throws IOException {
        count++;
        res.getWriter().println("Count: " + count);
    }
}
Answer

First visit: Count: 1. Second visit: Count: 2. Third visit: Count: 3. The count keeps incrementing because servlets are singletons. There is only one instance of TestServlet, and count is shared across all requests. This is a threading bug — in production, two users could hit this simultaneously and get wrong counts.

Challenge 2: EL Scope Resolution

What does ${name} show? 
// In servlet:
request.setAttribute("name", "Request-Rahul");
session.setAttribute("name", "Session-Amit");
// forward to JSP...
Answer

${name} shows "Request-Rahul". EL searches scopes in order: page → request → session → application. It finds "name" in request scope first and stops. To explicitly get the session value, use ${sessionScope.name}.

Challenge 3: Spring Controller Return

What URL does the browser show? 
@PostMapping("/booking/new")
public String book(@ModelAttribute Parcel parcel) {
    parcelDAO.save(parcel);
    return "redirect:/dashboard";
}

@GetMapping("/dashboard")
public String dashboard() {
    return "customer-dashboard";
}
Answer

After submitting the booking form, the browser URL changes to /dashboard. The redirect: prefix causes a 302 redirect. The browser makes a new GET request to /dashboard, which returns the customer-dashboard.jsp view. The URL bar shows /dashboard, not /booking/new.

Challenge 4: Filter Chain Order

What order do these print? 
// FilterA
public void doFilter(...) {
    System.out.println("A-before");
    chain.doFilter(request, response);
    System.out.println("A-after");
}

// FilterB
public void doFilter(...) {
    System.out.println("B-before");
    chain.doFilter(request, response);
    System.out.println("B-after");
}

// web.xml: FilterA mapped first, FilterB mapped second
Answer

Output: A-before, B-before, (servlet executes), B-after, A-after. Filters form a chain. Each filter wraps around the next like Russian dolls. The order is determined by the order of <filter-mapping> declarations in web.xml.
