Section 01

Why Sessions Exist

HTTP is stateless. Every request is a blank slate. The server has zero memory of previous requests. So how does it know you're logged in?

Restaurant Analogy

Imagine a restaurant where the waiter has no memory. Every time you call them over, they have no idea who you are, what you ordered, or that you're even a customer. You'd have to show your receipt and re-explain your entire order every single time.

That's HTTP. Every request is a brand-new waiter who knows nothing about you. Sessions solve this by giving you a wristband (session ID) that the waiter checks against a list in the kitchen.

The Problem: Statelessness

Consider this scenario without sessions:

  1. User sends POST /login with username "john" and password "secret". Server validates. Success! But...
  2. User sends GET /dashboard. Server receives this request. But it has no idea who sent it. The login from step 1 is forgotten. "Who are you? Login again."
  3. User sends GET /my-bookings. Same problem. "Who are you?" This would happen on every single request.

Sessions solve this by creating a link between multiple requests from the same user.

Without Sessions every request = stranger
  vs  
With Sessions server remembers you
Sessions turn stateless HTTP into a stateful experience
Section 02

Cookies

Cookies are small pieces of data that the server asks the browser to store. The browser automatically sends them back with every subsequent request.

How Cookies Work

1. Browser sends request GET /login (no cookies yet)
→
Server responds Set-Cookie: JSESSIONID=abc123
↓
2. Browser stores cookie saves abc123 locally
→
Every future request Cookie: JSESSIONID=abc123
Set-Cookie (server → browser) vs Cookie (browser → server)

Set-Cookie Header (Server → Browser)

Server Response with Set-Cookie 
HTTP/1.1 200 OK
Set-Cookie: JSESSIONID=abc123; Path=/; HttpOnly; Secure; Max-Age=1800
Cookie attributes explained
1
JSESSIONID=abc123
The cookie name and value. JSESSIONID is the standard name Java/Tomcat uses for session IDs.
2
Path=/
Send this cookie for ALL paths on this domain. If it was Path=/admin, it would only be sent for URLs starting with /admin.
3
HttpOnly
JavaScript CANNOT read this cookie (document.cookie won't show it). Prevents XSS attacks from stealing session IDs. Always use this for session cookies.
4
Secure
Only send this cookie over HTTPS. Prevents session IDs from being sent over unencrypted HTTP.
5
Max-Age=1800
Cookie expires in 1800 seconds (30 minutes). After that, the browser deletes it. If omitted, the cookie dies when the browser closes (session cookie).

Types of Cookies

TypeLifetimeUse Case
Session Cookie Until browser closes (no Max-Age or Expires) Login session, shopping cart
Persistent Cookie Until Max-Age or Expires date "Remember me" login, user preferences, theme

Setting Cookies in Servlets

Creating and Reading Cookies in Java 
// CREATING a cookie (in a Servlet)
Cookie cookie = new Cookie("theme", "dark");
cookie.setMaxAge(60 * 60 * 24 * 30);  // 30 days in seconds
cookie.setPath("/");
cookie.setHttpOnly(true);
response.addCookie(cookie);

// READING cookies from request
Cookie[] cookies = request.getCookies();
if (cookies != null) {
    for (Cookie c : cookies) {
        if ("theme".equals(c.getName())) {
            String theme = c.getValue();  // "dark"
        }
    }
}

// DELETING a cookie (set Max-Age to 0)
Cookie deleteCookie = new Cookie("theme", "");
deleteCookie.setMaxAge(0);
deleteCookie.setPath("/");
response.addCookie(deleteCookie);
Section 03

HttpSession — Server-Side Sessions

While cookies store data in the browser, HttpSession stores data on the server. The browser only holds a session ID (in a cookie). The real data lives server-side.

Analogy

A cookie is like a coat check ticket. The ticket itself has no value — it's just a number. But when you hand it to the coat check person (the server), they look up your number and give you your coat (your session data). The coat was always stored safely behind the counter, not in your pocket.

How HttpSession Works

Browser holds only: JSESSIONID=abc123
→
Server (Tomcat) Map: abc123 → {user=John, role=customer}
Browser stores the key. Server stores the data.

Core Methods

HttpSession API — The methods you'll use 
// GET or CREATE a session
HttpSession session = request.getSession();       // creates if none exists
HttpSession session = request.getSession(false);  // returns null if none exists

// STORE data in the session
session.setAttribute("loggedInUser", user);
session.setAttribute("role", "customer");
session.setAttribute("loginTime", System.currentTimeMillis());

// READ data from the session
User user = (User) session.getAttribute("loggedInUser");
String role = (String) session.getAttribute("role");

// REMOVE a specific attribute
session.removeAttribute("tempData");

// GET the session ID
String sessionId = session.getId();  // "abc123"

// SET session timeout (in seconds)
session.setMaxInactiveInterval(30 * 60);  // 30 minutes

// DESTROY the session (logout)
session.invalidate();
Key methods explained
2
request.getSession()
If a session already exists for this user (identified by JSESSIONID cookie), returns it. If not, creates a brand-new session and sends a Set-Cookie: JSESSIONID=... header automatically.
3
request.getSession(false)
Same, but does NOT create a new session. Returns null if no session exists. Use this on pages that should only work if you're already logged in.
6
session.setAttribute("loggedInUser", user)
Stores the User object in the session under the key "loggedInUser". Think of sessions as a HashMap<String, Object>. You put things in with a key, get them out with the same key.
11
(User) session.getAttribute("loggedInUser")
Retrieves the stored object. Returns Object, so you must cast it to the correct type. Returns null if the key doesn't exist.
23
session.invalidate()
Destroys the entire session. All stored attributes are gone. The JSESSIONID cookie becomes invalid. The user is effectively logged out.
Section 04

Session Lifecycle

A session is born, lives, and dies. Understanding this lifecycle is critical for building secure login systems.

1. Creation
When request.getSession() is called for the first time, Tomcat creates a new session, generates a unique ID (like A1B2C3D4E5F6), and sends Set-Cookie: JSESSIONID=A1B2C3D4E5F6 to the browser.
2. Active Use
Every request from this browser includes Cookie: JSESSIONID=A1B2C3D4E5F6. Tomcat looks up this ID in its internal map and gives your Servlet the matching HttpSession object. You can read/write attributes freely.
3. Timeout
If the user doesn't send any request for the timeout period (default: 30 minutes in Tomcat), the session is automatically destroyed. Configurable via session.setMaxInactiveInterval(seconds) or in web.xml.
4. Invalidation (Logout)
Calling session.invalidate() destroys the session immediately. The session ID is removed from Tomcat's map. The next request with that JSESSIONID will not find a session — the user must log in again.

Configuring Session Timeout

web.xml — Set timeout for ALL sessions 
<session-config>
    <session-timeout>30</session-timeout>  <!-- minutes, not seconds -->
</session-config>
Java Code — Set timeout for ONE session 
// Set timeout to 15 minutes for this specific session
session.setMaxInactiveInterval(15 * 60);  // seconds

Section 06

Nimbus Login Session Code

Complete working code for the Nimbus login system using Servlets, HttpSession, and JSP.

LoginServlet — Handle Login

LoginServlet.java 
@WebServlet("/login")
public class LoginServlet extends HttpServlet {
    private UserDAO userDAO = new UserDAO();

    // Show login page
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {
        request.getRequestDispatcher("/WEB-INF/views/login.jsp")
               .forward(request, response);
    }

    // Handle login form submission
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        String username = request.getParameter("username");
        String password = request.getParameter("password");

        // Validate against database
        User user = userDAO.findByUsername(username);

        if (user != null && user.getPassword().equals(password)) {
            // SUCCESS: Create session and store user
            HttpSession session = request.getSession();
            session.setAttribute("loggedInUser", user);
            session.setAttribute("role", user.getRole());
            session.setMaxInactiveInterval(30 * 60); // 30 min

            // Redirect based on role
            if ("officer".equals(user.getRole())) {
                response.sendRedirect("officer/dashboard");
            } else {
                response.sendRedirect("customer/dashboard");
            }
        } else {
            // FAILURE: Show error on login page
            request.setAttribute("error", "Invalid username or password");
            request.getRequestDispatcher("/WEB-INF/views/login.jsp")
                   .forward(request, response);
        }
    }
}
Key lines explained
26
if (user != null && user.getPassword().equals(password))
Checks two things: 1) Does this username exist in the database? 2) Does the password match? In production, you'd hash the password. Never compare plain text passwords.
28
HttpSession session = request.getSession();
Creates a new session. Tomcat generates a unique JSESSIONID and will send it to the browser as a cookie. This is the moment "state" begins.
29
session.setAttribute("loggedInUser", user);
Stores the entire User object in the session. Now every page can do session.getAttribute("loggedInUser") to know who's logged in and access their name, email, role, etc.
34
response.sendRedirect("officer/dashboard");
Sends a 302 Found redirect. The browser makes a new GET request to /officer/dashboard. This is the Post-Redirect-Get (PRG) pattern — prevents form resubmission on refresh.

DashboardServlet — Check Session

DashboardServlet.java 
@WebServlet("/customer/dashboard")
public class DashboardServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // Check if user is logged in
        HttpSession session = request.getSession(false);

        if (session == null || session.getAttribute("loggedInUser") == null) {
            // Not logged in: redirect to login page
            response.sendRedirect("../login");
            return;
        }

        // Logged in: get user from session
        User user = (User) session.getAttribute("loggedInUser");
        String role = (String) session.getAttribute("role");

        // Check authorization (customer can't access officer pages)
        if (!"customer".equals(role)) {
            response.sendError(HttpServletResponse.SC_FORBIDDEN);
            return;
        }

        // All good: show dashboard
        request.setAttribute("user", user);
        request.getRequestDispatcher("/WEB-INF/views/dashboard.jsp")
               .forward(request, response);
    }
}

LogoutServlet — Destroy Session

LogoutServlet.java 
@WebServlet("/logout")
public class LogoutServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        HttpSession session = request.getSession(false);
        if (session != null) {
            session.invalidate();  // Destroy session + all data
        }
        response.sendRedirect("login");  // Back to login page
    }
}

dashboard.jsp — Using Session Data

dashboard.jsp 
<%@ page language="java" contentType="text/html" %>
<html>
<body>
    <h1>Welcome, ${sessionScope.loggedInUser.name}!</h1>
    <p>Role: ${sessionScope.role}</p>
    <p>Session ID: ${pageContext.session.id}</p>
    <a href="../logout">Logout</a>
</body>
</html>
In Nimbus

This is exactly how Nimbus handles authentication. The LoginServlet validates credentials, stores the User object in the session, and redirects to the role-specific dashboard. Every protected page checks session.getAttribute("loggedInUser"). Logout calls session.invalidate().

Section 07

Session vs Cookie

Both maintain state, but they work very differently. Use the right one for the right job.

Feature Cookie HttpSession
Where data lives Browser (client-side) Server (server-side)
Size limit ~4 KB per cookie No practical limit (server RAM)
Security Less secure — user can read/modify More secure — data never leaves server
Data types Strings only Any Java object
Lifetime Controlled by Max-Age/Expires Until timeout or invalidation
Use for Preferences, "remember me", tracking Login state, user data, shopping cart
Server memory None (stored on client) Uses server memory per user
They Work Together

Sessions USE cookies. The session ID is stored in a cookie (JSESSIONID). So cookies are the transport mechanism, and sessions are the storage mechanism. You don't choose one or the other — sessions need cookies to work.

Section 08

URL Rewriting

What happens when cookies are disabled? Sessions break because the JSESSIONID cookie can't be stored. URL rewriting is the fallback.

How It Works

Instead of storing the session ID in a cookie, the server appends it to every URL:

URL with Session ID Appended 
// Normal URL (with cookies)
http://nimbus.com/dashboard

// URL rewritten (without cookies)
http://nimbus.com/dashboard;jsessionid=A1B2C3D4E5F6

Using URL Rewriting in Servlets

response.encodeURL() and encodeRedirectURL() 
// For links in your page
String url = response.encodeURL("/dashboard");
// If cookies work: returns "/dashboard"
// If cookies disabled: returns "/dashboard;jsessionid=A1B2C3D4E5F6"

// For redirects
response.sendRedirect(response.encodeRedirectURL("/dashboard"));
URL Rewriting Drawbacks
  • Session ID visible in the URL — can be shared/bookmarked (security risk)
  • Every single link in every page must use encodeURL()
  • Session ID appears in server logs and browser history

In practice, most modern apps require cookies and show a "Please enable cookies" message if they're disabled.

Section 09

@SessionAttributes in Spring

Spring provides a cleaner way to work with sessions using the @SessionAttributes annotation on controllers.

Spring Controller with @SessionAttributes 
@Controller
@SessionAttributes("loggedInUser")  // "keep 'loggedInUser' in session"
public class LoginController {

    @PostMapping("/login")
    public String handleLogin(@RequestParam String username,
                              @RequestParam String password,
                              Model model) {
        User user = userService.authenticate(username, password);
        if (user != null) {
            model.addAttribute("loggedInUser", user);  // goes to session!
            return "redirect:/dashboard";
        }
        model.addAttribute("error", "Invalid credentials");
        return "login";
    }
}

@Controller
public class DashboardController {

    @GetMapping("/dashboard")
    public String showDashboard(@SessionAttribute("loggedInUser") User user,
                                Model model) {
        // Spring automatically gets user from session
        // Throws exception if not in session (user not logged in)
        model.addAttribute("bookings", bookingService.getBookings(user.getId()));
        return "dashboard";
    }
}
Spring session handling explained
2
@SessionAttributes("loggedInUser")
Tells Spring: "Whenever model.addAttribute("loggedInUser", ...) is called in this controller, also store it in the session. It persists across requests." Without this, model attributes only live for one request.
22
@SessionAttribute("loggedInUser") User user
Spring retrieves the "loggedInUser" from the session and injects it as a method parameter. If it's not in the session (user not logged in), Spring throws a 400 Bad Request. Clean alternative to manually calling session.getAttribute().
In Nimbus

When Nimbus moves to Spring MVC, the raw HttpSession calls in LoginServlet/DashboardServlet get replaced with @SessionAttributes and @SessionAttribute. The logic is the same, but Spring handles the plumbing. No more request.getSession(false) and null checks.

Section 10

Session Security

Sessions are the keys to your users' accounts. If an attacker gets a session ID, they become that user. Here are the attacks and defenses.

Session Fixation

Attack: Session Fixation

How it works: Attacker creates a session on the server, gets a session ID (say EVIL123), and tricks the victim into using that same session ID (via a crafted URL). When the victim logs in, their authenticated session uses ID EVIL123. The attacker already has that ID and is now logged in as the victim.

Defense: Generate a NEW session ID after successful login. Invalidate the old session first.

Preventing Session Fixation 
// After successful authentication:
// 1. Invalidate old session
request.getSession().invalidate();
// 2. Create a brand-new session with a new ID
HttpSession newSession = request.getSession(true);
// 3. Store user in the NEW session
newSession.setAttribute("loggedInUser", user);

Session Hijacking

Attack: Session Hijacking

How it works: Attacker steals the victim's session ID by sniffing network traffic (on public WiFi), XSS attack (stealing via JavaScript), or reading server logs.

Defenses:

  • HTTPS everywhere — encrypts all traffic, session ID can't be sniffed
  • HttpOnly cookies — JavaScript can't read the session cookie, blocking XSS theft
  • Secure flag — cookie only sent over HTTPS
  • Short session timeout — limits the window of attack
  • Regenerate session ID on privilege escalation (login, role change)

Security Checklist

MeasureWhat It PreventsHow to Implement
HTTPS Network sniffing SSL certificate + redirect HTTP to HTTPS
HttpOnly cookie flag XSS stealing session cookie.setHttpOnly(true)
Secure cookie flag Cookie sent over HTTP cookie.setSecure(true)
Session regeneration Session fixation Invalidate + new session after login
Short timeout Stale sessions setMaxInactiveInterval(900) (15 min)

Practice Exercises

Build session management skills from basics to full implementation.

Exercise 1: Session Basics (Easy)

Exercise 1

Create a CounterServlet that counts how many times the user has visited the page. Store the count in the session. Display: "You have visited this page X times." Refreshing the page should increment the counter. Opening a new browser (different session) should start at 1.

Exercise 2: Login + Logout (Medium)

Exercise 2

Build a complete login system:

  • A LoginServlet that accepts username/password (hardcode one valid user: admin/admin123)
  • On success: create session, store username, redirect to /welcome
  • A WelcomeServlet that checks if the user is logged in (session exists). If not, redirect to login.
  • A LogoutServlet that invalidates the session and redirects to login.

Test: login → see welcome → logout → try to visit /welcome directly → should redirect to login.

Exercise 3: Session vs Cookie (Medium)

Exercise 3

Create two Servlets:

  • PreferenceServlet — stores the user's preferred theme ("dark" or "light") in a persistent cookie (30 days)
  • ProfileServlet — stores sensitive user info (name, email) in the HttpSession

Explain in comments: why is theme in a cookie (harmless, persists across browser restarts) and user info in a session (sensitive, server-side only)?

Exercise 4: Session Timeout Handling (Medium-Hard)

Exercise 4

Set the session timeout to 1 minute. Build a protected page that:

  • Shows the user's name and a "Session active" message if logged in
  • Shows "Session expired. Please login again." if the session timed out
  • Displays the session creation time and last accessed time (use session.getCreationTime() and session.getLastAccessedTime())

Test: login, wait over 1 minute without any activity, refresh — you should see the expired message.

Exercise 5: Secure Login with Session Fixation Prevention (Hard)

Exercise 5

Modify your login system from Exercise 2 to include session fixation prevention:

  1. Before authentication, print the current session ID (if any)
  2. After successful authentication, invalidate the old session and create a new one
  3. Print the new session ID
  4. Verify they're different

Also add the HttpOnly flag to the JSESSIONID cookie. (Hint: configure this in web.xml using <cookie-config>.)
