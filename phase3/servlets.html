Section 01

What is a Servlet?

A Servlet is a Java class that lives inside a web server and handles HTTP requests from browsers. It is the bridge between the web and your Java code.

The Core Idea

Right now, Nimbus is a console app. The user types choices into a terminal, and NimbusApp.java reads them with Scanner. That works for learning, but real users use browsers. They click buttons, fill forms, and hit submit.

A Servlet is the Java class that receives those browser requests and sends back responses. Every form submission, every link click, every page load — a Servlet handles it on the server side.

Analogy: The Hotel Receptionist

Think of a web server as a hotel. Guests (browsers) walk in and make requests: "I need a room," "Where is the pool?" "Check me out."

The receptionist is the Servlet. She doesn't own the hotel (that's Tomcat). She doesn't clean the rooms (that's the Service/DAO layer). But she is the person who takes the request, figures out what to do, talks to the right department, and gives the guest an answer.

Each type of request can have its own receptionist. One handles check-ins (LoginServlet), another handles bookings (BookingServlet), another handles complaints (ErrorServlet). But they all follow the same process: receive → process → respond.

In Nimbus

Your console app has NimbusApp.java with a switch statement: case 1 = login, case 2 = register, case 3 = book a parcel. In the web version, each case becomes its own Servlet: LoginServlet, RegisterServlet, BookingServlet. The logic stays the same — only the input/output changes from terminal to HTTP.

Layer Responsibilities

Browser
HTML forms, links, buttons. Sends HTTP requests. Displays HTML responses. Knows nothing about Java.
Tomcat
Web server. Receives HTTP requests, finds the right Servlet based on URL mapping, creates HttpServletRequest and HttpServletResponse objects, calls the Servlet.
Servlet
Controller. Reads form data from request, calls the Service/DAO layer, puts results into request attributes, forwards to a JSP for display. No business logic here. No SQL here.
Service
Business logic. Validates data, calculates costs, enforces rules (e.g., "no duplicate emails"). Calls DAOs for database work.
DAO
Data Access. Runs SQL queries via JDBC. Returns Java objects (User, Booking). Knows nothing about HTTP or Servlets.
Common Mistake

Beginners often put SQL queries directly inside the Servlet. Never do this. The Servlet's only job is to handle HTTP — read request data and send responses. All database work goes through DAO classes, just like in your console app.

In Nimbus

Your existing UserDAO, BookingDAO, TrackingService, and all your model classes (User, Booking) stay exactly the same. The Servlet is just a new front door. Instead of Scanner feeding data into the DAO, the Servlet reads from HttpServletRequest and feeds it into the same DAO. Your entire backend is reused.

Section 03

Servlet Lifecycle

Tomcat manages the birth, life, and death of every Servlet. Understanding this lifecycle tells you when your code runs.

The Four Stages

A Servlet goes through four stages, managed entirely by Tomcat. You never call new LoginServlet() yourself — Tomcat does it.

1. Loading & Instantiation
Tomcat loads the Servlet class and creates one instance using the no-arg constructor. This happens either when the server starts (if load-on-startup is set) or when the first request hits that URL. Only one instance per Servlet is created — all requests share it.
2. Initialization — init()
Called once after the instance is created. Use this for one-time setup: loading config, opening database connection pools, initializing resources. If init() fails, the Servlet is not put into service.
3. Request Handling — service() → doGet()/doPost()
For every request, Tomcat calls service(), which checks the HTTP method and routes to doGet() or doPost(). This is where your actual logic runs. Called thousands of times — once per request. Each request runs in its own thread but uses the same Servlet instance.
4. Destruction — destroy()
Called once when the server shuts down or the app is undeployed. Use this to release resources: close database connections, flush logs. After destroy(), the instance is garbage collected.

One Servlet instance handles all requests. Each request gets its own thread.
Critical: Single Instance, Multiple Threads

Tomcat creates one LoginServlet object. When 50 users log in at the same time, 50 threads run doPost() on that same object. This means you must never store request-specific data in instance variables. Use local variables inside doGet()/doPost() instead — each thread gets its own copy of local variables.

Lifecycle Demo 
public class LoginServlet extends HttpServlet {

    // Called ONCE when Tomcat creates this Servlet
    @Override
    public void init() throws ServletException {
        System.out.println("LoginServlet initialized");
        // One-time setup: load config, etc.
    }

    // Called for EVERY GET request to this Servlet's URL
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {
        // Show the login page
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);
    }

    // Called for EVERY POST request to this Servlet's URL
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {
        // Process login form submission
        String email = request.getParameter("email");
        String password = request.getParameter("password");
        // ... validate, create session, redirect
    }

    // Called ONCE when the server shuts down
    @Override
    public void destroy() {
        System.out.println("LoginServlet destroyed");
        // Cleanup resources
    }
}

Section 04

HttpServletRequest — Reading Incoming Data

The request object contains everything the browser sent: form data, URL parameters, headers, cookies, and session info. This is how your Servlet knows what the user wants.

The Most Important Methods

MethodReturnsWhat It Does
getParameter("name") String Gets a form field or query string value. Returns null if the parameter doesn't exist.
getParameterValues("name") String[] Gets multiple values for the same parameter name (e.g., checkboxes).
getAttribute("key") Object Gets data set by a Servlet (not from the browser). Used to pass data from Servlet to JSP.
setAttribute("key", value) void Stores data in the request. The JSP can then read it with ${key}.
getSession() HttpSession Gets the current user's session (creates one if none exists). Used to track logged-in users.
getSession(false) HttpSession Gets existing session or null — does NOT create a new one. Use this to check if user is logged in.
getRequestDispatcher("path") RequestDispatcher Gets a dispatcher to forward or include another resource (JSP, Servlet).
getMethod() String Returns "GET", "POST", "PUT", etc. Usually you don't need this since doGet/doPost already separate them.
getRequestURI() String Returns the URL path, e.g., /nimbus/login.
getHeader("name") String Reads HTTP headers (e.g., User-Agent, Content-Type).
getContextPath() String Returns the app's base path, e.g., /nimbus. Useful for building URLs.

getParameter() — The Method You'll Use Most

Every form field has a name attribute in HTML. When the form is submitted, those names become the parameter keys you read in the Servlet.

login.jsp — HTML Form 
<form action="login" method="post">
    <input type="email" name="email" placeholder="Email" />
    <input type="password" name="password" placeholder="Password" />
    <button type="submit">Login</button>
</form>
LoginServlet.java — Reading Form Data 
@Override
protected void doPost(HttpServletRequest request,
                      HttpServletResponse response)
        throws ServletException, IOException {

    // These names MUST match the HTML input "name" attributes
    String email = request.getParameter("email");       // from name="email"
    String password = request.getParameter("password"); // from name="password"

    // Always check for null/empty
    if (email == null || email.trim().isEmpty()) {
        request.setAttribute("error", "Email is required");
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);
        return;
    }

    // Now use email and password with your existing UserDAO
    UserDAO userDAO = new UserDAO();
    User user = userDAO.authenticate(email, password);
    // ...
}
Parameter vs Attribute — Don't Confuse Them

Parameters come from the browser (form fields, query strings). You only read them: getParameter().

Attributes are set by your Java code to pass data to JSPs. You read and write them: setAttribute() and getAttribute().

Think of it this way: Parameters = what the user sent you. Attributes = what you're sending to the JSP.

setAttribute() — Passing Data to JSP

TrackingServlet.java — Setting Attributes for JSP 
// In TrackingServlet.doGet()
String bookingId = request.getParameter("bookingId"); // from query string

TrackingService trackingService = new TrackingService();
Booking booking = trackingService.getTrackingInfo(bookingId);

// Store data as request attributes — JSP will read these
request.setAttribute("booking", booking);
request.setAttribute("status", booking.getStatus());
request.setAttribute("estimatedDelivery", booking.getEstimatedDate());

// Forward to JSP — it can now access ${booking}, ${status}, etc.
request.getRequestDispatcher("/WEB-INF/tracking.jsp")
       .forward(request, response);
Section 05

HttpServletResponse — Sending Data Back

The response object is how your Servlet sends data back to the browser. You can send HTML, redirect to another URL, set cookies, or return error codes.

Key Methods

MethodWhat It Does
setContentType("text/html") Tells the browser what kind of data you're sending. text/html for web pages, application/json for API responses.
getWriter() Returns a PrintWriter to write text/HTML directly to the response. Rarely used when you have JSP.
sendRedirect("url") Tells the browser to go to a different URL. Sends HTTP 302. The URL in the browser changes.
sendError(404) Sends an HTTP error status code to the browser.
setStatus(200) Sets the HTTP status code. 200 = OK, 404 = Not Found, 500 = Server Error.
addCookie(cookie) Adds a cookie to the response. The browser will store it and send it back with future requests.
setHeader("name", "value") Sets an HTTP response header (e.g., caching, content-disposition for downloads).

Writing HTML Directly (Quick & Dirty)

You can write HTML directly from a Servlet, but you almost never should. JSPs are much better for this. Still, it's important to understand how it works:

Writing HTML from a Servlet (NOT recommended) 
@Override
protected void doGet(HttpServletRequest request,
                     HttpServletResponse response)
        throws ServletException, IOException {

    response.setContentType("text/html");
    PrintWriter out = response.getWriter();

    out.println("<html>");
    out.println("<body>");
    out.println("<h1>Welcome to Nimbus</h1>");
    out.println("<p>Booking ID: " + request.getParameter("id") + "</p>");
    out.println("</body>");
    out.println("</html>");
}
Why This is Bad

Mixing Java code and HTML is a maintenance nightmare. Imagine styling a full page with CSS inside Java strings. That's why JSPs exist — they let you write HTML normally and embed small amounts of Java. In Nimbus, Servlets handle logic, JSPs handle display.

sendRedirect() — Sending User to Another URL

Redirect after successful login 
// After successful login, redirect based on role
User user = userDAO.authenticate(email, password);

if (user != null) {
    HttpSession session = request.getSession();
    session.setAttribute("loggedInUser", user);

    // Redirect based on user role
    if (user.getRole().equals("ADMIN")) {
        response.sendRedirect("admin/dashboard");
    } else if (user.getRole().equals("DELIVERY_AGENT")) {
        response.sendRedirect("agent/deliveries");
    } else {
        response.sendRedirect("customer/home");
    }
} else {
    request.setAttribute("error", "Invalid email or password");
    request.getRequestDispatcher("/WEB-INF/login.jsp")
           .forward(request, response);
}
Section 06

doGet() vs doPost()

GET is for reading. POST is for writing. Getting this right is fundamental to building web applications correctly.

When to Use Each

doGet() — Reading Data

Use when: User is requesting/viewing data.

  • Navigating to a page
  • Clicking a link
  • Searching for something
  • Tracking a parcel (just viewing)

Data goes in: URL query string
/track?bookingId=NB-1042

Browser behavior:

  • Can be bookmarked
  • Appears in browser history
  • Can be refreshed without warning
  • Data is visible in the URL
doPost() — Sending Data

Use when: User is submitting/modifying data.

  • Submitting a login form
  • Registering a new account
  • Booking a parcel
  • Updating delivery status

Data goes in: Request body (hidden)
POST /login (data in body)

Browser behavior:

  • Cannot be bookmarked
  • Not in browser history
  • Refreshing shows "re-submit?" warning
  • Data is hidden from the URL

How the HTML Form Decides

The HTML form's method attribute determines which Servlet method gets called:

Form method determines Servlet method 
<!-- This triggers doGet() on the Servlet -->
<form action="track" method="get">
    <input name="bookingId" />
    <button>Track Parcel</button>
</form>
<!-- URL becomes: /nimbus/track?bookingId=NB-1042 -->


<!-- This triggers doPost() on the Servlet -->
<form action="login" method="post">
    <input name="email" />
    <input name="password" type="password" />
    <button>Login</button>
</form>
<!-- URL stays: /nimbus/login (data hidden in body) -->
In Nimbus — Which Method for Each Servlet

Rule of Thumb

If the action changes data on the server (insert, update, delete) → POST.
If the action only reads data (view, search, display) → GET.

Never use GET for passwords, payments, or anything that modifies the database. That data would appear in the URL, browser history, and server logs.

Section 07

web.xml — Servlet Configuration

The deployment descriptor tells Tomcat which Servlet handles which URL. It's the routing table of your web application.

How Tomcat Knows Which Servlet to Call

When a browser requests /nimbus/login, how does Tomcat know to call LoginServlet? Two ways:

  1. web.xml — An XML file that maps URLs to Servlet classes. The older, traditional way.
  2. @WebServlet annotation — A modern annotation directly on the Servlet class. Simpler, no XML needed.

Most real projects use annotations, but you must understand web.xml because interview questions love it, and some enterprise projects still use it.

@WebServlet Annotation — The Modern Way

Instead of writing XML for every Servlet, you can use a single annotation on the class itself. Same result, much less config.

web.xml vs @WebServlet

web.xml (Old Way)
<servlet>
  <servlet-name>LoginServlet</servlet-name>
  <servlet-class>
    com.nimbus.servlet.LoginServlet
  </servlet-class>
</servlet>
<servlet-mapping>
  <servlet-name>LoginServlet</servlet-name>
  <url-pattern>/login</url-pattern>
</servlet-mapping>

10 lines of XML per Servlet. 8 Servlets = 80 lines of boilerplate.

@WebServlet (Modern Way)
@WebServlet("/login")
public class LoginServlet
    extends HttpServlet {
    // ...
}

1 line per Servlet. The URL mapping lives right next to the code it controls.

// Simple — single URL pattern

public class LoginServlet extends HttpServlet { ... }

// Multiple URL patterns — same Servlet handles both

public class LoginServlet extends HttpServlet { ... }

// With name and load-on-startup
@WebServlet(
    name = "LoginServlet",
    urlPatterns = "/login",

)
public class LoginServlet extends HttpServlet { ... }

// Wildcard pattern — matches /api/anything
@WebServlet("/api/*")
public class ApiServlet extends HttpServlet { ... }

// Extension pattern — matches any .do URL
@WebServlet("*.do")
public class ActionServlet extends HttpServlet { ... }

Don't Mix Both

If you use @WebServlet annotations, you can remove the <servlet> and <servlet-mapping> entries from web.xml for those Servlets. You can still keep web.xml for things like error pages and welcome files. But never register the same Servlet in both places — it will cause conflicts.

Section 09

Request Parameters — Reading User Input

Parameters are how user input arrives at your Servlet. They come from form fields (POST) or query strings (GET).

Where Parameters Come From

GET — Query String

Parameters are in the URL after ?:

/track?bookingId=NB-1042&format=detail

Multiple params separated by &. Visible to everyone. Used for searches, filters, links.

POST — Request Body

Parameters are hidden in the HTTP body:

email=john@test.com&password=secret

Not visible in URL. Used for form submissions with sensitive data.

Both are read the same way in your Servlet: request.getParameter("name"). The Servlet doesn't care whether it came from a query string or a form body.

Handling Missing Parameters

If a parameter doesn't exist, getParameter() returns null. If it exists but is empty (user submitted blank field), it returns an empty string "". You must handle both.

Safe parameter reading pattern 
@Override
protected void doPost(HttpServletRequest request,
                      HttpServletResponse response)
        throws ServletException, IOException {

    String email = request.getParameter("email");
    String password = request.getParameter("password");

    // Check for null (parameter not sent at all)
    // Check for empty (parameter sent but blank)
    if (email == null || email.trim().isEmpty()) {
        request.setAttribute("error", "Email is required");
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);
        return;  // IMPORTANT: stop processing after forward
    }

    if (password == null || password.trim().isEmpty()) {
        request.setAttribute("error", "Password is required");
        request.setAttribute("email", email); // Keep the email they typed
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);
        return;
    }

    // Both fields have values — safe to proceed
    // ...
}

Reading Numeric Parameters

Forwarding to JSP — RequestDispatcher

After the Servlet processes the request, it forwards to a JSP page for display. This is the core of the MVC pattern: Servlet = Controller, JSP = View.

The Forward Process

BrowserPOST /login
→
LoginServletdoPost() validates user
→
setAttribute()Stores user in request
→
forward()To dashboard.jsp
→
BrowserSees dashboard HTML
The browser URL stays as /login — it doesn't know about the JSP

The Two Steps: setAttribute() + forward()

HistoryServlet.java — Complete forward example 
@WebServlet("/history")
public class HistoryServlet extends HttpServlet {

    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // Step 1: Read parameters
        String userIdStr = request.getParameter("userId");
        String fromDate = request.getParameter("from");
        String toDate = request.getParameter("to");

        // Step 2: Call DAO to get data
        BookingDAO bookingDAO = new BookingDAO();
        int userId = Integer.parseInt(userIdStr);
        List<Booking> bookings = bookingDAO.getHistory(userId, fromDate, toDate);

        // Step 3: Store data as request attributes
        request.setAttribute("bookings", bookings);     // JSP reads as ${bookings}
        request.setAttribute("totalBookings", bookings.size());
        request.setAttribute("fromDate", fromDate);
        request.setAttribute("toDate", toDate);

        // Step 4: Forward to JSP for display
        request.getRequestDispatcher("/WEB-INF/history.jsp")
               .forward(request, response);
    }
}
Why /WEB-INF/ for JSPs?

Files inside /WEB-INF/ cannot be accessed directly from a browser. If someone types http://localhost:8080/nimbus/WEB-INF/history.jsp, they get a 404. This is good — it forces users to go through your Servlet first, where you can validate input and check permissions. Always put JSPs in /WEB-INF/.

Section 11

Redirect vs Forward — The Critical Difference

These two concepts confuse almost every beginner. Getting them wrong causes duplicate form submissions, lost data, and broken navigation. This is a must-know for interviews.

Visual Comparison

  FORWARD (Server-side)
  ======================

  Browser                    Server
  +--------+                +------------------+
  |        | --- Request -->| LoginServlet     |
  |        |                |   validates...   |
  |        |                |   forward() ---->| dashboard.jsp
  |        | <--- HTML -----|                  | generates HTML
  +--------+                +------------------+

  URL in browser: /login     (UNCHANGED!)
  Number of requests: 1
  Request attributes: PRESERVED (JSP can read them)


  REDIRECT (Client-side)
  =======================

  Browser                    Server
  +--------+                +------------------+
  |        | --- POST ----->| LoginServlet     |
  |        |                |   validates...   |
  |        | <-- 302 ------| sendRedirect()   |
  |        |                +------------------+
  |        | --- GET ------>| DashboardServlet |
  |        | <--- HTML -----|                  |
  +--------+                +------------------+

  URL in browser: /dashboard (CHANGED!)
  Number of requests: 2
  Request attributes: LOST (new request = clean slate)
Forward = one trip. Redirect = two trips.

When to Use Each

ScenarioUseWhy
Login failed — show error on login page Forward Need to pass the error message via request attribute. Same request.
Login succeeded — go to dashboard Redirect New page. Clean URL. Prevents "re-submit form?" on refresh.
Booking created — show confirmation Redirect POST-Redirect-GET pattern. Refreshing won't create duplicate booking.
Tracking result — show tracking page Forward Passing tracking data via request attributes to JSP.
User not logged in — go to login page Redirect Clean URL change. User sees /login in browser.

POST-Redirect-GET Pattern (PRG)

This is one of the most important patterns in web development. It solves the "double submission" problem.

The Problem Without PRG

User books a parcel (POST). Server creates booking, forwards to confirmation page. User refreshes the page. Browser asks "Re-submit form data?" User clicks Yes. Another booking is created. Duplicate charge.

The Solution: POST-Redirect-GET

User books a parcel (POST). Server creates booking, redirects to /booking-confirmation?id=42. Browser makes a new GET request. User refreshes the page. Browser just re-sends the GET — no form data, no duplicate. Safe.

PRG Pattern in BookingServlet 
@WebServlet("/book")
public class BookingServlet extends HttpServlet {

    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        // 1. Read form data
        String senderName = request.getParameter("senderName");
        String receiverName = request.getParameter("receiverName");
        double weight = Double.parseDouble(request.getParameter("weight"));

        // 2. Process booking
        BookingDAO bookingDAO = new BookingDAO();
        Booking booking = new Booking(senderName, receiverName, weight);
        int bookingId = bookingDAO.insertBooking(booking);

        // 3. REDIRECT (not forward!) — PRG pattern
        // Store success message in session (survives redirect)
        HttpSession session = request.getSession();
        session.setAttribute("successMessage", "Booking created! ID: " + bookingId);

        // Redirect to confirmation page (GET request)
        response.sendRedirect("invoice?bookingId=" + bookingId);
        // User refreshes? Just re-loads the invoice. No duplicate booking.
    }
}
Why each step matters
1
Read form data from POST
The form was submitted with method="post", so data is in the request body. getParameter() reads it.
2
Process — insert into database
The booking is saved. The database returns the generated booking ID. At this point, the data is committed.
3
Redirect instead of forward
This is the key step. sendRedirect() sends HTTP 302 to the browser. The browser makes a new GET request to /invoice?bookingId=42. If the user refreshes, they only repeat the GET — the POST is gone from browser history.
4
Session for flash messages
Since redirect creates a new request, request.setAttribute() won't survive. So we use session.setAttribute() for the success message. The JSP reads it and then removes it (so it only shows once).

Full Nimbus LoginServlet

The complete LoginServlet with form display, validation, session creation, and role-based redirect. This is the first Servlet you build for Nimbus.

com/nimbus/servlet/LoginServlet.java 
package com.nimbus.servlet;

import com.nimbus.dao.UserDAO;
import com.nimbus.model.User;
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;
import java.io.IOException;

@WebServlet("/login")
public class LoginServlet extends HttpServlet {

    private UserDAO userDAO;

    @Override
    public void init() throws ServletException {
        userDAO = new UserDAO();
    }

    // GET /login — Show the login form
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // If already logged in, redirect to home
        HttpSession session = request.getSession(false);
        if (session != null && session.getAttribute("loggedInUser") != null) {
            response.sendRedirect("home");
            return;
        }

        // Show login page
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);
    }

    // POST /login — Process login form
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        // 1. Read form data
        String email = request.getParameter("email");
        String password = request.getParameter("password");

        // 2. Validate input
        if (email == null || email.trim().isEmpty()
                || password == null || password.trim().isEmpty()) {
            request.setAttribute("error", "Email and password are required");
            request.getRequestDispatcher("/WEB-INF/login.jsp")
                   .forward(request, response);
            return;
        }

        // 3. Authenticate using existing UserDAO
        User user = userDAO.authenticate(email.trim(), password);

        if (user == null) {
            // Login failed — forward back to login with error
            request.setAttribute("error", "Invalid email or password");
            request.setAttribute("email", email); // Keep the email they typed
            request.getRequestDispatcher("/WEB-INF/login.jsp")
                   .forward(request, response);
            return;
        }

        // 4. Login succeeded — create session
        HttpSession session = request.getSession();
        session.setAttribute("loggedInUser", user);
        session.setAttribute("userId", user.getUserId());
        session.setAttribute("userName", user.getName());
        session.setAttribute("userRole", user.getRole());
        session.setMaxInactiveInterval(30 * 60); // 30 minutes

        // 5. Redirect based on role (PRG pattern)
        switch (user.getRole()) {
            case "ADMIN":
                response.sendRedirect("admin/dashboard");
                break;
            case "DELIVERY_AGENT":
                response.sendRedirect("agent/deliveries");
                break;
            case "CUSTOMER":
            default:
                response.sendRedirect("customer/home");
                break;
        }
    }
}
LoginServlet line-by-line
1
private UserDAO userDAO; + init()
The DAO is created once in init(), not in every request. This is efficient — all threads share one DAO instance. The DAO itself is stateless (no instance variables holding request data), so this is safe.
2
doGet() — request.getSession(false)
Passing false means "give me the existing session or null, don't create a new one." If the user is already logged in (session exists with user data), we skip the login page and redirect to home.
3
doPost() — Validation
Always validate before processing. If input is missing, we forward (not redirect) back to login.jsp. Forward preserves the error message we set as a request attribute.
4
request.setAttribute("email", email)
When login fails, we send the email back so the user doesn't have to retype it. The JSP reads this: <input value="${email}">. Small UX detail that matters.
5
session.setMaxInactiveInterval(30 * 60)
Session expires after 30 minutes of inactivity. After that, getSession(false) returns null, and the user is effectively logged out. The value is in seconds.
6
response.sendRedirect() after successful login
PRG pattern. After POST login, we redirect (not forward). The browser makes a new GET request to the dashboard URL. Refreshing won't re-submit the login form.
In Nimbus — Console vs Web Login

Console: Scanner reads email/password → userDAO.authenticate() → System.out.println("Welcome").

Web: request.getParameter() reads email/password → same userDAO.authenticate() → session.setAttribute() + response.sendRedirect().

The DAO call is identical. The only change is where input comes from and where output goes.

Full Nimbus BookingServlet

The BookingServlet handles parcel booking: reads form data, calculates cost, saves to database, and redirects to invoice.

com/nimbus/servlet/BookingServlet.java 
package com.nimbus.servlet;

import com.nimbus.dao.BookingDAO;
import com.nimbus.model.Booking;
import com.nimbus.model.User;
import jakarta.servlet.ServletException;
import jakarta.servlet.annotation.WebServlet;
import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import jakarta.servlet.http.HttpSession;
import java.io.IOException;

@WebServlet("/book")
public class BookingServlet extends HttpServlet {

    private BookingDAO bookingDAO;

    @Override
    public void init() throws ServletException {
        bookingDAO = new BookingDAO();
    }

    // GET /book — Show the booking form
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        // Must be logged in
        HttpSession session = request.getSession(false);
        if (session == null || session.getAttribute("loggedInUser") == null) {
            response.sendRedirect("login");
            return;
        }

        request.getRequestDispatcher("/WEB-INF/booking.jsp")
               .forward(request, response);
    }

    // POST /book — Process the booking
    @Override
    protected void doPost(HttpServletRequest request,
                          HttpServletResponse response)
            throws ServletException, IOException {

        // Must be logged in
        HttpSession session = request.getSession(false);
        if (session == null || session.getAttribute("loggedInUser") == null) {
            response.sendRedirect("login");
            return;
        }

        User loggedInUser = (User) session.getAttribute("loggedInUser");

        try {
            // 1. Read form parameters
            String senderName = request.getParameter("senderName");
            String senderAddress = request.getParameter("senderAddress");
            String senderPhone = request.getParameter("senderPhone");
            String receiverName = request.getParameter("receiverName");
            String receiverAddress = request.getParameter("receiverAddress");
            String receiverPhone = request.getParameter("receiverPhone");
            double weight = Double.parseDouble(request.getParameter("weight"));
            String parcelType = request.getParameter("parcelType");

            // 2. Create Booking object
            Booking booking = new Booking();
            booking.setSenderId(loggedInUser.getUserId());
            booking.setSenderName(senderName);
            booking.setSenderAddress(senderAddress);
            booking.setSenderPhone(senderPhone);
            booking.setReceiverName(receiverName);
            booking.setReceiverAddress(receiverAddress);
            booking.setReceiverPhone(receiverPhone);
            booking.setWeight(weight);
            booking.setParcelType(parcelType);

            // 3. Calculate cost (same logic as console app)
            double baseCost = weight * 50; // Rs. 50 per kg
            if ("EXPRESS".equals(parcelType)) {
                baseCost *= 1.5;
            }
            booking.setCost(baseCost);
            booking.setStatus("BOOKED");

            // 4. Save to database
            int bookingId = bookingDAO.insertBooking(booking);

            // 5. PRG — Redirect to invoice (not forward!)
            session.setAttribute("successMessage",
                "Parcel booked successfully! Booking ID: NB-" + bookingId);
            response.sendRedirect("invoice?bookingId=" + bookingId);

        } catch (NumberFormatException e) {
            request.setAttribute("error", "Invalid weight value");
            request.getRequestDispatcher("/WEB-INF/booking.jsp")
                   .forward(request, response);
        } catch (Exception e) {
            request.setAttribute("error", "Booking failed: " + e.getMessage());
            request.getRequestDispatcher("/WEB-INF/booking.jsp")
                   .forward(request, response);
        }
    }
}
In Nimbus — What Changed from Console

Console: Scanner reads each field one by one with prompts. Cost calculation inline. Result printed to terminal.

Web: All fields arrive at once from the form. Same cost calculation. Result saved, then redirect to invoice page (PRG pattern). The Booking model and BookingDAO are the exact same classes.

com/nimbus/servlet/TrackingServlet.java 
package com.nimbus.servlet;

import com.nimbus.dao.BookingDAO;
import com.nimbus.model.Booking;
import com.nimbus.service.TrackingService;
import com.nimbus.service.TrackingInfo;
import jakarta.servlet.ServletException;

import jakarta.servlet.http.HttpServlet;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.io.IOException;

@WebServlet("/track")
public class TrackingServlet extends HttpServlet {

    private BookingDAO bookingDAO;
    private TrackingService trackingService;

    @Override
    public void init() throws ServletException {
        bookingDAO = new BookingDAO();
        trackingService = new TrackingService();
    }

    // GET /track?bookingId=NB-1042
    @Override
    protected void doGet(HttpServletRequest request,
                         HttpServletResponse response)
            throws ServletException, IOException {

        String bookingIdParam = request.getParameter("bookingId");

        // No bookingId provided — show the search form
        if (bookingIdParam == null || bookingIdParam.trim().isEmpty()) {
            request.getRequestDispatcher("/WEB-INF/tracking-search.jsp")
                   .forward(request, response);
            return;
        }

        try {
            // Parse the booking ID (strip "NB-" prefix if present)
            String cleanId = bookingIdParam.replace("NB-", "");
            int bookingId = Integer.parseInt(cleanId);

            // Get booking from database
            Booking booking = bookingDAO.findById(bookingId);

            if (booking == null) {
                request.setAttribute("error", "Booking not found: " + bookingIdParam);
                request.getRequestDispatcher("/WEB-INF/tracking-search.jsp")
                       .forward(request, response);
                return;
            }

            // POLYMORPHISM: TrackingService decides which tracker to use
            // based on the parcel type (standard, express, fragile).
            // This is the SAME polymorphism from your console app.
            TrackingInfo trackingInfo = trackingService.track(booking);

            // Set attributes for JSP
            request.setAttribute("booking", booking);
            request.setAttribute("trackingInfo", trackingInfo);
            request.setAttribute("statusHistory", trackingInfo.getStatusHistory());
            request.setAttribute("estimatedDelivery", trackingInfo.getEstimatedDate());
            request.setAttribute("currentLocation", trackingInfo.getCurrentLocation());

            // Forward to tracking result page
            request.getRequestDispatcher("/WEB-INF/tracking-result.jsp")
                   .forward(request, response);

        } catch (NumberFormatException e) {
            request.setAttribute("error", "Invalid booking ID format");
            request.getRequestDispatcher("/WEB-INF/tracking-search.jsp")
                   .forward(request, response);
        }
    }
}

Section 15

Error Handling in Servlets

Things will go wrong: database down, invalid input, null pointers. A well-built web app handles errors gracefully instead of showing stack traces to users.

Three Levels of Error Handling

Level 1: Try-Catch Inside the Servlet

Handle specific errors where they happen. This is your first line of defense.

Handling errors in the Servlet 
@Override
protected void doPost(HttpServletRequest request,
                      HttpServletResponse response)
        throws ServletException, IOException {

    try {
        String email = request.getParameter("email");
        String password = request.getParameter("password");

        User user = userDAO.authenticate(email, password);

        if (user == null) {
            // Expected error — user-friendly message
            request.setAttribute("error", "Invalid email or password");
            request.getRequestDispatcher("/WEB-INF/login.jsp")
                   .forward(request, response);
            return;
        }

        // ... success logic ...

    } catch (SQLException e) {
        // Database error — log it, show generic message
        log("Database error during login", e);
        request.setAttribute("error", "Service temporarily unavailable. Please try again.");
        request.getRequestDispatcher("/WEB-INF/login.jsp")
               .forward(request, response);

    } catch (Exception e) {
        // Unexpected error — let it bubble up to error page
        throw new ServletException("Login processing failed", e);
    }
}

Level 2: Custom Error Pages in web.xml

When an unhandled exception escapes the Servlet, or when Tomcat generates an HTTP error (404, 500), web.xml error pages catch it.

web.xml — Error page configuration 
<!-- When user visits a URL that doesn't exist -->
<error-page>
    <error-code>404</error-code>
    <location>/WEB-INF/error/404.jsp</location>
</error-page>

<!-- When server encounters an internal error -->
<error-page>
    <error-code>500</error-code>
    <location>/WEB-INF/error/500.jsp</location>
</error-page>

<!-- When any Java exception goes unhandled -->
<error-page>
    <exception-type>java.lang.Exception</exception-type>
    <location>/WEB-INF/error/500.jsp</location>
</error-page>

<!-- Specific exception type -->
<error-page>
    <exception-type>com.nimbus.exception.DuplicateEmailException</exception-type>
    <location>/WEB-INF/error/duplicate-email.jsp</location>
</error-page>

Level 3: sendError() for Programmatic Errors

Using sendError() in Servlets 
// Missing required parameter
if (bookingIdParam == null) {
    response.sendError(HttpServletResponse.SC_BAD_REQUEST,
        "Missing required parameter: bookingId");
    return; // This triggers the 400 error page
}

// Resource not found
Booking booking = bookingDAO.findById(bookingId);
if (booking == null) {
    response.sendError(HttpServletResponse.SC_NOT_FOUND,
        "Booking not found: " + bookingId);
    return; // This triggers the 404 error page
}

// Unauthorized access
if (!"ADMIN".equals(session.getAttribute("userRole"))) {
    response.sendError(HttpServletResponse.SC_FORBIDDEN,
        "Admin access required");
    return; // This triggers the 403 error page
}
Never Show Stack Traces to Users

When an exception happens, log the full stack trace on the server (using log() or a logger) but show the user a friendly message like "Something went wrong. Please try again." Stack traces in the browser expose your internal code structure — a security risk and terrible UX.
