Section 01

Client-Server Model

Every time you open a website, two computers are talking to each other. One asks, one answers. That's the entire internet in two sentences.

Restaurant Analogy

You (the client) sit at a table and place an order. The kitchen (the server) prepares your food and sends it back. You never walk into the kitchen. The kitchen never walks to your table uninvited. There's a clear protocol: you ask, they deliver.

A web browser is you at the table. A web server is the kitchen. HTTP is the waiter — the rules for how orders are placed and food is delivered.

The Two Players

  • Client — the program that sends requests. Usually a web browser (Chrome, Firefox), but can also be a mobile app, Postman, or another server.
  • Server — the program that receives requests, processes them, and sends back responses. Runs 24/7 waiting for incoming connections. Examples: Apache Tomcat, Nginx, Node.js.
Client Browser / Mobile App
→ HTTP Request →
Server Tomcat / Nginx
Client Renders the page
← HTTP Response ←
Server Processes & returns data
Request-response cycle — the foundation of every web interaction

Key principle: the client always initiates. The server never sends data unless asked. This is called a request-response model.

Section 02

What is HTTP?

HTTP stands for HyperText Transfer Protocol. It's the set of rules that clients and servers follow to communicate. Without these rules, they wouldn't understand each other.

Breaking Down the Name

  • HyperText — originally designed for HTML documents (text with links). Now carries JSON, images, videos, anything.
  • Transfer — moving data from one computer to another over the internet.
  • Protocol — a set of rules both sides agree to follow. Like how a phone call has rules: you say "hello," the other person responds, you take turns talking.
Key Properties of HTTP
  • Stateless — every request is independent. The server doesn't remember previous requests. Each request must contain ALL the information the server needs.
  • Text-based — HTTP messages are plain text (human-readable). You can literally read a raw HTTP request.
  • Port 80 (HTTP) or Port 443 (HTTPS) by default.
Section 03

URL Anatomy

A URL (Uniform Resource Locator) is the address of a resource on the web. Every part has a specific purpose.

  https://www.nimbus.com:8080/api/bookings?status=active&page=2#results
  ──┬──   ──────┬──────  ─┬─  ─────┬─────  ──────────┬──────────  ──┬───
protocol      host      port     path            query string     fragment
Every URL follows this structure. Most parts are optional.
Part Example Purpose
protocol https:// Which rules to use (HTTP or HTTPS for encrypted)
host www.nimbus.com Which server to talk to (domain name or IP address)
port :8080 Which door on the server (default 80 for HTTP, 443 for HTTPS, 8080 for Tomcat)
path /api/bookings Which resource on the server (like a file path)
query string ?status=active&page=2 Extra parameters sent as key=value pairs, separated by &
fragment #results Jump to a section on the page (handled by browser only, NOT sent to server)
In Nimbus

When you run Nimbus locally, the URL is http://localhost:8080/nimbus/bookings?userId=5. Here: protocol = http, host = localhost (your own machine), port = 8080 (Tomcat's default), path = /nimbus/bookings, query = userId=5.

Section 04

HTTP Request

When your browser asks the server for something, it sends an HTTP request. Every request has three parts: request line, headers, and (optionally) a body.

Structure of an HTTP Request

Raw HTTP Request — What the browser actually sends 
GET /api/bookings?userId=5 HTTP/1.1
Host: www.nimbus.com
Accept: text/html, application/json
User-Agent: Mozilla/5.0 (Windows NT 10.0)
Cookie: JSESSIONID=abc123
Connection: keep-alive
Line-by-line explanation
1
GET /api/bookings?userId=5 HTTP/1.1
Request line. Three parts: method (GET = "give me data"), path (/api/bookings?userId=5 = which resource), protocol version (HTTP/1.1).
2
Host: www.nimbus.com
Which server this request is for. Required in HTTP/1.1 because one server can host multiple websites.
3
Accept: text/html, application/json
Tells the server what formats the client can understand. "I can read HTML and JSON — send me whichever you prefer."
4
User-Agent: Mozilla/5.0 (Windows NT 10.0)
Identifies the client. Servers use this to send different content to mobile vs desktop browsers.
5
Cookie: JSESSIONID=abc123
Sends stored cookies back to the server. This is how sessions work — the server gave you this ID earlier, and you send it back so it knows who you are.

Request with a Body (POST)

When you send data to the server (like submitting a form or creating a booking), the request includes a body:

Raw HTTP POST Request — Sending data to create a booking 
POST /api/bookings HTTP/1.1
Host: www.nimbus.com
Content-Type: application/json
Content-Length: 98

{
    "senderName": "John Doe",
    "receiverName": "Jane Smith",
    "weight": 2.5,
    "destination": "Mumbai"
}

Notice the blank line between headers and body. That blank line is required — it tells the server "headers are done, body starts now."

Section 05

HTTP Response

The server processes your request and sends back a response. Same structure: status line, headers, body.

Raw HTTP Response — What the server sends back 
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 156
Set-Cookie: JSESSIONID=xyz789; Path=/; HttpOnly
Date: Wed, 23 Apr 2026 10:30:00 GMT

{
    "bookingId": 1001,
    "senderName": "John Doe",
    "receiverName": "Jane Smith",
    "status": "BOOKED",
    "trackingNumber": "NIM-20260423-1001"
}
Line-by-line explanation
1
HTTP/1.1 200 OK
Status line. Protocol version + status code (200) + reason phrase (OK). This means "your request succeeded."
2
Content-Type: application/json
Tells the browser what format the body is in. The browser uses this to decide how to render it (parse as JSON, display as HTML, show as image, etc.).
3
Content-Length: 156
Size of the body in bytes. The browser knows exactly how many bytes to read.
4
Set-Cookie: JSESSIONID=xyz789; Path=/; HttpOnly
Server tells the browser: "Store this cookie. Send it back with every future request to /. HttpOnly means JavaScript can't access it (security)."
Section 06

HTTP Methods Deep Dive

HTTP methods (also called "verbs") tell the server what action to perform. Think of them as different types of orders at a restaurant.

GET — Read Data

  • Purpose: Retrieve/read a resource. "Give me this data."
  • Has body? No. Data goes in the URL (query string).
  • Cacheable? Yes. Browsers can cache GET responses.
  • Bookmarkable? Yes. The full URL with parameters can be saved.
  • Safe? Yes. Should never modify server data.
  • Idempotent? Yes. Calling it 10 times gives the same result as calling it once.
GET Examples 
GET /api/bookings              // Get all bookings
GET /api/bookings/1001         // Get booking #1001
GET /api/users?role=customer   // Get all customers
GET /api/tracking?id=NIM-001   // Track a parcel

POST — Create / Submit

  • Purpose: Submit data to create a new resource. "Here's data, do something with it."
  • Has body? Yes. Data goes in the request body.
  • Cacheable? No.
  • Bookmarkable? No. Data is in the body, not the URL.
  • Safe? No. Creates or modifies data on the server.
  • Idempotent? No. Calling it twice creates two resources.
POST Example 
POST /api/bookings
Content-Type: application/json

{"senderName": "John", "destination": "Mumbai", "weight": 2.5}

PUT — Update (Full Replace)

  • Purpose: Replace an entire resource with new data. "Here's the complete updated version."
  • Has body? Yes. The complete updated resource.
  • Idempotent? Yes. Calling it 10 times gives the same result as once.
PUT Example 
PUT /api/bookings/1001
Content-Type: application/json

{"bookingId": 1001, "status": "IN_TRANSIT", "currentLocation": "Delhi Hub"}

DELETE — Remove

  • Purpose: Delete a resource. "Remove this."
  • Has body? Usually no.
  • Idempotent? Yes. Deleting something twice still results in it being deleted.
DELETE Example 
DELETE /api/bookings/1001      // Cancel booking #1001
CRUD Mapping

The four methods map directly to database operations:

  • Create → POST
  • Read → GET
  • Update → PUT
  • Delete → DELETE
Section 07

HTTP Status Codes

Every response includes a 3-digit status code that tells the client what happened. The first digit tells you the category.

1xx — Informational

Rarely seen directly. The server is saying "I got your request, still working on it."

CodeNameMeaning
100ContinueServer received headers, client should send the body
101Switching ProtocolsServer is switching to WebSocket

2xx — Success

Your request worked.

CodeNameWhen It Happens
200OKStandard success. GET returned data. PUT updated successfully.
201CreatedPOST created a new resource. "Your booking was created."
204No ContentSuccess but nothing to return. Common after DELETE.

3xx — Redirection

The resource moved. The browser should go somewhere else.

CodeNameWhen It Happens
301Moved PermanentlyThis URL has permanently moved to a new one. Bookmarks should update.
302Found (Temporary)Temporarily redirecting. After login, redirect to dashboard.
304Not ModifiedThe cached version is still valid. Don't download again.

4xx — Client Error

YOU (the client) made a mistake.

CodeNameWhen It Happens
400Bad RequestYour request is malformed. Missing required fields, bad JSON, etc.
401UnauthorizedYou're not logged in. "Who are you? Show your ID."
403ForbiddenYou're logged in but don't have permission. "I know who you are, but you can't do this."
404Not FoundThe resource doesn't exist. Wrong URL or deleted resource.
401 vs 403 — Common Confusion

401 Unauthorized = "I don't know who you are" (not authenticated). Fix: log in.

403 Forbidden = "I know who you are, but you can't access this" (not authorized). Fix: get the right permissions. In Nimbus, a customer trying to access the officer dashboard gets 403.

5xx — Server Error

THE SERVER made a mistake. Not your fault.

CodeNameWhen It Happens
500Internal Server ErrorSomething broke on the server. A bug in the code, unhandled exception.
502Bad GatewayServer A asked Server B for data, and Server B failed.
503Service UnavailableServer is overloaded or under maintenance. "Try again later."
In Nimbus

When a user enters wrong credentials, Nimbus returns 401. When a customer tries to access /officer/dashboard, it returns 403. When booking ID doesn't exist, it returns 404. When the database is down, it returns 500.

Section 08

HTTP Headers

Headers are metadata — extra information about the request or response. They're key-value pairs sent alongside the actual data.

HeaderDirectionPurposeExample
Content-Type Both Format of the body application/json, text/html, application/x-www-form-urlencoded
Accept Request What formats the client accepts text/html, application/json
Authorization Request Credentials for authentication Bearer eyJhbGciOi... (JWT token)
Cookie Request Send stored cookies to server JSESSIONID=abc123; theme=dark
Set-Cookie Response Tell browser to store a cookie JSESSIONID=abc123; Path=/; HttpOnly
Content-Length Both Size of body in bytes 256
Cache-Control Response How long the browser can cache this max-age=3600 (1 hour)
Content-Type Matters

The most important header for web development. It tells the receiver how to parse the body:

  • application/json — JSON data (APIs)
  • text/html — HTML page (browsers)
  • application/x-www-form-urlencoded — HTML form submission (key=value&key=value)
  • multipart/form-data — File uploads
Section 09

GET vs POST — Comparison

The two most common methods. This is a question that comes up in every interview.

Feature GET POST
Purpose Read / Retrieve data Create / Submit data
Data location In the URL (query string) In the request body
Data visible? Yes — in URL bar, browser history, server logs No — hidden in body (but NOT encrypted unless HTTPS)
Data limit ~2,048 characters (URL length limit) No practical limit
Cacheable? Yes No
Bookmarkable? Yes No
Safe? Yes (should not modify data) No (modifies server state)
Idempotent? Yes (10 GETs = same as 1 GET) No (10 POSTs may create 10 records)
Back button Harmless (just re-fetches) Browser warns "Resubmit form data?"
Use for passwords? NEVER — visible in URL Yes (with HTTPS)
Common Mistake

Never send sensitive data via GET. The URL /login?username=john&password=secret123 exposes the password in browser history, server access logs, and proxy logs. Always use POST for login forms.

Full journey: URL → IP address → connection → request → response → render

Step-by-Step Breakdown

  1. DNS Lookup: Your browser asks a DNS server: "What's the IP address of nimbus.com?" DNS responds: "93.184.216.34". Domain names are for humans; computers use IP addresses.
  2. TCP Connection: Your browser opens a connection to the server at that IP address on port 80 (HTTP) or 443 (HTTPS). Uses a "3-way handshake": SYN → SYN-ACK → ACK. Like calling someone: ring → "hello?" → "hi, it's me."
  3. TLS Handshake (HTTPS only): Browser and server agree on encryption. They exchange certificates and encryption keys. Everything after this is encrypted.
  4. HTTP Request: Browser sends GET / HTTP/1.1 with headers (Host, User-Agent, Accept, Cookies).
  5. Server Processes: The web server (Tomcat) receives the request, finds the right Servlet, runs your Java code, queries the database, builds the HTML response.
  6. HTTP Response: Server sends back HTTP/1.1 200 OK with headers and the HTML body.
  7. Browser Renders: Browser parses HTML into a DOM tree, applies CSS styling, executes JavaScript, and paints pixels on screen. If the HTML references CSS/JS/images, the browser sends additional GET requests for each one.
Multiple Requests Per Page

Loading a single web page typically sends 10-100+ HTTP requests. One for the HTML, one for each CSS file, one for each JS file, one for each image. Chrome DevTools (F12 → Network tab) shows you every single request.

Section 11

Nimbus: GET vs POST Endpoints

How HTTP methods map to real Nimbus functionality.

In Nimbus

Every user action in Nimbus maps to either a GET or POST request. Here's how:

ActionMethodURLWhy
View login page GET /login Reading a page, no data submitted
Submit login form POST /login Sending credentials (username + password)
View dashboard GET /dashboard Reading data to display
Create a booking POST /bookings Submitting form data to create a record
View booking details GET /bookings/1001 Reading a specific booking
Update booking status POST /bookings/1001/status Submitting new status (officer action)
Track a parcel GET /track?id=NIM-001 Reading tracking information
Register new user POST /register Submitting registration form

Pattern: Viewing pages and fetching data = GET. Submitting forms and creating/updating data = POST.
