Section 01

What is Polymorphism?

The most powerful concept in Object-Oriented Programming. One name, many behaviors. This is what separates amateur code from professional, flexible, extensible software.

The Word Itself

Polymorphism comes from two Greek words:

  • Poly = many
  • Morph = forms

So polymorphism literally means "many forms." In Java, it means that a single method name or a single variable type can take on different behaviors depending on the context.

That is the entire concept in one sentence. But understanding how it works, why it matters, and when to use it — that is what the rest of this page covers in extreme detail.

Real-World Analogy: The Word "Play"

Analogy

Think about the English word "play":

  • "Play music" — your hands move across a guitar, sound comes out
  • "Play a game" — you kick a football around a field
  • "Play a role" — you act as a character in a movie
  • "Play a video" — you press a button and a video starts

Same word. Four completely different actions. Your brain figures out which "play" is meant based on context — what comes after the word.

Java does the exact same thing with methods. Same method name, but the behavior changes based on which object calls it or what arguments you pass.

Another Analogy: A Remote Control

You have a universal remote with a "power" button. Press it while pointing at:

  • A TV — the TV turns on, screen lights up
  • An AC — the compressor starts, cold air flows
  • A speaker — music starts playing

Same button. Same action (press). But the behavior depends on which device receives the signal. In Java terms: same method call, but the actual code that runs depends on which object is behind the reference.

Why Polymorphism Matters

Without polymorphism, your code is rigid. Every time you add a new type, you must change existing code everywhere. With polymorphism, you write code once and it automatically works with new types you add later. This is the foundation of flexible, professional software.

Without Polymorphism

You write a separate method for each type:

  • trackForCustomer()
  • trackForOfficer()
  • trackForAdmin()

Adding a new role? Go find every place in the codebase that has if/else for roles, and add another case. Miss one? Bug.

With Polymorphism

You write ONE method call:

  • tracker.trackParcel()

Adding a new role? Create a new class that extends TrackingService. Zero changes to existing code. Zero risk of breaking anything.

In Nimbus

Nimbus has three user roles: Customer, Officer, and Support. Each tracks parcels differently. Instead of writing if (role == "customer") { ... } else if (role == "officer") { ... } everywhere, Nimbus uses polymorphism: TrackingService is the parent, and CustomerTrackingService and OfficerTrackingService each override trackParcel(). One method call, different behavior based on the object.

Section 02

Two Types of Polymorphism

Polymorphism in Java comes in exactly two flavors. They work differently, they are resolved at different times, and they serve different purposes. You must understand both deeply.

The Two Types at a Glance

Polymorphism "many forms"
↓
Compile-Time (Static Polymorphism)
&
Runtime (Dynamic Polymorphism)
↓
Method Overloading Same name, different params
&
Method Overriding Child redefines parent's method
The two branches of polymorphism in Java

Detailed Comparison

Aspect Compile-Time (Overloading) Runtime (Overriding)
Also called Static polymorphism, early binding Dynamic polymorphism, late binding
Mechanism Method overloading Method overriding
When is it resolved? At compile time (before program runs) At runtime (while program is running)
What decides which method? The arguments you pass (number, type, order) The actual object behind the variable
Requires inheritance? No — can be in the same class Yes — must have parent-child relationship
Method signature Same name, different parameters Same name, same parameters
Return type Can be different (but won't resolve alone) Must be same or covariant (subtype)
Example calculateCost(int) vs calculateCost(int, String) Parent trackParcel() vs child trackParcel()
Power level Convenient — makes API cleaner Powerful — enables extensible architecture

Why These Names Make Sense

Compile-Time (Static)

When you write calculateCost(10) vs calculateCost(10, "express"), the Java compiler can look at the arguments right now, at compile time, and decide which version to call. It does not need to wait until the program actually runs. The decision is "static" — fixed before execution.

Runtime (Dynamic)

When you write tracker.trackParcel(bookingId, userId), the compiler has no idea which version of trackParcel() to call. Why? Because tracker could be pointing to a CustomerTrackingService object or an OfficerTrackingService object — and that is only decided while the program is running (based on who logged in, for example). The JVM checks the actual object at runtime and calls the correct version. The decision is "dynamic" — made during execution.

Key Insight

Compile-time polymorphism is about giving the programmer convenience — multiple versions of a method for different inputs. Runtime polymorphism is about giving the program flexibility — the ability to behave differently based on which object is actually present. Runtime polymorphism is the far more powerful and important concept. It is what makes OOP truly powerful.

Section 03

Compile-Time Polymorphism — Method Overloading

Same method name, different parameter lists. The compiler decides which version to call based on the arguments you pass. It is like ordering food — you say "I want a coffee" but the specifics change: black coffee, coffee with milk, coffee with milk and sugar.

What is Method Overloading?

Method overloading means you have multiple methods with the same name in the same class, but each version takes a different set of parameters. Java uses the parameters to figure out which one you mean.

Think of it like a restaurant order:

  • orderCoffee() — plain black coffee
  • orderCoffee("milk") — coffee with milk
  • orderCoffee("milk", 2) — coffee with milk and 2 sugars

Same action (ordering coffee), but different levels of detail change the result.

The Rules of Overloading

For two methods to be valid overloads, their parameter lists must differ in at least one of these ways:

Can Differ In Example Valid?
Number of parameters add(int a) vs add(int a, int b) Yes
Type of parameters print(String s) vs print(int n) Yes
Order of parameter types display(String s, int n) vs display(int n, String s) Yes
Only return type int getValue() vs String getValue() NO! Compile error
Only parameter names add(int x) vs add(int y) NO! Same signature
Common Mistake

Changing ONLY the return type is NOT valid overloading. If you have int calculate(int x) and try to add double calculate(int x), Java will give you a compile error. Why? Because when you call calculate(5), Java has no way to know which one you want — you might not even use the return value! The compiler can only look at the arguments to decide, not at what you do with the result.

Example 1: PricingCalculator with Different Cost Methods

PricingCalculator.java 
public class PricingCalculator {

    // Version 1: Just weight
    public double calculateCost(int weightInGrams) {
        double baseRate = 50.0;  // Rs. 50 base
        return baseRate + (weightInGrams * 0.1);
    }

    // Version 2: Weight + service type
    public double calculateCost(int weightInGrams, String serviceType) {
        double base = calculateCost(weightInGrams);  // reuse version 1!
        if (serviceType.equals("express")) {
            return base * 1.5;  // 50% extra for express
        } else if (serviceType.equals("overnight")) {
            return base * 2.0;  // double for overnight
        }
        return base;
    }

    // Version 3: Weight + service type + packing
    public double calculateCost(int weightInGrams, String serviceType, String packingType) {
        double base = calculateCost(weightInGrams, serviceType);  // reuse version 2!
        if (packingType.equals("fragile")) {
            return base + 100.0;  // extra for fragile packing
        } else if (packingType.equals("insured")) {
            return base + 200.0;  // extra for insurance
        }
        return base;
    }
}
Line-by-line explanation
1
public double calculateCost(int weightInGrams)
Version 1 — takes only weight. The simplest form. Returns a cost based on base rate + weight factor.
2
public double calculateCost(int weightInGrams, String serviceType)
Version 2 — takes weight AND service type. Calls version 1 internally (reuse!), then applies a multiplier for express/overnight.
3
public double calculateCost(int weightInGrams, String serviceType, String packingType)
Version 3 — takes weight, service type, AND packing. Calls version 2 internally, then adds packing surcharge. Each version builds on the previous one — elegant reuse.

Now when you use this class:

Usage 
PricingCalculator calc = new PricingCalculator();

double cost1 = calc.calculateCost(500);                              // Version 1
double cost2 = calc.calculateCost(500, "express");                   // Version 2
double cost3 = calc.calculateCost(500, "express", "fragile");        // Version 3

// Java looks at the ARGUMENTS and picks the right version at COMPILE TIME
In Nimbus

This is exactly how PricingCalculator.calculateServiceCost() works in the Nimbus codebase. Customers can request a basic delivery (just weight), an express delivery (weight + service type), or a fragile insured delivery (weight + service type + packing). One method name, three versions, each handling different levels of detail. The caller uses whichever version matches their needs.

Logger.java 
public class Logger {

    public void log(String message) {

    }
}

Example 3: Constructor Overloading

Constructors can be overloaded too — and this is very common. It lets you create objects with different levels of information.

User.java 
public class User {
    private String name;
    private String email;
    private String phone;
    private String role;

    // Constructor 1: Minimal — just a name
    public User(String name) {
        this.name = name;
        this.email = "not provided";
        this.phone = "not provided";
        this.role = "customer";  // default role
    }

    // Constructor 2: Name + email
    public User(String name, String email) {
        this.name = name;
        this.email = email;
        this.phone = "not provided";
        this.role = "customer";
    }

    // Constructor 3: Full details
    public User(String name, String email, String phone, String role) {
        this.name = name;
        this.email = email;
        this.phone = phone;
        this.role = role;
    }
}
Line-by-line explanation
1
public User(String name)
Minimal constructor — when you only know the name. Sets defaults for everything else. Useful for quick testing or when a customer signs up with just a name.
2
public User(String name, String email)
Partial constructor — name and email. Phone and role get defaults. Common in registration forms where phone is optional.
3
public User(String name, String email, String phone, String role)
Full constructor — everything provided. Used by admin when creating any type of user (customer, officer, support).

Usage:

Usage

User u2 = new User("Priya", "priya@email.com");                 // Constructor 2
User u3 = new User("Admin", "admin@nimbus.com", "999", "officer"); // Constructor 3

Example 4: Order of Parameter Types Matters

OrderMatters.java 
public class Display {

    // Version A: String first, then int
    public void show(String label, int value) {
        System.out.println(label + ": " + value);
    }

    // Version B: int first, then String
    public void show(int value, String label) {
        System.out.println(value + " (" + label + ")");
    }
}

// Usage:
Display d = new Display();
d.show("Weight", 500);   // calls Version A → "Weight: 500"
d.show(500, "Weight");   // calls Version B → "500 (Weight)"

Same number of parameters, same types — but different order. Java sees (String, int) and (int, String) as different signatures, so this is valid overloading.

Example 5: Java's Own Overloading — System.out.println()

You have been using method overloading since day one without realizing it. System.out.println() has 10 overloaded versions built into Java:

Java's println overloads 
// All of these call DIFFERENT versions of println:
System.out.println();                // println()         — empty line
System.out.println("Hello");         // println(String)
System.out.println(42);              // println(int)
System.out.println(3.14);            // println(double)
System.out.println(true);            // println(boolean)
System.out.println('A');             // println(char)
System.out.println(new User("D"));   // println(Object) — calls toString()

When you write System.out.println(42), Java sees the argument is an int and picks println(int). When you write System.out.println("Hello"), it picks println(String). This is compile-time polymorphism happening right under your nose.

How the Compiler Decides: The Resolution Process

When Java encounters a method call like calc.calculateCost(500, "express"), the compiler follows these steps:

  1. Look at the method name: calculateCost
  2. Look at the arguments: (int, String)
  3. Find all methods named calculateCost in the class
  4. Filter to ones whose parameter types match (int, String)
  5. If exactly one match found: that is the one. If zero: compile error. If multiple (ambiguous): compile error.

This all happens at compile time — before the program ever runs. That is why it is called static/compile-time polymorphism. The decision is baked into the compiled bytecode.

Tricky Case: Type Widening

What happens if you call log(5) but there is no log(int), only log(double)? Java will automatically widen the int to a double and call log(double). This is called implicit type widening. It can cause subtle bugs when you have multiple overloads because Java prefers the closest match: int → long → float → double.

Section 04

Runtime Polymorphism — Method Overriding

This is the BIG one. Method overriding is what makes Object-Oriented Programming truly powerful. A child class provides its own implementation of a method that already exists in the parent class. Same name, same parameters, different behavior.

What is Method Overriding?

When a parent class has a method, and a child class defines a method with the exact same name and exact same parameters, the child's version replaces (overrides) the parent's version for objects of the child type.

Analogy

Your father taught you how to make chai. He uses ginger and elaichi. You learned his recipe, but you make it differently — you add tulsi leaves and less sugar. Same recipe name ("make chai"), same inputs (water, milk, tea leaves), but your implementation is different. You overrode your father's version.

The Rules of Overriding

Rule Details
Method name Must be EXACTLY the same
Parameters Must be EXACTLY the same (number, types, order)
Return type Must be the same, OR a subtype (covariant return)
Access modifier Can be SAME or MORE visible. Cannot reduce visibility. (If parent is protected, child can be protected or public, but NOT private)
Requires inheritance Yes — the method must exist in a parent class or interface
static methods Cannot be overridden (they are hidden, not overridden — different concept)
final methods Cannot be overridden (that is the whole point of final)
private methods Cannot be overridden (child class cannot even see them)
Constructors Cannot be overridden (constructors are not inherited)

The @Override Annotation

Java provides the @Override annotation. You put it right above the method you intend to override:

@Override usage 
@Override
public void trackParcel(String bookingId, String userId) {
    // child's own implementation
}

What does @Override actually do? It is a safety check. It tells the compiler: "I intend this to be an override. If the parent class does NOT have a method with this exact signature, give me a compile error." Without it, if you accidentally misspell the method name (e.g., trackParcle instead of trackParcel), Java would silently create a NEW method instead of overriding. You would never know until a bug shows up in production.

Always Use @Override

You should put @Override on every single method that you intend to override. It is not technically required, but skipping it is a recipe for bugs. Every professional Java codebase enforces this as a rule. It costs nothing to write and can save hours of debugging.

Full Nimbus Example: TrackingService Hierarchy

TrackingService.java (Parent) 
public class TrackingService {

    public String trackParcel(String bookingId, String userId) {
        // Default: basic tracking info available to everyone
        return "Booking " + bookingId + " — Status: In Transit";
    }

    public void displayTrackingInfo(String bookingId, String userId) {
        String result = trackParcel(bookingId, userId);
        System.out.println("=== Tracking Result ===");
        System.out.println(result);
    }
}
CustomerTrackingService.java (Child) 
public class CustomerTrackingService extends TrackingService {

    @Override
    public String trackParcel(String bookingId, String userId) {
        // Customer can only see THEIR OWN parcels
        // First verify this booking belongs to this user
        if (!belongsToUser(bookingId, userId)) {
            return "Access denied: This booking does not belong to you.";
        }
        return "Booking " + bookingId + " — Status: In Transit | ETA: 2 days";
    }

    private boolean belongsToUser(String bookingId, String userId) {
        // Database check: does this booking belong to this user?
        // Simplified for learning
        return true;
    }
}
OfficerTrackingService.java (Child) 
public class OfficerTrackingService extends TrackingService {

    @Override
    public String trackParcel(String bookingId, String userId) {
        // Officer can see ANY parcel with FULL details
        return "Booking " + bookingId
            + " — Status: In Transit"
            + " | Sender: User#" + userId
            + " | Current Location: Mumbai Hub"
            + " | Assigned Officer: OFC-2045"
            + " | Weight: 2.5kg"
            + " | Route: Mumbai → Delhi → Lucknow";
    }
}
How overriding works here
1
TrackingService (parent)
Has a basic trackParcel() that returns minimal info. This is the default behavior if no child overrides it.
2
CustomerTrackingService (child)
Overrides trackParcel() to add ownership verification. A customer should NOT see someone else's parcel. Returns limited info + ETA.
3
OfficerTrackingService (child)
Overrides trackParcel() to show FULL details. An officer needs to see location, route, weight, assigned officer — everything.
4
Same method signature everywhere
All three versions have the SAME name (trackParcel) and SAME parameters (String, String). That is what makes it overriding, not overloading.

The Magic: Which Method Actually Runs?

This is the most important concept in all of runtime polymorphism. Read this carefully:

The Magic of Runtime Binding 
// The variable TYPE is TrackingService (parent)
// But the ACTUAL OBJECT is CustomerTrackingService (child)
TrackingService tracker = new CustomerTrackingService();

// Which trackParcel() runs?
// Answer: CustomerTrackingService's version!
tracker.trackParcel("BK001", "USR42");
// → "Access denied..." or "Booking BK001 — Status: In Transit | ETA: 2 days"

// Now point the SAME variable at a DIFFERENT object
tracker = new OfficerTrackingService();

// Which trackParcel() runs now?
// Answer: OfficerTrackingService's version!
tracker.trackParcel("BK001", "USR42");
// → "Booking BK001 — Status: In Transit | Sender: ... | Current Location: ..."
The Golden Rule

The method that runs depends on the ACTUAL OBJECT (right side of =), NOT the variable type (left side of =).

In TrackingService tracker = new CustomerTrackingService():

  • Variable type (TrackingService) determines what methods you can call
  • Actual object (CustomerTrackingService) determines what code actually runs

This distinction is EVERYTHING in runtime polymorphism. The variable is like a label on a box. The object is what is actually inside the box. When you call a method, Java opens the box, looks at the real object, and calls that object's version.

How the JVM Decides at Runtime

tracker.trackParcel(...) Method call in your code
↓
JVM checks: What object is "tracker" pointing to? Looks at the actual object in heap memory
↓
Found: CustomerTrackingService object The real object behind the variable
↓
Runs CustomerTrackingService.trackParcel() The overridden version in the child class
This decision happens every time the method is called, at runtime

Calling the Parent's Version: super

Sometimes you do not want to completely replace the parent's behavior — you want to extend it. Use super.methodName() to call the parent's version from within the child's override:

Using super in an override 
public class CustomerTrackingService extends TrackingService {

    @Override
    public String trackParcel(String bookingId, String userId) {
        // First, get the basic info from parent
        String basicInfo = super.trackParcel(bookingId, userId);

        // Then add customer-specific details
        return basicInfo + " | ETA: 2 days | Contact support: 1800-NIMBUS";
    }
}

super.trackParcel() explicitly calls the parent class's version. The child adds extra info on top. This is common when the parent has useful logic you want to keep, and the child just needs to add more.

Section 05

Upcasting and Downcasting

Polymorphism relies on treating a child object as its parent type. This is called upcasting. Sometimes you need to go the other way — downcasting. Understanding both is essential for writing polymorphic code.

Upcasting: Child Treated as Parent

Upcasting means storing a child object in a parent-type variable. It is automatic, safe, and the foundation of runtime polymorphism.

Upcasting 
// Upcasting — happens AUTOMATICALLY (implicit)
TrackingService tracker = new CustomerTrackingService();
//   ^^ parent type           ^^ child object

// This works because: a CustomerTrackingService IS-A TrackingService
// Just like: a Dog IS-AN Animal, a Car IS-A Vehicle

Why does upcasting work? Because of the IS-A relationship from inheritance. A CustomerTrackingService IS a TrackingService. It has everything a TrackingService has (and possibly more). So storing it in a TrackingService variable is perfectly safe — you are just choosing to see it as the simpler parent type.

What you can access after upcasting: Only the methods defined in the parent type. Even though the actual object is a CustomerTrackingService and might have extra methods like belongsToUser(), you cannot call them through a TrackingService variable. The variable type determines what you can call; the object determines what runs.

Analogy

Think of upcasting like describing yourself on a form. You are a "Java Developer" (specific), but on the form you write "Software Engineer" (general). You have not lost any of your Java skills — the form just sees you as the broader category. If someone asks "Software Engineer, can you write code?" you say yes. Your specific Java skills are still there — the label just does not show them.

Downcasting: Parent Variable Back to Child

Downcasting means converting a parent-type variable back to the child type. It is manual (you must explicitly cast), and it is risky.

Downcasting 
TrackingService tracker = new CustomerTrackingService();  // upcasted

// To access CustomerTrackingService-specific methods, you must downcast:
CustomerTrackingService cst = (CustomerTrackingService) tracker;  // explicit cast
// Now you can call customer-specific methods:
// cst.belongsToUser(...);

When Downcasting Goes Wrong: ClassCastException

ClassCastException — runtime crash 
TrackingService tracker = new OfficerTrackingService();  // actual object is Officer

// DANGER: trying to cast an Officer object to Customer type
CustomerTrackingService cst = (CustomerTrackingService) tracker;
// ^^^ RUNTIME CRASH! ClassCastException!
// OfficerTrackingService is NOT a CustomerTrackingService

The compiler will not catch this — it only checks if the cast is theoretically possible (both are subtypes of TrackingService, so maybe). But at runtime, the JVM sees that the actual object is an OfficerTrackingService, which is NOT a CustomerTrackingService, and throws a ClassCastException.

The Safe Way: instanceof

Always check with instanceof before downcasting:

Safe downcasting with instanceof 
TrackingService tracker = getTrackerForRole(role);  // could be any subtype

if (tracker instanceof CustomerTrackingService) {
    CustomerTrackingService cst = (CustomerTrackingService) tracker;
    // Safe! We verified the type first
    cst.showCustomerDashboard();
} else if (tracker instanceof OfficerTrackingService) {
    OfficerTrackingService ost = (OfficerTrackingService) tracker;
    ost.showOfficerPanel();
}
Line-by-line explanation
1
tracker instanceof CustomerTrackingService
Returns true if the actual object behind tracker is a CustomerTrackingService (or a subclass of it). Returns false otherwise. Safe, never crashes.
2
(CustomerTrackingService) tracker
Now we KNOW it is a CustomerTrackingService (because instanceof returned true), so the cast is guaranteed safe. We can call customer-specific methods.

Nimbus Example: Choosing Tracker by Role

RoleBasedTracker.java 
public class ParcelController {

    public TrackingService getTrackerForRole(String role) {
        // Upcasting happens automatically in each return
        if (role.equals("customer")) {
            return new CustomerTrackingService();   // upcast to TrackingService
        } else if (role.equals("officer")) {
            return new OfficerTrackingService();    // upcast to TrackingService
        } else {
            return new TrackingService();           // base version
        }
    }

    public void handleTrackRequest(String role, String bookingId, String userId) {
        TrackingService tracker = getTrackerForRole(role);

        // This single line calls DIFFERENT code depending on the role!
        String result = tracker.trackParcel(bookingId, userId);
        System.out.println(result);
    }
}
In Nimbus

This pattern is the heart of Nimbus's role-based system. getTrackerForRole() uses upcasting to return any tracker type as a TrackingService. The handleTrackRequest() method does not care which type it gets — it just calls trackParcel() and the correct version runs automatically. No if/else in the tracking logic. Clean, extensible, professional.

Upcasting vs Downcasting Summary

Aspect Upcasting Downcasting
Direction Child → Parent Parent → Child
Implicit/Explicit Implicit (automatic) Explicit (manual cast required)
Safe? Always safe Can crash with ClassCastException
Use case Treat all children uniformly Access child-specific methods
How common? Very common — used constantly Rare — avoid if possible
Best practice Use freely Always use instanceof first
Section 06

Polymorphism with Abstract Classes

Abstract classes take overriding to the next level: instead of providing a default implementation that children can override, you provide no implementation at all and force children to override. The parent says "you MUST implement this method" and refuses to let you forget.

What is an Abstract Method?

An abstract method is a method that is declared but has no body. No curly braces, no code inside — just the signature followed by a semicolon.

Abstract method syntax 
// Regular method — has a body
public String trackParcel(String bookingId, String userId) {
    return "Basic tracking info";
}

// Abstract method — NO body, just declaration
public abstract String trackParcel(String bookingId, String userId);

What is an Abstract Class?

If a class has even one abstract method, the class itself must be declared abstract. An abstract class:

  • Cannot be instantiated. You cannot write new TrackingService() if it is abstract.
  • Can have both abstract and regular methods. Some methods have implementations (shared code), some do not (forced overrides).
  • Can have fields, constructors, everything else. It is still a class. Just an incomplete one.
  • Children MUST override all abstract methods, or the child must also be declared abstract.

Nimbus: Making TrackingService Abstract

TrackingService.java (abstract version) 
public abstract class TrackingService {

    // Abstract — every child MUST implement this
    public abstract String trackParcel(String bookingId, String userId);

    // Concrete — shared by all children, no need to override
    public void displayTrackingInfo(String bookingId, String userId) {
        String result = trackParcel(bookingId, userId);  // calls child's version!
        System.out.println("=== Tracking Result ===");
        System.out.println(result);
        System.out.println("=======================");
    }

    // Concrete — shared utility method
    public String formatBookingId(String rawId) {
        return "BK-" + rawId.toUpperCase();
    }
}
What each part does
1
public abstract class TrackingService
The abstract keyword on the class means: you cannot do new TrackingService(). It is incomplete — it is a blueprint that children must finish.
2
public abstract String trackParcel(...);
Abstract method — no body. Every non-abstract child MUST provide its own implementation. The parent is saying: "I know you need this method, but I do not know what it should do for each role. You figure it out."
3
public void displayTrackingInfo(...)
Concrete method — has a body, is shared by all children. Notice it calls trackParcel() which is abstract! At runtime, the child's version of trackParcel() will run. This is called the Template Method pattern.
Children must implement abstract methods 
public class CustomerTrackingService extends TrackingService {

    @Override
    public String trackParcel(String bookingId, String userId) {
        // MUST implement — compiler forces you
        return "Booking " + bookingId + " — Customer view | ETA: 2 days";
    }
    // displayTrackingInfo() and formatBookingId() are inherited — no need to override
}

public class OfficerTrackingService extends TrackingService {

    @Override
    public String trackParcel(String bookingId, String userId) {
        // MUST implement — compiler forces you
        return "Booking " + bookingId + " — Officer view | Full route + details";
    }
}

// What happens if you forget to implement trackParcel()?
// public class BrokenService extends TrackingService { }
// ^^^ COMPILE ERROR: "BrokenService must implement abstract method trackParcel()"
Why Use Abstract?

Without abstract, a developer could create a CustomerTrackingService and forget to override trackParcel(). The parent's default version would silently run, returning generic info. The bug would not show up until a customer complains "Why can I see everyone's parcels?" With abstract, the compiler catches the mistake immediately. You cannot even compile until you provide an implementation. That is the safety net.

Another Nimbus Example: BaseHistoryService

BaseHistoryService.java 
public abstract class BaseHistoryService {

    // Abstract — each role sees different bookings
    public abstract List<Booking> getMyBookings(String userId);

    // Concrete — shared display logic
    public void displayHistory(String userId) {
        List<Booking> bookings = getMyBookings(userId);  // calls child's version
        if (bookings.isEmpty()) {
            System.out.println("No bookings found.");
            return;

        System.out.println("Found " + bookings.size() + " bookings:");

            System.out.println("  - " + b.toString());
        }

public class CustomerHistoryService extends BaseHistoryService {

    @Override

        // Customer sees ONLY their own bookings

In Nimbus

BaseHistoryService has shared display logic (displayHistory()) that all roles use. But which bookings to fetch is role-specific, so getMyBookings() is abstract. CustomerHistoryService overrides it to only return that customer's bookings. If an OfficerHistoryService is added later, it would return all bookings in their assigned area. The display code never changes.

Section 07

Polymorphism with Interfaces

An interface is a pure contract — it says "any class that claims to be this type MUST have these methods." No implementation, no fields (mostly), just method signatures. Interfaces enable the most flexible form of polymorphism.

What is an Interface?

An interface defines what a class must do, without specifying how. Every method in an interface is implicitly public and abstract (you do not need to write those keywords, though you can).

Interface syntax 
public interface Trackable {
    String getTrackingStatus();
    String getLocation();
    String getEstimatedDelivery();
}

public interface Printable {
    void printDetails();
    String toFormattedString();
}

Implementing an Interface

A class uses the implements keyword (not extends) to adopt an interface. It must provide implementations for every method in the interface.

Booking implementing Trackable and Printable 
public class Booking implements Trackable, Printable {

    private String bookingId;
    private String status;
    private String currentLocation;
    private String eta;

    public Booking(String bookingId, String status, String location, String eta) {
        this.bookingId = bookingId;
        this.status = status;
        this.currentLocation = location;
        this.eta = eta;
    }

    // ====== Trackable methods ======
    @Override
    public String getTrackingStatus() {
        return status;
    }

    @Override
    public String getLocation() {
        return currentLocation;
    }

    @Override
    public String getEstimatedDelivery() {
        return eta;
    }

    // ====== Printable methods ======
    @Override
    public void printDetails() {
        System.out.println(toFormattedString());
    }

    @Override
    public String toFormattedString() {
        return String.format("[%s] Status: %s | Location: %s | ETA: %s",
            bookingId, status, currentLocation, eta);
    }
}
Key concepts
1
implements Trackable, Printable
A class can implement MULTIPLE interfaces (separated by commas). This is unlike extends where you can only extend ONE class. This is one of the biggest advantages of interfaces.
2
@Override on every method
Every interface method implementation should have @Override. The compiler will error if the method signature does not match the interface.
3
Must implement ALL methods from BOTH interfaces
Booking must implement all 3 Trackable methods AND both Printable methods. Miss one? Compile error.

The Key Difference: extends vs implements

extends (Inheritance)
  • Used with classes
  • Can extend only ONE class
  • Inherits implementation (actual code)
  • IS-A relationship with shared behavior

class Dog extends Animal

implements (Interface)
  • Used with interfaces
  • Can implement MULTIPLE interfaces
  • No implementation inherited (must write your own)
  • CAN-DO relationship (capability contract)

class Dog implements Trainable, Feedable

Polymorphism Through Interfaces

Just like with classes, you can use an interface type as the variable type:

Interface polymorphism 
// Different classes can implement the same interface
Trackable item1 = new Booking("BK001", "In Transit", "Mumbai", "2 days");
Trackable item2 = new Vehicle("VH100", "En Route", "Highway NH48", "4 hours");

// Both can be treated as Trackable — same method calls, different behavior
System.out.println(item1.getTrackingStatus());  // "In Transit"
System.out.println(item2.getTrackingStatus());  // "En Route"

// You can put them in a list and process uniformly
List<Trackable> trackableItems = new ArrayList<>();
trackableItems.add(item1);
trackableItems.add(item2);

for (Trackable item : trackableItems) {
    System.out.println(item.getLocation());  // each one runs its own version
}

You Can Use Both: extends + implements

Combining class inheritance with interfaces 
// A class can extend ONE class AND implement MULTIPLE interfaces
public class CustomerTrackingService extends TrackingService
    implements Trackable, Printable {

    @Override
    public String trackParcel(String bookingId, String userId) { ... }

    @Override
    public String getTrackingStatus() { ... }

    @Override
    public String getLocation() { ... }

    @Override
    public String getEstimatedDelivery() { ... }

    @Override
    public void printDetails() { ... }

    @Override
    public String toFormattedString() { ... }
}

Abstract Class vs Interface — When to Use Which

This is one of the most frequently asked interview questions and one of the most confusing topics. Here is the complete breakdown with 10+ differences, decision rules, and Nimbus examples.

Complete Comparison Table

Nimbus Examples

Abstract Class: BaseHistoryService

Why abstract class?

  • Has shared code: displayHistory() is the same for all roles
  • Has shared state: could have a dateFormat field
  • Children are closely related: all are "history services"

Abstract method: getMyBookings() — each role fetches different bookings.

Interface: Trackable

Why interface?

  • No shared code: each implementor has completely different tracking logic
  • Unrelated classes: Booking, Vehicle, and Package can all be Trackable
  • Just a contract: "if you are Trackable, you must provide these methods"

Pure capability: any class can declare "I am Trackable."

Java 8+ Default Methods: Blurring the Line

Since Java 8, interfaces can have default methods — methods with actual implementations. This blurs the distinction between interfaces and abstract classes.

Default method in interface 
public interface Trackable {
    String getTrackingStatus();   // abstract — must implement
    String getLocation();         // abstract — must implement

    // Default — has implementation! Implementors get this for free.
    default String getTrackingSummary() {
        return "Status: " + getTrackingStatus() + " | Location: " + getLocation();
    }
}

// Booking does NOT need to implement getTrackingSummary() — it's inherited
// But it CAN override it if it wants a custom version
So When Do You Use What Now?

Even with default methods, the rule of thumb remains: use abstract classes when you have shared state (fields) and a strong IS-A hierarchy. Use interfaces when you want to define capabilities that unrelated classes can adopt, or when a class needs multiple contracts. Default methods are mainly for backward compatibility — adding a method to an existing interface without breaking all implementors.

The Power — Real Polymorphism in Action

Everything you have learned comes together here. This is where you see WHY polymorphism matters and how it makes your code elegant, extensible, and professional.

The Complete Nimbus Polymorphism Example

NimbusApp.java — Polymorphism in action 
public class NimbusApp {

    public static void main(String[] args) {
        String role = "customer";       // could come from login
        String bookingId = "BK-2045";
        String userId = "USR-007";

        // POLYMORPHISM: one variable, behavior decided at runtime
        TrackingService tracker;

        if (role.equals("customer")) {
            tracker = new CustomerTrackingService();
        } else if (role.equals("officer")) {
            tracker = new OfficerTrackingService();
        } else {
            tracker = new SupportTrackingService();
        }

        // THIS is the power — SAME method call, DIFFERENT behavior
        tracker.trackParcel(bookingId, userId);
        // If customer logged in → shows limited info + ownership check
        // If officer logged in  → shows full route + all details
        // If support logged in  → shows customer info + complaint history
    }
}
Why this is powerful
1
TrackingService tracker;
Variable type is the PARENT. This means it can hold ANY child type. The variable does not care which specific tracking service it holds.
2
if (role.equals(...)) tracker = new ___TrackingService();
The decision of WHICH object to create happens at runtime based on the logged-in user's role. This is the only place that knows about specific types.
3
tracker.trackParcel(bookingId, userId);
ONE line of code. But it runs completely different logic depending on the role. No if/else in the tracking logic itself. The polymorphic dispatch handles everything.

The Real Power: Adding a New Role

Imagine the Nimbus product manager says: "We need an Admin role that can see all parcels across all regions with analytics." Without polymorphism, you would have to:

Without Polymorphism
  1. Find every if/else that checks for roles
  2. Add else if (role == "admin") everywhere
  3. Hope you did not miss any
  4. Test all existing roles again because you touched their code
  5. Risk breaking customer/officer tracking

Changes needed: Dozens of files

With Polymorphism
  1. Create AdminTrackingService extends TrackingService
  2. Override trackParcel() with admin logic
  3. Add one line in the factory/if-else that creates trackers
  4. Done. Customer and officer code untouched.

Changes needed: 1 new file + 1 line in factory

// Before:

else if (role.equals("officer")) tracker = new OfficerTrackingService();

// After — just add one else-if:
if (role.equals("customer")) tracker = new CustomerTrackingService();

The Open/Closed Principle

Design Principle

What you just saw is the Open/Closed Principle — one of the five SOLID principles of software design. It states:

"Software entities should be open for extension (you can add new behavior) but closed for modification (you do not change existing working code)."

Polymorphism makes this possible. New roles, new payment types, new shipping methods — all added by creating new classes that extend/implement existing ones. The rest of the system does not even know they exist until they are plugged in.

Overriding equals()

By default, equals() checks if two variables point to the same object in memory (same as ==). But usually, you want to check if two objects have the same field values.

Default equals() problem 
User u1 = new User("Darshan", "d@email.com", "999", "customer");
User u2 = new User("Darshan", "d@email.com", "999", "customer");

System.out.println(u1.equals(u2));  // false !!!
// ^^^ Two DIFFERENT objects in memory, even though fields are identical
User.java — equals() override 
@Override
public boolean equals(Object obj) {
    // 1. Same reference? Same object.
    if (this == obj) return true;

    // 2. Null or different class? Not equal.
    if (obj == null || getClass() != obj.getClass()) return false;

    // 3. Cast and compare fields
    User other = (User) obj;
    return this.name.equals(other.name)
        && this.email.equals(other.email)
        && this.role.equals(other.role);
}

// Now:
System.out.println(u1.equals(u2));  // true — same field values!
equals() step by step
1
if (this == obj) return true;
Optimization: if both variables point to the exact same object in memory, they are obviously equal. No need to check fields.
2
if (obj == null || getClass() != obj.getClass()) return false;
Safety checks: if the other object is null, or is a different class entirely (comparing a User to a Booking), return false immediately.
3
User other = (User) obj; return this.name.equals(other.name) ...
Downcast to User (safe because we checked the class), then compare the fields that define equality. You choose which fields matter.

The equals-hashCode Contract

Critical Rule

If you override equals(), you MUST also override hashCode(). This is a contract in Java. If two objects are equal (according to equals()), they must return the same hashCode(). If you break this contract, HashMap, HashSet, and other hash-based collections will not work correctly — you will get bugs like "I added the user to the set but contains() says it is not there."

hashCode() override 
@Override
public int hashCode() {
    return Objects.hash(name, email, role);
    // Uses the same fields as equals() — this is the rule
}
Section 11

Common Mistakes

These are the mistakes every beginner makes with polymorphism. Learn to recognize them so you can avoid hours of debugging.

Mistake 1: Confusing Overloading with Overriding

Overloading
  • Same class
  • Same name, DIFFERENT parameters
  • Compile-time decision
add(int a, int b)
add(int a, int b, int c)
Overriding
  • Parent-child classes
  • Same name, SAME parameters
  • Runtime decision
Parent: trackParcel(String, String)
Child:  trackParcel(String, String)

Mistake 2: Forgetting @Override (Silent Bug)

The silent bug 
public class CustomerTrackingService extends TrackingService {

    // TYPO: "Parcle" instead of "Parcel" — but no @Override, so no error!
    public String trackParcle(String bookingId, String userId) {
        return "Customer view...";
    }
    // This is NOT an override — it is a completely NEW method.
    // The parent's trackParcel() still runs when you call tracker.trackParcel()
    // The customer version NEVER executes. Good luck finding this bug.
}

// WITH @Override, the compiler catches it:
// @Override
// public String trackParcle(...)  → COMPILE ERROR: method does not override
//                                    anything from superclass

Mistake 3: Trying to Override static Methods

Static methods cannot be overridden 
class Parent {
    public static void greet() {
        System.out.println("Hello from Parent");
    }
}

class Child extends Parent {
    // This looks like an override, but it is NOT.
    // This is "method hiding" — a completely different concept.
    public static void greet() {
        System.out.println("Hello from Child");
    }
}

Parent ref = new Child();
ref.greet();  // Prints "Hello from Parent" !!!
// ^^^ Because static methods use the VARIABLE type, not the object type.
// Static binding, not dynamic binding. No polymorphism for static methods.

Why? Static methods belong to the class, not to an object. Polymorphism works with objects. Since static methods are resolved at compile time based on the declared variable type, there is no "which object is behind the variable?" question to ask at runtime.

Mistake 4: Trying to Override final Methods

final prevents overriding 
class Parent {
    public final void validate() {
        System.out.println("Validation logic — CANNOT be changed");
    }
}

class Child extends Parent {
    @Override
    public void validate() {  // COMPILE ERROR!
        // "validate() in Child cannot override validate() in Parent
        //  — overridden method is final"
    }
}

// Use "final" when you want to LOCK a method —
// no child should ever change this behavior.
// Common for security checks, validation, audit logging.

Mistake 5: Reducing Visibility in Override

Cannot make an override less visible 
class Parent {
    public void display() {   // public visibility
        System.out.println("Parent display");
    }
}

class Child extends Parent {
    @Override
    private void display() {   // COMPILE ERROR!
        // Cannot reduce visibility from public to private
    }

    @Override
    protected void display() { // ALSO COMPILE ERROR!
        // Cannot reduce visibility from public to protected
    }

    @Override
    public void display() {    // OK — same visibility
        System.out.println("Child display");
    }
}

Why this rule? Imagine code that has TrackingService tracker = new CustomerTrackingService() and calls tracker.trackParcel(). The code sees trackParcel() as public (because it is public in TrackingService). If the child made it private, this call would suddenly fail at runtime. Java prevents this by enforcing: you can keep the same visibility or make it more visible (protected → public is OK), but never less.

Mistake 6: Thinking Return Type Alone Creates an Overload

Return type does NOT create overload 
public class Calculator {
    public int calculate(int x) { return x * 2; }
    public double calculate(int x) { return x * 2.0; }  // COMPILE ERROR!
    // Both take (int) — return type alone is not enough to differentiate
}

Quick Reference: What Can and Cannot Be Overridden

Method Type Can Override? Reason
Regular instance method Yes This is what overriding is for
static method No (hidden, not overridden) Belongs to class, not object
final method No Explicitly locked by parent
private method No Not visible to child class
Constructor No Not inherited
abstract method Must override No implementation exists yet

This is overloading, not overriding. The abstract method is area() with no parameters. area(double r) is a new method. Circle still has not implemented the abstract area(), so this will not compile. Fix: public double area() { return 3.14 * radius * radius; }

Bug 2: describe() is final in Shape
describe() is marked final in the parent class Shape, which means it cannot be overridden. Compile error. Fix: either remove final from Shape, or remove the override from Circle.
3
Bug 3: @Override on a method that does not exist in parent
perimeter() does not exist in Shape. You cannot override something that was never defined in the parent. Also, private would reduce visibility. Fix: remove @Override and make it public — this is just a new method, not an override.

Exercise 6: Interface Implementation

Exercise

Create an interface called Notifiable with methods:

  • sendNotification(String message, String recipient)
  • getNotificationType() — returns a String like "EMAIL", "SMS", "PUSH"

Implement it in three classes: EmailNotifier, SMSNotifier, PushNotifier.

Write a method that takes a List<Notifiable> and sends the same message through ALL notification channels.

Exercise 7: Overloading Challenge

Exercise

Create a BookingSearch class with overloaded search() methods:

  • search(String bookingId) — search by exact booking ID
  • search(String city, String status) — search by city and status
  • search(String userId, int limit) — search by user, return limited results
  • search(double minWeight, double maxWeight) — search by weight range

Each method should print what it is searching by. Then call all four from main.

Exercise 8: Full Polymorphism Challenge

Exercise

This is the big one. Build a complete Notification System for Nimbus:

  1. Abstract class NotificationService with abstract send(String to, String message) and concrete log(String message)
  2. Interface Schedulable with scheduleAt(String time)
  3. Class EmailService extends NotificationService implements Schedulable
  4. Class SMSService extends NotificationService
  5. Class WhatsAppService extends NotificationService implements Schedulable
  6. A method that takes NotificationService[] and sends a booking confirmation to all
  7. A method that takes Schedulable[] and schedules a reminder at "9:00 AM"

This exercise uses: inheritance, abstract classes, interfaces, overriding, upcasting, and processing collections polymorphically — all in one program.
