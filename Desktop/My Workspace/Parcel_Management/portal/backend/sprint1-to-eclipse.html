Section 01

What is Eclipse?

Before you write a single line of Java, you need the right tool. Eclipse is that tool.

Eclipse is an IDE — an Integrated Development Environment. That is a fancy way of saying it is a code editor, compiler, runner, and debugger all packed into one window.

When you write JavaScript, you open a .js file in VS Code, save it, then open a terminal to run node app.js. You jump between the editor and the terminal constantly.

Eclipse removes that juggling. You write your Java code, hit Ctrl+F11, and Eclipse compiles it, runs it, and shows the output — all without leaving the window. If there is an error, it underlines the exact line in red before you even try to run it.

Think of it this way

VS Code is like a kitchen counter — you can cook on it, but you need to bring your own stove, pans, and utensils (extensions). Eclipse is like a fully equipped kitchen — the stove, oven, fridge, and dishwasher are already there. For Java, that means the compiler, build tools, server support, and debugger are all built in.

Section 02

VS Code vs Eclipse

You already know VS Code. Here is why Eclipse is the right choice for this project.

VS Code
  • A general-purpose text editor
  • Needs extensions for every language
  • Great for JavaScript, Python, HTML/CSS
  • No built-in Java compilation
  • No built-in Maven support
  • No built-in servlet/JSP server
Eclipse
  • A Java-specific IDE
  • Understands Java natively — zero setup
  • Built-in Maven integration
  • Built-in Java compilation + debugging
  • Built-in servlet/JSP support
  • Built-in Spring Boot runner
Why you can't use VS Code for this project

This project needs: Maven integration (dependency management), Java compilation (converting .java to .class), servlet/JSP support (web pages in Phase 3), and a Spring Boot runner (Phase 4). Eclipse has all of this built in. VS Code would need 5+ extensions, manual configuration, and would still not work as smoothly.

Section 03

Installing Eclipse

Getting Eclipse on your machine takes about 5 minutes. No installer — just download and extract.

  1. Go to eclipse.org/downloads/packages
  2. Find "Eclipse IDE for Enterprise Java and Web Developers" and click the download link for your OS (Windows, Mac, or Linux)
  3. Extract the downloaded zip file to any folder you like (e.g., C:\Eclipse or D:\Tools\Eclipse)
  4. Open the extracted folder and double-click eclipse.exe (Windows) or Eclipse.app (Mac)
  5. Eclipse asks for a "workspace" location. Pick any folder like C:\eclipse-workspace. This is just the folder where your projects will live.
  6. Check "Use this as the default and do not ask again" so it does not ask every time you open Eclipse
Download the RIGHT version

There are multiple versions on the downloads page. You must download "Eclipse IDE for Enterprise Java and Web Developers" — not "Eclipse IDE for Java Developers" (the basic one). The Enterprise version includes servlet, JSP, and Spring support that you will need in Phase 3 and 4. If you download the wrong one, you will have to reinstall later.

What is a "workspace"?

A workspace is just a regular folder on your computer. Eclipse stores all your projects inside it. Think of it like a desk — all your project folders sit on this desk. You can have multiple workspaces, but one is enough for now.

Section 04

Package Explorer

The left panel of Eclipse — your file browser for the project.

When Eclipse opens, you will see a panel on the left side called Package Explorer. This is like the file explorer in VS Code's sidebar, but smarter — it understands Java project structure.

What you will see

  • Your project name at the top level
  • Click the arrow (▶) next to any item to expand it and see what is inside
  • src/main/java — this is where your Java code lives
  • src/main/resources — config files and static assets
  • Maven Dependencies — libraries Maven downloaded for you
  • pom.xml — your project's dependency file (like package.json)

Key actions

  • Right-click anything to see options: New (create files/folders), Delete, Rename, Run As, etc.
  • Right-click → New → Class creates a new Java file
  • Right-click → New → Package creates a new package (folder)
  • Double-click a file to open it in the editor
Analogy

Package Explorer is like the "Files" sidebar in VS Code, except it groups things by Java conventions instead of raw folders. It hides the messy internal folders and shows you only what matters.

Section 05

Editor

The center panel — where you actually write your code.

Double-click any file in Package Explorer, and it opens here. If you open multiple files, they appear as tabs at the top of the editor — just like browser tabs or VS Code tabs.

What to look for

  • Red squiggly underlines = errors. Your code will not compile until you fix these. Hover over the red line to see what is wrong.
  • Yellow squiggly underlines = warnings. Usually safe to ignore, but worth understanding.
  • Line numbers on the left side — error messages reference these
  • The red circle with an X in the left margin = there is an error on that line
  • The yellow triangle in the left margin = there is a warning on that line
Eclipse catches errors BEFORE you run

This is a major difference from JavaScript. In JS, you find errors only when you run the code. Eclipse checks your Java code as you type and shows errors immediately. If you see a red underline, hover over it — Eclipse often suggests a fix. Click the suggestion and it fixes it for you.

Section 06

Console

The bottom panel — where your program's output appears.

When you run your Java program (Ctrl+F11), any output goes to the Console panel at the bottom of Eclipse.

  • System.out.println("Hello") output appears here in normal text
  • Error messages (stack traces) appear here in red text
  • If your program asks for input (using Scanner), you type into this console
Console not visible?

If you cannot see the Console panel, go to Window → Show View → Console. Eclipse lets you rearrange or close any panel, so sometimes it gets hidden.

In Nimbus

In Sprint 2, you will build a console-based menu app. All the menu options ("1. Register", "2. Book Parcel", etc.) will print to this console, and user input will be typed here.

Section 07

Problems Tab

A centralized list of every error and warning in your entire project.

The Problems tab sits at the bottom of Eclipse (next to the Console tab). While the editor shows errors for the file you are looking at, the Problems tab shows errors across ALL files in your project.

  • Red circle with X = Error. Your code will not compile until you fix this.
  • Yellow triangle = Warning. Your code will compile, but something might not be ideal.
  • Double-click any error in the Problems tab — Eclipse jumps to the exact file and line where the error is.
Common beginner errors you will see here

cannot be resolved to a type — you forgot to import a class. Fix: Ctrl+Shift+O.
Syntax error on token — you forgot a semicolon, brace, or parenthesis.
The method X is undefined — you misspelled a method name or called it on the wrong object.

Section 09

Create Your First Maven Project

Time to create the Nimbus backend project inside Eclipse. Follow each step exactly.

  1. In Eclipse, go to File → New → Maven Project
  2. Check "Create a simple project (skip archetype selection)" and click Next
  3. Fill in the form:
    • Group ID: com.nimbus
    • Artifact ID: nimbus-backend
    • Packaging: war
  4. Click Finish
What is an "archetype"?

An archetype is a project template. Maven has templates for different project types (web app, console app, library, etc.). We skip it because we are building our structure from scratch — it is easier to understand when you create every folder yourself.

What do Group ID and Artifact ID mean?

Group ID is like your organization name, written in reverse domain format. com.nimbus means "Nimbus project." If you worked at Google, it would be com.google.

Artifact ID is your project name. nimbus-backend tells anyone looking at the project that this is the backend part of Nimbus.

Packaging: war means Web Application Archive. This tells Maven that your project is a web application (needed for servlets and JSP in Phase 3).

What Maven just created

After clicking Finish, look in Package Explorer. Maven created this folder structure for you:

nimbus-backend/
  pom.xml              ← Project config (your dependencies go here)
  src/
    main/
      java/             ← Your Java code goes here
      resources/        ← Config files (may need to create)
      webapp/           ← Web files like JSP pages
    test/
      java/             ← Tests (skip for now)
The default Maven project structure
Section 10

What is Maven?

Maven is the tool that manages your project's libraries. Think of it as "npm for Java."

You already know this pattern from JavaScript:

JavaScript (npm)
  1. Run npm install express
  2. package.json lists your dependencies
  3. node_modules/ has the actual library files
Java (Maven)
  1. Add a dependency to pom.xml
  2. pom.xml lists your dependencies
  3. "Maven Dependencies" folder has the actual .jar files

That is literally it. You list what you need in pom.xml, Maven downloads it. When you need a new library (like a database driver, or Spring Boot), you add a few lines to pom.xml and press Alt+F5 (Maven Update). Maven handles the rest.

Analogy

Maven is like a personal shopper. You write a grocery list (pom.xml) with items like "MySQL driver version 8.0" and "Spring Boot version 3.2". Maven goes to the store (the internet), downloads exactly those items, and puts them in your pantry (the Maven Dependencies folder). You never manually download .jar files.

Section 11

What is pom.xml?

POM stands for Project Object Model. It is your project's config file — like package.json.

The pom.xml file is the single most important file in a Maven project. It tells Maven three things:

  1. Who is this project? (Group ID, Artifact ID, version)
  2. What libraries does it need? (dependencies)
  3. How should it be built? (plugins, packaging type)

Here is what the pom.xml looks like right now after creating the project:

pom.xml 
<project xmlns="http://maven.apache.org/POM/4.0.0">
  <modelVersion>4.0.0</modelVersion>

  <groupId>com.nimbus</groupId>
  <artifactId>nimbus-backend</artifactId>
  <version>0.0.1-SNAPSHOT</version>
  <packaging>war</packaging>

  <dependencies>
    <!-- Libraries you need go here -->
    <!-- Example: In Sprint 2, you will add the MySQL driver here -->
  </dependencies>
</project>
Line-by-line explanation
1
<project>
The root element. Everything about your project goes inside this tag.
2
<modelVersion>4.0.0</modelVersion>
The POM format version. Always 4.0.0. Never change this.
3
<groupId>com.nimbus</groupId>
Your organization/project identifier. Like a domain name in reverse.
4
<artifactId>nimbus-backend</artifactId>
Your project's name. This becomes the name of the final .war file.
5
<version>0.0.1-SNAPSHOT</version>
Project version. SNAPSHOT means "still in development." Like version 0.1.0-beta.
6
<packaging>war</packaging>
Tells Maven to package this as a WAR (Web Application Archive) for deployment on a web server.
7
<dependencies> ... </dependencies>
All your external libraries go between these tags. Each library is a <dependency> block with a groupId, artifactId, and version.
In Nimbus

In Sprint 2, you will add the MySQL JDBC driver as a dependency here. In Sprint 3, you will add servlet and JSP dependencies. In Sprint 4, Spring Boot. Each time, you add a few lines inside <dependencies> and press Alt+F5.

Section 12

What are Java Packages?

A package is literally just a folder for your Java files. That is all.

When you see com.nimbus.model, that is a package. Under the hood, it is just a folder structure:

com/
  nimbus/
    model/
      User.java
      Booking.java
      Issue.java
com.nimbus.model = a folder at com/nimbus/model/

Every Java file starts with a package line at the very top. This tells Java which folder the file belongs to:

User.java 
package com.nimbus.model;  // This file lives in com/nimbus/model/

public class User {
    // ...
}
The package line MUST match the folder

If your file says package com.nimbus.model; but the file is sitting in com/nimbus/service/, Eclipse will show an error. The package declaration and the actual folder location must always match.

Section 14

Complete Folder Structure

Here is everything your project will look like after setup. Create the resource folders now.

Create resource folders

  1. If src/main/resources does not exist: Right-click src/main → New → Folder → type resources
  2. Inside resources: Right-click → New → Folder → type static (this is where your frontend files will go)
  3. If src/main/webapp does not exist: Right-click src/main → New → Folder → type webapp
  4. Inside webapp: Right-click → New → Folder → type WEB-INF
  5. Inside WEB-INF: Right-click → New → Folder → type jsp

The complete project structure

nimbus-backend/
  pom.xml
  src/main/
    java/com/nimbus/
      model/              ← User.java, Booking.java, Issue.java
      dao/                ← UserDAO.java, BookingDAO.java, IssueDAO.java
      repository/         ← UserRepository.java, etc.
      service/            ← UserService.java, etc.
      servlet/            ← LoginServlet.java, etc.
      controller/         ← AuthController.java, etc.
      config/             ← CorsConfig.java, etc.
      dto/                ← LoginRequest.java, etc.
      util/               ← IdGenerator.java, etc.
      console/            ← ConsoleApp.java, ConsoleMenu.java
      NimbusApplication.java
    resources/
      application.properties
      schema.sql
      data.sql
      static/             ← Copy nimbus/ frontend here
    webapp/WEB-INF/jsp/
      login.jsp
      register.jsp
      etc.
The complete Nimbus backend structure — each folder has a clear purpose
Why so many folders?

It looks like a lot, but each folder has one job. model/ holds data shapes. dao/ talks to the database. service/ contains business rules. This separation is called layered architecture — it keeps your code organized as the project grows. You will not put files in all these folders right away. Sprint 2 only uses model/, dao/, service/, util/, and console/.

Section 15

Copy nimbus/ Files

Your Phase 1 frontend is ready to move into the Eclipse project.

Remember the nimbus/ folder you built in Phase 1? It has all your HTML, CSS, and JS files — login.html, register.html, the customer/, officer/, and support/ dashboards. Now you will copy them into the Eclipse project so everything lives in one place.

  1. Open File Explorer (Windows) or Finder (Mac)
  2. Navigate to your nimbus/ folder — the one with login.html, register.html, customer/, officer/, support/, css/, js/
  3. Select ALL files and folders inside nimbus/ (Ctrl+A)
  4. Copy them (Ctrl+C)
  5. Navigate to your Eclipse project folder: nimbus-backend/src/main/resources/static/
  6. Paste everything here (Ctrl+V)
  7. Go back to Eclipse and press F5 to refresh Package Explorer — you should see all the files appear under src/main/resources/static/
Section 16

The static/ Folder

Why static/? Because Spring Boot serves files from this folder automatically.

In Phase 3 and 4, when you run your backend server (Spring Boot), it will automatically look inside src/main/resources/static/ and serve any file there as a web page. No configuration needed.

So if you have:

static/
  login.html
  register.html
  customer/
    dashboard.html
  css/
    style.css
  js/
    auth.js
Files inside static/ are served as web pages

Then when your server runs at http://localhost:8080, you can access:

  • http://localhost:8080/login.html
  • http://localhost:8080/register.html
  • http://localhost:8080/customer/dashboard.html
Analogy

Think of the static/ folder as the "public" folder in a Node.js/Express project. Any file you put in there is accessible via a URL. HTML, CSS, JS, images — anything.

Section 17

Run Frontend from Eclipse

You will not run the frontend from Eclipse just yet — here is why.

Right now, your frontend uses LocalStorage to store data. It does not need a server — you can just open the HTML files in a browser directly (like you have been doing).

The frontend will start running through Eclipse in Phase 3, when you set up a Tomcat server or Spring Boot. At that point, your backend Java code will replace LocalStorage with a real database, and the frontend will talk to the backend through API calls.

For now, keep opening your HTML files directly in the browser. The files are in the Eclipse project for organization — they do not need to run through Eclipse yet.

In Nimbus

Sprint 2: You build a console-based Java app. The frontend stays unchanged.
Sprint 3: You replace LocalStorage with servlets + database. The frontend starts running through the server.
Sprint 4: The frontend talks to REST APIs. Full stack.
