Overview

What PRA Asks

PRA sometimes asks you to create a custom exception class and use it. Don't panic — it's one of the easiest things in the exam. The entire exception class is just 4 lines of code. The real challenge is knowing WHEN to throw it and how to catch it.

The pattern in PRA is always the same:

  1. Create a class that extends Exception
  2. Add a constructor that takes a String message
  3. Call super(message) inside the constructor
  4. That's it — 4 lines of code

Then in your main code:

  1. Calculate something (sum, product, comparison)
  2. Check a condition — if true, throw the exception
  3. Wrap it in try-catch in the main method
📚 Analogy

Think of a custom exception like a custom alarm. Java comes with built-in alarms (NullPointerException, ArrayIndexOutOfBoundsException). But PRA asks you to build your own alarm with a custom name and a custom message. When something bad happens, you trigger (throw) this alarm, and somewhere else in your code, you listen for it (catch) and handle it.

Step 1

Creating the Custom Exception Class

The simplest class you'll ever write. Literally 4 lines.

OddEvenException.java 
class OddEvenException extends Exception {
    public OddEvenException(String message) {
        super(message);
    }
}
Line-by-line explanation
1
class OddEvenException extends Exception {
extends Exception makes this a checked exception. Java forces you to either catch it or declare it with throws. The name can be anything the question asks for.
2
public OddEvenException(String message) {
Constructor that takes a message string. When you throw this exception, you pass in the error message here.
3
super(message);
Passes the message up to the parent Exception class. This is what makes e.getMessage() work later when you catch the exception.

Exception Class Hierarchy

ThrowableThe root of all errors
↓
ErrorSystem crashes (don't catch)
    
ExceptionYOUR code extends this
↓
RuntimeExceptionUnchecked (NullPointer, etc.)
    
OddEvenExceptionYour custom exception!
PRA always asks you to extend Exception (checked), never RuntimeException (unchecked)
🎯 PRA Tip

Always extend Exception, not RuntimeException. PRA wants checked exceptions. If you extend RuntimeException, the compiler won't force you to handle it, and you might miss the try-catch which loses marks.

Step 2

When to Throw

PRA gives you a condition: "If X happens, throw the exception with this message." The pattern is always: calculate something, compare, throw if the condition is met.

The Throw Statement

Throwing an Exception 
if (sumOfOdds > productOfEvens) {
    throw new OddEvenException("Sum is greater");
}

Breaking Down the Throw

Part What It Does
throw Java keyword that triggers the exception. Execution stops here and jumps to the nearest catch block.
new OddEvenException(...) Creates a new instance of your custom exception. It's an object, just like any other class.
"Sum is greater" The message string. This is what e.getMessage() returns later. Use the EXACT message from the question.
⚠ Common Mistake

Don't confuse throw (the action) with throws (the declaration). throw actually triggers the exception. throws goes in the method signature to warn callers. They're different keywords!

throw (action)

throw new OddEvenException("msg");

Used inside the method body

Actually triggers the exception

Execution stops here

throws (declaration)

public void method() throws OddEvenException

Used in the method signature

Just a warning: "this method might throw"

No exception is actually thrown here

Step 3

Try-Catch in Main

The main method is where you catch the exception and handle it. PRA always specifies the exact output format for the catch block.

Main Method with Try-Catch 
public static void main(String[] args) {
    try {
        // ... read input, do calculations ...

        if (sumOfOdds > productOfEvens) {
            throw new OddEvenException("Sum is greater");
        } else {
            System.out.println(productOfEvens);
        }
    } catch (OddEvenException e) {
        System.out.println("OddEvenException: " + e.getMessage());
    }
}

How Try-Catch Works

try block startsCode runs normally
→
Condition checkedif (sumOfOdds > productOfEvens)
↓
Condition FALSEprint result, skip catch
    
Condition TRUEthrow exception, jump to catch
↓
Program continues after try-catch
When throw executes, Java immediately jumps to the matching catch block — all remaining code in try is skipped
🎯 PRA Tip

The catch block print format is ALWAYS specified in the question. Common patterns:

  • "OddEvenException: " + e.getMessage()
  • "Exception: " + e.getMessage()
  • Just e.getMessage()

Copy the exact format from the question. One wrong space or missing colon = wrong answer.

Step 4

The throws Keyword

If your method can throw a checked exception but doesn't catch it itself, you MUST declare it in the method signature with throws. This tells the caller: "Hey, I might throw this exception — you need to handle it."

Method with throws Declaration 
public static void processNumbers(Collection<Integer> numbers) throws OddEvenException {
    // ... calculations ...
    if (sumOfOdds > productOfEvens) {
        throw new OddEvenException("Sum is greater");
    }
    // This method throws but doesn't catch
    // The caller (main) must catch it
}

When Do You Need throws?

Scenario Need throws? Why
Method throws AND catches the exception No The exception is handled internally
Method throws but does NOT catch Yes Caller needs to know this method can explode
main() method with try-catch No main catches it, so no one else needs to know
main() throws without catching Yes (but unusual in PRA) public static void main(String[] args) throws OddEvenException
💡 In PRA Context

Most PRA exception questions have you throw AND catch inside the same main method. You usually don't need throws on main. But if the question structures it as a separate method that throws, add throws ExceptionName to that method's signature.

Full Solution

Complete OddEven Exception Solution

This is the complete working solution from Mock 1 Q2. Read n integers, separate odds and evens, sum odds, multiply evens, compare and throw/print. Study this as your exception template.

Solution.java — Complete OddEven Exception 
import java.util.*;

// Step 1: Custom Exception Class
class OddEvenException extends Exception {
    public OddEvenException(String message) {
        super(message);
    }
}

public class Solution {
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);

        // Step 2: Read input
        int n = sc.nextInt();
        ArrayList<Integer> numbers = new ArrayList<>();
        for (int i = 0; i < n; i++) {
            numbers.add(sc.nextInt());
        }

        // Step 3: Separate odds and evens
        int sumOfOdds = 0;
        int productOfEvens = 1;   // start at 1 for multiplication!
        boolean hasEven = false;

        for (int num : numbers) {
            if (num % 2 != 0) {
                sumOfOdds += num;           // odd: add to sum
            } else {
                productOfEvens *= num;      // even: multiply
                hasEven = true;
            }
        }

        // If no evens found, product stays at 1 (initial)
        if (!hasEven) {
            productOfEvens = 0;
        }

        // Step 4: Compare and throw or print
        try {
            if (sumOfOdds > productOfEvens) {
                throw new OddEvenException("Sum of odd numbers is greater");
            } else {
                System.out.println(productOfEvens);
            }
        } catch (OddEvenException e) {
            System.out.println("OddEvenException: " + e.getMessage());
        }
    }
}
Key decisions explained
1
int productOfEvens = 1;
Initialize product at 1, not 0. If you start at 0, multiplying anything by 0 gives 0. Product starts at 1 because 1 is the identity element for multiplication (1 * x = x).
2
boolean hasEven = false;
Edge case handling: if there are NO even numbers, productOfEvens stays at 1 (initial value). We use this flag to detect that case and set product to 0 instead.
3
num % 2 != 0
Modulo operator: if a number divided by 2 has a remainder, it's odd. If remainder is 0, it's even.
4
"OddEvenException: " + e.getMessage()
Exact output format from the question. Notice the colon, space, then the message. Always copy this format precisely.

Checklist

Exception Checklist

Before submitting your exception solution on HackerRank, verify every item on this list. Missing any one of these is the difference between full marks and zero.

🎯 Pre-Submit Checklist
  • Did you extend Exception (not RuntimeException)?
  • Did you call super(message) in the constructor?
  • Did you add try-catch in main?
  • Does your catch print in the EXACT format the question asks?
  • Is the exception class name exactly what the question specifies?
  • Is the error message string exactly what the question specifies?
  • Product initialized at 1 (not 0) if you're multiplying?
  • Edge case handled for empty categories (no odds / no evens)?

📋 Universal Exception Template

Copy this template for any PRA exception question. Just change the class name, condition, and message.

Template — Custom Exception 
// 1. Exception class (change name)
class YourException extends Exception {
    public YourException(String message) {
        super(message);
    }
}

// 2. In main method:
try {
    // ... your calculations ...
    if (/* bad condition */) {
        throw new YourException("Your error message");
    } else {
        System.out.println(/* normal result */);
    }
} catch (YourException e) {
    System.out.println("YourException: " + e.getMessage());
}
